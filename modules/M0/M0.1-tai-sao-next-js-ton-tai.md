# M0.1 — Tại Sao Next.js Tồn Tại

> **Senior Mindset:** Trước khi học một framework, câu hỏi quan trọng nhất không phải là _"Nó hoạt động thế nào?"_ mà là _"Vấn đề gì nó đang giải quyết?"_ — và quan trọng hơn: _"Vấn đề đó có phải vấn đề của tôi không?"_



## 1. Lesson Goal

Sau bài học này, bạn phải có khả năng:

- **Giải thích** tại sao React SPA thuần túy không đủ cho production web applications cần SEO hoặc performance
- **Nhận diện** 6 pain points cốt lõi của manual SSR mà Next.js ra đời để giải quyết
- **Phân biệt** khi nào Next.js là lựa chọn đúng và khi nào nó là over-engineering
- **Trả lời** câu hỏi phỏng vấn _"Tại sao bạn chọn Next.js thay vì React thuần?"_ với chiều sâu thực chất, không phải thuộc lòng bullet points



## 2. Evolution Context

### Bản đồ lịch sử (2010 → 2016)

Để hiểu Next.js, bạn cần hiểu **cơn đau** mà các engineer đã sống qua trước nó. Đây không phải lịch sử để thuộc. Đây là ngữ cảnh để mọi quyết định thiết kế của Next.js trở nên _có lý_.

```
[2010–2013]             [2013–2016]              [2016]
Server-rendered HTML  →   React SPA          →   "Chúng ta cần thứ gì đó tốt hơn"
PHP / Rails / Django      CRA / Webpack            Next.js 1.0 (Oct 25, 2016)
         ↓                      ↓                           ↓
✅ Good SEO              ❌ Terrible SEO           ✅ Pre-rendered HTML = SEO
✅ Fast first paint      ❌ Blank screen 3–5s      ✅ Content visible ngay lập tức
❌ Slow dynamic UI       ✅ Instant interactions   ✅ Instant interactions
❌ Full page reloads     ✅ SPA navigation         ✅ SPA navigation
❌ Tight server coupling ✅ Decoupled frontend     ✅ Decoupled + fullstack flexible
❌ N/A                   ❌ DIY SSR = hell         ✅ Zero-config SSR built-in
```

Đây là một **zigzag**, không phải tiến hóa thẳng. React SPA giải quyết vấn đề của PHP nhưng tạo ra vấn đề mới. Next.js là nỗ lực lấy lại những gì tốt nhất từ cả hai thế giới.

**Câu hỏi cốt lõi:** Nếu vấn đề đã tồn tại từ 2013, tại sao đến 2016 mới có Next.js? _(Bài M0.2 sẽ phân tích toàn bộ timeline này)_



## 3. Mental Model

### Framework = Kết tinh của nỗi đau tập thể

> _"Every framework is someone else's answer to a problem they got tired of solving manually."_

Next.js không phải "React với thêm tính năng". Next.js là **sự đồng thuận được code hóa** về cách giải quyết những vấn đề mà hàng ngàn team đã tự giải quyết — mỗi team theo một cách khác nhau, tốn thời gian khác nhau, và tạo ra bugs khác nhau.

```
Vấn đề lặp đi lặp lại ở hàng ngàn codebases
          ↓
Giải pháp tốt nhất được tìm ra qua thử-và-sai tập thể
          ↓
Được đóng gói thành conventions + tooling
          ↓
Framework
```

**Implication thực tế:** Khi bạn thấy một tính năng trong Next.js, câu hỏi đầu tiên phải là:

> _"Pain point nào đã dẫn đến tính năng này?"_

Không phải: _"Tính năng này hoạt động thế nào?"_

Hiểu pain point trước → hiểu tính năng sâu hơn → biết khi nào không dùng nó → đây là điểm khác biệt giữa mid-level và senior.



## 4. Core Concepts — 6 Pain Points Thực Sự

### Pain Point #1: The Empty HTML Problem (SEO Death)

Đây là pain point số một khiến Next.js ra đời.

**Kịch bản thực tế:** Bạn build một e-commerce site bằng React + Create React App. Sản phẩm ra mắt tháng 1. Tháng 6, Google Search Console báo 0 pages indexed. Organic traffic giảm 80%.

**Nguyên nhân:** CSR trả về HTML shell rỗng. Content chỉ tồn tại sau khi JavaScript execute xong.

```html
<!-- Đây là thứ Googlebot nhìn thấy khi crawl CSR React app -->
<!-- (và vẫn còn đúng cho nhiều bots ngày nay) -->
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <title>My E-commerce Store</title>
    <!-- ❌ Không có meta description -->
    <!-- ❌ Không có Open Graph tags → Facebook/Twitter share không có preview -->
    <!-- ❌ Không có structured data → Google không hiểu đây là product page -->
  </head>
  <body>
    <div id="root">
      <!-- ⚠️ TRỐNG HOÀN TOÀN -->
      <!-- Toàn bộ content nằm trong bundle.js -->
      <!-- Bot phải execute JavaScript MỚI thấy được content -->
      <!-- Và hầu hết bots không làm vậy, hoặc không chờ đủ lâu -->
    </div>
    <script src="/static/js/bundle.js"></script>
    <!-- 2–4MB JavaScript phải download + parse + execute -->
    <!-- trước khi user hoặc bot thấy bất cứ thứ gì có ý nghĩa -->
  </body>
</html>
```

**Hậu quả cascade:**
- Google rank sản phẩm của bạn về 0 (không index được)
- Facebook/Slack share preview không có ảnh, không có title, không có description
- `curl https://yoursite.com/products/123` trả về trang trống → mọi bot đều thấy trang trống
- Bing, DuckDuckGo, LinkedIn scraper: tất cả gặp cùng vấn đề

### Pain Point #3: Manual SSR — Khi Bạn Cố Tự Giải Quyết

Khi các team nhận ra vấn đề CSR, nhiều người cố tự implement SSR với Express + ReactDOMServer. Đây là khi mọi thứ trở nên **thực sự tồi tệ**:

```javascript
// server.js — "DIY SSR" với Express + React (2014–2016)
// Code này THỰC SỰ đã tồn tại trong nhiều production codebases

const express = require('express');
const React = require('react');
const ReactDOMServer = require('react-dom/server');

// Phải tự import từng component (không có code splitting)
const App = require('./src/App');

const app = express();

app.get('*', async (req, res) => {
  // ❌ Vấn đề 1: Không có convention nào cho "fetch data trước khi render"
  // Mỗi engineer tự quyết định cách implement
  // Mỗi route phải được handle thủ công
  let initialData = {};

  if (req.path === '/') {
    initialData.home = await fetchHomeData();
  } else if (req.path.startsWith('/products/')) {
    const productId = req.path.split('/')[2]; // ❌ Parsing thủ công, dễ sai
    initialData.product = await fetchProduct(productId);
  } else if (req.path.startsWith('/category/')) {
    const slug = req.path.split('/')[2];
    initialData.category = await fetchCategory(slug);
  }
  // ...tiếp tục với hàng chục routes khác
  // ❌ Nếu quên một route → server trả về trang thiếu data, không có error rõ ràng

  // ❌ Vấn đề 2: renderToString là SYNCHRONOUS và BLOCKING
  // Nó chiếm toàn bộ Node.js event loop trong suốt thời gian render
  // 1000 concurrent users render cùng lúc → server không responsive
  // Không có timeout, không có streaming
  const html = ReactDOMServer.renderToString(
    React.createElement(App, {
      data: initialData,
      url: req.url, // ❌ Phải tự truyền URL để React Router hoạt động server-side
    })
  );

  // ❌ Vấn đề 3: Phải tự "inject" data để client biết hydrate với data gì
  // Không có cơ chế official. Mỗi team dùng convention riêng.
  // window.__INITIAL_DATA__ là pattern phổ biến nhất — không có gì enforce nó
  const safeData = JSON.stringify(initialData)
    .replace(/</g, '\\u003c') // ❌ Phải tự escape để tránh XSS
    .replace(/>/g, '\\u003e') // ❌ Nếu quên → security vulnerability
    .replace(/&/g, '\\u0026');

  res.send(`
    <!DOCTYPE html>
    <html>
      <head>
        <title>My App</title>
        <!-- ❌ Meta tags phải hardcode hoặc inject thủ công -->
      </head>
      <body>
        <div id="root">${html}</div>
        <script>
          window.__INITIAL_DATA__ = ${safeData};
        </script>
        <!-- ❌ Tất cả JS trong 1 bundle duy nhất, không có code splitting -->
        <script src="/bundle.js"></script>
      </body>
    </html>
  `);
});
```

```javascript
// client.js — Phía client phải "biết" convention mà server đã dùng
// ❌ Không ai enforce điều này
// Người mới join team phải đọc code để hiểu — nếu có tài liệu thì may

const initialData = window.__INITIAL_DATA__;
// ❌ Nếu server quên inject data → initialData là undefined → crash

ReactDOM.hydrate(
  // ❌ Component phải được setup để chấp nhận data theo format này
  // Không có type checking, không có validation
  <App data={initialData} url={window.location.pathname} />,
  document.getElementById('root')
);
```

### Pain Point #4: Routing Conflict — Hai Source of Truth

```
Khi dùng Express + React Router cùng nhau:

Server routes (Express):         Client routes (React Router):
────────────────────────         ────────────────────────────
app.get('/products/:id', ...)    <Route path="/products/:id" />
app.get('/category/:slug', ...)  <Route path="/category/:slug" />
app.get('/about', ...)           <Route path="/about" />

❌ Hai nơi hoàn toàn riêng biệt định nghĩa routes
❌ Phải sync thủ công mỗi khi thêm/sửa route
❌ Thêm route trong React Router nhưng quên Express:
   → Navigation từ trang khác: HOẠT ĐỘNG
   → Direct URL hoặc F5: 404 NOT FOUND
   → Bug cực kỳ khó debug vì "nó hoạt động với tôi"
❌ Thêm route trong Express nhưng quên React Router:
   → Page load lần đầu: HOẠT ĐỘNG
   → Navigate bằng React Router: 404 hoặc render sai component
```

**Bug kinh điển nhất của era này:** App hoạt động hoàn hảo khi navigate từ trang này sang trang kia. Nhưng khi user bookmark URL và mở lại → 404. Tại sao? Server không biết route đó. Chỉ client-side router biết.

### Pain Point #5: Code Splitting — Mọi Thứ Trong 1 Bundle

```
CRA default: TẤT CẢ code → 1 file bundle.js

Cấu trúc app:
├── /home           ┐
├── /products       │ Tất cả code của tất cả routes
├── /category       │ đều nằm trong
├── /checkout       │ MỘT file bundle.js duy nhất
├── /admin          │ (thường 2–5MB sau một vài tháng develop)
├── /profile        │
└── /settings       ┘

Hậu quả:
User vào /home → phải download code của /admin (mà họ không bao giờ dùng)
User vào /products → phải download code của /checkout (chưa cần dùng)

Webpack lazy loading thủ công:
const Admin = React.lazy(() => import('./pages/Admin'));
// ❌ Phải tự làm cho TỪNG route
// ❌ Phải config Webpack đúng cách (không trivial)
// ❌ Dễ bị miss, không có gì nhắc nhở
// ❌ Mỗi người setup theo cách khác nhau
```

Đây là pain point về **thời gian và sự không nhất quán**, không phải về technical correctness.

```bash

Step 1:  Cấu hình Webpack cho CLIENT bundle

Step 9:  Setup production build (minification, asset hashing, source maps)
Step 10: Handle SSR error boundaries (server crash → thế nào?)
Step 11: Cấu hình TypeScript (nếu dùng) cho dual environment
Step 12: Setup CI/CD pipeline hiểu cả server lẫn client bundle

Thời gian thực tế: 1–2 tuần chỉ để có boilerplate chạy được.

Và quan trọng hơn: KHÔNG AI LÀM GIỐNG AI.
```

## 6. Modern Approach — Next.js Giải Quyết Thế Nào

Mỗi pain point có một solution cụ thể được baked in:

```
Pain Point                    → Next.js Solution
──────────────────────────────────────────────────────────────────────
#1 Empty HTML (SEO)          → Pre-render HTML trên server/build time
#2 Blank screen 3.5s         → HTML sẵn sàng trước khi JS load xong
#3 Manual SSR complexity     → Built-in SSR, zero config, conventions
#4 Routing conflict          → File-based routing (1 source of truth)
#5 1 giant bundle            → Automatic per-page code splitting
#6 DX hell (2 tuần setup)    → 5 phút với npx create-next-app@latest
```

```javascript
// pages/products/[id].js — Next.js Pages Router
// 6 pain points biến mất trong 1 file

// ✅ Convention rõ ràng: Next.js tự biết gọi hàm này trước khi render
// ✅ Params được parse tự động từ filename [id].js
// ✅ Data được inject vào props tự động (không cần window.__INITIAL_DATA__)
export async function getServerSideProps({ params }) {
  const product = await fetchProduct(params.id);

  if (!product) {
    return { notFound: true }; // ✅ 404 handling built-in
  }

  return { props: { product } };
}

// ✅ Component này được render trên server → HTML đầy đủ → SEO hoạt động
// ✅ Code của page này được split tự động → bundle nhỏ hơn
// ✅ Chỉ 1 file = 1 route (server + client cùng nhau, không conflict)
export default function ProductPage({ product }) {
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.price}</p>
    </div>
  );
}
```

**Điều quan trọng nhất Next.js làm:** Đưa ra **conventions** — cách làm được đồng thuận, không cần mỗi team tự phát minh lại bánh xe. Khi một engineer từ công ty A join công ty B, họ nhận ra ngay codebase vì cùng dùng Next.js conventions.



## 7. Production Scenario

### Case Study: E-commerce mất 60% organic traffic vì CSR

**Bối cảnh:** 2017. Một startup e-commerce build platform bằng React CRA. Organic traffic từ Google Search chiếm 55% tổng traffic.

**Timeline sự cố:**

- **Tháng 1:** Launch. Team hài lòng với speed development.
- **Tháng 3:** Organic traffic bắt đầu giảm nhẹ. Không ai để ý.
- **Tháng 6:** Organic traffic giảm 70%. Google Search Console: phần lớn product pages "Discovered – currently not indexed".
- **Tháng 7:** Emergency all-hands. CTO yêu cầu fix ASAP.

**Investigation:**

```bash
# Bước 1: Confirm hypothesis
curl -A "Googlebot/2.1" https://store.example.com/products/bestseller-2024

# Output thực tế:
# <!DOCTYPE html><html><head><title>Store</title></head>
# <body><div id="root"></div><script src="/bundle.js"></script></body></html>
#
# → Googlebot thấy TRANG TRỐNG. Không có product name, không có price, không có description.

# Bước 2: Kiểm tra social preview
# Chia sẻ URL lên Slack/Facebook → Không có thumbnail, không có title
# og:title, og:description, og:image đều trống

# Bước 3: Lighthouse SEO audit
# SEO score: 42/100
# "Document does not have a meta description"
# "Links are not crawlable"
```

**Root cause:** Toàn bộ React app là CSR. Không có pre-rendering. Googlebot crawl và chỉ thấy HTML shell.

**Solution:** Migrate product pages và category pages sang Next.js với `getServerSideProps`.

**Kết quả sau 3 tháng:** Organic traffic phục hồi về 85% mức cũ, rồi vượt lên 120% (vì các page giờ được index đúng).

**Lesson cho Engineer:** Khi nhận task _"improve SEO"_, câu hỏi đầu tiên phải là: _"App có pre-render HTML không?"_ Đây là foundation — không có nó, mọi SEO optimization khác đều vô nghĩa.

### Debugging Hydration Mismatch (Pain Point của DIY SSR)

Lỗi này là "ác mộng" của era DIY SSR:

```
Warning: Text content did not match. Server: "Monday" Client: "Tuesday"
Warning: Prop `className` did not match. Server: "dark" Client: "light"
```

```javascript
// Nguyên nhân phổ biến #1: Date/Time
// Server chạy UTC, client chạy timezone local
function BadDateComponent() {
  return <div>{new Date().toLocaleDateString()}</div>;
  // ❌ Server: "1/1/2024" (UTC)
  // ❌ Client: "1/2/2024" (ICT +7) → Hydration mismatch
}

// Nguyên nhân phổ biến #2: Browser APIs không tồn tại trên server
function BadThemeComponent() {
  // ❌ localStorage, window, document không tồn tại trong Node.js
  const theme = localStorage.getItem('theme') || 'light';
  return <div className={theme}>Content</div>;
  // Server crash với: "ReferenceError: localStorage is not defined"
}

// Nguyên nhân phổ biến #3: Math.random() và IDs
function BadListItem() {
  // ❌ Server generate ID = "abc123"
  // ❌ Client generate ID = "xyz789"
  return <li id={Math.random().toString(36)}>Item</li>;
}

// Nguyên nhân phổ biến #4: User-agent based rendering
function BadResponsiveComponent() {
  // ❌ Server không biết screen width của client
  const isMobile = window.innerWidth < 768;
  return isMobile ? <MobileView /> : <DesktopView />;
}
```

### Mindset khi tiếp nhận Legacy CSR Codebase

Đây không phải bài dạy migration chi tiết (xem M2.10 và M17). Đây là **tư duy** của một engineer nhìn vào legacy code.

**Checklist 5 phút khi tiếp nhận codebase mới:**

```javascript
// Nhìn vào những thứ này theo thứ tự:

// 1. package.json — Era indicator
{
  "next": "xxx"            // → Next.js project, xem version để biết era
  "react-scripts": "xxx"  // → CRA, likely CSR-only
  "express": "xxx"         // → Có server tự viết, có thể là DIY SSR
}

// 2. Cấu trúc thư mục — Convention indicator
pages/          // → Next.js Pages Router (era 2016–2022)
app/            // → Next.js App Router (era 2023+)
src/pages/      // → Có thể là CRA với react-router
server.js       // → DIY SSR hoặc custom server

// 3. index.html hoặc document.js
// <div id="root"></div> trống → CSR không có SSR
// <div id="root">${html}</div> → DIY SSR với template strings

// 4. Data fetching pattern
getInitialProps   // → Next.js pre-13, universal (server + client)
getServerSideProps // → Next.js Pages Router, server-only SSR
getStaticProps    // → Next.js Pages Router, build-time SSG
fetch() trong RSC  // → Next.js App Router, modern

// 5. Test nhanh nhất:
// curl <url> → thấy content → SSR/SSG
// curl <url> → thấy trống → CSR
```

**Câu hỏi trước khi propose migration:**

```
❓ Migration có giải quyết được business problem không?
   (Nếu app là internal dashboard → SEO không quan trọng → không cần migrate)

❓ Risk vs reward?
   (Migration lớn = risk cao. Cần proof of concept với 1–2 pages trước)

❓ Timeline và resources?
   (Incremental migration luôn tốt hơn big-bang rewrite)

❓ Dependencies có compat issues không?
   (Một số libraries không work với SSR)
```



## 10. Architecture Notes

### Khi nào KHÔNG cần Next.js

Đây là điều quan trọng nhất mà các tutorial thường bỏ qua:

```
✅ Dùng Next.js khi:
   - Public-facing pages cần SEO (landing pages, blogs, e-commerce, news)
   - App cần performance tốt trên mobile/slow networks
   - Team lớn cần conventions thống nhất
   - Cần hybrid: một số pages static, một số pages dynamic
   - Fullstack: cần cả API routes và UI trong cùng 1 project

❌ Không dùng Next.js khi:
   - Internal dashboard (chỉ dùng sau login, SEO không quan trọng)
     → Vite + React: setup đơn giản hơn, HMR nhanh hơn, build nhanh hơn

   - Heavy real-time app (live trading, collaborative editing, live chat)
     → WebSocket server tùy chỉnh hoặc framework chuyên biệt phù hợp hơn

   - Simple content site (blog cá nhân, docs, portfolio)
     → Astro: zero JS by default, cực kỳ nhẹ, tốt cho content-heavy sites

   - Mobile app
     → React Native, Expo — Next.js là web framework

   - Team deadline gấp, không ai có Next.js experience
     → Learning curve của App Router không nên underestimate
```

**Trade-off thực tế của Next.js:**

```
✅ Pros:
   - Pre-rendering out of the box
   - File-based routing (zero config)
   - Automatic code splitting
   - Image, Font optimization built-in
   - API routes (fullstack trong 1 repo)
   - Vercel ecosystem nếu deploy Vercel
   - Community lớn, ecosystem trưởng thành

❌ Cons:
   - Build time lâu hơn Vite cho dev
   - Complexity cao hơn CRA cho apps đơn giản
   - App Router có learning curve đáng kể (Server Components, caching)
   - Tie-in với Vercel (mặc dù self-hosting được)
   - Debugging SSR issues phức tạp hơn CSR
   - Một số libraries không work với SSR/RSC
```

**Senior Engineer Mindset:** Next.js là công cụ xuất sắc, không phải công cụ đúng cho mọi tình huống. Biết khi nào _không_ dùng nó quan trọng không kém biết cách dùng nó.



## 11. Interview Questions

### Junior Level

**Q1: React và Next.js khác nhau thế nào?**

<details>
<summary>Đáp án</summary>

React là một UI library — nó chỉ xử lý rendering UI. Next.js là một fullstack framework xây dựng trên React, thêm vào:

- **Server-side rendering (SSR):** HTML được tạo trên server, tốt cho SEO
- **Static site generation (SSG):** HTML được tạo lúc build time, cực nhanh
- **File-based routing:** Không cần setup react-router thủ công
- **API routes:** Viết backend endpoints trong cùng project
- **Automatic code splitting:** Mỗi page chỉ load code của chính nó
- **Image/Font optimization:** Built-in, không cần config

Analogy: React là động cơ xe hơi. Next.js là chiếc xe hoàn chỉnh — có bánh, có ghế, có điều hòa — sẵn sàng lăn bánh. Bạn vẫn cần biết động cơ hoạt động thế nào, nhưng không phải tự lắp ráp từng bộ phận.

</details>

---

**Q2: Tại sao React SPA có vấn đề SEO?**

<details>
<summary>Đáp án</summary>

React SPA (Single Page Application) trả về HTML shell gần như trống từ server. Tất cả content được render bởi JavaScript sau khi browser download và execute bundle.

Vấn đề: Search engine crawlers, social media bots (Facebook, Twitter), và hầu hết automated tools không execute JavaScript — hoặc không chờ đủ lâu để JavaScript render xong. Kết quả là chúng chỉ thấy trang trống.

**Test nhanh:**
```bash
curl https://yoursite.com
# CSR app: <div id="root"></div> — Trống
# Next.js app: <div id="root"><h1>Actual Content</h1>...</div>
```

**Hệ quả thực tế:**
- Pages không được index hoặc index với content sai
- Social share không có preview (no OG image, no title)
- Organic traffic giảm mạnh

</details>

---

### Mid Level

**Q3: Giải thích "hydration" trong context của SSR. Tại sao nó phức tạp?**

<details>
<summary>Đáp án</summary>

**Hydration là gì:**

Khi dùng SSR, server render HTML đầy đủ và gửi về client. Client nhận HTML này và hiển thị ngay (user thấy content). Sau đó JavaScript load xong và React cần "attach" vào HTML đã có — thêm event listeners, kết nối state — mà KHÔNG re-render lại từ đầu (vì sẽ gây flicker). Quá trình này gọi là hydration.

**Tại sao phức tạp:**

**1. Mismatch risk — Nguyên nhân số 1 gây bugs:**
HTML server render phải giống HỆTT với gì React sẽ render trên client. Nếu khác → React báo lỗi hoặc re-render toàn bộ (mất hết lợi ích SSR).

Common mismatches:
- Dates: Server (UTC) vs Client (local timezone)
- `window`/`localStorage`/`document` không tồn tại trên server
- `Math.random()` cho kết quả khác nhau mỗi lần

**2. "Uncanny Valley" — UX problem:**
Sau khi HTML render, trước khi JS load xong, page trông có vẻ interactive nhưng thực ra không hoạt động. User click button → không có gì xảy ra. Thời gian này gọi là "Time to Interactive gap".

**3. Complexity trong DIY SSR:**
Phải tự đảm bảo data server dùng để render giống hệt data client dùng để hydrate. Trong Next.js, framework handle điều này tự động qua `getServerSideProps`/`getStaticProps` → component props.

</details>

---

**Q4: Một team quyết định tự implement SSR với Express + ReactDOMServer thay vì dùng Next.js. Bạn thấy rủi ro gì?**

<details>
<summary>Đáp án</summary>

Đây là câu hỏi về trade-offs, không có đáp án duy nhất. Tôi sẽ phân tích theo nhiều chiều:

**Technical risks:**

- `renderToString` là synchronous, block Node.js event loop trong suốt thời gian render một page. Với high concurrency → performance bottleneck nghiêm trọng
- Phải tự manage routing giữa Express và React Router → dễ desync, gây 404 khi direct access
- Data injection thủ công (`window.__INITIAL_DATA__`) → XSS risk nếu không escape đúng
- Webpack dual config (server bundle + client bundle) → phức tạp, error-prone
- Không có official solution cho code splitting trong SSR context

**Team và operational risks:**

- 1–2 tuần setup vs 5 phút với `create-next-app`
- Engineer mới join phải học custom infrastructure, không có docs chuẩn
- Không có community support khi gặp vấn đề
- Security updates và performance improvements phải tự implement

**Khi nào DIY có thể hợp lý (hiếm):**

- App có rendering requirements đặc biệt mà Next.js không support
- Team có deep expertise và dedicated bandwidth để maintain
- Cần control hoàn toàn rendering pipeline vì compliance/security requirements

**Câu trả lời thực tế của Senior Engineer:**

"Trong 95% cases, tự build SSR infrastructure là premature optimization và tạo ra technical debt đáng kể. Next.js đã giải quyết những vấn đề này qua nhiều năm với battle-tested solutions, cộng đồng lớn, và active maintenance. Proposal của tôi: dùng Next.js, và chỉ xem xét DIY nếu có requirements cụ thể, đã được document rõ ràng, mà Next.js chứng minh được là không thể đáp ứng."

</details>

---

### Senior Level

**Q5: Bạn join một team đang dùng CRA cho public-facing e-commerce site với 1M monthly visitors. Organic traffic giảm 40% trong 6 tháng. Team đề xuất migrate sang Next.js. Bạn sẽ approach thế nào?**

<details>
<summary>Đáp án</summary>

**Bước 1: Diagnose trước, giả định sau**

```bash
# Confirm hypothesis: CSR có phải nguyên nhân không?
curl https://store.example.com/products/top-seller | grep -i "product-name"

# Check Google Search Console:
# - "Coverage" → bao nhiêu pages "Discovered – currently not indexed"?
# - "URL Inspection" → Google thấy gì khi crawl?

# Lighthouse audit trên 3–5 representative pages
# Core Web Vitals: LCP, CLS, FID/INP
```

Traffic giảm có thể có nhiều nguyên nhân: algorithm update, technical SEO issues, competitor, content quality. Không nhảy thẳng vào "migrate sang Next.js" nếu chưa confirm CSR là root cause.

**Bước 2: Scope assessment nếu CSR confirmed**

- Bao nhiêu unique routes? (20? 200? 2000?)
- Route nào critical cho SEO? (Product pages, Category pages, Landing pages)
- Route nào không cần SSR? (Cart, Checkout, User profile, Admin)
- Dependencies phức tạp? (State management library, Auth solution, UI components)
- Team Next.js experience level?

**Bước 3: Propose incremental migration, không big-bang rewrite**

```
Phase 1 (2 tuần): Setup Next.js, migrate 5 highest-traffic product pages
                  → Measure SEO impact sau 4 tuần
Phase 2 (4 tuần): Migrate toàn bộ product catalog + category pages
                  → Monitor metrics
Phase 3 (6–8 tuần): Migrate remaining pages, retire CRA
```

Strangler Fig pattern: Next.js và CRA có thể chạy song song. Reverse proxy route từng path sang đúng app.

**Bước 4: Define success metrics trước khi bắt đầu**

- Google Search Console: indexing rate
- Google Analytics: organic sessions
- Core Web Vitals: LCP < 2.5s, CLS < 0.1
- TTFB (Time to First Byte)

**Bước 5: Risk identification**

- Breaking URL changes → 301 redirects
- Third-party libraries incompatible với SSR
- Auth flow cần review cho SSR context
- Data fetching patterns phải refactor

**Senior mindset được thể hiện ở đây:** Không giả định, đo lường trước. Không big-bang, incremental. Không chỉ technical solution, còn cần business metrics để track ROI của migration effort.

</details>

## 13. Summary

```
Next.js ra đời tháng 10/2016 để giải quyết 6 pain points cốt lõi
của React SPA và DIY SSR:

#1 Empty HTML        → ✅ Pre-rendered HTML (SEO hoạt động)
#2 Blank screen 3.5s → ✅ Content visible ngay lập tức
#3 Manual SSR hell   → ✅ Zero-config SSR với conventions rõ ràng
#4 Routing conflict  → ✅ File-based routing (1 source of truth)
#5 1 giant bundle    → ✅ Automatic per-page code splitting
#6 DX hell           → ✅ 5 phút setup, conventions nhất quán
```

**Takeaway #1 — Framework là convention, không phải magic:**
Next.js không làm gì mà bạn không thể tự làm. Nó làm những thứ đó theo một cách chuẩn, được đồng thuận, đủ tốt cho 95% use cases.

**Takeaway #2 — Hiểu pain point = Hiểu sâu hơn:**
Mỗi tính năng trong Next.js tồn tại vì một lý do cụ thể. Khi bạn hiểu lý do đó, bạn sẽ biết khi nào dùng, khi nào không dùng, và khi nào cần workaround.

**Takeaway #3 — Công cụ đúng, không phải công cụ tốt nhất:**
Next.js là công cụ xuất sắc — khi dùng đúng chỗ. Internal dashboard không cần Next.js. Simple blog có thể dùng Astro. Heavy real-time app cần kiến trúc khác. Senior engineer biết phân biệt.
