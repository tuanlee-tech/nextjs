# [M0.1] Next.js trong bức tranh lớn — Why Exists?

> 🎯 **Lesson Goal:** Hiểu tại sao React cần meta-framework, phân biệt được 4 chiến lược rendering (CSR/SSR/SSG/ISR) ở mức conceptual, nhận diện anti-pattern "Next.js = backend" và "dùng là tự động fast", và biết cách đọc một codebase legacy React SPA để lên kế hoạch migration.

---

## 1. 🕰️ Evolution Context

### Pain Point gốc rễ: React không đủ để làm production web

React tự định nghĩa là _"library for web and native user interfaces"_. Đúng nghĩa đen — nó chỉ là library. Không có routing, không có SSR, không có image optimization, không có font loading strategy, không có build pipeline chuẩn hóa.

Để ship một production web app với React thuần, team phải tự tổ hợp:

- **Routing**: React Router (hoặc tự viết)
- **SSR**: Custom Express server + `renderToString()`
- **Bundler**: Webpack config tự write (hoặc CRA nếu chấp nhận được giới hạn của nó)
- **SEO / Meta**: React Helmet
- **Code splitting**: Tự config

Mỗi thứ một tool riêng, một config riêng, một đội phải maintain riêng. Vấn đề không phải là không làm được. Vấn đề là **mỗi team lại reinvent the wheel theo cách khác nhau** — và một số wheel được reinvent sai.

### SPA + CSR: SEO là điểm chết

SPA (Single Page App) với CSR (Client-Side Rendering) có một vấn đề cơ bản: **HTML gần như rỗng khi server trả về**.

```
GET /blog/nextjs-la-gi
← <html><body><div id="root"></div></body></html>  +  bundle.js (3MB)
```

Browser nhận HTML rỗng, tải JavaScript, chạy JavaScript, rồi mới render content. Google Crawler thấy HTML rỗng — hoặc phải tự render JavaScript để thấy nội dung, và điều đó không phải lúc nào cũng xảy ra.

Hệ quả trực tiếp:

- **FCP (First Contentful Paint)** chậm — người dùng nhìn màn hình trắng chờ JS load xong
- **LCP (Largest Contentful Paint)** chậm theo — content chính xuất hiện muộn
- **SEO không ổn định** — indexing phụ thuộc vào Google có render JS không

### Các giải pháp cũ — và tại sao thất bại

**Custom SSR bằng Express + `renderToString`:**
Setup nặng, khó maintain. Không có chuẩn hóa cho caching, code splitting, error handling, streaming, hay prefetch. Mỗi team tự quyết định cách handle — mỗi team có bug riêng. Hydration mismatch (server render khác client render) là lỗi phổ biến và khó debug.

**SSG bằng Gatsby hoặc custom build scripts:**
Build time tăng phi mã khi số trang lớn. Site có 50,000 product pages phải rebuild toàn bộ mỗi lần update 1 sản phẩm. Không có ISR — không có cách update nội dung mà không trigger full rebuild.

**SPA bằng CRA + React Router + React Helmet:**
Chỉ là CSR. React Helmet chỉ inject meta tags trên client sau khi JavaScript đã chạy — crawler vẫn thấy HTML rỗng. Performance kém trên content-heavy sites. Không có server data fetching.

### Timeline Next.js — 30 giây

| Version / Năm      | Thay đổi quan trọng                                                                                                              |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **2016**            | Next.js ra mắt. SSR-first, file-based routing (Pages Router). Giải quyết được custom SSR complexity.                            |
| **v9.3 (2019)**     | SSG stable. Hybrid: một app có thể mix SSR + SSG.                                                                               |
| **v9.5 (2020)**     | ISR stable — update static page sau build mà không rebuild toàn bộ.                                                              |
| **v12.1 (2022)**    | On-demand ISR — revalidate theo trigger (webhook), không chờ timeout.                                                            |
| **v13–13.4 (2022–2023)** | App Router beta → stable. React Server Components mặc định. SSR đổi nghĩa từ "page-level per-request" sang "component-level, có thể cache". |
| **v16 (2025)**      | Turbopack mặc định, Cache Components / PPR (Partial Prerendering), explicit caching, `proxy.ts` thay `middleware.ts`.            |

> ⚠️ **Timeline note:** On-demand ISR ra mắt ở v12.1 (tháng 3/2022), không phải 2020. ISR *concept* lần đầu ở v9.5 (2020). Tài liệu cũ thường nhầm hai mốc này — đọc cẩn thận khi research.

## 2. 🧠 Mental Model

### Next.js là orchestrator, không phải replacement

[MODEL] Next.js không replace React. Next.js *orchestrate* React cùng với nhiều thứ khác:

```
Next.js = React (UI)
        + Turbopack/webpack (bundler)
        + File-based routing
        + Rendering strategies (CSR / SSR / SSG / ISR)
        + Caching layer
        + API layer (Route Handlers)
        + Image optimization (next/image)
        + Font optimization (next/font)
```

React vẫn là nền tảng. Next.js thêm vào những thứ production cần mà React không cung cấp.

[MODEL] **Meta-framework** = framework xây trên top của một framework hoặc library khác. Next.js là React-based meta-framework — thêm routing, SSR, SSG, API layer, và deployment conventions lên React. Ví dụ tương tự: Nuxt.js là Vue meta-framework, SvelteKit là Svelte meta-framework.

### 4 chiến lược rendering — conceptual, không code

[ANALOGY] Hãy tưởng tượng bạn đang in tài liệu để phát cho khách:

```
┌──────────────────────────────────────────────────────────────────────┐
│ SSG  = Bản in sẵn                                                    │
│        In tất cả trước khi khách đến, khách lấy ngay.                │
│        Nhanh nhất, nhưng nội dung cố định đến khi in lại.            │
├──────────────────────────────────────────────────────────────────────┤
│ ISR  = Bản in sẵn + tái bản khi hết hạn                              │
│        In sẵn, nhưng sau X giờ thì in bản mới nếu có người yêu cầu.  │
│        Gần như nhanh như SSG, nhưng nội dung được cập nhật.          │
├──────────────────────────────────────────────────────────────────────┤
│ SSR  = In theo yêu cầu mỗi khách                                     │
│        Mỗi khách muốn trang, server in ngay lúc đó.                  │
│        Luôn fresh, nhưng khách phải đợi máy in.                      │
├──────────────────────────────────────────────────────────────────────┤
│ CSR  = Khách tự viết từ mẫu                                          │
│        Server chỉ đưa tờ mẫu rỗng + bút.                             │
│        Khách tự điền — chỉ khách đó mới thấy kết quả lúc điền.       │
└──────────────────────────────────────────────────────────────────────┘
```

[MODEL] Trade-off cốt lõi giữa SSR và CSR:

```
TTFB nhanh ◄────────────────────────────────► FCP / LCP tốt

    CSR ─────────────────────────────────────── SSR
Server ít tính toán                  HTML đầy đủ về đến browser ngay
```

- **CSR**: TTFB nhanh vì server chỉ trả về HTML rỗng + JavaScript bundle. Nhưng FCP/LCP chậm vì browser phải tải và chạy JS trước khi có content hiển thị.
- **SSR**: HTML đầy đủ về đến browser ngay — FCP/LCP tốt hơn. Nhưng TTFB cao hơn vì server phải tính toán và render HTML trước.
- **SSG/ISR**: Gần như tốt cả hai mặt — HTML pre-built từ CDN, TTFB cực thấp, FCP/LCP tốt. Trade-off là nội dung có thể stale.

## 3. 📚 Core Concepts

### 3.1 The React Gap

[FACT] Theo official React docs, React tự định nghĩa là _"library for web and native user interfaces"_ — không bao gồm routing, data fetching, project structure, hay build pipeline.

[FACT] Theo official Next.js docs, Next.js cung cấp out-of-the-box: file-based routing (Pages & App Router), image optimization (`next/image`), font optimization (`next/font`), automatic code splitting, và tất cả rendering strategies (SSR/SSG/ISR/CSR).

Hệ quả thực tế: team dùng React thuần phải tự quyết định và maintain routing, SEO, optimization. Team dùng Next.js nhận conventions chuẩn hóa — giảm bikeshedding, tăng consistency.

Đây là ví dụ cụ thể về khoảng cách đó — image optimization:

```ts [src/components/ProductImage.tsx]
import Image from 'next/image'
import { useState } from 'react'

interface ProductImageProps {
  src: string
  alt: string
  width: number
  height: number
  // priority: chỉ đặt true cho LCP element (hero image above-the-fold).
  // KHÔNG true cho tất cả image — làm delay preload của resource quan trọng khác.
  priority?: boolean
}

export function ProductImage({
  src,
  alt,
  width,
  height,
  priority = false,
}: ProductImageProps) {
  const [hasError, setHasError] = useState(false)

  // Fallback khi image load fail.
  // Tại sao cần: broken image icon phá UX, đặc biệt trên mobile.
  const fallbackSrc = '/images/fallback-product.webp'

  return (
    <Image
      // Tại sao next/image thay vì <img> thuần:
      // 1. Auto convert sang WebP/AVIF — giảm 30–50% file size
      // 2. Lazy loading built-in — chỉ load khi image gần viewport
      // 3. Reserve aspect ratio → ngăn CLS (Cumulative Layout Shift)
      // 4. Serve đúng kích thước theo viewport — không load ảnh 2000px cho mobile 400px
      // 5. Nếu src lỗi, có thể xử lý fallback thay vì broken image icon
      src={hasError ? fallbackSrc : src}
      alt={alt}
      width={width}
      height={height}
      priority={priority}
      // sizes: hint cho browser chọn đúng breakpoint image.
      // Không set sizes → browser assume image chiếm full viewport → tải image lớn hơn cần.
      sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw"
      placeholder="blur" // Giảm CLS bằng cách giữ chỗ trước khi ảnh load
      blurDataURL="data:image/jpeg;base64,/9j/4AAQSkZJRgABAQAAAQ..." // Base64 encoded tiny version
      onError={() => {
        // Log để monitor broken image URLs — không swallow error silently.
        // Production: gửi event về error tracking (Sentry / Datadog / ...).
        console.error(`[ProductImage] Failed to load: ${src}`)
        setHasError(true)
      }}
    />
  )
}
```

> **Tại sao không dùng `<img>` thuần?** Browser tải full-size image gốc, không lazy load, không tối ưu format, không prevent CLS. Trên mobile 4G, product page có 20 images × 1MB = 20MB wasted bandwidth. `next/image` giải quyết tất cả với 1 component.

### 3.2 Next.js là gì — Framework vs Library

[FACT] Theo official Next.js docs: _"Next.js is a React framework for building full-stack web applications… automatically configures lower-level tools like bundlers and compilers."_

[FACT] Next.js 16 dùng **Turbopack** làm bundler mặc định. Turbopack là incremental bundler viết bằng Rust — dev startup nhanh hơn webpack đáng kể trên large codebase.

> **Overlap Lock:** Chi tiết về Turbopack và bundler internals thuộc M0.3. Bài này chỉ cần biết: "Next.js 16 dùng Turbopack by default."

Đây là `next.config.ts` production-ready với validation môi trường tại build time:

```ts [next.config.ts]
import type { NextConfig } from 'next'

// Validate required env vars tại build time — không phải runtime.
// Tại sao: fail fast ở build tốt hơn fail im lặng ở runtime production.
// Deploy lên staging mà thiếu env → build fail rõ ràng, không tốn deploy slot.
const requiredEnvVars = ['NEXT_PUBLIC_API_URL'] as const

for (const envVar of requiredEnvVars) {
  if (!process.env[envVar]) {
    throw new Error(
      `[next.config] Missing required environment variable: ${envVar}\n` +
        `Check your .env.local or deployment environment settings.`,
    )
  }
}

const nextConfig: NextConfig = {
  // Next.js 16: Turbopack là default cho `next dev`.
  // Không cần flag `--turbopack` nữa.
  // Nếu gặp incompatibility với webpack plugin cũ:
  // → Dùng tạm `next dev --no-turbopack` và mở issue.
  turbopack: {
    // Tăng memory nếu project lớn — tránh OOM trong CI/CD
    memoryLimit: 4096,
  },

  images: {
    // remotePatterns thay thế `domains` (deprecated từ v13.4+).
    // Chặt hơn `domains`: lock theo protocol + hostname + path pattern.
    // Mục đích: 
    // - Tránh cho phép toàn bộ internet dùng image optimizer của bạn.
    // - Chỉ cho phép domains tin cậy — tránh attacker dùng image proxy để SSRF
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'cdn.yourapp.com',
        port: '',
        pathname: '/images/**',
      },
    ],
    // webp trước avif: browser support webp > avif (số liệu 2024).
    // Điều chỉnh nếu analytics cho thấy audience chủ yếu dùng modern browsers.
    // WebP/AVIF giảm 30-50% size so với JPEG
    formats: ['image/webp', 'image/avif'],
  },
  // Experimental flags — KHÔNG bật tất cả, chỉ bật cái team cần và đã test
  experimental: {
    // Chỉ bật nếu team đã đọc docs và hiểu rõ trade-offs
    // ppr: true, // Partial Prerendering — Next.js 16+
  },
  // Headers bảo mật cơ bản — không thay thế WAF nhưng là layer phòng thủ đầu tiên
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
        ],
      },
    ];
  },
  
  // Redirect cũ → mới — critical cho SEO migration
  async redirects() {
    return [
      {
        source: '/old-product/:slug',
        destination: '/products/:slug',
        permanent: true, // 301 — chuyển hết link juice
      },
    ];
  },
}

export default nextConfig
```

### 3.3 Vercel Relationship — Open Source vs Platform

[FACT] Next.js là open-source React framework. Repository công khai tại `github.com/vercel/next.js` — bất kỳ ai cũng có thể contribute, fork, và self-host.

[FACT] Vercel là managed platform — _"Made by the creators of Next.js… the native Next.js platform."_ Vercel optimize deploy experience cho Next.js, đặc biệt là ISR và Edge Runtime.

[FACT] Next.js **không phải** Vercel, và Vercel **không phải** điều kiện bắt buộc. Next.js có **Ecosystem Working Group** và **Build Adapters** để các platform khác (Netlify, Firebase, AWS Amplify, Cloudflare, Railway...) implement cùng deployment contract.

Nhiều junior nhầm lẫn: "Deploy Next.js phải dùng Vercel." Sai. Next.js chạy trên bất kỳ server nào hỗ trợ Node.js.

```ts [next.config.ts]
import type { NextConfig } from 'next'

// DEPLOY_TARGET cho phép CI/CD quyết định output format.
// Tại sao dùng env var: không hardcode deployment target — cùng codebase
// deploy được lên nhiều platforms khác nhau tùy môi trường.
const deployTarget = process.env.DEPLOY_TARGET ?? 'server'

const validTargets = ['server', 'static', 'standalone'] as const
type DeployTarget = (typeof validTargets)[number]

if (!validTargets.includes(deployTarget as DeployTarget)) {
  throw new Error(
    `[next.config] Invalid DEPLOY_TARGET: "${deployTarget}"\n` +
      `Valid options: ${validTargets.join(', ')}`,
  )
}

const nextConfig: NextConfig = {
  // 'server' (default): cần Node.js runtime.
  //   → Vercel, Railway, Fly.io, self-hosted Node.js.
  //
  // 'standalone': Next.js bundle dependencies vào .next/standalone/.
  //   → Docker, Kubernetes.
  //   → Giảm Docker image size đáng kể — không cần copy cả node_modules.
  //   Deploy lên AWS/GCP/Azure
  //
  // 'export': pure static HTML/CSS/JS — không cần Node.js server.
  //   → S3 + CloudFront, GitHub Pages, Netlify static.
  //   → Giới hạn: KHÔNG có ISR, KHÔNG có API routes, KHÔNG có Image Optimization.
  output:
    deployTarget === 'static'
      ? 'export'
      : deployTarget === 'standalone'
        ? 'standalone'
        : undefined, // undefined = default 'server' behavior

  // Với output 'export': Image Optimization cần server — phải disable.
  // Dùng external CDN (Cloudinary, imgix) hoặc chấp nhận unoptimized images.
  ...(deployTarget === 'static' && {
    images: {
      unoptimized: true,
    },
  }),
    // Nếu export static, distDir sẽ chứa pure HTML/JS/CSS
  distDir: 'dist',
  
  // Trailing slash quan trọng cho SEO và static hosting consistency
  trailingSlash: true,
  
  // Asset prefix nếu dùng CDN riêng cho static assets
  assetPrefix: process.env.CDN_URL || undefined,
}

export default nextConfig
```

> **Điểm cần nhớ:** Nếu team quyết định rời Vercel, Next.js app có thể migrate sang self-hosted hoặc platform khác — chỉ cần đổi `output` config và setup adapter. Không bị vendor lock-in ở framework level (mặc dù một số tính năng như ISR optimize tốt nhất trên Vercel).

## 4. 🚫 Anti-Patterns in the Wild

### Anti-Pattern 1: "Next.js là backend framework"

**❌ Code Smell — Junior:**

```ts [src/app/api/orders/route.ts]
// ❌ Toàn bộ business logic nhét vào Next.js API route.
// Junior làm thế này vì "tiện" — 1 repo, 1 deploy, 1 CI/CD.
// Nhưng đây là monolith FE+BE deploy chung, scale chung, fail chung.

export async function POST(request: Request) {
  const body = await request.json()

  // ❌ Complex auth logic trong Next.js
  const user = await validateJWTAndLoadPermissions(body.token)
  if (!user) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  // ❌ Domain business logic và DB transactions trong Next.js
  const inventory = await checkAndReserveInventory(body.items) // phức tạp, có retry
  const pricing = await applyTierPricingAndDiscounts(body.items, user.tier) // pricing engine
  const order = await createOrderWithSagaPattern(body, pricing, inventory)

  // ❌ Side effects và background jobs — blocking, delay response
  await sendOrderConfirmationEmail(user.email, order)
  await syncWithERP(order) // external system, có thể timeout
  await updateInventoryAnalytics(body.items)

  return Response.json({ orderId: order.id })
}
// Nếu Next.js server down → toàn bộ order system down.
// Scale order processing → phải scale cả FE, lãng phí.
```

**✅ Staff Refactor — BFF Pattern:**

```ts [src/app/api/orders/route.ts]
// ✅ Next.js chỉ là BFF (Backend-for-Frontend).
/**
 * Tại sao chỉ dùng Next.js làm BFF (Backend-for-Frontend)?
 * 1. Next.js nên chỉ là thin proxy + UI orchestration
 * 2. Domain logic, transactions, background jobs giữ ở backend service riêng (NestJS/Express/Go)
 * 3. FE và BE scale độc lập — BE nặng thì scale BE, FE cần CDN thì CDN
 * 4. Failure isolation — BE sập không kéo FE chết
 * 5. Security — không lộ domain logic trong FE bundle
 */

// Business logic nằm ở backend service riêng (NestJS / Go / Java / ...).
// Next.js: validate input nhanh → proxy sang backend → format response cho client.

export async function POST(request: Request) {
  const orderServiceUrl = process.env.ORDER_SERVICE_URL
  if (!orderServiceUrl) {
    // Fail fast. Không để request proxy đến URL undefined và crash im lặng.
    console.error('[OrderRoute] ORDER_SERVICE_URL not configured')
    return Response.json({ error: 'Service configuration error' }, { status: 500 })
  }

  let body: unknown
  try {
    body = await request.json()
  } catch {
    return Response.json({ error: 'Invalid request body' }, { status: 400 })
  }

  try {
    const response = await fetch(`${orderServiceUrl}/orders`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        // Forward auth header — backend service tự validate JWT.
        Authorization: request.headers.get('Authorization') ?? '',
        // Correlation ID: trace request xuyên suốt FE → BFF → BE → downstream.
        // Dùng ID này khi search logs trên monitoring (Datadog / Loki / Grafana).
        'X-Request-ID': crypto.randomUUID(),
        'X-Source': 'nextjs-bff',
      },
      body: JSON.stringify(body),
      // Timeout: tránh Next.js giữ connection mãi khi backend chậm hoặc hang.
      signal: AbortSignal.timeout(10_000), // 10 giây
    })

    if (!response.ok) {
      console.error(`[OrderRoute] Backend returned HTTP ${response.status}`)
      // Không expose backend error message ra client — potential security leak.
      return Response.json({ error: 'Order creation failed' }, { status: response.status })
    }

    return Response.json(await response.json())
  } catch (error) {
    if (error instanceof DOMException && error.name === 'TimeoutError') {
      console.error('[OrderRoute] Backend timeout after 10s')
      return Response.json({ error: 'Service timeout' }, { status: 504 })
    }
    console.error('[OrderRoute] Unexpected error proxying to order service:', error)
    return Response.json({ error: 'Internal server error' }, { status: 500 })
  }
}
```

**🧠 Trade-off của việc refactor:**

*Pros:* Scale FE và BE độc lập — BE order service có thể scale lên 10x mà không đụng FE. BE down → Next.js vẫn serve static content. Domain logic rõ ràng, không pha trộn UI concerns. Monitor, alert, deploy cycle riêng cho từng service.

*Cons:* Thêm 1 service phải vận hành, deploy, monitor. Mỗi request thêm 1 network hop (BFF → BE) — latency tăng 5–50ms tùy infrastructure. Cần setup inter-service communication (HTTP, gRPC, message queue).

**🔥 Incident Risk nếu giữ code cũ:**

- Deploy Next.js version mới có bug → sập cả FE lẫn order processing cùng một lúc — shared failure domain, không có isolation. Điều tra incident phức tạp hơn vì không biết lỗi từ FE hay logic order.
- Một API route bị DDoS → cả Next.js instance down, FE không thể phục vụ static pages.
- Security: nhầm tưởng Server Components là "private" nên để database connection string trong component → lộ trong bundle (Server Components không gửi JS nhưng vẫn có thể leak nếu serialize nhầm).

### Anti-Pattern 2: "Dùng Next.js là tự động fast"

**❌ Code Smell — Junior:**

```ts [src/app/blog/[slug]/page.tsx]
// ❌ Blog post tĩnh nhưng không có cache directive.
// App Router mặc định: nếu không opt vào static, page là dynamic (SSR mỗi request).
// Mỗi visitor đều hit database — dù bài viết không thay đổi trong nhiều tuần.

async function BlogPost({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params

  // ❌ Không có next: { revalidate } hoặc cache tag → dynamic rendering
  // 10,000 visitor/ngày = 10,000 database queries cho cùng 1 bài blog.
  const post = await fetch(`${process.env.API_URL}/posts/${slug}`).then((r) =>
    r.json(),
  )

  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.content }} />
    </article>
  )
}

export default BlogPost
// Hệ quả: TTFB cao, server load tăng, Vercel Function invocations bill tăng.
// Next.js không tự động làm nhanh — bạn phải chọn đúng strategy.
```

**✅ Staff Refactor — ISR với On-demand Revalidation:**

```ts [src/app/blog/[slug]/page.tsx]
// ✅ ISR: cache bài viết 1 giờ + on-demand revalidation khi CMS publish.
// 10,000 visitor/ngày → nhiều lắm 24 lần render thực sự (1 lần/giờ).
// Phần còn lại: serve từ cache, gần như free.


export const revalidate = 3600; // 1 giờ — điều chỉnh theo business SLA

// generateStaticParams: pre-render bài phổ biến tại build time.
// Các bài chưa trong list: render lần đầu → cache → serve từ cache từ lần 2 trở đi.
export async function generateStaticParams() {
  // Chỉ pre-render top 100 — tránh build time dài cho site lớn.
  // Các post còn lại sẽ được ISR on first visit
  const posts = await fetch(`${process.env.API_URL}/posts/popular?limit=100`, {
    next: { revalidate: 86400 }, // Cache danh sách 1 ngày
  }).then((r) => r.json());
  return posts.map((post: { slug: string }) => ({ slug: post.slug }))
}

export async function generateMetadata({ params }: { params: { slug: string } }) {
  const post = await getPost(params.slug);
  
  return {
    title: post.title,
    description: post.content.slice(0, 160),
    openGraph: {
      title: post.title,
      type: 'article',
      publishedTime: post.publishedAt,
    },
  };
}

async function getPost(slug: string): Promise<BlogPost> {
  const res = await fetch(`${process.env.API_URL}/posts/${slug}`, {
    next: {
      // revalidate: 3600 giây = 1 giờ.
      // Chọn giá trị này dựa trên: bài blog thay đổi bao lâu 1 lần?
      // Nếu CMS dùng on-demand revalidation → có thể set cao hơn (24h, 7 ngày).
      revalidate: 3600,
      // tags: khi CMS publish bài mới → webhook gọi revalidateTag('blog-posts').
      // Content update ngay, không cần đợi timeout.
      tags: [`blog-post-${slug}`, 'blog-posts'],
    },
  });
  
  if (!res.ok) {
    if (res.status === 404) notFound();
    throw new Error(`Failed to fetch post: ${res.status}`);
  }
  
  return res.json();
}

async function BlogPost({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params

  const post = await getPost(params.slug);

  return (
    <article className="prose lg:prose-xl">
      <h1>{post.title}</h1>
      <time className="text-gray-500">
        {new Date(post.publishedAt).toLocaleDateString('vi-VN')}
      </time>
      <div 
        className="mt-8"
        // Nếu content từ CMS, cần sanitize trước khi render — tránh XSS
        dangerouslySetInnerHTML={{ __html: post.content }} 
      />
    </article>
  )
}

export default BlogPost
```

**🧠 Trade-off của việc refactor:**

*Pros:* TTFB thấp (serve từ cache / CDN), server load giảm mạnh, scale tốt với traffic lớn, Vercel/hosting cost giảm đáng kể.

*Cons:* Content có thể stale trong khoảng `revalidate`. First visitor *sau khi cache expire* thấy stale content và trigger background regeneration — visitor đó phải serve version cũ. Cần build và maintain on-demand revalidation endpoint nếu muốn update ngay khi publish.

**🔥 Incident Risk nếu giữ code cũ:**

- Blog scale lên 50k daily visitors, không có cache → TTFB tăng, server overload, không scale được. Trên Vercel: Function invocations tính tiền theo lần gọi — chi phí tăng phi lý so với ISR. Trên self-host: cần horizontal scale server để xử lý load không cần thiết.
- Không set revalidate → mỗi request đều compute → bill tăng phi mã. Tôi đã thấy team nhận bill Vercel $3k/tháng vì "quên" cache.
- ISR với revalidate quá ngắn (10s cho blog) → vô nghĩa, gần như SSR nhưng thêm complexity cache.

## 5. ⚖️ Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| --- | --- | --- | --- | --- | --- |
| **CSR** (React SPA) | Navigation nhanh sau lần đầu; deploy đơn giản (static hosting); code familiar; bundle có thể tree-shake tốt | SEO kém — HTML rỗng cho crawler; FCP/LCP chậm; cần JS để render bất kỳ content nào | Dashboard nội bộ; app-type SPA; authenticated users; không cần SEO | Public content site; SEO quan trọng; content-heavy marketing/blog; slow network users | Build nhẹ; không cần server runtime; mọi compute trên client |
| **SSR** (Server Render) | HTML đầy đủ mỗi request → SEO tốt; phù hợp personalization và realtime data; dễ migration từ custom SSR | TTFB cao hơn SSG/ISR; server load lớn hơn; scale tốn kém hơn; khó cache đúng | User-specific pages; realtime data; A/B testing per-user; khi data thay đổi mỗi request | Page nội dung tĩnh ít thay đổi; traffic rất lớn mà server resource có hạn | Server compute mỗi request; cần edge cache để giảm TTFB |
| **SSG** (Static) | TTFB thấp nhất (từ CDN); server load gần bằng 0 lúc runtime; SEO tốt; scale tốt với traffic lớn | Build time tăng khi số trang lớn; cập nhật nội dung phải rebuild; không personalization | Blog; docs; marketing site; catalog ít thay đổi; traffic lớn; SEO quan trọng | Nội dung thay đổi liên tục; cần per-user data; data fresh mỗi phút | Build time và storage; cần ISR / PPR nếu content hay update |
| **ISR** (SSG + revalidation) | Gần như pros của SSG + update không full rebuild; giảm server load so SSR; CDN-friendly; SEO tốt | Stale content trong khoảng revalidate; first visitor sau expire thấy version cũ; host phải hỗ trợ ISR | E-commerce catalog; blog lớn; content update vừa phải; tránh full rebuild với site nhiều trang | Realtime data; cần fresh 100% mỗi request; host không support ISR | Cần host hỗ trợ ISR (Vercel / adapter); complexity caching và revalidation logic |

> **Rule of thumb:** Hỏi 3 câu trước khi chọn strategy — (1) Data thay đổi bao lâu 1 lần? (2) Có cần SEO không? (3) Có per-user data không? Câu trả lời dẫn đến strategy đúng, không phải "dùng SSR cho chắc ăn".

## 6. 🏛️ Legacy Recognition

Khi vào một codebase React SPA cũ, đây là 3 dấu hiệu nhận diện ngay:

**Dấu hiệu 1 — Routing bằng React Router:**

```
src/
  App.tsx          ← <BrowserRouter> + <Routes> ở đây
  pages/           ← convention tự đặt, không phải Next.js convention
    Home.tsx
    About.tsx
    Product.tsx
```

Không có `app/` hoặc `pages/` theo Next.js file-based routing convention. URL structure không ánh xạ 1-1 với file system. Routing logic nằm trong JavaScript component, không phải file structure.

**Dấu hiệu 2 — SEO bằng React Helmet:**

```tsx
import { Helmet } from 'react-helmet'

function BlogPost() {
  return (
    <>
      <Helmet>
        <title>Bài viết của tôi</title>
        <meta name="description" content="..." />
      </Helmet>
      {/* content */}
    </>
  )
}
```

React Helmet inject meta tags trên client *sau khi JavaScript chạy xong*. Crawler nhận HTML không có meta. SEO không ổn định — tùy Google có render JS không.

**Dấu hiệu 3 — Data fetching toàn bộ trên client:**

```tsx
function ProductList() {
  const [products, setProducts] = useState([])

  useEffect(() => {
    // Mọi data fetch đều ở đây — chạy sau khi HTML về đến browser
    fetch('/api/products').then((r) => r.json()).then(setProducts)
  }, [])

  if (!products.length) return <div>Loading...</div>
  return <div>{/* render */}</div>
}
```

Không có server data fetching. Cache không chuẩn hóa. Loading state xuất hiện mỗi lần page load — ngay cả với data không thay đổi.

**Terminology cũ thường thấy trong legacy codebase:**

- `getServerSideProps` — Pages Router SSR (M11.x sẽ cover)
- `getStaticProps` / `getStaticPaths` — Pages Router SSG (M11.x sẽ cover)
- `React Helmet` — thay bằng Next.js Metadata API (App Router) hoặc `<Head>` (Pages Router)
- `renderToString` — custom SSR cũ, trước Next.js
- `create-react-app` (CRA) — deprecated, không còn được maintain
- `React Router DOM v5 / v6` — client-side routing trong SPA

## 7. 🚀 Modern Approach

### Migration strategy — Strangler Fig, không Big Bang

Staff Engineer không bao giờ rewrite toàn bộ. Đây là migration an toàn từ CRA SPA sang Next.js theo kiểu **Strangler Fig**:

**Bước 1: Parallel deployment với proxy**

Tạo Next.js app song song với SPA cũ. Dùng nginx / Cloudflare / reverse proxy để route một phần traffic sang Next.js. SPA cũ vẫn chạy, không có downtime.

```
Traffic → Proxy / CDN
                ├── /blog/*     → Next.js (mới — SSG/ISR)
                ├── /products/* → Next.js (mới — ISR)
                └── /*          → SPA cũ (fallback mọi route còn lại)
```

**Bước 2: Migration route-by-route**

Chuyển từng React Router route sang file trong `app/` hoặc `pages/`. Thay `useEffect` data fetch bằng Server Components fetch hoặc `getServerSideProps`. Không migrate tất cả cùng lúc.

**Bước 3: Thay React Helmet bằng Metadata API**

```tsx
// Trước (React Helmet — client-only):
<Helmet><title>Blog của tôi</title></Helmet>

// Sau (App Router Metadata API — server-rendered):
//Thay React Helmet bằng Metadata API (App Router: export const metadata hoặc 
//generateMetadata) hoặc <Head> (Pages Router).

export const metadata = {
  title: 'Blog của tôi',
  description: '...',
}
// Metadata object được Next.js inject vào <head> ở server — crawler thấy ngay.
// Không cần JavaScript chạy để có title/description trong HTML.
```

**Bước 4: Chọn đúng rendering strategy cho từng route**

- Content pages (blog, docs, marketing) → SSG hoặc ISR
- Auth dashboard, user-specific data → SSR hoặc CSR
- Static pages (about, contact, terms) → SSG
- Checkout, cart → SSR (per-user, không cache)

**Bước 5: Remove legacy dependencies**

Sau khi tất cả routes đã migrate và ổn định trong vài tuần: remove React Router, CRA / Vite config cũ, React Helmet. Update CI/CD để deploy Next.js. Đây là bước cuối — không rush.

### Rollback plan nếu migration fail

Không bao giờ đặt cược tất cả vào migration mà không có escape hatch:

- **Giữ SPA cũ hoạt động độc lập** — không phụ thuộc vào Next.js API hoặc shared state
- **Feature flag tại proxy level**: `X-Use-Next: true` header → route sang Next.js; tắt flag → toàn bộ traffic về SPA cũ trong vài giây, không cần redeploy
- **Dark launch**: Next.js chỉ expose qua subdomain (`next.yourapp.com`) hoặc chỉ cho internal testers — không public cho real users đến khi stable
- **Monitor và rollback trigger**: Core Web Vitals và error rate trước/sau migration. Rollback nếu error rate tăng >1% hoặc LCP tăng >20% so với baseline

## 8. 🔥 Production Scenario

### Case Study 1: Team React SPA muốn add SEO → Next.js migration

**Bối cảnh:** E-commerce site dùng CRA + React Router + React Helmet. Google Analytics báo organic traffic flat dù content tốt. Core Web Vitals audit: LCP 4.1s, CLS 0.21 — cả hai đều fail "Poor" threshold.

**Sau migration Next.js** (SSG cho content pages + `next/image` + `next/font`):

- LCP: 4.1s → 1.9s (SSG serve từ CDN + image optimization giảm payload)
- CLS: 0.21 → 0.04 (`next/image` reserve layout space → không còn layout shift)
- Organic sessions: +38% sau 6 tháng (content được index đầy đủ)

**Pattern đã dùng:** Product listing, blog, category pages → SSG/ISR. Auth dashboard → giữ CSR. Checkout → SSR (per-user, không cache).

### Case Study 2: Tech Lead hỏi "Why Next.js instead of Vite + React Router?"

Câu trả lời không phải "Next.js tốt hơn." Câu trả lời là **trade-off cụ thể theo nhu cầu**:

| | Vite + React Router | Next.js |
| --- | --- | --- |
| SSR / SSG / ISR | Tự setup (renderToString, custom server) | Built-in, conventions chuẩn |
| Image optimization | Tự config hoặc dùng external service | `next/image` out-of-the-box |
| Font optimization | Tự xử lý FOUT/FOIT | `next/font` với preloading tự động |
| SEO / Meta | React Helmet (client-only) | Server-rendered metadata |
| File-based routing | Không có | Có (Pages & App Router) |

Khi nào vẫn chọn Vite + React Router: internal tool không cần SSR, team đã optimize tốt stack hiện tại, muốn tránh Next.js conventions và server runtime overhead.

### Case Study 3: E-commerce lớn rebuild với Next.js

**Pattern:** Next.js chỉ là storefront (UI + BFF). Backend riêng — Shopify headless hoặc custom services. ISR cho product pages (content ít thay đổi, scale tốt qua CDN). SSR cho cart và checkout (per-user, không cache).

Kết quả từ các case studies tương tự: giảm code surface đáng kể, ship feature nhanh hơn, product pages scale qua CDN mà không tăng server cost tuyến tính.

### 🐛 Debugging: Organic traffic giảm sau migration Next.js

**Symptom:** Sau migration, Google Search Console báo indexed pages giảm, organic traffic giảm 15% trong 2 tuần đầu.

**Root causes thường gặp:**

1. **URL không match:** React Router dùng `/blog/:id`, Next.js migrate thành `/blog/[id]` nhưng quên setup 301 redirect từ URL cũ → crawler gặp 404, mất SEO juice tích lũy.
2. **Metadata thiếu hoặc sai:** `metadata` export chưa đủ fields, hoặc bị overwrite bởi parent layout — title/description trong HTML không đúng.
3. **CSR thay vì SSR/SSG:** Một số content pages vẫn có `'use client'` với data fetch trong `useEffect` → crawler thấy HTML rỗng.
4. **Canonical sai trong giai đoạn parallel deploy:** Duplicate content giữa SPA cũ và Next.js mới — chưa set canonical URL đúng dẫn đến Google chọn nhầm version.

**Cách detect:**

```bash
# Kiểm tra server có trả về metadata trong HTML không
curl https://yoursite.com/blog/some-post | grep -i "<title>"

# Kiểm tra redirect chain từ URL cũ
curl -IL https://yoursite.com/old-url-format
```

- Google Search Console → URL Inspection → xem Google "thấy" gì (rendered HTML sau JS)
- So sánh `view-source:` trước/sau hydration bằng cách tắt JavaScript trong browser
- Audit sitemap.xml — đảm bảo URLs khớp với routes thực tế trong Next.js

## 9. 📊 Performance & Overhead

[MODEL] Từ nghiên cứu benchmark SSR vs SSG trên cloud infrastructure:
- **SSG cải thiện 50–70% TTFB/FCP/LCP** so với SSR trên cùng content, đồng thời giảm server resource utilization đáng kể — CDN serve static HTML thay vì server phải compute mỗi request.

[MODEL] PPR / Cache Components (Next.js 16) — đo lường trên 1 product page thực tế:
- TTFB: ~420ms → 85ms
- FCP: 1.2s → 0.6s
- LCP: giảm ~50%

> 🔴 **PRODUCTION REALITY:** Những con số trên là **kết quả của 1 case study cụ thể**, không phải guaranteed behavior. Thực tế phụ thuộc vào: database query time, network latency, content size, CDN coverage, và cold start frequency. SSG/ISR giảm server cost đáng kể nhưng tăng build time và complexity caching. SSR đơn giản về code nhưng là "tax" trên mỗi request — tính toán server cost trước khi chọn, đặc biệt trên Vercel nơi Function invocations tính tiền theo lần gọi.

**Cách benchmark đúng cách, không đoán mò:**

1. Dùng **Lighthouse** hoặc **WebPageTest** để đo FCP, LCP, TTFB, CLS trên staging environment — không phải localhost
2. So sánh trước/sau thay đổi rendering strategy — không assume, measure
3. Kiểm tra **cold start** (first request sau deploy hoặc sau idle) — thường cao hơn warm requests nhiều, gây surprise khi deploy production
4. Monitor **percentile 95 và 99**, không chỉ median — outliers ảnh hưởng UX nhiều hơn average value

## 10. 🏗️ Architecture Notes

Phần này cho ai cần đưa ra quyết định kiến trúc — không chỉ implement theo spec.

**Next.js nên là BFF + UI layer, không phải backend monolith.**

Khi hệ thống cần những thứ sau, tách backend riêng:
- Complex distributed transactions (saga pattern, 2-phase commit)
- Background jobs (queue workers, cron jobs, event-driven processing)
- Long-lived processes (WebSocket server, streaming data pipeline)
- Sophisticated ORM với database migrations phức tạp
- Multi-tenancy domain logic hoặc DDD (Domain-Driven Design)

**Khi nào KHÔNG chọn Next.js:**

- Internal SPA không cần SEO, team đã có infrastructure CSR tốt — overhead Next.js không justified
- Cần backend phức tạp và Next.js chỉ làm phức tạp thêm — không có domain layer, không có DDD support
- Pure static site đơn giản không cần React — Astro hoặc Hugo có thể phù hợp hơn và nhẹ hơn

**Scale pattern điển hình:**

```
User
  └── Next.js (BFF + UI / Edge Runtime)
        ├── Static assets → CDN
        ├── Dynamic pages → Node.js Runtime (full Node.js APIs)
        └── Auth / routing → Edge Runtime (fast, global, ít latency)
                └── Backend Services (via API)
                      ├── Order Service (NestJS / Go)
                      ├── Product Service
                      └── User Service
```

Dùng **Edge Runtime** cho middleware nhẹ: authentication redirect, geo-routing, A/B testing. Edge Runtime không có full Node.js APIs — không dùng cho heavy computation hay complex database queries. Dùng **Node.js Runtime** cho SSR / Server Components cần full Node.js APIs.

## 11. 🎤 Interview Questions

### Q1: "Next.js khác React ở điểm gì? Tại sao cần meta-framework?"

<details>
<summary>Đáp án</summary>

React là UI library — chỉ handle rendering components. Không có routing, SSR, build pipeline, hay optimization nào. Khi build production web app, cần tất cả những thứ đó.

Next.js là React meta-framework: orchestrate React + routing (file-based) + rendering strategies (SSR/SSG/ISR/CSR) + image/font optimization + build pipeline chuẩn hóa. Không phải replace React — là layer trên top của React.

Cần meta-framework vì production web cần nhiều hơn UI: SEO, performance, developer experience, scalability. Thay vì mỗi team tự tổ hợp và maintain stack riêng, Next.js cung cấp conventions battle-tested ở production.

</details>

### Q2: "Khi nào bạn KHÔNG chọn Next.js?"

<details>
<summary>Đáp án</summary>

Ba trường hợp chính:

Internal SPA không cần SEO — nếu app là internal dashboard, authenticated users, SEO không quan trọng, team đã có infrastructure CSR tốt → overhead Next.js server runtime không justified.

Backend phức tạp — nếu app cần complex transactions, background jobs, domain-driven design, Next.js thiếu layer này. Nên tách backend riêng. Khi đó Next.js chỉ là BFF — có thể cân nhắc liệu overhead Next.js có worthwhile không so với Vite + React Router đơn giản hơn.

Pure static site không cần React interactivity — Astro hoặc Hugo nhẹ hơn, build nhanh hơn, không có JavaScript overhead.

Key insight: chọn Next.js khi cần SSR/SSG/ISR + React + production conventions. Không chọn vì "mọi người dùng Next.js."

</details>

### Q3: "SSR trong Pages Router khác SSR trong App Router thế nào?"

<details>
<summary>Đáp án</summary>

Pages Router SSR: page dùng `getServerSideProps` → HTML render mỗi request, không cache. SSR = "page-level per-request rendering." Đây là nghĩa cổ điển.

App Router: nghĩa SSR bị redefined. Server Components render trên server theo mặc định — nhưng *có thể* được cache, static generate, hoặc render per-request tùy config. SSR không còn đồng nghĩa "render mỗi request."

Confusion phổ biến nhất: Junior đọc "Server Components render on server" → nghĩ "vậy là SSR như cũ" → expect mỗi request đều re-render. Sai. App Router Server Components mặc định là static (tương tự SSG) trừ khi có dynamic data hoặc dynamic functions.

</details>

### Q4: "ISR trade-offs là gì? Khi nào bạn chọn ISR thay vì SSR/SSG?"

<details>
<summary>Đáp án</summary>

ISR giữ pros của SSG (TTFB thấp, CDN-friendly, SEO tốt) nhưng cho phép update content sau build mà không rebuild toàn bộ.

Trade-off: stale content trong khoảng `revalidate`. First visitor sau khi cache expire thấy version cũ và trigger background regeneration — không phải realtime update.

Chọn ISR khi: data thay đổi vừa phải (hàng giờ, hàng ngày), không cần realtime, cần tránh full rebuild (site 100k+ pages), nhưng cần content tươi hơn SSG thuần.

Không chọn ISR: data cần fresh 100% mỗi request (stock price, live sports score) → SSR. Data gần như không đổi (legal docs, terms) → SSG thuần, không cần complexity của ISR.

</details>

### Q5: "Tại sao nói Next.js không phải backend framework?"

<details>
<summary>Đáp án</summary>

Next.js có API routes và Route Handlers — nó có server capabilities. Nhưng "backend framework" implies: domain layer, long-lived processes, background job queue, robust ORM, transaction management, message broker integration.

Next.js thiếu tất cả. API routes là stateless, short-lived. Không có built-in queue, không có worker processes, không có sophisticated transaction support, không có domain model.

Đúng cách dùng Next.js: BFF (Backend-for-Frontend) — proxy requests, aggregate data từ nhiều services, handle auth cookies, format response cho UI. Business logic nặng → backend service riêng (NestJS / Go / Java / ...).

Dấu hiệu đang dùng sai: đặt pricing engine, inventory management, email queue, hoặc saga pattern trong Next.js API routes.

</details>

### 🟡 Exercise 2: Fix Hydration Mismatch

**Đề bài:** Component dưới đây gây lỗi `"Text content does not match server-rendered HTML"` trong Next.js App Router. Refactor để fix hydration mismatch mà vẫn giữ đúng chức năng ban đầu:

```tsx
// ❌ Tìm bug và sửa — có 2 vấn đề trong file này
'use client'

import { useState } from 'react'

function SessionInfo() {
  const isClient = typeof window !== 'undefined'
  const screenWidth = isClient ? window.innerWidth : 0

  return (
    <div>
      <p>Môi trường: {isClient ? 'Browser' : 'Server'}</p>
      <p>Chiều rộng màn hình: {screenWidth}px</p>
    </div>
  )
}

function LoadTimestamp() {
  const [loadTime] = useState(Date.now())
  return <p>Trang load lúc: {new Date(loadTime).toLocaleTimeString('vi-VN')}</p>
}

export { SessionInfo, LoadTimestamp }
```

**Acceptance Criteria:**
- [ ] Không còn hydration mismatch error trong browser console
- [ ] `SessionInfo` vẫn hiển thị đúng thông tin platform và screen width sau hydration
- [ ] `LoadTimestamp` hiển thị đúng thời điểm client load (không phải server render)
- [ ] Có comment giải thích *tại sao* fix đó hoạt động — không chỉ *làm gì*
- [ ] Không dùng `suppressHydrationWarning` để bịt lỗi (đó là che giấu bug, không phải fix)

<details>
<summary>Hint — đọc nếu stuck sau 15 phút</summary>

Vấn đề cốt lõi: server và client phải render *cùng HTML* tại thời điểm initial render. Bất cứ giá trị nào phụ thuộc vào browser environment — `window`, thời gian thực, random value — sẽ cho kết quả khác nhau giữa server và client → mismatch.

Giải pháp: dùng `useEffect` để defer browser-only code sang *sau* khi hydration xong. `useEffect` không chạy trên server — chỉ chạy trên client sau khi React đã "take over" DOM từ server HTML.

Pattern chuẩn: `const [value, setValue] = useState(stableDefaultValue)` kết hợp với `useEffect(() => { setValue(browserOnlyValue) }, [])`. Initial render dùng stable default (khớp với server), sau hydration mới update với browser value.

Với `LoadTimestamp`: `null` là stable default tốt hơn `0` — tránh hiển thị giá trị sai trong fraction of second trước khi effect chạy. Handle `null` state bằng loading indicator hoặc placeholder.
</details>

### 🔴 Exercise 3: Design Migration Plan từ CRA sang Next.js

**Đề bài:** Bạn vừa join một team có codebase CRA + React Router v6 + React Helmet + client-side fetch. Site là **e-commerce catalog** với:

- ~200 trang sản phẩm (xem được không cần đăng nhập, SEO quan trọng, data update vài lần/ngày)
- 1 trang `/dashboard` cho seller (cần đăng nhập, data per-user, không cần SEO)
- 1 trang `/checkout` (per-user, cần cart data mới nhất, không cache được)
- Các trang `/about`, `/contact`, `/terms` (content tĩnh, gần như không thay đổi)

Yêu cầu từ Tech Lead: không có downtime, không big-bang rewrite, rollback phải thực hiện được trong vòng 5 phút.

Viết migration plan bao gồm:

1. **Route mapping:** mỗi route sẽ dùng rendering strategy gì sau migration và tại sao
2. **Metadata migration:** xử lý SEO/meta tags như thế nào trong quá trình chuyển đổi
3. **Data fetching migration:** mỗi loại data migrate theo cách nào
4. **Rollback plan:** cơ chế cụ thể để rollback trong 5 phút nếu migration fail
5. **Thứ tự ưu tiên:** migrate cái gì trước, cái gì sau và lý do

**Acceptance Criteria:**
- [ ] Route mapping đầy đủ — mỗi route có strategy + lý do cụ thể
- [ ] Không có bước nào "migrate tất cả cùng lúc"
- [ ] Rollback plan có cơ chế kỹ thuật cụ thể (không chỉ "rollback nếu có vấn đề")
- [ ] Thứ tự migration có logic rõ ràng — route ít risk nhất trước
- [ ] Đề cập đến giai đoạn parallel deployment (SPA cũ vẫn chạy khi Next.js mới đang migrate)

<details>
<summary>Hint — đọc nếu stuck sau 20 phút</summary>

Về thứ tự: bắt đầu từ pages ít risk nhất, ít dynamic nhất. `/about`, `/contact`, `/terms` (SSG, không có data) → product pages (ISR) → `/dashboard` (SSR hoặc CSR) → `/checkout` cuối cùng vì đây là critical path, sai là mất revenue.

Về parallel deployment: Next.js chạy song song với SPA cũ. Proxy (nginx hoặc Cloudflare) route từng path prefix một sang Next.js. SPA cũ là fallback cho tất cả routes chưa được migrate. Không public Next.js cho real users cho đến khi ổn định qua dark launch hoặc internal test.

Về rollback trong 5 phút: feature flag tại proxy level đơn giản và nhanh nhất. Thay đổi 1 dòng trong nginx config hay 1 toggle trong Cloudflare → toàn bộ traffic về SPA cũ ngay lập tức, không cần redeploy Next.js. Đây nhanh hơn nhiều so với rollback bằng git revert + redeploy.

Về metadata trong giai đoạn transition: khi một route đã migrate sang Next.js, metadata phải được migrate cùng lúc — không để route live trên Next.js mà vẫn dùng React Helmet client-side. Verify bằng `curl` và `view-source` trước khi public.
</details>

## 13. 🎯 Summary

- **React là library, Next.js là orchestrator** — React handle UI, Next.js thêm routing, rendering strategies, optimization, và build pipeline. Không phải replacement, là layer trên top.

- **Chọn rendering strategy theo data requirements, không phải theo default hay "cho chắc"** — hỏi data freshness, SEO requirement, và traffic scale trước khi quyết định. SSR không phải luôn tốt nhất.

- **Next.js là BFF + UI layer, không phải backend replacement** — API routes phù hợp cho proxy và simple CRUD. Business logic phức tạp (transactions, background jobs, domain logic) thuộc về backend service riêng.

- **Migration từ legacy SPA phải dùng Strangler Fig, không Big Bang** — route-by-route, parallel deployment, với rollback plan có cơ chế kỹ thuật cụ thể. Staff Engineer không bet toàn bộ production vào 1 migration không có escape hatch.

## 14. 📖 Terminology Evolution

### SSR (Server-Side Rendering)

```
┌──────────────┬───────────────────┬───────────────────────────────────────────────────┐
│ Era          │ Status            │ Ý nghĩa thực tế                                  │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ Pages Router │ Active            │ Page dùng getServerSideProps → HTML render mỗi   │
│              │                   │ request, không cache. "Page-level per-request."   │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ App Router   │ Redefined (!)     │ Server Components render trên server — nhưng có  │
│              │                   │ thể static, cached, hoặc dynamic tùy config.      │
│              │                   │ SSR ≠ "render mỗi request" nữa.                  │
└──────────────┴───────────────────┴───────────────────────────────────────────────────┘
```

**Lịch sử thay đổi:** "Page-level per-request rendering" (Pages Router) → "server-rendered components có thể static/dynamic" (App Router). Đây là nguồn confusion lớn nhất khi đọc tài liệu Next.js viết ở các giai đoạn khác nhau — luôn check version khi đọc docs.

**Codebase legacy:** Tìm `getServerSideProps` trong thư mục `pages/` — đây là SSR Pages Router kiểu cũ.

---

### Hydration

```
┌──────────────┬───────────────────┬───────────────────────────────────────────────────┐
│ Era          │ Status            │ Ý nghĩa thực tế                                  │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ Tất cả       │ Stable —          │ React "takes over" DOM từ HTML server đã render:  │
│ versions     │ nghĩa không đổi   │ attach event handlers, reconcile virtual DOM với  │
│              │                   │ server HTML, biến static HTML thành interactive.  │
└──────────────┴───────────────────┴───────────────────────────────────────────────────┘
```

**Lịch sử thay đổi:** Khái niệm ổn định qua mọi version. Điều thay đổi là App Router giảm lượng JavaScript cần hydrate — Server Components không gửi JS sang client, không tự hydrate. Ít JS hơn → hydration nhanh hơn.

**Codebase legacy:** Lỗi `"Text content does not match server-rendered HTML"` — nguyên nhân phổ biến: `typeof window`, `Date.now()`, `Math.random()`, `localStorage`, hay bất kỳ browser-only API nào được dùng trong render cycle.

---

### Meta-framework

```
┌──────────────┬───────────────────┬───────────────────────────────────────────────────┐
│ Era          │ Status            │ Ý nghĩa thực tế                                   │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ Tất cả       │ Active            │ Framework xây trên top của framework / library    │
│ versions     │                   │ khác. Next.js = React meta-framework: thêm        │
│              │                   │ routing, SSR, SSG, API layer, deployment          │
│              │                   │ conventions lên React.                            │
└──────────────┴───────────────────┴───────────────────────────────────────────────────┘
```

**Lịch sử thay đổi:** Khái niệm ổn định. Ví dụ khác trong ecosystem: Nuxt.js (Vue meta-framework), SvelteKit (Svelte), Remix (React, cạnh tranh với Next.js).

**Codebase legacy:** Khái niệm lý thuyết — không có code signature cụ thể để nhận diện.

---

### RSC (React Server Components)

```
┌──────────────┬───────────────────┬───────────────────────────────────────────────────┐
│ Era          │ Status            │ Ý nghĩa thực tế                                   │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ App Router   │ Active (default)  │ Components render trên server, không gửi JS sang  │
│              │                   │ client, không tự hydrate. Giảm bundle size.       │
│              │                   │ KHÔNG phải "SSR cũ" — model hoàn toàn mới.        │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ Pages Router │ N/A               │ Không có RSC — dùng traditional SSR / CSR model.  │
└──────────────┴───────────────────┴───────────────────────────────────────────────────┘
```

**Lịch sử thay đổi:** RSC là model mới trong App Router — không có tương đương trong Pages Router. Junior thường nhầm "render on server" với "SSR kiểu getServerSideProps". Đây là hai thứ hoàn toàn khác: RSC là về *where* component renders và *what JS gets sent to client*, không phải về caching behavior.

**Codebase legacy:** RSC chỉ tồn tại trong App Router. Không xuất hiện trong thư mục `pages/`.

---

### PPR (Partial Prerendering)

```
┌──────────────┬───────────────────┬───────────────────────────────────────────────────┐
│ Era          │ Status            │ Ý nghĩa thực tế                                   │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ Next.js 16+  │ Active            │ Static shell render lúc build time, dynamic       │
│              │                   │ "islands" stream vào sau. Kết hợp tốc độ SSG      │
│              │                   │ với flexibility của SSR trong cùng 1 page.        │
├──────────────┼───────────────────┼───────────────────────────────────────────────────┤
│ Pre-16       │ Experimental      │ Có thể có nhưng không stable — không dùng         │
│              │                   │ production.                                       │
└──────────────┴───────────────────┴───────────────────────────────────────────────────┘
```

**Lịch sử thay đổi:** PPR là feature mới trong Next.js 16 — không có tương đương trong Pages Router hay Next.js phiên bản cũ. Measurement từ case study: TTFB giảm ~420ms → 85ms, FCP 1.2s → 0.6s.

**Codebase legacy:** N/A — feature mới, không xuất hiện trong legacy codebase.
