**2. Evolution Context (Bài này xuất hiện để giải quyết vấn đề gì?)**

**Trước đó**

- 2010–2013: Server-rendered HTML (PHP/Rails) → SEO tốt, nhưng UX chậm (full page reload)
- 2013–2016: React SPA (CRA) → UX mượt, nhưng HTML rỗng, SEO chết, blank screen 3–5s

↓

**Vấn đề**

- CSR trả về `<div id="root"></div>` trống → Googlebot thấy trang trống
- Team tự implement SSR = Express + ReactDOMServer → routing conflict, XSS, blocking event loop, 1–2 tuần setup
- Mỗi team làm khác nhau → không nhất quán, không scalable

↓

**Giải pháp**

- Next.js 1.0 (Oct 2016): Pre-render HTML + SPA navigation + zero-config SSR
- Lấy cái hay từ cả 2 thế giới: SEO của server-rendered + UX của SPA

**4. Core Concepts (20% kiến thức quan trọng nhất)**

| #   | Pain Point                                            | Next.js Solution                       |
| --- | ----------------------------------------------------- | -------------------------------------- |
| 1   | **Empty HTML** → SEO death                            | Pre-rendered HTML (SSR/SSG)            |

**5. Legacy Recognition (Nhìn code biết gì?)**

✅ **CSR-only (CRA):**

- `public/index.html` có `<div id="root"></div>` trống
- `package.json` có `"react-scripts"` (không có `"next"`)
- `curl <url>` → HTML trống

✅ **DIY SSR:**

- `server.js` dùng `ReactDOMServer.renderToString()`
- `window.__INITIAL_DATA__` pattern
- Routes định nghĩa ở 2 nơi (Express + React Router)

❌ **Không phải Next.js:**

- Không có `pages/` hoặc `app/`
- Không có `getServerSideProps` / `getStaticProps` / `generateStaticParams`

---

**6. Modern Approach (Ngày nay làm thế nào?)**

| Legacy                                   | Modern                                          |
| ---------------------------------------- | ----------------------------------------------- |
| CRA → CSR, HTML trống                    | Next.js → Pre-render, SEO hoạt động             |
| Express + React Router (2 nguồn sự thật) | File-based routing (1 nguồn sự thật)            |
| `window.__INITIAL_DATA__` thủ công       | `getServerSideProps` / `getStaticProps` tự động |
| Webpack config tay cho client + server   | Zero-config, built-in                           |
| 1 bundle 2–5MB                           | Auto code-splitting theo page                   |
| 1–2 tuần setup                           | 5 phút `create-next-app`                        |

**7. Production Scenario (Thực tế)**

**Case:** E-commerce CRA, organic traffic giảm 60% trong 6 tháng.

- **Context:** Public-facing site, 55% traffic từ Google
- **Problem:** `curl` trả về `<div id="root"></div>` trống → Google không index
- **Root Cause:** CSR, không có pre-render
- **Solution:** Migrate product/category pages sang Next.js `getServerSideProps`
- **Lesson:** Khi nhận task "improve SEO", câu hỏi đầu tiên: _"App có pre-render HTML không?"_

👉 **Rule:**

> Nếu gặp **public-facing pages + SEO quan trọng** hãy nghĩ ngay đến **pre-rendering (Next.js/Astro/SSR)**.

**8. Debugging Scenario (Debug nhanh)**

**Checklist — CSR hay SSR?**

- □ `curl <url> | head -50` → thấy content? (SSR) hay trống? (CSR)
- □ Tắt JavaScript → reload → content vẫn hiển thị? (SSR) hay trắng? (CSR)
- □ `Ctrl+U` (View Source) → khác với Inspect? (CSR = giống nhau, SSR = khác)
- □ Network → document request → Response tab → raw HTML có content?

**Lệnh hay dùng:**

```bash
# Test nhanh nhất
curl -s https://yoursite.com | grep -o '<div id="root">.*</div>'

# Kiểm tra meta tags (OG, SEO)
curl -s https://yoursite.com | grep -E 'og:|description'

# Lighthouse SEO audit
npx lighthouse https://yoursite.com --only-categories=seo
```

**9. Refactoring Scenario (Nhìn code cũ sẽ làm gì?)**

**Checklist khi tiếp nhận legacy codebase:**

1. `package.json` → có `"next"`? version nào? (Pages Router vs App Router)
2. Thư mục `pages/` hay `app/`? → Next.js. `src/pages/`? → CRA + react-router
3. `curl <url>` → content trong HTML? → SSR/SSG. Trống? → CSR
4. Data fetching: `getInitialProps`? `getServerSideProps`? `fetch()` trong RSC? → biết era
5. Có `server.js` tự viết? → DIY SSR, cần audit security/performance

**Không sửa khi:**

- App là internal dashboard (không cần SEO) → Vite + React đủ
- Team deadline gấp, không ai có Next.js experience → learning curve cao
- Heavy real-time app (WebSocket) → Next.js không phải công cụ đúng

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng Next.js:**

- Public pages cần SEO (landing, blog, e-commerce, news)
- Cần performance tốt trên mobile/slow networks
- Team lớn cần conventions thống nhất
- Hybrid: một số pages static, một số dynamic
- Fullstack: API routes + UI trong cùng repo

**Khi KHÔNG nên dùng:**

- Internal dashboard (sau login, SEO không quan trọng) → **Vite + React**
- Heavy real-time (live trading, collaborative editing) → **Custom WebSocket**
- Simple content site (blog cá nhân) → **Astro** (zero JS by default)
- Mobile app → **React Native / Expo**

**Trade-off:**

| Pros                                    | Cons                               |
| --------------------------------------- | ---------------------------------- |
| Pre-rendering out of the box            | Build time lâu hơn Vite            |
| File-based routing, auto code-splitting | App Router learning curve cao      |
| Image/Font optimization built-in        | Tie-in với Vercel ecosystem        |
| API routes (fullstack 1 repo)           | Debug SSR phức tạp hơn CSR         |
| Community lớn, ecosystem trưởng thành   | Một số libs không work với SSR/RSC |

**11. Interview Notes**

**Junior:**

- React = UI library. Next.js = fullstack framework trên React (SSR, SSG, routing, API routes)
- CSR trả về HTML trống → bot không thấy content → SEO chết

**Mid:**

- Hydration = React "attach" vào HTML server-rendered. Phức tạp vì mismatch risk (date, random, browser APIs)
- DIY SSR rủi ro: XSS, blocking event loop, routing desync, 1–2 tuần setup

**Senior:**

- Approach migration: diagnose trước (confirm CSR là root cause), incremental (Strangler Fig), define metrics trước (indexing rate, organic sessions, Core Web Vitals)
- 95% cases: tự build SSR = premature optimization. Next.js đã battle-tested.

## 4. Core Concepts — 6 Pain Points Thực Sự

### Pain Point #1: The Empty HTML Problem (SEO Death)

Đây là pain point số một khiến Next.js ra đời.

**Kịch bản thực tế:** Bạn build một e-commerce site bằng React + Create React App. Sản phẩm ra mắt tháng 1. Tháng 6, Google Search Console báo 0 pages indexed. Organic traffic (Lượng truy cập tự nhiên) giảm 80%.

**Nguyên nhân:** CSR trả về HTML shell rỗng. Content chỉ tồn tại sau khi JavaScript execute xong.

```html
<!-- Đây là thứ Googlebot nhìn thấy khi crawl CSR React app -->
<!-- (và vẫn còn đúng cho nhiều bots ngày nay) -->
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <title>My E-commerce Store</title>
    <!-- ❌ Không có meta description -->
    <!-- ❌ Không có Open Graph tags → Facebook/Twitter share không có preview -->
    <!-- ❌ Không có structured data → Google không hiểu đây là product page -->
  </head>
  <body>
    <div id="root">
      <!-- ⚠️ TRỐNG HOÀN TOÀN -->
      <!-- Toàn bộ content nằm trong bundle.js -->
      <!-- Bot phải execute JavaScript MỚI thấy được content -->
      <!-- Và hầu hết bots không làm vậy, hoặc không chờ đủ lâu -->
    </div>
    <script src="/static/js/bundle.js"></script>
    <!-- 2–4MB JavaScript phải download + parse + execute -->
    <!-- trước khi user hoặc bot thấy bất cứ thứ gì có ý nghĩa -->
  </body>
</html>
```

**Hậu quả cascade:**

- Google rank sản phẩm của bạn về 0 (không index được)
- Facebook/Slack share preview không có ảnh, không có title, không có description
- `curl https://yoursite.com/products/123` trả về trang trống → mọi bot đều thấy trang trống
- Bing, DuckDuckGo, LinkedIn scraper: tất cả gặp cùng vấn đề

**Phân tích Pain Point :**

| Thời điểm                                             | Đánh giá                                                                                                                                                         | Kết luận                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **2016 (khi Next.js ra đời)**                         | ✅ **ĐÚNG**                                                                                                                                                      | Googlebot thời đó (Chrome 41, headless) có hạn chế nghiêm trọng. "Two-wave indexing" chưa tồn tại. CSR gây vấn đề SEO thực sự.                                                                                                                                                                                                                                               |
| **Two-wave Indexing (2018 - khi Google I/O công bố)** | **Wave 1**: Crawl raw HTML (trống với CSR) → index ngay với content thiếu hoặc không index . **Wave 2**: Render JS → re-index sau (có thể vài giây đến vài tuần) | Khái niệm này chưa tồn tại khi Next.js ra đời 2016. Googlebot sử dụng **two-wave indexing** (Google I/O 2018, Tom Greenaway). Trong khoảng thời gian giữa 2 waves, page có thể bị index với title sai, description thiếu, hoặc không được index. Đây là lý do pre-rendered HTML vẫn là best practice — không phải vì Googlebot 'không render được', mà vì 'render bị delay'. |
| **Hiện tại (2024–2026)**                              | ❌ **SAI**                                                                                                                                                       | Googlebot đã evergreen (tự động update Chrome) từ 2019. CÓ render JS hoàn toàn.                                                                                                                                                                                                                                                                                              |

**Cách giải quyết tại thời điểm đó (2016):** ✅ **ĐÚNG** — Pre-rendered HTML là giải pháp hợp lý.

**Còn đúng ở hiện tại không?** ⚠️ **CẦN BỔ SUNG** — Vẫn cần pre-render, nhưng lý do đã thay đổi:

> "**Lưu ý (cập nhật 2024–2026):** Googlebot đã có khả năng render JavaScript hoàn toàn từ 2019 (evergreen Chrome). Tuy nhiên, vấn đề CSR vẫn tồn tại vì:
>
> 1. **Two-wave indexing delay**: Khái niệm 'two-wave indexing' đã được Google chính thức bác bỏ từ 2020. Martin Splitt (Google) tuyên bố đây là 'oversimplification'. Googlebot hiện render JS với median delay ~5 giây (2019). Tuy nhiên, pre-rendered HTML VẪN là best practice
> 2. **AI crawlers** (GPTBot, ClaudeBot, PerplexityBot — 2024–nay): **KHÔNG render JavaScript** → invisible với AI search. Theo Vercel research (Dec 2024): GPTBot traffic tăng 305% YoY, chiếm 30% AI crawler traffic. Phân tích 500M+ GPTBot fetches cho thấy zero evidence of JavaScript execution. ClaudeBot download JS files 23.84% thời gian nhưng không execute. Đây là lý do pre-render còn quan trọng hơn cả SEO truyền thống.
> 3. **Social bots** (Facebook, Twitter, LinkedIn): Vẫn không render JS → preview thiếu
> 4. **Crawl budget**: Render JS tốn nhiều resource hơn → fewer pages crawled"
>
> **Kết luận**: Dù Googlebot đã cải thiện, pre-rendered HTML vẫn là best practice cho SEO và AI visibility."

### Pain Point #2: Blank Screen — Business Problem, không phải Technical Problem

```
Timeline người dùng thấy khi vào CSR React App (Mobile 3G, 2015):

 0ms  ──── Browser gửi request đến server
│
 150ms ─── Nhận HTML response (trống rỗng)
│           Browser bắt đầu download bundle.js
│
│     ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  ← Màn hình trắng
│
 3200ms ── bundle.js (2MB) download xong
│
 3400ms ── JavaScript parse & compile
│
 3500ms ── React render → DOM update
│           [User cuối cùng thấy content]
│
└──────────────────────────────────────────
         3.5 GIÂY nhìn màn hình TRẮNG
```

Google research (2016) — báo cáo "The Need for Mobile Speed": **53% người dùng mobile bỏ trang nếu load > 3 giây.**

Với CSR app không được optimize, 3–5 giây trước khi thấy content là bình thường. Đây không còn là technical problem — đây là **revenue problem**.

### Pain Point #3: Manual SSR — Khi Bạn Cố Tự Giải Quyết

Khi các team nhận ra vấn đề CSR, nhiều người cố tự implement SSR với Express + ReactDOMServer. Đây là khi mọi thứ trở nên **thực sự tồi tệ**:

```javascript
// DIY : Do it yourself
// server.js — "DIY SSR" với Express + React (2014–2016)
// Code này THỰC SỰ đã tồn tại trong nhiều production codebases

const express = require('express');
const React = require('react');
const ReactDOMServer = require('react-dom/server');

// Phải tự import từng component (không có code splitting)
const App = require('./src/App');

const app = express();

app.get('*', async (req, res) => {
  // ❌ Vấn đề 1: Không có convention nào cho "fetch data trước khi render"
  // Mỗi engineer tự quyết định cách implement
  // Mỗi route phải được handle thủ công
  let initialData = {};

  if (req.path === '/') {
    initialData.home = await fetchHomeData();
  } else if (req.path.startsWith('/products/')) {
    const productId = req.path.split('/')[2]; // ❌ Parsing thủ công, dễ sai
    initialData.product = await fetchProduct(productId);
  } else if (req.path.startsWith('/category/')) {
    const slug = req.path.split('/')[2];
    initialData.category = await fetchCategory(slug);
  }
  // ...tiếp tục với hàng chục routes khác
  // ❌ Nếu quên một route → server trả về trang thiếu data, không có error rõ ràng

  // ❌ Vấn đề 2: renderToString là SYNCHRONOUS và BLOCKING
  // Nó chiếm toàn bộ Node.js event loop trong suốt thời gian render
  // 1000 concurrent users render cùng lúc → server không responsive
  // Không có timeout, không có streaming
  const html = ReactDOMServer.renderToString(
    React.createElement(App, {
      data: initialData,
      url: req.url, // ❌ Phải tự truyền URL để React Router hoạt động server-side
    }),
  );

  // ❌ Vấn đề 3: Phải tự "inject" data để client biết hydrate với data gì
  // Không có cơ chế official. Mỗi team dùng convention riêng.
  // window.__INITIAL_DATA__ là pattern phổ biến nhất — không có gì enforce nó
  const safeData = JSON.stringify(initialData)
    .replace(/</g, '\\u003c') // ❌ Phải tự escape để tránh XSS
    .replace(/>/g, '\\u003e') // ❌ Nếu quên → security vulnerability
    .replace(/&/g, '\\u0026');

  res.send(`
    <!DOCTYPE html>
    <html>
      <head>
        <title>My App</title>
        <!-- ❌ Meta tags phải hardcode hoặc inject thủ công -->
      </head>
      <body>
        <div id="root">${html}</div>
        <script>
          window.__INITIAL_DATA__ = ${safeData};
        </script>
        <!-- ❌ Tất cả JS trong 1 bundle duy nhất, không có code splitting -->
        <script src="/bundle.js"></script>
      </body>
    </html>
  `);
});
```

```javascript
// client.js — Phía client phải "biết" convention mà server đã dùng
// ❌ Không ai enforce điều này
// Người mới join team phải đọc code để hiểu — nếu có tài liệu thì may

const initialData = window.__INITIAL_DATA__;
// ❌ Nếu server quên inject data → initialData là undefined → crash

ReactDOM.hydrate(
  // ❌ Component phải được setup để chấp nhận data theo format này
  // Không có type checking, không có validation
  <App
    data={initialData}
    url={window.location.pathname}
  />,
  document.getElementById('root'),
);
```

### Pain Point #4: Routing Conflict — Hai Source of Truth

```
Khi dùng Express + React Router cùng nhau:

Server routes (Express):         Client routes (React Router):
────────────────────────         ────────────────────────────
app.get('/products/:id', ...)    <Route path="/products/:id" />
app.get('/category/:slug', ...)  <Route path="/category/:slug" />
app.get('/about', ...)           <Route path="/about" />

❌ Hai nơi hoàn toàn riêng biệt định nghĩa routes
❌ Phải sync thủ công mỗi khi thêm/sửa route
❌ Thêm route trong React Router nhưng quên Express:
   → Navigation từ trang khác: HOẠT ĐỘNG
   → Direct URL hoặc F5: 404 NOT FOUND
   → Bug cực kỳ khó debug vì "nó hoạt động với tôi"
❌ Thêm route trong Express nhưng quên React Router:
   → Page load lần đầu: HOẠT ĐỘNG
   → Navigate bằng React Router: 404 hoặc render sai component
```

**Bug kinh điển nhất của era này:** App hoạt động hoàn hảo khi navigate từ trang này sang trang kia. Nhưng khi user bookmark URL và mở lại → 404. Tại sao? Server không biết route đó. Chỉ client-side router biết.

## 6. Modern Approach — Next.js Giải Quyết Thế Nào

Mỗi pain point có một solution cụ thể được baked in (tích hợp sẵn):

```
Pain Point                    → Next.js Solution
──────────────────────────────────────────────────────────────────────
#1 Empty HTML (SEO)          → Pre-render HTML trên server/build time
#2 Blank screen 3.5s         → HTML sẵn sàng trước khi JS load xong
#3 Manual SSR complexity     → Built-in SSR, zero config, conventions
#4 Routing conflict          → File-based routing (1 source of truth)
#5 1 giant bundle            → Automatic per-page code splitting
#6 DX hell (2 tuần setup)    → 5 phút với npx create-next-app@latest
```

```javascript
// pages/products/[id].js — Next.js Pages Router
// 6 pain points biến mất trong 1 file

// ✅ Convention rõ ràng: Next.js tự biết gọi hàm này trước khi render
// ✅ Params được parse tự động từ filename [id].js
// ✅ Data được inject vào props tự động (không cần window.__INITIAL_DATA__)
export async function getServerSideProps({ params }) {
  const product = await fetchProduct(params.id);

  if (!product) {
    return { notFound: true }; // ✅ 404 handling built-in
  }

  return { props: { product } };
}

// ✅ Component này được render trên server → HTML đầy đủ → SEO hoạt động
// ✅ Code của page này được split tự động → bundle nhỏ hơn
// ✅ Chỉ 1 file = 1 route (server + client cùng nhau, không conflict)
export default function ProductPage({ product }) {
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.price}</p>
    </div>
  );
}
```

**Điều quan trọng nhất Next.js làm:** Đưa ra **conventions** — cách làm được đồng thuận, không cần mỗi team tự phát minh lại bánh xe. Khi một engineer từ công ty A join công ty B, họ nhận ra ngay codebase vì cùng dùng Next.js conventions.

## 7. Production Scenario

### Case Study: E-commerce mất 60% organic traffic vì CSR

**Bối cảnh:** 2017. Một startup e-commerce build platform bằng React CRA. Organic traffic từ Google Search chiếm 55% tổng traffic.

**Timeline sự cố:**

- **Tháng 1:** Launch. Team hài lòng với speed development.
- **Tháng 3:** Organic traffic bắt đầu giảm nhẹ. Không ai để ý.
- **Tháng 6:** Organic traffic giảm 70%. Google Search Console: phần lớn product pages "Discovered – currently not indexed".
- **Tháng 7:** Emergency all-hands. CTO yêu cầu fix ASAP.

**Investigation:**

```bash
# Bước 1: Confirm hypothesis - Kiểm chứng giả thuyết
curl -A "Googlebot/2.1" https://store.example.com/products/bestseller-2024

# Output thực tế:
# <!DOCTYPE html><html><head><title>Store</title></head>
# <body><div id="root"></div><script src="/bundle.js"></script></body></html>
#
# → Googlebot thấy TRANG TRỐNG. Không có product name, không có price, không có description.

# Bước 2: Kiểm tra social preview
# Chia sẻ URL lên Slack/Facebook → Không có thumbnail, không có title
# og:title, og:description, og:image đều trống

# Bước 3: Lighthouse SEO audit
# SEO score: 42/100
# "Document does not have a meta description"
# "Links are not crawlable"
```

**Root cause:** Toàn bộ React app là CSR. Không có pre-rendering. Googlebot crawl và chỉ thấy HTML shell.

**Solution:** Migrate product pages và category pages sang Next.js với `getServerSideProps`.

**Kết quả sau 3 tháng:** Organic traffic phục hồi về 85% mức cũ, rồi vượt lên 120% (vì các page giờ được index đúng).

**Lesson cho Engineer:** Khi nhận task _"improve SEO"_, câu hỏi đầu tiên phải là: _"App có pre-render HTML không?"_ Đây là foundation — không có nó, mọi SEO optimization khác đều vô nghĩa.

## 10. Architecture Notes

### Khi nào KHÔNG cần Next.js

Đây là điều quan trọng nhất mà các tutorial thường bỏ qua:

```
✅ Dùng Next.js khi:
   - Public-facing pages cần SEO (landing pages, blogs, e-commerce, news)
   - App cần performance tốt trên mobile/slow networks
   - Team lớn cần conventions (Quy ước) thống nhất
   - Cần hybrid: một số pages static, một số pages dynamic
   - Fullstack: cần cả API routes và UI trong cùng 1 project

❌ Không dùng Next.js khi:
   - Internal dashboard (chỉ dùng sau login, SEO không quan trọng)
     → Vite + React: setup đơn giản hơn, HMR nhanh hơn, build nhanh hơn

   - Heavy real-time app (live trading, collaborative editing, live chat)
     → WebSocket server tùy chỉnh hoặc framework chuyên biệt phù hợp hơn

   - Simple content site (blog cá nhân, docs, portfolio)
     → Astro: zero JS by default, cực kỳ nhẹ, tốt cho content-heavy sites

   - Mobile app
     → React Native, Expo — Next.js là web framework

   - Team deadline gấp, không ai có Next.js experience
     → Learning curve (độ dốc học tập) của App Router không nên underestimate (xem nhẹ)
```

**Trade-off thực tế của Next.js:**

```
✅ Pros:
   - Pre-rendering out of the box
   - File-based routing (zero config)
   - Automatic code splitting
   - Image, Font optimization built-in
   - API routes (fullstack trong 1 repo)
   - Vercel ecosystem nếu deploy Vercel
   - Community lớn, ecosystem trưởng thành

❌ Cons:
   - Build time lâu hơn Vite cho dev
   - Complexity (Độ phức tạp) cao hơn CRA cho apps đơn giản
   - App Router có learning curve đáng kể (Server Components, caching, etc .)
   - Tie-in (gắn bó chặt chẽ) với Vercel (mặc dù self-hosting được)
   - Debugging SSR issues phức tạp hơn CSR
   - Một số libraries không work với SSR/RSC
```

**Senior Engineer Mindset:** Next.js là công cụ xuất sắc, không phải công cụ đúng cho mọi tình huống. Biết khi nào _không_ dùng nó quan trọng không kém biết cách dùng nó.

**Q1: React và Next.js khác nhau thế nào?**

<details>
<summary>Đáp án</summary>

### Mid Level

**Q3: Giải thích "hydration" trong context của SSR. Tại sao nó phức tạp?**

<details>
<summary>Đáp án</summary>

**Hydration là gì:**

Khi dùng SSR, server render HTML đầy đủ và gửi về client. Client nhận HTML này và hiển thị ngay (user thấy content). Sau đó JavaScript load xong và React cần "attach" vào HTML đã có — thêm event listeners, kết nối state — mà KHÔNG re-render lại từ đầu (vì sẽ gây flicker). Quá trình này gọi là hydration.

**Tại sao phức tạp:**

**1. Mismatch risk — Nguyên nhân số 1 gây bugs:**
HTML server render phải giống HỆTT với gì React sẽ render trên client. Nếu khác → React báo lỗi hoặc re-render toàn bộ (mất hết lợi ích SSR).

Common mismatches:

- Dates: Server (UTC) vs Client (local timezone)
- `window`/`localStorage`/`document` không tồn tại trên server
- `Math.random()` cho kết quả khác nhau mỗi lần

**2. "Uncanny Valley" — UX problem:**
Sau khi HTML render, trước khi JS load xong, page trông có vẻ interactive nhưng thực ra không hoạt động. User click button → không có gì xảy ra. Thời gian này gọi là "Time to Interactive gap".

**3. Complexity trong DIY SSR:**
Phải tự đảm bảo data server dùng để render giống hệt data client dùng để hydrate. Trong Next.js, framework handle điều này tự động qua `getServerSideProps`/`getStaticProps` → component props.

</details>

---

**Q4: Một team quyết định tự implement SSR với Express + ReactDOMServer thay vì dùng Next.js. Bạn thấy rủi ro gì?**

<details>
<summary>Đáp án</summary>

Đây là câu hỏi về trade-offs, không có đáp án duy nhất. Tôi sẽ phân tích theo nhiều chiều:

**Technical risks:**

- `renderToString` là synchronous, block Node.js event loop trong suốt thời gian render một page. Với high concurrency → performance bottleneck nghiêm trọng
- Phải tự manage routing giữa Express và React Router → dễ desync, gây 404 khi direct access
- Data injection thủ công (`window.__INITIAL_DATA__`) → XSS risk nếu không escape đúng
- Webpack dual config (server bundle + client bundle) → phức tạp, error-prone
- Không có official solution cho code splitting trong SSR context

**Team và operational risks:**

- 1–2 tuần setup vs 5 phút với `create-next-app`
- Engineer mới join phải học custom infrastructure (hạ tầng tuỳ chỉnh), không có docs chuẩn
- Không có community support (sự hỗ trợ của cộng đồng) khi gặp vấn đề
- Security updates (cập nhật bảo mật) và performance improvements (cải tiến hiệu năng) phải tự implement (thực hiện)

**Khi nào DIY có thể hợp lý (hiếm):**

- App có rendering requirements (yêu cầu hiển thị) đặc biệt mà Next.js không support
- Team có deep expertise (chuyên môn sâu) và dedicated bandwidth (băng thông dành riêng) để maintain (duy trì)
- Cần control hoàn toàn rendering pipeline (đường ống kết xuất) vì compliance/security requirements (yêu cầu tuân thủ/bảo mật)

**Câu trả lời thực tế của Senior Engineer:**

"Trong 95% cases (trường hợp), tự build SSR infrastructure (hạ tầng kết xuất phía máy chủ) là premature optimization (tối ưu hóa sớm) và tạo ra technical debt (nợ kỹ thuật) đáng kể. Next.js đã giải quyết những vấn đề này qua nhiều năm với battle-tested solutions (giải pháp đã được thử nghiệm), cộng đồng lớn, và active maintenance (bảo trì tích cực). Proposal của tôi: dùng Next.js, và chỉ xem xét DIY nếu có requirements (yêu cầu) cụ thể, đã được document rõ ràng, mà Next.js chứng minh được là không thể đáp ứng."

| 300 content creators, content thay đổi thường xuyên | **ISR (Incremental Static Regeneration)** → rebuild tự động khi content update, không cần redeploy |
| Editor dashboard (private)                          | **Server Components** cho data fetching, **Client Components** cho rich text editor                |
| Image-heavy blog                                    | `next/image` → automatic optimization, WebP/AVIF, responsive sizes                                 |
| OG tags, structured data                            | `generateMetadata` API → type-safe, automatic                                                      |

**CMS choice — Sanity vs Strapi:**

|                 | Sanity                               | Strapi                     |
| --------------- | ------------------------------------ | -------------------------- |
| **Hosting**     | Managed (SaaS)                       | Self-hosted (Node.js + DB) |
| **Client B**    | ✅ Không cần maintain infrastructure | ❌ Cần DevOps resource     |
| **Real-time**   | ✅ Live preview built-in             | ❌ Cần config thêm         |
| **Cost**        | Pay per usage                        | Free, nhưng server cost    |
| **300 editors** | ✅ Scales automatically              | Cần scale manually         |

→ **Sanity** cho Client B vì budget không cho phép DevOps overhead.

**App Router vs Pages Router:**

- **App Router (2023+):** Server Components mặc định → less JS shipped to client → faster TTI cho readers
- **Nested layouts:** Blog layout (header, footer, sidebar) + Article layout (TOC, related posts) → shared UI không re-render
- **Streaming:** Dashboard có thể stream từng section (stats, recent posts, notifications) thay vì chờ tất cả data

**Trade-offs chấp nhận**

- ❌ **Vendor lock-in Vercel** — App Router hoạt động tốt nhất trên Vercel. Self-host được nhưng config phức tạp hơn.
- ❌ **Learning curve** — App Router (Server Components, caching behavior) đòi hỏi team có experience hoặc thời gian học
- ❌ **Sanity cost** — 300 editors có thể vượt free tier, cần budget recurring
- ✅ **Gain:** SEO tốt nhất có thể, editor experience professional, scale đến 500K readers không cần refactor

**Decision: Tldraw/Yjs + Custom WebSocket Server + Vite/React Frontend**

// Stack gợi ý

  "frontend": "Vite + React + Tldraw (canvas) hoặc Yjs (text)",

**Reasoning**

**Tại sao KHÔNG Next.js:**

| Next.js Feature               | Reality cho Client C                                              |
| ----------------------------- | ----------------------------------------------------------------- |
| SSR/SSG                       | Không cần — app sau login, content là user-generated              |
| API Routes                    | HTTP request/response model, không phù hợp persistent WebSocket   |
| File-based routing            | 5,000 concurrent connections → routing không phải bottleneck      |
| **What Next.js doesn't have** | **Built-in WebSocket support, CRDT integration, presence system** |

**Real-time architecture — đây là core problem:**

```
Client A ──┐
Client B ──┼──► WebSocket Server Cluster ──► CRDT Merge ──► PostgreSQL
Client C ──┘         (uWebSockets.js)        (Yjs)           (Source of truth)
     ▲                                              │
     └──────────────────────────────────────────────┘
              (Operational Transform / CRDT sync)
```

**Tech stack chi tiết:**

| Layer                | Technology                                     | Lý do                                                                                                       |
| -------------------- | ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **CRDT Engine**      | Yjs                                            | Industry standard cho collaborative editing. Conflict resolution automatic, offline support built-in        |
| **WebSocket Server** | uWebSockets.js (Node.js) hoặc Phoenix (Elixir) | 5,000 concurrent connections cần high-perf WebSocket. uWebSockets.js handle 100K+ connections trên 1 server |
| **Frontend**         | Vite + React + Tldraw                          | Tldraw là canvas-based editor với built-in multiplayer (dùng Yjs). Không cần reinvent                       |
| **Database**         | PostgreSQL + Redis                             | PG lưu document snapshots. Redis lưu user presence (who's online, cursor positions)                         |

**Tại sao không dùng Next.js API Routes cho WebSocket:**

- Next.js API Routes là **serverless functions** — stateless, timeout sau vài giây
- WebSocket cần **persistent connection** → không tương thích với serverless model
- Có thể dùng **Next.js cho marketing site** (landing page, docs) nhưng **không cho editor**

**Architecture đề xuất**

```
┌─────────────────────────────────────────────┐
│           Next.js (Landing, Docs, Pricing)  │  ← SEO pages, static
│           Deploy: Vercel                    │
└─────────────────────────────────────────────┘
                      │
┌─────────────────────────────────────────────────┐
│           Vite + React (Editor App)             │  ← Auth required, SPA
│           Deploy: CDN (CloudFront/Cloudflare)   │
│           Real-time: WebSocket to custom server │
└─────────────────────────────────────────────────┘
                      │
┌──────────────────────────────────────────────┐
│           WebSocket Cluster (uWebSockets.js) │  ← Horizontal scaling
│           Load Balancer (sticky sessions)    │
│           Redis (presence, pub/sub)          │
└──────────────────────────────────────────────┘
                      │
┌─────────────────────────────────────────────┐
│           PostgreSQL (document storage)     │  ← Source of truth
│           Yjs (CRDT merge logic)            │
└─────────────────────────────────────────────┘
```

**Trade-offs chấp nhận**

- ❌ **Split codebase** — 2 repos (Next.js cho marketing, Vite cho editor) → overhead maintain
- ❌ **Custom infrastructure** — Cần DevOps expertise cho WebSocket scaling, không có managed solution plug-and-play
- ❌ **Complexity cao** — CRDT, operational transform, conflict resolution là domain knowledge sâu
- ✅ **Gain:** Performance và reliability là competitive moat. Google Docs không dùng Next.js — họ dùng custom infrastructure vì lý do này.

|                   | SSE                                                             | WebSocket                        |

| Complexity        | Lower (HTTP-based)                                              | Higher                           |
| Scale             | 100K students trong 1 session?                                  | Cần horizontal scaling           |
| **Decision**      | **SSE cho broadcast**, **WebSocket (Ably) cho interactive Q&A** |

**Caching strategy phức tạp:**

- ❌ **Next.js App Router learning curve** — Team cần hiểu Server Components, caching, streaming. Không phải "zero-config" ở scale này.
- ✅ **Gain:** SEO cho catalog, security cho video, scale cho live events — tất cả trong 1 framework, 1 deployment platform.

## 13. Summary (Tóm tắt)

```
Next.js ra đời tháng 10/2016 để giải quyết 6 pain points cốt lõi
của React SPA và DIY SSR:

#1 Empty HTML        → ✅ Pre-rendered HTML (SEO hoạt động)
#2 Blank screen 3.5s → ✅ Content visible ngay lập tức
#3 Manual SSR hell   → ✅ Zero-config SSR với conventions rõ ràng
#4 Routing conflict  → ✅ File-based routing (1 source of truth)
#5 1 giant bundle    → ✅ Automatic per-page code splitting
#6 DX hell           → ✅ 5 phút setup, conventions nhất quán
```

**Takeaway #1 — Framework là convention, không phải magic:**
Next.js không làm gì mà bạn không thể tự làm. Nó làm những thứ đó theo một cách chuẩn, được đồng thuận, đủ tốt cho 95% use cases.

**Takeaway #2 — Hiểu pain point = Hiểu sâu hơn:**
Mỗi tính năng trong Next.js tồn tại vì một lý do cụ thể. Khi bạn hiểu lý do đó, bạn sẽ biết khi nào dùng, khi nào không dùng, và khi nào cần workaround (giải quyết tạm thời).

**Takeaway #3 — Công cụ đúng, không phải công cụ tốt nhất:**
Next.js là công cụ xuất sắc — khi dùng đúng chỗ. Internal dashboard không cần Next.js. Simple blog có thể dùng Astro. Heavy real-time app cần kiến trúc khác. Senior engineer biết phân biệt.
