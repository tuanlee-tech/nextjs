# M0.3: Toolchain & Project Anatomy

> 🎯 **Lesson Goal:** Scaffold project với `create-next-app`, giải thích role của Turbopack vs webpack, đọc `next.config.ts` và hiểu các field cơ bản — sẵn sàng cho production setup từ ngày đầu tiên.

---

## 1. Evolution Context

Trước khi Next.js tồn tại, để có một React app chạy được, bạn phải tự cấu hình Webpack, Babel, TypeScript compiler, PostCSS, và tất cả các plugin liên quan. Không nói đến việc chúng phải làm việc được với nhau — mà còn phải maintain chúng khi version bump.

**Cách cũ (2016–2022): `create-react-app` (CRA)**

CRA ẩn đi toàn bộ config đó. Nghe hay. Nhưng khi bạn cần custom bất cứ thứ gì — `eject`. Và sau khi `eject`, bạn sở hữu toàn bộ đống config đó mãi mãi, không có đường quay lại. Ngoài ra, Webpack ở thời điểm đó bắt đầu chạm trần hiệu suất: cold start trên codebase lớn mất 30–60 giây, HMR lag. Team dành nhiều thời gian chờ bundler hơn là viết code.

**Timeline thực tế:**

- **2016–2023:** `create-next-app` + Webpack ra đời như một giải pháp zero-config thực sự. Không cần eject. Config ẩn đi, nhưng vẫn override được qua `next.config.js`.
- **2024:** Webpack chạm trần. Vercel phát triển Turbopack (viết bằng Rust) như một bundler thế hệ mới. Được ship dưới dạng `experimental` trong Next.js 15.
- **2025 (Next.js 16):** Turbopack trở thành default. `next.config.ts` được support native (không còn chỉ `.js`). Node.js ≥ 20.9.0 là yêu cầu bắt buộc. Khái niệm "setup phức tạp" về cơ bản đã bị xóa bỏ cho Next.js project mới.

Bài này không phải dạy bạn "Next.js dùng thế nào". Bài này dạy bạn hiểu **tại sao** cái scaffold bạn vừa tạo ra trông như vậy — và tại sao những quyết định đó được đưa ra.

## 2. Mental Model

### [ANALOGY] Bundler là gì?

Hãy tưởng tượng bạn có một nhà máy đóng hàng. Nguyên liệu đầu vào là hàng trăm file TypeScript, CSS, hình ảnh, font. Đầu ra là một vài file JavaScript nhỏ gọn mà browser có thể chạy được. Bundler là dây chuyền lắp ráp của nhà máy đó.

Webpack là dây chuyền lắp ráp thế hệ cũ — cực kỳ linh hoạt, có thể gắn thêm mọi loại thiết bị (plugins/loaders), nhưng chậm vì được xây bằng JavaScript. Mỗi lần khởi động, nó phải đọc lại và phân tích toàn bộ đầu vào từ đầu.

Turbopack là dây chuyền lắp ráp thế hệ mới, xây bằng Rust — ngôn ngữ hệ thống chạy gần với phần cứng hơn. Nó biết file nào thay đổi, chỉ rebuild phần đó, và thực hiện song song. Cold start nhanh hơn, HMR gần như tức thì.

### [MODEL] Tại sao pnpm chặn được phantom dependencies?

npm dùng "flat `node_modules`": tất cả package (kể cả dependencies của dependencies) đều nằm cùng một cấp trong `node_modules/`. Kết quả: bạn có thể `import` một package mà bạn **không khai báo** trong `package.json` của mình — vì nó đã được flatten lên đó bởi package khác.

Đây là "phantom dependency". Nó chạy tốt trên máy bạn. Nhưng khi package kia bump version và không còn kéo theo dependency đó nữa — code của bạn break ngầm lúc production.

pnpm dùng symlink structure với strict hoisting: package của bạn chỉ "thấy" những gì bạn đã khai báo tường minh trong `package.json`. Phantom dependency sẽ fail ngay lúc dev, không phải lúc production.

## 3. Core Concepts

### 3.1 `create-next-app` Scaffold

[FACT] Chạy `create-next-app` với Next.js 16 tạo ra cấu trúc sau (Theo official docs: `nextjs.org/docs/app/api-reference/cli/create-next-app`):

```
my-app/
├── app/                  ← App Router entry point
│   ├── layout.tsx        ← Root layout
│   ├── page.tsx          ← Root page (route "/")
│   └── globals.css
├── public/               ← Static assets (ảnh, favicon, robots.txt)
├── next.config.ts        ← Next.js framework config
├── tsconfig.json         ← TypeScript compiler config
├── package.json          ← Dependencies & scripts
└── .gitignore            ← Và .next/ phải có ở đây
```

**4 file/thư mục cần hiểu ngay hôm nay — không cần đọc gì thêm:**

| File/Thư mục | Role | Đọc hôm nay? |
| :--- | :--- | :--- |
| `app/` | Nơi bạn viết UI. Mọi route đều bắt đầu từ đây. | ✅ Hiểu cấu trúc cơ bản |
| `public/` | Static assets — browser fetch trực tiếp, không qua bundler | ✅ Biết để đặt file đúng chỗ |
| `next.config.ts` | Config của Next framework (không phải TypeScript compiler) | ✅ Đọc ngay bên dưới |
| `package.json` | Dependencies, scripts, Node version requirement | ✅ Xem scripts section |
| `tsconfig.json` | TypeScript compiler config | ⏸️ Khi cần path alias mới động vào |
| `app/layout.tsx` | Root layout chi tiết | ⏸️ M1.x |
| `.eslintrc.json` | ESLint rules | ⏸️ Khi lint bắt đầu annoy team |

Risk Flag: Đừng đọc hết toàn bộ scaffold ngay hôm nay. Đây là bẫy phổ biến nhất khiến fresher mất focus. Bạn sẽ học từng file khi cần dùng đến nó.

### 3.2 Turbopack, Scripts & Node.js Runtime

[FACT] Next.js 16 set Turbopack làm **default bundler**. Chạy `next dev` sẽ tự động dùng Turbopack — không cần flag, không cần config. (Nguồn: Next.js 16 Release Notes)

[FACT] Next.js 16 yêu cầu **Node.js ≥ 20.9.0**. (Nguồn: Official Installation docs)

[FACT] Ba scripts chính và sự khác biệt quan trọng (Nguồn: Next.js CLI Docs):

```json [package.json]
{
  "scripts": {
    // next dev: Development server. Compile on-demand, HMR bật sẵn, không optimize.
    // KHÔNG dùng cái này trên production — không bao giờ.
    "dev": "next dev",

    // next build: Tạo optimized production bundles trong .next/
    // Đây là bước transform code của bạn thành thứ browser thực sự chạy.
    // Output là static files + server-side bundles.
    "build": "next build",

    // next start: Chạy production Node server, đọc từ .next/ đã build sẵn.
    // Phải chạy `next build` TRƯỚC khi chạy `next start`.
    // Đây là gì chạy trên server của bạn.
    "start": "next start",

    // next lint: Chạy ESLint với ruleset của next (eslint-config-next)
    "lint": "next lint"
  },
  // engines field: Contract với CI/CD về Node version.
  // Nếu CI runner dùng Node version khác, `pnpm install` sẽ fail ngay — không fail ngầm lúc runtime.
  "engines": {
    "node": ">=20.9.0"
  }
}
```

**Tại sao pnpm thay vì npm?**

```bash
# ❌ npm — flat node_modules, phantom dependencies có thể sneak in
npm install

# ✅ pnpm — strict hoisting, chỉ thấy những gì bạn khai báo
pnpm install

# pnpm chạy script tương đương npm run dev
pnpm dev
pnpm build
pnpm start
```

**Khi nào webpack thay vì Turbopack?**

[FACT] Webpack vẫn có thể dùng với `--webpack` flag nếu bạn phụ thuộc vào một Webpack loader chưa được port sang Turbopack. Đây là con đường thoát, không phải lựa chọn mặc định. (Xem Trade-offs section để hiểu khi nào dùng.)

### 3.3 `next.config.ts` Basics & `next/font`

[FACT] `next.config.ts` hỗ trợ TypeScript native từ Next.js 16. Không phải `tsconfig.json` (file đó là config của TypeScript compiler, khác hoàn toàn). (Nguồn: `nextjs.org/docs/app/api-reference/config/next-config-js`)

```ts [next.config.ts]
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  // reactStrictMode: Cố tình unmount/remount component ở dev để detect side-effect leaks.
  // Bạn sẽ thấy console.log in ra 2 lần — đó là đúng, không phải bug.
  // Chỉ xảy ra ở development. Production không bị ảnh hưởng.
  // Để true — đừng tắt nó đi.
  reactStrictMode: true,

  images: {
    // remotePatterns: Next.js Image Optimization block external images theo mặc định.
    // Đây là bảo vệ chống SSRF — không phải Next.js khó dùng.
    // Phải khai báo domain tường minh trước khi dùng <Image src="https://..." />
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'images.unsplash.com',
        // pathname: '/photos/**', // Optional: restrict thêm nếu cần
      },
    ],
  },

  // turbopack: Top-level config trong Next.js 16.
  // KHÔNG phải experimental.turbopack (đó là config của Next.js 15 — xem Anti-Patterns).
  // Object rỗng là đủ nếu không cần custom gì.
  turbopack: {},
}

export default nextConfig
```

**`next/font` — tại sao không dùng Google Fonts CDN?**

[FACT] `next/font` tự động download font tại build time, inject vào layout dưới dạng CSS variables, self-hosted hoàn toàn — không gọi external CDN khi user load page. (Nguồn: `nextjs.org/docs/app/building-your-application/optimizing/fonts`)

```ts [app/layout.tsx]
import { Inter, Geist_Mono } from 'next/font/google'

// next/font download font tại BUILD TIME, không phải runtime.
// Browser nhận được font từ server của bạn, không phải fonts.googleapis.com.
// Benefit: Loại bỏ network round-trip tới Google CDN.
// Benefit quan trọng hơn: Pre-compute font metrics → eliminate Layout Shift (CLS).
const inter = Inter({
  subsets: ['latin'],
  // display: 'swap' là mặc định và đúng — text vẫn hiển thị với fallback font
  // trong khi custom font đang load.
  display: 'swap',
  // variable: Inject vào CSS variable để dùng trong Tailwind hoặc CSS modules.
  variable: '--font-inter',
})

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en" className={inter.variable}>
      <body>{children}</body>
    </html>
  )
}
```

Nếu bạn dùng `<link href="https://fonts.googleapis.com/..." />` trong `<head>`, Lighthouse sẽ penalize CLS score vì browser không biết trước kích thước font trước khi render. `next/font` giải quyết đúng tầng: build time, không phải runtime.

## 4. Anti-Patterns in the Wild

### AP1: Config `experimental.turbopack` từ Next.js 15 vào Next.js 16

❌ **Code Smell — Junior copy config cũ không check changelog:**

```ts [next.config.ts]
// ❌ Config từ Next.js 15 — KHÔNG dùng trong Next.js 16
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  experimental: {
    turbopack: true, // ← Field này đã bị remove khỏi experimental
  },
}

export default nextConfig
```

✅ **Staff Refactor — Move lên top-level, xóa experimental nếu trống:**

```ts [next.config.ts]
// ✅ Next.js 16 — turbopack là top-level config
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  reactStrictMode: true,
  // turbopack giờ ở top-level, không phải trong experimental.
  // Nếu không cần custom gì, object rỗng là đủ.
  // Turbopack tự động chạy khi gọi `next dev`.
  turbopack: {},
}

export default nextConfig
```

🧠 **Why:** Breaking change có chủ đích. Vercel muốn force cleanup: không thể âm thầm giữ experimental flag sau khi feature đã stable. Nếu giữ config cũ, dev server có thể báo lỗi "unrecognised config field" hoặc silently fallback về Webpack mà không báo gì — bạn ngồi chờ cold start 30 giây và không hiểu tại sao.

🔥 **Incident Risk:** Nếu Webpack fallback xảy ra âm thầm, dev performance tụt xuống đột ngột trên toàn team mà không ai hiểu tại sao. Check symptom: `next dev` log hiển thị `Using Webpack` thay vì `Using Turbopack`.

**Trade-off khi refactor:** Zero trade-off. Config cũ không có runtime benefit nào. Refactor ngay.

### AP2: Bỏ TypeScript vì "setup phức tạp"

❌ **Code Smell — Junior dùng `--js` flag để tránh TypeScript:**

```bash
# ❌ Dùng JavaScript để "đơn giản hóa"
npx create-next-app@latest my-app --js
# Kết quả: Không có type safety. Runtime errors phát hiện muộn.
```

✅ **Staff Refactor — Dùng default (TypeScript là mặc định từ Next.js 16):**

```bash
# ✅ Default command — TypeScript được setup zero-config
npx create-next-app@latest my-app
# Hoặc với pnpm
pnpm create next-app my-app

# Không cần đụng vào tsconfig.json cho đến khi cần path alias.
# Next.js đã cấu hình TypeScript đúng cho App Router từ đầu.
```

🧠 **Why:** Next.js 16 zero-config TypeScript nghĩa là bạn không cần làm gì. Không có `tsconfig.json` phức tạp để config, không có Babel preset TS nào để install. Đổi lại bạn nhận được type safety, autocomplete, và lỗi được phát hiện tại compile time thay vì production runtime.

🔥 **Incident Risk:** `TypeError: Cannot read properties of undefined` lúc 2 giờ sáng production. TypeScript sẽ bắt cái đó ngay lúc bạn viết code.

**Trade-off khi refactor:** Fresher cần thêm 15–30 phút đọc TS error messages lần đầu. Đó là cost duy nhất. Benefit: mọi runtime error được đẩy sớm về compile time.

### AP3: Commit `.next/` folder

❌ **Code Smell — `git add .` ngay sau `next build`:**

```bash
# ❌ Sau khi chạy next build lần đầu
git add .
git commit -m "initial commit"
# Bạn vừa commit hàng nghìn file hashed vào repo.
```

✅ **Staff Refactor — Verify `.gitignore` trước commit đầu tiên:**

```bash
# ✅ Trước khi commit lần đầu, luôn verify
cat .gitignore | grep ".next"
# Output phải chứa: .next

# Nếu chưa có (ví dụ: bạn tự init repo, không dùng create-next-app):
echo ".next" >> .gitignore
echo "out" >> .gitignore     # Export output nếu dùng static export
echo ".env*.local" >> .gitignore  # Environment variables

# Sau đó mới commit
git add .
git commit -m "chore: initial Next.js 16 scaffold"
```

```gitignore [.gitignore]
# Build output — KHÔNG commit
.next/
out/

# Dependencies — KHÔNG commit
node_modules/
.pnpm-store/

# Environment variables — KHÔNG commit
.env*.local
.env.local
.env.development.local
.env.test.local
.env.production.local

# OS files
.DS_Store
Thumbs.db
```

🧠 **Why:** `.next/` chứa hàng nghìn file hashed (`_buildManifest.js`, chunks, etc.) được generate lại mỗi lần build — kể cả khi code không thay đổi. Hash khác nhau trên từng máy. Kết quả: merge conflicts vô nghĩa trên mọi PR.

🔥 **Incident Risk:** Repo phình to gigabytes sau vài tuần. Git clone mất 10 phút. CI cache bị pollute. Merge conflicts xuất hiện mọi PR dù không ai đụng vào file đó. Không có trade-off — đây là sai hoàn toàn.

### AP4: Install global Next.js

❌ **Code Smell — Cài Next.js global rồi chạy trực tiếp:**

```bash
# ❌ Install global
npm install -g next
# Sau đó chạy
next dev
# Bạn đang dùng global Next.js version, không phải version trong package.json
```

✅ **Staff Refactor — Dùng local binary qua package.json scripts:**

```bash
# ✅ Dùng pnpm (hoặc npm run) — luôn dùng local binary
pnpm dev      # ← Đây là `next dev` từ node_modules/.bin/next
pnpm build
pnpm start

# Nếu cần chạy lệnh next trực tiếp mà không có script:
pnpm exec next info   # Verify Next.js version đang dùng
npx next info         # Tương đương với npm

# KHÔNG bao giờ: next dev (global), npm install -g next
```

🧠 **Why:** Khi bạn dùng global `next`, bạn dùng version bạn install lần cuối trên máy — không phải version trong `package.json`. Teammate khác có thể có global version khác. CI không có global Next.js. Kết quả: build output khác nhau giữa các môi trường, và không ai hiểu tại sao.

🔥 **Incident Risk:** Local: Next.js 16 (global cũ). Team member: Next.js 15 (global cũ hơn). CI: Next.js 16 (từ `package.json`). Ba môi trường, ba behavior. Deploy fail trên production mà dev environment pass — đây là dạng bug tốn nhiều thời gian nhất để debug.

**Trade-off khi refactor:** Gõ `pnpm dev` thay vì `next dev` — 3 ký tự thêm. Benefit: deterministic environment trên mọi máy và CI.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Turbopack (Next 16 Default)** | Cold start cực nhanh, HMR gần như instant, zero-config, Rust-based | Ecosystem plugins chưa phong phú bằng Webpack; một số loader chưa được port | 99% Next.js dev environments; new projects | Khi phụ thuộc một Webpack loader cụ thể chưa được port sang Turbopack | Không có |
| **Webpack (`--webpack` flag)** | Cực kỳ ổn định, mọi loader/plugin đều có, documentation đồ sộ | Cold start 10–30s trên codebase lớn, config phức tạp khi cần custom | Legacy migration, phụ thuộc plugin độc lạ (SVGR customized, Module Federation) | New projects, standard features, team không có Webpack expertise | Dev waiting time liên tục — productivity cost thực sự |
| **Vite + React (không dùng Next.js)** | HMR nhanh nhất ecosystem, config tối giản cho SPA | Mất App Router, mất SSR/SSG zero-config, tự quản lý routing, tự build SSR nếu cần | SPAs thuần, Design Systems standalone, Chrome Extension | SEO-required apps, heavy server-side data fetching, team nhỏ không muốn tự manage SSR | Phải tự build và maintain SSR logic nếu cần |
| **Custom Webpack (CRA style)** | 100% control mọi thứ | Maintenance nightmare, dễ bị lỗi bảo mật do outdated dependency, chi phí maintain cao | Không bao giờ trong context Next.js | Always | Toàn bộ thời gian của một Build Engineer chỉ để maintain toolchain |

## 6. Legacy Recognition

Khi bạn vào một codebase Next.js cũ, đây là những dấu hiệu nhận biết:

```ts [next.config.js]
// 🔴 Legacy signal 1: File là .js thay vì .ts
// → Next.js 15 hoặc cũ hơn

// 🔴 Legacy signal 2: experimental.turbopack
const nextConfig = {
  experimental: {
    turbopack: true,
  },
}
// → Next.js 15 config. Cần move lên top-level cho Next.js 16.

module.exports = nextConfig
```

```bash
# 🔴 Legacy signal 3: node_modules flat structure (npm)
# Check: ls node_modules | head -20 — thấy packages không khai báo trong package.json
# → npm thay vì pnpm

# 🔴 Legacy signal 4: Node version
node -v
# v18.x.x → CI sẽ fail với Next.js 16 (yêu cầu >=20.9.0)
# v20.x.x nhưng < 20.9.0 → Cũng fail

# 🔴 Legacy signal 5: .next/ có trong Git history
git log --all --full-history -- ".next/*" | head -5
# Nếu có output → Ai đó đã commit .next/ trước đây
# Cần: git filter-branch hoặc BFG Repo Cleaner để purge history
```

## 7. Modern Approach

### Migration từ Next.js 15 → 16

**Bước 1: Update Node.js version TRƯỚC (quan trọng — làm trước khi bump Next.js)**

```bash
# Check version hiện tại
node -v

# Update Node.js qua nvm (hoặc package manager tương đương)
nvm install 20
nvm use 20
nvm alias default 20

# Update .nvmrc để team cùng dùng một version
echo "20" > .nvmrc

# Update engines field trong package.json
# "engines": { "node": ">=20.9.0" }
```

```yaml [.github/workflows/ci.yml]
# Update CI runner — TRƯỚC khi merge bump Next.js version
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          # Đổi từ node-version: '18' sang '20'
          node-version: '20'
          cache: 'pnpm'
```

**Bước 2: Bump Next.js version và update config**

```bash
pnpm add next@latest react@latest react-dom@latest
```

## 8. Production Scenario

> 🔴 **Scenario 1: "Build thất bại vì Node version"**
>
> **Symptom:** CI pipeline throw `Error: The engine "node" is incompatible with this module. Expected version ">=20.9.0". Got "18.19.0"`.
>
> **Root cause:** `package.json` đã khai báo `"engines": { "node": ">=20.9.0" }` — đúng. Nhưng CI runner (GitHub Actions, CircleCI) vẫn đang dùng Node 18 image mặc định vì chưa update workflow file.
>
> **Detect:** Check log bước `pnpm install` trong CI. Tìm dòng `node -v` hoặc xem `runner.tool_cache`. Lỗi thường xuất hiện ở bước install, không phải bước build.
>
> **Fix:** Update `actions/setup-node` version trong workflow trước khi bump Next.js version. Thứ tự quan trọng.

---

> 🔴 **Scenario 2: "Dev chậm bất thường sau khi merge"**
>
> **Symptom:** Cold start từ 2–3 giây (bình thường với Turbopack) tụt xuống 20–30 giây. Save file phải chờ 3–5 giây mới reflect trên browser. Không ai thay đổi máy tính hay settings.
>
> **Root cause:** Next.js 16 silently fallback về Webpack nếu phát hiện config trong `next.config.ts` không tương thích với Turbopack. Ai đó vừa merge một thay đổi config có vấn đề.
>
> **Detect:** Nhìn terminal output ngay khi `pnpm dev` khởi động. Nếu thấy `▲ Next.js 16.x.x (Webpack)` thay vì `▲ Next.js 16.x.x (Turbopack)` → Bị fallback. Tìm commit gần nhất thay đổi `next.config.ts`.

---

> 🔴 **Scenario 3: Font loading gây Layout Shift (CLS) trên mobile**
>
> **Symptom:** Text nhảy vị trí khi load page trên mobile — từ fallback font (Arial/sans-serif) sang custom font. Lighthouse CLS score > 0.1 (Poor).
>
> **Root cause:** Dùng `<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=..." />` trong `<head>`. Browser render text với fallback font trước, sau đó download custom font và re-render — gây shift.
>
> **Detect:** Mở DevTools → Network tab → Filter by "Font". Nếu thấy request tới `fonts.googleapis.com` hoặc `fonts.gstatic.com` → Đây là vấn đề. Chạy `pnpm build && pnpm start` rồi Lighthouse audit để thấy CLS score.
>
> **Fix:** Migrate sang `next/font` (xem Section 3.3). Build time font loading + pre-computed metrics loại bỏ hoàn toàn CLS.

## 9. Performance & Overhead

**Turbopack vs Webpack — development cold start:**

| Codebase size | Webpack cold start | Turbopack cold start | Saving |
| :--- | :--- | :--- | :--- |
| Small (< 50 routes) | 5–10s | < 2s | 3–8s |
| Medium (50–200 routes) | 15–30s | 2–5s | 10–25s |
| Large (> 200 routes) | 30–60s | 5–10s | 20–50s |

*Số liệu dựa trên Next.js benchmark công khai và internal team experience. Số thực tế phụ thuộc vào hardware và complexity của từng module.*

Nếu bạn cần benchmark cụ thể cho project của mình:

```bash
# Benchmark cold start Turbopack
time pnpm dev --no-open
# Ctrl+C sau khi thấy "ready" message, đọc elapsed time

# Switch sang Webpack để so sánh
time pnpm dev --webpack --no-open
```

**`next/font` CLS elimination:**

`next/font` pre-compute `size-adjust`, `ascent-override`, `descent-override`, `line-gap-override` tại build time. Inject vào `@font-face` CSS trên server. Browser biết kích thước exact của font trước khi render HTML — không có layout shift vì không có bước re-calculate layout.

**pnpm disk saving:**

pnpm dùng content-addressable store với hard links. Package chỉ download và store một lần, dù được dùng trong 10 project. `node_modules/` trong từng project là symlink tới store. Trong môi trường CI với nhiều project, saving có thể lên 40–60% disk space.

## 10. Architecture Notes

**Khi nào buộc phải dùng `--webpack` trong Next.js 16:**

- Phụ thuộc vào `@svgr/webpack` với config custom (không phải default setup)
- Dùng Module Federation (Webpack 5 feature, chưa có Turbopack equivalent)
- Plugin xử lý file format đặc thù chưa có Turbopack loader tương đương

Trong những trường hợp này: dùng `--webpack` flag và track Turbopack compatibility page của Next.js để biết khi nào có thể migrate.

**Monorepo workspace — pnpm là bắt buộc:**

Trong monorepo với `packages/ui`, `packages/api`, `apps/web`, npm flat hoisting sẽ tạo ra phantom dependency ở mức monorepo — package trong `apps/web` có thể vô tình import code từ `packages/api` mà không khai báo dependency, và chỉ phát hiện khi deploy từng package riêng lẻ.

pnpm workspace với `shamefully-hoist: false` (default) enforce đúng boundary giữa các package.

**Deterministic builds — contract bắt buộc:**

```
package.json (version range) + lockfile (exact version) + engines field (runtime version) = Reproducible build
```

Thiếu bất kỳ một trong ba: build có thể pass local, fail CI, hoặc tệ hơn — pass mọi nơi nhưng behavior khác nhau giữa environments.

## 11. Interview Questions

### Q1: Sự khác biệt giữa `next dev` và `next build && next start` là gì? (Fresher)

<details>
<summary>Đáp án</summary>

`next dev` chạy development compiler với HMR bật sẵn, compile on-demand (chỉ compile route bạn đang request), không optimize output. Không bao giờ dùng trên production.

`next build` tạo optimized production bundles trong `.next/` — tree-shaking, minification, code splitting, static generation. Đây là bước transform code thành artifact deploy được.

`next start` chạy Node.js production server đọc từ `.next/` đã build sẵn. Không làm gì ngoài serve request. Phải chạy `next build` trước.

Ba lệnh, ba mục đích khác nhau hoàn toàn. `next start` mà không có `.next/` sẽ fail ngay.
</details>

---

### Q2: Tại sao pnpm được khuyến khích hơn npm trong Next.js project? (Junior)

<details>
<summary>Đáp án</summary>

npm dùng flat `node_modules` — tất cả package bao gồm transitive dependencies được flatten vào cùng một level. Kết quả: bạn có thể import package chưa khai báo trong `package.json` của mình vì nó đã được flatten lên bởi package khác. Đây là "phantom dependency".

Phantom dependency chạy tốt cho đến khi package kia thay đổi dependency tree — lúc đó code của bạn break ngầm trên CI mà không có lý do rõ ràng.

pnpm dùng symlink structure với strict hoisting. Package của bạn chỉ "thấy" những gì bạn khai báo tường minh. Phantom dependency sẽ throw `Module not found` ngay lúc dev, không phải production.

Trong monorepo workspaces, vấn đề này nghiêm trọng hơn nhiều — npm không enforce boundary giữa các workspace packages.
</details>

---

### Q3: Tại sao Next.js 16 đổi `turbopack` config lên top-level thay vì giữ trong `experimental`? (Mid)

<details>
<summary>Đáp án</summary>

Hai lý do:

Một: Đánh dấu Turbopack đã reach stability — không còn experimental. Semantic của `experimental` là "có thể thay đổi bất kỳ lúc nào, không có guarantee". Turbopack giờ là default bundler với stable API.

Hai: Force cleanup. Nếu giữ trong `experimental`, team có thể không bao giờ biết Turbopack đã stable và tiếp tục dùng config cũ. Breaking change này buộc mọi người phải đọc upgrade guide và dọn dẹp config legacy khi upgrade — thay vì âm thầm giữ flag experimental đến đời sau.

Đây là design decision: tạo ma sát nhỏ (breaking change) để enforce hygiene lớn (không giữ config deprecated).
</details>

---

### Q4: Nếu commit `.next/` folder vào Git, hậu quả tệ nhất là gì? (Mid)

<details>
<summary>Đáp án</summary>

Không phải chỉ repo phình to — tệ hơn là: merge conflicts liên tục trên file hashed (`_buildManifest.js`, `BUILD_ID`, chunk files) vì hash thay đổi mỗi lần build, trên mỗi máy khác nhau, dù code không thay đổi.

Kết quả thực tế: mọi PR đều có conflict. `git merge` và `git rebase` fail trên những file không ai đụng vào. CI/CD pipeline bị tắc nghẽn. Team mất 30–60 phút mỗi ngày chỉ để resolve conflict vô nghĩa.

Nếu đã xảy ra: phải dùng `git filter-branch` hoặc BFG Repo Cleaner để purge `.next/` khỏi toàn bộ Git history — đây là operation nguy hiểm, cần coordinate toàn team.
</details>

---

### Q5: `next/font` solve được vấn đề Performance gì mà CDN font hay self-host thủ công không làm được? (Senior)

<details>
<summary>Đáp án</summary>

Self-host thủ công giải quyết được vấn đề network round-trip tới CDN. Nhưng vẫn còn Layout Shift (CLS).

Vấn đề CLS xảy ra vì browser không biết kích thước của custom font trước khi download và parse nó. Browser render text với fallback font (Arial, sans-serif), sau đó khi custom font load xong, re-calculate layout với kích thước mới — gây shift.

`next/font` giải quyết ở tầng khác: nó pre-compute `size-adjust`, `ascent-override`, `descent-override` của custom font tại BUILD TIME, rồi inject vào `@font-face` CSS rule được serve từ server. Khi browser nhận HTML, nó đã có thông tin kích thước exact của custom font — fallback font được render với cùng bounding box. Không có re-layout, không có shift.

Đây không phải chỉ là optimization — đây là loại bỏ hoàn toàn một class of bug.
</details>

## 13. Summary

- **Next.js 16 = zero-config TypeScript + Turbopack default + `next.config.ts` native.** Không còn lý do để avoid TypeScript hay setup Turbopack thủ công. Nếu bạn đang làm điều đó, bạn đang làm thêm việc không cần thiết.
- **`pnpm` + `engines` field + `.gitignore` đúng là 3 việc PHẢI làm trước commit đầu tiên.** Không thỏa hiệp. Làm đúng từ đầu rẻ hơn nhiều so với fix sau.
- **Turbopack ≠ Turborepo.** Turbopack là bundler (thay webpack). Turborepo là monorepo build orchestrator. Hai tool khác nhau hoàn toàn, cùng nhà sản xuất. Nhầm lẫn này xảy ra trong mọi team mới.
- **`next/font` và `images.remotePatterns` là hai config không thể thiếu ngay từ scaffold.** Thiếu một trong hai là CLS incident hoặc build crash khi production.

## 14. Terminology Evolution

**Turbopack**
| Era | Status | Dùng khi nào |
| :--- | :--- | :--- |
| Next.js 14–15 | `experimental` | Dùng flag `experimental: { turbopack: true }` khi muốn test performance |
| Next.js 16+ | Default bundler — stable | Dùng mặc định, config top-level `turbopack: {}`, zero-config |

Lịch sử: Rust-based bundler thay thế Webpack làm default từ Next.js 16. Được build bởi Vercel với mục tiêu cold start nhanh hơn và HMR instant. **KHÔNG PHẢI Turborepo** — Turborepo là monorepo build orchestrator cũng của Vercel, giải quyết bài toán hoàn toàn khác.

Codebase legacy: `experimental: { turbopack: true }` → Xóa `experimental` block, thêm top-level `turbopack: {}`

---

**next.config.ts**
| Era | Status | Dùng khi nào |
| :--- | :--- | :--- |
| Pre-Next.js 16 | `next.config.js` — JavaScript only | Config bundler, images, headers, redirects |
| Next.js 16+ | `next.config.ts` — TypeScript native | Config Next framework với full type safety |

Lịch sử: File config của Next.js framework — bundler behavior, image optimization, headers, redirects. **KHÔNG PHẢI `tsconfig.json`** — `tsconfig.json` là config của TypeScript compiler, kiểm soát cách TypeScript compile code của bạn. Hai file, hai role, không liên quan.

Codebase legacy: Rename `next.config.js` → `next.config.ts` + `import type { NextConfig } from 'next'` + type annotation cho config object + đổi `module.exports` → `export default`

---

**HMR / Fast Refresh**
| Era | Status | Dùng khi nào |
| :--- | :--- | :--- |
| Webpack era | HMR (Hot Module Replacement) — generic | Reload chỉ module thay đổi, không reload toàn bộ page |
| Turbopack (Next.js 16+) | Fast Refresh — React-specific | HMR cho React components, giữ nguyên component state khi sửa code |

Lịch sử: HMR là cơ chế chung — thay thế module tại runtime mà không reload page. Fast Refresh là implementation cụ thể của Next.js/React cho HMR: khi sửa component, state không bị reset (ví dụ: text bạn đang gõ trong input không biến mất sau khi save file). Turbopack Fast Refresh nhanh hơn Webpack HMR vì incremental compilation được optimize tốt hơn. Junior thường nhầm lẫn với full page reload — đó là hoàn toàn khác, xảy ra khi sửa file ngoài component tree.
