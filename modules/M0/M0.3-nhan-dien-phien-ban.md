# M0.3 — Nhận Diện Phiên Bản Next.js: Đọc Kỷ Nguyên Từ Code



## 1. Lesson Goal

Sau bài này, bạn có thể:

- **Nhìn vào bất kỳ file Next.js nào** và xác định được nó thuộc era nào
- **Biết ngay constraints** — cái gì có thể làm, cái gì không thể làm, cái gì đang bị deprecated
- **Đọc code của đồng nghiệp** mà không cần hỏi "codebase này dùng Next.js bao nhiêu?"
- **Phát hiện code trộn lẫn nhiều era** — nguyên nhân phổ biến nhất của production bugs khi migration

> **Kỹ năng thực chiến:** Khi bạn join một công ty mới, khả năng đọc codebase trong 15 phút đầu tiên và hiểu bức tranh toàn cảnh là thứ phân biệt senior engineer với người chỉ biết code theo tutorial.

---

## 2. Evolution Context

### Tại sao cần nhận diện era?

Bạn không bao giờ bắt đầu từ zero. 99% công việc thực tế là:

- **Maintain** codebase đã có từ 2–5 năm trước
- **Review PR** của đồng nghiệp viết theo style cũ
- **Debug bug** mà nguyên nhân là mixing era patterns
- **Plan migration** mà không biết điểm xuất phát

Next.js thay đổi **rất nhanh và rất nhiều** so với hầu hết frameworks. Một feature hoạt động hoàn hảo ở v12 có thể bị **deprecated hoàn toàn** ở v13, và **bị remove** ở v15. Việc không nhận diện được era đồng nghĩa với việc bạn đang debug trong bóng tối.

### Timeline tổng quan

```
2016  Next.js 1.0    — SSR cơ bản, pages/ directory
2018  Next.js 7.0    — Error overlay, hooks support
2019  Next.js 9.0    — API Routes, getInitialProps chính thức
2020  Next.js 10.0   — getStaticProps/getServerSideProps, next/image
2021  Next.js 12.0   — Middleware, SWC compiler, React 18 concurrent
2022  Next.js 13.0   — App Router (BETA), Server Components, Turbopack
2023  Next.js 13.4   — App Router STABLE, Server Actions (beta)
2023  Next.js 14.0   — Server Actions STABLE, Partial Prerendering (exp)
2024  Next.js 15.0   — React 19, Async APIs, after()
2025  Next.js 16+    — "use cache" stable, cacheTag, updateTag
```

Mỗi bước nhảy tạo ra **một era riêng** với mental model, data fetching pattern, và performance trade-offs hoàn toàn khác nhau.

## 3. Mental Model — Code Archaeology

Nghĩ mình là **nhà khảo cổ học đọc tầng địa chất**.

Địa chất học có các tầng đất, mỗi tầng chứa dấu vết của một thời kỳ. Code cũng vậy — mỗi era để lại **dấu vân tay (fingerprints)** đặc trưng không thể nhầm lẫn.

```
                    Era Strata (Tầng kỷ nguyên)
    ┌──────────────────────────────────────────────────┐
    │  v16+   │  "use cache", cacheTag, updateTag      │  ← Hiện tại
    ├──────────────────────────────────────────────────┤
    │  v15    │  await cookies(), after(), async params │
    ├──────────────────────────────────────────────────┤
    │  v14    │  "use server" stable, useFormState      │
    ├──────────────────────────────────────────────────┤
    │  v13.4  │  app/ stable, Server Components        │
    ├──────────────────────────────────────────────────┤
    │  v13.0  │  app/ beta, "use client" xuất hiện     │
    ├──────────────────────────────────────────────────┤
    │  v10-12 │  getSSP, getStaticProps, ISR            │
    ├──────────────────────────────────────────────────┤
    │  v9     │  getInitialProps, pages/ thuần túy     │  ← Cổ nhất
    └──────────────────────────────────────────────────┘
```

**Quy tắc vàng:** Mỗi tầng có thể chứa code của tầng cũ hơn (backward compat), nhưng KHÔNG thể chứa code của tầng mới hơn. Nếu bạn thấy code từ nhiều tầng trộn lẫn — đó là codebase đang trong quá trình migration.

**Tín hiệu phụ trong `package.json`:**

| Package có mặt | Suy ra gì |
|---|---|
| `@next/bundle-analyzer` | Team quan tâm performance |
| `next-auth` hoặc `@auth/nextjs` | Có auth layer |
| `prisma` hoặc `drizzle-orm` | Full-stack app, DB layer trực tiếp |
| `axios` nhưng KHÔNG có `swr`/`@tanstack/react-query` | Pages Router era (fetch từ client phổ biến) |
| `@tanstack/react-query` | Data fetching phức tạp, có thể Pages hoặc App Router |
| `react-query` (v3, không phải @tanstack) | Code cũ, v10-12 era |

### 🔍 Lớp 4: Data Fetching Patterns — Trái Tim Của Era

Đây là lớp nhận diện **quan trọng và chi tiết nhất**. Data fetching pattern phản ánh trực tiếp mental model của mỗi era.

#### Era 1: `getInitialProps` (v9–v10, cách cũ)

```javascript
// pages/products.js
// getInitialProps chạy BOTH phía server (initial load) VÀ client (navigation)
// Đây là nguyên nhân chính tại sao nó bị deprecated

class ProductsPage extends React.Component {
  // ← Class component thường gặp ở era này
  static async getInitialProps(context) {
    // context: { req, res, pathname, query, asPath }
    // Vấn đề: context.req chỉ có khi render phía server
    // Khi navigate phía client: req === undefined → bug khó debug
    const res = await fetch(`https://api.example.com/products`);
    const products = await res.json();

    return { products }; // ← Trả về object, tự động merge vào props
  }

  render() {
    const { products } = this.props;
    return <div>{/* ... */}</div>;
  }
}

export default ProductsPage;

// NHẬN DIỆN: static async getInitialProps = Era 1 chắc chắn
// Nếu còn thấy class component = rất cũ (pre-hooks)
```

```javascript
// _app.js với getInitialProps — pattern rất phổ biến era này
// pages/_app.js
class MyApp extends App {
  static async getInitialProps({ Component, ctx }) {
    let pageProps = {};
    if (Component.getInitialProps) {
      pageProps = await Component.getInitialProps(ctx);
    }
    return { pageProps };
  }

  render() {
    const { Component, pageProps } = this.props;
    return <Component {...pageProps} />;
  }
}

export default MyApp;
```

#### Era 2: `getServerSideProps` & `getStaticProps` (v10–v12)

```javascript
// pages/products/[id].js
// getServerSideProps: Chạy trên server MỖI request — không bao giờ xuống client
// ← Đây là điểm khác biệt cốt lõi so với getInitialProps

export async function getServerSideProps(context) {
  // context: { params, req, res, query, preview, previewData, locale }
  const { id } = context.params;

  // Có thể dùng req.headers, cookies một cách AN TOÀN
  // vì đây LUÔN LUÔN chạy trên server
  const token = context.req.cookies['auth-token'];

  const res = await fetch(`https://api.example.com/products/${id}`, {
    headers: { Authorization: `Bearer ${token}` },
  });

  if (!res.ok) {
    return { notFound: true }; // ← Trả 404 page
  }

  const product = await res.json();

  return {
    props: { product },
    // KHÔNG có revalidate ở đây — đây là SSR, không phải ISR
  };
}

export default function ProductPage({ product }) {
  // ← Functional component với hooks — era này đã dùng hooks phổ biến
  return <div>{product.name}</div>;
}

// NHẬN DIỆN: export async function getServerSideProps = Era 2, SSR pattern
```

```javascript
// pages/blog/[slug].js
// getStaticProps + getStaticPaths = SSG pattern
// Cực kỳ phổ biến cho content sites, docs, blogs

export async function getStaticProps(context) {
  const { slug } = context.params;
  const post = await fetchPost(slug);

  return {
    props: { post },
    revalidate: 60, // ← revalidate = ISR! Tái tạo sau 60 giây nếu có request
    // Nếu không có revalidate → thuần SSG, không tự update
  };
}

export async function getStaticPaths() {
  const posts = await fetchAllPosts();

  return {
    paths: posts.map((post) => ({
      params: { slug: post.slug },
    })),
    fallback: 'blocking', // ← 'blocking' | false | true
    // false: 404 nếu path không có trong list
    // true: Show fallback UI, build in background
    // 'blocking': SSR cho path mới, không show fallback
  };
}

export default function BlogPost({ post }) {
  return <article>{post.content}</article>;
}

// NHẬN DIỆN:
// getStaticProps BÊN NGOÀI component = Static generation
// + revalidate = ISR
// + getStaticPaths = Dynamic SSG routes
```

#### Era 3: `next/link` Evolution — Chi Tiết Nhỏ, Bug Lớn

```jsx
// ERA 1-2 (trước v13): next/link YÊU CẦU <a> con
// pages/nav.js
import Link from 'next/link';

function Navigation() {
  return (
    <nav>
      {/* CÁCH CŨ: Phải có <a> bên trong */}
      <Link href="/about">
        <a className="nav-link">About</a>
      </Link>

      {/* Nếu KHÔNG có <a>, sẽ bị warning — và link không work đúng */}
      <Link href="/contact">
        Contact {/* ← BUG trong v12 trở về trước */}
      </Link>
    </nav>
  );
}

// ERA 4+ (v13+): next/link KHÔNG cần <a> nữa
// app/nav.tsx
import Link from 'next/link';

function Navigation() {
  return (
    <nav>
      {/* CÁCH MỚI: Link trực tiếp render <a> */}
      <Link href="/about" className="nav-link">
        About
      </Link>

      {/* Nếu vẫn có <a> bên trong → Bị DOUBLE anchor warning */}
      <Link href="/contact">
        <a>Contact</a> {/* ← BUG trong v13+ */}
      </Link>
    </nav>
  );
}

// NHẬN DIỆN:
// Link > a = Code cũ viết cho v12-
// Link không có a = Code mới, v13+
// Thấy cả hai trộn lẫn = Codebase đang migrate dở
```

#### Era 4: App Router Fetching (v13.4+)

```typescript
// app/products/page.tsx
// Server Component mặc định — KHÔNG cần export gì đặc biệt cho data fetching
// Chỉ cần async/await trong component!

async function ProductsPage() {
  // fetch() trong Server Component tự động được cache bởi Next.js
  // Đây là sự thay đổi cơ bản nhất so với Pages Router
  const products = await fetch('https://api.example.com/products', {
    // Cache options — không có gì tương tự trong Pages Router
    cache: 'no-store',          // ← Giống getServerSideProps: fresh mỗi request
    // next: { revalidate: 60 } // ← Giống ISR: revalidate sau 60 giây
    // (không có option nào) = cached mãi, giống getStaticProps
  }).then(res => res.json());

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>{product.name}</li>
      ))}
    </ul>
  );
}

export default ProductsPage;

// NHẬN DIỆN:
// async function Component() = Server Component, App Router
// fetch() với { cache } option = App Router data fetching
// Không có getServerSideProps/getStaticProps = KHÔNG phải Pages Router
```

```typescript
// app/products/[id]/page.tsx
// Dynamic route trong App Router
interface PageProps {
  params: { id: string };         // ← v13-v14: params là object bình thường
  searchParams: { [key: string]: string };
}

// v15 BREAKING CHANGE: params TRỞ THÀNH Promise!
// Đây là một trong những thay đổi gây confusion nhiều nhất
interface PagePropsV15 {
  params: Promise<{ id: string }>;    // ← v15+: phải await!
  searchParams: Promise<{ [key: string]: string }>;
}

// App Router v13-14
export default async function ProductPage({ params }: PageProps) {
  const { id } = params;                     // ← Sync, trực tiếp destructure
  const product = await fetchProduct(id);
  return <div>{product.name}</div>;
}

// App Router v15+
export default async function ProductPage({ params }: PagePropsV15) {
  const { id } = await params;              // ← PHẢI await! Nếu không → bug
  const product = await fetchProduct(id);
  return <div>{product.name}</div>;
}

// NHẬN DIỆN:
// params không await = v13-14 style
// await params = v15+ — đây là breaking change quan trọng
```

### 🔍 Lớp 5: Directives & Imports — Dấu Hiệu Rõ Ràng Nhất

```typescript
// "use client" — Xuất hiện từ v13
// ======================================
'use client'; // ← Dòng này = Component này là Client Component
              // KHÔNG có dòng này trong Pages Router — không tồn tại khái niệm

import { useState, useEffect } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

// ======================================
// "use server" — Xuất hiện stable từ v14
// ======================================
'use server'; // ← File này chứa Server Actions

export async function createProduct(formData: FormData) {
  // Hàm này chạy TRÊN SERVER khi được gọi từ client
  const name = formData.get('name');
  await db.product.create({ data: { name } });
  revalidatePath('/products');
}

// Hoặc inline trong Server Component:
async function ProductForm() {
  async function handleSubmit(formData: FormData) {
    'use server'; // ← Inline directive = Server Action ngay trong component
    await createProduct(formData);
  }

  return <form action={handleSubmit}>{/* ... */}</form>;
}

// "use cache" — Xuất hiện stable từ v16
// ======================================
'use cache'; // ← File này có thể cache kết quả

import { cacheTag, cacheLife } from 'next/cache'; // ← Import mới v16+

export async function getProducts() {
  cacheTag('products');          // ← Gắn tag để invalidate
  cacheLife('minutes');          // ← Định nghĩa TTL

  return products;
}

// Import patterns theo era
```

## 5. Legacy Recognition — Bản Đồ Fingerprints

### Bảng Tổng Hợp Nhận Diện Nhanh

| Signal | Era | Version |
|---|---|---|
| `class Component` + `getInitialProps` | Pre-modern | v9–v10 |
| `getInitialProps` trong functional component | Legacy | v9–v12 |
| `getServerSideProps` | Pages Router | v10–v12 |

## 9. Refactoring Scenario — Migrate `getServerSideProps` sang App Router

**Task:** Migrate một page từ Pages Router sang App Router, giữ nguyên behavior.

```typescript
// BEFORE: pages/user/[id].tsx — Pages Router
import type { GetServerSideProps } from 'next';

interface Props {
  user: User;
  posts: Post[];
}

// Sequential data fetching — waterfall pattern
export const getServerSideProps: GetServerSideProps<Props> = async ({ params, req }) => {
  const userId = params?.id as string;
  const token = req.cookies['auth-token'];

  // Fetch user trước
  const user = await fetch(`/api/users/${userId}`, {
    headers: { Authorization: `Bearer ${token}` },
  }).then(r => r.json());

  // Sau đó mới fetch posts — waterfall!
  const posts = await fetch(`/api/users/${userId}/posts`).then(r => r.json());

  return { props: { user, posts } };
};

export default function UserPage({ user, posts }: Props) {
  return (
    <div>
      <h1>{user.name}</h1>
      {posts.map(post => <PostCard key={post.id} post={post} />)}
    </div>
  );
}
```

```typescript
// AFTER: app/user/[id]/page.tsx — App Router
// Cơ hội để optimize: parallel fetching

import { cookies } from 'next/headers';  // ← App Router API

export default async function UserPage({
  params,
}: {
  params: { id: string }; // v14 style — nếu v15 thì params: Promise<{id: string}>
}) {
  const cookieStore = cookies(); // v14: sync — v15: await cookies()
  const token = cookieStore.get('auth-token')?.value;

  const userId = params.id;

  // Parallel fetching — không còn waterfall!
  // App Router cho phép điều này tự nhiên hơn
  const [user, posts] = await Promise.all([
    fetch(`/api/users/${userId}`, {
      headers: { Authorization: `Bearer ${token ?? ''}` },
      cache: 'no-store', // ← Tương đương getServerSideProps: fresh mỗi request
    }).then(r => r.json()),

    fetch(`/api/users/${userId}/posts`, {
      cache: 'no-store',
    }).then(r => r.json()),
  ]);

  return (
    <div>
      <h1>{user.name}</h1>
      {posts.map((post: Post) => (
        <PostCard key={post.id} post={post} />
      ))}
    </div>
  );
}

// NHẬN XÉT: Việc migrate này không chỉ là đổi syntax
// Đây là cơ hội để:
// 1. Loại bỏ waterfall fetching → performance tốt hơn
// 2. Tách thành multiple Server Components với Suspense boundaries
// 3. Chỉ rõ cache behavior một cách explicit
```

**Migrate từng bước an toàn:**

```
Bước 1: Tạo file app/user/[id]/page.tsx
  → Giữ nguyên pages/user/[id].tsx (Next.js cho phép cả hai cùng tồn tại)
  → Test app version trong dev

Bước 2: Xác nhận behavior giống nhau
  → Auth works? Data loading? Error states?

Bước 3: Xóa pages/user/[id].tsx
  → Chú ý: Next.js ưu tiên app/ route nếu cả hai cùng tồn tại
  → Nhưng không nên để cả hai — confusing và có thể có side effects

Bước 4: Update tests
  → Integration tests có thể cần update nếu dùng next/router mock
  → Chuyển sang next/navigation mock cho App Router
```

## 10. Architecture Notes

### Tại sao companies bị stuck ở era cũ?

**1. Migration risk quá cao:**

```
Codebase Pages Router → 50,000 LOC → 200 pages
Migration sang App Router → Rewrite 80% → Risk production outage
Business: "Tại sao phải làm vậy? App đang chạy tốt mà?"

→ Kết quả: Codebase bị frozen ở v12 dù Next.js đã lên v15
```

**2. Incremental migration trap:**

```
Tháng 1: Migrate 10 pages
Tháng 2: Tìm ra pages Router components không tương thích với App Router context
Tháng 3: Fix compatibility shim
Tháng 4: Shim gây bug mới
Tháng 5: Team burn out, migration bị đình chỉ
→ Kết quả: Codebase mắc kẹt ở trạng thái hybrid bất định
```

**3. "Good enough" trap:**

```
getServerSideProps: Vẫn hoạt động tốt cho hầu hết use cases
ISR: Vẫn là cách hợp lý để handle content sites
App Router: Mang lại benefits thực sự cho những app phức tạp
→ Nếu app không phức tạp, Pages Router vẫn là lựa chọn tốt
→ Không phải mọi codebase đều PHẢI migrate
```

### Khi nào PHẢI migrate, khi nào KHÔNG cần:

```
PHẢI migrate khi:
  → Cần Server Components để reduce JS bundle size đáng kể
  → Cần Streaming cho user experience tốt hơn
  → Cần "use cache" patterns (v16+)
  → App phức tạp với nhiều nested layouts động

KHÔNG cần migrate khi:
  → App đơn giản, Pages Router đủ dùng
  → Team nhỏ, migration cost cao hơn benefit
  → Codebase ổn định, ít feature mới
  → Business không muốn risk
```

## 11. Interview Questions

### Junior Level

**Q1: Sự khác biệt giữa `pages/` và `app/` directory trong Next.js là gì?**

<details>
<summary>Đáp án</summary>

`pages/` directory là **Pages Router** — hệ thống routing cổ điển của Next.js từ v9. Mỗi file trong `pages/` tự động trở thành một route. Data fetching được thực hiện thông qua các function đặc biệt export từ file: `getServerSideProps`, `getStaticProps`, `getInitialProps`.

`app/` directory là **App Router** — được giới thiệu stable từ v13.4. Sử dụng các convention files đặc biệt: `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`. Component mặc định là **Server Components** — render hoàn toàn trên server, không gửi JavaScript xuống client. Data fetching được thực hiện trực tiếp trong component với `async/await`.

Hai hệ thống có thể cùng tồn tại trong một project — thường thấy trong các codebase đang migration.

</details>

---

**Q2: Bạn thấy một file Next.js có dòng `export async function getServerSideProps`. File này có thể nằm trong `app/` directory không?**

<details>
<summary>Đáp án</summary>

**Không.** `getServerSideProps` là Pages Router API — chỉ hoạt động trong `pages/` directory. Nếu file này nằm trong `app/`, function đó sẽ bị **ignore hoàn toàn** — không có lỗi, nhưng cũng không chạy, gây ra bug không có dữ liệu một cách im lặng.

Trong App Router, data fetching được thực hiện trực tiếp trong component:
```typescript
// Tương đương getServerSideProps trong App Router:
export default async function Page() {
  const data = await fetch('...', { cache: 'no-store' });
  // ...
}
```

Đây là một trong những điểm confusing nhất khi migration và là nguồn gốc của nhiều silent bugs.

</details>

### Mid Level

**Q3: Tại sao `getInitialProps` bị deprecated và nên thay bằng gì?**

<details>
<summary>Đáp án</summary>

`getInitialProps` có một **vấn đề kiến trúc nghiêm trọng**: nó chạy **cả ở server lẫn client**. Điều này tạo ra:

**1. Code phải handle cả hai môi trường:**
```javascript
static async getInitialProps(ctx) {
  const isServer = !!ctx.req; // Phải check điều kiện liên tục
  if (isServer) {
    // Dùng ctx.req.cookies
  } else {
    // Dùng document.cookie
  }
}
```

**2. Bundle size tăng:** Mọi logic data fetching đều được gửi xuống client, kể cả những thứ chỉ cần chạy trên server.

**3. Automatic Static Optimization bị disabled:** Khi một page dùng `getInitialProps`, Next.js không thể static optimize page đó nữa.

**Thay thế:**
- `getServerSideProps` — chỉ chạy trên server, luôn luôn, đảm bảo
- `getStaticProps` — chạy lúc build time
- Trong App Router: data fetching trực tiếp trong Server Component

</details>

**Q4: Một developer viết code sau trong `app/profile/page.tsx`. Có gì sai không?**

```typescript
import { useRouter } from 'next/router';

export default function ProfilePage() {
  const router = useRouter();
  return <div>Profile</div>;
}
```

<details>
<summary>Đáp án</summary>

Có **hai vấn đề**:

**1. Import sai package:**
- `next/router` là **Pages Router** hook
- Trong `app/` directory phải dùng `next/navigation`:

```typescript
import { useRouter } from 'next/navigation'; // ← Đúng cho App Router
```

**2. Hooks không dùng được trong Server Component:**
- File trong `app/` mặc định là **Server Component**
- Server Components không thể dùng hooks (`useState`, `useEffect`, `useRouter`,...)
- Nếu cần router, phải thêm `'use client'` directive:

```typescript
'use client'; // ← Cần thiết nếu dùng hooks
import { useRouter } from 'next/navigation';

export default function ProfilePage() {
  const router = useRouter();
  return <div>Profile</div>;
}
```

Đây là pattern rất hay gặp khi developer quen Pages Router chuyển sang App Router.

</details>

### Senior Level

**Q5: Một team đang migrate codebase từ Next.js 12 (Pages Router) sang Next.js 14 (App Router). Họ muốn migrate từng page incrementally. Những rủi ro chính cần cân nhắc là gì?**

<details>
<summary>Đáp án</summary>

**Rủi ro 1: Context/State không tương thích giữa hai hệ thống**

`pages/_app.tsx` providers (Auth, Cart, Theme...) **không tự động áp dụng** cho `app/` routes. Developer phải setup lại providers trong `app/layout.tsx`. Nếu quên, các trang đã migrate sẽ thiếu context — thường biểu hiện dưới dạng "user bị logout" hoặc "cart bị reset" khi navigate đến trang mới.

**Rủi ro 2: Session/Cookie handling khác nhau**

Pages Router: `getServerSideProps` nhận `req.cookies`
App Router: `import { cookies } from 'next/headers'`

Nếu auth logic dùng session strategy khác nhau cho hai phần, user experience sẽ không nhất quán.

**Rủi ro 3: Shared component compatibility**

Các shared components dùng `useRouter` từ `next/router` sẽ crash khi render trong App Router (phải đổi sang `next/navigation`). Cần audit toàn bộ shared components.

**Rủi ro 4: CSS/Style isolation**

Global CSS từ `pages/_app.tsx` vẫn áp dụng, nhưng `app/globals.css` chỉ áp dụng nếu được import trong `app/layout.tsx`. Nếu cả hai import cùng một CSS file, styles có thể bị duplicate hoặc conflict.

**Chiến lược giảm rủi ro:**
- Feature flag để control việc serve page từ pages/ hay app/
- E2E tests cover critical paths trước khi migrate
- Migrate theo feature domain, không theo file order
- Dùng Strangler Fig pattern: migrate từng route, không migrate cả page cùng lúc

</details>

**Q6: Bạn thấy code sau trong một Next.js 15 codebase. Nó có bug không?**

```typescript
// app/user/[id]/page.tsx
export default async function UserPage({
  params,
}: {
  params: { id: string };
}) {
  const cookieStore = cookies();
  const session = cookieStore.get('session');

  const user = await db.user.findUnique({
    where: { id: params.id },
  });
```

<details>
<summary>Đáp án</summary>

Có **hai bugs** liên quan đến Next.js 15 breaking changes:

**Bug 1: `cookies()` phải được `await` trong v15**

```typescript
// v14 (đúng):
const cookieStore = cookies();

// v15 (đúng):
const cookieStore = await cookies(); // ← Phải await!
```

**Bug 2: `params` phải được `await` trong v15**

```typescript
// v14 (đúng):
params: { id: string }
const id = params.id;

// v15 (đúng):
params: Promise<{ id: string }>
const { id } = await params; // ← Phải await!
```

Code đúng cho Next.js 15:

```typescript
export default async function UserPage({
  params,
}: {
  params: Promise<{ id: string }>; // ← Promise type
}) {
  const { id } = await params;          // ← await params
  const cookieStore = await cookies();  // ← await cookies()
  const session = cookieStore.get('session');

  const user = await db.user.findUnique({ where: { id } });
  return <div>{user?.name}</div>;
}
```

Đây là **breaking change** trong v15, không có deprecation warning trong v14 — code v14 compile được nhưng có thể có subtle bugs khi migrate lên v15.

</details>

## 13. Summary

### Bảng Cheat Sheet — Nhận Diện Trong 60 Giây

```
Thấy pages/_app.js         → Pages Router, ngừng tìm kiếm app/ features
Thấy getInitialProps       → v9–v10 era, oldest pattern
Thấy getServerSideProps    → v10–v12, Pages Router SSR
Thấy getStaticProps        → v10–v12, Pages Router SSG
+ revalidate trong trên    → ISR pattern
Thấy app/layout.tsx        → App Router (v13.4+)
Thấy "use client"          → App Router, component cần browser
Thấy "use server"          → Server Actions (v14+)
Thấy await cookies()       → v15+ (sync = v13-14)
Thấy await params          → v15+ (sync = v13-14)
Thấy "use cache"           → v16+, Cache Components
Thấy cacheTag/cacheLife    → v16+, fine-grained cache control
```

### Kết Luận

**Code không chỉ là instructions cho máy tính. Code là lịch sử của những quyết định thiết kế.**

Khi bạn đọc `getServerSideProps`, bạn đang đọc quyết định năm 2020: "Chúng ta cần SSR nhưng getInitialProps có vấn đề." Khi bạn đọc `"use client"`, bạn đang đọc quyết định năm 2022: "Server Components là default, client là exception."

Kỹ năng nhận diện era không phải là việc nhớ syntax. Đó là khả năng **đọc ý định của người viết code**, hiểu **constraints họ đang làm việc trong đó**, và **biết trade-offs họ đã phải chấp nhận**.

Đây là thứ phân biệt người biết code với kỹ sư.
