**5. Legacy Recognition (Nhìn code biết gì?)**

✅ **Nhận diện từ cấu trúc thư mục:**

- `pages/` + `app/` cùng tồn tại → đang migration
- Chỉ `pages/` → Pages Router only
- Chỉ `app/` → Full App Router

✅ **Nhận diện từ data fetching:**

- `Component.getInitialProps` → Era 1 (v1–v8)
- `getServerSideProps` / `getStaticProps` → Era 2 (v9.3+)
- `async function Page()` → Era 3+ (v13+)
- `'use cache'` → Era 4 (v16)

✅ **Test 30 giây:**

```bash
cat package.json | grep '"next"'  # version
ls -la | grep -E 'pages|app'      # cấu trúc
grep -r "getInitialProps\|getServerSideProps\|getStaticProps" .  # pattern cũ
```

❌ **Không nhận diện bằng version một mình** — cal.com chạy v16 nhưng vẫn có `getInitialProps` (do dependency lock-in)

**6. Modern Approach (Ngày nay làm thế nào?)**

**Codebase Next.js 2026:**

```
app/
├── layout.tsx              ← Server Component (mặc định)
├── page.tsx                ← Server Component
├── (auth)/                 ← Route Group
├── dashboard/
│   ├── layout.tsx          ← Nested layout
│   └── @notifications/     ← Parallel Routes
└── api/
    └── webhooks/route.ts   ← Route Handler
```

```typescript
// Modern pattern (v16)
import { unstable_cacheTag as cacheTag } from 'next/cache';

async function getProduct(id: string) {
  'use cache';
  cacheTag(`product-${id}`);
  return db.products.findById(id);
}

export default async function ProductPage({ params }: { params: { id: string } }) {
  const product = await getProduct(params.id);
  return <article><h1>{product.name}</h1></article>;
}
```

**9. Refactoring Scenario (Nhìn code cũ sẽ làm gì?)**

**Checklist khi tiếp nhận legacy codebase:**

1. Version Next.js → biết era
2. Có `pages/` không? Có `app/` không? → hybrid hay single
3. Data fetching pattern → `getInitialProps`? `getXxxProps`? `async`?
4. Có `middleware.ts` hay `proxy.ts`? → v12+ hay v16+
5. Có experimental flags không? → bleeding edge features

**Không sửa khi:**

- `pages/` vẫn work tốt, không cần migrate (Next.js cam kết support lâu dài)
- Team chưa quen App Router → learning curve cao
- Third-party libs chưa compat với RSC

**Migration strategy:** Coexistence (chạy song song) → migrate từng phần → không big-bang

**10. Architecture Notes (Decision Framework)**

**Khi nào dùng Pages Router vs App Router (2026):**

| Dùng App Router                         | Dùng Pages Router           |
| --------------------------------------- | --------------------------- |
| Project mới từ đầu                      | Maintain legacy codebase    |
| Cần Server Components giảm JS bundle    | Libs chưa compat với RSC    |
| Cần nested layouts phức tạp             | Team chưa quen mental model |
| Cần streaming + Suspense                |                             |
| Cần colocation (data ở component level) |                             |

**Hybrid:** Dùng cho migration, không dùng làm long-term strategy

**Trade-off:**

| Pros App Router               | Cons App Router                     |
| ----------------------------- | ----------------------------------- |
| Less JS to client             | Learning curve cao                  |
| Component-level data fetching | Caching behavior phức tạp (đến v15) |
| Nested layouts                | Debug SSR issues khó hơn            |
| Streaming built-in            | Một số libs không work với RSC      |

## 4. Core Concepts — Timeline Đầy Đủ Theo Era

---

### 🏗️ Era 1 — Pages Router Genesis (v1–v8 | 2016–2019)

**Sứ mệnh của era này:** Chứng minh rằng SSR React không cần phải set-up một cách phức tạp và tốn nhiều thời gian

---

#### Next.js 1.0 — Tháng 10/2016

**Pain point giải quyết:** DIY SSR setup tốn 1–2 tuần, không có convention (quy ước) nào.

**Những thứ quan trọng được giới thiệu:**

- Zero-config SSR (không cần viết server.js thủ công)
- File-based routing: `pages/index.js` → route `/`, `pages/about.js` → route `/about`
- Hot module replacement built-in ( HMR ): là tính năng tự động cập nhật phần code vừa thay đổi trên trình duyệt khi đang development mà không cần reload toàn bộ trang, giúp thấy kết quả ngay lập tức và thường vẫn giữ được state hiện tại.
- Automatic code splitting theo từng page
- `_app.js` để wrap toàn bộ ứng dụng

**Code signature của era này:**

```javascript
// pages/index.js — Cách viết Next.js 1.x
// Không có TypeScript, không có dynamic routing

MyPage.getInitialProps = async (ctx) => {
  // getInitialProps chạy trên CẢ server lẫn client
  // ctx.req tồn tại trên server, undefined trên client
  // → Cần check thủ công: if (ctx.req) { ... }
  const res = await fetch('https://api.example.com/data');
  const data = await res.json();
  return { data };
};

export default function MyPage({ data }) {
  return <div>{data.title}</div>;
}
```

**Mental model của era này:** _"Đặt file vào pages/, mọi thứ sẽ hoạt động."_

#### Next.js 8.0 — Tháng 2/2019

**Pain point giải quyết:** Không thể deploy Next.js lên serverless platforms (Lambda, Netlify Functions).

**Thay đổi quan trọng:** Serverless mode — mỗi page trở thành 1 Lambda function độc lập.

```javascript
// next.config.js — v8 serverless mode
module.exports = {
  target: 'serverless', // ← Deprecated sau này nhưng là bước ngoặt đầu tiên
};
```

**Ý nghĩa:** Next.js bắt đầu hướng đến deployment linh hoạt, không chỉ traditional server.

### 🚀 Era 2 — Pages Router Maturity (v9–v12 | 2019–2022)

**Sứ mệnh của era này:** Biến Next.js từ "SSR framework" thành "fullstack framework".

#### Next.js 9.0 — Tháng 7/2019 \* ⭐ Milestone (Cột mốc) lớn

**Pain points giải quyết:**

- Dynamic routes buộc phải dùng custom Express server
- TypeScript phải config thủ công
- Không có built-in API layer (phải dùng server riêng hoặc Express)

**Ba tính năng thay đổi Next.js mãi mãi:**

**1. Dynamic Routing:**

```javascript
// pages/products/[id].js ← Tên file chứa [bracket]
// → Tự động match /products/1, /products/abc, /products/anything
// Không cần custom server nữa

export default function ProductPage({ product }) {
  return <div>{product.name}</div>;
}
// params.id được truyền tự động vào getInitialProps/getServerSideProps
```

**2. Built-in TypeScript:**

```bash
# Trước v9: phải tự cài @types/react, tsconfig.js, babel preset
# v9+: Chỉ cần đổi tên file .js → .tsx và Next.js tự detect
touch pages/index.tsx
# Next.js tự tạo tsconfig.json với config tối ưu
```

**3. API Routes:**

```javascript
// pages/api/users.js ← Đây là backend endpoint, chạy trên server
// → GET /api/users trả về JSON

export default function handler(req, res) {
  if (req.method === 'GET') {
    res.json({ users: [...] });
  }
}
```

**Ý nghĩa của API Routes:** Next.js không còn là frontend framework. Đây là thời điểm "fullstack với một codebase" trở thành thực tế.

---

#### Next.js 9.3 — Tháng 3/2020 \* ⭐ Milestone quan trọng

**Pain point giải quyết:** `getInitialProps` chạy trên cả server lẫn client, không thể dùng Node.js APIs thuần (như `fs`) một cách an toàn. Không có sự phân biệt rõ ràng giữa SSR và SSG.

**Ba data fetching functions thay thế `getInitialProps`:**

```javascript
// getServerSideProps — Chạy trên server MỖI REQUEST
// Dùng khi data thay đổi liên tục (user-specific, real-time)
export async function getServerSideProps(context) {
  const data = await fetchFromDB(context.params.id);
  return { props: { data } };
}

// getStaticProps — Chạy lúc BUILD TIME, KHÔNG chạy mỗi request
// Dùng khi data ít thay đổi (blog posts, product catalog)
export async function getStaticProps() {
  const posts = await fetchAllPosts();
  return { props: { posts } };
}

// getStaticPaths — Đi kèm getStaticProps cho dynamic routes
// Định nghĩa TRƯỚC danh sách paths cần generate lúc build
export async function getStaticPaths() {
  const posts = await fetchAllPosts();
  return {
    paths: posts.map((post) => ({ params: { slug: post.slug } })),
    fallback: false, // hoặc true, hoặc 'blocking'
  };
}
```

**Ý nghĩa:** Lần đầu tiên Next.js có **clear separation** giữa SSR và SSG. Đây là nền tảng cho mọi thứ sau này.

#### Next.js 9.5 — Tháng 7/2020

// middleware.ts — Chạy trên Edge Runtime và được thực thi trước khi request đến page, route handler hoặc API route.
// Không phải Node.js server, không phải client
// Dùng Web APIs (Request, Response, Headers)

/*
 *   Request
 *      ↓
 *   middleware.ts (Edge)
 *      ↓
 *   Rewrite / Redirect / Continue
 *      ↓
 *   Page / Layout / Route Handler / API
 *      ↓
 *   Response

import { NextResponse } from 'next/server';

export function middleware(request) {
  const token = request.cookies.get('auth-token');

  if (!token && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/dashboard/:path*'],
};

**Ý nghĩa của Middleware:** Auth check không còn cần custom Express server. Logic chạy gần user nhất (edge) mà không tốn server round-trip.

**So sánh Edge Runtime và Node.js Runtime**

| Edge Runtime                       | Node.js Runtime                       |
| ---------------------------------- | ------------------------------------- |
| Khởi động rất nhanh (low latency)  | Khởi động chậm hơn                    |

| Hỗ trợ Web APIs                    | Hỗ trợ đầy đủ Node APIs               |
| Có nhiều giới hạn                  | Ít giới hạn hơn                       |
| Phù hợp middleware, auth, redirect | Phù hợp xử lý business logic phức tạp |

#### Next.js 15.0 — Tháng 10/2024

**Pain point giải quyết:** Caching trong v13/v14 quá "magic" ( ảo lòi ) — opt-out thay vì opt-in gây nhiều surprise bugs ( bugs bất ngờ ).

**Thay đổi caching philosophy (tư duy caching ):**

```javascript
// v13/v14: fetch() cache theo mặc định (opt-out)

// v15: Không cache theo mặc định (opt-in)

fetch('/api/data', { cache: 'force-cache' }); // → Phải opt-in
```

**Tính năng mới:**

- React 19 support
- Turbopack stable cho `next dev`
- `after()` API - Non-blocking side effects: chạy code sau khi response đã được gửi. Request đã được trả response về client, nhưng server vẫn tiếp tục chạy thêm logic nền (background task). Dùng `after(() => { logic background })`. ❗ after() KHÔNG đảm bảo 100% chạy. Chỉ nên chạy cho các tác vụ không quan trọng (Logging, analytics, Debug tracing...) Có thể bị kill process (serverless cold start / shutdown) Không phù hợp cho logic quan trọng (payment, database transaction, queue job,...)
- Cải thiện error messages cho hydration mismatch

#### Next.js 16 — Tháng 10/2025 (21/10/2025)

**Pain point giải quyết:** Cache control còn implicit (ngầm định) và khó debug. Cần cache primitives rõ ràng hơn.

**`"use cache"` directive (chỉ thị) — Cache trở thành explicit (rõ ràng):**

```javascript
// Trước v16: Cache behavior phụ thuộc vào nhiều factors ngầm định
// v16: Khai báo rõ ràng những gì cần cache

'use cache'; // ← Directive ở đầu file hoặc function

export async function getProductData(id) {
  'use cache'; // ← Cache function-level
  const data = await db.products.findById(id);
  return data;
}
```

**`cacheTag` và `cacheLife`:**

```javascript
import {
  unstable_cacheTag as cacheTag,
  unstable_cacheLife as cacheLife,
} from 'next/cache';

export async function getProductData(id) {
  'use cache';
  cacheTag(`product-${id}`); // ← Tag để invalidate (vô hiệu hóa) theo nhóm
  cacheLife('hours'); // ← Thời gian sống của cache

  return await db.products.findById(id);
}
```

**`revalidateTag` và `updateTag`:**

```javascript
// Invalidate cache theo tag
import { revalidateTag } from 'next/cache';
revalidateTag(`product-123`); // → Tất cả data có tag này bị invalidate

// updateTag — "Đọc những gì bạn vừa viết"
// Đảm bảo ngay sau khi mutation, user thấy data mới nhất
// Không cần wait cho revalidation cycle
```

// app/products/[id]/page.tsx — Modern Next.js pattern

async function getProduct(id: string) {

## 9. Refactoring Scenario

### Coexistence — Pages Router và App Router cùng tồn tại

Từ v13, Next.js hỗ trợ chạy song song cả hai:

```
project/
├── pages/
│   ├── _app.tsx     ← Pages Router vẫn hoạt động
│   └── old-page.tsx ← Route cũ vẫn chạy
└── app/
    └── new-page/    ← Route mới dùng App Router
        └── page.tsx
```

**Nguyên tắc coexistence:**

- `pages/` và `app/` không được có cùng route path
- `pages/` vẫn dùng `_app.tsx`, `_document.tsx`
- `app/` dùng `layout.tsx` riêng
- Shared components nên ở `components/` hoặc `lib/` bên ngoài cả hai

**Đây chính là chiến lược migration thực tế** — không cần rewrite toàn bộ cùng lúc.

## 10. Architecture Notes

### Khi nào dùng Pages Router vs App Router trong 2026

```
Dùng App Router khi:
✅ Project mới từ đầu (không lý do gì để dùng Pages Router)
✅ Cần Server Components để giảm JS bundle
✅ Cần nested layouts phức tạp
✅ Cần streaming và Suspense boundaries
✅ Cần colocation (data fetching ở component level)

Dùng Pages Router khi:
✅ Maintain legacy codebase (không refactor khi không cần)
✅ Một số third-party libraries chưa compat với RSC
✅ Team chưa quen với Server Components mental model (short-term)

Dùng hybrid (cả hai) khi:
✅ Đang trong quá trình migration (intentional transition)
⚠️ KHÔNG dùng hybrid làm long-term strategy
   → Tăng complexity, hai mental models trong cùng project
   → Có migration deadline rõ ràng
```

### Việc deprecated và removal trong Next.js

```
getInitialProps: Không bị remove, nhưng không nên dùng trong code mới

pages/ directory: Không bị remove trong v13/v14/v15/v16.
Team Next.js đã cam kết support Pages Router "for multiple major versions" (từ blog v13.4, May 2023).
Đến 2026 vẫn chưa có timeline removal.

target: 'serverless': Deprecated từ v12 (không cần thiết nữa)

→ Next.js ít khi breaking remove. Nhưng không dùng patterns cũ trong code mới.
```

## 11. Interview Questions

**Q1: Next.js 9 có ý nghĩa gì với hệ sinh thái Next.js?**

<details>
<summary>Đáp án</summary>

Next.js 9 (2019) là release quan trọng nhất trong giai đoạn đầu vì ba lý do:

**1. Dynamic Routing `[id].js`:** Xóa bỏ nhu cầu cuối cùng phải dùng custom Express server. Trước v9, dynamic routes buộc phải viết thêm server riêng. Sau v9, hoàn toàn zero-config.

**2. API Routes:** Biến Next.js từ frontend framework thành fullstack framework. Giờ đây một developer có thể xây dựng toàn bộ ứng dụng — frontend, backend API, data fetching — trong một codebase Next.js duy nhất.

**3. TypeScript native:** Không cần config thủ công. Đây là thời điểm TypeScript adoption trong Next.js tăng vọt.

Ba tính năng này cùng nhau định nghĩa "Next.js Pages Router" như chúng ta biết ngày nay.

</details>

---

**Q2: Tại sao Next.js 13 được coi là "paradigm shift", không phải chỉ là feature update?**

<details>
<summary>Đáp án</summary>

Next.js 13 (App Router + Server Components) không chỉ thêm tính năng — nó thay đổi **fundamental mental model**:

|                           | Pages Router                  | App Router                      |
| ------------------------- | ----------------------------- | ------------------------------- |
| Default rendering         | Client Component              | Server Component                |
| Data fetching             | Page level (`getStaticProps`) | Component level (`async/await`) |
| Layouts                   | Tự implement trong `_app.js`  | Built-in nested layouts         |
| JavaScript sent to client | Tất cả components             | Chỉ `'use client'` components   |
| Streaming                 | Limited                       | First-class với Suspense        |

Đây là shift từ **"React chạy trên server lần đầu rồi hydrate"** sang **"Components thực sự chạy hoàn toàn trên server, không gửi JavaScript xuống client"**.

Một developer Pages Router giỏi cần học lại cách tư duy khi chuyển sang App Router. Đó là dấu hiệu của paradigm shift thực sự.

</details>

---

**Q3: Tại sao Next.js 15 đảo ngược caching behavior (từ opt-out sang opt-in)?**

<details>
<summary>Đáp án</summary>

Trong Next.js 13/14, `fetch()` cache theo mặc định. Ý định ban đầu tốt: tự động optimize performance.

**Vấn đề thực tế nảy sinh:**

- Developer không nhận ra data đang được cache → bug "tại sao data không update?"
- Phải ghi nhớ thêm `{ cache: 'no-store' }` cho mọi dynamic fetch
- Debugging cache behavior cực kỳ khó vì không explicit
- Team mới join không hiểu tại sao behavior khác nhau giữa dev và production

**Quyết định của v15:** Opt-in caching — rõ ràng hơn, ít surprise hơn.

```javascript
// v13/v14: Surprise cache
fetch('/api/data'); // ← Đang cache! Bạn có biết không?

// v15: Explicit intent
fetch('/api/data', { cache: 'force-cache' }); // ← Rõ ràng là đang cache
fetch('/api/data'); // ← Rõ ràng là không cache
```

Đây là ví dụ điển hình về "framework học từ production feedback của cộng đồng". Nguyên tắc "explicit tốt hơn implicit" đôi khi phải học qua đau thực tế.

</details>

<details>

<summary>Đáp án Exercise 1 — Era Dating</summary>

**Code A:** `MyPage.getInitialProps` → **Era 1 — Pages Router Genesis (v1–v8, 2016–2019)**

```javascript
MyPage.getInitialProps = async ({ req, query }) => {
  const isServer = !!req;
  return { data: await fetchData(query.id), isServer };
};
```

**Bằng chứng nhận diện:**

| Dấu hiệu                                                            | Ý nghĩa                                                                                                                         |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `MyPage.getInitialProps = ...`                                      | Gắn function vào component như property — đây là API duy nhất để fetch data **trước v9.3**                                      |
| `isServer = !!req`                                                  | Dòng kinh điển của era này. Vì `getInitialProps` chạy **cả server lẫn client**, dev phải tự check `req` để biết mình đang ở đâu |
| `query.id` (không phải `params.id`)                                 | Trước v9 dynamic routing chưa tồn tại → query string là cách duy nhất lấy dynamic segment. V9+ mới có `[id].js`                 |
| Không có TypeScript, không có `getServerSideProps`/`getStaticProps` | Hai function này chỉ xuất hiện từ v9.3, sau era này                                                                             |

**Mental model của era này:** "Một function duy nhất xử lý mọi nơi, dev tự phân biệt server/client." Đây là lý do v9.3 phải ra đời — pattern này quá dễ gây bug.

---

**Code B:** `getStaticProps` + `revalidate` → **Era 2 — Pages Router Maturity, cụ thể ISR (v9.5+, 2020)**

```javascript
export async function getStaticProps() {
  return { props: { data: await fetchAll() }, revalidate: 3600 };
}
export async function getStaticPaths() {
  return { paths: [], fallback: 'blocking' };
}
```

**Bằng chứng nhận diện:**

| Dấu hiệu                                | Ý nghĩa                                                                                                                     |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `getStaticProps`                        | Tách riêng SSG khỏi SSR — chỉ có từ **v9.3+** (ra mắt T3/2020)                                                              |
| `revalidate: 3600`                      | ISR (Incremental Static Regeneration) — **chỉ có từ v9.5+** (ra mắt T7/2020). Trước v9.5 chỉ có SSG thuần (build-time only) |
| Cặp `getStaticProps` + `getStaticPaths` | Pattern đặc trưng cho dynamic routes cần pre-render. `getStaticPaths` cũng từ v9.3                                          |
| `fallback: 'blocking'`                  | Ba giá trị `'blocking'` được thêm ở các minor versions sau; việc dùng nó cho thấy code này từ thời kỳ v9.5 trở đi           |
| Vẫn ở `pages/` directory                | Chưa có `app/` → vẫn thuộc Pages Router, không phải App Router                                                              |

**Mental model của era này:** "Build-time là chính, nhưng cho phép tái sinh theo chu kỳ." Đây là khái niệm Next.js tiên phong — nhiều framework khác sau này copy (Astro, Nuxt...).

---

**Code C:** `async function Page({ params })` → **Era 3 — App Router Revolution (v13+, 2022 trở đi)**

```javascript
export default async function Page({ params }) {
  const data = await fetch(`/api/${params.id}`, { cache: 'no-store' });
  return <div>{(await data.json()).name}</div>;
}
```

**Bằng chứng nhận diện:**

| Dấu hiệu                                                               | Ý nghĩa                                                                                                                                 |
| ---------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| `export default async function Page`                                   | Server Component mặc định trong `app/` — **chỉ có từ v13+**                                                                             |
| Nhận `params` (không phải `query`)                                     | App Router đổi từ `query.id` (Pages Router) sang `params.id` (App Router)                                                               |
| `await` trực tiếp trong component, không qua `useEffect`               | Component-level data fetching — paradigm mới của App Router                                                                             |
| `await fetch(...)` chứ không gọi `getServerSideProps`/`getStaticProps` | Hai function export này **không tồn tại trong `app/`**                                                                                  |
| `{ cache: 'no-store' }` explicit                                       | Web Fetch API chuẩn. Tồn tại cả ở v13/v14 (opt-out) và v15+ (opt-in), nhưng việc **ghi rõ** cho thấy dev hiểu caching — phong cách v15+ |

**Xác định cụ thể hơn:**

- Nếu không có `{ cache: 'no-store' }` → có thể là v13/v14 (default cache)
- Có `{ cache: 'no-store' }` explicit → nghiêng về **v15+** (triết lý explicit is better than implicit)

→ **Kết luận:** **Era 3 (v13+)**, khả năng cao là **v15+** vì cách viết cache rất có ý thức.

**Mental model của era này:** "Component tự lo data của nó, server render một lần, chỉ gửi JS thật sự cần tương tác xuống client."

---

**Code D:** `'use cache'` + `cacheTag` → **Era 4 — New Cache Primitives (v15/v16, 2024–2026)**

```javascript
async function getData(id) {
  'use cache';
  cacheTag(`item-${id}`);
  return db.items.findById(id);
}
```

**Bằng chứng nhận diện:**

| Dấu hiệu                                | Ý nghĩa                                                                               |
| --------------------------------------- | ------------------------------------------------------------------------------------- |
| `'use cache';` directive                | **Đây là dấu hiệu chắc chắn nhất.** Directive này **chỉ tồn tại từ v16** (2025/2026)  |
| `cacheTag(item-${id})`                  | API tagging cho invalidation theo nhóm — đặc trưng của cache primitives mới trong v16 |
| Hàm thuần (pure function) với directive | Pattern mới: cache ở **function/file level**, không qua config ẩn                     |

**Tại sao không phải era khác:**

- `'use cache'` không tồn tại trong v13/v14/v15 — không thể nhầm
- `cacheTag()` từ module `next/cache` với tên `unstable_cacheTag` (alias) — chỉ v16
- Không có `'use server'` → đây không phải Server Action
- Không có `getStaticProps`/`getServerSideProps` → không phải Pages Router
- Có `async function` → không phải Pages Router cũ

**Mental model của era này:** "Cache là quyết định có chủ đích, không phải default ẩn. Tag cache để invalidate chính xác, không phải đoán."

---

**Bảng tổng hợp nhanh**

| Code  | Era                                 | Phiên bản                     | Từ khóa fingerprint                                             |
| ----- | ----------------------------------- | ----------------------------- | --------------------------------------------------------------- |
| **A** | Era 1 — Pages Router Genesis        | v1–v8 (2016–2019)             | `getInitialProps` + `isServer = !!req` + `query.id`             |
| **B** | Era 2 — Pages Router Maturity (ISR) | v9.5+ (T7/2020)               | `getStaticProps` + `revalidate` + `getStaticPaths`              |

**Bài học rút ra:** Khi gặp code Next.js bất kỳ, nhìn 3 thứ là đủ đoán era:

1. **Có `pages/` hay `app/` directory?**

**b. Legacy pattern sống trong modern codebase:**

- `app/layout.tsx` có `getInitialProps()` → đây là **Era 1 pattern** (v1–v8) vẫn còn tồn tại trong Era 4
- Lý do: thư viện `next-translate` (i18n) cần `getInitialProps` để detect locale từ request khi render trên server. Đây là ví dụ thực tế về **dependency lock-in**: framework tiến hóa, nhưng thư viện chưa theo kịp

**c. Áp dụng một số tính năng v16:**

- `proxy.ts` xuất hiện → đây là **rename mới của `middleware.ts` trong Next.js 16** (theo xu hướng làm rõ role: proxy = ở trước, chỉ làm việc routing/rewrite/redirect, không phải "middleware" chạy giữa request và response)
- NHƯNG chưa dùng `'use cache'`, `cacheTag`, `cacheLife`, hay bật PPR
- React vẫn 18.2.0 (chưa lên 19) → adoption có chọn lọc, không rush bleeding edge

**Kết luận**

> **cal.com thuộc Era 4 (v16.2.3) nhưng đang trong giai đoạn "selective adoption"** — vừa có bleeding edge features mới (proxy.ts), vừa giữ legacy pattern (getInitialProps), vừa migration dở dang (pages/ + app/ coexist), vừa chưa adopt new cache primitives. Đây là codebase thực tế của một product lớn, lâu năm, nhiều người contribute — và nó phản ánh chính xác thực tế: **không phải project nào cũng đi đầu mọi paradigm shift**.

**🧠 3 bài học rút ra từ "đi thực địa"**

**1. Era không quyết định bởi version một cách tuyến tính**

cal.com đang ở **v16** (cao nhất) nhưng lại có **nhiều pattern cũ hơn** shadcn-ui/taxonomy (v13.3). Điều này cho thấy:

- **Version cao ≠ codebase hiện đại**
- Một codebase có thể vừa có `getInitialProps` (v1–v8), vừa có `pages/` (v9+), vừa có `app/` async components (v13+), vừa có `proxy.ts` (v16)
- Khi đọc codebase, **không nhìn version, nhìn từng file cụ thể**

**2. Quy trình 5 phút trong bài học áp dụng thực tế 100%**

1. `cat package.json | grep "next"` → có version
2. `ls -la` → có pages/ không, có app/ không
3. `grep getServerSideProps...` → còn dùng pattern cũ không
4. Đọc 1 page đại diện → confirm pattern
5. Đọc `next.config.*` → biết thêm experimental flags

5 phút là đủ. Không cần đọc toàn bộ codebase.

**3. Migration là quá trình, không phải sự kiện**

Cả 3 project đều cho thấy migration Pages → App Router **không bao giờ xảy ra overnight**:

- **taxonomy**: 2023, đang giữa chừng (UI mới, API cũ)
- **commerce**: ~2024, đã xong (chỉ còn app/)
- **cal.com**: 2026, vẫn chưa xong (cả hai coexist, có cả getInitialProps)

Khi join team maintain codebase Next.js, **đừng mong mọi thứ "sạch"**. Một codebase thực tế luôn có lớp lang lịch sử. Việc của bạn là đọc hiểu từng lớp, không phải ép refactor tất cả.

## 13. Summary

```
4 Era của Next.js:

Era 1 — Pages Router Genesis (v1–v8, 2016–2019):
   Zero-config SSR, file-based routing, getInitialProps universal

Era 2 — Pages Router Maturity (v9–v12, 2019–2021):
   Dynamic routing [id].js, API Routes, TypeScript native,
   getServerSideProps/getStaticProps/ISR, SWC compiler, Middleware

Era 3 — App Router Revolution (v13–v14, 2022–2023):
   app/ directory, Server Components, Nested layouts,
   Server Actions, Partial Prerendering

Era 4 — New Cache Primitives (v15–v16, 2024–2026):
   Opt-in caching, 'use cache', cacheTag, updateTag,
   Build Adapters API stable (v16.2), Partial Prefetching (v16.3),
   proxy.ts thay middleware.ts, Turbopack default

2 Paradigm Shifts thực sự:
   v9 (2019): Pages Router hoàn chỉnh, fullstack capability
   v13 (2022): App Router, Server Components, mental model mới hoàn toàn
```

**Takeaway duy nhất cần nhớ:**

Khi gặp Next.js code bất kỳ, câu hỏi đầu tiên: _"Pages Router hay App Router?"_. Câu trả lời đó quyết định toàn bộ cách bạn đọc và hiểu code tiếp theo.
