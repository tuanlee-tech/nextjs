# M0.5: Next.js Ecosystem & Tooling

> 🎯 **Lesson Goal:** Liệt kê 5 tools thường dùng với Next.js, cấu hình ESLint/Prettier/TypeScript, hiểu Turbopack vs Webpack trong context tooling, và chọn đúng runtime (Node.js vs Edge) cho use case đơn giản.

---

## 1. Evolution Context

Cái đau đầu thật sự không phải là "Next.js khó dùng". Là một project React/Next.js hiện đại bắt buộc phải có 5-6 tool nói chuyện được với nhau: TypeScript compile, ESLint lint, Prettier format, PostCSS xử lý CSS, và một bundler đóng gói tất cả lại. Những tool này được viết bởi các team khác nhau, không ai cam kết tương thích 100% với ai, và version của chúng update lệch pha liên tục.

**Cách cũ (React thuần, pre-Next.js, trước 2016):** Bạn tự wiring từng mảnh. Cài `ts-loader` cho Webpack để compile TypeScript, tự thêm `eslint-plugin-react-hooks` vì ESLint mặc định không biết React Hooks là gì, tự setup PostCSS pipeline cho Tailwind/Sass. Không có convention chung — mỗi project tự bịa ra cấu trúc riêng.

**Tại sao thất bại:** "Config hell". Bạn lên Stack Overflow copy 1 đoạn `.babelrc`, copy thêm 1 đoạn `webpack.config.js` từ project khác, rồi 2 cái đó conflict version với nhau. ESLint legacy `.eslintrc` cascading rules đè lên nhau không đoán trước được thứ tự. Prettier và ESLint cùng can thiệp vào dấu nháy đơn/đôi, file save xong là báo lỗi ngược lại.

**Timeline (đã verify thứ tự, KHÔNG suy diễn khoảng cách thời gian chính xác giữa các mốc):**
- Manual Webpack/TS wiring (trước 2016) — React thuần, tự ráp toolchain.
- Next.js zero-config TypeScript & Webpack (2016-2021) — Next.js tự bundle TS compiler, bạn không cần `ts-loader` nữa.
- Next.js tích hợp Tailwind & `eslint-config-next` (2021+) — `create-next-app` generate sẵn ESLint rules đặc thù cho App Router.
- Next.js 15.5 đưa ra deprecation warning cho `next lint`, mở thêm lựa chọn Biome [FACT — Source: Next.js 15.5 Blog, nextjs.org/blog/next-15-5].
- Next.js 16 (hiện tại, bản bạn đang dùng theo Bible) **xoá hẳn** `next lint`, Turbopack trở thành bundler mặc định ổn định cho cả `next dev` và `next build` [FACT — Source: Next.js 16 Blog & Upgrading Guide v16, nextjs.org/docs/app/guides/upgrading/version-16].

## 2. Mental Model

[MODEL] Tooling trong Next.js ecosystem vận hành theo nguyên tắc **"Convention over Configuration" tối đa**. Next.js cố tình KHÔNG bắt bạn khai báo loader cho TypeScript, KHÔNG bắt bạn khai báo PostCSS plugin riêng cho từng file CSS, KHÔNG bắt bạn tự resolve path alias trong bundler. Mọi convention chuẩn (file `tsconfig.json`, `postcss.config.mjs`, `eslint.config.mjs` ở root) đều được framework tự động đọc.

Hệ quả thực dụng cho bạn: **chỉ viết custom config khi có business requirement cụ thể** — ví dụ bạn cần alias `@/legacy` trỏ tới một package monorepo ngoài root, hoặc bạn cần loader riêng cho file `.svg`. Nếu bạn đang viết config "cho chắc" mà không có lý do cụ thể, đó là dấu hiệu bạn đang chống lại convention thay vì dùng nó — và đó chính là cách AP2 (`next.config.ts` phình to) xảy ra trong thực tế.

[ANALOGY] Coi 5 tool tooling này như 5 trạm kiểm soát trên cùng một băng chuyền: TypeScript kiểm tra type trước khi code chạy được, ESLint kiểm tra pattern/bug, Prettier chuẩn hoá hình thức, PostCSS biến class thành CSS thật, bundler gói mọi thứ thành artifact chạy được. Vấn đề không phải là từng trạm — là khi 2 trạm tranh nhau quyền quyết định trên cùng một dòng code (ESLint vs Prettier ở Scenario 1) hoặc một trạm bị đặt sai vị trí (Edge runtime ở chỗ cần Node.js APIs).

## 3. Core Concepts

Theo Topics trong Bible, bài này KHÔNG đào sâu Turbopack internals, performance tuning, hay security config — những phần đó thuộc M9/M10. Ở đây bạn chỉ cần nắm tooling ở mức "biết cấu hình đúng, biết khi nào không nên cấu hình".

Trước khi vào chi tiết từng phần, đây là 5 tool cốt lõi và vai trò của chúng trong mọi Next.js project production:

1. **TypeScript** — compiler kiêm type-checker. Bắt lỗi type ở local trước khi nó crash production.
2. **ESLint** — linter. Bắt code smell và bug pattern (stale closure trong `useEffect`, unused import...) trước khi merge.
3. **Prettier** — formatter. Chuẩn hoá hình thức code (quote, indent, semicolon) để code review không tốn thời gian cãi style.
4. **PostCSS** (chạy qua Tailwind CSS) — CSS processor. Biến utility class thành CSS thật, xử lý vendor-prefix.
5. **Bundler — Turbopack hoặc Webpack** — đóng gói module thành bundle chạy được trên browser/server.

Riêng tool số 5: Next.js 16 dùng **Turbopack làm bundler mặc định ổn định cho cả `next dev` và `next build`**, không cần flag `--turbopack` nữa [FACT — Source: Next.js 16 Blog, nextjs.org/blog/next-16]. Bạn vẫn opt-out về Webpack bằng `next dev --webpack` / `next build --webpack` nếu project có custom webpack config chưa migrate được [FACT — Source: Next.js Turbopack API Reference]. Ở mức tooling (không phải internals), điều bạn cần biết chỉ là: cả hai bundler đọc `tsconfig.json` paths và `postcss.config.mjs` theo đúng cùng convention, nên đổi bundler không kéo theo việc viết lại ESLint hay Tailwind config.

### 3.1 TypeScript & Path Aliases

`tsconfig.json` là nguồn sự thật duy nhất cho path alias. Next.js đọc `compilerOptions.paths` trực tiếp khi resolve module — qua cả Webpack và Turbopack — bạn KHÔNG cần cài `tsconfig-paths` hay khai báo lại alias trong `next.config.ts` [FACT — Source: Next.js Config Docs, nextjs.org/docs/app/api-reference/config/typescript].

`next.config.ts` (TypeScript config file) đã được Next.js hỗ trợ native từ v12, và v16 tiếp tục khuyến nghị dùng `.ts` thay vì `.js` để có type safety ngay trong IDE khi bạn gõ sai key config [FACT — Source: Next.js Config Docs].

```json [tsconfig.json]
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["dom", "dom.iterable", "esnext"],
    "module": "esnext",
    "moduleResolution": "bundler",
    "baseUrl": ".",
    // Path alias là nguồn sự thật DUY NHẤT — Next.js tự đọc paths này,
    // không cần khai báo lại ở next.config.ts hay cài tsconfig-paths.
    "paths": {
      "@/components/*": ["./src/components/*"],
      "@/lib/*": ["./src/lib/*"]
    },
    "strict": true,
    // noUncheckedIndexedAccess bắt lỗi access array/object key có thể undefined —
    // tắt nó là tự tay tạo runtime crash mà type-checker lẽ ra đã bắt được.
    "noUncheckedIndexedAccess": true,
    "jsx": "preserve",
    "incremental": true,
    "plugins": [{ "name": "next" }]
  },
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx", ".next/types/**/*.ts"],
  "exclude": ["node_modules"]
}
```

```ts [next.config.ts]
import type { NextConfig } from 'next'

// next.config.ts KHÔNG cần khai báo lại path alias — Next.js đọc trực tiếp
// compilerOptions.paths trong tsconfig.json, kể cả khi build bằng Turbopack.
// Source: nextjs.org/docs/app/api-reference/config/typescript
const nextConfig: NextConfig = {
  typescript: {
    // Không tắt type-check khi build trừ khi có lý do CỤ THỂ (VD: tách bước
    // type-check ra một job CI riêng để build song song). Tắt mặc định
    // = nợ kỹ thuật âm thầm tích lũy, sớm muộn cũng nổ ở production.
    ignoreBuildErrors: false,
  },
}

export default nextConfig
```

### 3.2 ESLint, Prettier & `eslint-config-next`

`next lint` từng là wrapper quanh ESLint CLI. Next.js 15.5 đưa ra deprecation warning, và Next.js 16 (bản bạn đang dùng) **xoá hẳn lệnh này** — không phải "không khuyến khích", mà là không còn tồn tại. Tuỳ chọn `eslint` trong `next.config.ts` cũng bị xoá theo, vì nó chỉ tồn tại để cấu hình cho wrapper đã biến mất [FACT — Source: Next.js Upgrading Guide v16, nextjs.org/docs/app/guides/upgrading/version-16].

Thay vào đó, bạn chạy ESLint CLI trực tiếp với flat config (`eslint.config.mjs`), import rules từ `eslint-config-next`. Một điểm dễ nhầm: gói này export theo từng preset riêng — `eslint-config-next/core-web-vitals` và `eslint-config-next/typescript` — KHÔNG có export tên `/flat` như một số tutorial/AI-generated snippet hay dùng. Nếu bạn thấy `import x from 'eslint-config-next/flat'` trong codebase, đó là code lỗi thời hoặc bị hallucinate, sửa lại thành `/core-web-vitals` [FACT — Source: Next.js ESLint Config Docs, nextjs.org/docs/app/api-reference/config/eslint].

`eslint-config-prettier` là package KHÔNG nằm trong danh sách Terminology của Bible, nhưng bắt buộc phải biết: nó tắt toàn bộ formatting rule của ESLint đang xung đột với Prettier (xem Scenario 1). Phải đặt **cuối mảng config** vì entry sau override entry trước.

```js [eslint.config.mjs]
import { defineConfig, globalIgnores } from 'eslint/config'
import nextVitals from 'eslint-config-next/core-web-vitals'
import nextTs from 'eslint-config-next/typescript'
import prettier from 'eslint-config-prettier/flat'

// Thứ tự trong array LÀ thứ tự override — entry càng về sau quyền càng cao.
// `prettier` phải nằm CUỐI vì nhiệm vụ của nó là tắt mọi rule formatting
// (quote, semi, indent) mà các config trước đã bật, tránh ESLint và Prettier
// "đánh nhau" trên cùng một dòng code mỗi lần save.
// Source: nextjs.org/docs/app/api-reference/config/eslint
export default defineConfig([
  ...nextVitals,
  ...nextTs,
  prettier,
  globalIgnores([
    '.next/**',
    'out/**',
    'build/**',
    'next-env.d.ts',
  ]),
])
```

```json [package.json]
{
  "scripts": {
    "lint": "eslint . --max-warnings 0",
    "lint:fix": "eslint . --fix"
  }
}
```

Lưu ý ngoài lề: từ 15.5, `create-next-app` cho chọn **Biome** thay ESLint (nhanh hơn, ít rule hơn, tự kèm formatter nên không cần `eslint-config-prettier`) [FACT — Source: Next.js 15.5 Blog]. Bible yêu cầu track ESLint + Prettier vì đây vẫn là combo phổ biến nhất trong codebase production hiện tại, nên bài này không đi sâu Biome.

### 3.3 Tailwind CSS & Runtime Selection

**Tailwind CSS:** Setup hiện tại (Tailwind v4, default khi chạy `create-next-app --tailwind`) chỉ cần 2 package: `tailwindcss` và `@tailwindcss/postcss`. Không cần `autoprefixer` riêng, không cần `tailwind.config.js` — Tailwind v4 dùng engine CSS-first (`@theme` ngay trong CSS) và tự detect content path, không cần khai báo `content: [...]` thủ công như v3 [FACT — Source: Next.js Getting Started CSS Docs, nextjs.org/docs/app/getting-started/css; Tailwind CSS v4.0 Blog, tailwindcss.com/blog/tailwindcss-v4]. Next.js không cần custom Webpack/Turbopack config gì thêm cho việc này.

```js [postcss.config.mjs]
// Tailwind v4: chỉ 1 plugin duy nhất. Vendor-prefix đã được xử lý nội bộ
// bởi engine của Tailwind — không cần autoprefixer tách riêng như v3.
export default {
  plugins: {
    '@tailwindcss/postcss': {},
  },
}
```

```css [app/globals.css]
/* Thay cho 3 dòng @tailwind base/components/utilities của v3 */
@import "tailwindcss";
```

**Runtime selection (Node.js vs Edge):** Edge Runtime chạy trên V8 isolate — KHÔNG có Node.js APIs (`fs`, `path`, `child_process`), chỉ có subset Web APIs (`fetch`, `Request`, `Response`) [FACT — Source: Next.js Edge Runtime Docs, nextjs.org/docs/app/api-reference/edge]. Node.js runtime là default khi bạn không khai báo gì.

> ⚠️ **Cập nhật thực tế production (2026), không có trong raw material gốc:** Lý do chọn Edge cho "auth check, redirect đơn giản" từng dựa trên việc Edge cold-start nhanh hơn Node.js rất nhiều. Hiện tại Vercel đã đưa **Fluid Compute** làm execution model mặc định cho mọi project mới, giúp Node.js gần như loại bỏ cold start nhờ pre-warm instance và bytecode cache. Vercel hiện khuyến nghị **migrate từ Edge về Node.js** cho phần lớn route handler vì performance/reliability tốt hơn, và Edge Functions độc lập (route-level) đã ở trạng thái deprecated trên platform này — ngoại lệ duy nhất là Routing Middleware, vẫn default Edge và "expected and supported" [FACT — Source: Vercel Edge Runtime Docs & Edge Functions (Deprecated) Docs, vercel.com/docs/functions/runtimes/edge, cập nhật tháng 6/2026]. Đây là platform-level guidance của Vercel, không phải Next.js framework xoá feature — `export const runtime = 'edge'` vẫn là syntax hợp lệ trong Next.js 16. Nhưng nếu bạn đang quyết định runtime cho route mới mà không có constraint cụ thể (geo-routing, region-pinned), mặc định Node.js là lựa chọn an toàn hơn bây giờ so với 2-3 năm trước.

```ts [src/app/api/health/route.ts]
import { NextResponse } from 'next/server'

// Health check không đụng fs/path/native binding → an toàn cho Edge.
// Đây là use case Edge vẫn hợp lý: latency thấp, logic cực mỏng.
export const runtime = 'edge' // mặc định là 'nodejs' nếu bỏ dòng này

export async function GET() {
  try {
    return NextResponse.json({ status: 'ok', runtime: 'edge' }, { status: 200 })
  } catch (error) {
    // Structured logging — KHÔNG console.log thô, để log parser (Vercel/Datadog)
    // đọc được field thay vì phải regex chuỗi text.
    console.error(JSON.stringify({ level: 'error', msg: 'health-check-failed', error: String(error) }))
    return NextResponse.json({ status: 'error' }, { status: 500 })
  }
}
```

```ts [src/app/api/reports/route.ts]
import { NextResponse } from 'next/server'
import { writeFile } from 'node:fs/promises'

// Cần fs để ghi report tạm ra /tmp trước khi upload lên object storage
// → BẮT BUỘC Node.js runtime. Set 'edge' ở đây build vẫn pass (xem Scenario 2)
// nhưng request sẽ crash 500 ngay khi chạm dòng writeFile.
export const runtime = 'nodejs' // default, viết rõ ra để intent không bị hiểu lầm khi review

export async function POST(request: Request) {
  try {
    const body = await request.json()
    await writeFile(`/tmp/report-${Date.now()}.json`, JSON.stringify(body))
    return NextResponse.json({ status: 'queued' }, { status: 202 })
  } catch (error) {
    console.error(JSON.stringify({ level: 'error', msg: 'report-write-failed', error: String(error) }))
    return NextResponse.json({ status: 'error' }, { status: 500 })
  }
}
```

## 4. Anti-Patterns in the Wild

- ❌ **Code Smell:** "Dùng next lint"
  ```json [package.json]
  {
    "scripts": {
      "lint": "next lint"
    }
  }
  ```
- ✅ **Staff Refactor:** ESLint trực tiếp với flat config
  ```json [package.json]
  {
    "scripts": {
      "lint": "eslint . --max-warnings 0",
      "lint:fix": "eslint . --fix"
    }
  }
  ```
  (kèm `eslint.config.mjs` import `eslint-config-next/core-web-vitals` — xem mục 3.2)
- 🧠 **Why:** Mất 1 lớp abstraction mỏng (wrapper `next lint`), đổi lại quyền control trực tiếp lên ESLint CLI — không bị block khi Next.js xoá hẳn binary, và bạn thấy rõ rule nào đang chạy thay vì tin tưởng "magic" của wrapper.
- 🔥 **Incident Risk:** CI fail cứng (`command not found`) ngay khi upgrade lên Next.js 16+, vì lệnh `next lint` không còn tồn tại trong binary — không phải warning, là lỗi chặn build.

- ❌ **Code Smell:** "Config quá phức tạp trong next.config.ts"
  ```ts [next.config.ts]
  import type { NextConfig } from 'next'

  const nextConfig: NextConfig = {
    async redirects() {
      return [ /* 15 redirect rule nhồi thẳng vào đây */ ]
    },
    async headers() {
      return [ /* 20 header rule khác, security + CORS trộn lẫn */ ]
    },
    webpack: (config) => {
      // 80 dòng custom webpack logic không ai nhớ tại sao cần
      return config
    },
    env: { /* 30 biến env parse tay, không validate */ },
    // ...200+ dòng tổng cộng, ai sửa cũng sợ đụng phần người khác
  }

  export default nextConfig
  ```
- ✅ **Staff Refactor:** Tách logic ra các file riêng
  ```ts [next.config.ts]
  import type { NextConfig } from 'next'
  import { redirects } from './config/redirects'
  import { headers } from './config/headers'

  // Mỗi concern là 1 module riêng — next.config.ts chỉ còn việc lắp ráp.
  const nextConfig: NextConfig = {
    redirects,
    headers,
  }

  export default nextConfig
  ```
  ```ts [config/redirects.ts]
  import type { NextConfig } from 'next'

  // Tách riêng để: (1) import trực tiếp và unit test bằng Vitest mà không cần
  // spin up cả Next.js build pipeline, (2) giảm khả năng merge conflict —
  // conflict chỉ xảy ra khi 2 branch cùng sửa CÙNG MỘT file nhỏ này.
  export const redirects: NonNullable<NextConfig['redirects']> = async () => {
    return [
      { source: '/old-blog/:slug', destination: '/blog/:slug', permanent: true },
    ]
  }
  ```
- 🧠 **Why:** Tăng số lượng file đổi lại tính dễ đọc, dễ unit test độc lập, giảm merge conflict — nhưng trade-off là bạn phải điều hướng nhiều file hơn khi debug, không phải lúc nào tách file cũng đáng (project nhỏ 1-2 redirect không cần làm vậy).
- 🔥 **Incident Risk:** Git merge conflict kéo dài ở `next.config.ts` khi 2 feature branch cùng sửa config trong cùng 1 file 200+ dòng — review PR vì vậy cũng chậm vì reviewer phải đọc lại toàn bộ diff thay vì chỉ phần liên quan.

- ❌ **Code Smell:** "Edge runtime cho mọi thứ"
  ```ts [src/app/api/checkout/route.ts]
  export const runtime = 'edge' // "nghe nói edge nhanh hơn"

  export async function POST(request: Request) {
    const bcrypt = await import('bcrypt') // native binding — KHÔNG chạy được trên Edge
    // ...
  }
  ```
- ✅ **Staff Refactor:** Chọn runtime đúng use case, mặc định Node.js
  ```ts [src/app/api/checkout/route.ts]
  import { NextResponse } from 'next/server'
  import bcrypt from 'bcrypt'

  // Node.js là default — checkout cần bcrypt (native binding) và có thể cần
  // DB pool nặng, đây chính xác là use case Edge KHÔNG hỗ trợ.
  // Khai báo rõ runtime để người review không phải đoán intent.
  export const runtime = 'nodejs'

  export async function POST(request: Request) {
    try {
      const { password } = await request.json()
      const hashed = await bcrypt.hash(password, 10)
      return NextResponse.json({ status: 'ok' })
    } catch (error) {
      console.error(JSON.stringify({ level: 'error', msg: 'checkout-failed', error: String(error) }))
      return NextResponse.json({ status: 'error' }, { status: 500 })
    }
  }
  ```
- 🧠 **Why:** Edge cho cold start nhanh và chạy gần user theo region, đổi lại mất toàn bộ Node.js ecosystem API. Với Fluid Compute hiện tại (xem mục 3.3), lợi thế cold-start của Edge với Node.js đã hẹp lại nhiều so với vài năm trước, nên với route không có constraint geo/region rõ ràng, Node.js mặc định thường là lựa chọn an toàn hơn — không phải lúc nào "nhanh hơn trên giấy" cũng đáng đánh đổi.
- 🔥 **Incident Risk:** Production crash 500 ngay lập tức khi gọi `fs.readFileSync` hoặc import thư viện có native binding (`bcrypt`, `sharp`...) trong route được set `runtime = 'edge'` — local có thể chạy được vì dev server fallback hành vi khác, nhưng deploy lên thì fail.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Node.js Runtime (default)** | Full Node.js API, mọi npm package chạy được, ổn định; trên Vercel còn chạy kèm Fluid Compute giúp giảm mạnh cold start | Vẫn nặng hơn Edge cho workload siêu nhẹ nếu hosting không có cơ chế warm-instance tương đương | Mọi route cần DB driver, native binding (`bcrypt`, `sharp`), file processing, hoặc khi không có constraint region rõ ràng | Khi bạn cần logic chạy ở edge node trước khi request chạm origin (Routing Middleware) | Tính theo Active CPU time trên Vercel — chỉ trả phí lúc CPU thực sự xử lý, không tính lúc đợi I/O |
| **Edge Runtime** | Cold start cực nhanh, chạy gần user theo region | Không có Node built-in (`fs`, `child_process`), giới hạn bundle size chặt; nhiều nền tảng (gồm Vercel) đang khuyến nghị Node.js + Fluid Compute thay thế cho hầu hết route handler | Routing Middleware (vẫn default Edge), redirect/geo-check siêu đơn giản không cần thư viện nặng | Khi dùng `fs`, native binding, ORM client nặng, hoặc khi hosting đã có cơ chế warm-instance khiến lợi thế cold-start không còn rõ | Phải audit bundle size cho từng dependency mới thêm, dễ vượt limit mà không biết trước khi deploy |

## 6. Legacy Recognition

Nhận diện code legacy khi review hoặc audit codebase:

- `"lint": "next lint"` trong `package.json` — **đã bị xoá hoàn toàn ở Next.js 16**, không còn chạy được, không chỉ là deprecated.
- `.eslintrc` / `.eslintrc.json` — legacy ESLint config, ESLint 10 sẽ bỏ hẳn hỗ trợ format này.
- `next.config.js` (không phải `.ts`) — vẫn chạy được, nhưng mất type safety khi gõ sai key.
- `tailwind.config.js` đi kèm `content: [...]` thủ công + `autoprefixer` riêng — dấu hiệu project chưa migrate Tailwind v3 → v4.
- `tsconfig-paths` trong dependencies của một project Next.js — dư thừa, Next.js đọc `paths` natively từ lâu.
- `import x from 'eslint-config-next/flat'` — export này **không tồn tại** trong package hiện tại (chỉ có `/core-web-vitals` và `/typescript`); nếu thấy trong code review, đây là snippet lỗi thời hoặc do AI generate sai, cần sửa lại.

## 7. Modern Approach

Migration path từ legacy sang Next.js 15+/16+:

- **`next lint` → ESLint trực tiếp:** Xoá script cũ trong `package.json`. Cài `eslint` + `eslint-config-next`, tạo `eslint.config.mjs`, import `eslint-config-next/core-web-vitals` (và `/typescript` nếu dùng TS). Test bằng `eslint .`.
- **`.eslintrc` → Flat Config:** Dùng tool chính thức `@eslint/eslintrc` (qua `FlatCompat`) để convert tự động cho plugin cũ chưa có bản flat-config native. Xoá file `.eslintrc` sau khi convert xong. Thêm `eslint-config-prettier/flat` vào cuối mảng config.
- **`next.config.js` → `next.config.ts`:** Rename file, thêm `import type { NextConfig } from 'next'`, type-cast object config: `const nextConfig: NextConfig = {...}`. Xoá luôn key `eslint` trong config object nếu có — key này đã bị remove ở v16 vì không còn liên quan đến wrapper nào.
- **Tailwind v3 → v4:** Chạy `npx @tailwindcss/upgrade` (codemod chính thức), review lại phần `@theme` được generate ra từ `tailwind.config.js` cũ, xoá `autoprefixer` khỏi `postcss.config.mjs`.
- **Rollback plan:** Giữ lại file config cũ trong repo (comment out hoặc git stash riêng 1 commit dễ revert), KHÔNG xoá ngay. Nếu migration block deployment gấp, dùng `--no-verify` tạm thời để qua pre-commit hook, nhưng phải fix triệt để trong 24h — không để `--no-verify` trở thành convention ngầm của team.

## 8. Production Scenario

> 🔴 **Scenario 1: "ESLint conflict với Prettier"**
> - Symptom: File bị Prettier format lại lúc save, nhưng ESLint lại báo lỗi formatting (VD: rule `quotes` báo sai dù Prettier vừa fix xong).
> - Root cause: Config ESLint đang có formatting rule active, xung đột trực tiếp với Prettier — 2 tool cùng muốn quyết định style trên 1 dòng code.
> - Detect: Check error code ESLint trả về (VD: `@typescript-eslint/quotes`). Fix: thêm `eslint-config-prettier/flat` vào **cuối** mảng `defineConfig([...])`.

> 🔴 **Scenario 2: "Edge runtime lỗi với Node.js API"**
> - Symptom: Local chạy bình thường, deploy lên Vercel thì crash 500.
> - Root cause: Code dùng `fs` hoặc module cần native binding trong file đã set `runtime = 'edge'` — dev server local có thể không strict bằng môi trường production khi check API availability.
> - Detect: Xem function logs trên Vercel, lỗi điển hình là `TypeError: xxx is not a function` hoặc `Module not found: fs`. Fix nhanh nhất hiện tại: nếu route không có constraint region cụ thể, bỏ `runtime = 'edge'` luôn, chuyển về Node.js — vừa fix bug vừa khớp khuyến nghị hiện tại của Vercel về Fluid Compute (xem mục 3.3).

> 🔴 **Scenario 3: Path alias không hoạt động ở ESLint**
> - Symptom: Import `@/components/Foo` báo lỗi "Unable to resolve path" từ ESLint, dù `tsc`/Next.js build vẫn pass bình thường.
> - Root cause: ESLint không tự đọc `tsconfig.json` paths — đây là behavior khác hẳn so với bundler (Webpack/Turbopack), dễ gây hiểu lầm là "alias bị lỗi" trong khi chỉ là ESLint resolver chưa biết alias tồn tại.
> - Detect: Cài `eslint-import-resolver-typescript`, khai báo resolver này trong `eslint.config.mjs` ở phần settings của plugin import.

## 9. Performance & Overhead

- **Turbopack vs Webpack:** Theo công bố chính thức của Next.js 16, Turbopack cho Fast Refresh nhanh hơn tới 5-10x và build nhanh hơn 2-5x so với Webpack [FACT — Source: Next.js 16 Blog]. Đây là số liệu benchmark công bố, không phải cam kết cho mọi codebase — muốn đo trên project của bạn: xoá `.next/` rồi chạy `time next build` trước/sau khi migrate, so sánh wall-clock time để có số liệu thật cho team mình.
- **Edge vs Node.js cold start (Vercel-specific):** Trước Fluid Compute, Edge cold start nhanh hơn rõ rệt vì không cần boot container. Với Fluid Compute hiện là default cho project mới, Vercel claim loại bỏ cold start cho phần lớn traffic sản xuất ổn định nhờ giữ sẵn instance warm và bytecode cache [FACT — Source: Vercel Fluid Compute Docs]. Số liệu này là đặc thù platform Vercel, KHÔNG generalize cho self-hosted Node.js server hoặc nền tảng khác.
- **Edge bundle size limit:** Lịch sử Vercel giới hạn Edge Function độc lập ở mức khoảng 1MB (Hobby) / 2MB (Pro) / 4MB (Enterprise) — vượt limit sẽ fail deploy với lỗi rõ ràng (VD: `Edge Function size is X MB and your plan size limit is Y MB`) [FACT — Source: Vercel Edge Functions Docs, đã verify thay placeholder NEEDS VERIFICATION trong raw material gốc]. Sản phẩm Edge Function độc lập hiện ở trạng thái deprecated trên Vercel; Node.js runtime qua Fluid Compute có limit uncompressed cao hơn nhiều (tới 250MB).
- **Edge Runtime constraint khác cần biết:** tối đa 950 fetch request mỗi invocation, tối đa 6 connection mở đồng thời, phải bắt đầu gửi response trong 25 giây [FACT — Source: Vercel Edge Functions Limits Docs].
- **Tailwind v4 build time:** Theo Tailwind Labs, full build nhanh hơn tới 5x và incremental build nhanh hơn hơn 100x so với v3 nhờ engine mới [FACT — Source: Tailwind CSS v4.0 Blog]. Muốn benchmark trên project bạn: so sánh `time npm run build` trước/sau migrate, đo riêng bước CSS bằng `--profile` nếu cần breakdown sâu hơn.
- **ESLint Flat Config:** Không có số liệu chính thức so trực tiếp với legacy cascading config. Cách benchmark: chạy `TIMING=1 eslint .` để xem rule nào tốn thời gian nhất, so sánh trước/sau khi migrate hoặc khi thêm plugin mới.

## 10. Architecture Notes

- **Tổ chức `next.config.ts`:** Tách pure function ra module riêng (`redirects.ts`, `headers.ts`), import và gán vào object config chính. Mục tiêu: unit test độc lập bằng Vitest/Jest mà không cần spin up Next.js build pipeline, và giảm merge conflict khi nhiều branch cùng sửa config.
- **Runtime selection strategy:** Node.js là default cho mọi route trừ khi có lý do cụ thể (region-pinning, edge-only API). Với Fluid Compute hiện là execution model mặc định trên Vercel cho project mới, lợi thế "Edge luôn nhanh hơn" không còn chắc đúng như trước — quyết định runtime nên dựa vào: route có dùng Node.js API/native binding không (nếu có → Node.js, hết tranh luận), và route có cần chạy sát edge node vì lý do region/latency cụ thể không (nếu không → giữ default Node.js).
- **ESLint strict rules (`core-web-vitals`):** Bật `--max-warnings 0` ngay từ ngày 1 cho project mới để chặn nợ kỹ thuật từ linter tích lũy. Chỉ hạ xuống mức warning trên CI khi đang inherit một legacy codebase quá lớn để fix toàn bộ ngay.
- **Tooling integration order:** TypeScript → ESLint Flat Config + Prettier → Tailwind PostCSS → Runtime decision. Thứ tự này phản ánh dependency thực tế (ESLint cần biết TS path alias để resolver hoạt động đúng; Tailwind không phụ thuộc 2 bước trước nhưng nên setup sau để tránh phải sửa lại cấu hình ESLint cho file `.css`). Không đảo ngược thứ tự nếu bạn đang setup project từ đầu.

## 13. Summary

- `next lint` không còn tồn tại từ Next.js 16 — dùng ESLint CLI trực tiếp với flat config (`eslint.config.mjs`) ngay từ ngày đầu project, import `eslint-config-next/core-web-vitals` (không phải `/flat`).
- `next.config.ts` phải gọn — tách logic phức tạp ra module riêng (`redirects.ts`, `headers.ts`) để tránh merge conflict hell và để unit test được độc lập.
- Edge Runtime không phải "silver bullet": chỉ phù hợp cho handler siêu nhẹ, region-specific. Với Fluid Compute hiện là default trên Vercel, Node.js đã thu hẹp khoảng cách cold-start, nên mặc định Node.js là lựa chọn an toàn hơn cho route không có constraint region rõ ràng.
- `eslint-config-prettier/flat` là bắt buộc nếu dùng cả ESLint và Prettier, không phải optional — và phải đặt cuối mảng config.
- Tailwind v4 (`@tailwindcss/postcss`) là convention hiện tại, không cần `tailwind.config.js` hay `autoprefixer` riêng như v3.

Lịch sử: Runtime dựa trên V8 isolate, không có Node.js built-in (`fs`, `path`, `child_process`). Cold start nhanh, auto-scale global theo region. Diễn biến mới nhất (2026): hosting platform lớn nhất của Next.js (Vercel) đã đưa Fluid Compute làm default cho Node.js runtime, khiến lợi thế cold-start truyền thống của Edge không còn rõ rệt như trước cho phần lớn use case ngoài Middleware.
Codebase legacy: Không có legacy code tương đương trực tiếp. Junior thường nhầm "Edge = Node.js nhanh hơn" và set `runtime = 'edge'` cho mọi route mà không kiểm tra dependency có dùng native binding hay không.
