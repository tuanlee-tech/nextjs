# M0.4: File Conventions & Routing Mental Model

> 🎯 **Lesson Goal:** Giải thích App Router file conventions, so sánh với Pages Router, vẽ routing tree từ folder structure, và hiểu tại sao `page.tsx` không phải `index.tsx`.

---

## 1. Evolution Context

Routing trong web framework về cơ bản phải trả lời 1 câu hỏi: **URL này → render component nào?** Nghe đơn giản. Nhưng khi app có 50–100 routes với layouts lồng nhau, câu trả lời đó bắt đầu thối ra.

### Cách cũ: Config file và `_app.tsx` wrapper

Có 2 trường phái giải quyết:

**Trường phái 1 — Config-based routing (React Router thuần):** Viết 1 file object định nghĩa tất cả routes.

```ts [src/routes.tsx]
// Config routing tập trung — từng hợp lý, sau 1 năm thành thế này
const routes = [
  {
    path: '/',
    element: <RootLayout />,
    children: [
      {
        path: 'dashboard',
        element: <DashboardLayout />,
        children: [
          { path: 'analytics', element: <AnalyticsPage /> },
          {
            path: 'settings',
            element: <SettingsLayout />,       // Layout lồng tiếp
            children: [
              { path: 'billing', element: <BillingPage /> }
            ]
          }
        ]
      }
      // ... 80 routes nữa, 1 file, merge conflict mỗi sprint
    ]
  }
];
```

**Trường phái 2 — Pages Router (`_app.tsx` wrapper):** Next.js Pages Router dùng folder structure cho routing, nhưng mọi layout logic phải chui vào `_app.tsx` — global wrapper duy nhất.

```tsx [pages/_app.tsx]
// pages/_app.tsx — Pages Router: 1 file gánh toàn bộ layout logic
// Đây là codebase thực tế sau 1 năm phát triển không kiểm soát
export default function MyApp({ Component, pageProps }) {
  const router = useRouter();

  // Mỗi route mới = thêm 1 conditional ở đây
  // Không có cách tự nhiên nào khác để chia sẻ layout theo segment
  const isAdmin = router.pathname.startsWith('/admin');
  const isDashboard = router.pathname.startsWith('/dashboard');
  const isMarketing = router.pathname === '/' || router.pathname === '/about';

  return (
    <SessionProvider>
      <ThemeProvider>
        {isAdmin ? (
          <AdminShell>
            {isDashboard ? (
              <DashboardShell>           // Wrapper lồng wrapper
                <Component {...pageProps} />
              </DashboardShell>
            ) : (
              <Component {...pageProps} />
            )}
          </AdminShell>
        ) : isMarketing ? (
          <MarketingShell>
            <Component {...pageProps} />
          </MarketingShell>
        ) : (
          <Component {...pageProps} />
        )}
      </ThemeProvider>
    </SessionProvider>
  );
  // Đây là Wrapper Hell — và chỉ ngày càng tệ hơn
}
```

### Tại sao cả 2 cách đều thất bại

3 failure mode được ghi nhận ở mọi codebase lớn:

**1. Merge conflicts không dứt:** 1 file config routing trung tâm = bottleneck cho toàn team. 2 dev thêm route cùng lúc → conflict mỗi sprint.

**2. Wrapper Hell không lối thoát:** Nested layouts trong Pages Router chỉ làm được bằng conditional rendering trong `_app.tsx`. Không scale. Không trace được.

**3. Không colocate được:** Loading state của `/dashboard` nằm trong `_app.tsx`, không nằm cạnh component dashboard. Mở codebase lúc 2 giờ sáng on-call, không biết trace từ đâu.

### App Router v13+: Convention-over-Configuration

Next.js giải quyết bằng cách loại bỏ hoàn toàn config file routing. Thay vào đó: **folder structure = routing**, **tên file = behavior**. Không cần đọc docs mỗi lần thêm route, không merge conflict vào config trung tâm, layout colocate ngay cạnh page.

## 2. Mental Model

[MODEL] **URL tree = Folder tree.** Mỗi folder trong `app/` là một node trên URL tree. Không có folder = không có URL segment.

[ANALOGY] Convention là **biển tên đường cố định** — đường nào tên gì, mọi người biết không cần giải thích. Config-based routing giống như mỗi lần đi phải **tự vẽ bản đồ**. Convention-over-configuration trao cho team một bản đồ dùng chung, ai mở codebase cũng đọc được ngay.

[ANALOGY] **Segment** là một "nhánh cây" trên URL tree. `/dashboard/settings` có 2 segments — mỗi cái tương ứng với 1 folder trong `app/`.

```
URL:    /  dashboard  /  settings  /  billing
                │              │             │
           segment 1      segment 2     segment 3
                │              │             │
         app/dashboard/  settings/      billing/
```

[MODEL] **Tên file = Behavior.** Đây là contract không thể thương lượng. Next.js chỉ nhận ra đúng tên file đã quy định — bất kỳ tên nào khác bị bỏ qua hoàn toàn, không error, không warning.

```
app/dashboard/
├── page.tsx          ← ✅ Next.js nhận ra → tạo route /dashboard
├── layout.tsx        ← ✅ Next.js nhận ra → shared UI cho /dashboard/*
├── loading.tsx       ← ✅ Next.js nhận ra → Suspense boundary
├── error.tsx         ← ✅ Next.js nhận ra → Error boundary
├── not-found.tsx     ← ✅ Next.js nhận ra → 404 handler trong segment
├── template.tsx      ← ✅ Next.js nhận ra → re-mounting UI
│
├── index.tsx         ← ❌ Bị bỏ qua hoàn toàn (bẫy cổ điển từ Pages Router)
├── Dashboard.tsx     ← ❌ Bị bỏ qua hoàn toàn
└── home.tsx          ← ❌ Bị bỏ qua hoàn toàn
```

## 3. Core Concepts

### 3.1 Special Files & Segment Behavior

[FACT] Theo [Next.js Docs — Defining Routes](https://nextjs.org/docs/app/building-your-application/routing#defining-routes): Next.js **chỉ tạo route publicly accessible** nếu folder đó chứa `page.tsx`. Đặt `index.tsx` sẽ bị bỏ qua — folder trở thành dead route, URL trả về 404.

Dưới đây là implementation đầy đủ cho một dashboard segment trên production:

```tsx [app/dashboard/layout.tsx]
import type { ReactNode } from 'react';
// Import từ _components/ — private folder, không tạo route
import { DashboardNav } from './_components/DashboardNav';
import { DashboardSidebar } from './_components/DashboardSidebar';

interface DashboardLayoutProps {
  children: ReactNode; // BẮT BUỘC — Next.js inject page.tsx content vào đây
}

// layout.tsx này bọc TOÀN BỘ /dashboard/* routes:
// /dashboard, /dashboard/analytics, /dashboard/settings, /dashboard/settings/billing
//
// [FACT] Nguồn: Next.js Docs — Layouts and Templates:
// layout.tsx KHÔNG re-mount khi navigate giữa các children trong cùng segment
// → sidebar state, scroll position, mọi React state đều được preserve tự động
//
// Hệ quả trực tiếp: layout này phải LIGHTWEIGHT
// Bất kỳ blocking operation nào ở đây = block toàn bộ /dashboard/* subtree
export default function DashboardLayout({ children }: DashboardLayoutProps) {
  return (
    <div className="dashboard-shell">
      <DashboardNav />
      <div className="dashboard-body">
        <DashboardSidebar />
        <main className="dashboard-main">{children}</main>
      </div>
    </div>
  );
}
```

```tsx [app/dashboard/page.tsx]
// page.tsx: điểm duy nhất tạo public route /dashboard
// File này tồn tại = URL /dashboard accessible
// File này không tồn tại = URL /dashboard → 404, dù folder và mọi file khác hoàn toàn hợp lệ
export default function DashboardPage() {
  return (
    <section>
      <h1>Dashboard</h1>
      {/* Data fetching trong page.tsx thuộc M2 — không đào sâu ở bài này */}
    </section>
  );
}
```

```tsx [app/dashboard/loading.tsx]
// [FACT] Nguồn: Next.js Docs — Loading UI and Streaming:
// loading.tsx tự động tạo React <Suspense> boundary cho segment
// → khi page.tsx (Server Component) đang resolve, Next.js stream loading.tsx ra browser
//   ngay lập tức thay vì để user nhìn blank screen
//
// Lưu ý: loading.tsx giảm PERCEIVED latency (blank screen time), không giảm actual fetch time
export default function DashboardLoading() {
  return (
    <div role="status" aria-label="Đang tải Dashboard..." className="loading-skeleton">
      <div className="skeleton-header" aria-hidden="true" />
      <div className="skeleton-content" aria-hidden="true" />
    </div>
  );
}
```

```tsx [app/dashboard/error.tsx]
'use client';
// BẮT BUỘC "use client" — error.tsx phải là Client Component
// Lý do: React Error Boundaries là client-side API,
// không thể chạy trên server (không có DOM, không có event handling)
//
// Chi tiết error handling strategy thuộc M1.3 — bài này chỉ nhận diện file này tồn tại

import { useEffect } from 'react';

interface DashboardErrorProps {
  error: Error & { digest?: string }; // digest: server-side error ID để correlate với server logs
  reset: () => void;                  // Next.js inject: function để retry render segment
}

export default function DashboardError({ error, reset }: DashboardErrorProps) {
  useEffect(() => {
    // KHÔNG swallow error — luôn log hoặc report lên monitoring service
    console.error('[DashboardError]', {
      message: error.message,
      digest: error.digest, // dùng để trace trong server logs
    });
  }, [error]);

  return (
    <div role="alert" aria-live="assertive">
      <h2>Dashboard gặp lỗi</h2>
      <button onClick={reset} type="button">
        Thử lại
      </button>
    </div>
  );
}
```

```tsx [app/dashboard/not-found.tsx]
// Segment-level not-found.tsx: giữ nguyên DashboardLayout khi 404 trong /dashboard/*
//
// Khi component trong segment gọi hàm notFound() → file này được render
// → user vẫn thấy sidebar, nav của dashboard (context-aware 404)
//
// Nếu không có file này: Next.js bubble lên root app/not-found.tsx
// → user bị ném ra page 404 không có layout dashboard
import Link from 'next/link';

export default function DashboardNotFound() {
  return (
    <div className="not-found-container">
      <h2>Trang không tồn tại trong Dashboard</h2>
      <p>Link này đã bị xóa hoặc chưa được tạo.</p>
      <Link href="/dashboard">← Quay lại Dashboard</Link>
    </div>
  );
}
```

```tsx [app/dashboard/template.tsx]
import type { ReactNode } from 'react';

// [FACT] Nguồn: Next.js Docs — Layouts and Templates:
// template.tsx KHÁC layout.tsx: tạo instance mới (re-mount hoàn toàn) mỗi lần navigate
//
// Dùng khi cần re-mount có chủ đích:
// - CSS page transition animations (enter effect trigger mỗi lần navigate)
// - Analytics event per-page view (cần side effect reset mỗi trang)
//
// KHÔNG dùng khi cần giữ state giữa các routes → dùng layout.tsx
// Template là edge case, không phải default choice
export default function DashboardTemplate({ children }: { children: ReactNode }) {
  // Mỗi navigate vào /dashboard/* → component này destroy và tạo lại từ đầu
  return (
    <div className="page-enter-animation">
      {children}
    </div>
  );
}
```

**Tóm tắt behavior của 6 special files:**

| File | Behavior | Khi nào dùng |
| :--- | :--- | :--- |
| `page.tsx` | Tạo public route — bắt buộc để URL accessible | Mọi route cần render UI |
| `layout.tsx` | Shared UI, **KHÔNG re-mount** khi navigate | Nav, sidebar, persistent state |
| `template.tsx` | Shared UI, **re-mount** mỗi navigate | Page animations, per-page analytics |
| `loading.tsx` | Suspense boundary tự động | Loading skeleton khi server fetch |
| `error.tsx` | Error boundary (**bắt buộc `"use client"`**) | Catch và hiển thị lỗi trong segment |
| `not-found.tsx` | 404 handler trong segment | Context-aware 404 UX |

### 3.2 Route Organization Conventions

Ngoài special files, App Router có 2 convention để tổ chức code mà không tạo routes:

**Route Groups `(groupName)/`:**

[FACT] Theo [Next.js Docs — Route Groups](https://nextjs.org/docs/app/building-your-application/routing/route-groups): Folder với cú pháp `(groupName)` **không thêm segment vào URL path**. Dùng để nhóm routes share cùng layout mà không muốn group name xuất hiện trong URL.

**Private Folders `_folderName/`:**

[FACT] Theo [Next.js Docs — Colocation](https://nextjs.org/docs/app/building-your-application/routing/colocation): Folder prefix `_` bị **exclude hoàn toàn khỏi routing system**. Dùng để colocate utility components, helpers cạnh route mà không tạo dead routes.

**Parallel Routes `@folder`:**

[MODEL] Nhận diện: folder bắt đầu bằng `@` (VD: `@modal`) là parallel route slot — cho phép render nhiều page cùng 1 layout slot. Implementation chi tiết thuộc M8.1, bài này chỉ cần nhận ra khi đọc codebase người khác viết.

```tsx [app/(dashboard)/layout.tsx]
import type { ReactNode } from 'react';
// File này nằm trong Route Group (dashboard)
//
// URL thực tế của routes bên trong KHÔNG có /(dashboard)/ prefix:
// app/(dashboard)/settings/page.tsx   → URL: /settings
// app/(dashboard)/profile/page.tsx    → URL: /profile
//
// Dùng khi: /settings và /profile cần share cùng DashboardLayout
// mà URL vẫn là /settings và /profile (không phải /dashboard/settings)

import { DashboardNav } from './_components/DashboardNav';
// _components/ là Private Folder
// → KHÔNG tạo route /(dashboard)/components hoặc /components
// → Chỉ để colocate code cạnh route, không thể accessible từ URL

export default function DashboardLayout({ children }: { children: ReactNode }) {
  return (
    <div className="dashboard-shell">
      <DashboardNav />
      <main>{children}</main>
    </div>
  );
}
```

Folder structure tương ứng:

```
app/
├── layout.tsx                    ← Root layout: bắt buộc, bọc toàn bộ app
│
├── (marketing)/                  ← Route Group: không xuất hiện trong URL
│   ├── layout.tsx                ← MarketingLayout cho / và /about
│   ├── page.tsx                  ← URL: /
│   └── about/
│       └── page.tsx              ← URL: /about
│
└── (dashboard)/                  ← Route Group: không xuất hiện trong URL
    ├── layout.tsx                ← DashboardLayout cho /dashboard và /settings
    ├── _components/              ← Private Folder: không tạo route
    │   ├── DashboardNav.tsx
    │   └── Sidebar.tsx
    ├── dashboard/
    │   └── page.tsx              ← URL: /dashboard
    └── settings/
        └── page.tsx              ← URL: /settings
```

> ⚠️ **Pitfall cần nhớ:** Route Groups `(folder)` không tồn tại trong URL, nhưng **vẫn tồn tại trong filesystem path**. Import ngang hàng giữa các file trong `(dashboard)/` vẫn cần include `(dashboard)/` trong relative import path — đây là nguồn gốc của nhiều lỗi `Module not found` khó debug.

### 3.3 Pages Router vs App Router Conventions

[FACT] Theo [Next.js Docs — Upgrading](https://nextjs.org/docs/app/building-your-application/upgrading): Next.js hỗ trợ hybrid mode — chạy song song Pages Router (`pages/`) và App Router (`app/`) trong cùng project.

```tsx [pages/_app.tsx]
// pages/_app.tsx — Pages Router: global wrapper duy nhất cho toàn app
//
// Vấn đề cốt lõi: không có nested layout tự nhiên
// Mọi layout logic buộc phải conditional render trong file này
// → Wrapper Hell khi app phình to
import type { AppProps } from 'next/app';
import { SessionProvider } from 'next-auth/react';
import '../styles/globals.css';

export default function MyApp({ Component, pageProps }: AppProps) {
  return (
    <SessionProvider session={pageProps.session}>
      {/* Không có cách tự nhiên để chia layout theo route segment */}
      <Component {...pageProps} />
    </SessionProvider>
  );
}
```

```tsx [app/layout.tsx]
// app/layout.tsx — App Router: root layout thay thế _app.tsx
//
// 3 điều BẮT BUỘC mà root layout phải có:
// (1) Export default function
// (2) Nhận và render children prop
// (3) Render <html> và <body> tags
//
// Không có 1 trong 3 → Next.js throw build error hoặc runtime error
import type { ReactNode } from 'react';
import { SessionProvider } from 'next-auth/react';
import './globals.css'; // Global CSS import chuyển từ _app.tsx sang đây

interface RootLayoutProps {
  children: ReactNode; // Next.js inject page content vào đây — bỏ prop này = blank app
}

export default function RootLayout({ children }: RootLayoutProps) {
  // Root layout = shell tối thiểu: html, body, global providers
  // KHÔNG fetch data ở đây → block toàn bộ app mỗi hard navigation
  return (
    <html lang="vi">
      <body>
        <SessionProvider>
          {children}
        </SessionProvider>
      </body>
    </html>
  );
}
```

**Mapping Pages Router → App Router:**

| Pages Router | App Router | Lưu ý migration |
| :--- | :--- | :--- |
| `pages/index.tsx` | `app/page.tsx` | Route `/` |
| `pages/about.tsx` | `app/about/page.tsx` | Route `/about` |
| `pages/_app.tsx` | `app/layout.tsx` | Thêm `children` prop bắt buộc, thêm `<html>`, `<body>` |
| `pages/_document.tsx` | `app/layout.tsx` (HTML shell) | `<html>`, `<body>`, meta tags vào root layout |
| `pages/_error.tsx` | `app/error.tsx` | Thêm `"use client"` directive — bắt buộc |
| `pages/404.tsx` | `app/not-found.tsx` | Đổi tên file, có thể nested ở segment levels |
| `pages/api/` | Route Handlers | Thuộc M11 — không cover ở bài này |

## 4. Anti-Patterns in the Wild

### ❌ Anti-Pattern 1: Đặt tên file tùy ý trong `app/`

```tsx [app/dashboard/index.tsx]
// ❌ Junior thường viết: thói quen từ Pages Router và thông lệ web cũ
// Hoặc: app/dashboard/Dashboard.tsx, app/dashboard/Home.tsx
//
// Next.js im lặng bỏ qua file này.
// KHÔNG error. KHÔNG warning. Chỉ... không nhận ra.
export default function DashboardPage() {
  return <div>Dashboard</div>;
}
// Kết quả: URL /dashboard → 404, dù component hoàn toàn đúng
```

```tsx [app/dashboard/page.tsx]
// ✅ Staff refactor: đổi tên về đúng convention
// Trade-off có ý thức: mất tự do đặt tên file
// đổi lấy tính dự đoán 100% của routing system —
// bất kỳ ai mở repo đều biết page.tsx = route entry, không cần đọc docs
export default function DashboardPage() {
  return <div>Dashboard</div>;
}
```

🧠 **Trade-off:** Tự do đặt tên file vs tính dự đoán của routing. Trong team 5+ người, tính dự đoán luôn thắng. "Tôi đặt tên file theo ý mình cho dễ nhớ" không scale khi codebase có 3 người mới join.

🔥 **Incident Risk:** URL `/dashboard` trả về 404 trên production dù code không có bug. Debug mất 30 phút vì không có error message — Next.js không phàn nàn, chỉ không nhận ra file. Chạy `find app/ -name "page.tsx"` để phát hiện: `/dashboard` không có trong list.

### ❌ Anti-Pattern 2: Layout quá nhiều logic

```tsx [app/dashboard/layout.tsx]
// ❌ Junior thường nhồi vào layout để "tái sử dụng logic"
import { getUser } from '@/lib/auth';
import { getDashboardConfig } from '@/lib/config';
import { useEffect } from 'react'; // ❌ Sai thêm: async Server Component không dùng được React hooks

export default async function DashboardLayout({ children }) {
  // ❌ Data fetching nặng trong layout
  // Layout không re-mount giữa navigations →
  // data này chỉ fetch 1 lần khi hard-navigate, không update khi click sang tab
  const user = await getUser();              // DB call: ~100–500ms
  const config = await getDashboardConfig(); // API call: ~200–800ms
  // Tổng: 300–1300ms BLOCK mỗi lần hard navigate vào bất kỳ /dashboard/* route

  return (
    <div>
      <nav>{user.name} | {config.dashboardTitle}</nav>
      {children}
    </div>
  );
}
```

```tsx [app/dashboard/layout.tsx]
// ✅ Staff refactor: layout chỉ là shell UI
// Auth check → middleware.ts (chạy ở Edge, không block layout)
// Data fetching → page.tsx hoặc Server Components con (không block navigation)
import type { ReactNode } from 'react';
import { DashboardNav } from './_components/DashboardNav';

export default function DashboardLayout({ children }: { children: ReactNode }) {
  // Layout không async, không data fetch, không business logic
  // → không thể block navigation
  // → DashboardNav nhận data từ chính nó (Server Component riêng)
  return (
    <div className="dashboard-shell">
      <DashboardNav />   {/* DashboardNav tự fetch data của mình nếu cần */}
      <main>{children}</main>
    </div>
  );
}
```

🧠 **Trade-off:** Gom logic vào 1 chỗ (layout) có vẻ DRY, nhưng đổi lấy việc block toàn bộ subtree. Layout không re-mount = data fetch trong layout chỉ chạy lúc hard navigate, không refresh khi soft navigate giữa tabs. Sai về cả performance lẫn correctness.

🔥 **Incident Risk:** Mọi navigation trong `/dashboard` chậm 400ms+. User report "app chậm khi đổi tab". Team profile `page.tsx`, không tìm ra gì — vì bottleneck nằm ở layout, không phải page. Mất thêm 2 giờ debug sai chỗ.

### ❌ Anti-Pattern 3: Quên `not-found.tsx` ở segment level

```
❌ Junior setup — chỉ có root level not-found:

app/
├── not-found.tsx       ← root: layout trống, không có dashboard context
├── layout.tsx
└── dashboard/
    ├── layout.tsx      ← DashboardLayout đẹp, có Sidebar và Nav
    ├── page.tsx
    └── settings/
        └── page.tsx
        ← Không có not-found.tsx ở đây
```

```
✅ Staff setup — not-found ở đúng segment level:

app/
├── not-found.tsx                ← root: fallback cuối cùng
├── layout.tsx
└── dashboard/
    ├── not-found.tsx            ← giữ DashboardLayout khi 404 trong /dashboard/*
    ├── layout.tsx
    ├── page.tsx
    └── settings/
        ├── not-found.tsx        ← giữ context khi 404 trong /dashboard/settings/*
        └── page.tsx
```

```tsx [app/dashboard/not-found.tsx]
// ✅ Segment-level not-found: user vào /dashboard/some-deleted-report
// → vẫn thấy sidebar, nav của dashboard
// → biết mình đang trong dashboard context, không bị mất orientation
import Link from 'next/link';

export default function DashboardNotFound() {
  return (
    <div className="not-found-container">
      <h2>Trang không tồn tại trong Dashboard</h2>
      <p>Link này đã bị xóa hoặc chưa được tạo.</p>
      <Link href="/dashboard">← Quay lại Dashboard</Link>
    </div>
  );
}
// So sánh với root not-found.tsx: user thấy blank page, không có sidebar,
// không biết mình đang ở đâu trong app
```

🧠 **Trade-off:** Tăng số file đổi lấy UX 404 context-aware. Trade-off này luôn đáng — không ai muốn bị ném ra trang trắng tinh khi click nhầm 1 link trong dashboard.

🔥 **Incident Risk:** User click link `/dashboard/settings/deleted-report`. App ném họ ra root 404 — mất hoàn toàn layout dashboard. User screenshot gửi support: "app bị vỡ rồi, nhìn kìa". Support ticket tăng, team mất 30 phút giải thích đây là "feature, không phải bug" — rồi đi fix.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **App Router (Convention)** | Zero-config routing, nested layouts tự nhiên, colocate file cạnh route, không merge conflict vào config trung tâm | Bắt buộc nhớ naming rules, silent fail khi sai tên file, debug khó nếu chưa quen | Mọi Next.js project từ v13+ | Khi team nhỏ không có thời gian invest vào learning conventions ngắn hạn (không scalable lâu dài) | ~2–3 ngày learning curve để internalize 6 special file names |
| **Pages Router (Legacy)** | Đơn giản, flat structure, ít magic, dễ onboard người mới | Nested layouts cực khó (Wrapper Hell), `_app.tsx` phình to, khó chia sẻ state theo route segment | Maintain legacy codebase chưa có timeline migrate | Bất kỳ project mới nào — End of Life roadmap rõ ràng | 2–5 wrapper components thêm mỗi level nesting, conditional render logic |
| **Config-based (React Router)** | Linh hoạt 100%, routing hoàn toàn tách biệt filesystem, dễ generate routes dynamically | Phải maintain file config 500+ dòng ở app lớn, không colocate được, merge conflict thường xuyên | SPAs không dùng Next.js, apps cần routing dynamic tại runtime | Bất kỳ Next.js project nào — chống lại conventions của framework | File config tăng tuyến tính với số routes; bottleneck merge conflict |
| **Flat Folder (No nesting)** | Tìm file nhanh bằng IDE search, cấu trúc đơn giản, dễ giải thích cho người mới | Mất tính phân cấp URL, không dùng được nested layouts tự nhiên, layout sharing phải code thủ công | Tiny apps (1–3 routes), MVP kiểm chứng idea nhanh | Apps có depth > 2 levels (dashboard → settings → billing) | Layout sharing = 2–3× code thủ công, phải define layout logic ở mỗi page |

## 6. Legacy Recognition

Khi nhận handover codebase Pages Router, đây là những file cần nhận ra ngay:

```
pages/
├── index.tsx           ← Route / — flat routing entry
├── about.tsx           ← Route /about — file tương ứng trực tiếp với URL
├── dashboard/
│   └── index.tsx       ← Route /dashboard — folder + index.tsx pattern
├── _app.tsx            ← Global wrapper, providers, global CSS (không có URL)
├── _document.tsx       ← Custom HTML shell (không có URL)
├── _error.tsx          ← Custom error page (không có URL)
├── 404.tsx             ← Custom 404 (không có URL)
└── api/                ← API routes — thuộc M11
    └── users.ts
```

Đặc điểm nhận dạng nhanh: folder `pages/` ở root, files có underscore prefix (`_app`, `_document`, `_error`), routing flat bằng filesystem không có nested layout tự nhiên.

## 7. Modern Approach

Migrate từ Pages Router sang App Router không phải "rewrite toàn bộ" — đó là cách nhanh nhất để tạo ra incident trên production. Strategy đúng là **incremental migration** với hybrid mode.

### Strategy: Incremental Migration + Hybrid Mode

**Phase 1: Tạo `app/` cạnh `pages/` — hybrid mode**

Sau khi `app/` folder tồn tại, `app/layout.tsx` là bắt buộc. Tạo nó trước:

```tsx [app/layout.tsx]
// Bước đầu tiên trong mọi App Router migration
// File này BẮT BUỘC tồn tại nếu app/ folder tồn tại
// Providers và global CSS từ pages/_app.tsx chuyển vào đây
import type { ReactNode } from 'react';
import { SessionProvider } from 'next-auth/react';
import './globals.css'; // Chuyển global CSS import từ _app.tsx sang đây

export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html lang="vi">
      <body>
        <SessionProvider>
          {children}
        </SessionProvider>
      </body>
    </html>
  );
}
```

Trong cùng phase này, Pages Router vẫn chạy bình thường — không có route nào bị gián đoạn.

**Phase 2: Migrate từng route, leaf routes trước**

Sau khi root layout tồn tại và stable, migrate từng route theo thứ tự leaf → root. Verify mỗi route trên staging trước khi tiếp tục:

```
pages/dashboard/index.tsx  →  app/dashboard/page.tsx
pages/about.tsx            →  app/about/page.tsx
pages/index.tsx            →  app/page.tsx         (cuối cùng — root route)
pages/404.tsx              →  app/not-found.tsx
```

**Phase 3: Rollback plan nếu staging fail**

Rollback = rename hoặc xóa folder `app/`. Pages Router tiếp tục serve toàn bộ routes ngay lập tức. Không cần rollback database, không downtime, không ảnh hưởng user.

```bash
# Rollback nhanh trong hybrid mode:
mv app/ app.bak/
# → Pages Router tiếp quản toàn bộ, app/ không còn tồn tại
# → Verify production, sau đó quyết định có tiếp tục migrate không
```

**Chỉ xóa `pages/` sau khi toàn bộ routes đã verify trên staging** — không bao giờ xóa trước.

### Migration chi tiết: `pages/_app.tsx` → `app/layout.tsx`

```tsx [pages/_app.tsx]
// TRƯỚC — Pages Router: tất cả providers và CSS ở đây
import type { AppProps } from 'next/app';
import { SessionProvider } from 'next-auth/react';
import { ThemeProvider } from '@/components/ThemeProvider';
import '../styles/globals.css';
import '../styles/theme.css';

export default function MyApp({ Component, pageProps }: AppProps) {
  return (
    <SessionProvider session={pageProps.session}>
      <ThemeProvider defaultTheme="light">
        <Component {...pageProps} />
      </ThemeProvider>
    </SessionProvider>
  );
}
```

```tsx [app/layout.tsx]
// SAU — App Router: tương đương 1:1 với _app.tsx trên
// Thay đổi duy nhất: (1) thêm children prop, (2) thêm html/body, (3) CSS import path
import type { ReactNode } from 'react';
import { SessionProvider } from 'next-auth/react';
import { ThemeProvider } from '@/components/ThemeProvider';
import './globals.css';   // Đường dẫn relative thay đổi vì file ở app/ thay vì pages/
import './theme.css';

export default function RootLayout({ children }: { children: ReactNode }) {
  return (
    <html lang="vi">
      <body>
        <SessionProvider>
          <ThemeProvider defaultTheme="light">
            {children}
          </ThemeProvider>
        </SessionProvider>
      </body>
    </html>
  );
}
```

## 8. Production Scenario

> 🔴 **Scenario 1: "Route không render đúng — URL /dashboard trả về 404"**
>
> **Symptom:** Developer viết component, chạy `npm run dev`, vào `/dashboard` thấy Next.js 404 page.
>
> **Root cause:** File đặt tên `app/dashboard/index.tsx` hoặc `app/dashboard/Dashboard.tsx`. Next.js không error — im lặng bỏ qua.
>
> **Detect:** Kiểm tra terminal output khi start dev server. Next.js log danh sách routes nó nhận ra. Nếu `/dashboard` không có trong list → folder `app/dashboard/` không có `page.tsx`. Xác nhận nhanh bằng: `find app/ -name "page.tsx" | sort` — list này phải có đầy đủ routes cần thiết.
>
> **Fix:** Đổi tên file thành `page.tsx`. Convention không thể thương lượng, không có workaround.

---

> 🔴 **Scenario 2: "Header/Sidebar bị nháy mỗi lần chuyển tab trong dashboard"**
>
> **Symptom:** Navigate từ `/dashboard/analytics` sang `/dashboard/settings` — header unmount rồi mount lại, sidebar mất state, animation reset.
>
> **Root cause:** `layout.tsx` đặt sai segment level — nằm sâu trong từng subfolder thay vì ở `dashboard/` chung.
>
> ```
> ❌ Sai: layout.tsx trong từng subfolder → 2 layout riêng biệt, re-mount khi navigate
> app/dashboard/
> ├── analytics/
> │   ├── layout.tsx    ← Layout chỉ apply cho /dashboard/analytics
> │   └── page.tsx
> └── settings/
>     ├── layout.tsx    ← Layout khác → unmount analytics layout, mount settings layout
>     └── page.tsx
>
> ✅ Đúng: layout.tsx ở folder cha chung → share, không re-mount
> app/dashboard/
> ├── layout.tsx        ← Share cho /dashboard/analytics VÀ /dashboard/settings
> ├── analytics/
> │   └── page.tsx
> └── settings/
>     └── page.tsx
> ```
>
> **Detect:** Vẽ routing tree ra giấy, kiểm tra vị trí `layout.tsx` so với các routes cần share. Trong React DevTools, bật "Highlight updates when components render" — nếu Nav/Sidebar highlight đỏ khi navigate giữa tabs → đang re-mount sai.
>
> **Fix:** Di chuyển `layout.tsx` lên đúng segment level (folder cha chung của routes cần share).

---

> 🔴 **Scenario 3: "URL sai trên production do nhầm Route Group"**
>
> **Symptom:** Developer tạo `app/(dashboard)/settings/page.tsx` và link trong code là `href="/dashboard/settings"` → 404 trên production. Hoặc ngược lại: nghĩ URL là `/settings` nhưng lại link `/dashboard/settings`.
>
> **Root cause:** Nhầm `(dashboard)` là URL segment thật. Folder này tồn tại trên filesystem, không tồn tại trong URL path.
>
> **Detect:** Mở Network tab trong Chrome DevTools — URL bar không có `(dashboard)`. Kiểm tra tất cả `<Link href=...>` và `redirect(...)` trong codebase: `grep -r "dashboard/settings" src/` → nếu có link `/dashboard/settings` nhưng route thực là `/settings` → đây là bug.
>
> **Fix:** Cập nhật tất cả links về URL thực tế (không có Route Group name). Dùng `next/link` hoặc `useRouter().push()` với URL đúng.

## 9. Performance & Overhead

**`layout.tsx` preserve state vs `template.tsx` re-mount:**

`layout.tsx` giữ nguyên React component tree, DOM nodes, và mọi state khi navigate giữa các children trong cùng segment. Không có React reconciliation overhead cho layout tree — chỉ có `children` (page content) thay đổi. Phù hợp cho sidebar, navigation, mọi state cần persist.

`template.tsx` destroy và tạo lại toàn bộ component subtree mỗi navigate. Trigger full React reconciliation cycle, GC thu dọn instance cũ, khởi tạo instance mới. Phù hợp cho CSS animations (enter effect trigger lại), per-page side effects. Không bao giờ là default choice — chỉ dùng khi có lý do cụ thể.

**Root `layout.tsx` data fetching — impact thực tế trên production:**

Root layout bọc 100% routes. 1 database query 200ms trong root layout = 200ms thêm vào TTFB (Time to First Byte) cho mọi hard navigation (F5, direct URL, link từ external site).

```
Ví dụ tác động thực tế:
Database query trong root layout: 200ms
User trên mobile 4G (round-trip latency ~80ms): 200 + 80 = ~280ms thêm vào TTFB
User trên slow 3G (~200ms RTT): 200 + 200 = ~400ms thêm vào TTFB

Nếu app có 10.000 hard navigations/ngày:
→ 10.000 × 200ms = 2.000 giây user time bị lãng phí mỗi ngày
```

> 📊 **Benchmark guide:** Nghi ngờ root layout block? Thêm timestamps trong server logs:
> ```ts
> export default async function RootLayout({ children }) {
>   const start = Date.now();
>   // ... data fetch nếu có
>   console.log(`[RootLayout] render time: ${Date.now() - start}ms`);
>   return (...);
> }
> ```
> Metric cần theo dõi: TTFB trong Chrome DevTools → Network tab → chọn document request → xem "Waiting (TTFB)". Nếu TTFB > 200ms, profile server-side trước.

**`loading.tsx` Suspense — clarification quan trọng:**

`loading.tsx` không giảm actual server fetch time. Nó giảm **perceived latency**: Next.js stream HTML skeleton ra browser ngay lập tức khi layout resolve, trước khi page data sẵn sàng. Impact thực tế:
- Giảm thời gian blank screen (cải thiện LCP — Largest Contentful Paint)
- Giảm CLS (Cumulative Layout Shift) nếu skeleton đúng kích thước với content thật
- Actual data fetch time không thay đổi

## 10. Architecture Notes

*(Đọc để biết tồn tại — không cần deep-dive ở M0.4)*

**Template vs Layout decision tree:**

Câu hỏi duy nhất cần trả lời: "Component này có cần giữ state/DOM khi user navigate không?"
- Có → `layout.tsx`
- Không, và có lý do cụ thể cần re-mount (animations, analytics) → `template.tsx`

Mặc định luôn là `layout.tsx`. `template.tsx` là exception có tài liệu, không phải alternative.

**Route Groups `(folder)` — khi nào thực sự cần:**

Khi: 2+ group routes cần layout khác nhau, nhưng URL structure phải flat. Ví dụ: marketing pages (`/`, `/about`, `/pricing`) và app pages (`/dashboard`, `/settings`) cần 2 layout hoàn toàn khác nhau mà không muốn group name xuất hiện trong URL.

Không khi: muốn "làm phẳng" URL có nested intent (`/dashboard/settings` thực sự là nested) hoặc để workaround folder structure.

**Private folders `_components/` — scope và limit:**

`_components/` cạnh route = colocated components chỉ dùng cho route đó. Nếu component được dùng ở 3+ routes khác nhau → nó thuộc `src/components/` (shared), không phải `_components/`. Private folders cho local code, không phải shared utilities.

**Root layout — golden rule:**

Root `app/layout.tsx` phải lightweight nhất có thể: HTML shell + global providers không async. Mọi blocking operation = penalty cho 100% app. Auth check → `middleware.ts`. Config → environment variables. Data fetch → page.tsx hoặc Server Component con.

## 11. Interview Questions

### Q1: Sự khác biệt giữa `layout.tsx` và `template.tsx`? (Junior)

<details>
<summary>Đáp án</summary>

`layout.tsx` **không re-mount** khi navigate giữa các children trong cùng segment — React giữ nguyên component instance, DOM tree, và state. Phù hợp cho sidebar, navigation, persistent state.

`template.tsx` **tạo instance mới** (re-mount hoàn toàn) trên mỗi navigation. Dùng khi cần: CSS enter/exit animations, analytics event per-page, bất kỳ side effect nào cần reset khi chuyển trang.

Rule of thumb trong production: dùng `layout.tsx` mặc định. Chỉ chuyển sang `template.tsx` khi có lý do cụ thể, rõ ràng cần re-mount.

</details>

---

### Q2: Tại sao App Router dùng convention thay vì configuration? (Mid)

<details>
<summary>Đáp án</summary>

3 lý do cụ thể, có thể dẫn chứng bằng production experience:

**Colocation:** UI, layout, loading state, error handler của cùng 1 route nằm cùng folder. Trace từ URL → folder → file mà không cần đọc file config riêng.

**Scale ngang không có merge conflict:** Thêm 1 route = tạo 1 folder + `page.tsx`. Không sửa file config trung tâm. Team 10 người làm song song 10 features, không conflict.

**Tính dự đoán shared across team:** `page.tsx` luôn là route entry, mọi nơi, mọi project Next.js. Onboard người mới không cần giải thích từng project setup khác nhau.

</details>

---

### Q3: Tại sao `page.tsx` là tên file bắt buộc thay vì `index.tsx` như thông lệ web? (Mid)

<details>
<summary>Đáp án</summary>

`index.tsx` có nghĩa "file mặc định của folder" — không nói gì về routing intent. `page.tsx` có nghĩa rõ ràng hơn: "folder này là một route segment có UI render."

Tên `page.tsx` tạo không gian cho các special files cùng tồn tại trong folder: `loading.tsx`, `error.tsx`, `not-found.tsx`, `template.tsx` — tất cả cùng cấp, rõ ràng về purpose. Với `index.tsx`, không có convention tự nhiên để phân biệt "đây là route entry" vs "đây là special file."

Thực tế: naming rõ ràng giảm câu hỏi "file này làm gì?" trong code review.

</details>

---

### Q4: Khi nào bạn dùng Route Groups `(folder)` thay vì nested folders bình thường? (Senior)

<details>
<summary>Đáp án</summary>

Khi cần: **chia routes vào layouts khác nhau mà không muốn group name xuất hiện trong URL.**

Ví dụ production thực tế: marketing pages (`/`, `/about`, `/blog`) và dashboard pages (`/dashboard`, `/settings`, `/profile`) dùng 2 layout hoàn toàn khác nhau. Nếu dùng nested folders bình thường, URL phải là `/marketing/about` hoặc `/app/dashboard` — không đúng với business requirement. Route Groups cho phép share layout theo nhóm mà URL vẫn là `/about` và `/dashboard`.

Khi không dùng: nếu URL hierarchy thực sự là nested (`/dashboard/settings` là có ý nghĩa business là "settings trong dashboard"), dùng nested folders bình thường — không cần Route Groups.

</details>

---

### Q5: Hệ quả performance nếu để data fetching nặng trong root `layout.tsx`? (Senior)

<details>
<summary>Đáp án</summary>

Root layout bọc **toàn bộ app**. Mọi hard navigation (F5, external link, direct URL) phải chờ root layout resolve trước khi render bất kỳ content nào.

Database query 300ms trong root layout: mọi hard navigation cộng thêm 300ms vào TTFB. Trên mobile với latency cao, thực tế có thể 500–800ms. Với 10.000 hard navigations/ngày, đó là hàng giờ user time bị lãng phí.

Soft navigation (client-side routing) thường không bị ảnh hưởng vì layout không re-mount — nhưng users mobile xài hard navigate nhiều hơn desktop users.

Fix: data fetch xuống `page.tsx` hoặc Server Component con. Auth check → `middleware.ts` (Edge Runtime, không block layout). Root layout = HTML shell + async-free providers, không hơn.

</details>

Vẽ tree ra giấy trước khi gõ code. `layout.tsx` "bọc" tất cả `page.tsx` và subfolder nằm trong cùng folder với nó — đặt nó ở folder cha chung của tất cả routes cần share.

`/dashboard/settings/billing` là 3 segment lồng nhau: `dashboard` → `settings` → `billing`. Mỗi segment = 1 folder riêng. Chỉ folder `billing/` cần `page.tsx` — các folder cha không cần `page.tsx` nếu URL đó không cần accessible riêng.

`loading.tsx` và `not-found.tsx` đặt ở `dashboard/` level sẽ tự động áp dụng cho toàn bộ `/dashboard/*` routes — không cần lặp lại ở mỗi subfolder (trừ khi muốn custom riêng).

</details>

import { getUser } from '@/lib/auth';
import { getAnnouncements } from '@/lib/cms';
import { useState, useEffect } from 'react';

export default async function DashboardLayout({ children }) {
  const user = await getUser();                    // DB call
  const announcements = await getAnnouncements();  // CMS API call

  const [collapsed, setCollapsed] = useState(false); // Hook trong async component

  }, [user]);

  return (
    <div className={collapsed ? 'collapsed' : ''}>
      <nav>
        <span>{user.displayName}</span>
        {announcements.map(a => <p key={a.id}>{a.text}</p>)}
        <button onClick={() => setCollapsed(!collapsed)}>Toggle</button>
      </nav>
      {children}
    </div>
  );
}
```

- [ ] Vấn đề 2a: `layout.tsx` không còn dùng React hooks (`useState`, `useEffect`) — async Server Component không thể dùng hooks

- [ ] `layout.tsx` sau fix: không async, không hooks, không data fetching

Lỗi đầu tiên: `async function` + `useState`/`useEffect` là không thể. Async Server Component không có lifecycle hooks — chúng chạy trên server, không có DOM, không có React event loop. Chọn 1 trong 2: hoặc async (Server Component), hoặc hooks (Client Component).

## 13. Summary

- `page.tsx` là tên duy nhất Next.js nhận ra để tạo public route trong App Router — `index.tsx` bị bỏ qua hoàn toàn, im lặng, không error, không warning. Đây là bẫy kinh điển nhất khi migrate từ Pages Router.
- `layout.tsx` persist state, không re-mount — giữ lightweight, không data fetching. `template.tsx` re-mount mỗi navigate — edge case cho animations, không phải default choice.
- Route Groups `(folder)` và Private Folders `_folder` là công cụ tổ chức code, không phải URL segments thật — `(dashboard)` không xuất hiện trong URL, nhưng vẫn tồn tại trong filesystem path cho imports.
- `not-found.tsx` ở segment level là ranh giới giữa UX context-aware (user giữ được sidebar khi 404) và "app bị vỡ" (user bị ném ra blank page không có navigation).

## 14. Terminology Evolution

**page.tsx**
```
┌──────────────┬──────────────────────┬────────────────────────────────────────────┐
│ Era          │ Convention           │ Dùng khi nào                               │
├──────────────┼──────────────────────┼────────────────────────────────────────────┤
│ Pages Router │ pages/index.tsx      │ Route entry trong flat folder structure    │
│ App Router   │ page.tsx             │ UI unique to route segment, tạo public URL │
└──────────────┴──────────────────────┴────────────────────────────────────────────┘
```

*Lịch sử:* `index.tsx` là thông lệ web từ thời file server — "file mặc định của folder". Next.js App Router đổi thành `page.tsx` để phân biệt "folder là route segment có UI" vs "folder chứa assets hoặc utilities". Tên rõ ràng hơn tạo không gian cho các special files cùng cấp.

*Codebase legacy:* `pages/index.tsx` → `app/page.tsx`. `pages/about/index.tsx` → `app/about/page.tsx`.

---

**layout.tsx**
```
┌──────────────┬──────────────────────┬────────────────────────────────────────────┐
│ Era          │ Convention           │ Dùng khi nào                               │
├──────────────┼──────────────────────┼────────────────────────────────────────────┤
│ Pages Router │ pages/_app.tsx       │ Global wrapper duy nhất, providers, CSS    │
│ App Router   │ layout.tsx           │ Shared UI across routes, không re-mount    │
└──────────────┴──────────────────────┴────────────────────────────────────────────┘
```

*Lịch sử:* `_app.tsx` là global wrapper — mọi layout logic phải chui vào 1 file, dẫn đến Wrapper Hell khi app phình to. `layout.tsx` cho phép nested layouts tự nhiên theo folder structure: mỗi segment có layout riêng, preserve state khi navigate giữa children.

*Codebase legacy:* `pages/_app.tsx` → `app/layout.tsx`. Lưu ý bắt buộc: thêm `children` prop, thêm `<html>` và `<body>` tags trong root layout.

---

**template.tsx**
```
┌──────────────┬──────────────────────┬────────────────────────────────────────────┐
│ Era          │ Convention           │ Dùng khi nào                               │
├──────────────┼──────────────────────┼────────────────────────────────────────────┤
│ Pages Router │ Không tồn tại        │ N/A                                        │
│ App Router   │ template.tsx         │ Re-mount on navigation, animations, resets │
└──────────────┴──────────────────────┴────────────────────────────────────────────┘
```

*Lịch sử:* Khái niệm hoàn toàn mới từ App Router v13+. Giải quyết nhu cầu "layout-like nhưng cần re-mount" mà Pages Router không có cách handle tự nhiên (phải hack bằng `useEffect` + `key` prop).

*Codebase legacy:* Không có tương đương. Convention mới hoàn toàn, không cần migration từ Pages Router.

---

**loading.tsx / error.tsx / not-found.tsx**
```
┌──────────────┬───────────────────────────────────┬──────────────────────────────┐
│ Era          │ Convention                        │ Dùng khi nào                 │
├──────────────┼───────────────────────────────────┼──────────────────────────────┤
│ Pages Router │ pages/_error.tsx                  │ Error handling tập trung     │
│ Pages Router │ pages/404.tsx                     │ 404 page duy nhất, toàn app  │
│ App Router   │ loading.tsx                       │ Suspense boundary cho segment│
│ App Router   │ error.tsx (bắt buộc "use client") │ Error boundary cho segment   │
│ App Router   │ not-found.tsx                     │ 404 handler phân cấp         │
└──────────────┴───────────────────────────────────┴──────────────────────────────┘
```

*Lịch sử:* Pages Router xử lý error/404 tập trung — 1 file cho toàn app, không có context của route nào đang bị lỗi. App Router colocate `loading/error/not-found.tsx` cạnh `page.tsx` để mỗi segment tự handle, giữ nguyên layout context khi lỗi hoặc 404.

*Codebase legacy:* `pages/_error.tsx` → `app/error.tsx` (thêm `"use client"` directive — bắt buộc, không có exception). `pages/404.tsx` → `app/not-found.tsx` (có thể thêm nested ở các segment levels quan trọng).

---

**Segment**
```
┌──────────────┬──────────────────────┬────────────────────────────────────────────┐
│ Era          │ Concept              │ Ý nghĩa                                    │
├──────────────┼──────────────────────┼────────────────────────────────────────────┤
│ Pages Router │ Không có chính thức  │ Chỉ gọi là "route" hoặc "path"            │
│ App Router   │ Segment              │ 1 phần URL path = 1 folder trong app/      │
└──────────────┴──────────────────────┴────────────────────────────────────────────┘
```

*Lịch sử:* Terminology chính thức từ App Router để mô tả phân cấp URL tree. `/dashboard/settings` = 2 segments: `dashboard` và `settings`. Mỗi segment tương ứng 1 folder, có thể có layout/loading/error/not-found riêng — đây là foundation của toàn bộ mental model nested routing.

*Codebase legacy:* Không có tương đương trực tiếp trong Pages Router. Khi migrate, cần internalize concept này trước — nếu không, sẽ không hiểu tại sao `layout.tsx` đặt ở folder này lại áp dụng cho URL kia.
