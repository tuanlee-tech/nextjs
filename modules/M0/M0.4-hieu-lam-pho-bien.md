**3. Mental Model (Ý tưởng cốt lõi)**

> **Cú pháp giống nhau ≠ Execution model giống nhau.** Server Component và Client Component chia sẻ JSX nhưng là hai runtime khác nhau với quy tắc khác nhau — coi chúng như một là nguồn gốc của đa số lỗi nghiêm trọng.

---

**4. Core Concepts (20% kiến thức quan trọng nhất)**

**Nhóm A — Bản chất framework**

- **#1** Next.js ≠ React + Router. RSC là runtime riêng (async, no state, no hook) — không phải "React thiếu tính năng"
- **#2** Tutorial cũ (v9–12) không áp được cho App Router. `getServerSideProps` trong `app/` = dead code, fail silently

**Nhóm B — Migration**

- **#3** App Router ≠ đổi folder. Rendering model khác: async component, streaming, params là Promise (v15+)
- **#4** Migrate route-by-route ảnh hưởng shared code (`lib/auth.ts` phải support 2 paradigm → dual maintenance tax)

**Nhóm C — Server/Client Components**

- **#5** `'use client'` ≠ chỉ chạy browser. Component vẫn SSR trên server trước hydration → hydration mismatch nếu dùng `Date`, `window`, `Math.random()`
- **#6** `'use client'` ở cha → kéo **toàn bộ subtree** vào client bundle. Đặt boundary càng gần leaf càng tốt
- **#7** RSC không phải "luôn tốt hơn". Tương tác <100ms → bắt buộc Client Component (không phải lựa chọn)

**Nhóm D — Caching**

- **#8** `revalidate` = ngưỡng chấp nhận cũ theo TIME, không phải đảm bảo tươi mới. Mutation quan trọng cần `revalidateTag`/`revalidatePath`
- **#9** `force-dynamic` ≠ default an toàn. Chỉ dành cho per-user/real-time. Content tĩnh → SSG/ISR, không thì biến Next.js thành "Express chậm"

**Nhóm E — Kiến trúc**

- **#10** Next.js cung cấp building blocks, không quyết định architecture. Flat `app/` đúng cho 1–2 dev, nhưng không phải "set once forever" cho 500k users

**6. Modern Approach (Ngày nay làm thế nào?)**

| Legacy                                       | Modern                                                   |
| -------------------------------------------- | -------------------------------------------------------- |
| `getServerSideProps`/`getStaticProps` export | `async function` page, fetch trực tiếp                   |
| `getStaticPaths`                             | `generateStaticParams`                                   |
| `'use client'` ở page/layout cha             | `'use client'` ở component nhỏ nhất cần interactivity    |
| `useEffect` fetch data                       | Server Component `async` fetch                           |
| `revalidate` thuần time-based                | `revalidate` + `revalidateTag` cho mutation              |
| `force-dynamic` toàn site                    | `force-dynamic` chỉ per-user route, còn lại SSG/ISR      |
| Business logic trong `actions.ts`            | Tách `features/[domain]/service.ts` (framework-agnostic) |

**8. Debugging Scenario (Debug nhanh)**

**"Tại sao Dashboard hiện data cũ?"**

Checklist:

- □ Có bao nhiêu layer cache active? (Full Route, Data Cache, Router Cache client)
- □ Timer mỗi layer là bao nhiêu? Có fetch lồng nhau không? (route revalidate 60s vs fetch 900s)
- □ Mutation có gọi `revalidateTag`/`revalidatePath` không?
- □ Default cache behavior của version đang dùng là gì? (v13/v14: cached by default; v15+: no-store by default)
- □ "User thấy cũ sau 2 giờ" = request mới hay chỉ client-side navigation? (Router Cache)

Lệnh hay dùng:

```bash

# Tìm 'use client' ở layout (creep)
grep -rln "'use client'" app/**/layout.tsx

# Kiểm tra độ lớn actions.ts
find app -name "actions.ts" | xargs wc -l | sort -rn

# Kiểm tra revalidate/force-dynamic
grep -rn "revalidate\|force-dynamic" app/
```

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng**

- Server Component: chỉ render data, không cần browser API/interactivity <100ms
- Client Component: cần `useState`, `useEffect`, event handler, browser API
- `force-dynamic`: per-user data, real-time, phụ thuộc cookies/headers
- SSG/ISR: content giống nhau cho mọi user, ít đổi

**Khi không nên dùng**

- `'use client'` ở ancestor chỉ để "fix lỗi hooks" — đặt ở leaf
- `force-dynamic` cho blog/marketing/docs — tắt caching layer vô cớ
- Flat structure cho team > 4 người, > 2 domain

Trade-off

| Pros                                                                | Cons                                                         |
| ------------------------------------------------------------------- | ------------------------------------------------------------ |
| Server Component: zero JS bundle, direct DB access, an toàn secrets | Không có state, không tương tác tức thì                      |
| Client Component: interactivity, phản hồi <100ms                    | Tăng bundle size, cần hydration                              |
| `force-dynamic`: luôn fresh, đúng correctness                       | Mất CDN cache, TTFB cao, server load tuyến tính theo traffic |
| SSG/ISR: CDN edge, miễn phí scale, TTFB thấp                        | Có thể stale, cần invalidation strategy                      |

## 1. Lesson Goal

Đây là bài cuối của Module M0 — đóng vai trò **radar**, không phải **thuốc**. Sau bài này, bạn cần làm được:

- Nhận ra khi MỘT TRONG 10 hiểu lầm này đang điều khiển một quyết định — của bạn, đồng nghiệp, hoặc chính đoạn code bạn đang đọc. Mỗi hiểu lầm sẽ có một "câu nói báo động" cụ thể — khi nghe thấy câu đó trong standup/code review, bạn biết phải dừng lại.
- Giải thích được **tại sao** hiểu lầm đó "có lý" trong một ngữ cảnh nào đó. Không ai "ngu" khi tin vào những điều này — họ đang áp dụng một mental model ĐÚNG-Ở-QUÁ-KHỨ vào HIỆN-TẠI. Hiểu được nguồn gốc giúp bạn sửa tận gốc, không chỉ "sửa triệu chứng".
- Dùng **"Lăng kính 3 chiều"** (mental model của bài này) để tự kiểm tra bất kỳ assumption MỚI nào về Next.js trong tương lai — không chỉ 10 cái này.
- Biết **module nào sẽ "trị" tận gốc** từng hiểu lầm (M2, M6, M7, M15...) — bài này chỉ chẩn đoán, việc điều trị nằm ở các module chuyên sâu.

Lưu ý quan trọng: cả 10 hiểu lầm dưới đây đều **từng là sự thật** ở một thời điểm/context nào đó của Next.js. Đó chính là lý do chúng nguy hiểm — chúng không phải "sai hoàn toàn", mà là "đúng nhưng đã hết hạn" hoặc "đúng một phần nhưng thiếu điều kiện áp dụng".

## 2. Evolution Context — "4 Lớp Di Sản Tư Duy"

Next.js đã trải qua ít nhất **4 lần đổi mô hình tư duy** (paradigm shift) lớn. Mỗi lần đổi, một lớp kiến thức cũ — từng đúng, từng được dạy rộng rãi — bị bỏ lại phía sau. Nhưng internet không "quên" theo kịp framework: blog cũ, course cũ, video YouTube cũ, và cả câu trả lời của AI model được train trên dữ liệu cũ vẫn tiếp tục lan truyền lớp kiến thức đó như thể nó còn đúng.

**Lớp 1 — Pre-Next.js / CSR React thuần** (~trước 2020, nhưng vẫn là mental model mặc định của hầu hết React dev mới)

> Niềm tin nền: "Component = hàm JavaScript chạy trong browser, render ra DOM, hết chuyện."
> Đúng 100% trong thế giới Create React App / Vite SPA.

**Lớp 2 — Pages Router Next.js với `getServerSideProps`/`getStaticProps` (9–12, ~2020–2022)** — vẫn là **MAJORITY content** trên internet tính đến hiện tại

> Niềm tin nền: "Data fetching = các function export đặc biệt (`getServerSideProps`, `getStaticProps`), còn component vẫn là pure React nhận `props` như bình thường."
> Đúng 100% trong kỷ nguyên Pages Router — và đây là lý do M2 (Next.js Kế thừa) tồn tại như một module riêng.

**Lớp 3 — Early App Router (13.4–14, ~May 2023–2024)** — giai đoạn "cả cộng đồng vừa học vừa dạy lại nhau"

> Niềm tin nền: "App Router = đổi thư mục + Server Components = server-side nên nhanh hơn, dùng cho tất cả + `'use client'` = code chạy ở client."
> Đây là giai đoạn các **oversimplification** lan truyền mạnh nhất — vì độ chính xác trung bình của nội dung "App Router tutorial" thời điểm đó rất thấp, nhưng số lượng lại rất nhiều.

Mỗi hiểu lầm trong bài này sẽ được gắn nhãn **"bắt nguồn từ lớp nào"** — để bạn thấy rõ: không phải bạn "không học kỹ", mà là bạn đang mang một bộ giả định từ một kỷ nguyên khác sang kỷ nguyên hiện tại.

Riêng **Nhóm D (Caching)** có một tính chất khác biệt: đây không chỉ là "tư duy cũ" mà là **"tư duy đúng-ở-version-trước, sai-ở-version-sau"** — vì _default behavior_ của caching đã thay đổi qua các major version (13 → 14 → 15 → 16). Đây là loại hiểu lầm "động", cần re-verify liên tục — sẽ quay lại sâu ở M7.

**Lớp 4 — Cache Components Era (Next.js 16, ~Oct 2025–nay)**

> Niềm tin nền: "`'use cache'` directive thay thế hoàn toàn fetch caching và route segment config. Mọi thứ dynamic by default, opt-in để cache."
> Đây là paradigm shift lớn thứ 4: từ "static by default, opt-out để dynamic" (v14) → "dynamic by default, opt-in để cache" (v15) → "explicit cache với compiler-generated keys" (v16). Điều này làm cho nhiều pattern về `revalidate`, `dynamic`, `fetchCache` trở nên legacy.

## 3. Mental Model — "Lăng Kính 3 Chiều"

Đây là công cụ chính của bài học, và cũng là công cụ bạn dùng **cho mọi bài học sau này** trong chương trình. Với BẤT KỲ API, pattern, hay đoạn code Next.js nào, hãy hỏi 3 câu:

### 🔍 Chiều 1 — Vị trí thực thi (Execution Location)

> _Đoạn code này chạy ở đâu, và khi nào?_

- **Server hay Client?** (Không phải "trông giống React" là chạy ở client)
- **Build-time, Request-time, hay cả hai?** (SSG vs SSR vs ISR)
- **Node.js runtime hay Edge runtime?** (khác nhau về API khả dụng)

### 🔍 Chiều 2 — Vòng đời Cache (Cache Lifecycle)

> _Kết quả này được lưu ở đâu, tồn tại bao lâu, và ai/khi nào xoá nó?_

- Request Memoization? Data Cache? Full Route Cache? Router Cache (client)?
- Điều gì **trigger invalidation** — thời gian (`revalidate`), tag (`revalidateTag`), path, hay một lần deploy mới?

### 🔍 Chiều 3 — Tính ổn định theo Version (Version Stability)

> _Mental model này dựa trên hành vi MẶC ĐỊNH của version nào — và default đó còn đúng ở version bạn đang dùng không?_

---

**Cách dùng:** mỗi hiểu lầm dưới đây sẽ được đánh dấu **vi phạm chiều nào**. Bạn sẽ thấy một pattern lặp lại: phần lớn lỗi tư duy nghiêm trọng về Next.js đều quy về việc **collapse 3 chiều này thành 1** — coi mọi đoạn code đều "chỉ là JavaScript, chạy ở đâu đó, lúc nào đó, luôn giống nhau".

### Hiểu lầm #1: "Next.js chỉ là React + Router + SSR — học React kỹ là đủ"

🏷️ **Vi phạm:** Chiều 1 (Execution Location) + Chiều 2 (Cache Lifecycle)

📜 **Nguồn gốc:** Lớp 1 (CSR React)

🚨 **Câu nói báo động:** _"Tôi viết React giỏi rồi, Next.js chỉ là thêm routing với SSR thôi, có gì khó."_

**❌ Niềm tin:**

Next.js là một "lớp vỏ" mỏng bọc ngoài React — thêm file-based routing, thêm khả năng render trên server, nhưng bản chất component vẫn là React component bình thường, tuân theo đúng quy tắc React mà bạn đã biết.

**✅ Sự thật:**

Từ App Router, Next.js định nghĩa **mô hình thực thi riêng của nó** — không phải "React chạy ở 2 nơi", mà là **hai runtime khác nhau với quy tắc khác nhau, chia sẻ một cú pháp giống nhau**. Server Components là `async function` chạy trên server, không có state, không có lifecycle, không có event handler — đây không phải "React thiếu tính năng", mà là một **execution model khác hẳn**, có quy tắc riêng (data fetching trực tiếp trong component, serialization boundary khi truyền props sang Client Component, v.v.)

**💻 Minh chứng:**

```tsx
// app/dashboard/page.tsx
import { useState } from 'react';

export default async function DashboardPage() {
  // 💥 Lỗi: useState chỉ hoạt động trong Client Component
  const [filter, setFilter] = useState('all');

  const data = await fetchDashboardData(filter);
  return <div>{/* render data */}</div>;
}
```

Với mental model "Next.js = React + Router", lỗi này **vô lý** — "tôi vẫn viết React đúng cú pháp mà, sao lỗi?". Nhưng với mental model đúng (Chiều 1: đây là Server Component, chạy trên server, không có vòng đời client), lỗi này hoàn toàn logic: `useState` đòi hỏi một instance sống trên client để giữ state qua các lần render — thứ không tồn tại trong execution model của Server Component.

**💥 Hệ quả production:**

Dev mang mental model React thuần sẽ debug **sai tầng**. Gặp lỗi, họ tìm "React hook rules", "tại sao state không update" — trong khi vấn đề thực sự là "đoạn code này đang chạy ở runtime nào". Hậu quả: hàng giờ debug sai hướng, và khi "fix" được (thường bằng cách thêm `'use client'` bừa — xem Hiểu lầm #6), root cause vẫn chưa được hiểu, tích lũy thành nợ kỹ thuật.

### Hiểu lầm #2: "Code/pattern từ tutorial cũ (Next 9–12) áp dụng được nguyên vẹn cho version mới"

📜 **Nguồn gốc:** Lớp 2 (Pages Router) — đây là lý do M0.2 (Nhận diện Phiên bản) và toàn bộ M2 tồn tại

### Hiểu lầm #3: "App Router chỉ là đổi `pages/` → `app/`, logic fetch data giữ nguyên"

🏷️ **Vi phạm:** Chiều 1 (Execution Location) + Chiều 3 (Version Stability)

📜 **Nguồn gốc:** Lớp 3 (Early App Router messaging — "App Router = thư mục mới")

🚨 **Câu nói báo động:** _"Migrate dễ thôi, mình đổi tên folder rồi sửa import là xong."_

**❌ Niềm tin:**

Sự khác biệt giữa Pages Router và App Router chủ yếu là **routing convention** — `pages/blog/[slug].js` → `app/blog/[slug]/page.tsx`. Logic bên trong (data fetching, component structure) có thể giữ nguyên, chỉ cần "đổi vỏ".

**✅ Sự thật:**

Routing chỉ là phần **nhìn thấy được** của thay đổi. Bên dưới là một thay đổi sâu hơn nhiều: rendering model chuyển từ "Client Component nhận props từ một special function" sang "Server Component tự fetch async, có thể stream, có layout không re-mount khi navigate". Đây là lý do `getStaticPaths` → `generateStaticParams`, và `params`/`searchParams` trong App Router (từ Next.js 15, Oct 2024) là **Promise** cần `await`. Đây là một trong những breaking changes lớn của v15, cùng với việc `cookies()`, `headers()`, và `draftMode()` cũng trở thành async — một breaking change phản ánh đúng việc rendering model đã đổi, không chỉ là rename

**💻 Minh chứng:**

```tsx
// pages/blog/[slug].js — Pages Router
export async function getStaticPaths() {
  const posts = await getAllPosts();
  return {
    paths: posts.map((p) => ({ params: { slug: p.slug } })),
    fallback: 'blocking',

}

export async function getStaticProps({ params }) {
  const post = await getPostBySlug(params.slug);
  return { props: { post }, revalidate: 60 };
}

export default function BlogPost({ post }) {
  return <article>{post.content}</article>;
}
```

```tsx
  /* ... */
} // 💥 Next.js không gọi
export async function getStaticProps({ params }) {
  /* ... */
} // 💥 Next.js không gọi
```

### Hiểu lầm #4: "Migrate từng route lẻ tẻ, không cần kế hoạch tổng thể, không ảnh hưởng phần còn lại"

🏷️ **Vi phạm:** Chiều 1 (Execution Location — shared code giữa 2 paradigm)

📜 **Nguồn gốc:** Lớp 2 + Lớp 3 (incremental adoption được marketing đúng ở cấp độ ROUTE, bị hiểu sai ở cấp độ SHARED CODE)

🚨 **Câu nói báo động:** _"Next.js support cả 2 router cùng lúc mà, mình cứ migrate dần từng page, không sao."_

**❌ Niềm tin:**

Next.js CHO PHÉP `app/` và `pages/` cùng tồn tại — đây là tính năng chính thức, được document rõ ràng cho incremental migration. Vậy migrate "page by page", mỗi page độc lập, không ảnh hưởng gì tới các page khác.

### Hiểu lầm #5: "`'use client'` nghĩa là code đó CHỈ chạy ở browser, server không render gì"

🏷️ **Vi phạm:** Chiều 1 (Execution Location) — đây là **mẹ của mọi hiểu lầm RSC**, gốc rễ của #6 và #7

📜 **Nguồn gốc:** Lớp 3 — chính cái TÊN "use client" đã gây hiểu sai, vì nó nghe như một lệnh "chạy ở client"

🚨 **Câu nói báo động:** _"Component này có `'use client'` nên server không đụng vào — sao log lại xuất hiện trong terminal?"_

**❌ Niềm tin:**

`'use client'` là một flag phân vùng môi trường thực thi: file có flag này → code chạy ở browser; file không có → code chạy ở server. Hai thế giới tách biệt hoàn toàn.

**✅ Sự thật:**

`'use client'` là **boundary marker cho bundler**, không phải instruction về nơi-chạy. Một Client Component vẫn được **render trên server lần đầu** (để tạo HTML cho SSR), sau đó **render lại trên client** (hydration) để gắn interactivity. `'use client'` chỉ đánh dấu: "module này (và mọi thứ nó import) sẽ được đóng gói gửi xuống client, và được PHÉP dùng hook/browser API" — nó không hề tắt server-render.

**💻 Minh chứng:**

```tsx
// app/components/Counter.tsx
'use client';

import { useState } from 'react';

export default function Counter() {
  console.log('[Counter] render, typeof window =', typeof window);
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount((c) => c + 1)}>Count: {count}</button>;
}
```

Load trang lần đầu, bạn sẽ thấy **2 dòng log ở 2 nơi khác nhau**:

```
# Terminal (server, lúc SSR):
[Counter] render, typeof window = undefined

# Browser console (lúc hydration):
[Counter] render, typeof window = object
```

Component được render **2 lần, ở 2 môi trường**. Lần 1 (server) tạo HTML ban đầu — `'use client'` không ngăn điều này. Nó chỉ đảm bảo: (a) `useState`/`onClick` được phép dùng (chỉ có ý nghĩa ở lần render 2), và (b) JS của component này được gửi xuống client để lần render 2 diễn ra.

**💥 Hệ quả production:**

```tsx
'use client';
export default function Greeting() {
  // 💥 Giá trị KHÁC NHAU giữa lần render server và lần render client
  return <p>Bây giờ: {new Date().toLocaleTimeString()}</p>;
}
```

HTML server gửi xuống ("10:42:01") khác HTML client tính ra khi hydrate ("10:42:03") → **Hydration Mismatch Error**. Người tin "use client = client only" sẽ bối rối: _"code chạy ở client thì so sánh với cái gì mà mismatch?"_ — trong khi đây CHÍNH XÁC là hệ quả của việc component chạy ở cả hai nơi. Đây là root cause của phần lớn lỗi trong **M16.1 (Hydration Errors)**.

### Hiểu lầm #6: "Dính lỗi hooks → cứ thêm `'use client'` lên đầu file, không có hệ quả"

🏷️ **Vi phạm:** Chiều 1 — hệ quả trực tiếp của #5: nếu không hiểu `'use client'` là gì, nó trở thành "nút tắt báo lỗi"

📜 **Nguồn gốc:** Lớp 3

🚨 **Câu nói báo động:** _"Lỗi `useState is not allowed in Server Components` → thêm `'use client'` đầu file, build pass, xong."_

**❌ Niềm tin:**

`'use client'` là fix cục bộ, chỉ ảnh hưởng file đang sửa. Build pass = đã giải quyết xong.

**✅ Sự thật:**

Boundary `'use client'` **lan xuống toàn bộ subtree** mà component đó render — trừ khi children được truyền vào qua `children`/props (composition pattern). Thêm `'use client'` ở một component CHA sẽ kéo TẤT CẢ component CON vào client bundle, dù chúng không cần interactivity gì.

Sau vài tháng, `app/` có cấu trúc App Router nhưng client bundle to như một SPA — trả giá hạ tầng cho App Router (build phức tạp, RSC payload) mà KHÔNG nhận lợi ích (zero-JS subtree). Đây là nguồn dữ liệu chính cho **M11.1 (Bundle Analysis)** và **M6.6 (Phân tích Bundle — Zero JS to client)**.

### Hiểu lầm #7: "RSC luôn nhanh hơn → nên ép TẤT CẢ component thành Server Component"

🏷️ **Vi phạm:** Chiều 1 — overcorrection ngược với #6, biến Server/Client thành "thang điểm performance" thay vì "yêu cầu kỹ thuật"

📜 **Nguồn gốc:** Lớp 3 — messaging "zero bundle size!" bị hiểu thành "luôn ưu tiên tuyệt đối"

🚨 **Câu nói báo động:** _"Component này sao cần `'use client'`? Cố giữ Server Component được không?"_ — khi component đó là nút Like, dropdown, search input...

**❌ Niềm tin:**

Server Component = tốt hơn về mọi mặt (bundle nhỏ, an toàn, nhanh). Mục tiêu nên là tối đa hoá Server Components, tối thiểu hoá Client Components — codebase "nhiều `'use client'`" = codebase tệ.

**✅ Sự thật:**

Server vs Client **không phải một cái nút chỉnh performance** — nó là câu hỏi về YÊU CẦU của component. Cần state/effect/event handler/browser API → **bắt buộc** Client Component, đây không phải lựa chọn. Cố ép một nút Like (cần phản hồi tức thì <100ms) thành Server Component nghĩa là biến mọi click thành một network round-trip đầy đủ.

**💻 Minh chứng:**

```tsx
// ❌ Ép thành Server Component: Like button qua Server Action + full reload
// KHÔNG có 'use client'
import { likePost } from './actions';

export default async function LikeButton({ postId, liked, count }: Props) {
  return (
    <form action={likePost.bind(null, postId)}>
      <button type='submit'>
        {liked ? '❤️' : '🤍'} {count}
      </button>
    </form>
  );
}
```

Mỗi click → submit → re-render TOÀN BỘ route trên server → trả lại HTML mới. Không có optimistic update — độ trễ network lộ rõ trên một hành động lẽ ra phải tức thì.

```tsx
// ✅ Client Component nhỏ + useOptimistic — UX tức thì, vẫn gọi Server Action
'use client';
import { useOptimistic } from 'react';
import { likePost } from './actions';

export function LikeButton({ postId, liked, count }: Props) {
  const [state, addOptimistic] = useOptimistic(
    { liked, count },
    (s, newLiked: boolean) => ({
      liked: newLiked,
      count: s.count + (newLiked ? 1 : -1),
    }),
  );

  return (
    <button
      onClick={() => {
        addOptimistic(!state.liked);
        likePost(postId);
      }}
    >
      {state.liked ? '❤️' : '🤍'} {state.count}
    </button>
  );
}
```

**💥 Hệ quả production:**

Đây là nguồn gốc của phàn nàn _"Next.js app cảm giác nặng hơn SPA"_ — không phải vì RSC chậm, mà vì các tương tác <100ms bị ép vào model request/response, tạo round-trip không cần thiết cho mọi thao tác nhỏ. Nghệ thuật không phải "tối đa hoá RSC" mà "đặt boundary đúng chỗ" — chủ đề chính của **M6.5 (Boundaries)**.

### Hiểu lầm #8: "Set `revalidate` một lần là cache 'đúng mãi mãi', không cần nghĩ lại"

🏷️ **Vi phạm:** Chiều 2 (Cache Lifecycle) + Chiều 3 (Version Stability)

📜 **Nguồn gốc:** Lớp 2 — mental model ISR của Pages Router, nhưng số lớp cache đã nhân lên

🚨 **Câu nói báo động:** _"Mình set `revalidate: 3600` từ lúc setup, xong, không đụng nữa"_ / _"Sao data cũ vẫn hiện sau khi update DB?"_

**❌ Niềm tin:**

`revalidate: N` là cấu hình một lần — đặt theo "dữ liệu này đổi bao lâu một lần", hệ thống tự lo phần còn lại mãi mãi.

**✅ Sự thật:**

`revalidate` là **ngưỡng chấp nhận cũ theo thời gian**, không phải đảm bảo tươi mới. Có cả một **cache hierarchy** (Request Memoization, Data Cache, Full Route Cache, Router Cache phía client) — mỗi tầng có timing riêng. Quan trọng hơn: time-based revalidation nghĩa là **luôn có thể tồn tại độ trễ tới N giây**, kể cả ngay sau khi data vừa đổi. Nếu yêu cầu là "vừa publish phải thấy ngay" (admin preview), `revalidate` thuần KHÔNG đáp ứng được — cần on-demand invalidation.

**💻 Minh chứng:**

```tsx
// app/products/[id]/page.tsx
export const revalidate = 60; // set 1 lần lúc setup

export default async function ProductPage({ params }) {
  const { id } = await params;
  const res = await fetch(`https://api.shop.com/products/${id}`);
  return <ProductDetail product={await res.json()} />;
}
```

Admin sửa giá lúc 10:00:00 → khách (và cả admin refresh trang preview) vẫn thấy giá CŨ tới 10:01:00. Với flash-sale, "tới 60 giây" là khoảng thời gian khách có thể mua với giá sai — vấn đề business, không chỉ "cache hơi cũ".

```ts
// app/actions/update-product.ts
'use server';
import { revalidateTag } from 'next/cache';

export async function updateProductPrice(id: string, newPrice: number) {
  await db.product.update({ where: { id }, data: { price: newPrice } });
  revalidateTag(`product-${id}`); // ✅ invalidate NGAY — đây là việc revalidate: 60 không làm được
}
```

```tsx
const res = await fetch(`https://api.shop.com/products/${id}`, {
  next: { tags: [`product-${id}`], revalidate: 3600 }, // 3600 = "tối đa cũ bao lâu NẾU không ai update"
});
```

**💥 Hệ quả production:**

_"Tại sao data không update?"_ là một trong những bug report phổ biến nhất (chủ đề chính **M7.11**, **M16.2**). Root cause hầu như luôn là: `revalidate` set một lần, không có on-demand invalidation cho mutation, và không ai còn nhớ con số "60" từ đâu ra hay còn hợp lý không.

**⚠️ Context version quan trọng:**

Caching behavior đã thay đổi đáng kể qua các version:

- Next.js 13–14: Fetch cached by default (`force-cache`). `revalidate` là opt-out mechanism.
- Next.js 15+: Fetch **KHÔNG cached by default** (`no-store`). Muốn cache phải explicit opt-in với `cache: 'force-cache'` hoặc `next: { revalidate }`.

### Hiểu lầm #9: "Không hiểu cache → `force-dynamic` toàn bộ cho 'an toàn', mất hết lợi ích Next.js"

🏷️ **Vi phạm:** Chiều 2 — overcorrection ngược với #8: từ "cũ mãi" sang "chậm mãi nhưng đúng"

📜 **Nguồn gốc:** Lớp 3 — phản ứng sợ sau khi từng bị bug như #8

🚨 **Câu nói báo động:** _"Bị bug cache hoài, giờ cứ `force-dynamic` hết cho chắc"_

**❌ Niềm tin:**

Cache là "kẻ thù" — nguồn gốc của bug khó hiểu. Lựa chọn "an toàn" là tắt hết (`force-dynamic` mọi route, `cache: 'no-store'` mọi fetch). Performance là vấn đề "tối ưu sau".

**✅ Sự thật:**

Điều này biến MỌI request thành full server round-trip — kể cả cho content GIỐNG NHAU với hàng nghìn user (blog post, trang marketing). Next.js trở thành "Express app chậm vẽ React" — trả full chi phí SSR (query DB, gọi API, render) cho mỗi request, kể cả với một bài blog publish 6 tháng trước không hề đổi.

**💻 Minh chứng:**

```tsx
// app/blog/[slug]/page.tsx
// ❌ "An toàn": force-dynamic cho blog — content tĩnh, publish xong không đổi
export const dynamic = 'force-dynamic';

export default async function BlogPost({ params }) {
  const { slug } = await params;
  const post = await getPostBySlug(slug); // query DB MỖI request, dù content tĩnh
  return <article>{post.content}</article>;
}
```

Nếu bài viết lên top Hacker News (traffic spike), MỖI request đập vào DB → connection pool exhausted → toàn site sập, không chỉ trang blog.

```tsx
// ✅ Đúng đối tượng: SSG/ISR — build 1 lần, revalidate khi publish mới
export async function generateStaticParams() {
  const posts = await getAllPostSlugs();
  return posts.map(({ slug }) => ({ slug }));
}

export default async function BlogPost({ params }) {
  const { slug } = await params;
  const post = await getPostBySlug(slug);
  return <article>{post.content}</article>;
}
// Publish mới: revalidatePath('/blog') — không cần force-dynamic
```

```tsx
// Cái THỰC SỰ cần force-dynamic: data per-user, real-time
export const dynamic = 'force-dynamic';

export default async function Dashboard() {
  const session = await getSession();
  const liveData = await getLiveMetrics(session.userId);
  return <DashboardView data={liveData} />;
}
```

**Phân biệt mấu chốt:** `force-dynamic` không phải "default an toàn" — nó là công cụ cho ĐÚNG một loại trang (per-request, per-user, real-time). Áp cho toàn site nghĩa là coi trang marketing và dashboard cá nhân hoá như nhau.

**💥 Hệ quả production:**

TTFB tăng vọt toàn site, infra cost tăng tuyến tính theo traffic (không gì serve được từ CDN edge) — cùng traffic đó, với cache đúng, phần lớn gần như miễn phí. Quyết định "đúng correctness, sai kinh tế" mà **M11** và **M7.10** phải gỡ lại.

### Hiểu lầm #10: "Next.js dùng được cho MỌI loại project theo cùng kiểu kiến trúc — landing page và enterprise system như nhau"

🏷️ **Vi phạm:** Cả 3 chiều, nhưng ở cấp độ KIẾN TRÚC, không phải code-level

📜 **Nguồn gốc:** Lớp 3 — messaging "full-stack framework" bị hiểu thành "one architecture fits all"

🚨 **Câu nói báo động:** _"Mình dùng Next.js cho mọi project luôn, kiến trúc set sẵn rồi, cứ thêm feature vào"_

**❌ Niềm tin:**

Vì Next.js "full-stack" (routing, rendering, Server Actions đủ làm mutation), nó có thể là kiến trúc cho MỌI quy mô — từ landing page 5 trang tới SaaS multi-team — theo cùng một cách: "mọi thứ nằm trong `app/`, tổ chức theo route".

**✅ Sự thật:**

"Dùng được" (capability) và "nên cấu trúc/scale như nhau" (architecture) là hai câu hỏi khác nhau. Next.js cung cấp BUILDING BLOCKS — không quyết định: domain logic phức tạp nên nằm ở đâu, route nào team nào own, khi nào "Next.js làm backend luôn" là hợp lý vs khi nào cần service riêng. Một SaaS multi-tenant tổ chức giống landing page (mọi thứ phẳng, `actions.ts` chứa hết business logic) sẽ "chạy được" — cho tới khi team scale qua ~5 người, kiến trúc sụp đổ không phải vì Next.js "không làm được", mà vì kiến trúc không scale theo TEAM.

**💻 Minh chứng:**

```
// ❌ "Next.js architecture" cho landing page, áp cho SaaS đa team
app/
├── dashboard/
│   ├── page.tsx       ← UI + fetch + business logic + validation, tất cả trộn
│   └── actions.ts      ← 800 dòng, mọi domain logic của cả app dồn vào đây
├── billing/
│   └── page.tsx        ← copy-paste pattern từ dashboard
└── lib/
    └── db.ts            ← mọi feature import trực tiếp, không qua layer nào
```

2 team cùng sửa `actions.ts` của 2 feature khác nhau → merge conflict liên tục. Logic "tính billing cycle" rải rác trong `actions.ts` của nhiều route — đổi rule phải tìm-sửa nhiều nơi, dễ sót.

```
// ✅ Kiến trúc tính tới scale theo TEAM (M15.2)
app/
├── (dashboard)/dashboard/page.tsx   ← chỉ UI + compose
├── (dashboard)/billing/page.tsx
└── (dashboard)/settings/page.tsx

features/
├── billing/
│   ├── actions.ts    ← Server Actions, GỌI service layer
│   ├── service.ts     ← business logic, KHÔNG biết gì về Next.js
│   └── schema.ts       ← Zod validation
├── dashboard/{actions.ts, service.ts}
└── settings/{actions.ts, service.ts}
```

`service.ts` không import gì từ `next/*` — lớp này tái dùng được nếu sau cần expose API cho mobile app, hoặc tách microservice — điều cấu trúc "mọi thứ trong route" không cho phép mà không viết lại từ đầu.

**💥 Hệ quả production:**

Chủ đề chính **M15**, nhưng hậu quả xuất hiện sớm hơn nhiều người nghĩ. Một startup 2 người chọn flat structure là HOÀN TOÀN ĐÚNG cho giai đoạn đó — over-engineer sớm cũng là lỗi. Hiểu lầm thực sự không phải "nên dùng kiến trúc nào", mà là **tin rằng đây là quyết định "set once, forever"** — không checkpoint để re-evaluate khi team/scope đổi. Kết quả: "big rewrite" panic, thường trùng đúng lúc traffic/business pressure cao nhất.

### Ngày 1–3: "Migrate nhanh route đầu tiên"

Mid developer A nhận route `/products`. Copy code từ một YouTube tutorial "Next.js 14 App Router" (thực ra tutorial viết cho Next 13, `params` chưa là Promise):

```tsx
// app/products/[id]/page.tsx — tuần 1
export async function getStaticProps({ params }) {
  // ❌ Dead code, không bao giờ được gọi
  return { props: { id: params.id } };
```

Không có cấu hình cache → Next.js cache theo default của version đang dùng (behavior này khác nhau giữa v13/v14/v15 — **Chiều 3**).

**Hiểu lầm kích hoạt:** #8 (không có on-demand invalidation), #9 (`force-dynamic` lan ra toàn site)

## 8. Debugging Scenario — Walkthrough: "Tại Sao Dashboard Hiện Data Cũ?"

### Bug Report

> **Người báo:** Admin  
> **Triệu chứng:** Cập nhật giá sản phẩm trong CMS lúc 09:15. Refresh trang `/dashboard/products` nhiều lần. Đến 09:28 vẫn thấy giá cũ. Sau 09:29 thì thấy giá mới.  
> **Môi trường:** Production, Next.js 15, `revalidate: 60` trên route `/dashboard/products`

### Bước 1 — Định vị sai lệch thời gian

```
09:15 — Admin update giá
09:16 — Admin refresh: vẫn giá cũ     ← có thể chấp nhận (revalidate: 60)
09:17 — Admin refresh: vẫn giá cũ     ← route đã revalidate, sao vẫn cũ?
09:28 — Admin refresh: vẫn giá cũ     ← 13 phút, không phải 1 phút
09:29 — Admin refresh: giá mới ✅
```

Nếu chỉ có `revalidate: 60`, bug lý ra phải fix tại 09:16. Tại sao data cũ tồn tại 13 phút?

### Bước 2 — Trace qua cache hierarchy (Chiều 2)

```
Request 09:16:
→ [Full Route Cache] Stale → trigger background revalidation
   → Revalidation bắt đầu: gọi lại page component, fetch data...
   → Nhưng fetch bên trong có cache riêng:

   fetch('https://pricing-api/products', {
     next: { revalidate: 900 }  // ← 15 phút
   })

   → [Data Cache] pricing-api response còn fresh (chỉ mới 1 phút / 900s)
   → Data Cache KHÔNG gọi pricing-api thật
   → Background revalidation build HTML mới với giá từ Data Cache — VẪN GIÁ CŨ
   → Full Route Cache cập nhật, nhưng chứa giá cũ

Request 09:28:
→ [Full Route Cache] Hit → trả HTML chứa giá cũ (được tính lúc 09:16)

Request 09:29:
→ Data Cache của pricing-api hết hạn (09:15 + 15 phút = 09:30 gần kề)
  → Revalidation lần này gọi pricing-api thật → giá mới
  → Full Route Cache cập nhật với giá mới ✅
```

### Bước 3 — Root cause

```
2 revalidate timer lồng nhau, không được design cùng nhau:
  Full Route Cache:  revalidate: 60  (1 phút)
  Data Cache:        revalidate: 900 (15 phút) — trong fetch bên trong

Khi route revalidate sau 1 phút, nó tái sử dụng Data Cache còn fresh
→ HTML "mới" nhưng data vẫn cũ.
Window bug: 09:16 – 09:30 (cho đến khi Data Cache của pricing-api hết hạn).
```

**Hiểu lầm kích hoạt:** #8 — đặt `revalidate` không nghĩ đến hierarchy, nghĩ "1 layer" khi thực ra có nhiều layer lồng nhau.

**⚠️ Kiểm tra version-specific behavior:**

Nếu bạn đang dùng Next.js 15+, hãy kiểm tra:

1. Fetch có explicit `cache: 'force-cache'` hoặc `next: { revalidate }` không? (Vì default là `no-store`)
2. `cookies()`, `headers()`, `draftMode()` có được `await` không? (Vì chúng là async từ v15)
3. `params`/`searchParams` có được `await` không?

Nếu không, bug "data không update" có thể là do **không có cache nào hoạt động** (v15 default no-store) chứ không phải do cache stale.

## 10. Architecture Notes

### "Quy Tắc Đặt Boundary" — Khi Nào Server, Khi Nào Client

Không phải công thức cứng, nhưng 3 câu hỏi theo thứ tự ưu tiên:

```
1. Component này có cần browser API không?
   (localStorage, geolocation, window, ResizeObserver, WebSocket...)
   → CÓ → Client Component, không thảo luận.

2. Component này có cần phản hồi < 100ms không?
   (click, hover, drag, type, keypress, animation trực quan...)
   → CÓ → Client Component, không thảo luận.

3. Component này chỉ render data, không cần 2 điều trên?
   → Server Component — đây là default trong App Router.
```

**Suy ra (Corollary):** đặt `'use client'` càng gần **leaf** (lá) của component tree càng tốt. Mọi component tổ tiên (ancestor) đều không nên có `'use client'` trừ khi CHÍNH NÓ — không phải con của nó — cần state hoặc effect.

### Khi nào nên xem xét Cache Components (Next.js 16+)

Nếu bạn đang dùng Next.js 16 với `cacheComponents: true`, các quy tắc trên thay đổi:

- `export const revalidate = 60` và `export const dynamic = 'force-dynamic'` **không còn hoạt động** — thay bằng `'use cache'` directive và `cacheLife()`.
- `fetch` caching vẫn hoạt động như một layer riêng, nhưng primary caching mechanism là `use cache`.
- PPR trở thành default behavior, không cần flag.

Điều này ảnh hưởng trực tiếp đến Hiểu lầm #8 và #9 — các pattern `revalidate`/`force-dynamic` đang được dạy ở đây có thể là legacy trong v16+.

---

### "Quy Tắc Chọn Rendering Strategy"

```
Data giống nhau cho mọi user + ít đổi?
→ generateStaticParams + revalidate dài (SSG/ISR)
→ Ví dụ: blog, trang marketing, docs, catalog sản phẩm thường.

Data giống nhau cho mọi user + đổi theo event (admin action)?
→ revalidate dài + revalidateTag khi mutation
→ Ví dụ: landing page edit từ CMS, product page cập nhật khi admin sửa.

Data per-user hoặc cần real-time?
→ force-dynamic + streaming (Suspense)
→ Ví dụ: dashboard cá nhân hoá, feed notification, live stock price.

Data mix — phần static + phần per-user trên cùng trang?
→ PPR (Partial Prerendering) — experimental từ Next.js 14 hoặc layout tĩnh + slot dynamic
→ Ví dụ: trang sản phẩm (static) + badge "Đã mua" (per-user).
```

**Tiêu chí quyết định nhanh (Quick Heuristic):** Nếu 1000 user truy cập cùng một URL mà họ có thể nhận cùng một HTML/response giống nhau, thì trang đó không cần render riêng cho từng user → không nên dùng `force-dynamic`.

```ts
'use server';

import { db } from '@/lib/db';
import { revalidatePath } from 'next/cache';
import { z } from 'zod';

const CreateOrderSchema = z.object({
  productId: z.string(),
  quantity: z.number().min(1),
});

export async function createOrder(input: unknown) {
  // 1. Validation
  const data = CreateOrderSchema.parse(input);

  // 2. Business logic
  const product = await db.product.findUnique({
    where: {
      id: data.productId,
    },
  });

  if (!product) {
    throw new Error('Product not found');
  }

  if (product.stock < data.quantity) {
    throw new Error('Not enough stock');
  }

  const total = product.price * data.quantity;

  // 3. Raw DB query
  const order = await db.order.create({
    data: {
      productId: product.id,
      quantity: data.quantity,
      total,
    },
  });

  // 4. Side effect của Next.js
  revalidatePath('/orders');

  return order;
}
```

## 11. Interview Questions — Junior đến Architect

<details>
<summary><strong>Q1 [Junior]</strong> — Giải thích sự khác nhau giữa Server Component và Client Component. Khi nào dùng cái nào?</summary>

**Điểm cần có trong câu trả lời:**

Server Component chạy trên server, không có state/effect/event handler, có thể `async`, fetch data trực tiếp, không gửi JS xuống client. Client Component (có `'use client'`) được render trên server (SSR để tạo HTML) VÀ render lại trên client (hydration để gắn interactivity) — **không phải "chỉ chạy ở browser"**.

Dùng Client Component khi: cần `useState`, `useEffect`, event handler (`onClick`, `onSubmit`...), hoặc browser API. Dùng Server Component khi: chỉ render data, không cần interactivity — đây là default, không cần khai báo gì.

**Câu trả lời sai thường gặp:** _"Client Component chỉ chạy ở browser, Server Component chỉ chạy ở server"_ — bỏ qua SSR của Client Component, dẫn đến mù hydration mismatch.

**Follow-up tốt để hỏi thêm:** _"Nếu một Server Component render một Client Component qua `import`, điều gì xảy ra với bundler?"_ → Bundler đóng gói Client Component và dependencies của nó để gửi xuống client. Server Component render trên server, truyền kết quả qua serialization boundary.

</details>

<summary><strong>Q2 [Junior]</strong> — Team vừa migrate sang App Router nhưng còn <code>getServerSideProps</code> trong <code>app/</code>. Có vấn đề gì không?</summary>

**Điểm cần có:**

`getServerSideProps`, `getStaticProps`, `getStaticPaths` **không được gọi** trong `app/` directory — chúng là API độc quyền của Pages Router. Next.js không throw error: chúng chỉ là named exports thông thường, bị ignore hoàn toàn. Props nhận vào component sẽ là `undefined`.

Đây chính xác là lý do hiểu lầm này nguy hiểm: **fail silently** — build pass, TypeScript pass, chỉ data là `undefined`.

**Fix:** xóa các export cũ, viết page component thành `async function` và fetch data trực tiếp bên trong.

**Câu trả lời sai thường gặp:** _"Build sẽ throw lỗi ngay"_ — không, đây là điểm mấu chốt.

</details>

<details>
<summary><strong>Q3 [Junior-Mid]</strong> — <code>'use client'</code> được thêm vào một layout component ở cấp cao. Hậu quả là gì?</summary>

**Điểm cần có:**

`'use client'` tạo boundary cho bundler: mọi component được **import bởi** file đó đều bị kéo vào client JavaScript bundle — dù chúng không cần state hay event handler gì. Exception (ngoại lệ) duy nhất: component được truyền qua `children` prop (composition pattern) — vì khi đó chúng được render bởi một Server Component ở tầng trên, không phải bởi component có `'use client'`.

Hậu quả thực tế: bundle size tăng đột ngột, mọi Server Component con bị mất lợi ích zero-JS. Với root layout, điều này ảnh hưởng toàn bộ app.

**Fix:** tách phần cần interactivity thành Client Component nhỏ, layout quay về Server Component. Phát hiện nhanh bằng `grep -rln "'use client'" app/**/layout.tsx`.

</details>

**Q4 [Mid]** — Tại sao đoạn code này gây Hydration Mismatch Error?

```tsx
'use client';
export function CurrentTime() {
  return <p>Bây giờ: {new Date().toLocaleTimeString()}</p>;
}
```

<details>
<summary>Đáp án Q4</summary>

**Điểm cần có:**

Client Component vẫn được render trên server (SSR) để tạo HTML ban đầu.

`Server render` lúc T=10:42:01 → HTML có chuỗi "10:42:01".

`Client hydrate` lúc T=10:42:03 → tính ra "10:42:03"

→ **không khớp** với HTML server → React throw `Hydration Mismatch`.

`'use client'` **không ngăn server render** — nó chỉ đánh dấu component được phép dùng hooks và sẽ được gửi JS xuống client.

**Fix:** render rỗng trên server, set time thật sau hydration:

```tsx
'use client';
import { useState, useEffect } from 'react';

export function CurrentTime() {
  const [time, setTime] = useState(''); // rỗng trên server → không có gì để mismatch

  useEffect(() => {
    // Chỉ chạy trên client, sau khi hydration xong
    setTime(new Date().toLocaleTimeString());
    const id = setInterval(
      () => setTime(new Date().toLocaleTimeString()),
      1000,
    );
    return () => clearInterval(id);
  }, []);

  return <p>Bây giờ: {time || '--:--:--'}</p>;
}
```

</details>

<details>
<summary><strong>Q5 [Mid]</strong> — Route <code>/blog/[slug]</code> có <code>revalidate: 3600</code> (1 giờ), nhưng sau khi publish bài viết mới, người dùng vẫn thấy nội dung cũ ngay cả sau hơn 2 giờ. Hướng debug như thế nào?</summary>

**Điểm cần có:**

Trace qua cache hierarchy theo thứ tự:

**Full Route Cache** — trước tiên xác nhận route có thực sự đang sử dụng Full Route Cache hay không. `revalidate: 3600` có đang được áp dụng đúng không? Lần revalidation gần nhất diễn ra khi nào? Kiểm tra xem route có vô tình opt-out khỏi Full Route Cache bởi `fetch({ cache: 'no-store' })`, `dynamic = 'force-dynamic'`, hoặc các dynamic APIs khác như `headers()`, `cookies()` hay không. Nếu route đã opt-out (không có tác dụng cache nên sẽ không thể bị stale ở đây), thì nguyên nhân không nằm ở Full Route Cache và cần chuyển sang điều tra các lớp cache khác bên dưới.

**Data Cache** — fetch bên trong có `revalidate` riêng lớn hơn 3600 không? Nếu có, thì do route revalidate sau 1 giờ nhưng dùng data từ cache vẫn fresh của fetch bên trong → Nên là HTML mới nhưng content cũ (chính xác là root cause của Debugging Scenario).

**Router Cache (phía client)** — xác nhận "content vẫn cũ sau 2 giờ" được đo như thế nào. User có thực sự reload trang (hard refresh) hay chỉ điều hướng trong ứng dụng bằng `Link`, `router.push()`, hoặc `back/forward` button? Trong trường hợp điều hướng phía client, nội dung có thể đang được phục vụ từ `Router Cache` thay vì `request mới` lên server. Chỉ khi tái hiện được vấn đề bằng một request mới hoàn toàn (reload hoặc mở tab mới), mới nên kết luận nguyên nhân nằm ở `server-side` cache.

```txt
User:
9:00 mở bài viết

9:05 publish content mới

11:00 bấm Back
11:00 bấm vào bài viết lần nữa
```

Người dùng nói:

> "Tôi vẫn thấy content cũ sau 2 giờ."

Nhưng thực tế:

```txt
11:00
↓
Không có request mới lên server
↓
Router Cache trả dữ liệu đã lưu
```

Nghĩa là **"2 giờ" là thời gian người dùng quan sát được**, không phải thời gian dữ liệu đã nằm trong cache server.

Vì vậy khi debug stale content, câu hỏi đầu tiên luôn là:

> "Người dùng thấy dữ liệu cũ sau 2 giờ" có nghĩa là sau 2 giờ và **một request mới**, hay chỉ sau 2 giờ và **vẫn ở trong cùng một phiên điều hướng client-side**?

---

**Câu hỏi chẩn đoán thêm** — đang dùng Next.js phiên bản nào? (Next.js 15 thay đổi một số hành vi caching mặc định so với v14, nên cần loại trừ khác biệt do version. Không caching `no-store` mặc định thay vì cache mặc định như trước). Khi publish bài viết mới, hệ thống có gọi `revalidateTag()` hoặc `revalidatePath()` hay không? Nếu không có cache invalidation chủ động, toàn bộ cơ chế cập nhật nội dung sẽ phụ thuộc vào timer (`revalidate: 3600`), dẫn đến việc người dùng phải chờ tới chu kỳ revalidation tiếp theo mới thấy dữ liệu mới.

> **Giải pháp** — không nên chỉ dựa vào time-based revalidation cho nội dung được cập nhật bởi CMS. Gắn tag cho dữ liệu bài viết và gọi `revalidateTag()` (hoặc `revalidatePath()`) ngay khi publish. Cách này chủ động invalidation Data Cache và Full Route Cache tại thời điểm mutation xảy ra, thay vì chờ hết 1 giờ mới được refresh.

Ví dụ:

```ts
// page.tsx
const post = await fetch(`https://cms/posts/${slug}`, {
  next: {
    tags: [`post:${slug}`],
    revalidate: 3600,
  },
});
```

```ts
// publish action / webhook
import { revalidateTag } from 'next/cache';

export async function publishPost(slug: string) {
  await cms.publish(slug);

  revalidateTag(`post:${slug}`);
}
```

Kết quả:

```txt
❌ Chỉ dùng revalidate: 3600

Publish
    ↓
Chờ tối đa 1 giờ
    ↓
User mới thấy content mới


✅ Dùng revalidateTag()

Publish
    ↓
revalidateTag()
    ↓
Cache bị invalid ngay
    ↓
Request kế tiếp thấy content mới
```

Điểm cần ứng viên nhận ra là: **`revalidate: 3600` là chiến lược "eventually fresh", còn `revalidateTag()` / `revalidatePath()` là chiến lược "fresh ngay sau mutation".** Đây thường là đáp án mong đợi trong các hệ thống CMS, blog, e-commerce admin panel.

</details>

<details>
<summary><strong>Q7 [Mid-Senior]</strong> — Giải thích "serialization boundary" trong RSC. Những loại data nào KHÔNG thể truyền từ Server Component sang Client Component qua props?</summary>

**Điểm cần có:**

Server Component render trên server và gửi kết quả (RSC payload) sang client qua một format đặc biệt. Props truyền qua boundary phải **serializable** — có thể chuyển thành format gửi qua network.

**Không thể truyền:**

- **Function** — bao gồm callback handler thông thường (không thể serialize). _Exception:_ Server Action được serialize như một reference endpoint URL.
- **Class instance** với method
- **Date object thô** — cần `.toISOString()` trước
- **`undefined`** — serialize thành `null` trong transport, dễ gây bug ngầm
- **Symbol, WeakMap, WeakSet**
- **React element chứa Server Component** bên trong

**Có thể truyền:** string, number, boolean, null, array, plain object (JSON-serializable)

```tsx
// Server Component
async function ServerParent() {
  const product = await db.product.findFirst({ where: { id } });
  return (
    <ClientChild
      name={product.name} // ✅ string
      price={product.price} // ✅ number
      createdAt={product.createdAt.toISOString()} // ✅ string — KHÔNG phải Date object
      onSave={handleSave} // ❌ function không serialize được
      instance={new ProductModel(product)} // ❌ class instance
    />
  );
}
```

<summary><strong>Q8 [Senior]</strong> — Team đang migrate codebase Pages Router sang App Router. Auth lib hiện tại expose <code>getUserFromRequest(req: NextApiRequest)</code>. Kế hoạch migrate như thế nào để tránh dual maintenance?</summary>

**Điểm cần có:**

Chiến lược: tách auth lib thành 2 lớp — core logic (framework-agnostic) và adapter cho từng paradigm.

```ts
// lib/auth/core.ts — Pure business logic, KHÔNG import gì từ next/*
// Testable với unit test thuần Jest, không cần mock
export async function verifyAndDecodeToken(
  token: string | undefined,
): Promise<User | null> {
  if (!token) return null;
  try {
    return await jwtVerify(token, process.env.JWT_SECRET!);
  } catch {
    return null;
  }
}
```

```ts
// lib/auth/server.ts — Adapter cho App Router
import { cookies } from 'next/headers';
import { verifyAndDecodeToken } from './core';

export async function getCurrentUser(): Promise<User | null> {
  const cookieStore = await cookies();
  return verifyAndDecodeToken(cookieStore.get('token')?.value);
}
```

```ts
// lib/auth/pages.ts — Adapter cho Pages Router (xóa khi migrate xong)
import type { NextApiRequest } from 'next';
import { verifyAndDecodeToken } from './core';

export async function getUserFromRequest(
  req: NextApiRequest,
): Promise<User | null> {
  return verifyAndDecodeToken(req.cookies.token);
}
```

**Lợi ích:** mọi thay đổi logic xác thực (rotate secret, thêm claim, đổi JWT library) chỉ sửa `core.ts`. Khi migrate xong, xóa `pages.ts` — không sửa gì ở `core.ts`. Business logic không bao giờ phải tồn tại song song ở 2 nơi.

<summary><strong>Q9 [Senior]</strong> — Giải thích Request Memoization trong Next.js. Khi nào hữu ích, khi nào không hoạt động như bạn nghĩ?</summary>

**Điểm cần có:**

Trong cùng một server request, nếu nhiều Server Component gọi `fetch()` với **cùng URL + cùng options**, Next.js chỉ thực sự thực hiện network call 1 lần — lần sau trả từ in-memory. Cơ chế này dùng React's `cache()` và Next.js's `fetch` monkey-patching.

```tsx
// RootLayout và DashboardPage cùng cần user — không double-fetch
async function RootLayout({ children }) {
  const user = await getCurrentUser(); // fetch /api/me — thực sự gọi network
  return <html>...</html>;
}

async function DashboardPage() {
  const user = await getCurrentUser(); // cùng fetch /api/me → được memoize, trả từ memory
  return <div>Hello {user.name}</div>;
}
```

**Khi KHÔNG hoạt động:**

- Fetch method là `POST` (không được deduplicate)
- Dùng `axios`, `got`, hoặc bất kỳ HTTP client nào khác — Next.js chỉ patch **native `fetch`**
- Fetch có `cache: 'no-store'` (opt out deduplication)
- **Memoization reset sau mỗi request** — không persist qua request khác (khác với Data Cache)

**Fix cho non-fetch function** (ví dụ: query DB trực tiếp):

```ts
import { cache } from 'react';

export const getCurrentUser = cache(async () => {
  return db.user.findFirst({ where: { id: getSessionId() } });
});
// Mỗi server request: getCurrentUser() gọi nhiều lần → DB query chỉ 1 lần
```

</details>

<summary><strong>Q10 [Senior-Architect]</strong> — Bạn review kiến trúc một Next.js SaaS: mọi business logic nằm trong <code>app/</code> route handlers và Server Actions, không có service layer. App đang ở 5.000 users, dự kiến scale tới 500.000. Bạn raise concern gì và đề xuất roadmap refactor thế nào?</summary>

**Điểm cần có:**

**Concerns kỹ thuật:**

Business logic trong route/action không testable với unit test — phụ thuộc vào Next.js internals (`next/cache`, `next/headers`). Nếu sau này cần expose API cho mobile app hoặc tách microservice, phải rewrite lại logic (vì hiện nó coupled với framework layer). Breaking change trong Next.js buộc refactor cả business logic lẫn infra cùng lúc.

**Concerns về team:**

Nhiều developer đụng vào cùng `actions.ts` của shared domain → merge conflict tăng tuyến tính. Không có clear ownership: _"ai chịu trách nhiệm khi billing logic thay đổi — nó nằm rải rác trong 3 file `actions.ts` khác nhau?"_

**Roadmap đề xuất — không "big rewrite":**

```
Phase 1 (2 tuần, không break production):
→ Tạo features/ directory
→ Với MỌI feature mới, viết service layer trong features/ TRƯỚC
→ Server Action gọi service — không implement logic trực tiếp
→ Quy tắc: actions.ts không được import prisma/db trực tiếp

Phase 2 (song song feature work, ~3 tháng):
→ Với mỗi actions.ts > 100 dòng: extract business logic sang features/[domain]/service.ts
→ Thêm unit test cho service layer — không cần mock Next.js
→ Ưu tiên domain quan trọng nhất (billing, auth) trước

Phase 3 (chỉ khi cần — mobile API, microservice):
→ Service layer đã "Next.js-free" → wrap bằng REST/tRPC mà không rewrite logic
→ Zero coupling: thay Next.js version hay thậm chí framework → service layer không đổi
```

**Không đề xuất:** _"Rewrite toàn bộ kiến trúc ngay"_ — quá rủi ro ở giai đoạn growth. 5k users đang tăng là thời điểm tệ nhất để freeze feature dev.

**Câu phản biện hay gặp:** _"Tại sao không tách microservice ngay?"_ → Ở 5k users, overhead của microservice (network latency, distributed tracing, eventual consistency, devops complexity) hiếm khi justified. Service layer trong monorepo cho phép scale team và logic rõ ràng, giữ deployment đơn giản cho đến khi **thực sự** cần tách.

</details>

Phân loại từng component: **(a) nên là Server Component, (b) phải là Client Component, (c) đang sai — cần refactor:**

```tsx
// Component A: SearchBar với debounce
'use client';
import { useState } from 'react';
export function SearchBar({ onSearch }: { onSearch: (q: string) => void }) {
  const [query, setQuery] = useState('');
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    setQuery(e.target.value);
    onSearch(e.target.value);
  };
  return (
    <input
      value={query}
      onChange={handleChange}
      placeholder='Tìm kiếm...'
    />
  );
}
```

**Component A — (b) + cần xem xét thêm:** `'use client'` đúng vì cần `useState`. Nhưng prop `onSearch` là function — nếu parent là Server Component, đây là serialization boundary problem (function không serialize được). **Component D — (c) sai:** `'use client'` ở page-level kéo `SummaryChart` và `RecentActivity` vào bundle không cần thiết. Fix: giữ page là Server Component, tách `DateRangePicker` thành Client Component riêng — hai component kia không bị ảnh hưởng.

### Exercise 2 — Cache Audit [~45 phút]

Xác định cache layer đang active, vấn đề tiềm ẩn, và đề xuất fix:

```tsx
// app/shop/[category]/page.tsx
export const revalidate = 1800; // 30 phút

export default async function CategoryPage({
  params,
}: {
  params: Promise<{ category: string }>;
}) {
  const { category } = await params;

  const [products, banner] = await Promise.all([
    fetch(`https://api.shop.com/products?category=${category}`, {
      next: { revalidate: 300 }, // 5 phút
    }).then((r) => r.json()),

    fetch(`https://cms.shop.com/banners/${category}`, {
      cache: 'no-store', // luôn fresh
    }).then((r) => r.json()),
  ]);

  return (
    <CategoryView
      products={products}
      banner={banner}
    />
  );
}
```

**Câu hỏi:**

- `revalidate: 1800` (route) vs `revalidate: 300` (fetch products) — layer nào quyết định?
- `cache: 'no-store'` trên banner fetch ảnh hưởng gì đến Full Route Cache của cả route?
- Admin thay banner lúc 10:00 → khách thấy banner mới khi nào?
- Products có thực sự được cache 5 phút như dự định không?

<details>
<summary>Đáp án</summary>

**1. `revalidate: 1800` (route) vs `revalidate: 300` (fetch products) — Layer nào quyết định?**

| Layer           | `revalidate: 1800` trên route | `revalidate: 300` trên fetch |
| --------------- | ----------------------------- | ---------------------------- |
| **Cache Layer** | Full Route Cache              | Data Cache                   |
| **Scope**       | HTML + RSC Payload toàn route | Response của fetch products  |
| **Ý nghĩa**     | Route re-render sau 30 phút   | Products re-fetch sau 5 phút |

**Tuy nhiên**, vì route có `cache: 'no-store'` trên banner fetch → **Full Route Cache bị opt-out toàn bộ** — route này thực chất chạy như `force-dynamic`, dù không khai báo. Mỗi request rebuild toàn trang. `revalidate: 1800` trên route **không còn hiệu lực** vì route không còn được static render.

---

**2. `cache: 'no-store'` trên banner fetch ảnh hưởng gì đến Full Route Cache?**

**Full Route Cache bị opt-out toàn bộ route.**

Theo Next.js docs, một component trở thành **dynamic** khi sử dụng:

- `cookies()`, `headers()`, `connection()`, `draftMode()`
- `searchParams` prop
- `unstable_noStore`
- `fetch` với `{ cache: 'no-store' }`

Chỉ cần **một fetch `no-store` trong route** là đủ để opt-out Full Route Cache — route chạy như `force-dynamic`, dù không khai báo. Mỗi request rebuild toàn trang server-side.

---

**3. Admin thay banner lúc 10:00 → khách thấy banner mới khi nào?**

**Ngay lập tức** (request tiếp theo). Vì `no-store` đảm bảo fetch luôn đi thẳng đến CMS, không qua Data Cache.

**Nhưng chi phí:** Mọi request đều hit server, không ai được serve từ CDN edge. TTFB cao hơn đáng kể.

---

**4. Products có thực sự được cache 5 phút như dự định không?**

**Products được cache ở Data Cache, nhưng hiệu quả khác với kỳ vọng:**

| Khía cạnh            | Thực tế                                                                                                                                              |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Data Cache**       | `revalidate: 300` vẫn hoạt động — products được lưu trong Data Cache                                                                                 |
| **Full Route Cache** | Đã bị opt-out nên không có static HTML/RSC                                                                                                           |
| **Hiệu quả thực tế** | Mỗi request vẫn trigger server render, nhưng products lấy từ Data Cache (không gọi API) trong vòng 5 phút. Sau 5 phút, API products mới được gọi lại |

Products **có được cache**, nhưng ở Data Cache chứ không phải Full Route Cache. Điều này khác với kỳ vọng "cả trang được cache 5 phút".

---

**5. Fix caching — Có thể fix lại với phương án sau đây không ?**

**Static Shell** là phần HTML/RSC Payload của một route được **prerender** (tạo sẵn) ở **build time** hoặc **ISR time** — tức là nội dung "cứng" không đổi giữa các request, được cache ở CDN, serve ngay lập tức cho user mà không cần chạy server.

**Mental Model**

```
┌─────────────────────────────────────────┐
│           STATIC SHELL                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ Header  │  │ Sidebar │  │ Product │  │  ← Prerender ở build time
│  │ (logo)  │  │ (menu)  │  │  Grid   │  │     Cache ở CDN
│  └─────────┘  └─────────┘  └─────────┘  │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │     🕳️ DYNAMIC HOLE             │    │
│  │     (Suspense fallback)         │    │  ← Stream ở request time
│  │     ┌─────────────────────┐     │    │     (banner, user data...)
│  │     │  <BannerSkeleton /> │     │    │
│  │     └─────────────────────┘     │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
              ↓
         [STREAMING]
              ↓
┌─────────────────────────────────────────┐
│  Dynamic content thay thế fallback      │
│  (banner mới nhất từ CMS)               │
└─────────────────────────────────────────┘
```

**Tại sao cần Static Shell?**

| Vấn đề                            | Giải pháp                              |
| --------------------------------- | -------------------------------------- |
| User mở trang → phải đợi API chậm | Shell hiển thị ngay lập tức            |
| API fail → trang trắng            | Shell vẫn render, chỉ dynamic part lỗi |
| TTFB cao                          | Shell serve từ CDN edge (cache)        |
| Layout shift khi content load     | Skeleton giữ chỗ trước                 |

**Đáp án đề xuất**: Tách banner thành Suspense slot riêng

> **Context:** Next.js 14, App Router, chưa bật PPR (config mặc định)

```tsx
// app/shop/[category]/page.tsx
export const revalidate = 300;

export default async function CategoryPage({ params }) {
  const { category } = await params;
  const products = await fetch(`...products?category=${category}`, {
    next: { revalidate: 300, tags: [`category-${category}`] },
  }).then((r) => r.json());

  return (
    <div>
      <Suspense fallback={<BannerSkeleton />}>
        <CategoryBanner category={category} />
      </Suspense>
      <ProductGrid products={products} />
    </div>
  );
}
```

**Đánh giá:**

✅ **Nhìn thì có vẻ đúng:**

**Nguyên tắc:** Tách phần dynamic (`no-store`) ra khỏi shell static. Đây là pattern chính thức được khuyến nghị trong Next.js để tránh "drag down" cả route.

**Cơ chế:** `<CategoryBanner />` là Server Component riêng, khi được wrap trong `<Suspense>`, nó sẽ stream riêng biệt. Phần products (static) được cache ở Full Route Cache, còn banner (dynamic) được fetch fresh mà không ảnh hưởng đến shell.

⚠️ **Nhưng đây là cái bẫy:** `Suspense` chỉ giúp **streaming** (gửi phần static trước, dynamic sau), chứ **không giúp caching** ở Full Route Cache level. Next.js 14 quyết định static/dynamic ở **Route level**, không phải Component level.

❌ **Kết luận — vẫn sai:** Dù bọc `<CategoryBanner />` trong `<Suspense>`, vì nó vẫn là **Server Component** trong cùng route và dùng `cache: 'no-store'`, toàn bộ route vẫn bị đánh dấu dynamic. Static shell bị "drag down" hoàn toàn.

> **Tóm lại:** Suspense slot chỉ giúp UI không bị block, không giúp route thoát khỏi dynamic mode. **Context (Next.js 14, không PPR) quyết định kết quả.**

---

**Fix tối ưu: Tách dynamic hoàn toàn ra khỏi static shell**

**Nguyên tắc:** Phần dynamic (`no-store`) phải được tách ra khỏi route để route chính vẫn static.

**Option A: PPR (Partial Prerendering) — Next.js 15+**

> **Context:** Next.js 15+, đã bật PPR. Đây là cách **gần giống nhất** với đáp án đề xuất, nhưng **đúng** nhờ PPR.

**Bước 1: Bật PPR**

```js
// next.config.js
const nextConfig = {
  experimental: {
    ppr: true, // Bật PPR toàn app
  },
};

module.exports = nextConfig;
```

Hoặc per-route:

```tsx
// app/shop/[category]/page.tsx
export const experimental_ppr = true; // Chỉ bật cho route này
```

```
⚠️ Lưu ý quan trọng: PPR chỉ hoạt động khi:
- Next.js 15+ với `experimental_ppr = true` (hoặc Next.js 16 với
`cacheComponents: true`)
- Component dynamic được wrap trong `<Suspense>` với fallback
- Build output phải show `◐` (partial prerender) thay vì `○` (static)
hoặc `λ` (dynamic)
```

**Bước 2: Code giống đáp án đề xuất, nhưng hoạt động đúng**

```tsx
// app/shop/[category]/page.tsx
export const revalidate = 300;
// export const experimental_ppr = true  // Nếu chưa bật global

export default async function CategoryPage({ params }) {
  const { category } = await params;

  const products = await fetch(
    `https://api.shop.com/products?category=${category}`,
    { next: { revalidate: 300 } },
  ).then((r) => r.json());

  return (
    <div>
      {/* Static shell được prerender ở build time */}
      <ProductGrid products={products} />

      {/* Dynamic hole: stream ở request time, KHÔNG drag down static shell */}
      <Suspense fallback={<BannerSkeleton />}>
        <CategoryBanner category={category} />
      </Suspense>
    </div>
  );
}
```

```tsx
// app/shop/[category]/CategoryBanner.tsx
export default async function CategoryBanner({ category }) {
  // Trong PPR, component này được đánh dấu là dynamic hole
  // và được stream riêng biệt, không làm dynamic cả route
  const banner = await fetch(`https://cms.shop.com/banners/${category}`, {
    cache: 'no-store',
  }).then((r) => r.json());

  return <BannerView banner={banner} />;
}
```

**Build output:** `◐` (partial prerender) — static shell + dynamic hole

**Option B: Client Component (Không cần PPR, Next.js 14/15 đều chạy)**

> **Context:** Mọi version Next.js. Tách dynamic ra khỏi route hoàn toàn bằng cách đẩy xuống client.

```tsx
// app/shop/[category]/page.tsx
export const revalidate = 300;

export default async function CategoryPage({ params }) {
  const { category } = await params;

  const products = await fetch(
    `https://api.shop.com/products?category=${category}`,
    { next: { revalidate: 300, tags: [`category-${category}`] } },
  ).then((r) => r.json());

  return (
    <div>
      {/* Banner là Client Component → fetch ở client, 
      không ảnh hưởng route cache */}
      <BannerClient category={category} />
      <ProductGrid products={products} />
    </div>
  );
}
```

```tsx
// components/BannerClient.tsx
'use client';

import { useEffect, useState } from 'react';

export default function BannerClient({ category }: { category: string }) {
  const [banner, setBanner] = useState(null);

  useEffect(() => {
    fetch(`https://cms.shop.com/banners/${category}`, { cache: 'no-store' })
      .then((r) => r.json())
      .then(setBanner);
  }, [category]);

  if (!banner) return <BannerSkeleton />;
  return <BannerView banner={banner} />;
}
```

**Build output:** `○` (static) — route hoàn toàn static, banner fetch ở client

---

**Trade-off: Option A (PPR) vs Option B (Client Component)**

| Tiêu chí                 | Option A: PPR                   | Option B: Client Component     |
| ------------------------ | ------------------------------- | ------------------------------ |
| **Next.js version**      | 15+                             | 14+                            |
| **Config cần thiết**     | Bật PPR (global hoặc per-route) | Không cần                      |
| **Static Shell**         | ✅ Có (prerender ở build time)  | ✅ Có (prerender ở build time) |
| **Banner render**        | Server stream (SSR)             | Client fetch (CSR)             |
| **Banner Freshness**     | ✅ Real-time                    | ✅ Real-time                   |
| **SEO (banner content)** | ✅ Có trong HTML                | ❌ Không có trong HTML ban đầu |
| **Layout shift**         | ⚠️ Có thể (skeleton → content)  | ⚠️ Có thể (skeleton → content) |
| **CDN Cache**            | ✅ Static shell cache           | ✅ Toàn trang cache            |
| **Build output**         | `◐`                             | `○`                            |
| **Độ phức tạp**          | Medium (cần hiểu PPR)           | Low (đơn giản)                 |

**Khi nào chọn gì?**

- **Muốn banner có trong HTML (SEO), dùng Next.js 15+:** → **Option A (PPR)**
- **Muốn đơn giản, không phụ thuộc version/config:** → **Option B (Client Component)**
- **Luôn verify:** Chạy `next build` và kiểm tra build output — `○` (static) hoặc `◐` (partial) là đúng, `λ` (dynamic) là sai

**Option C: Route Handler + Client fetch (Nâng cao)**

```tsx
// app/api/banner/[category]/route.ts
export const dynamic = 'force-dynamic';
// Tác dụng: Bắt Next.js không prerender route này ở build time
// Nếu không có dòng này, Next.js có thể tự động phát hiện:
// "À, route này chỉ fetch data, không dùng cookies/headers → static được"
// Khi đó ở Build time: Next.js gọi API 1 lần, cache kết quả vào Route Handler Cache
// Dẫn tới ở Request time: serve cache → không fresh

export async function GET(
  request: Request,
  { params }: { params: Promise<{ category: string }> },
) {
  const { category } = await params;
  const banner = await fetch(`https://cms.shop.com/banners/${category}`, {
    cache: 'no-store',
    // Tác dụng: Bắt Next.js không cache response này vào Data Cache
    // Nếu không có cache: 'no-store', fetch tự động cache response vào Data Cache (default behavior của Next.js)
    // Dù Route Handler chạy mỗi request, nhưng fetch bên trong vẫn lấy từ cache → không fresh
  }).then((r) => r.json());

  return Response.json(banner);
}
```

```ts
export async function verifyAndDecodeToken(
  token: string | undefined,
): Promise<User | null> {
  if (!token) return null;
  try {
    return await jwtVerify(token, process.env.JWT_SECRET!);
  } catch {
    return null;
  }
}
```

**Câu 2: `CheckoutPage` là Server hay Client Component sau khi migrate? Tại sao?**

**Trả lời:** **CẢ HAI** — nhưng ở **2 component khác nhau**, không phải 1 component như Pages Router.

| Phần                                                                        | Loại             | Lý do                                                           |
| --------------------------------------------------------------------------- | ---------------- | --------------------------------------------------------------- |
| **Shell** (layout, summary, form structure)                                 | Server Component | Fetch data, render static structure                             |
| **Form interactivity** (shipping selection, voucher input, payment, submit) | Client Component | Cần `useState`, `useEffect`, event handlers, gọi API khi submit |

**Tại sao không thể là 1 component?**

Pages Router: `CheckoutPage` nhận props từ `getServerSideProps` → render 1 lần ở server → hydrate ở client → dùng `useState`/`useEffect`.

App Router: Không có `getServerSideProps`. Nếu viết `async function CheckoutPage()` để fetch data → **không thể dùng `useState`** (Server Component không có hooks). Nếu thêm `'use client'` để dùng hooks → **không thể `async`** để fetch data trực tiếp.

> **Hiểu lầm liên quan:** #1 (Next.js = React + Router), #5 (`'use client'` = chỉ browser), #6 (thêm `'use client'` không hệ quả).

```tsx
// app/checkout/CheckoutForm.tsx — Client Component (CHỈ phần này có 'use client')
'use client';

import { useState } from 'react';
import { useRouter } from 'next/navigation';
import { checkoutAction } from './actions';

interface Props {
  user: User;
  cart: Cart;
  shippingOptions: ShippingOption[];
}

export function CheckoutForm({ user, cart, shippingOptions }: Props) {
  const [selectedShipping, setSelectedShipping] = useState(
    shippingOptions[0]?.id,
  );
  const [voucherCode, setVoucherCode] = useState('');
  const [voucherApplied, setVoucherApplied] = useState(false);
  const [paymentMethod, setPaymentMethod] = useState('cod');
  const [agreed, setAgreed] = useState(false);
  const [submitting, setSubmitting] = useState(false);
  const router = useRouter();

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    if (!agreed) return;

    setSubmitting(true);
    try {
      const result = await checkoutAction({
        cartId: cart.id,
        shippingMethodId: selectedShipping,
        voucherCode: voucherApplied ? voucherCode : undefined,
        paymentMethod,
      });

      if (result.success) {
        router.push(`/checkout/success?order=${result.orderId}`);
      } else {
        alert(result.error);
      }
    } finally {
      setSubmitting(false);
    }
  };

  return (
    <form
      onSubmit={handleSubmit}
      className='space-y-4'
    >
      {/* Shipping */}
      <div>
        <h3>Phương thức vận chuyển</h3>
        {shippingOptions.map((opt) => (
          <label
            key={opt.id}
            className='block'
          >
            <input
              type='radio'
              name='shipping'
              value={opt.id}
              checked={selectedShipping === opt.id}
              onChange={() => setSelectedShipping(opt.id)}
            />
            {opt.name} — {opt.price.toLocaleString('vi-VN')} đ
          </label>
        ))}
      </div>

      {/* Voucher */}
      <div>
        <h3>Mã giảm giá</h3>
        <input
          value={voucherCode}
          onChange={(e) => setVoucherCode(e.target.value)}
          placeholder='Nhập mã'
        />
        <button
          type='button'
          onClick={() => setVoucherApplied(true)}
          disabled={!voucherCode}
        >
          Áp dụng
        </button>
      </div>

      {/* Payment */}
      <div>
        <h3>Thanh toán</h3>
        <select
          value={paymentMethod}
          onChange={(e) => setPaymentMethod(e.target.value)}
        >
          <option value='cod'>COD</option>
          <option value='bank'>Chuyển khoản</option>
          <option value='card'>Thẻ tín dụng</option>
        </select>
      </div>

      {/* Terms + Submit */}
      <label className='block'>
        <input
          type='checkbox'
          checked={agreed}
          onChange={(e) => setAgreed(e.target.checked)}
        />
        Tôi đồng ý với điều khoản
      </label>

      <button
        type='submit'
        disabled={!agreed || submitting}
        className='bg-blue-500 text-white px-4 py-2 rounded'
      >
        {submitting ? 'Đang xử lý...' : 'Đặt hàng'}
      </button>
    </form>
  );
}
```

```ts
// app/checkout/actions.ts — Server Action
'use server';

import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';
import { getCurrentUser } from '@/lib/auth/server';
import { createOrderService } from '@/features/orders/service';

export async function checkoutAction(input: CheckoutInput) {
  const user = await getCurrentUser();
  if (!user) throw new Error('Unauthorized');

  // Business logic ở service layer — KHÔNG ở đây
  const result = await createOrderService({
    ...input,
    userId: user.id,
  });

  if (result.success) {
    // Bust cache: giỏ hàng đã clear + đơn mới tạo → 2 path này stale
    // Invalidate Full Route Cache cho các path liên quan
    // Lý do: sau khi tạo order thành công, data ở 2 path này đã thay đổi:
    // - /checkout: giỏ hàng đã clear → cần hiển thị giỏ trống
    // - /orders: có đơn hàng mới → cần hiển thị trong list
    // Không gọi revalidatePath → user quay lại vẫn thấy data cũ từ cache
    revalidatePath('/checkout');
    revalidatePath('/orders');
  }

  return result;
}
```

```ts
// features/orders/service.ts — Pure business logic
export async function createOrderService(input: CreateOrderInput) {
  // Validation
  const cart = await db.cart.findUnique({ where: { id: input.cartId } });
  if (!cart || cart.userId !== input.userId) {
    return { success: false, error: 'Giỏ hàng không hợp lệ' };
  }

  // Calculate
  const shipping = await db.shippingMethod.findUnique({
    where: { id: input.shippingMethodId },
  });
  let total = cart.total + shipping.price;

  // Apply voucher
  if (input.voucherCode) {
    const voucher = await db.voucher.findUnique({
      where: { code: input.voucherCode, used: false },
    });
    if (voucher && voucher.expiresAt > new Date()) {
      total -= voucher.discount;
    }
  }

  // Create order
  const order = await db.order.create({
    data: {
      userId: input.userId,
      items: cart.items,
      shippingMethodId: shipping.id,
      paymentMethod: input.paymentMethod,
      total,
      status: 'PENDING',
    },
  });

  // Clear cart
  await db.cart.update({
    where: { id: cart.id },
    data: { items: [], total: 0 },
  });

  return { success: true, orderId: order.id };
}
```

**Câu 4: Middleware có thể thay thế redirect check không?**

**Trả lời:** **Có** — và thường **nên dùng** cho pattern "check auth → redirect".

**So sánh 2 cách:**

| Cách                                  | Vị trí          | Ưu điểm                          | Nhược điểm                                |
| ------------------------------------- | --------------- | -------------------------------- | ----------------------------------------- |
| **Inline trong page** (như code trên) | `page.tsx`      | Rõ ràng, dễ đọc                  | Lặp lại ở mọi page cần auth               |
| **Middleware**                        | `middleware.ts` | Một nơi, áp dụng cho nhiều route | Chạy trước mọi request, có thể over-match |

**Code middleware:**

```ts
// middleware.ts
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const token = request.cookies.get('token')?.value;

  // Các route cần auth
  const protectedPaths = ['/checkout', '/orders', '/profile'];
  const isProtected = protectedPaths.some((path) =>
    request.nextUrl.pathname.startsWith(path),
  );

  if (isProtected && !token) {
    return NextResponse.redirect(new URL('/login', request.url));
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/checkout/:path*', '/orders/:path*', '/profile/:path*'],
};
```

**Khi nào dùng gì?**

| Scenario                                                           | Khuyến nghị                                                                              |
| ------------------------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| Auth check đơn giản (có token không)                               | Middleware                                                                               |
| Auth check phức tạp (verify token, check role, check subscription) | Inline trong page hoặc Server Component                                                  |
| Cần data user để render (ví dụ: hiển thị "Xin chào, [tên]")        | Inline — middleware không truyền data xuống page                                         |
| Route cần auth + cần data user                                     | Cả hai: middleware redirect nhanh nếu không token, page fetch user chi tiết nếu có token |

> **Lưu ý:** Trong code `page.tsx` ở Câu 3, tôi vẫn giữ `if (!user) redirect('/login')` — để minh họa pattern inline. Nếu dùng middleware, có thể bỏ check này (vì middleware đã đảm bảo user có token). Nhưng vẫn nên giữ `getCurrentUser()` để lấy data user cho render.

## 13. Summary

### 3 Điều Quan Trọng Nhất Từ Bài Này

**1. Nguồn gốc của hiểu lầm quan trọng hơn bản thân hiểu lầm.**

Không ai "sai" khi dùng `getServerSideProps` trong App Router — họ đang áp một pattern từ kỷ nguyên khác. Khi bạn thấy mình hoặc đồng nghiệp làm điều gì đó "có vẻ sai", câu hỏi đầu tiên không phải _"tại sao làm vậy?"_ mà là _"pattern này đúng ở context nào, và tại sao context đó không còn apply nữa?"_ — điều này chuyển code review từ chỉ trích thành chuyển giao mental model.

**2. Lăng Kính 3 Chiều là công cụ kiểm tra chủ động, không phải checklist thụ động.**

Với MỌI đoạn code Next.js bạn viết hoặc review, 30 giây hỏi 3 câu: Chiều 1 — _đoạn code này chạy ở đâu, khi nào?_ Chiều 2 — _cache layer nào active, hết hạn thế nào, ai invalidate?_ Chiều 3 — _behavior này còn đúng ở version đang dùng không?_ Có thể ngăn nhiều giờ debug sau đó.

**3. Hiểu lầm về caching là loại "động" — cần re-verify theo version.**

Khác với các nhóm khác (sai về execution model, về migration), caching behavior đã thay đổi **mặc định** qua từng major version. Điều đúng ở Next.js 13 về fetch caching có thể sai ở Next.js 15. Bất cứ khi nào upgrade version hoặc đọc tutorial về cache, hỏi ngay: _"tutorial này viết cho version nào?"_
