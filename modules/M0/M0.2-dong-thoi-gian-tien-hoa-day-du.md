# M0.2: Next.js Version Timeline & Era Classification

> **Level:** Fresher | **Module:** M0 | **Prerequisite:** M0.1 | **Unlocks:** M0.3
> **Đọc xong bài này, bạn có thể:** Nhìn vào bất kỳ codebase Next.js nào và xác định được đang ở era nào trong vòng 5 phút.

---

## 1. Evolution Context — Tại Sao Bài Này Tồn Tại

### Pain Point

Bạn join team mới, clone repo, mở file đầu tiên thấy `getServerSideProps`. Google ra "deprecated". Mở PR để migrate. Lead review xong comment: *"Đừng đụng vào đó, app đang chạy prod, đụng là chết."*

Hoặc ngược lại: copy tutorial App Router từ blog vào codebase v12, build fail với lỗi không hiểu tại sao, mất 2 tiếng debug.

Root cause của cả hai: **bạn không biết mình đang đứng ở era nào.**

Next.js thay đổi nhanh hơn hầu hết framework frontend. Từ 2016 đến 2025, có ít nhất **5 paradigm shift lớn** — mỗi shift thay đổi cách fetch data, render page, bundle code. Không có mental model về era, bạn sẽ:

- Dùng API Pages Router trong App Router project → runtime crash hoặc silent fail
- Copy tutorial v14 vào codebase v11 → build fail, error message vô nghĩa
- Upgrade version không đọc era changes → production incident lúc 2 giờ sáng
- Không biết khi nào migrate, khi nào ở lại → tech debt chồng chất

### Cách Cũ — Blog Chasing

Trước khi community có era classification rõ ràng, "cập nhật kiến thức" về Next.js = subscribe blog Vercel, đọc từng release note, tự ghép mental model.

Vấn đề với approach này:

- Release note viết cho người đã có context. Fresher đọc không hiểu *tại sao* feature X quan trọng.
- Ranh giới beta vs stable mờ nhạt. App Router beta (v13.0) và stable (v13.4) có nhiều breaking change. Ai không theo dõi sát bị kẹt.
- Tutorial community lag 3–6 tháng. "Hướng dẫn Next.js mới nhất" trên Medium có thể đang dùng API đã deprecated.

### Tại Sao Nó Thất Bại

Không có **codebase fingerprint** rõ ràng. Mở repo: thấy `pages/` vs `app/`, thấy `next.config.js` vs `next.config.ts`, thấy `middleware.ts` ở root — không biết đây là Next.js của năm nào, era nào, và cần làm gì tiếp theo.

Bài này cho bạn cái bản đồ đó.

## 2. Mental Model — 5 Eras Của Next.js

[MODEL] Nhìn Next.js như nhìn địa tầng địa chất. Mỗi era để lại dấu vết riêng trong file structure, config, và API. Đọc dấu vết → biết mình đang đứng đâu.

```
2016         2020      2021-22     2022-24       2025+
  │            │          │           │             │
Era 1        Era 2      Era 3       Era 4         Era 5
SSR + SSG    ISR +      Rust +      App Router    Turbopack
             Serverless  Edge        + RSC         Default +
                                                   Cache
                                                   Components
  v1–v8     v9.3–v11    v12        v13–v15         v16+
```

**5 Major Shifts:**

| Shift | Version | Thay đổi |
|-------|---------|----------|
| 1 | v9.3 | `getStaticProps`/`getServerSideProps` thay `getInitialProps` |
| 2 | v12 | Rust-based SWC compiler, Edge Functions, RSC alpha |
| 3 | v13 | App Router + RSC stable, component-level data fetching |
| 4 | v15 | `params` là `Promise` (breaking), `fetch()` default no-cache |
| 5 | v16 | Turbopack default bundler, Cache Components + PPR stable |

### Era 1 — SSR + SSG (v1 → v8, 2016–2019)

**Vấn đề giải quyết:** React app hoàn toàn client-side. SEO fail. TTFB chậm. Next.js ra đời để server-render React.

**Fingerprint trong codebase:**
- `pages/` directory, chỉ có `pages/`
- `getInitialProps` (pre-v9.3) — chạy cả server lẫn client khi navigate
- Webpack 4 mặc định
- `target: 'serverless'` trong `next.config.js` (v8)
- `next.config.js` đơn giản, không có `swcMinify`, không có `appDir`

### Era 2 — ISR + Serverless (v9.3 → v11, 2020–2021)

**[FACT]** v9.3 introduce `getServerSideProps` và `getStaticProps`, thay thế `getInitialProps` cho SSR/SSG. v9.5 (2020-07) introduce stable Incremental Static Regeneration, rewrites/redirects, Webpack 5 beta. v11 (2021) Webpack 5 trở thành default. Nguồn: Next.js 9.5 blog, v11 blog.

**Vấn đề giải quyết:** Static site có hàng triệu page không thể rebuild toàn bộ mỗi lần data thay đổi. ISR cho phép rebuild từng page sau deploy mà không cần rebuild toàn site.

**Fingerprint:**
- `pages/` directory
- `getServerSideProps`, `getStaticProps` (không còn `getInitialProps` trong code mới)
- `revalidate` property trong `getStaticProps`
- `next.config.js` có thể có `webpack5: true` (v9.5–v10) hoặc không (v11 mặc định)
- `rewrites()`, `redirects()` functions trong config

### Era 3 — Rust + Edge (v12, 2021–2022)

**[FACT]** v12 (2021-10) giới thiệu Rust-based SWC compiler thay Babel, Edge Functions/Middleware beta, và RSC alpha (experimental). Theo Vercel official benchmark: khoảng 3x faster Fast Refresh, khoảng 5x faster production builds so với Babel+Webpack. Nguồn: Next.js 12 blog.

**Vấn đề giải quyết:** Build chậm là developer experience killer. Webpack + Babel không scale tốt với codebase lớn.

**Fingerprint:**
- `pages/` directory vẫn là primary
- `next.config.js` có `swcMinify: true`
- `middleware.js` hoặc `middleware.ts` ở root level
- `experimental: { serverComponents: true }` nếu team thử RSC alpha (rất hiếm gặp)
- Không có `app/` directory (RSC còn quá experimental để production)

### Era 4 — App Router + RSC (v13 → v15, 2022–2024)

**[FACT]** v13 (2022-10) ra App Router beta kèm RSC. v13.4 (2023-05) App Router stable, Turbopack beta. v14 (2023-10) Server Actions stable, PPR preview (`experimental.ppr`). v15 (2024-10) Turbopack dev stable. **Breaking change quan trọng trong v15:** `params` và `searchParams` trong page/layout trở thành `Promise` — phải `await params` trước khi destructure. Nguồn: Next.js 13 blog, 13.4 blog, 14 blog, 15 blog.

**Vấn đề giải quyết:** Pages Router không support React 18 natively — RSC, Streaming, Suspense-based hydration. App Router xây lại routing layer để leverage full React 18.

**Fingerprint:**
- `app/` directory tồn tại (có thể song song với `pages/`)
- `layout.tsx`, `page.tsx`, `loading.tsx`, `error.tsx`
- `"use client"` directive trong component files
- `next.config.ts` (TypeScript config, bắt đầu phổ biến từ v15)
- `experimental: { appDir: true }` (v13 beta) hoặc không có flag (v13.4+ stable)
- `experimental: { ppr: true }` nếu đang dùng PPR (v14–v15)
- v15+: `const { id } = await params;` thay vì `const { id } = params;`

### Era 5 — Turbopack Default + Cache Components (v16, 2025+)

**[FACT]** v16.0.0: Turbopack trở thành default bundler (không cần `--turbo` flag). Cache Components với `"use cache"` directive và PPR stable thay thế implicit caching model của App Router cũ. AMP support bị remove hoàn toàn. Nguồn: Next.js 16 blog, Turbopack API Reference.

**[FACT]** `middleware.ts` deprecated trong v16, `proxy.ts` là network boundary mới. `experimental.ppr` bị remove, thay bằng `cacheComponents: true` trong config. Nguồn: Next.js 16 blog.

**Vấn đề giải quyết:** Caching model implicit của App Router trước v16 khó reasoning — "mặc định cache hay không cache?" là câu hỏi phổ biến nhất. Cache Components chuyển sang explicit opt-in với `"use cache"`.

**Fingerprint:**
- `proxy.ts` ở root level (thay `middleware.ts`)
- `next.config.ts` có `cacheComponents: true`
- `"use cache"` directive trong Server Component functions
- Không có `experimental.ppr`, không có AMP-related code
- Turbopack chạy mặc định, không cần `--turbo` flag

## 3. Core Concepts

### 3.1 Version Timeline

**[FACT]** Next.js release timeline (tổng hợp từ official changelog và Next.js Wikipedia):

| Version | Release | Major Feature | Era |
|---------|---------|--------------|-----|
| v1 | 2016-10 | SSR cho React | Era 1 |
| v7 | 2018 | Webpack 4 default | Era 1 |
| v9.3 | 2020-02 | `getStaticProps`, `getServerSideProps` | Era 2 |
| v9.5 | 2020-07 | ISR stable, Webpack 5 beta | Era 2 |
| v11 | 2021-06 | Webpack 5 default | Era 2 |
| v12 | 2021-10 | SWC compiler, Middleware, RSC alpha | Era 3 |
| v13 | 2022-10 | App Router beta, RSC | Era 4 |
| v13.4 | 2023-05 | App Router stable, Turbopack beta | Era 4 |
| v14 | 2023-10 | Server Actions stable, PPR preview | Era 4 |
| v15 | 2024-10 | Turbopack dev stable, async `params` | Era 4 |
| v16 | 2025-10 | Turbopack default, Cache Components, `proxy.ts` | Era 5 |

Đây không phải list để học thuộc. Đây là lookup table cho hai trường hợp thực tế: (1) identify era khi đọc codebase, (2) trả lời câu hỏi phỏng vấn về history.

### 3.2 Era Characteristics Matrix

**[MODEL]** 3 fingerprint nhanh nhất để identify era:

| Era | File Structure | Data Fetching Pattern | Bundler/Compiler |
|-----|--------------|----------------------|-----------------|
| Era 1 | `pages/` only | `getInitialProps` | Webpack 4 + Babel |
| Era 2 | `pages/` + `revalidate` | `getStaticProps`/`getServerSideProps` + ISR | Webpack 5 + Babel/SWC |
| Era 3 | `pages/` + `middleware.ts` | `getServerSideProps`/`getStaticProps` | SWC + Webpack |
| Era 4 | `app/` (± `pages/`) | Server Components + `fetch()` | SWC + Turbopack (dev beta) |
| Era 5 | `app/` + `proxy.ts` | Server Components + `"use cache"` | Turbopack (default) |

### 3.3 Rendering Model Evolution

**[MODEL]** Shift lớn nhất không phải về syntax mà về **nơi data fetching xảy ra**:

**Era 1–3 (Pages Router paradigm) — function-level data fetching:**

```
Request → getServerSideProps (function riêng) → truyền props → Component render → HTML
```

Data fetching tách biệt khỏi component. Component nhận data qua props. Đây là mental model Pages Router.

**Era 4–5 (App Router paradigm) — component-level data fetching:**

```
Request → Server Component (async function) → fetch data bên trong → Stream HTML → Client
```

Component là async function, fetch data trực tiếp bên trong. Server Component không gửi JS xuống client.

**Cách đọc nhanh:** Thấy `getServerSideProps` → Pages Router (Era 1–3). Thấy `export default async function Page()` với `fetch()` bên trong → App Router (Era 4–5). Thấy `"use cache"` directive → Era 5.

> **Overlap Lock — RSC:** Tại sao Server Component không gửi JS xuống client, hydration hoạt động thế nào, serialization boundary — đây là nội dung của M4.1. Bài này chỉ cần biết: Server Component chạy trên server, không có JavaScript ở phía client.

## 4. Anti-Patterns

### ❌ Anti-Pattern 1 — Dùng Pages Router APIs Trong App Router Project

**Code Junior thường viết:**

```ts [app/products/page.tsx]
// ❌ WRONG: getServerSideProps không tồn tại trong App Router
// Runtime behavior: Next.js KHÔNG throw error ngay — function bị silently ignore
// Kết quả: products = undefined → TypeError khi .map() hoặc page trống hoàn toàn

export async function getServerSideProps() {
  const res = await fetch('https://api.example.com/products');
  const products = await res.json();
  return { props: { products } };
}

export default function ProductsPage({ products }: { products: Product[] }) {
  // products là undefined vì getServerSideProps không chạy trong app/
  // Lỗi xuất hiện ở đây, không phải ở getServerSideProps bên trên
  return (
    <ul>
      {products.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  );
}
```

**Vấn đề sâu hơn:** Junior copy sai API nhưng root cause là không hiểu tại sao data fetching phải viết khác trong App Router. Họ nghĩ App Router = Pages Router nhưng đổi tên thư mục từ `pages/` sang `app/`.

**Staff refactor:**

```ts [app/products/page.tsx]
// ✅ CORRECT: Server Component trong App Router
// async function là điểm khác biệt cơ bản nhất so với Pages Router component
// Không cần props, không cần getServerSideProps, fetch data trực tiếp

import { notFound } from 'next/navigation';

interface Product {
  id: string;
  name: string;
}

// async Server Component: chạy trên server, không gửi JS này xuống client
export default async function ProductsPage() {
  const res = await fetch('https://api.example.com/products', {
    // App Router v15+: fetch() không cache mặc định
    // Phải explicit về caching intent
    next: { revalidate: 3600 } // cache 1 tiếng, ISR behavior
  });

  // Không swallow error — fail rõ ràng hơn là fail ngầm
  if (!res.ok) {
    // notFound() render 404 page — không throw, không crash vô kiểm soát
    // Đây là cách App Router handle not-found, khác hoàn toàn Pages Router
    notFound();
  }

  const products: Product[] = await res.json();

  return (
    <ul>
      {products.map(p => (
        <li key={p.id}>{p.name}</li>
      ))}
    </ul>
  );
}
```

**Incident nếu để lại code cũ:** `getServerSideProps` trong `app/` bị Next.js silently ignore — không throw, không warn rõ ràng. Component render với `products = undefined`. Production không crash ngay; page trống hoặc throw `TypeError: Cannot read properties of undefined (reading 'map')` — lỗi xuất hiện xa điểm gây ra, khó debug.

**Trade-off khi refactor:** Refactor data fetching của một page phức tạp (nhiều data source, nhiều loading state) tốn effort không nhỏ. Nếu deadline gấp, giải pháp hợp lý là **giữ route đó trong `pages/`** và dùng Pages Router tiếp — Next.js hỗ trợ hai router chạy song song. Không bao giờ dùng `getServerSideProps` trong `app/`.

> **Overlap Lock:** Cách implement `getServerSideProps` đúng cách trong Pages Router, cách dùng `generateStaticParams` thay `getStaticProps` — đây là nội dung M11. Bài này chỉ nhận diện pattern, không dạy cách dùng API.

### ❌ Anti-Pattern 2 — Upgrade Version Mà Không Kiểm Tra Era Changes

**Approach Junior thường làm:**

```bash
# ❌ WRONG: Upgrade trực tiếp không đọc changelog
npm install next@latest

# Dev server chạy được → nghĩ là ổn
npm run dev

# Build production → fail hoặc silent regression với error không rõ nguyên nhân
npm run build
```

**Config sai sau khi "upgrade" lên v16 không đọc breaking changes:**

```ts [next.config.ts]
// ❌ WRONG: Config pha trộn hai era khác nhau
// Team copy config cũ từ v14/v15, không check v16 breaking changes

import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  experimental: {
    ppr: 'incremental', // ❌ experimental.ppr REMOVED trong v16
                        // Không throw error → silent fail
                        // PPR không được enable → page không có static shell
                        // TTFB regression, không có error message rõ ràng
  },
};

export default nextConfig;
```

**`middleware.ts` vẫn ở root sau upgrade v16:**

```ts [middleware.ts]
// ❌ Deprecated trong v16, sẽ bị remove trong v17
// Không crash ngay nhưng behavior không guaranteed
// Deprecated warning trong build output — dễ bỏ qua

import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export async function middleware(request: NextRequest) {
  // Logic auth check này cần migrate sang proxy.ts trong v16
  if (!request.cookies.get('auth-token')) {
    return NextResponse.redirect(new URL('/login', request.url));
  }
  return NextResponse.next();
}
```

**Staff approach — checklist bắt buộc trước khi upgrade:**

```bash
# Bước 1: Đọc upgrading guide TRƯỚC khi chạy bất kỳ lệnh nào
# Cho mỗi major version jump, đọc docs.nextjs.org/docs/app/building-your-application/upgrading

# v14 breaking changes cần check:
# - `next export` command bị remove → dùng output: 'export' trong config
# - `@next/font` deprecated → dùng `next/font`

# v15 breaking changes cần check:
# - fetch() default behavior thay đổi: không còn cache mặc định
# - params và searchParams là Promise → phải await trước khi dùng
# - next lint config thay đổi

# v16 breaking changes cần check:
# - experimental.ppr removed → dùng cacheComponents: true
# - middleware.ts deprecated → migrate sang proxy.ts
# - AMP support removed hoàn toàn
# - Turbopack là default (không cần --turbo flag)

# Bước 2: Dùng codemods chính thức để auto-fix các change có thể tự động
npx @next/codemod@latest upgrade

# Bước 3: Upgrade từng major version, KHÔNG skip
# v13.4 → v14 → v15 → v16, không phải v13.4 → v16 trực tiếp
# Mỗi step: chạy build + test suite, fix broken → commit → tiếp tục

# Bước 4: Staging environment trước production
# Nếu không có staging: feature flag trong production (xem Rollback Plan ở Section 7)
```

**Config đúng cho v16:**

```ts [next.config.ts]
// ✅ CORRECT: Config v16
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  // Cache Components thay thế experimental.ppr
  // Cần bật để dùng "use cache" directive và PPR stable
  cacheComponents: true,

  // Turbopack là default — không cần config thêm
  // Nếu cần fallback webpack (plugin webpack chưa tương thích với Turbopack):
  // $ next dev --webpack
  // $ next build --webpack
};

export default nextConfig;
```

**Incident nếu để lại config cũ:** Build succeed nhưng PPR không hoạt động — page vẫn render dynamic toàn bộ thay vì static shell + dynamic holes. TTFB tăng, không có error message rõ ràng, chỉ có warning trong build output mà Junior thường bỏ qua.

**Trade-off khi refactor:** Đọc changelog và upgrade từng step tốn nhiều thời gian hơn upgrade "big bang". Nhưng upgrade hai major version cùng lúc tạo ra bugs khó isolate — không biết breaking change đến từ v14→v15 hay v15→v16. **Staff rule:** Upgrade từng major version một, mỗi step có commit riêng và test suite pass trước khi tiếp tục.

## 5. Trade-offs — 4 Approaches Quản Lý Version/Era

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead |
|----------|------|------|--------------|---------------------|----------|
| **Ở lại Pages Router, chỉ upgrade minor/patch** | Ổn định; ít code change; team không cần học RSC ngay; không rủi ro migration | Không có RSC/PPR/Cache Components; docs ngày càng mỏng cho Pages Router; khó recruit khi JD đòi App Router | App nhỏ, ít dynamic content; team chưa sẵn sàng; deadline gấp, ưu tiên ổn định | Cần streaming, RSC, PPR dài hạn; app cần scale lớn; team có bandwidth học | Thấp ban đầu, tech debt tăng dần theo thời gian |
| **Migration toàn bộ sang App Router cùng lúc (big-bang)** | Codebase đồng nhất; tận dụng đầy đủ RSC/streaming/PPR; dễ recruit | Rủi ro cao; nhiều breaking changes cùng lúc; cần refactor data fetching, routing, auth, tests; team cần đào tạo lại | Dự án mới; codebase nhỏ (< 20 routes) với test coverage tốt; team strong và có thời gian | Codebase lớn (> 50 routes); thiếu test; deadline ngắn; team chưa hiểu RSC | Cao — effort lớn, risk production; giảm tech debt dài hạn |
| **Migration incremental (route-by-route)** | Giảm rủi ro; hai router chạy song song; ưu tiên route có ROI cao nhất; team học dần | Phức tạp về config; hai mental model cùng lúc; dễ nhầm lẫn API giữa Pages và App; cần coordination | Hệ thống lớn, nhiều route; cần giữ production ổn định; có thể chia migration theo domain/team | Dự án nhỏ (migration một lần nhanh hơn); không có owner rõ ràng cho từng route | Trung bình — risk phân tán, cần coordination tốt |
| **Lock version, không upgrade** | Ổn định tuyệt đối; không breaking changes | Không có security patches; CVE risk tăng dần; không có Turbopack/RSC/PPR; khó recruit; forced migration sau nhiều năm sẽ đau hơn nhiều | POC nội bộ ngắn hạn; tool không public-facing; không yêu cầu security patches | Bất kỳ app public-facing nào; app cần security; app dùng Vercel/hosting mới với constraints | Thấp ban đầu, cực cao khi buộc phải migrate sau nhiều năm |

**Hai anti-pattern phổ biến nhất khi nhìn vào bảng này:**

- **"Cứ dùng latest là đúng":** Enterprise app chạy v13.x không phải vì team lười upgrade. Có thể vì dependencies chưa support v16, migration cost cao hơn ROI, hoặc team đang focus vào business feature. Staff đánh giá migration cost trước khi propose upgrade.

- **"Pages Router = legacy = xấu":** Pages Router vẫn perfectly valid cho nhiều use case. Không phải mọi app đều cần RSC hay PPR. Staff chọn tool theo requirements, không theo "mới nhất".

## 6. Legacy Recognition — Đọc Codebase Như Đọc Địa Tầng

Khi join team mới hoặc nhận legacy codebase, 4 pattern cần check ngay:

### Pattern 1: `getInitialProps` (Era 1 Artifact)

```ts [pages/index.tsx]
// Fingerprint Era 1: getInitialProps — API pre-v9.3
```

### Cache Components Pattern (v16)

```ts [app/dashboard/page.tsx]
// ✅ v16: Cache Components với "use cache" directive
// Explicit caching — không còn implicit cache behavior như App Router cũ
// "use cache" trong function body, không phải wrapper function bên ngoài

import {
  unstable_cacheTag as cacheTag,
  unstable_cacheLife as cacheLife,
} from 'next/cache';

// Chỉ cache function có data thực sự đắt — không cache mọi thứ
async function getDashboardMetrics(userId: string) {
  'use cache'; // ← directive opt-in: explicit, co-located với data logic

  // cacheLife: predefined strategy hoặc custom {maxAge, staleWhileRevalidate}
  cacheLife('hours'); // maxAge: 3600, staleWhileRevalidate: 900 (mặc định)

  // cacheTag: identifier để invalidate targeted cache entries
  cacheTag(`dashboard:${userId}`);

  const res = await fetch(`https://api.example.com/metrics/${userId}`);

  if (!res.ok) {
    // Error trong cached function cũng được cache — cân nhắc kỹ strategy
    throw new Error(`Metrics fetch failed: ${res.status}`);
  }

  return res.json();
}

export default async function DashboardPage({
  params,
}: {
  // v15+: params là Promise — breaking change từ v14
  params: Promise<{ userId: string }>;
}) {
  const { userId } = await params; // phải await trước khi destructure
  const metrics = await getDashboardMetrics(userId);

  return <DashboardMetrics data={metrics} />;
}
```

### proxy.ts (v16) — Thay middleware.ts

```ts [proxy.ts]
// ✅ v16: proxy.ts là network boundary mới
// Đặt ở root, cùng level với next.config.ts
// [THEORY — NEEDS VERIFICATION]: Exact API signature của proxy.ts
// cần verify với Next.js 16 official docs — check docs.nextjs.org trước khi deploy

import { NextResponse } from 'next/server';

    return NextResponse.redirect(new URL('/login', request.url));

  return NextResponse.next();
```

### 3 Giai Đoạn Migration (Incremental — Không Big-Bang)

**Giai đoạn 1: Stabilize trong era hiện tại**
- Upgrade minor/patch trong cùng era, không cross major version
- Loại `getInitialProps` → thay bằng `getServerSideProps`/`getStaticProps`
- Bật SWC nếu đang dùng Babel (`swcMinify: true` trong v12+)
- Loại AMP code nếu có (hard blocker cho v16)
- Đảm bảo test suite pass 100% trước khi tiếp tục

**Giai đoạn 2: Bật App Router song song**
- Route **mới** tạo trong `app/` — không đụng route cũ trong `pages/`
- Hai router chạy song song — Next.js hỗ trợ natively, không cần config đặc biệt
- Team học App Router patterns qua route mới, không rủi ro route production cũ
- Khi team confident về App Router patterns: bắt đầu giai đoạn 3

**Giai đoạn 3: Migrate route cũ và hoàn thiện**
- Migrate từng route một (không phải từng file)
- Xóa `pages/[route]` khi `app/[route]/page.tsx` đã stable trên staging
- Sau khi route cuối migrate xong: xóa `pages/` directory
- Migrate `middleware.ts` → `proxy.ts`
- Migrate `experimental.ppr` → `cacheComponents: true`
- Migrate `unstable_cache` → `"use cache"` directive

### Rollback Plan

```bash
# Git strategy: commit + tag sau mỗi giai đoạn hoàn chỉnh
git tag v-pre-migration          # Trước khi bắt đầu giai đoạn 1
git tag v-phase1-stable          # Sau khi giai đoạn 1 pass CI
git tag v-phase2-stable          # Sau khi giai đoạn 2 pass CI

# Nếu production incident ở giai đoạn 3:
# Option 1: git revert về v-phase2-stable (rollback code)
# Option 2: tắt feature flag (rollback behavior, không cần redeploy)
```


## 8. Production Scenarios

### Case Study 1: Pages Router v11 → v14 (Upgrade Không Migration Router)

**Context:** E-commerce platform, 80 routes trong `pages/`, team 6 người, Next.js v11.

**Decision:** Ở lại Pages Router, upgrade lên v14 để có security patches và SWC performance.

**Trong cùng phase migration:**
- v11 → v12: Bật SWC (`swcMinify: true`) → build time từ ~4 phút xuống ~45 giây (theo team report)
- v12 → v13: Giữ Pages Router, tạo route mới trong `app/` cho product detail pages mới
- v13 → v14: Server Actions cho form submissions trong route mới → giảm client JS bundle

**Kết quả:** Team học App Router từng bước qua route mới, không rủi ro 80 route cũ đang chạy prod. Performance cải thiện rõ rệt từ SWC mà không cần migration router toàn bộ.

**Lesson:** Upgrade version và migrate router là hai quyết định độc lập. Có thể upgrade v11→v14 mà vẫn giữ Pages Router.

---

### Case Study 2: App Router v13 Beta → v14 Stable (Tắt Experimental Feature)

**Context:** SaaS dashboard, migrate từ Pages Router lên App Router v13 beta. Bật PPR preview (`experimental.ppr: true`) ngay khi v14 ra.

**Incident sau 2 tuần prod:** PPR gây hydration mismatch lác đác — static shell không match dynamic content trong một số edge case với certain user session states. Khó reproduce, càng khó debug trong production.

**Decision:** Tắt `experimental.ppr`, giữ Server Components và Server Actions stable nhưng không PPR.

**Sau khi tắt PPR:**
```ts [next.config.ts]
// ✅ Safe config cho giai đoạn này: tắt experimental, giữ stable features
const nextConfig: NextConfig = {
  // Không có experimental.ppr
  // Server Components và Server Actions vẫn hoạt động bình thường
};
```

**Kết quả:** App ổn định. Team tiếp tục dùng App Router stable features. Khi v16 ra với PPR stable (`cacheComponents: true`), bật lại có kiểm soát.

**Lesson:** Experimental flag trong Next.js production = rủi ro không có SLA. **Staff rule:** Không bật `experimental.*` trong production trừ khi có team capacity debug 24/7 nếu fail.

---

### Case Study 3: ISR v9.5 → Turbopack v15 (Performance Upgrade Incremental)

**Context:** News site, 100k+ article pages, ISR từ v9.5, team 4 người. Build time ngày càng chậm.

**Trong cùng phase migration:**
- v9.5 → v12: SWC compiler → build time drop đáng kể
- v12 → v15: Bật Turbopack dev (`--turbo` flag trong v13–v15) → HMR từ ~3 giây xuống < 1 giây
- ISR logic không thay đổi — `revalidate` vẫn hoạt động trong App Router qua `next: { revalidate: X }` trong `fetch()`

**Gotcha gặp phải:** Một số image optimization config không tương thích với Turbopack build beta. Team dùng `--webpack` cho production build trong khi dùng `--turbo` cho dev — merge hoàn toàn sau khi Turbopack build stable trong v15.5+.

**Lesson:** Dev experience (HMR) có thể dùng Turbopack sớm hơn. Production build nên chờ Turbopack build stable (v15.5+) hoặc verify compatibility trước.

## 10. Architecture Notes — Decision Trees

### Khi Nào PHẢI Dùng App Router

App Router không phải "luôn tốt hơn" — nhưng có những use case Pages Router không handle tốt:

- **Streaming:** Real-time dashboard với nhiều data source loading độc lập → App Router + Suspense boundaries
- **RSC:** Cần giảm client JS bundle đáng kể → Server Component không gửi JS xuống client
- **PPR (v16):** Static shell với dynamic holes → `cacheComponents: true` + Cache Components
- **Nested layouts với data riêng:** Sidebar, header, content đều load data riêng → App Router layout system
- **Long-term roadmap:** Nếu cần các feature trên trong 6–12 tháng tới → migrate sớm tốt hơn migrate gấp sau

### Khi Nào Pages Router Vẫn Là Lựa Chọn Hợp Lý

- App marketing/blog đơn giản: SSG, không cần streaming, không cần RSC
- Legacy codebase lớn với test coverage thấp: migration risk > benefit rõ ràng
- Team không có bandwidth học RSC patterns đúng cách: forced migration tạo bugs tệ hơn Pages Router
- Deadline gấp: ổn định ưu tiên hơn feature mới

### Turbopack vs Webpack — Quyết Định Cho Production Build

```bash
# Next.js 16: Turbopack là default cho cả dev và build
npm run dev    # → Turbopack
npm run build  # → Turbopack

# Check compatibility trước: https://areweturbopackyet.com

# Dấu hiệu cần fallback --webpack:
# 1. Custom webpack loader trong next.config.ts chưa có Turbopack equivalent
# 2. Build output khác biệt đáng kể so với Webpack (kiểm tra bundle sizes)
# 3. Specific CSS/asset transform fail với Turbopack

# Fallback khi cần

npm run build -- --webpack
```

## 11. Interview Questions

**Câu 1 (Junior hỏi):** App Router ra đời để giải quyết vấn đề gì của Pages Router?

<details>
<summary>Đáp án Staff</summary>

Pages Router không thể leverage React 18 natively — RSC, Streaming, Suspense-based hydration không thể implement clean với Pages Router architecture. Data fetching trong Pages Router tách rời khỏi component: `getServerSideProps` là function riêng, component nhận data qua props, không phải async. App Router xây lại từ đầu để component là async function, fetch data inline. Server Component không gửi JS xuống client. Shift từ "function-level data fetching" sang "component-level async data fetching". Đây không phải "version mới tốt hơn" — đây là paradigm shift để enable React 18 features.

</details>

---

**Câu 2 (Mid hỏi):** Next.js đã có những paradigm shift nào? Tại sao?

<details>
<summary>Đáp án Staff</summary>

3 paradigm shifts quan trọng nhất:

**1. v9.3 — `getStaticProps`/`getServerSideProps` thay `getInitialProps`:** `getInitialProps` chạy cả server lẫn client khi navigate, gây confusion về behavior và security (data nhạy cảm có thể leak). `getServerSideProps`/`getStaticProps` chỉ chạy trên server — rõ ràng, predictable, tách biệt hơn.

**2. v13 — App Router + RSC:** Pages Router không support React 18 natively. App Router + RSC cho phép component-level async data fetching, streaming, nested layouts với data riêng. Server Component không có JS ở client — giảm bundle size. Đây là shift cơ bản về mô hình component.

**3. v16 — Cache Components + Turbopack default:** Caching implicit của App Router cũ gây confusion ("có cache hay không?" là câu hỏi phổ biến nhất). Cache Components làm explicit với `"use cache"` opt-in. Turbopack thay Webpack làm default bundler.

Mỗi shift xuất phát từ pain point production thực tế, không phải feature marketing.

</details>

---

**Câu 3 (Mid hỏi):** Nếu join team đang dùng Next.js 14, bạn sẽ làm gì trước tiên?

<details>
<summary>Đáp án Staff</summary>

4 bước theo thứ tự:

**1. Identify era và router:** Check có `app/` hay `pages/` hay cả hai? Có `experimental.ppr` trong config không? Nếu cả hai router tồn tại, team đang ở giai đoạn nào của migration?

**2. Đọc `package.json` và `next.config.ts`:** Version chính xác? Có lock version không? Config flags nào đang bật? Có `@next/bundle-analyzer` không?

**3. Đọc routing structure:** Bao nhiêu route tổng cộng? Route nào trong `app/`, route nào trong `pages/`? Đây là migration state hiện tại.

**4. Hỏi team về context:** Tại sao đang ở v14? Có plan lên v15/v16 không? Pain points hiện tại là gì? Dependencies nào chưa support v16?

Không propose "upgrade lên v16 ngay" mà chưa hiểu context. Migration cost phụ thuộc vào số route, test coverage, team capacity, và dependencies.

</details>

**Câu 4 (Senior hỏi):** Khi nào nên dùng Pages Router thay vì App Router?

<details>
<summary>Đáp án Staff</summary>

Pages Router vẫn hợp lý khi:

**Migration cost > benefit:** Codebase 100+ routes, test coverage thấp, không có ROI rõ ràng từ RSC/PPR trong 12 tháng tới. Migration toàn bộ có thể tốn 3–6 tháng với team 5 người — nếu business không cần RSC/streaming trong thời gian đó, ở lại là quyết định đúng.

**Team capacity thấp:** Forced migration với team chưa hiểu RSC patterns tạo bugs tệ hơn Pages Router. Biết dùng sai Server Component, nhầm Client/Server boundary — những bugs này khó debug hơn bugs trong Pages Router quen thuộc.

**App đơn giản:** Blog, landing page, admin tool nội bộ — không cần streaming, không cần client JS reduction đáng kể, không cần PPR.

Senior không nói "App Router luôn tốt hơn". Senior nói: "Với codebase này, requirements này, team này — App Router có justify migration cost không?" Nếu không → Pages Router là đúng.

</details>

---

**Câu 5 (Mid hỏi):** ISR khác SSR và SSG thế nào? ISR xuất hiện từ version nào?

<details>
<summary>Đáp án Staff</summary>

**SSG:** Build time → HTML → CDN. Không thay đổi cho đến rebuild toàn bộ app. Nhanh nhất, nhưng data stale sau mỗi deploy. Phù hợp: content tĩnh, ít thay đổi.

**SSR:** Request time → fetch data → HTML. Fresh data mỗi request, nhưng latency cao hơn, không cache được ở CDN edge. Phù hợp: data personalized, real-time.

**ISR:** Static như SSG nhưng rebuild từng page sau khoảng thời gian nhất định. Người dùng đầu tiên sau X giây nhận HTML cũ (stale-while-revalidate), kích hoạt background rebuild. Người dùng tiếp theo nhận HTML mới. Hybrid giữa SSG performance và SSR freshness.

**ISR stable từ v9.5 (2020-07).** Vẫn tồn tại trong App Router qua `next: { revalidate: X }` trong `fetch()` options hoặc `cacheLife('hours')` trong Cache Components (v16).

ISR không phải silver bullet: nếu data phải fresh 100% mỗi request (stock price, real-time inventory), dùng SSR (`cache: 'no-store'`). Nếu data chấp nhận stale vài phút, ISR là sweet spot.

</details>

### 🔴 Bài 3: Debug the Era Mismatch

**Đề bài:**

Bạn nhận bug report từ production sau khi team upgrade lên v16:

> *"Hai vấn đề xảy ra đồng thời: (1) Dashboard TTFB tăng 3x — static shell không có nữa, toàn bộ dynamic render lại. (2) Request logging middleware không ghi log — không có log nào từ middleware layer trong tuần này dù monitoring vẫn nhận traffic. Không có error trong Sentry."*

```ts [next.config.ts]
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  experimental: {
    ppr: true,
  },
};

export default nextConfig;
```

```ts [middleware.ts]
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';
import { logger } from '@/lib/logger';

export async function middleware(request: NextRequest) {
  logger.info({
    method: request.method,
    path: request.nextUrl.pathname,
    timestamp: Date.now(),
  });

  return NextResponse.next();
}

export const config = {
  matcher: ['/((?!_next|favicon.ico).*)'],
};
```

```ts [app/dashboard/page.tsx]
import { unstable_cache } from 'next/cache';

const getCachedDashboardData = unstable_cache(
  async (userId: string) => {
    const res = await fetch(`https://api.example.com/dashboard/${userId}`);
    if (!res.ok) throw new Error('Failed');
    return res.json();
  },
  ['dashboard'],
  { revalidate: 3600, tags: ['dashboard'] }
);

export default async function DashboardPage({
  params,
}: {
  params: { userId: string }; // Type annotation này cũng sai cho v15+
}) {
  const { userId } = params; // Thiếu await
  const data = await getCachedDashboardData(userId);
  return <DashboardUI data={data} />;
}
```

**Yêu cầu:**
1. Identify TẤT CẢ mismatch giữa code và v16 expectations (ít nhất 4 issues)
2. Viết lại `next.config.ts`, `proxy.ts` (thay `middleware.ts`), và `app/dashboard/page.tsx` đúng cho v16
3. Giải thích tại sao bug (2) về logging có thể silent fail mà không có error trong Sentry
4. Giải thích điểm khác nhau về reasoning giữa `unstable_cache` và `"use cache"` directive

**Acceptance Criteria:**
- [ ] Identify đúng: `experimental.ppr` removed trong v16 (issue 1 — silent fail)
- [ ] Identify đúng: `middleware.ts` deprecated trong v16 → behavior không guaranteed (issue 2 — silent fail)
- [ ] Identify đúng: `unstable_cache` không phải Cache Components pattern cho v16 (issue 3)
- [ ] Identify đúng: `params` type và usage sai cho v15+ (`Promise<{userId: string}>` + `await`) (issue 4)
- [ ] Viết lại đúng cả 3 artifacts: `next.config.ts`, `proxy.ts`, `app/dashboard/page.tsx`
- [ ] Giải thích được tại sao cả hai bugs đều "silent fail" — không có error nhưng behavior sai

<details>
<summary>Hint — chỉ mở khi đã identify ít nhất 2 issues</summary>

Silent fail là pattern nguy hiểm nhất trong era mismatch. `experimental.ppr` trong v16: không throw error, bị ignore → PPR không enable → dynamic render → TTFB tăng mà không có error. `middleware.ts` deprecated: không crash ngay, nhưng behavior không guaranteed trong v16 → logger không được gọi mà không có error. Về `"use cache"` vs `unstable_cache`: `unstable_cache` là wrapper function bên ngoài (explicit wrapping), `"use cache"` là directive bên trong function body (co-located với logic). Trong v16, Cache Components model dùng directive, không phải wrapper. Về `params`: v15+ breaking change — type là `Promise<{userId: string}>`, phải `await params` trước khi destructure. Không await → `userId` là Promise object, không phải string → `fetch()` URL sai nhưng không crash nếu API trả về fallback. Xem lại Section 7 cho pattern đúng của cả 3 files.

</details>

## 13. Summary

- **Era, không phải version number** là mental model đúng để đọc codebase. Thấy `getServerSideProps` → Pages Router era. Thấy `export default async function Page()` với `fetch()` → App Router era. Thấy `"use cache"` → Cache Components era (v16+). 5 phút fingerprinting tiết kiệm hàng giờ confusion.

- **5 Eras, 3 paradigm shifts thực sự:** v9.3 (function-level data fetching, tách logic khỏi component), v13 (component-level data fetching + RSC, co-located logic), v16 (explicit caching + Turbopack default). Mỗi shift giải quyết pain point production cụ thể, không phải feature marketing.

- **"Latest version" ≠ "đúng nhất" cho mọi context.** Enterprise app đang chạy v13.x hoàn toàn có thể là quyết định đúng. Staff đánh giá migration cost, team capacity, production risk trước khi propose upgrade. Pages Router vẫn valid cho nhiều use case.

- **Migration incremental luôn an toàn hơn big-bang.** Hai router chạy song song trong Next.js là feature có chủ ý, không phải workaround. Dùng nó để migrate route-by-route, với rollback plan ở mỗi giai đoạn.

## 14. Terminology Evolution

| Term | Era | Status | Context | Lịch sử thay đổi | Codebase legacy |
|------|-----|--------|---------|-----------------|----------------|
| **Pages Router** | Era 1 (v1+) | [LEGACY_CORE] — supported trong v16, không nhận features mới | Routing system dựa trên `pages/` directory | Không thay đổi API; không được prioritize từ v13+; docs ngày càng thin | `pages/index.tsx`, `pages/api/users.ts` |
| **App Router** | Era 4 (v13 beta → v13.4 stable) | [MODERN] | Routing dựa trên `app/` directory; layouts, Server/Client Components, Streaming | v13.0 beta → v13.4 stable; `experimental: { appDir: true }` không cần từ v13.4 | `app/layout.tsx`, `app/page.tsx`, `app/(group)/` |
| **RSC (React Server Components)** | Era 3 (v12 alpha), Era 4 (v13 stable) | [MODERN] | Component chạy exclusively trên server, không có JS bundle ở client | v12 experimental → v13 stable với App Router; không available trong Pages Router | Component không có `"use client"` trong `app/` = Server Component mặc định |
| **`getServerSideProps`** | Era 2 (v9.3) | [LEGACY_CORE] [DEPRECATED in App Router] | SSR data fetching trong Pages Router | Từ "pattern chính cho SSR" → "legacy pattern, thay bằng Server Component + `cache: 'no-store'`" | `export async function getServerSideProps(ctx)` trong `pages/` |
| **`getStaticProps`** | Era 2 (v9.3) | [LEGACY_CORE] [DEPRECATED in App Router] | SSG data fetching trong Pages Router | Từ "build-time data fetching" → "replaced by `generateStaticParams` + `fetch()` với cache" | `export async function getStaticProps()` với `revalidate` trong `pages/` |
| **ISR (Incremental Static Regeneration)** | Era 2 (v9.5 stable) | [STILL_ALIVE] evolved | Pages Router: `revalidate` trong `getStaticProps`; App Router: `next: { revalidate: X }` trong `fetch()` | Concept giữ nguyên, API thay đổi theo Router; v16 có `cacheLife()` trong Cache Components | `revalidate: 60` trong `getStaticProps` hoặc `fetch(url, { next: { revalidate: 60 } })` |
| **SSG (Static Site Generation)** | Era 1 | [STILL_ALIVE] | Build-time HTML generation | Pages Router: `getStaticProps`; App Router: default behavior của Server Component không có dynamic data + `generateStaticParams` | `getStaticProps` hoặc `output: 'export'` trong config |
| **SSR (Server-Side Rendering)** | Era 1 | [STILL_ALIVE] evolved | Request-time rendering | Pages Router: `getServerSideProps`; App Router: Server Component với `fetch(url, { cache: 'no-store' })` | `getServerSideProps` trong `pages/` hoặc `fetch(url, { cache: 'no-store' })` trong `app/` |
| **Turbopack** | Era 3 (v12 alpha), Era 4 (v13.4 beta), Era 5 (v16 default) | [MODERN] | Rust-based incremental bundler thay Webpack | v12 alpha → v13.4 beta → v15.0 dev stable → v15.3 build experimental → v15.5 build beta → v16.0 default | `--turbo` flag (v13–v15); không cần flag từ v16 |
| **Cache Components / PPR** | Era 5 (v16) | [MODERN] | Explicit caching với `"use cache"` directive; PPR = static shell + dynamic holes via Suspense | `experimental.ppr` (v14–v15) → `cacheComponents: true` (v16); `unstable_cache` → `"use cache"` | `"use cache"` trong function body; `cacheLife()`, `cacheTag()` calls |
| **`proxy.ts`** [BIBLE MISSING] | Era 5 (v16) | [MODERN] | Network boundary mới thay `middleware.ts` | `middleware.ts` deprecated trong v16 → `proxy.ts` là replacement; `middleware.ts` sẽ bị remove trong v17 | `proxy.ts` ở root level, cùng level với `next.config.ts` |
| **`experimental.ppr` / `cacheComponents`** [BIBLE MISSING] | Era 4 (v14–v15) → Era 5 (v16) | [REFERENCE_ONLY] → [MODERN] | PPR config evolution | `experimental: { ppr: true }` (v14) → `experimental: { ppr: 'incremental' }` (v15) → removed (v16) → `cacheComponents: true` | Config cũ trong `next.config.ts`; không dùng trong code mới |
| **`getInitialProps`** | Era 1 (v1–v8) | [HISTORY_ONLY] | SSR trước khi có `getServerSideProps`; chạy cả server lẫn client khi navigate | Superseded bởi `getServerSideProps` từ v9.3; vẫn technically hoạt động nhưng không recommended | `MyPage.getInitialProps = async (ctx) => {}` |
| **`target: 'serverless'`** | Era 1 (v8) | [HISTORY_ONLY] | Serverless deployment mode trong v8 | Removed trong v9; Next.js hỗ trợ serverless natively mà không cần flag | `target: 'serverless'` trong `next.config.js` — rất cũ |
| **AMP (`useAmp`)** | Era 1–4 | [HISTORY_ONLY] removed trong v16 | AMP page support | Deprecated warnings từ v15.5 → removed hoàn toàn trong v16, không có graceful migration | `useAmp()`, `export const config = { amp: true }` trong `pages/` |

Có một thứ tôi thấy nhiều hơn bugs trong production: developer bị paralyzed vì không biết mình đang đứng ở đâu.

"Codebase này dùng gì?" → Era 4, v14, App Router với `experimental.ppr`. "Nên upgrade không?" → Check 3 breaking changes của v15→v16, estimate effort migrate 5 PPR routes, đánh giá với PM về timeline. "API này còn dùng được không?" → `getServerSideProps` trong `pages/` vẫn dùng được, trong `app/` thì không.

Đó là sự khác biệt giữa Fresher và Junior — không phải ở chỗ biết nhiều API hơn, mà ở chỗ biết **đang đứng đâu để hỏi câu hỏi đúng.**
