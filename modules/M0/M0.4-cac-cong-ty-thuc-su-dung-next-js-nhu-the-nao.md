# M0.4 — Các Công Ty Thực Sự Dùng Next.js Như Thế Nào



## 1. Lesson Goal

Sau bài này, bạn có thể:

- **Nhìn vào job description** và biết ngay công ty đó dùng Next.js theo kiến trúc nào
- **Hiểu tại sao** hai công ty cùng dùng Next.js nhưng code lại khác nhau hoàn toàn
- **Không bị shock** khi codebase thực tế khác xa những gì tutorial dạy
- **Đặt câu hỏi đúng** trong technical interview: về infrastructure, team structure, deployment
- **Đưa ra quyết định kiến trúc** khi được hỏi "Chúng ta nên thiết kế hệ thống này như thế nào?"

> **Vấn đề cốt lõi:** Tutorial dạy bạn xây một thứ cụ thể. Production là vô số thứ cụ thể khác nhau. Bài này là bản đồ của khoảng cách đó.

## 2. Evolution Context

### Tại sao Next.js được chọn — và chọn vì lý do gì

Không phải công ty nào chọn Next.js đều vì cùng một lý do. Đây là điều quan trọng nhất để hiểu.

```
Công ty A (E-commerce):
  Vấn đề: Google không index được SPA React
  Giải pháp: Next.js vì SSR/SSG → SEO tốt
  Họ dùng: 80% SSG, 20% SSR, App Router optional

Công ty B (SaaS Dashboard):
  Vấn đề: Cần build admin panel nhanh, không cần SEO
  Giải pháp: Next.js vì routing, API routes, DX tốt
  Họ dùng: 90% CSR (Client Side Rendering), Next.js gần như chỉ là router

Công ty C (Startup):
  Vấn đề: Team 3 người, cần full-stack nhanh
  Giải pháp: Next.js vì không cần backend riêng ban đầu
  Họ dùng: Server Actions, API Routes, Prisma, Vercel

Công ty D (Enterprise):
  Vấn đề: Legacy .NET backend, cần modern frontend
  Giải pháp: Next.js là "frontend layer" trên existing API
  Họ dùng: Next.js chỉ fetch từ .NET API, không có API Routes
```

**Lesson:** Khi thấy Next.js trong job requirement, câu hỏi phải hỏi không phải "Họ dùng Next.js", mà là **"Họ dùng Next.js để làm gì?"**

### Tiến hóa của việc sử dụng Next.js trong industry

```
2019–2020: "SSR cho React" — Companies chọn Next.js vì SSR
  → Hầu hết dùng Pages Router + getServerSideProps
  → App Router chưa tồn tại

2021–2022: "Framework cho mọi thứ" — Companies dùng Next.js rộng hơn
  → API Routes cho simple backend
  → ISR cho content sites
  → Vercel deployment boom

2023–2024: "App Router revolution" — Kỳ vọng cao, adoption chậm
  → Nhiều công ty vẫn ở Pages Router (stable, production-proven)
  → Startup mới → App Router
  → Company cũ → "Sẽ migrate sau"

2025–hiện tại: "Pragmatic adoption" — Reality check
  → App Router phổ biến cho greenfield
  → Nhiều codebase vẫn Pages Router — không migrate vì cost > benefit
  → Next.js 15/16 features dành cho những ai thực sự cần
```

## 3. Mental Model — Next.js Là Infrastructure, Không Phải Sản Phẩm

**Sai lầm phổ biến nhất của developer mới:** Nghĩ rằng "dùng Next.js" có nghĩa là xây thứ gì đó trông giống tutorial.

**Thực tế:** Next.js là một **infrastructure layer**. Sản phẩm thực sự là business logic bên trên.

```
TUTORIAL MINDSET:
  Next.js App
  ├── pages (hoặc app)
  ├── components
  ├── styles
  └── public

PRODUCTION MINDSET:
  Business Problem
  └── Giải pháp kỹ thuật
      ├── Next.js (một phần của solution)
      ├── Database (Postgres, MySQL, MongoDB...)
      ├── Auth (NextAuth, Auth0, Clerk, custom...)
      ├── CDN (Cloudflare, CloudFront...)
      ├── Monitoring (Datadog, Sentry...)
      ├── CI/CD (GitHub Actions, Jenkins...)
      └── Infrastructure (Vercel, Docker, K8s...)
```

**Analogy:** Next.js giống như khung nhà. Tutorial dạy bạn dựng khung. Công ty thực tế phải lo: điện, nước, nội thất, hệ thống PCCC, an ninh, bảo trì — và tất cả những thứ đó có ảnh hưởng đến cách dựng khung.

## 4. Core Concepts — 5 Kiến Trúc Thực Tế

### 📐 Pattern A — Full-Stack Monolith (Startup / SMB)

**Phổ biến ở:** Startup giai đoạn đầu, freelance project, MVP, công ty <20 người

**Đặc điểm:** Next.js làm TẤT CẢ — frontend, backend logic, database access, auth

```
startup-app/
├── app/
│   ├── (auth)/
│   │   ├── login/page.tsx
│   │   └── register/page.tsx
│   ├── (dashboard)/
│   │   ├── layout.tsx
│   │   └── dashboard/page.tsx
│   └── api/           ← Backend sống trong Next.js
│       ├── auth/
│       │   └── [...nextauth]/route.ts
│       └── webhooks/
│           └── stripe/route.ts
├── actions/           ← Server Actions thay thế nhiều API endpoints
│   ├── user.ts
│   ├── product.ts
│   └── order.ts
├── lib/
│   ├── db.ts          ← Prisma client
│   ├── auth.ts        ← Auth config
│   └── stripe.ts      ← Payment
└── prisma/
    └── schema.prisma  ← Database schema ngay trong repo
```

```typescript
// actions/product.ts — Server Action trong Pattern A
// Database access TRỰC TIẾP từ Next.js layer
'use server';

import { db } from '@/lib/db';
import { auth } from '@/lib/auth';
import { revalidatePath } from 'next/cache';
import { z } from 'zod';

const CreateProductSchema = z.object({
  name: z.string().min(1).max(100),
  price: z.number().positive(),
  description: z.string().optional(),
});

export async function createProduct(formData: FormData) {
  // Auth check thẳng trong Server Action
  const session = await auth();
  if (!session?.user) throw new Error('Unauthorized');

  const input = CreateProductSchema.parse({
    name: formData.get('name'),
    price: Number(formData.get('price')),
    description: formData.get('description'),
  });

  // Prisma query thẳng — không qua API layer nào
  const product = await db.product.create({
    data: {
      ...input,
      createdById: session.user.id,
    },
  });

  revalidatePath('/dashboard/products');
  return product;
}
```

```tsx
      take: 10,
    }),
    db.order.count({
      where: { userId: session.user.id },
    }),
    db.order.aggregate({
      where: { userId: session.user.id, status: 'COMPLETED' },
      _sum: { total: true },
    }),
  ]);

  return (
    <div>
      <h1>Dashboard</h1>
      {/* Render data */}
    </div>
  );
}
```

**Tại sao Pattern A tồn tại:**

```
PRO:
  + Đơn giản nhất — một repo, một deploy, một team
  + Không có network latency giữa frontend và backend (cùng process)
  + Type safety end-to-end tự nhiên (Prisma → TypeScript → Component)
  + Vercel deployment trivial
  + Iteration speed cực nhanh ở giai đoạn đầu

CON:
  - Không scale tốt khi team lớn (ai cũng có access vào DB logic)
  - Database connection management phức tạp trên serverless
  - Không thể reuse API cho mobile app (nếu cần)
  - Vendor lock-in vào Vercel ecosystem (nếu không cẩn thận)
  - Một Next.js deploy = một monolith to tướng
```

**Signal trong job posting:**
> *"Next.js, Prisma, PostgreSQL, Vercel, Server Actions"*
→ Pattern A. Team nhỏ, move fast, full-stack engineers.

### 📐 Pattern B — Frontend Layer (Decoupled)

**Phổ biến ở:** Công ty đã có backend (Java, .NET, Python, Go), team frontend riêng

**Đặc điểm:** Next.js chỉ là **presentation layer**. Không có API Routes (hoặc rất ít). Không có DB access. Tất cả data từ external API.

```
enterprise-frontend/
├── app/
│   ├── layout.tsx
│   ├── page.tsx
│   └── products/
│       └── [id]/page.tsx
├── lib/
│   ├── api/
│   │   ├── client.ts        ← Configured API client (axios/fetch wrapper)
│   │   ├── products.ts      ← Product API calls
│   │   └── users.ts         ← User API calls
│   └── types/
│       ├── product.ts       ← Types mirror backend DTOs
│       └── user.ts
└── next.config.ts
# KHÔNG CÓ:
# - /prisma
# - /actions (hoặc rất ít)
# - /app/api (hoặc chỉ proxy routes)
```

```typescript
// lib/api/client.ts — Configured API client
// Đây là pattern rất phổ biến ở enterprise

const BASE_URL = process.env.API_URL; // Internal backend URL

// Server-side API calls (trong Server Components)
export async function serverFetch<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const response = await fetch(`${BASE_URL}${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      // Service-to-service auth — không phải user token
      'X-Service-Key': process.env.SERVICE_KEY!,
      ...options?.headers,
    },
  });

  if (!response.ok) {
    throw new Error(`API Error: ${response.status} ${endpoint}`);
  }

  return response.json();
}

// Client-side API calls (trong Client Components)
export async function clientFetch<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  // Next.js API route đóng vai trò PROXY
  // Không expose backend URL ra client
  const response = await fetch(`/api/proxy${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...options?.headers,
    },
    credentials: 'include', // Gửi session cookie
  });

  if (!response.ok) throw new Error(`API Error: ${response.status}`);
  return response.json();
}
```

```typescript
// app/api/proxy/[...path]/route.ts
// Proxy pattern — Next.js chỉ forward request
// Pattern B thường có proxy như thế này
import { NextRequest, NextResponse } from 'next/server';
import { auth } from '@/lib/auth';

export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ path: string[] }> }
) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  // Forward to backend với user token
  const { path } = await params;
  const endpoint = path.join('/');
  const url = `${process.env.API_URL}/${endpoint}${request.nextUrl.search}`;

  const response = await fetch(url, {
    headers: {
      Authorization: `Bearer ${session.accessToken}`, // Forward user token
      'X-User-Id': session.user.id,
    },
  });

  const data = await response.json();
  return NextResponse.json(data, { status: response.status });
}

// POST, PUT, DELETE tương tự...
export async function POST(request: NextRequest, { params }: { params: Promise<{ path: string[] }> }) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  const { path } = await params;
  const body = await request.json();

  const response = await fetch(`${process.env.API_URL}/${path.join('/')}`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${session.accessToken}`,
    },
    body: JSON.stringify(body),
  });

  const data = await response.json();
  return NextResponse.json(data, { status: response.status });
}
```

**Tại sao Pattern B tồn tại:**

```
PRO:
  + Backend team có thể làm việc độc lập (Go team, Java team...)
  + Next.js chỉ quan tâm đến UI — không bị lẫn business logic
  + Backend có thể serve nhiều client (web, mobile, partner API)
  + Dễ swap Next.js sang framework khác nếu cần
  + Security tốt hơn: DB credentials không cần ở frontend layer

CON:
  - Thêm một network hop (browser → Next.js → Backend)
  - Type duplication: phải tạo TS types mirror backend DTOs
  - Two team coordination: API contract thay đổi → frontend bị ảnh hưởng
  - Local development phức tạp hơn (cần chạy cả backend)
```

**Signal trong job posting:**
> *"Next.js frontend, REST API consumption, Java/Spring backend (separate team)"*
> *"Experience with API integration, OpenAPI/Swagger"*
→ Pattern B. Frontend-only role, backend là black box.

### 📐 Pattern C — BFF (Backend for Frontend)

**Phổ biến ở:** Công ty có microservices, cần aggregation layer, team medium-large

**Đặc điểm:** Next.js đóng vai trò **BFF** — nhận request từ client, gọi nhiều microservices, tổng hợp data, trả về.

```
BFF không phải là "proxy" — BFF có BUSINESS LOGIC:
  - Aggregation: Lấy data từ 3 services, merge thành 1 response
  - Transformation: Convert data format phù hợp với UI cần
  - Orchestration: Gọi service A, dùng kết quả gọi service B
  - Auth gateway: Verify token, enrich request với user context
```

```
bff-app/
├── app/
│   ├── layout.tsx
│   └── [feature]/page.tsx
├── app/api/
│   ├── dashboard/
│   │   └── route.ts      ← Aggregate từ 3 microservices
│   ├── checkout/
│   │   └── route.ts      ← Orchestrate cart + payment + inventory
│   └── search/
│       └── route.ts      ← Query search service + personalization
└── services/              ← Typed clients cho từng microservice
    ├── product-service.ts
    ├── inventory-service.ts
    ├── user-service.ts
    ├── order-service.ts
    └── recommendation-service.ts
```

```typescript
// services/product-service.ts — Typed client cho Product Microservice
const PRODUCT_SERVICE_URL = process.env.PRODUCT_SERVICE_URL;

interface Product {
  id: string;
  name: string;
  price: number;
  categoryId: string;
}

export const productService = {
  async getById(id: string): Promise<Product> {
    const res = await fetch(`${PRODUCT_SERVICE_URL}/products/${id}`, {
      headers: { 'X-Service-Name': 'nextjs-bff' },
      // Service mesh auth (mTLS trong production)
    });
    if (!res.ok) throw new Error(`Product not found: ${id}`);
    return res.json();
  },

  async getMany(ids: string[]): Promise<Product[]> {
    // Batch endpoint — tránh N+1 calls
    const res = await fetch(`${PRODUCT_SERVICE_URL}/products/batch`, {
      method: 'POST',
      body: JSON.stringify({ ids }),
      headers: {
        'Content-Type': 'application/json',
        'X-Service-Name': 'nextjs-bff',
      },
    });
    return res.json();
  },
};
```

```typescript
// app/api/dashboard/route.ts — BFF Aggregation
// Đây là trái tim của Pattern C
import { NextRequest, NextResponse } from 'next/server';
import { auth } from '@/lib/auth';
import { productService } from '@/services/product-service';
import { orderService } from '@/services/order-service';
import { analyticsService } from '@/services/analytics-service';
import { inventoryService } from '@/services/inventory-service';

export async function GET(request: NextRequest) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  const userId = session.user.id;

  // BFF PATTERN: Gọi nhiều services PARALLEL
  // Client chỉ cần 1 request thay vì 4 requests
  const [recentOrders, topProducts, analytics, lowStockAlerts] = await Promise.all([
    orderService.getRecent(userId, { limit: 5 }),
    productService.getTopSelling(userId, { limit: 10 }),
    analyticsService.getSummary(userId, { period: '30d' }),
    inventoryService.getLowStockAlerts(userId),
  ]);

  // BFF TRANSFORMATION: Merge và format data phù hợp với UI
  // Client không cần biết data đến từ 4 services khác nhau
  const dashboardData = {
    summary: {
      totalRevenue: analytics.revenue,
      orderCount: analytics.orderCount,
      topProductsCount: topProducts.length,
    },
    recentOrders: recentOrders.map(order => ({
      id: order.id,
      // Flatten nested structure — client nhận format đơn giản hơn
      customerName: `${order.customer.firstName} ${order.customer.lastName}`,
      total: order.lineItems.reduce((sum, item) => sum + item.price * item.quantity, 0),
      status: order.status,
      createdAt: order.createdAt,
    })),
    topProducts,
    alerts: lowStockAlerts.filter(alert => alert.quantity < 10),
    // Thêm computed fields mà không service nào cung cấp
    conversionRate: analytics.orderCount / analytics.visitCount,
  };

  return NextResponse.json(dashboardData);
}
```

**Tại sao Pattern C tồn tại:**

```
PRO:
  + Client chỉ cần 1 request thay vì nhiều requests riêng lẻ
  + Microservices thay đổi nội bộ không ảnh hưởng client
  + Dễ cache ở BFF layer
  + One place để handle cross-cutting concerns: logging, tracing, auth
  + Mobile và web có thể có BFF khác nhau (Mobile BFF vs Web BFF)

CON:
  - Thêm một layer cần maintain
  - BFF có thể trở thành bottleneck
  - Logic aggregation dễ bị trùng lặp nếu nhiều BFF
  - Team cần hiểu cả frontend lẫn service contracts
```

**Signal trong job posting:**
> *"Next.js, microservices architecture, API aggregation, GraphQL (optional)"*
> *"Experience with service-to-service communication"*
→ Pattern C. Thường gặp ở tech companies dùng microservices.

### 📐 Pattern D — Content Platform

**Phổ biến ở:** Media, news, blog, documentation site, marketing website

**Đặc điểm:** Tập trung vào **SEO, performance, CDN**. Pages được build trước (SSG/ISR). Content từ CMS.

```
content-platform/
├── app/
│   ├── layout.tsx
│   ├── page.tsx            ← Homepage, ISR 1 hour
│   ├── [category]/
│   │   └── page.tsx        ← Category page, ISR 30 min
│   └── [category]/[slug]/
│       └── page.tsx        ← Article page, ISR 5 min
├── lib/
│   └── cms/
│       ├── contentful.ts   ← Hoặc Sanity, Strapi, Prismic...
│       ├── queries.ts      ← GraphQL/REST queries for CMS
│       └── transforms.ts   ← Transform CMS data → app types
└── next.config.ts
# Đặc trưng: Không có user auth phức tạp
# Đặc trưng: Nhiều generateStaticParams
# Đặc trưng: revalidatePath/revalidateTag từ CMS webhook
```

```typescript
// lib/cms/contentful.ts — Headless CMS integration
import { createClient } from 'contentful';

const client = createClient({
  space: process.env.CONTENTFUL_SPACE_ID!,
  accessToken: process.env.CONTENTFUL_ACCESS_TOKEN!,
  // Preview token khác — dùng cho draft content
  ...(process.env.CONTENTFUL_PREVIEW_MODE && {
    accessToken: process.env.CONTENTFUL_PREVIEW_TOKEN!,
    host: 'preview.contentful.com',
  }),
});

// CMS query được wrap trong typed functions
export async function getArticle(slug: string) {
  const entries = await client.getEntries<ArticleFields>({
    content_type: 'article',
    'fields.slug': slug,
    include: 2, // Depth của linked entries
    limit: 1,
  });

  if (!entries.items.length) return null;
  return transformArticle(entries.items[0]);
}

export async function getAllArticleSlugs(): Promise<string[]> {
  const entries = await client.getEntries<ArticleFields>({
    content_type: 'article',
    select: ['fields.slug'], // Chỉ lấy slug — tối ưu bandwidth
    limit: 1000,
  });

  return entries.items.map(item => item.fields.slug);
}
```

```typescript
// app/[category]/[slug]/page.tsx — Article page, Pattern D
import { getArticle, getAllArticleSlugs } from '@/lib/cms/contentful';
import { notFound } from 'next/navigation';

// ISR: Tái tạo trang sau 5 phút nếu có request
export const revalidate = 300;

// Pre-render top articles tại build time
// Các bài khác sẽ được generate khi có request đầu tiên
export async function generateStaticParams() {
  const slugs = await getAllArticleSlugs();
  // Chỉ pre-render 200 bài phổ biến nhất — tránh build time quá dài
  return slugs.slice(0, 200).map(slug => ({
    category: 'news', // Simplified — thực tế phức tạp hơn
    slug,
  }));
}

export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const article = await getArticle(slug);
  if (!article) return {};

  return {
    title: `${article.title} | TechNews`,
    description: article.excerpt,
    openGraph: {
      title: article.title,
      description: article.excerpt,
      images: [{ url: article.coverImage.url }],
      type: 'article',
      publishedTime: article.publishedAt,
    },
```

```typescript
import { NextRequest, NextResponse } from 'next/server';
import { revalidatePath, revalidateTag } from 'next/cache';

export async function POST(request: NextRequest) {
  // Verify webhook secret — chống abuse
  const secret = request.headers.get('x-webhook-secret');
  if (secret !== process.env.CMS_WEBHOOK_SECRET) {
    return NextResponse.json({ error: 'Invalid secret' }, { status: 401 });
  }

  const body = await request.json();
  const { contentType, slug, action } = body;

  // Revalidate dựa trên loại content bị thay đổi
  if (contentType === 'article') {
    // Invalidate article page cụ thể
    revalidatePath(`/news/${slug}`);
    // Invalidate listing pages (có thể có article mới xuất hiện)
    revalidatePath('/news');
    revalidatePath('/');
    // Invalidate bằng tag — cách tốt hơn nếu dùng v13.4+
    revalidateTag('articles');
  }

  return NextResponse.json({
    revalidated: true,
    timestamp: new Date().toISOString(),
  });
}
```

### 📐 Pattern E — Multi-tenant SaaS

**Phổ biến ở:** B2B SaaS, platform products, agency tools

**Đặc điểm:** Một Next.js app phục vụ **nhiều tenants** (organizations). Mỗi tenant có thể có subdomain riêng, theme riêng, data isolation.

```typescript
// middleware.ts — Tenant detection — Trái tim của Pattern E
// Chạy trên EVERY request trước khi đến route handler
import { NextRequest, NextResponse } from 'next/server';

export function middleware(request: NextRequest) {
  const hostname = request.headers.get('host') || '';
  const url = request.nextUrl.clone();

  // Subdomain detection: acme.app.com → tenant: "acme"
  // Hoặc custom domain: acme.com → lookup in DB
  const mainDomain = process.env.NEXT_PUBLIC_ROOT_DOMAIN; // "app.com"

  let tenantSlug: string | null = null;

  if (hostname.endsWith(`.${mainDomain}`)) {
    // Subdomain pattern: acme.app.com
    tenantSlug = hostname.replace(`.${mainDomain}`, '');
  } else if (hostname !== mainDomain && hostname !== `www.${mainDomain}`) {
    // Custom domain: Cần lookup từ DB/cache
    // Thực tế: Cache tenant mapping để tránh DB hit mỗi request
    tenantSlug = getCachedTenantByDomain(hostname);
  }

  if (tenantSlug) {
    // Rewrite URL để route đến tenant-specific handler
    // User thấy URL: acme.app.com/dashboard
    // Next.js nhận: /tenants/acme/dashboard
    url.pathname = `/tenants/${tenantSlug}${url.pathname}`;
    return NextResponse.rewrite(url);
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)'],
};
```

## 9. Refactoring Scenario — Grow From Pattern A To Pattern B/C

**Kịch bản thực tế:** Startup bắt đầu với Pattern A. 2 năm sau: 50 người, cần mobile app, cần partner API, team backend riêng.

```
TRƯỚC (Pattern A — 2 năm trước):
  next-app/
  ├── app/
  ├── actions/   ← Business logic tất cả ở đây
  └── prisma/

VẤN ĐỀ:
  1. Mobile team muốn dùng cùng business logic → không thể, nó bị lock trong Server Actions
  2. Partner muốn API key access → cần expose API, nhưng actions không có auth API key
  3. Backend team mới muốn dùng Python → không tích hợp được
```

```
MIGRATION PATH: Pattern A → Pattern B (Incremental, không Big Bang)

BƯỚC 1: Extract API layer (Tháng 1-2)
  Tạo /app/api/v1/* Route Handlers mirror các Server Actions
  Mobile team bắt đầu dùng REST API mới
  Actions vẫn hoạt động song song

BƯỚC 2: Thin the Actions (Tháng 3-4)
  Server Actions không còn chứa business logic
  Actions chỉ call API endpoint nội bộ
  Business logic di chuyển vào API layer

BƯỚC 3: Extract Service (Tháng 5-6)
  API layer ready để tách ra thành separate service
  Hoặc giữ trong Next.js nếu không cần tách hoàn toàn
  Mobile, web, partner đều dùng cùng API
```

## 10. Architecture Notes

### Khi Nào Chọn Pattern Nào

```
Câu hỏi 1: Team size?
  1-10 người    →  Pattern A (ít overhead nhất)
  10-30 người   →  Pattern A → B transition
  30+ người     →  Pattern B hoặc C
  Enterprise    →  Pattern B hoặc C, tùy có microservices không

Câu hỏi 2: Có mobile app không?
  Không có, không plan  →  A (Server Actions OK)
  Có hoặc sẽ có        →  B (REST/GraphQL API từ đầu)

Câu hỏi 3: SEO có quan trọng không?
  Không (internal tool, dashboard)  →  Bất kỳ pattern nào
  Quan trọng (public site)          →  D (Content) hoặc SSG phần của A/B

Câu hỏi 4: Content có thay đổi thường xuyên không?
  Rất thường (news, blog)  →  D với ISR + webhook
  Ít thay đổi (docs)       →  D với thuần SSG

Câu hỏi 5: Có nhiều tenant/customer khác nhau không?
  Một tenant (internal)    →  A, B, hoặc C
  Nhiều tenant (SaaS)      →  E
```

### Điều Thực Sự Quyết Định Architecture

```
THỰC TẾ thường bị bỏ qua:

1. TEAM TOPOLOGY quan trọng hơn TECHNICAL PURITY
   "Pattern C (BFF) là tốt nhất" — nhưng nếu không có team riêng cho BFF,
   ai maintain nó? Ai on-call khi BFF down?

2. EXISTING INFRASTRUCTURE thường quyết định pattern
   Công ty đã có Java backend → Pattern B (không ai muốn rewrite)
   Công ty đã dùng AWS → Vercel ít likely hơn

3. MIGRATION COST có thể quá cao
   Pattern A → B tưởng dễ nhưng thường mất 6-12 tháng
   Nhiều công ty bị stuck ở Pattern A vì không có đủ resource migrate

4. "BORING TECHNOLOGY" thường win
   Pattern A với getServerSideProps/Pages Router vẫn serve hàng triệu user
   Không nhất thiết phải dùng Server Components mới thành công
```

### Junior Level

**Q1: Tại sao một công ty có thể dùng Next.js nhưng không có file nào trong `/app/api/`?**

<details>
<summary>Đáp án</summary>

Vì công ty đó đang dùng **Pattern B (Decoupled Frontend)** — Next.js chỉ là presentation layer, toàn bộ API được xử lý bởi một backend service riêng biệt (Java, Go, Python, .NET, etc.).

Trong pattern này:
- Server Components fetch trực tiếp từ external API URL trong môi trường server
- Client Components dùng một proxy route (ví dụ `/api/proxy/*`) để forward requests
- Không có business logic hay database access trong Next.js

Đây là pattern rất phổ biến tại các công ty lớn có team backend riêng, và là lý do tại sao "biết Next.js" không tự động có nghĩa là biết backend development.

</details>

---

**Q2: Developer mô tả project của họ: "Next.js app, PostgreSQL, Prisma, deploy trên Vercel, dùng Server Actions cho form submissions". Đây là pattern gì và trade-offs của nó?**

<details>
<summary>Đáp án</summary>

**Pattern A — Full-Stack Monolith.** Đây là pattern phổ biến nhất ở startup và small team.

**Trade-offs:**

*Ưu điểm:*
- Type safety end-to-end: Schema Prisma → TypeScript → Component
- Không có network hop giữa frontend và database (same process)
- Iteration speed rất nhanh
- Deploy trivial với Vercel

*Nhược điểm:*
- Database connection pooling là vấn đề trên serverless (mỗi Vercel function = new Prisma instance). Cần setup connection pooler (PgBouncer, Supabase pooler, Neon).
- Không thể reuse API cho mobile app nếu dùng Server Actions — phải build REST API riêng sau này
- Scale team khó: Khi team lớn, mọi người có access vào database logic, dễ inconsistent patterns
- Vendor lock-in vào Vercel ecosystem nếu không setup standalone output

</details>

### Mid Level

**Q3: Company có 3 microservices: user-service, product-service, order-service. CTO muốn dùng Next.js. Bạn sẽ suggest kiến trúc gì và tại sao?**

<details>
<summary>Đáp án</summary>

**Pattern C — BFF (Backend for Frontend).**

**Lý do:**

Với 3+ microservices riêng biệt, một client phải gọi nhiều services cho một trang (ví dụ order history cần cả user-service lẫn order-service lẫn product-service). Không có BFF dẫn đến:
- Client phải biết về tất cả service URLs
- Nhiều network round-trips từ browser
- Không có nơi tập trung xử lý auth token

**BFF implementation với Next.js:**

- `/app/api/dashboard/route.ts` → Gọi cả 3 services song song với `Promise.all()`, merge response
- `/services/*.ts` → Typed clients cho từng microservice
- Middleware xử lý auth: verify JWT, add user context vào service calls
- Service-to-service communication dùng internal URLs (không expose qua internet)

**Lưu ý quan trọng khi pitch:**
- BFF cần team riêng maintain (không phải frontend team hoặc backend team làm thêm)
- Cần monitoring và alerting cho BFF layer (nó là single point of failure)
- Circuit breaker pattern nếu một service down không nên làm cả page crash

</details>

### Senior Level

**Q4: Bạn join một team đang dùng Pattern A (Full-stack Monolith). 1 năm sau, company muốn launch mobile app. Bạn sẽ migrate như thế nào mà không break production?**

<details>
<summary>Đáp án</summary>

**Incremental migration, không Big Bang rewrite.**

**Giai đoạn 1 — Tạo API layer song song (2-3 tháng):**
- Tạo `/app/api/v1/*` Route Handlers expose business logic dưới dạng REST
- API handlers call cùng database queries với Server Actions
- Mobile team bắt đầu dùng API mới, web vẫn dùng Server Actions cũ
- Viết API documentation (OpenAPI/Swagger)

**Giai đoạn 2 — Thin the Actions (1-2 tháng):**
- Server Actions refactor để gọi internal API (`/api/v1/*`) thay vì DB trực tiếp
- Đây là bridge: Web dùng Actions, mobile dùng REST, nhưng cùng underlying logic
- Thêm API key auth cho mobile requests

**Giai đoạn 3 — Evaluate và decide (ongoing):**
- Nếu mobile team satisfised và API stable → Có thể giữ nguyên (API trong Next.js vẫn OK)
- Nếu cần tách hoàn toàn → Extract API sang separate service (NestJS, FastAPI, etc.), Next.js thành Pattern B

**Rủi ro phải monitor:**
- Data consistency khi cả Actions và API đều mutate DB
- Auth token management: Session cookie (web) vs API token (mobile)
- Rate limiting và API versioning từ ngày đầu

</details>

**Q5: "Next.js trên Vercel rất đắt khi scale. Công ty muốn self-host bằng Docker nhưng vẫn muốn giữ ISR và Image Optimization. Điều này khả thi không? Những gì bị mất?"**

<details>
<summary>Đáp án</summary>

**Khả thi — với một số lưu ý quan trọng.**

**Setup cơ bản:**
```typescript
// next.config.ts
const nextConfig: NextConfig = {
  output: 'standalone', // ← BẮT BUỘC cho Docker deployment
};
```

**Những gì hoạt động khi self-host:**
- ISR: Hoạt động, nhưng cache lưu trên local file system của container
- `revalidatePath`/`revalidateTag`: Hoạt động
- Image Optimization (`next/image`): Hoạt động (Next.js có built-in image server)

**Những gì BỊ MẤT hoặc CẦN SETUP THÊM:**
1. **Edge Runtime functions** — Vercel Edge Network không có khi self-host. Middleware chạy trên Node.js runtime, không phải Edge.
2. **ISR với multiple instances** — Nếu deploy 3 Docker containers, mỗi container có cache riêng → inconsistent ISR. Cần shared cache layer (Redis) để sync.
3. **Automatic scaling** — Vercel tự scale. Docker cần Kubernetes hoặc ECS setup.
4. **Analytics và Speed Insights** — Vercel-specific, không khả dụng.
5. **Automatic SSL, CDN, DDoS protection** — Phải tự setup (Cloudflare, Nginx, etc.).

**Về ISR với multiple instances — đây là vấn đề nghiêm trọng nhất:**
```
Container A: Article "ABC" → Cache HIT (fresh)
Container B: Article "ABC" → Cache MISS (chưa có cache)
→ Load balancer gửi user đến B → User thấy stale content

Fix: Dùng Redis làm shared cache, hoặc sticky sessions trên load balancer
```

**Bottom line:** Self-host được, nhưng phải tự build nhiều infrastructure mà Vercel cung cấp sẵn. Tính toán TCO (Total Cost of Ownership) bao gồm: engineer time setup infrastructure, monitoring, on-call.

</details>

## 13. Summary

### Bản Đồ 5 Patterns

```
Pattern A — Full-Stack Monolith
  Nhận diện:  prisma/ + actions/ + Vercel deployment
  Tốt cho:    Startup, MVP, small team
  Risk:       DB connections, không reusable API

Pattern B — Decoupled Frontend
  Nhận diện:  lib/api/client.ts, KHÔNG có prisma/
  Tốt cho:    Công ty có existing backend, enterprise
  Risk:       Type duplication, coordination overhead

Pattern C — BFF
  Nhận diện:  /services/*.ts, Promise.all trong API routes
  Tốt cho:    Microservices architecture
  Risk:       BFF là single point of failure

Pattern D — Content Platform
  Nhận diện:  generateStaticParams nhiều, CMS client, revalidate webhook
  Tốt cho:    Media, docs, marketing, blog
  Risk:       ISR invalidation complexity, multi-instance cache

Pattern E — Multi-tenant SaaS
  Nhận diện:  middleware.ts detect subdomain, tenant DB queries
  Tốt cho:    B2B SaaS, platform products
  Risk:       Data isolation, tenant cache pollution
```

### Câu Hỏi Phải Hỏi Khi Join Công Ty Mới

```
1. "Backend là gì? Team backend riêng hay full-stack?"
   → Xác định Pattern B/C (có backend riêng) vs A (monolith)

2. "Deploy như thế nào? Vercel hay Docker?"
   → Quyết định nhiều thứ: Middleware có dùng Edge không? ISR setup?

3. "Có mobile app không hay chỉ web?"
   → Pattern A có thể không đủ nếu mobile app đang planned

4. "Database access như thế nào?"
   → ORM nào? Connection pooling? (Quan trọng cho serverless)

5. "Migration plan từ Pages Router sang App Router đang ở giai đoạn nào?"
   → Biết mình sẽ làm việc với code era nào (bài M0.3)
```

### Điều Quan Trọng Nhất Của Bài Này

Tutorial dạy bạn xây **một** thứ.

Production là bạn phải hiểu **tại sao** thứ đó được xây theo cách đó, và **khi nào** nên xây theo cách khác.

Biết 5 patterns này không phải để bạn "chọn đúng pattern". Đó là để bạn **đọc được context** khi bước vào bất kỳ codebase nào — hiểu những quyết định đã được đưa ra, tại sao, và trade-offs nào đang được chấp nhận.

Đó là tư duy của engineer, không phải coder.
