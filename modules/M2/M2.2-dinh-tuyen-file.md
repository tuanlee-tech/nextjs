# M2.2 — Định tuyến File: Quy ước thư mục `pages/`

## 1. Lesson Goal

M2.1 đã cho bạn quy tắc mapping cơ bản: file → URL, 1:1. Nhưng quy tắc đơn giản đó **vỡ** ngay khi nhiều file có thể cùng khớp một URL. Sau bài này bạn phải trả lời được:

- Khi `pages/post/create.js` và `pages/post/[id].js` cùng tồn tại, ai thắng khi truy cập `/post/create`? Theo thứ tự ưu tiên nào?
- Tại sao Next.js **build fail** khi bạn đặt `pages/blog/[id].js` và `pages/blog/[slug].js` cùng lúc, dù về mặt code mỗi file hoàn toàn hợp lệ riêng lẻ?
- `pageExtensions` trong `next.config.js` giải quyết đúng vấn đề gì đã nêu ở M2.1 (file thường vô tình thành route), và cái giá phải trả là gì?
- Shallow Routing (`router.push(..., { shallow: true })`) khác gì với điều hướng route bình thường, và vì sao nó là một đặc trưng riêng của Pages Router?

## 2. Evolution Context

**Một chi tiết lịch sử quan trọng giải thích tại sao chương trình này chọn mốc Next.js 9 làm điểm khởi đầu "Pages Router Era":** trước Next.js 9, **không có Dynamic Routes**. Muốn một trang hiển thị theo tham số, bạn phải dùng query string thuần (`/post?id=123`) hoặc tự dựng một custom Express server đứng trước Next.js để parse URL rồi render trang tương ứng bằng tay.

**Pain Point gốc:** Query string (`?id=123`) không có hình dạng URL "đẹp" (`/post/123`), không tốt cho SEO, không match được thói quen người dùng web, và đòi hỏi code rewrite-URL bằng tay nếu muốn URL sạch.

**Next.js 9 (10/2019)** giới thiệu **Dynamic Route Segments** (`[param].js`) như tính năng chủ lực của bản phát hành — chính là tính năng biến file-based routing từ "chỉ xử lý được URL tĩnh" thành "xử lý được toàn bộ resource có tham số", và đây là lý do M2 của khóa học này bắt đầu chính xác từ Next.js 9.

**Nhưng giải quyết được "có tham số trong URL" lại sinh ra một bài toán mới:** khi route tree có cả static, dynamic, và (sau đó) catch-all segments, **ai khớp trước khi nhiều thứ cùng khớp một URL?** Nếu không có một thuật toán rõ ràng, file-based routing sẽ mất chính lợi thế nó từng có — "đọc file là biết route" — vì bạn phải đoán xem Next.js ưu tiên file nào.

**Cách mới:** Next.js định nghĩa một **thứ tự ưu tiên cố định, không phụ thuộc thứ tự khai báo hay alphabet**: cụ thể nhất thắng trước.

## 3. Mental Model

Đừng nghĩ route resolution là "Next.js thử từng file theo thứ tự ngẫu nhiên cho đến khi khớp". Hãy nghĩ nó là **một cây quyết định có phân cấp độ cụ thể (specificity), luôn duyệt từ cụ thể nhất đến tổng quát nhất:**

```
Mức độ cụ thể:  Static route  >  Dynamic route  >  Catch-all route
                (literal path)   ([param])         ([...param])
```

Và một ràng buộc đi kèm bắt buộc phải nhớ: **tại một vị trí (segment position) trong cây route, chỉ được tồn tại MỘT tên tham số động duy nhất.** Hãy hình dung mỗi vị trí trong cây giống một ô trong từ điển — ô đó chỉ được gán một khóa (key) duy nhất; bạn không thể vừa gọi nó là `id` vừa gọi nó là `slug` cùng lúc, vì về bản chất hai file đó đang tranh nhau quyền sở hữu **cùng một vị trí URL**.

## 4. Core Concepts

**4.1. Thứ tự ưu tiên khi nhiều route cùng khớp**

```
pages/post/create.js        → static, ƯU TIÊN CAO NHẤT
pages/post/[id].js          → dynamic, ưu tiên trung bình
pages/post/[...all].js      → catch-all, ƯU TIÊN THẤP NHẤT
```

Truy cập `/post/create` → khớp `create.js`, **không bao giờ** rơi vào `[id].js` với `id === 'create'`, dù về lý thuyết `[id].js` cũng "khớp được" nếu xét tách biệt. Next.js luôn ưu tiên route cụ thể nhất tồn tại trước khi xét tới route tổng quát hơn.

**4.2. Ràng buộc tên tham số phải đồng nhất theo vị trí**

```
pages/blog/[id].js
pages/blog/[slug].js
```

```
Error: You cannot use different slug names for the same dynamic path
('id' !== 'slug'). Use either 'id' or 'slug', not both.
```

Đây **không phải bug**, mà là một ràng buộc chủ ý: route tree của Next.js gán **một** tên thuộc tính (property key) cho `query` object tại mỗi vị trí — nếu cho phép hai tên khác nhau, hai file thực chất đang định nghĩa **hai sự thật khác nhau** cho cùng một không gian URL (`/blog/:???`), điều này không thể giải quyết một cách xác định (deterministic).

**4.3. Nested Dynamic Segments**

```
pages/posts/[postId]/comments/[commentId].js
                                              → /posts/:postId/comments/:commentId
```

```tsx
// pages/posts/[postId]/comments/[commentId].js
import { useRouter } from 'next/router';

export default function CommentDetail() {
  const { postId, commentId } = useRouter().query;
  return <p>Bài {postId}, bình luận {commentId}</p>;
}
```

**4.4. `index.js` trong folder động — mở rộng một resource thành nhiều sub-route**

```
pages/posts/[id]/index.js   → /posts/:id           (tương đương pages/posts/[id].js)
pages/posts/[id]/edit.js    → /posts/:id/edit
pages/posts/[id]/comments.js → /posts/:id/comments
```

Đây là pattern phổ biến khi một "resource" (bài viết, đơn hàng, user...) cần nhiều route con cùng chia sẻ một tham số động — gom vào một folder `[id]/` rõ ràng hơn việc đặt rời rạc nhiều file `[id].edit.js`, `[id].comments.js` ở cùng cấp.

**4.5. `pageExtensions` — giải pháp thực dụng cho rủi ro đã nêu ở M2.1**

Ở M2.1, ta đã thấy rủi ro: mọi file trong `pages/` đều là route, không có private folder. `pageExtensions` trong `next.config.js` là cách cộng đồng Next.js thực sự dùng để giả lập việc "đánh dấu file nào mới là route":

```js
// next.config.js
module.exports = {
  pageExtensions: ['page.tsx', 'page.ts', 'page.jsx', 'page.js'],
};
```

Sau khi cấu hình, **chỉ file có đuôi `.page.tsx` (v.v.) mới được coi là route** — mọi file khác, dù nằm ngay trong `pages/`, không còn bị framework công khai hóa thành URL:

```
pages/
  product/
    [id].page.tsx        ← route thật: /product/:id
    [id].test.tsx        ← KHÔNG còn là route, an toàn để colocate unit test
    useProductData.ts     ← KHÔNG còn là route, một custom hook hỗ trợ ngay cạnh
```

> ⚠️ Lưu ý bắt buộc: cấu hình này ảnh hưởng **toàn dự án**, bao gồm cả `_app`, `_document`, và mọi file trong `pages/api/` — phải đổi tên đồng bộ thành `_app.page.tsx`, `_document.page.tsx`, `pages/api/users.api.ts`, nếu không các file đặc biệt này sẽ không còn được framework nhận diện đúng.

**4.6. Programmatic Navigation & Shallow Routing — đặc trưng riêng của Pages Router**

```tsx
import { useRouter } from 'next/router';

function FilterPanel() {
  const router = useRouter();

  function applyFilter(category: string) {
    router.push(
      { pathname: '/products', query: { category } },
      undefined,
      { shallow: true }, // chỉ đổi URL + query, KHÔNG re-run getServerSideProps/getStaticProps
    );
  }

  return <button onClick={() => applyFilter('shoes')}>Lọc: Giày</button>;
}
```

`shallow: true` cập nhật URL hiển thị trên address bar và `router.query`, đồng bộ với nút back/forward của browser, **nhưng không kích hoạt lại vòng đời lấy dữ liệu** của trang (`getServerSideProps`/`getStaticProps` không chạy lại). Chỉ hoạt động khi điều hướng **trong cùng một page component** (không đổi sang component khác). Hữu ích cho filter, tab, phân trang client-side mà không muốn full data refetch — sẽ thấy lại tầm quan trọng của giới hạn này khi học `getServerSideProps` ở M2.4.

## 5. Legacy Recognition

- Thấy `pageExtensions` tùy biến trong `next.config.js`, kèm file tên `*.page.tsx` rải khắp `pages/` → đây là dấu hiệu rõ ràng team đã từng đối mặt đúng vấn đề "file thường vô tình thành route" ở M2.1 và chọn giải pháp thực dụng này — **không phải** quy ước ngẫu nhiên hay lỗi gõ nhầm tên file.
- Thấy `router.push(..., { shallow: true })` trong code, kèm URL đổi nhưng UI không có dấu hiệu loading lại data → đừng vội nghĩ đây là bug "quên fetch lại dữ liệu". Đây thường là chủ đích — đồng bộ URL với state UI (filter, pagination) mà không cần round-trip dữ liệu.
- Thấy cấu trúc `pages/<resource>/[id]/index.js` cùng các file con `edit.js`, `settings.js` cùng cấp `[id]/` → đây là cách tổ chức route theo resource có chủ đích, không phải file bị đặt lộn vị trí.
- Khi đọc thấy lỗi build "You cannot use different slug names..." trong log CI/CD của dự án cũ — đây gần như chắc chắn là hậu quả của việc 2 nhánh code (2 Pull Request riêng) cùng thêm dynamic route vào cùng một thư mục với tên tham số khác nhau, git merge "thành công" về text nhưng route tree raise lỗi ngay khi build lại (phân tích kỹ ở mục 8).

## 6. Modern Approach

App Router (M3) **giữ nguyên** quy tắc ưu tiên static > dynamic > catch-all và ràng buộc "một tên tham số mỗi vị trí" — đây **không phải** điểm khác biệt giữa hai router, mà là một nguyên lý xuyên suốt cả hai mô hình routing của Next.js.

Điểm khác biệt thật sự nằm ở cách giải quyết bài toán colocation (mục 4.5): App Router **không cần `pageExtensions`** — vì quy ước đã đổi gốc: chỉ file tên chính xác `page.tsx` mới công khai hóa route, mọi file khác trong cùng folder (`Button.tsx`, `useProductData.ts`, `Button.test.tsx`) **mặc định đã an toàn**, không cần đổi extension toàn dự án để đạt được điều đó.

Với nhu cầu UI phức tạp hơn (hiển thị modal trên route khác mà vẫn giữ nguyên URL nền, đồng bộ đúng với nút back) — điều Pages Router chỉ có thể giả lập bằng state cục bộ (không đồng bộ thật với URL) — App Router có Parallel Routes và Intercepting Routes (M3.11, M3.12) giải quyết tận gốc ở tầng routing, không phải tầng UI state.

## 7. Production Scenario

**Bối cảnh:** Một trang blog cần URL thân thiện SEO dạng slug (`/blog/cach-toi-hoc-nextjs`), nhưng hệ thống quản trị nội bộ (CMS, search) vẫn quen tra cứu bài viết theo numeric ID. Ý tưởng ban đầu của team: làm cả hai route `pages/blog/[id].js` (cho hệ thống nội bộ) và `pages/blog/[slug].js` (cho người dùng public) — **build fail ngay lập tức** vì vi phạm ràng buộc ở mục 4.2.

**Giải pháp đúng:** Chỉ giữ **một** route động duy nhất, đặt tên tổng quát hơn để phản ánh đúng bản chất "định danh, có thể là id hoặc slug":

```tsx
// pages/blog/[identifier].js
export async function getServerSideProps({ params }) {
  const { identifier } = params;
  const isNumericId = /^\d+$/.test(identifier);

  const post = isNumericId
    ? await getPostById(identifier)
    : await getPostBySlug(identifier);

  if (!post) return { notFound: true };

  // Chuẩn hóa: nếu người dùng vào bằng ID cũ, redirect 301 sang URL slug
  // chính thức — vừa giữ backward-compatible link cũ, vừa đảm bảo SEO
  // chỉ có MỘT URL canonical cho mỗi bài viết.
  if (isNumericId) {
    return {
      redirect: { destination: `/blog/${post.slug}`, permanent: true },
    };
  }

  return { props: { post } };
}
```

Kết quả: hệ thống nội bộ vẫn dùng được link dạng ID (tự động redirect sang slug chuẩn), người dùng public luôn thấy URL slug đẹp, và route tree không còn xung đột.

## 8. Debugging Scenario

**Hiện tượng:** Pull Request của dev A (thêm `pages/order/[orderId].js`) và Pull Request của dev B (thêm `pages/order/[id].js`, không biết PR của dev A) đều được review riêng, **đều pass CI riêng lẻ** (vì mỗi PR build độc lập trên nhánh của nó, chỉ có file của chính nó). Sau khi merge cả hai vào `main`, pipeline CI/CD báo lỗi:

```
Error: You cannot use different slug names for the same dynamic path
('orderId' !== 'id'). Use either 'orderId' or 'id', not both.
```

**Quy trình debug:**

1. Đọc kỹ thông báo lỗi — Next.js đã chỉ thẳng ra **vị trí xung đột** (route position `pages/order/`) và **hai tên tham số đang tranh nhau** (`orderId` vs `id`). Không cần đoán, chỉ cần `git log -- pages/order/` để tìm hai commit thêm hai file riêng biệt.
2. Hiểu rằng đây **không phải lỗi merge theo nghĩa git** (git không báo conflict vì hai file có tên khác nhau, hoàn toàn không đụng dòng code nào của nhau) — đây là lỗi ở tầng **route tree semantics**, một loại "conflict" mà git không có khả năng phát hiện.
3. Quyết định giữ tên nào (thường ưu tiên tên rõ nghĩa hơn, ví dụ `orderId` thay vì `id` chung), xóa file trùng, cập nhật lại mọi nơi gọi `router.query.id` thành `router.query.orderId` (hoặc ngược lại) trong cả hai nhánh code liên quan.
4. **Phòng ngừa về sau:** đây là lý do nhiều team viết hẳn quy ước tên tham số động (naming convention) vào style guide nội bộ — ví dụ "luôn dùng `[id]` cho mọi resource, không dùng `[entityId]` hay tên riêng" — để loại bỏ khả năng xung đột này ngay từ thiết kế, không phải xử lý hậu kỳ sau khi CI báo lỗi.

## 9. Refactoring Scenario

**Trước (hai dynamic route mâu thuẫn, build fail):**

```tsx
// pages/user/[id].js — xem theo numeric ID
export async function getServerSideProps({ params }) {
  const user = await getUserById(params.id);
  return { props: { user } };
}

// pages/user/[username].js — xem theo username, KHÔNG THỂ coexist với file trên
export async function getServerSideProps({ params }) {
  const user = await getUserByUsername(params.username);
  return { props: { user } };
}
```

**Sau (hợp nhất thành một route, phân biệt logic bên trong):**

```tsx
// pages/user/[identifier].js
export async function getServerSideProps({ params }) {
  const { identifier } = params;

  // Quy ước: ID nội bộ luôn là số nguyên dương, username không bao giờ
  // là một chuỗi số thuần — đây là cách phân biệt an toàn cho domain này
  const isNumericId = /^\d+$/.test(identifier);

  const user = isNumericId
    ? await getUserById(identifier)
    : await getUserByUsername(identifier);

  if (!user) return { notFound: true };
  return { props: { user } };
}

export default function UserProfile({ user }) {
  return <h1>{user.displayName}</h1>;
}
```

**Điểm khác biệt cốt lõi:** Logic phân biệt "đây là ID hay username" được chuyển từ **tầng routing** (vốn chỉ cho phép một tên tham số mỗi vị trí, không có cách "or" giữa hai pattern) xuống **tầng application logic**, nơi bạn có toàn quyền kiểm tra định dạng giá trị và quyết định cách xử lý — đây là nguyên tắc chung cần nhớ: khi route tree không cho phép thứ bạn muốn, đáp án thường là gộp route lại và đẩy nhánh rẽ logic vào trong handler, không phải tìm cách "lách" quy tắc routing.

## 10. Architecture Notes

- **Route tree statically analyzable = nền tảng cho build-time optimization.** Vì Next.js biết toàn bộ route tree ngay tại thời điểm build (xuất ra route manifest nội bộ), nó có thể quyết định trang nào tối ưu hóa tĩnh được (Automatic Static Optimization), trang nào cần code-splitting riêng, mà không cần chạy thử runtime. Đây là lý do ràng buộc "một tên tham số mỗi vị trí" tồn tại — nó giữ route tree **xác định (deterministic)** ngay từ build time, không phụ thuộc giá trị runtime.
- **`pageExtensions` là quyết định toàn dự án (project-wide), không thể áp dụng nửa vời.** Một khi đổi, mọi file trong `pages/` (kể cả `_app`, `_document`, `pages/api/`) phải tuân theo extension mới — đây là chi phí thực sự cần cân nhắc trước khi áp dụng cho một codebase lớn đang chạy production.
- **Liên kết tới M2.10 (Migration Strategy):** nếu dự án đang dùng `pageExtensions` tùy biến, cần kiểm tra kỹ tương thích khi bắt đầu cho `pages/` và `app/` coexist trong giai đoạn incremental migration — cấu hình này được thiết kế riêng cho `pages/`, không tự động áp dụng ý nghĩa tương đương cho `app/`.
- **Naming convention cho dynamic param nên là quyết định kiến trúc từ ngày đầu dự án**, viết rõ trong style guide team, không phải thứ để mỗi engineer tự chọn theo cảm tính — chi phí sửa lỗi xung đột tên tham số tăng theo số lượng route đã tồn tại trong dự án.

## 11. Interview Questions

**[Junior/Mid] Nếu `pages/shop/new.js` và `pages/shop/[id].js` cùng tồn tại, truy cập `/shop/new` sẽ render file nào? Vì sao?**

<details>
<summary>Đáp án</summary>

Render `pages/shop/new.js`. Next.js luôn ưu tiên static route (literal path) trước dynamic route, bất kể thứ tự khai báo hay tên file. `[id].js` chỉ được xét đến khi không có static route nào khớp chính xác với segment đó.
</details>

**[Mid] Vì sao Next.js không cho phép `pages/blog/[id].js` và `pages/blog/[slug].js` cùng tồn tại?**

<details>
<summary>Đáp án</summary>

Vì cả hai file đang định nghĩa cùng một vị trí trong route tree (`/blog/:???`), nhưng với hai tên thuộc tính khác nhau cho `query` object. Route tree của Next.js cần một tên tham số xác định, duy nhất cho mỗi vị trí ngay tại thời điểm build — cho phép hai tên khác nhau sẽ tạo ra một sự mơ hồ không thể giải quyết một cách xác định, nên framework chủ động raise lỗi build ngay từ đầu thay vì để hành vi runtime không nhất quán.
</details>

**[Mid/Senior] `pageExtensions` giải quyết vấn đề gì, và chi phí áp dụng nó là gì?**

<details>
<summary>Đáp án</summary>

`pageExtensions` cho phép giới hạn route chỉ còn những file có đuôi tùy biến (ví dụ `.page.tsx`), giúp colocate test file, custom hook, component phụ trợ ngay trong `pages/` mà không lo chúng vô tình trở thành route công khai — giải quyết đúng rủi ro đã nêu ở M2.1 (Pages Router không có private folder). Chi phí: đây là cấu hình toàn dự án, buộc phải đổi tên đồng bộ mọi page, `_app`, `_document`, và file trong `pages/api/` sang extension mới, không thể áp dụng từng phần.
</details>

**[Senior] Shallow Routing khác gì với điều hướng route thông thường? Khi nào nó KHÔNG hoạt động như mong đợi?**

<details>
<summary>Đáp án</summary>

Shallow Routing (`router.push(url, as, { shallow: true })`) chỉ cập nhật URL hiển thị và `router.query`, đồng bộ với lịch sử browser (back/forward), nhưng không re-run `getServerSideProps`/`getStaticProps` của trang — phù hợp cho filter, tab, phân trang không cần fetch lại dữ liệu server. Nó chỉ hoạt động khi điều hướng **trong cùng một page component đang mount** — nếu URL đích trỏ tới một page component khác (Next.js phải mount component mới), shallow routing không còn ý nghĩa, vì toàn bộ vòng đời trang vẫn chạy lại như điều hướng bình thường.
</details>

**[Staff/Architect] Tại sao việc route tree phải "statically analyzable" (xác định được tại build time) lại quan trọng, không chỉ là một chi tiết kỹ thuật vụn?**

<details>
<summary>Đáp án</summary>

Vì Next.js dựa vào việc biết toàn bộ route tree ngay tại build time để thực hiện các tối ưu hóa quan trọng: Automatic Static Optimization (xác định trang nào không cần data fetching động để pre-render tĩnh), tạo route manifest cho `next start`, và đảm bảo code-splitting theo route hoạt động đúng. Nếu route tree có thể mơ hồ tùy theo giá trị runtime (ví dụ hai tên tham số khác nhau cho cùng vị trí), framework không thể đưa ra quyết định tối ưu hóa một cách an toàn tại build time — đây là lý do ràng buộc tên tham số tồn tại như một nguyên lý thiết kế, không phải hạn chế tùy tiện.
</details>

## 12. Exercises

**Bài 1 (Mid):** Cho cấu trúc file sau, xác định file nào xử lý từng URL được hỏi.

```
pages/
  shop/
    index.js
    new.js
    [id].js
    [...rest].js
```

URL cần xác định: `/shop`, `/shop/new`, `/shop/42`, `/shop/a/b/c`.

<details>
<summary>Đáp án</summary>

`/shop` → `pages/shop/index.js` (route gốc của folder, không có segment con nào).

`/shop/new` → `pages/shop/new.js` (static route, ưu tiên cao nhất, dù `[id].js` về lý thuyết cũng khớp được với `id = 'new'`).

`/shop/42` → `pages/shop/[id].js` (không có static route nào tên `42`, dynamic route khớp).

`/shop/a/b/c` → `pages/shop/[...rest].js` (nhiều segment, chỉ catch-all mới khớp được; `[id].js` chỉ khớp đúng một segment).
</details>

**Bài 2 (Senior):** Team bạn đang có `pages/ticket/[ticketId].js` (xử lý ticket hỗ trợ khách hàng theo ID nội bộ). Một dev khác, không biết route này tồn tại, vừa tạo Pull Request thêm `pages/ticket/[code].js` để hỗ trợ tra cứu ticket bằng mã code công khai (gửi qua email cho khách hàng). Hãy dự đoán điều gì xảy ra khi merge, và đề xuất hướng xử lý đúng.

<details>
<summary>Đáp án</summary>

Khi merge cả hai file vào cùng nhánh, `next build` (hoặc `next dev`) sẽ báo lỗi: "You cannot use different slug names for the same dynamic path ('ticketId' !== 'code')" — vì cả hai file đang tranh nhau định nghĩa cùng một vị trí route `pages/ticket/:???` với hai tên tham số khác nhau.

Hướng xử lý đúng: hợp nhất thành một file duy nhất, ví dụ `pages/ticket/[identifier].js`, rồi trong `getServerSideProps` kiểm tra định dạng giá trị nhận được (ví dụ ticket ID nội bộ luôn là số nguyên hoặc theo một pattern cố định như `TCK-xxxxx`, còn mã code công khai theo định dạng khác) để quyết định tra cứu theo ID nội bộ hay theo code công khai — áp dụng đúng nguyên tắc đã học ở mục 9: đẩy nhánh rẽ logic vào application code khi route tree không cho phép hai tên tham số cùng vị trí.
</details>

## 13. Summary

- Khi nhiều route cùng có thể khớp một URL, Next.js áp dụng thứ tự ưu tiên cố định: static route > dynamic route > catch-all route — không phụ thuộc thứ tự khai báo file.
- Tại mỗi vị trí trong route tree, chỉ được tồn tại **một** tên tham số động duy nhất — hai file dùng tên khác nhau ở cùng vị trí sẽ gây lỗi build, vì đây là một loại "conflict" mà git không phát hiện được.
- `pageExtensions` trong `next.config.js` là giải pháp thực dụng cho rủi ro "mọi file là route" đã nêu ở M2.1, nhưng là một quyết định toàn dự án, ảnh hưởng cả `_app`, `_document`, `pages/api/`.
- Nested dynamic segments và `index.js` trong folder động cho phép tổ chức nhiều route con quanh một resource có tham số.
- Shallow Routing đồng bộ URL với UI state mà không re-run data fetching lifecycle của trang — chỉ hoạt động trong cùng một page component đang mount.
- Route tree statically analyzable là nền tảng cho các tối ưu hóa build-time của Next.js (Automatic Static Optimization, route manifest) — đây là lý do các ràng buộc tưởng chừng "cứng nhắc" của file-based routing tồn tại có chủ đích.
