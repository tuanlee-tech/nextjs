# M2.3 — `getInitialProps`: Universal Data Fetching (Legacy)

## 1. Lesson Goal

Sau bài này bạn phải:

- Giải thích được tại sao `getInitialProps` ra đời và vấn đề nó giải quyết
- Hiểu "universal execution" nghĩa là gì và tại sao nó nguy hiểm
- Đọc được legacy code dùng `getInitialProps` và nhận ra các anti-pattern
- Biết cách `_app.js` làm thay đổi toàn bộ hành vi của `getInitialProps`
- Hiểu lý do Next.js deprecated nó và lộ trình migration

---

## 2. Evolution Context

Để hiểu `getInitialProps`, bạn cần quay về năm 2016–2017 — thời điểm Next.js mới ra đời.

**Vấn đề của React thuần lúc đó:**

```
User request → Server gửi về HTML rỗng → Browser tải JS → React mount → 
fetch data → render lại → User thấy nội dung
```

Đây là CSR (Client-Side Rendering) thuần. Hệ quả:

- SEO không hoạt động (Googlebot thấy HTML rỗng)
- Người dùng thấy blank screen trong khi JS tải
- Không thể pre-render dữ liệu từ database

**Giải pháp thời đó:**

Cộng đồng đang dùng các hack như `react-helmet`, `react-snap`, hoặc tự viết SSR server bằng Express. Đây là công việc rất phức tạp và dễ sai.

Next.js ra đời với cam kết: *"SSR out of the box, zero config"*. Và `getInitialProps` là cơ chế trung tâm để làm điều đó.

**Timeline:**

| Version | Sự kiện |
|---------|---------|
| Next.js 1.0 (2016) | `getInitialProps` xuất hiện lần đầu |
| Next.js 3–8 | API duy nhất để fetch data, cực kỳ phổ biến |
| Next.js 9.3 (2020) | `getServerSideProps` và `getStaticProps` ra đời |
| Next.js 9.3+ | `getInitialProps` bị discouraged (không deprecated hoàn toàn) |
| Next.js 13+ | App Router ra đời, `getInitialProps` không tồn tại trong paradigm mới |

## 3. Mental Model

`getInitialProps` được thiết kế theo một triết lý đơn giản và nghe có vẻ hợp lý:

> "Hàm này chạy trước khi component render. Nếu request đến từ server, chạy trên server. Nếu client navigate bằng `<Link>`, chạy trên client."

Đây gọi là **"Universal" hay "Isomorphic" execution** — cùng một đoạn code, chạy được trên cả hai môi trường.

```
Lần đầu vào trang (hard navigation):
Browser → Server → getInitialProps chạy trên Server → HTML đã có data → gửi về Browser

Chuyển trang bằng Link (soft navigation):
Browser → getInitialProps chạy trên Browser → fetch API → render
```

Nghe hay đấy. Nhưng thực tế:

```
getInitialProps có thể chạy trên SERVER  →  có Node.js APIs
getInitialProps có thể chạy trên BROWSER →  không có Node.js APIs
```

Bạn phải viết code đồng thời satisfy cả hai môi trường. Đây là nguồn gốc của vô số bug.

## 4. Core Concepts

### 4.1 Cú pháp cơ bản

```javascript
// pages/products.js
function ProductsPage({ products }) {
  return (
    <ul>
      {products.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  );
}

// Đây là static method, không phải hook hay lifecycle thông thường
ProductsPage.getInitialProps = async (context) => {
  // context chứa thông tin về request
  const { req, res, query, pathname, asPath } = context;

  const response = await fetch('https://api.example.com/products');
  const products = await response.json();

  // Phải return một plain object — object này trở thành props của component
  return { products };
};

export default ProductsPage;
```

**Quan sát quan trọng:** `getInitialProps` là static method gắn vào component, không phải export riêng. Đây là khác biệt lớn so với `getServerSideProps` sau này.

### 4.2 Context Object

```javascript
ProductsPage.getInitialProps = async (context) => {
  const {
    req,       // IncomingMessage — CHỈ có trên server. Undefined ở client!
    res,       // ServerResponse — CHỈ có trên server. Undefined ở client!
    query,     // URL query parameters (object)
    pathname,  // Route path, ví dụ: "/products/[id]"
    asPath,    // URL thực tế trên browser, ví dụ: "/products/123?ref=home"
    err,       // Error object nếu có lỗi rendering
  } = context;

  // Đây là pattern ĐÚNG để detect môi trường:
  const isServer = typeof window === 'undefined';
  // Hoặc:
  const isServer = !!req;

  return {};
};
```

**Đây là điểm quan trọng cần nhớ:** `req` và `res` có thể là `undefined`. Bất kỳ code nào đọc `req.headers` mà không check sẽ crash khi client-side navigation.

### 4.3 Universal Execution — Bug thực tế

```javascript
// ❌ Anti-pattern cực kỳ phổ biến trong legacy code
ProductsPage.getInitialProps = async ({ req }) => {
  // Code này crash khi user click <Link> để đến trang này
  // req là undefined trong browser navigation!
  const token = req.cookies.auth_token;
  
  const response = await fetch('/api/products', {
    headers: { Authorization: `Bearer ${token}` }
  });
  
  return { products: await response.json() };
};
```

```javascript
// ✅ Pattern đúng để handle universal context
ProductsPage.getInitialProps = async ({ req }) => {
  // Cần lấy auth token theo cách khác nhau tùy môi trường
  let token;
  
  if (req) {
    // Server: đọc từ cookie trong HTTP header
    token = req.cookies?.auth_token;
  } else {
    // Client: đọc từ document.cookie hoặc localStorage
    token = getCookieFromDocument('auth_token');
  }

  const baseUrl = req 
    ? `http://localhost:${process.env.PORT || 3000}` // Server cần absolute URL
    : '';                                              // Client dùng relative URL được
  
  const response = await fetch(`${baseUrl}/api/products`, {
    headers: { Authorization: `Bearer ${token}` }
  });
  
  return { products: await response.json() };
};
```

Nhìn code trên, bạn thấy vấn đề ngay: logic data fetching bị nhiễm bởi environment detection. Code trở nên khó đọc, khó test, khó maintain.

### 4.4 `getInitialProps` trong `_app.js`

Đây là phần **nguy hiểm nhất** và ít được hiểu nhất.

```javascript
// pages/_app.js
function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}

MyApp.getInitialProps = async (appContext) => {
  // appContext khác với page context!
  const { Component, ctx } = appContext;
  
  // Gọi getInitialProps của page nếu có
  const pageProps = await Component.getInitialProps?.(ctx) || {};
  
  // Fetch data dùng chung cho toàn app (ví dụ: user session)
  const userRes = await fetch(`${getBaseUrl(ctx.req)}/api/me`);
  const user = await userRes.json();
  
  return { pageProps, user };
};
```

**Hệ quả nghiêm trọng khi để `getInitialProps` trong `_app.js`:**

Next.js có tính năng **Automatic Static Optimization** — nếu một page không có `getInitialProps` hay `getServerSideProps`, Next.js tự động build nó thành static HTML. Tốt hơn nhiều về performance.

Nhưng nếu `_app.js` có `getInitialProps`, **toàn bộ app mất đi Automatic Static Optimization**. Mọi page đều bị forced sang SSR. Đây là một trong những performance trap phổ biến nhất trong legacy Next.js codebase.

```
Không có _app.getInitialProps:
  → Page không có getInitialProps/getServerSideProps → Static HTML ✅ (fast)
  → Page có getInitialProps → SSR per request

Có _app.getInitialProps:
  → MỌI page → SSR per request ❌ (toàn bộ app chậm hơn)
```

### 4.5 Serialization Constraint

```javascript
// getInitialProps phải return một plain, serializable object
// Vì data được serialize thành JSON và nhúng vào HTML

ProductsPage.getInitialProps = async () => {
  // ✅ OK — plain objects, arrays, primitives
  return {
    products: [{ id: 1, name: 'Product A' }],
    totalCount: 42,
    isLoaded: true,
  };
  
  // ❌ KHÔNG OK — không serialize được
  return {
    date: new Date(),           // Mất type, thành string
    error: new Error('oops'),   // Mất prototype chain
    fn: () => {},               // Functions không serialize được
    map: new Map(),             // Map không serialize được thành JSON
  };
};
```

Next.js sẽ nhúng data vào HTML như này:

```html
<script id="__NEXT_DATA__" type="application/json">
{
  "props": {
    "pageProps": {
      "products": [{ "id": 1, "name": "Product A" }]
    }
  }
}
</script>
```

Sau đó React dùng `__NEXT_DATA__` để hydrate mà không cần fetch lại. Nếu data không serialize được, hydration sẽ fail.

## 5. Legacy Recognition

Khi bạn nhìn vào một codebase, đây là cách nhận ra code đang dùng `getInitialProps`:

**Signal 1: Static method gắn vào component**
```javascript
// Luôn là pattern: ComponentName.getInitialProps = async (ctx) => { ... }
SomePage.getInitialProps = async (ctx) => { ... };
```

**Signal 2: `typeof window === 'undefined'` để detect môi trường**
```javascript
// Khi thấy pattern này trong data fetching = đang handle universal execution
const isServer = typeof window === 'undefined';
```

**Signal 3: Absolute URL cho fetch**
```javascript
// Server-side fetch cần absolute URL, không thể dùng /api/...
const baseUrl = ctx.req ? 'http://localhost:3000' : '';
fetch(`${baseUrl}/api/products`);
```

**Signal 4: `_app.js` có `getInitialProps`**
```javascript
// Đây là red flag về performance
MyApp.getInitialProps = async (appContext) => { ... };
```

**Signal 5: Manual `Component.getInitialProps` call trong `_app.js`**
```javascript
// Phải tự gọi thủ công, dễ quên và gây bug
const pageProps = await Component.getInitialProps(ctx);
```

## 6. Modern Approach

Từ Next.js 9.3, bạn nên dùng:

| Tình huống | Trước (legacy) | Sau (modern) |
|-----------|---------------|-------------|
| Cần data từ server mỗi request | `getInitialProps` | `getServerSideProps` |
| Cần data tại build time | `getInitialProps` với workaround | `getStaticProps` |
| Cần auth trước khi render | `getInitialProps` + check `req` | `getServerSideProps` + redirect |
| Data dùng chung toàn app | `_app.getInitialProps` | Layout component + server component (App Router) |

**Lý do `getServerSideProps` tốt hơn:**

```javascript
// getServerSideProps — LUÔN chạy trên server, không bao giờ trên client
export async function getServerSideProps(context) {
  const { req, res, query, params } = context;
  
  // req LUÔN tồn tại — không cần check typeof window
  const token = req.cookies.auth_token;
  
  // Có thể dùng Node.js APIs thoải mái
  const fs = require('fs');
  
  const products = await db.query('SELECT * FROM products');
  
  return {
    props: { products }, // Phải wrap trong { props: ... }
  };
}
```

**Sự khác biệt key:**

```
getInitialProps:
  - Chạy trên server (hard navigation) VÀ client (soft navigation)
  - Static method trên component
  - Return plain object trực tiếp
  - Có thể block Automatic Static Optimization

getServerSideProps:
  - CHỈ chạy trên server, KHÔNG BAO GIỜ chạy trên client
  - Export riêng biệt, tách khỏi component
  - Return { props: { ... } }
  - Không ảnh hưởng đến Automatic Static Optimization của các page khác
```

## 7. Production Scenario

**Scenario: Bạn join một startup đang chạy Next.js 10, app bị chậm bất thường**

Bạn mở codebase, thấy:

```javascript
// pages/_app.js — đây là vấn đề
import { getSession } from 'next-auth/client';

function MyApp({ Component, pageProps, session }) {
  return (
    <SessionProvider session={session}>
      <Component {...pageProps} />
    </SessionProvider>
  );
}

MyApp.getInitialProps = async (appContext) => {
  const appProps = await App.getInitialProps(appContext);
  const session = await getSession(appContext.ctx);
  return { ...appProps, session };
};
```

**Chẩn đoán:**

`_app.js` có `getInitialProps` → toàn bộ app mất Automatic Static Optimization → mọi page đều bị SSR dù không cần → server load cao, TTFB cao, cache không hiệu quả.

**Các trang bị ảnh hưởng:**

```javascript
// pages/about.js — trang hoàn toàn static, không có data
function AboutPage() {
  return <div>About Us</div>;
}
// Trang này đáng ra nên là Static HTML, được cache CDN
// Nhưng vì _app.getInitialProps, nó bị SSR mỗi request!
export default AboutPage;
```

**Fix:**

```javascript
// pages/_app.js — sau khi fix
import { SessionProvider } from 'next-auth/react';
import { getServerSideProps } from 'next-auth/next'; // Dùng session trong từng page nếu cần

function MyApp({ Component, pageProps: { session, ...pageProps } }) {
  return (
    <SessionProvider session={session}>
      <Component {...pageProps} />
    </SessionProvider>
  );
}

// KHÔNG có getInitialProps ở đây nữa
// Session được lấy ở từng page cần auth, không phải toàn bộ app

export default MyApp;
```

## 8. Debugging Scenario

**Bug:** User report rằng sau khi login và navigate đến `/dashboard`, họ thấy dashboard của người khác trong vài giây.

```javascript
// pages/dashboard.js — code có bug
DashboardPage.getInitialProps = async ({ req, query }) => {
  const userId = req?.cookies?.user_id || query.uid; // ← Bug ở đây
  
  const data = await fetchUserDashboard(userId);
  return { data };
};
```

**Phân tích bug:**

```
Lần 1: User A (id=1) load /dashboard
  → Hard navigation → req có cookies → userId = 1 ✅
  → Server trả về data của User A

User A click link đến /about rồi click back/navigate đến /dashboard
  → Soft navigation (client-side) → req là undefined
  → userId = query.uid → query.uid không có → userId = undefined
  → fetchUserDashboard(undefined) → có thể trả về cache của user trước? 
     hoặc default user? → BUG!
```

**Fix:**

```javascript
// pages/dashboard.js — sau khi fix
// Cách 1: Dùng getServerSideProps thay thế
export async function getServerSideProps({ req, res }) {
  // LUÔN có req, không bao giờ undefined
  const userId = req.cookies.user_id;
  
  if (!userId) {
    return {
      redirect: { destination: '/login', permanent: false }
    };
  }
  
  const data = await fetchUserDashboard(userId);
  return { props: { data } };
}
```

## 9. Refactoring Scenario

**Nhiệm vụ:** Migrate một page dùng `getInitialProps` sang `getServerSideProps`

**Trước:**

```javascript
// pages/product/[id].js — legacy code
import { useRouter } from 'next/router';

function ProductPage({ product, relatedProducts, error }) {
  const router = useRouter();

  if (error) return <div>Lỗi: {error}</div>;
  if (!product) return <div>Không tìm thấy</div>;

  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <RelatedProducts items={relatedProducts} />
    </div>
  );
}

ProductPage.getInitialProps = async ({ req, query }) => {
  // Phải handle cả server lẫn client
  const baseUrl = req ? `http://localhost:${process.env.PORT || 3000}` : '';
  const { id } = query;

  try {
    const [productRes, relatedRes] = await Promise.all([
      fetch(`${baseUrl}/api/products/${id}`),
      fetch(`${baseUrl}/api/products/${id}/related`),
    ]);

    if (!productRes.ok) {
      return { product: null, relatedProducts: [], error: 'Not found' };
    }

    const product = await productRes.json();
    const relatedProducts = await relatedRes.json();

    return { product, relatedProducts, error: null };
  } catch (err) {
    return { product: null, relatedProducts: [], error: err.message };
  }
};

export default ProductPage;
```

**Sau:**

```javascript
// pages/product/[id].js — sau khi refactor
// Không cần import useRouter nếu dùng params từ getServerSideProps
function ProductPage({ product, relatedProducts }) {
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <RelatedProducts items={relatedProducts} />
    </div>
  );
}

export async function getServerSideProps({ params, res }) {
  const { id } = params; // Dùng params thay vì query cho dynamic routes

  try {
    // Không cần baseUrl nữa — LUÔN chạy trên server
    // Gọi trực tiếp database/service thay vì đi qua HTTP
    const [product, relatedProducts] = await Promise.all([
      db.products.findById(id),
      db.products.findRelated(id),
    ]);

    if (!product) {
      return { notFound: true }; // Next.js built-in 404 handling
    }

    return {
      props: {
        product,
        relatedProducts,
      },
    };
  } catch (err) {
    // Có thể return error page hoặc throw
    throw err; // Next.js sẽ hiển thị error page
  }
}

export default ProductPage;
```

**Điểm cải thiện:**

1. Không còn `baseUrl` hack — gọi thẳng DB/service
2. `notFound: true` thay vì check thủ công
3. Error handling rõ ràng hơn
4. Code chỉ có một execution context (server), không cần guard clause
5. Dễ test hơn

## 10. Architecture Notes

### Tại sao Next.js không xóa hẳn `getInitialProps`?

Vì `getInitialProps` vẫn là API duy nhất hoạt động trong `_app.js` và `_document.js` đối với Pages Router. Nếu bạn cần inject data vào toàn bộ app (ví dụ: theme, locale), bạn vẫn cần nó ở `_app.js`.

Tuy nhiên, từ Next.js 13+ với App Router, `_app.js` và `_document.js` không còn tồn tại. Root Layout thay thế vai trò này, và Server Components đảm nhiệm việc fetch data.

### Trade-off của Universal Execution

```
Universal Execution (getInitialProps):
  Ưu: Một đoạn code duy nhất, DRY
  Nhược:
    - Khó debug (bug xảy ra ở đâu? server hay client?)
    - Code phức tạp hơn (phải guard cho cả hai môi trường)
    - Performance không tối ưu (JS bundle lớn hơn vì phải ship code fetch lên client)
    - Security risk (dễ vô tình leak server-only logic lên client)

Server-only Execution (getServerSideProps):
  Ưu: 
    - Execution context rõ ràng, predictable
    - Code đơn giản hơn
    - Có thể dùng Node.js APIs thoải mái
    - Không ship data fetching code xuống client
  Nhược:
    - Mỗi navigation đều tốn network round-trip đến server
    - Latency cao hơn so với client-side fetch (nếu data thay đổi thường xuyên)
```

### `getInitialProps` vs `getServerSideProps` — Khi nào vẫn dùng `getInitialProps`?

Trong thực tế với Pages Router, **vẫn có một trường hợp buộc phải dùng `getInitialProps`:**

```javascript
// pages/_app.js
// getServerSideProps KHÔNG hoạt động ở đây
// getStaticProps KHÔNG hoạt động ở đây
// Chỉ có getInitialProps là được phép

MyApp.getInitialProps = async (appContext) => {
  // Fetch global data như locale, feature flags, user session
};
```

Đây là giới hạn của Pages Router. App Router giải quyết vấn đề này bằng cách cho phép Server Components ở mọi cấp độ layout.

## 11. Interview Questions

**Junior Level:**

**Q: `getInitialProps` là gì và nó khác React component lifecycle như thế nào?**

<details>
<summary>Đáp án</summary>

`getInitialProps` là một static method gắn vào Page component trong Next.js Pages Router. Nó chạy trước khi component được render lần đầu tiên.

Khác biệt so với React lifecycle:
- React lifecycle (componentDidMount, useEffect) chỉ chạy trên client sau khi component đã mount
- `getInitialProps` chạy trên server (khi hard navigation) hoặc client (khi soft navigation), và chạy *trước* khi render — data đã sẵn sàng ngay từ lần render đầu tiên
- Return value của nó trở thành props của component

```javascript
// Lifecycle thông thường — data đến sau
function Page() {
  const [data, setData] = useState(null);
  useEffect(() => {
    fetch('/api/data').then(r => r.json()).then(setData);
  }, []); // Chạy sau khi mount, user thấy blank state
  return <div>{data ? data.name : 'Loading...'}</div>;
}

// getInitialProps — data đến trước render
function Page({ data }) {
  return <div>{data.name}</div>; // Data sẵn sàng ngay
}
Page.getInitialProps = async () => {
  const res = await fetch('/api/data');
  return { data: await res.json() };
};
```

</details>

**Q: Tại sao để `getInitialProps` trong `_app.js` là một vấn đề về performance?**

<details>
<summary>Đáp án</summary>

Next.js có tính năng **Automatic Static Optimization**: nếu một page không có `getInitialProps` hay `getServerSideProps`, Next.js tự động build nó thành file HTML tĩnh và cache ở CDN. Page tĩnh load cực nhanh, không cần server processing.

Khi bạn thêm `getInitialProps` vào `_app.js`, Next.js bắt buộc phải chạy `_app.getInitialProps` cho mọi request, kể cả những page hoàn toàn static. Điều này làm mất Automatic Static Optimization cho **toàn bộ app** — mọi page đều bị SSR per-request, dù bản thân page đó không cần server-side data.

Hệ quả: Server load tăng, TTFB tăng, không thể cache ở CDN level, chi phí infrastructure cao hơn.

</details>

---

**Mid Level:**

**Q: Giải thích "universal execution" của `getInitialProps` và các vấn đề nó gây ra trong production.**

<details>
<summary>Đáp án</summary>

"Universal execution" hay "isomorphic" nghĩa là cùng một đoạn code chạy được trên cả server lẫn client.

`getInitialProps` chạy trên **server** khi user truy cập URL trực tiếp (hard navigation), và chạy trên **client** khi user navigate bằng `<Link>` hoặc `router.push()` (soft navigation).

**Vấn đề trong production:**

1. **Environment mismatch bug**: `req` và `res` chỉ có trên server. Code đọc `req.cookies` mà không check sẽ crash khi client-side navigate.

2. **Absolute URL requirement**: Server-side fetch cần URL tuyệt đối (`http://localhost:3000/api/...`) vì không có browser context. Client-side fetch dùng URL tương đối được. Phải viết logic để detect và handle cả hai.

3. **Node.js API leak**: Developer vô tình dùng `fs`, `path`, hay các Node.js APIs trong `getInitialProps`. Trên server thì OK, nhưng Next.js sẽ cố ship chúng vào bundle client → error hoặc bundle size tăng.

4. **Security risk**: Logic chỉ nên chạy server (query DB trực tiếp, check private keys) nhưng vô tình chạy cả ở client.

5. **Testing complexity**: Phải mock cả hai execution context khi unit test.

`getServerSideProps` giải quyết hoàn toàn bằng cách guarantee chỉ chạy trên server.

</details>

**Senior Level:**

**Q: Trong một codebase Next.js 10 với `_app.getInitialProps`, bạn sẽ approach migration như thế nào để không break production?**

<details>
<summary>Đáp án</summary>

Đây là bài toán incremental migration với zero-downtime. Tôi sẽ approach theo từng bước:

**Bước 1: Audit và phân loại**

Xác định `_app.getInitialProps` đang làm gì. Thường là một trong:
- Fetch user session/auth
- Fetch global config (feature flags, locale)
- Gọi `Component.getInitialProps` và pass xuống

**Bước 2: Phân tách responsibility**

```javascript
// Trước: một chỗ làm tất cả
MyApp.getInitialProps = async (appCtx) => {
  const pageProps = await App.getInitialProps(appCtx);
  const session = await getSession(appCtx.ctx);
  const config = await fetchConfig();
  return { pageProps, session, config };
};
```

Với session/auth: Chuyển sang middleware (Next.js 12+) hoặc để từng page tự handle qua `getServerSideProps`. Dùng NextAuth.js `SessionProvider` với `session` prop chỉ ở các page cần auth.

Với global config: Xem config có thay đổi không? Nếu static → dùng `getStaticProps` + ISR. Nếu cần mỗi request → context API từ server component.

**Bước 3: Xóa `_app.getInitialProps` và verify**

Sau khi đã move hết logic ra, xóa `getInitialProps` khỏi `_app.js`. Chạy build, kiểm tra Next.js output — các page static đáng ra phải xuất hiện với marker `○` (static) thay vì `λ` (server).

**Bước 4: Gradual rollout**

Dùng feature flag để test với một phần user trước khi full rollout. Monitor TTFB, error rate, và server load.

Điều quan trọng nhất là không rush — mỗi page có `getInitialProps` cần được test kỹ cả hard navigation lẫn soft navigation.

</details>

## 12. Exercises

**Bài 1:** Đọc đoạn code sau và liệt kê tất cả các vấn đề tiềm ẩn:

```javascript
UserPage.getInitialProps = async ({ req, query }) => {
  const userId = req.cookies.user_id;
  const token = req.headers.authorization;
  
  const user = await fetch(`/api/users/${userId}`, {
    headers: { Authorization: token }
  }).then(r => r.json());
  
  return { user, timestamp: new Date() };
};
```

<details>
<summary>Đáp án</summary>

Có 4 vấn đề:

**Vấn đề 1: `req` có thể là `undefined`**

`req` chỉ có khi hard navigation (server). Khi user click `<Link>` để đến trang này, `req` là `undefined`. Code `req.cookies.user_id` sẽ throw `TypeError: Cannot read property 'cookies' of undefined`.

Fix: `const userId = req?.cookies?.user_id || getCookieClient('user_id');`

**Vấn đề 2: Relative URL `/api/users/...` không hoạt động trên server**

Server-side fetch cần URL tuyệt đối. Fetch tới `/api/users/${userId}` sẽ fail trên server vì không có base URL.

Fix: Thay vì gọi API route, gọi thẳng DB/service layer: `const user = await db.users.findById(userId);`

**Vấn đề 3: `new Date()` không serialize an toàn**

`getInitialProps` return value được JSON.stringify rồi nhúng vào HTML. `new Date()` sẽ bị convert thành string `"2024-01-01T00:00:00.000Z"`. Khi component nhận được prop `timestamp`, nó là string chứ không phải `Date` object. Nếu code component sau đó gọi `timestamp.getTime()` sẽ crash.

Fix: Return `timestamp: Date.now()` (number) hoặc `timestamp: new Date().toISOString()` (string, nhưng explicit).

**Vấn đề 4: Không có error handling**

Nếu fetch fail, exception không được catch, Next.js sẽ hiển thị 500 error page. Không có graceful degradation.

Fix: Wrap trong try/catch và return appropriate error state.

</details>

**Bài 2:** Viết lại đoạn code dưới đây để migrate từ `getInitialProps` sang `getServerSideProps`. Đảm bảo handle đúng trường hợp product không tìm thấy:

```javascript
function ProductDetailPage({ product }) {
  if (!product) return <div>Không tìm thấy sản phẩm</div>;
  return <div>{product.name}</div>;

ProductDetailPage.getInitialProps = async ({ req, query }) => {
  const baseUrl = req ? 'http://localhost:3000' : '';
  const { id } = query;
  
  const res = await fetch(`${baseUrl}/api/products/${id}`);
  if (!res.ok) return { product: null };
  
  const product = await res.json();
  return { product };
};

```

<details>
<summary>Đáp án</summary>

```javascript
// pages/products/[id].js

function ProductDetailPage({ product }) {
  // Không cần check null nữa — getServerSideProps handle bằng notFound
  return <div>{product.name}</div>;
}

export async function getServerSideProps({ params }) {
  const { id } = params; // Dùng params cho dynamic route, không phải query

  try {
    // Gọi thẳng DB/service — không cần baseUrl
    const product = await db.products.findById(id);

    if (!product) {
      // Trả về notFound: true — Next.js tự render 404 page
      return { notFound: true };
    }

    return {
      props: { product },
    };
  } catch (error) {
    // Có thể log error ở đây trước khi throw
    console.error('Failed to fetch product:', error);
    throw error; // Next.js render 500 error page
  }
}

export default ProductDetailPage;
```

**Điểm cải thiện so với bản gốc:**

Không cần `baseUrl` vì luôn chạy trên server. Dùng `params` thay `query` để lấy dynamic route params. Dùng `notFound: true` thay vì check `!product` trong component — clean hơn và component không cần handle null state. Có error handling với try/catch. Component đơn giản hơn, chỉ lo về rendering.

</details>

**Bài 3 (Nâng cao):** Bạn inherit một codebase Next.js 11. `_app.js` có `getInitialProps` để fetch user session. 80% các trang không cần session. Đề xuất chiến lược migration mà không break production:

<details>
<summary>Đáp án</summary>

**Bối cảnh vấn đề:**

`_app.getInitialProps` đang làm tất cả app mất Automatic Static Optimization. 80% trang không cần session nhưng vẫn bị SSR mỗi request.

**Chiến lược migration 3 giai đoạn:**

**Giai đoạn 1 — Audit (1-2 ngày)**

Liệt kê tất cả pages, phân loại:
- Nhóm A: Hoàn toàn static (about, landing, blog posts) — không cần session
- Nhóm B: Cần session để hiển thị personalized UI nhưng content vẫn public (dashboard, profile)
- Nhóm C: Cần session để protect route (admin, settings)

**Giai đoạn 2 — Tách session logic (2-3 ngày)**

Cài NextAuth.js nếu chưa có, chuyển sang `SessionProvider` pattern:

```javascript
// pages/_app.js — sau khi refactor
import { SessionProvider } from 'next-auth/react';

function MyApp({ Component, pageProps: { session, ...pageProps } }) {
  return (
    // SessionProvider tự quản lý session client-side
    // Session được truyền từ page-level getServerSideProps
    <SessionProvider session={session}>
      <Component {...pageProps} />
    </SessionProvider>
  );
}

// KHÔNG có getInitialProps ở đây
export default MyApp;
```

**Giai đoạn 3 — Migrate từng nhóm**

Nhóm A: Không làm gì — tự động trở thành static sau khi xóa `_app.getInitialProps`.

Nhóm B & C: Thêm `getServerSideProps` vào từng page:

```javascript
// pages/dashboard.js
export async function getServerSideProps({ req }) {
  const session = await getServerSession(req);
  if (!session && isProtected) {
    return { redirect: { destination: '/login', permanent: false } };
  }
  return { props: { session: session || null } };
}
```

**Kiểm tra kết quả:**

Sau khi xóa `_app.getInitialProps`, chạy `next build` và xem output:
- `○` = static (đạt mục tiêu với nhóm A)
- `λ` = server (chỉ nhóm B và C)

**Rollout safety:**

Deploy lên staging, test mỗi page với cả hard navigation và soft navigation. Đặc biệt test các flow: login → navigate → logout. Dùng feature flag để gradual rollout nếu app lớn.

</details>

## 13. Summary

`getInitialProps` là API đặt nền móng cho Next.js SSR, nhưng thiết kế "universal execution" của nó mang theo hai mặt: tiện lợi (một đoạn code, hai môi trường) và nguy hiểm (code phải guard cho cả server lẫn client, dễ có bug khó debug).

**Điểm cốt lõi cần nhớ:**

`getInitialProps` trong `_app.js` là performance killer số một trong legacy Next.js — nó vô hiệu hóa Automatic Static Optimization cho toàn bộ app.

`req` và `res` có thể là `undefined` — đây là nguồn gốc của class bug phổ biến nhất khi dùng `getInitialProps`.

`getServerSideProps` ra đời để giải quyết vấn đề universal execution bằng cách đơn giản hóa: chỉ chạy trên server, không bao giờ trên client. Clear contract, dễ debug, dễ test.

Trong codebase hiện tại, bạn sẽ vẫn thấy `getInitialProps` trong `_app.js` và `_document.js` vì Pages Router không có lựa chọn nào khác cho global data. App Router (M3+) loại bỏ hoàn toàn giới hạn này.

**Mental model để nhớ:**

```
getInitialProps  =  "Tôi chạy ở đâu? Server hay Client? Phụ thuộc vào context."
getServerSideProps = "Tôi LUÔN chạy trên Server. Không cần hỏi."
```

Bài tiếp theo **M2.4** sẽ đào sâu vào `getServerSideProps` — cơ chế hoạt động, các patterns, và khi nào nó không phải lựa chọn tốt nhất.
