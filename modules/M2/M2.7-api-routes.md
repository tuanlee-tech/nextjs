# M2.7 — API Routes: Backend trong Next.js

## 1. Lesson Goal

Sau bài này bạn phải trả lời được:

- Tại sao API Routes ra đời? Vấn đề gì nó giải quyết so với Express riêng lẽ?
- Tại sao Next.js không có built-in method routing và bạn phải tự handle?
- HOF middleware pattern là gì và tại sao đây là pattern phổ biến nhất trong legacy codebase?
- Khi nào API Routes **không** phải lựa chọn đúng?

## 2. Evolution Context

Trước Next.js API Routes, một ứng dụng web thông thường trông như thế này:

```
Codebase cũ (pre-Next.js API Routes):
─────────────────────────────────────
frontend/          ← React app (port 3000)
  package.json
  src/
    components/

backend/           ← Express server (port 4000)
  package.json
  routes/
  middleware/
  controllers/
```

**Vấn đề:**

- Hai server riêng lẽ → hai process để chạy, hai deployment pipeline
- CORS phải config thủ công giữa localhost:3000 và localhost:4000
- Environment variables phải sync giữa hai nơi
- Developer mới onboard phải hiểu cả hai codebase
- Chia sẻ type giữa frontend và backend là cả một bài toán (shared types, monorepo, hoặc copy-paste)

Next.js 9.3 (2020) giải quyết bằng cách cho phép bạn viết API handler **ngay trong cùng project**:

```
nextjs-app/
  pages/
    api/              ← Backend sống ở đây
      users.ts        → GET/POST /api/users
      posts/
        [id].ts       → GET/PUT/DELETE /api/posts/:id
    index.tsx         ← Frontend
  next.config.js
```

**Một codebase. Một deployment. Không CORS.**

Timeline:

| Version               | Sự kiện                                                   |
| --------------------- | --------------------------------------------------------- |
| Next.js 9.0 (07/2019) | API Routes ra đời — production-ready với built-in parsing |
| Next.js 12.2 (2022)   | Edge Runtime option cho API Routes                        |
| Next.js 13 (2022)     | App Router: Route Handlers (`route.ts`) song song         |
| Next.js 13.4 (2023)   | Route Handlers stable, khuyến nghị dùng thay API Routes   |

## 3. Mental Model

Hãy nghĩ API Routes như **serverless functions được colocate với frontend**:

```
pages/api/posts.ts
       ↓
Khi request đến /api/posts:
  1. Next.js server (Node.js) nhận request
  2. Tìm file khớp: pages/api/posts.ts
  3. Gọi default export function với (req, res)
  4. Function chạy trên SERVER — không bao giờ ship đến browser
  5. Response trả về client

Quan trọng:
  ✅ Code trong pages/api/ không bao giờ vào client bundle
  ✅ Có thể import DB drivers, secrets, Node.js modules
  ❌ Không thể dùng browser APIs (window, document)
```

**API Routes là Node.js code, không phải React code.**

## 4. Core Concepts

### 4.1 Anatomy của một API Handler

```typescript
// pages/api/hello.ts
import type { NextApiRequest, NextApiResponse } from 'next';

// Type cho response data — tốt cho TypeScript
type ResponseData = {
  message: string;
};

// Default export bắt buộc — tên function không quan trọng
export default function handler(
  req: NextApiRequest, // Extends Node.js IncomingMessage
  res: NextApiResponse<ResponseData>, // Extends Node.js ServerResponse
) {
  res.status(200).json({ message: 'Hello World' });
}
```

**Không có `return` value.** Bạn gọi methods trên `res` để kết thúc request. Đây là Node.js HTTP convention, không phải React.

### 4.2 HTTP Method Routing — Pattern bắt buộc phải nhớ

Next.js **không có built-in method routing**. Một file handler nhận **tất cả** HTTP methods. Bạn phải tự kiểm tra `req.method`:

```typescript
// pages/api/posts.ts
import type { NextApiRequest, NextApiResponse } from 'next';
import { db } from '@/lib/db';

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse,
) {
  switch (req.method) {
    case 'GET':
      return handleGet(req, res);
    case 'POST':
      return handlePost(req, res);
    default:
      // 405: Method Not Allowed — bắt buộc set Allow header
      res.setHeader('Allow', ['GET', 'POST']);
      return res
        .status(405)
        .json({ error: `Method ${req.method} not allowed` });
  }
}

async function handleGet(req: NextApiRequest, res: NextApiResponse) {
  const posts = await db.post.findMany({ orderBy: { createdAt: 'desc' } });
  return res.status(200).json(posts);
}

async function handlePost(req: NextApiRequest, res: NextApiResponse) {
  const { title, content } = req.body;
  const post = await db.post.create({ data: { title, content } });
  return res.status(201).json(post);
}
```

**Tại sao Next.js không có built-in method routing như Express?**

> API Routes được thiết kế theo hướng serverless function — mỗi file là một endpoint đơn giản. Nếu thêm method routing, nó trở nên giống Express hơn. Quyết định thiết kế này giúp file nhỏ, focused, và dễ deploy lên serverless platforms.

### 4.3 Request Object — `NextApiRequest`

```typescript
export default function handler(req: NextApiRequest, res: NextApiResponse) {
  // URL query params: /api/posts?page=2&limit=10
  const { page, limit } = req.query;
  // req.query luôn trả về string | string[]
  // Phải tự parse sang number nếu cần
  const pageNum = parseInt(page as string, 10);

  // Request body (auto-parsed nếu Content-Type: application/json)
  const { title, content } = req.body;
  // body là undefined nếu bodyParser bị disable

  // Headers
  const authHeader = req.headers.authorization;
  const contentType = req.headers['content-type'];

  // Cookies (auto-parsed)
  const sessionToken = req.cookies['session-token'];

  // HTTP Method
  const method = req.method; // 'GET' | 'POST' | 'PUT' | 'DELETE' | ...

  // URL (không include query string)
  const url = req.url; // '/api/posts'
}
```

**`req.query` luôn là string** — không bao giờ auto-convert sang number/boolean:

```typescript
// URL: /api/posts?active=true&page=2

// ❌ Sai khi compare
if (req.query.active === true) { ... }  // Luôn false vì 'true' !== true

// ✅ Đúng
if (req.query.active === 'true') { ... }
const page = parseInt(req.query.page as string, 10)
```

**Lưu ý quan trọng về `req.query` trong API Routes:** `req.query` chứa **cả**
dynamic route parameters lẫn query string parameters. Điều này khác với Pages Router
UI routes nơi `query` chỉ chứa query string (và dynamic params được tách riêng ra `params`
trong `getServerSideProps`).

Ví dụ với file `pages/api/posts/[id].ts` và request đến `/api/posts/123?format=json`:

```javascript
req.query = { id: '123', format: 'json' };
```

Nếu dynamic param và query param có cùng tên, **dynamic param ưu tiên** — query param
bị override. Ví dụ: `/api/posts/123?id=456` → `req.query.id === '123'` (không phải '456').

### 4.4 Response Object — `NextApiResponse`

```typescript
export default function handler(req: NextApiRequest, res: NextApiResponse) {
  // Status code + JSON (phổ biến nhất)
  res.status(200).json({ data: 'something' });

  // Chỉ status, không body (dùng cho 204 No Content)
  res.status(204).end();

  // Redirect
  res.redirect(307, '/login'); // Temporary redirect
  res.redirect(301, '/new-path'); // Permanent redirect

  // Set header trước khi send
  res.setHeader('Cache-Control', 'public, max-age=3600');
  res.setHeader('X-Custom-Header', 'value');

  // Gửi raw string
  res.status(200).send('Plain text response');
}
```

**Lỗi cực phổ biến — gửi response hai lần:**

```typescript
// ❌ BUG: Gửi response hai lần → Error: "Cannot set headers after they are sent"
export default function handler(req: NextApiRequest, res: NextApiResponse) {
  if (!req.body.name) {
    res.status(400).json({ error: 'Name required' });
    // THIẾU return ở đây → code tiếp tục chạy!
  }

  res.status(200).json({ message: `Hello ${req.body.name}` });
  // Crash vì header đã được gửi ở dòng trên
}

// ✅ ĐÚNG: Luôn return sau khi gửi response
export default function handler(req: NextApiRequest, res: NextApiResponse) {
  if (!req.body.name) {
    return res.status(400).json({ error: 'Name required' });
    //     ↑ return bắt buộc
  }

  return res.status(200).json({ message: `Hello ${req.body.name}` });
}
```

### 4.5 Dynamic API Routes

File routing rules giống như Pages Router:

```
pages/api/posts/[id].ts         → /api/posts/1, /api/posts/abc
pages/api/users/[...path].ts    → /api/users/a/b/c
pages/api/[[...slug]].ts        → /api, /api/a, /api/a/b
```

```typescript
// pages/api/posts/[id].ts
import type { NextApiRequest, NextApiResponse } from 'next';

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse,
) {
  // req.query.id là string (không phải number!)
  const { id } = req.query;

  // Validate trước khi dùng
  if (!id || Array.isArray(id)) {
    return res.status(400).json({ error: 'Invalid id' });
  }

  switch (req.method) {
    case 'GET': {
      const post = await db.post.findUnique({ where: { id: parseInt(id) } });
      if (!post) return res.status(404).json({ error: 'Post not found' });
      return res.status(200).json(post);
    }
    case 'DELETE': {
      await db.post.delete({ where: { id: parseInt(id) } });
      return res.status(204).end();
    }
    default:
      res.setHeader('Allow', ['GET', 'DELETE']);
      return res.status(405).json({ error: 'Method not allowed' });
  }
}
```

### 4.6 Middleware Pattern — HOF (Higher-Order Function)

Next.js API Routes không có built-in middleware system như Express. Pattern phổ biến nhất trong legacy codebase là **Higher-Order Function (HOF)**:

```typescript
// lib/middleware/withAuth.ts
import type { NextApiRequest, NextApiResponse, NextApiHandler } from 'next';
import { verifyToken } from '@/lib/auth';

// HOF: nhận handler, trả về handler mới đã wrap authentication
export function withAuth(handler: NextApiHandler): NextApiHandler {
  return async (req: NextApiRequest, res: NextApiResponse) => {
    const token =
      req.cookies['auth-token'] || req.headers.authorization?.split(' ')[1];

    if (!token) {
      return res.status(401).json({ error: 'Authentication required' });
    }

    try {
      const user = await verifyToken(token);
      // Gắn user vào request để handler downstream dùng
      (req as any).user = user;
      return handler(req, res);
    } catch {
      return res.status(401).json({ error: 'Invalid token' });
    }
  };
}
```

```typescript
// lib/middleware/withValidation.ts
import type { NextApiHandler, NextApiRequest, NextApiResponse } from 'next';
import { ZodSchema } from 'zod';

export function withValidation<T>(
  schema: ZodSchema<T>,
  handler: NextApiHandler,
): NextApiHandler {
  return async (req: NextApiRequest, res: NextApiResponse) => {
    // Chỉ validate với methods có body
    if (['POST', 'PUT', 'PATCH'].includes(req.method || '')) {
      const result = schema.safeParse(req.body);
      if (!result.success) {
        return res.status(422).json({
          error: 'Validation failed',
          details: result.error.flatten(),
        });
      }
      req.body = result.data; // Gán lại body đã validate
    }
    return handler(req, res);
  };
}
```

```typescript
// Compose nhiều middleware — thứ tự thực thi từ ngoài vào trong
// pages/api/posts.ts
import { withAuth } from '@/lib/middleware/withAuth';
import { withValidation } from '@/lib/middleware/withValidation';
import { createPostSchema } from '@/lib/schemas';

async function handler(req: NextApiRequest, res: NextApiResponse) {
  // req.user đã có (từ withAuth)
  // req.body đã validate (từ withValidation với POST)
  const posts = await db.post.findMany({
    where: { authorId: (req as any).user.id },
  });
  return res.status(200).json(posts);
}

// Thứ tự: withAuth → withValidation → handler
export default withAuth(withValidation(createPostSchema, handler));
```

**Pattern compose nhiều middleware gọn hơn:**

```typescript
// lib/middleware/compose.ts
import type { NextApiHandler } from 'next';

type Middleware = (handler: NextApiHandler) => NextApiHandler;

// Compose từ phải sang trái (giống pipe operator)
export function compose(...middlewares: Middleware[]) {
  return (handler: NextApiHandler): NextApiHandler =>
    middlewares.reduceRight((acc, middleware) => middleware(acc), handler);
}

// Dùng:
export default compose(
  withAuth,
  withRateLimit({ max: 100, windowMs: 60_000 }),
  withValidation(createPostSchema),
)(handler);
```

### 4.8 Error Handling Pattern — Production-ready

```typescript
// lib/apiHandler.ts — Wrapper chuẩn hóa error handling
import type { NextApiRequest, NextApiResponse, NextApiHandler } from 'next';

export class ApiError extends Error {
  constructor(
    public statusCode: number,
    message: string,
    public details?: unknown,
  ) {
    super(message);
    this.name = 'ApiError';
  }
}

// HOF bọc error handling cho mọi handler
export function apiHandler(handler: NextApiHandler): NextApiHandler {
  return async (req: NextApiRequest, res: NextApiResponse) => {
    try {
      await handler(req, res);
    } catch (error) {
      // Lỗi đã biết — ApiError do code tự throw
      if (error instanceof ApiError) {
        return res.status(error.statusCode).json({
          error: error.message,
          ...(error.details && { details: error.details }),
        });
      }

      // Lỗi Zod validation
      if (error instanceof ZodError) {
        return res.status(422).json({
          error: 'Validation failed',
          details: error.flatten(),
        });
      }

      // Lỗi không biết — log và trả generic message
      console.error('[API Error]', {
        method: req.method,
        url: req.url,
        error,
      });

      return res.status(500).json({ error: 'Internal server error' });
    }
  };
}
```

```typescript
// pages/api/posts/[id].ts — Dùng apiHandler
import { apiHandler, ApiError } from '@/lib/apiHandler';

export default apiHandler(async (req, res) => {
  const { id } = req.query;

  if (req.method !== 'GET') {
    throw new ApiError(405, 'Method not allowed');
  }

  const post = await db.post.findUnique({
    where: { id: parseInt(id as string) },
  });

  if (!post) {
    throw new ApiError(404, 'Post not found');
    // ApiError tự động catch → trả 404 JSON
  }

  res.status(200).json(post);
});
```

## 5. Legacy Recognition

Pattern nào thường gặp trong codebase Pages Router cũ:

```typescript
// Pattern 1: Không handle method → nhận mọi method, có thể gây data mutation không mong muốn
export default async function handler(req, res) {
  const users = await getUsers();
  res.json(users); // Gọi được bằng DELETE cũng trả users
}

// Pattern 2: Try-catch thiếu → unhandled rejection, server crash
export default async function handler(req, res) {
  const data = await fetchData(); // Nếu throw → 500 unhandled
  res.json(data);
}

// Pattern 3: Thiếu return → gửi response 2 lần
export default function handler(req, res) {
  if (!req.body.name) {
    res.status(400).json({ error: 'required' });
    // Không return → tiếp tục xuống dưới
  }
  res.json({ ok: true }); // Crash: headers already sent
}

// Pattern 4: req.query không validate type
export default function handler(req, res) {
  const id = req.query.id; // string | string[]
  const post = getPost(id); // Nếu id là string[] → bug
  res.json(post);
}

// Pattern 5: Hardcode credentials (cực kỳ nguy hiểm)
export default async function handler(req, res) {
  const response = await fetch('https://api.example.com', {
    headers: { Authorization: 'Bearer sk-hardcoded-key-12345' },
    //                                 ↑ Bị lộ trong git history
  });
  res.json(await response.json());
}
```

**Cách nhận ra codebase dùng API Routes:**

```
pages/
  api/          ← Folder này có → đang dùng API Routes
    auth/
      [...nextauth].ts  ← NextAuth pattern phổ biến
    users/
      index.ts
      [id].ts
```

## 6. Modern Approach

### Route Handlers (App Router)

`pages/api/` → `app/api/.../route.ts` (hoặc bất cứ đâu trong `app/`):

```typescript
// app/api/posts/route.ts — Route Handler
import { NextRequest, NextResponse } from 'next/server';
import { db } from '@/lib/db';

// Method routing: named exports thay vì switch/case
export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url);
  const page = parseInt(searchParams.get('page') ?? '1');

  const posts = await db.post.findMany({
    skip: (page - 1) * 10,
    take: 10,
  });

  // NextResponse thay vì res.json()
  return NextResponse.json(posts);
}

export async function POST(request: NextRequest) {
  const body = await request.json();

  const post = await db.post.create({ data: body });
  return NextResponse.json(post, { status: 201 });
}
```

```typescript
// app/api/posts/[id]/route.ts — Dynamic Route Handler
import { NextRequest, NextResponse } from 'next/server';

export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }, // params từ URL
) {
  const post = await db.post.findUnique({ where: { id: parseInt(params.id) } });

  if (!post) {
    return NextResponse.json({ error: 'Not found' }, { status: 404 });
  }

  return NextResponse.json(post);
}

export async function DELETE(
  request: NextRequest,
  { params }: { params: { id: string } },
) {
  await db.post.delete({ where: { id: parseInt(params.id) } });
  return new Response(null, { status: 204 });
}
```

**Điểm khác biệt cốt lõi:**

| Pages Router API Routes              | App Router Route Handlers           |
| ------------------------------------ | ----------------------------------- |
| `NextApiRequest` / `NextApiResponse` | Web standard `Request` / `Response` |
| `switch (req.method)`                | Named exports: `GET`, `POST`, ...   |
| `res.json()`                         | `NextResponse.json()`               |
| `req.query`                          | `new URL(request.url).searchParams` |
| `req.cookies`                        | `request.cookies.get('name')`       |
| Node.js runtime only                 | Node.js hoặc Edge runtime           |
| HOF middleware pattern               | Middleware.ts + Next.js built-in    |

**Route Handlers là Web API chuẩn** — bạn có thể dùng cùng code ở Deno, Bun, Cloudflare Workers mà không cần thay đổi.

### Middleware (middleware.ts) — Không phải API Routes

Một điểm hay bị nhầm: `middleware.ts` ở root của project là **khác** với HOF middleware trong API Routes:

```typescript
// middleware.ts — Chạy TRƯỚC khi request đến page hoặc API route
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  // Chạy ở Edge runtime — không có DB access trực tiếp
  const token = request.cookies.get('auth-token');

  if (!token && request.nextUrl.pathname.startsWith('/api/protected')) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }
}

export const config = {
  matcher: ['/api/protected/:path*'],
};
```

`middleware.ts` sẽ được học chi tiết ở M8.5. Bài này tập trung vào API Routes handlers.

## 7. Production Scenario

### Xây dựng REST API hoàn chỉnh với authentication

```typescript
// pages/api/posts/index.ts
import type { NextApiRequest, NextApiResponse } from 'next';
import { apiHandler } from '@/lib/apiHandler';
import { withAuth, AuthenticatedRequest } from '@/lib/middleware/withAuth';
import { withRateLimit } from '@/lib/middleware/withRateLimit';
import { createPostSchema, listPostsSchema } from '@/lib/schemas/post';
import { db } from '@/lib/db';

async function handler(req: NextApiRequest, res: NextApiResponse) {
  const user = (req as AuthenticatedRequest).user;

  switch (req.method) {
    case 'GET': {
      // Validate query params
      const query = listPostsSchema.parse({
        page: req.query.page ?? '1',
        limit: req.query.limit ?? '10',
        status: req.query.status,
      });

      const [posts, total] = await Promise.all([
        db.post.findMany({
          where: { authorId: user.id, status: query.status },
          skip: (query.page - 1) * query.limit,
          take: query.limit,
          orderBy: { createdAt: 'desc' },
          include: { author: { select: { name: true } } },
        }),
        db.post.count({ where: { authorId: user.id } }),
      ]);

      // Pagination headers — best practice cho REST API
      res.setHeader('X-Total-Count', total);
      res.setHeader('X-Page', query.page);
      res.setHeader('X-Limit', query.limit);

      return res.status(200).json(posts);
    }

    case 'POST': {
      const data = createPostSchema.parse(req.body);

      const post = await db.post.create({
        data: { ...data, authorId: user.id },
      });

      // 201 Created với Location header
      res.setHeader('Location', `/api/posts/${post.id}`);
      return res.status(201).json(post);
    }

    default:
      res.setHeader('Allow', ['GET', 'POST']);
      return res.status(405).json({ error: 'Method not allowed' });
  }
}

// Apply middleware stack: rate limit → auth → error handling → handler
export default apiHandler(withRateLimit({ max: 60 })(withAuth(handler)));
```

### CORS cho third-party clients

```typescript
// lib/middleware/withCors.ts
import type { NextApiHandler, NextApiRequest, NextApiResponse } from 'next';

const ALLOWED_ORIGINS = process.env.CORS_ORIGINS?.split(',') ?? [];

export function withCors(handler: NextApiHandler): NextApiHandler {
  return (req: NextApiRequest, res: NextApiResponse) => {
    const origin = req.headers.origin;

    if (origin && ALLOWED_ORIGINS.includes(origin)) {
      res.setHeader('Access-Control-Allow-Origin', origin);
      res.setHeader(
        'Access-Control-Allow-Methods',
        'GET,POST,PUT,DELETE,OPTIONS',
      );
      res.setHeader(
        'Access-Control-Allow-Headers',
        'Content-Type, Authorization',
      );
      res.setHeader('Access-Control-Max-Age', '86400');
    }

    // Handle OPTIONS preflight — PHẢI trả về trước khi handler chạy
    if (req.method === 'OPTIONS') {
      return res.status(204).end();
    }

    return handler(req, res);
  };
}
```

```typescript
import { z } from 'zod';
import { signJWT } from '@/lib/auth';
import { db } from '@/lib/db';
import bcrypt from 'bcryptjs';

const loginSchema = z.object({
  email: z.string().email(),

  res: NextApiResponse,

    }
    return res.status(500).json({ error: 'Internal server error' });
```

```typescript
// app/api/auth/login/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { z } from 'zod';
import { signJWT } from '@/lib/auth';
import { db } from '@/lib/db';
import bcrypt from 'bcryptjs';
import { cookies } from 'next/headers';

const loginSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
});

// Named export thay vì switch/case
export async function POST(request: NextRequest) {
  try {
    const body = await request.json();
    const { email, password } = loginSchema.parse(body);

    const user = await db.user.findUnique({ where: { email } });
    if (!user || !(await bcrypt.compare(password, user.passwordHash))) {
      return NextResponse.json(
        { error: 'Invalid credentials' },
        { status: 401 },
      );
    }

    const token = await signJWT({ userId: user.id });

    // Dùng Next.js cookies() API thay vì Set-Cookie header thủ công
    cookies().set('auth-token', token, {
      httpOnly: true,
      secure: true,
      sameSite: 'strict',
      path: '/',
    });

    return NextResponse.json({ user: { id: user.id, email: user.email } });
  } catch (error) {
    if (error instanceof z.ZodError) {
      return NextResponse.json({ error: error.flatten() }, { status: 422 });
    }
    return NextResponse.json(
      { error: 'Internal server error' },
      { status: 500 },
    );
  }
}
// GET, PUT, DELETE không cần export → Next.js tự trả 405 nếu method không match
```

```
□ Di chuyển file: pages/api/X.ts → app/api/X/route.ts

□ Request object:
  □ req.body → await request.json()
  □ req.query → new URL(request.url).searchParams
  □ req.cookies → request.cookies.get('name')?.value
  □ req.headers → request.headers.get('name')

□ Response:
  □ res.json() → NextResponse.json()
  □ res.status(X).json() → NextResponse.json({}, { status: X })
  □ res.setHeader('Set-Cookie',...) → cookies().set(...)
  □ res.redirect() → NextResponse.redirect(new URL('/path', request.url))

□ Method routing:
  □ switch(req.method) → named exports GET/POST/PUT/DELETE

□ Middleware HOF:
  □ Giữ nguyên nếu chuyển dần (có thể adapt)
  □ Hoặc migrate sang middleware.ts cho auth logic chung

□ Config:
  □ export const config → route segment config hoặc edge runtime config
```

## 10. Architecture Notes

### Khi nào KHÔNG dùng API Routes

API Routes phù hợp với **"vừa vặn"** use case. Khi nào nên dùng backend riêng:

```
✅ Dùng API Routes khi:
  - CRUD đơn giản: users, posts, products
  - Auth callbacks: OAuth, JWT
  - Webhook receivers (Stripe, GitHub)
  - Proxy tới external API (ẩn API key)
  - Form submission handling
  - Small-medium project (< 20 endpoints)

⚠️ Cân nhắc backend riêng khi:
  - Cần WebSocket (API Routes không support)
  - Business logic phức tạp > 500 lines/endpoint
  - Cần share API với mobile app, third-party
  - > 50 endpoints với logic phức tạp

❌ Không dùng API Routes khi:
  - Cần streaming protocols (gRPC, WebSocket)
  - Team backend riêng, frontend riêng
  - Cần heavy computation (xử lý video, ML inference)
  - Microservices architecture
```

### API Routes và Serverless Implications

```
Trên Vercel (default deployment):
  Mỗi API Route → một Serverless Function riêng
  ├── Cold start: 100-500ms cho lần đầu
  ├── Stateless: không có in-memory state giữa requests
  ├── Timeout: 10s (Hobby), 60s (Pro), 900s (Enterprise)
  └── DB connection: phải dùng connection pooling (PgBouncer, Neon, etc.)

Self-hosted (Docker/VPS):
  Tất cả API Routes chạy trong cùng Node.js process
  ├── Không có cold start
  ├── Có thể dùng in-memory cache
  └── DB connection pool giữ nguyên giữa requests
```

**Một lưu ý quan trọng về `res.json()` và async callbacks:** Khi dùng các library
dựa trên callback (như `nodemailer`, `formidable`, `multer`) trong API Routes,
**không được** return `res.json()` bên trong callback mà không wrap trong Promise.
Next.js API Routes handler kết thúc khi function return, không phải khi response được gửi.
Nếu bạn dùng callback pattern không đồng bộ, handler sẽ return trước khi callback chạy,
dẫn đến response không được gửi hoặc gửi không đúng.

## 11. Interview Questions

**Junior level:**

**Q1: API Routes trong Next.js là gì? Tại sao chúng không cần server Express riêng?**

<details>
<summary>Đáp án</summary>

API Routes là files trong thư mục `pages/api/` cho phép viết backend HTTP endpoints ngay trong project Next.js mà không cần server riêng. Mỗi file map 1:1 với một URL endpoint.

Lý do không cần Express riêng:

- Next.js server (Node.js) tự handle routing, parse request body, serve response
- Code trong `pages/api/` chạy trên server, không ship đến client
- Cùng project → không CORS issue, cùng environment variables, một deployment
- Tuy nhiên, API Routes không phải full Express replacement — thiếu WebSocket, middleware system phức tạp, và không phù hợp cho enterprise backend lớn

</details>

**Q2: Tại sao phải `return res.status(...).json(...)` thay vì chỉ `res.status(...).json(...)`?**

<details>
<summary>Đáp án</summary>

`return` không phải để trả value cho `res.json()` — `res.json()` không dùng return value. `return` ở đây dùng để **dừng function execution** sau khi response được gửi.

Không có `return`, code tiếp tục chạy và có thể gọi `res.json()` lần thứ hai → Error: "Cannot set headers after they are sent to the client."

```typescript
// Ví dụ vấn đề:
if (error) {
  res.status(400).json({ error: 'Bad request' });
  // Không return → chạy tiếp xuống
}
res.status(200).json({ ok: true }); // 💥 Crash vì đã gửi response ở trên
```

Convention: luôn `return res.status().json()` để rõ ràng flow dừng ở đó.

</details>

**Mid level:**

**Q3: Giải thích HOF middleware pattern trong API Routes. Tại sao Next.js không có built-in middleware như Express?**

<details>
<summary>Đáp án</summary>

HOF (Higher-Order Function) middleware pattern: một function nhận `handler` và trả về `handler` mới đã được wrap thêm logic:

```typescript
function withAuth(handler) {
  return async (req, res) => {
    if (!req.cookies.token)
      return res.status(401).json({ error: 'Unauthorized' });
    req.user = await verifyToken(req.cookies.token);
    return handler(req, res);
  };
}
export default withAuth(myHandler);
```

Tại sao Next.js không có built-in middleware: API Routes được thiết kế hướng serverless — mỗi file là một isolated function. Thêm Express-style middleware system sẽ làm phức tạp mental model và không phù hợp với serverless. Thay vào đó:

- App Router có `middleware.ts` chạy ở Edge (trước khi request đến handler)
- Route Handlers dùng Web standard Request/Response không cần middleware pattern

HOF pattern có nhược điểm: compose nhiều middleware → callback nesting khó đọc, debug stack trace phức tạp. Đây là một trong những lý do App Router Route Handlers được thiết kế khác.

</details>

**Senior level:**

**Q4: API Routes chạy như thế nào trên Vercel vs self-hosted? Implication cho DB connection management là gì?**

<details>
<summary>Đáp án</summary>

**Vercel (Serverless Functions):**

- Mỗi API Route compile thành một serverless function độc lập
- Không có state giữa invocations — mọi in-memory data bị reset
- Cold start 100-500ms khi function không được call một thời gian
- Parallel requests → nhiều function instances chạy song song
- DB connections: mỗi instance tạo connection mới → có thể exhaust DB connection pool nhanh

**Self-hosted (Node.js process):**

- Tất cả API Routes chạy trong cùng long-running process
- In-memory state persist giữa requests (có thể dùng memory cache)
- Không có cold start
- DB connections: giữ connection pool nguyên vẹn giữa requests

**DB Connection Management cho Serverless:**

```typescript
// Singleton pattern với global variable để reuse connection
const globalForPrisma = global as unknown as { prisma: PrismaClient };
export const prisma = globalForPrisma.prisma ?? new PrismaClient();
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;
```

Với PostgreSQL: cần connection pooler (PgBouncer, Neon, Supabase pooler) vì serverless functions không giữ connection lâu và số lượng connections có thể vượt DB limit.

</details>

**Bài 1 — Cơ bản:** Viết API Route tại `pages/api/todos/[id].ts` hỗ trợ `GET` (lấy một todo), `PUT` (cập nhật), `DELETE` (xóa). Trả về 404 nếu todo không tồn tại, 405 với method không được hỗ trợ.

<details>
<summary>Đáp án</summary>

```typescript
// pages/api/todos/[id].ts
import type { NextApiRequest, NextApiResponse } from 'next';

// Mock DB — trong thực tế dùng DB thật
const todos: Record<string, { id: string; title: string; done: boolean }> = {
  '1': { id: '1', title: 'Learn Next.js', done: false },
  '2': { id: '2', title: 'Build a project', done: false },
};

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse,
) {
  const { id } = req.query;

  // Guard: id không hợp lệ
  if (!id || Array.isArray(id)) {
    return res.status(400).json({ error: 'Invalid id parameter' });
  }

  const todo = todos[id];

  switch (req.method) {
    case 'GET': {
      if (!todo) {
        return res
          .status(404)
          .json({ error: `Todo with id '${id}' not found` });
      }
      return res.status(200).json(todo);
    }

    case 'PUT': {
      if (!todo) {
        return res
          .status(404)
          .json({ error: `Todo with id '${id}' not found` });
      }
      const { title, done } = req.body;
      todos[id] = {
        ...todo,
        ...(title !== undefined && { title }),
        ...(done !== undefined && { done }),
      };
      return res.status(200).json(todos[id]);
    }

    case 'DELETE': {
      if (!todo) {
        return res
          .status(404)
          .json({ error: `Todo with id '${id}' not found` });
      }
      delete todos[id];
      return res.status(204).end();
    }

    default:
      res.setHeader('Allow', ['GET', 'PUT', 'DELETE']);
      return res
        .status(405)
        .json({ error: `Method ${req.method} not allowed` });
  }
}
```

Điểm cần chú ý: luôn `return` sau mỗi response, validate `id` trước khi dùng, set `Allow` header với 405, `DELETE` trả 204 No Content (không có body).

</details>

```typescript
// lib/middleware/withApiKey.ts
import type { NextApiHandler, NextApiRequest, NextApiResponse } from 'next';

export function withApiKey(handler: NextApiHandler): NextApiHandler {
  return async (req: NextApiRequest, res: NextApiResponse) => {
    const apiKey = req.headers['x-api-key'];
    const expectedKey = process.env.API_KEY;

    if (!expectedKey) {
      // Cấu hình server thiếu — log để ops biết
      console.error('API_KEY environment variable not configured');
      return res.status(500).json({ error: 'Server configuration error' });
    }

    if (!apiKey) {
      return res.status(401).json({ error: 'API key required' });
    }

    if (Array.isArray(apiKey) || apiKey !== expectedKey) {
      return res.status(401).json({ error: 'Invalid API key' });
    }

    return handler(req, res);
  };
}
```

**Bài 3 — Debug:** Đoạn code dưới nhận lỗi lạ trong production — đôi khi trả data đúng, đôi khi crash với "Cannot read properties of undefined". Tìm tất cả bugs:

```typescript
// pages/api/users.ts
export default async function handler(req, res) {
  if (req.method == 'GET') {
    const users = await db.user.findMany();
    res.json(users);
  }

  if (req.method == 'POST') {
    const user = await db.user.create({
      data: {
        name: req.body.name,
        email: req.body.email,
        role: req.body.role,
      },
    });
    res.status(201).json(user);
  }
}
```

<details>
<summary>Đáp án</summary>

Có 4 bugs:

**Bug 1: Thiếu `return` → gửi response hai lần**

Với GET request: `if (req.method == 'GET')` đúng → gửi response. Sau đó `if (req.method == 'POST')` sai → không gửi thêm. Tưởng ổn nhưng...

Với GET request sau khi `res.json(users)`: code tiếp tục xuống block POST. Block POST check `req.method == 'POST'` → sai → bỏ qua. Không crash nhưng là pattern nguy hiểm.

Nhưng crash thật sự đến từ Bug 2.

**Bug 2: Không có `else` → nếu thêm method khác, cả hai block đều bỏ qua, không gửi response → hanging request**

**Bug 3: Không handle method không biết → không có 405 response**

POST, GET hoạt động nhưng DELETE, PUT, ... → không có response → timeout.

**Bug 4: Không try-catch → unhandled rejection nếu DB lỗi**

Nếu `db.user.create()` throw (email duplicate, DB down...) → crash không có response.

**Fix hoàn chỉnh:**

```typescript
// pages/api/users.ts
import type { NextApiRequest, NextApiResponse } from 'next';

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse,
) {
  try {
    switch (req.method) {
      case 'GET': {
        const users = await db.user.findMany();
        return res.status(200).json(users);
        // return bắt buộc
      }

      case 'POST': {
        const { name, email, role } = req.body;
        if (!name || !email) {
          return res.status(400).json({ error: 'Name and email required' });
        }
        const user = await db.user.create({
          data: { name, email, role },
        });
        return res.status(201).json(user);
      }

      default:
        res.setHeader('Allow', ['GET', 'POST']);
        return res
          .status(405)
          .json({ error: `Method ${req.method} not allowed` });
    }
  } catch (error) {
    console.error('[/api/users]', error);
    return res.status(500).json({ error: 'Internal server error' });
  }
}
```

</details>

## 13. Summary

API Routes giải quyết đúng một vấn đề cụ thể: **colocate backend logic với frontend trong cùng Next.js project**, loại bỏ nhu cầu server Express riêng cho majority of use cases.

Ba điểm cốt lõi cần nhớ:

**Thứ nhất**, Next.js không có built-in method routing — bạn phải tự check `req.method` và luôn handle `default` case với 405. Đây là pattern xuất hiện trong 100% legacy codebase.

**Thứ hai**, HOF middleware pattern là cách duy nhất để reuse cross-cutting concerns (auth, validation, rate limiting) trong Pages Router. Nhận ra pattern này khi đọc code cũ là kỹ năng thiết yếu.

**Thứ ba**, `return` sau mỗi `res.json()` không phải optional — thiếu nó là nguồn gốc của bug "Cannot set headers after they are sent" phổ biến nhất.

**Khi nào API Routes không đủ:** WebSocket, heavy background jobs, cần share API với nhiều clients, hoặc team backend riêng — lúc đó cần backend service độc lập.

**Migration sang App Router:** Route Handlers (`route.ts`) dùng Web standard API — không còn `NextApiRequest/NextApiResponse`, method routing bằng named exports, middleware HOF không còn cần thiết cho phần lớn use cases.

Bài tiếp theo **M2.8 — ISR: Incremental Static Regeneration** sẽ đào sâu vào cơ chế "stale-while-revalidate" của Next.js — cách pages tĩnh được cập nhật mà không cần rebuild toàn bộ site.
