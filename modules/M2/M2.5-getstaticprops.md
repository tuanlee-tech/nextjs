# M2.5 — `getStaticProps`: Build-Time Data Fetching

## 1. Lesson Goal

Sau bài này bạn phải:

- Giải thích được tại sao build-time rendering tồn tại và bài toán nó giải quyết
- Hiểu sự khác biệt căn bản giữa "build time" và "request time"
- Biết đầy đủ context object và các return types — đặc biệt là `revalidate`
- Hiểu `fallback` trong `getStaticPaths` ở mức đủ để dùng với `getStaticProps`
- Nhận ra khi nào static generation là lựa chọn sai
- Biết Preview Mode là gì và tại sao editorial workflow cần nó

---

## 2. Evolution Context

Sau bài M2.4, bạn đã hiểu `getServerSideProps` — mỗi request đều hit server, render HTML, gửi về. Điều này đúng với dashboard, feed, order page. Nhưng hãy suy nghĩ về bài toán khác:

**Một trang blog có 10,000 bài viết. Mỗi bài viết thay đổi trung bình một lần mỗi tháng. Traffic: 500,000 lượt đọc mỗi ngày.**

Với `getServerSideProps`:

```
500,000 requests/ngày
→ 500,000 lần query database
→ 500,000 lần React render trên server
→ 500,000 lần gửi HTML về browser
→ Tất cả cho cùng nội dung không đổi trong 30 ngày
```

Đây là lãng phí tài nguyên thuần túy. Nội dung không đổi, nhưng server phải làm việc như thể nó đổi mỗi giây.

**Giải pháp:** Render HTML **một lần** vào thời điểm deploy, lưu thành file tĩnh, phục vụ từ CDN cho mọi request.

```
Build time: Render 10,000 bài viết → 10,000 file HTML
Deploy: Upload lên CDN

500,000 requests/ngày
→ CDN phục vụ file HTML có sẵn
→ 0 database queries
→ 0 server rendering
→ TTFB < 50ms
```

Đây là **Static Site Generation (SSG)**, và `getStaticProps` là cơ chế để làm điều đó trong Next.js.

**Timeline:**

| Version | Sự kiện |
|---------|---------|
| Next.js 1-9.2 | Phải dùng `getInitialProps` workaround cho SSG |
| Next.js 9.3 (2020) | `getStaticProps` ra đời, SSG trở thành first-class citizen |
| Next.js 9.5 (2020) | ISR (Incremental Static Regeneration) ra đời, giải quyết vấn đề stale data |
| Next.js 12+ | On-demand ISR — invalidate cache theo yêu cầu |

## 3. Mental Model

Để hiểu `getStaticProps`, bạn cần phân biệt rõ hai thời điểm hoàn toàn khác nhau:

```
BUILD TIME                          REQUEST TIME
(Khi bạn chạy `next build`)         (Khi user vào trang)

┌─────────────────────┐             ┌─────────────────────┐
│ getStaticProps chạy │             │ Không có gì chạy    │
│ Fetch từ CMS/DB     │             │ CDN phục vụ         │
│ Render HTML         │             │ file HTML có sẵn    │
│ Lưu thành file      │             │                     │
└─────────────────────┘             └─────────────────────┘
        │                                    ▲
        └── file HTML ──────────────────────►┘
             được tạo
             một lần
```

**Hệ quả quan trọng:**

Vì `getStaticProps` chạy tại **build time**, không phải request time:

- Không có `req` hay `res` — không có HTTP request nào đang diễn ra
- Không có session user — không biết ai đang xem
- Không có query string từ browser — URL chưa được truy cập
- Có `params` — biết page nào đang được build (cho dynamic routes)

Đây là ranh giới tư duy quan trọng nhất: **static = không có context về user hay request**.

## 4. Core Concepts

### 4.1 Cú pháp cơ bản

```javascript
// pages/about.js — trang hoàn toàn static

function AboutPage({ team, companyInfo }) {
  return (
    <div>
      <h1>{companyInfo.name}</h1>
      <p>{companyInfo.description}</p>
      <TeamList members={team} />
    </div>
  );
}

// Named export — Next.js nhận ra và chạy tại build time
export async function getStaticProps() {
  // Tại đây có thể:
  // - Query database trực tiếp (không qua API route)
  // - Đọc file system
  // - Gọi CMS API
  // - Fetch external API
  const team = await db.team.findAll({ where: { isActive: true } });
  const companyInfo = await cmsClient.getSingleton('company');

  return {
    props: {
      team,
      companyInfo,
    },
    // revalidate: 3600, // Tùy chọn — ISR sẽ học ở M2.8
  };
}

export default AboutPage;
```

### 4.2 Context Object

Context của `getStaticProps` **nghèo hơn** `getServerSideProps` — vì không có HTTP request đang diễn ra:

```javascript
export async function getStaticProps(context) {
  const {
    // Dynamic route params — chỉ có khi dùng với getStaticPaths
    params,        // { slug: 'my-post' } với route /blog/[slug]
                   // undefined với route không dynamic

    // Preview Mode — cho editorial workflow (xem mục 4.5)
    preview,       // true nếu đang ở Preview Mode, false/undefined nếu không
    previewData,   // Data được truyền từ preview API route

    // Internationalization
    locale,        // 'vi', 'en'
    locales,       // ['vi', 'en']
    defaultLocale, // 'en'
  } = context;

  // KHÔNG có:
  // ❌ req — không có HTTP request
  // ❌ res — không có HTTP response
  // ❌ query — không có URL query string
  // ❌ resolvedUrl — không có URL

  return { props: {} };
}
```

### 4.3 Return Types

`getStaticProps` có cùng ba return type với `getServerSideProps`, nhưng có thêm `revalidate`:

**Return 1: `{ props }` — Render và cache**

```javascript
export async function getStaticProps() {
  const data = await fetchData();

  return {
    props: { data },
    // revalidate: 60, // Giây — enable ISR (học ở M2.8)
  };
}
```

**Return 2: `{ notFound: true }` — Render 404**

```javascript
export async function getStaticProps({ params }) {
  const post = await db.posts.findBySlug(params.slug);

  if (!post || post.status === 'draft') {
    return { notFound: true };
    // File HTML của trang này sẽ không được tạo
    // Request đến URL này sẽ nhận 404
  }

  return { props: { post } };
}
```

**Return 3: `{ redirect }` — Redirect**

```javascript
export async function getStaticProps({ params }) {
  const post = await db.posts.findBySlug(params.slug);

  // Bài đã được move sang slug mới
  if (post?.newSlug) {
    return {
      redirect: {
        destination: `/blog/${post.newSlug}`,
        permanent: true, // 308 — bài cũ vĩnh viễn chuyển sang bài mới
      },
    };
  }

  return { props: { post } };
}
```

**Lưu ý về `permanent` trong static context:**

Với `getStaticProps`, `permanent: true` hợp lý hơn vì redirect được quyết định tại build time — nếu slug đã đổi thì đổi vĩnh viễn, không phải conditional. Ngược lại với `getServerSideProps` nơi `permanent: true` rất nguy hiểm.

**Return 4: `revalidate` — ISR (giới thiệu)**

```javascript
export async function getStaticProps() {
  const products = await db.products.findAll();

  return {
    props: { products },
    revalidate: 60, // Cho phép rebuild page này sau 60 giây nếu có request mới
  };
}
```

`revalidate` biến trang static thành **Incremental Static Regeneration** — giữ được lợi thế của static (CDN cache, fast TTFB) nhưng tự động cập nhật khi data thay đổi. ISR sẽ được học chi tiết ở M2.8, nhưng cần biết nó tồn tại để có mental model đầy đủ.

### 4.4 `getStaticProps` với Dynamic Routes

Với route tĩnh (`pages/about.js`), `getStaticProps` chạy một lần và tạo một file HTML.

Với dynamic route (`pages/blog/[slug].js`), Next.js cần biết sẽ build bao nhiêu trang, và build trang nào. Đây là lý do `getStaticPaths` tồn tại — sẽ học chi tiết ở M2.6.

Để M2.5 có ý nghĩa, cần hiểu cơ chế `fallback` ở mức cơ bản:

```javascript
// pages/blog/[slug].js

export async function getStaticProps({ params }) {
  const post = await db.posts.findBySlug(params.slug);

  if (!post) return { notFound: true };

  return { props: { post } };
}

// getStaticPaths — cần thiết khi getStaticProps dùng với dynamic route
export async function getStaticPaths() {
  const posts = await db.posts.findAll({ select: ['slug'] });

  return {
    paths: posts.map(post => ({ params: { slug: post.slug } })),

    // fallback quyết định điều gì xảy ra khi request đến slug KHÔNG có trong paths
    fallback: false,      // → 404 ngay lập tức
    // fallback: true,    // → Render loading state, build page ngầm, serve kết quả
    // fallback: 'blocking', // → Chờ server build page, rồi serve (không có loading state)
  };
}
```

**Ba giá trị `fallback` — tóm tắt để dùng với M2.5:**

```
fallback: false
  Phù hợp: Số lượng pages cố định, biết trước tất cả (ví dụ: danh sách sản phẩm nhỏ)
  Hành vi: Slug không có trong paths → 404 ngay

fallback: 'blocking'
  Phù hợp: Số lượng pages lớn, không build hết tại deploy (ví dụ: blog 100,000 bài)
  Hành vi: Slug chưa build → server build on-demand, user chờ, sau đó cache

fallback: true
  Phù hợp: Muốn hiển thị loading state trong khi build
  Hành vi: Slug chưa build → ngay lập tức trả về page với props rỗng, build ngầm
  Yêu cầu: Component phải handle trạng thái loading (dùng router.isFallback)
```

Chi tiết từng behavior sẽ cover ở M2.6.

### 4.5 Preview Mode

Đây là tính năng ít được biết đến nhưng rất quan trọng trong **editorial workflow**.

**Bài toán:** Content editor đang soạn một bài viết ở CMS, muốn xem preview trước khi publish. Nhưng trang đang là static, được build từ published content. Làm sao show draft content?

**Giải pháp: Preview Mode**

```javascript
// pages/api/preview.js — API route để kích hoạt Preview Mode
export default function handler(req, res) {
  // Verify secret để tránh abuse
  if (req.query.secret !== process.env.PREVIEW_SECRET) {
    return res.status(401).json({ message: 'Invalid token' });
  }

  // Set preview cookies — Next.js tự quản lý
  res.setPreviewData({
    // Data này sẽ được truyền vào context.previewData trong getStaticProps
    userId: req.query.userId, // Ai đang preview
  });

  // Redirect về trang cần preview
  res.redirect(req.query.slug || '/');
}
```

```javascript
// pages/api/exit-preview.js — Thoát Preview Mode
export default function handler(req, res) {
  res.clearPreviewData();
  res.redirect('/');
}
```

```javascript
// pages/blog/[slug].js — getStaticProps nhận biết Preview Mode
export async function getStaticProps({ params, preview, previewData }) {
  if (preview) {
    // Đang ở Preview Mode — fetch draft content
    const post = await cmsClient.getPostBySlug(params.slug, {
      status: 'draft', // Lấy draft thay vì published
      userId: previewData?.userId,
    });

    return { props: { post, isPreview: true } };
  }

  // Bình thường — chỉ fetch published content
  const post = await cmsClient.getPostBySlug(params.slug, {
    status: 'published',
  });

  if (!post) return { notFound: true };

  return { props: { post, isPreview: false } };
}
```

**Cơ chế hoạt động của Preview Mode:**

```
Editor click "Preview" trên CMS
→ CMS gọi /api/preview?secret=xxx&slug=/blog/my-post
→ Next.js set preview cookies (bypass static cache)
→ Redirect về /blog/my-post
→ Request đến /blog/my-post có preview cookies
→ Next.js KHÔNG serve static file
→ Chạy getStaticProps với preview: true
→ Fetch draft content → Render on-demand
→ Editor thấy draft content
```

Khi preview cookies tồn tại, Next.js tự động bypass static cache và chạy `getStaticProps` như SSR. Đây là behavior built-in của Next.js, không cần implement thủ công.

**App Router thay thế Preview Mode bằng Draft Mode** — cơ chế tương tự nhưng tên gọi khác và implementation clean hơn.

### 4.6 Đọc File System trong `getStaticProps`

Vì `getStaticProps` chạy trên Node.js tại build time, bạn có thể đọc file system trực tiếp — không cần API:

```javascript
// pages/docs/[slug].js — Documentation site từ Markdown files
import fs from 'fs';
import path from 'path';
import matter from 'gray-matter';      // Parse frontmatter YAML
import { remark } from 'remark';
import html from 'remark-html';

const docsDirectory = path.join(process.cwd(), 'docs');

export async function getStaticProps({ params }) {
  const filePath = path.join(docsDirectory, `${params.slug}.md`);

  // File không tồn tại
  if (!fs.existsSync(filePath)) {
    return { notFound: true };
  }

  const fileContents = fs.readFileSync(filePath, 'utf8');

  // Parse frontmatter và content
  const { data: frontmatter, content } = matter(fileContents);

  // Convert Markdown → HTML
  const processedContent = await remark().use(html).process(content);
  const contentHtml = processedContent.toString();

  return {
    props: {
      frontmatter,
      contentHtml,
      slug: params.slug,
    },
  };
}

export async function getStaticPaths() {
  const files = fs.readdirSync(docsDirectory);
  const slugs = files
    .filter(f => f.endsWith('.md'))
    .map(f => f.replace(/\.md$/, ''));

  return {
    paths: slugs.map(slug => ({ params: { slug } })),
    fallback: false,
  };
}
```

**Đây là pattern của nhiều documentation sites thực tế** (Next.js docs chính thức trước đây dùng pattern tương tự). File Markdown là source of truth, `getStaticProps` transform chúng thành HTML tại build time.

## 5. Legacy Recognition

**Signal 1: Named export `getStaticProps`**
```javascript
export async function getStaticProps(context) { ... }
// Không bao giờ có req hay res trong context
```

**Signal 2: Thường đi kèm `getStaticPaths` trong file**
```javascript
// Hai exports này thường xuất hiện cùng nhau với dynamic routes
export async function getStaticProps({ params }) { ... }
export async function getStaticPaths() { ... }
```

**Signal 3: Import Node.js modules trực tiếp**
```javascript
import fs from 'fs';
import path from 'path';
// fs và path được dùng tự nhiên — code biết mình chạy trên Node.js tại build time
```

**Signal 4: Gọi trực tiếp CMS SDK, không qua API route**
```javascript
import { createClient } from 'contentful';
// Gọi thẳng SDK — không cần đi qua /api/* vì chạy server-side tại build time
const client = createClient({ space: '...', accessToken: '...' });
```

**Signal 5: Không có `revalidate` → pure static, có `revalidate` → ISR**
```javascript
return { props: { ... } }                  // Pure static
return { props: { ... }, revalidate: 60 }  // ISR — rebuild mỗi 60s nếu có request
```

## 6. Modern Approach

Trong App Router (Next.js 13+), `getStaticProps` không còn tồn tại. Thay thế là fetch với cache config:

```javascript
// app/blog/[slug]/page.js — App Router

async function BlogPostPage({ params }) {
  // Mặc định: static (cache: 'force-cache')
  // Tương đương getStaticProps
  const post = await fetch(`https://cms.example.com/posts/${params.slug}`, {
    cache: 'force-cache', // Cached indefinitely (until revalidated)
  }).then(r => r.json());

  if (!post) notFound(); // Import từ 'next/navigation'

  return <PostContent post={post} />;
}

// generateStaticParams thay thế getStaticPaths
export async function generateStaticParams() {
  const posts = await fetch('https://cms.example.com/posts').then(r => r.json());
  return posts.map(post => ({ slug: post.slug }));
}

// ISR equivalent
export const revalidate = 3600; // Revalidate sau 1 giờ
```

**Mapping conceptual:**

```
Pages Router                    App Router
getStaticProps               →  fetch với cache: 'force-cache' (default)
getStaticPaths               →  generateStaticParams
revalidate trong return      →  export const revalidate = N
Preview Mode                 →  Draft Mode (tương tự nhưng clean hơn)
```

## 7. Production Scenario

**Scenario: E-commerce catalog với 50,000 sản phẩm**

Không thể build tất cả 50,000 trang tại deploy (quá lâu). Nhưng cũng không muốn SSR cho mọi request (quá tốn kém). Giải pháp: Static Generation với `fallback: 'blocking'`.

```javascript
// pages/products/[id].js

export async function getStaticProps({ params }) {
  const product = await db.products.findById(params.id, {
    include: ['images', 'variants', 'reviews'],
  });

  if (!product || !product.isPublished) {
    return { notFound: true };
  }

  // Serialize dates và các giá trị không JSON-safe
  return {
    props: {
      product: JSON.parse(JSON.stringify(product)), // Deep serialize
    },
    revalidate: 300, // Rebuild sau 5 phút nếu có request mới (ISR)
  };
}

export async function getStaticPaths() {
  // Chỉ pre-build 1,000 sản phẩm phổ biến nhất tại deploy
  const popularProducts = await db.products.findMostViewed(1000);

  return {
    paths: popularProducts.map(p => ({
      params: { id: String(p.id) },
    })),
    fallback: 'blocking',
    // 49,000 sản phẩm còn lại: build on-demand khi có request đầu tiên
    // Sau đó cached — request tiếp theo nhận file static
  };
}
```

**Trade-off của strategy này:**

```
1,000 sản phẩm phổ biến:
  → Build sẵn tại deploy
  → Mọi request → CDN cache → TTFB < 50ms

49,000 sản phẩm còn lại:
  → Request đầu tiên → Server build on-demand → ~500ms-2s
  → Request tiếp theo → CDN cache → TTFB < 50ms
  → Deploy time: nhanh hơn nhiều (không build 50,000 trang)

Với revalidate: 300:
  → Giá/stock cập nhật sau tối đa 5 phút
  → Không cần full rebuild khi giá thay đổi
```

Đây là pattern thực tế của nhiều e-commerce lớn dùng Next.js.

## 8. Debugging Scenario

**Bug: Data trên production luôn hiển thị thông tin cũ dù đã update trên CMS**

```javascript
// pages/homepage.js
export async function getStaticProps() {
  const hero = await cmsClient.getHeroSection();
  const features = await cmsClient.getFeatures();

  return {
    props: { hero, features },
    // Không có revalidate!
  };
}
```

**Chẩn đoán:**

Không có `revalidate` → trang là **pure static**. Next.js tạo file HTML một lần khi build, và file đó không bao giờ tự cập nhật. Dù CMS được update, file HTML cũ vẫn được phục vụ cho đến khi deploy lại.

```
Build lúc 09:00 → HTML tạo với hero "Summer Sale 50%"
Update CMS lúc 10:00 → hero mới "Back to School"
User vào lúc 11:00 → vẫn thấy "Summer Sale 50%" (file HTML cũ)
Deploy lại lúc 14:00 → lúc này mới thấy "Back to School"
```

**Fix 1: Thêm `revalidate` (ISR)**

```javascript
export async function getStaticProps() {
  const hero = await cmsClient.getHeroSection();
  const features = await cmsClient.getFeatures();

  return {
    props: { hero, features },
    revalidate: 60, // Rebuild trang sau 60 giây nếu có request mới
    // User thấy data mới tối đa sau 60 giây
  };
}
```

**Fix 2: On-demand ISR — Invalidate ngay khi CMS update (Next.js 12.2+)**

```javascript
// pages/api/revalidate.js — CMS gọi webhook này khi có update
export default async function handler(req, res) {
  if (req.query.secret !== process.env.REVALIDATION_SECRET) {
    return res.status(401).json({ message: 'Invalid token' });
  }

  try {
    await res.revalidate('/'); // Invalidate homepage
    return res.json({ revalidated: true });
  } catch (err) {
    return res.status(500).send('Error revalidating');
  }
}
```

```
CMS update hero section
→ CMS gọi webhook /api/revalidate?secret=xxx
→ Next.js rebuild /
→ User request tiếp theo → HTML mới với hero mới
→ Data cũ tồn tại: chỉ vài giây (thời gian rebuild)
```

Fix 2 là production-grade solution: data luôn cập nhật gần như real-time mà không tốn chi phí SSR.

## 9. Refactoring Scenario

**Nhiệm vụ: Migrate trang blog từ `getServerSideProps` sang `getStaticProps` + ISR**

Codebase hiện tại dùng `getServerSideProps` cho mọi trang blog — ngay cả những bài không có user-specific content.

**Trước:**

```javascript
// pages/blog/[slug].js — dùng SSR không cần thiết

function BlogPost({ post }) {
  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.contentHtml }} />
    </article>
  );
}

export async function getServerSideProps({ params }) {
  const post = await db.posts.findBySlug(params.slug, {
    where: { status: 'published' }
  });

  if (!post) return { notFound: true };

  return { props: { post } };
}
// Vấn đề: Mỗi trong 10,000 request/ngày đều hit DB và render trên server
```

**Sau:**

```javascript
// pages/blog/[slug].js — sau khi migrate

function BlogPost({ post }) {
  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.contentHtml }} />
    </article>
  );
}

export async function getStaticProps({ params, preview, previewData }) {
  // Handle Preview Mode cho editor
  if (preview) {
    const draftPost = await db.posts.findBySlug(params.slug, {
      where: { status: 'draft' }
    });
    return { props: { post: draftPost, isPreview: true } };
  }

  const post = await db.posts.findBySlug(params.slug, {
    where: { status: 'published' }
  });

  if (!post) return { notFound: true };

  return {
    props: { post, isPreview: false },
    revalidate: 3600, // Rebuild sau 1 giờ nếu có request mới
  };
}

export async function getStaticPaths() {
  // Chỉ pre-build 100 bài mới nhất — bài cũ build on-demand
  const recentPosts = await db.posts.findAll({
    where: { status: 'published' },
    orderBy: { publishedAt: 'desc' },
    take: 100,

  return {
    paths: recentPosts.map(p => ({ params: { slug: p.slug } })),
    fallback: 'blocking',
  };
}

export default BlogPost;
```

**Kết quả đo được (ví dụ thực tế):**

```
Trước (getServerSideProps):
  TTFB trung bình: 180–400ms
  Server CPU: cao
  DB queries: 10,000/ngày cho blog

Sau (getStaticProps + ISR):
  TTFB trung bình: 20–50ms (từ CDN)
  Server CPU: gần 0 (chỉ khi revalidate)
  DB queries: ~100/ngày (chỉ khi cache miss hoặc revalidate)
```

## 10. Architecture Notes

### Ranh giới giữa Static và Dynamic

**Nguyên tắc cơ bản:** Nếu output của một page phụ thuộc vào **ai đang xem**, thì không thể static. Nếu output phụ thuộc vào **nội dung**, có thể static.

```
Phụ thuộc vào AI đang xem → Dynamic (getServerSideProps / client-side)
  - Dashboard cá nhân: "Xin chào, Nguyễn Văn A"
  - Giỏ hàng: sản phẩm của user cụ thể
  - Feed mạng xã hội: bài của người user follow

Phụ thuộc vào NỘI DUNG → Static (getStaticProps)
  - Bài blog: nội dung không đổi theo người xem
  - Trang sản phẩm: giá công khai, mô tả sản phẩm
  - Trang tài liệu: documentation content
  - Landing page: marketing content
```

**Edge case phổ biến — "gần static":**

```javascript
// Trang sản phẩm có review count và average rating
// Review count thay đổi mỗi vài giờ — không phải mỗi request
// → Dùng getStaticProps + revalidate, KHÔNG phải getServerSideProps

export async function getStaticProps({ params }) {
  const [product, reviews] = await Promise.all([
    db.products.findById(params.id),
    db.reviews.getStats(params.id),
  ]);

  return {
    props: { product, reviews },
    revalidate: 300, // 5 phút — đủ fresh cho review stats
  };
}
```

### Build Time Implications

`getStaticProps` chạy tại build time — đây có implications quan trọng:

**1. Build time tăng theo số lượng pages:**

```
10 pages × 100ms/page = 1 giây build time
10,000 pages × 100ms/page = ~17 phút build time
```

Với site lớn, dùng `fallback: 'blocking'` để chỉ pre-build subset quan trọng.

**2. Môi trường build ≠ môi trường runtime:**

```javascript
export async function getStaticProps() {
  // Biến môi trường tại BUILD TIME (khi chạy next build)
  // Không phải biến môi trường của server đang chạy!
  const apiKey = process.env.CMS_API_KEY;

  // ⚠️ Nếu biến này thay đổi sau build, cần rebuild để có effect
  // Ngược lại với getServerSideProps: đọc env tại runtime mỗi request
}
```

**3. Không thể đọc request headers hay cookies:**

```javascript
export async function getStaticProps() {
  // ❌ Không thể đọc Accept-Language header từ browser
  // Giải pháp: dùng Next.js i18n config + locale trong context
  // hoặc detect language ở client-side

  return { props: {} };
}
```

### So sánh đầy đủ ba Data Fetching Methods

| Tiêu chí | `getInitialProps` | `getServerSideProps` | `getStaticProps` |
|---------|------------------|---------------------|-----------------|
| Chạy khi nào | Server + Client | Server (mỗi request) | Build time |
| Có `req`/`res` | Có (chỉ trên server) | Luôn có | Không có |
| Có `params` | Qua `query` | Có | Có |
| TTFB | Cao | Cao | Rất thấp (CDN) |
| Data freshness | Luôn fresh | Luôn fresh | Phụ thuộc revalidate |
| Server load | Cao | Cao | Gần 0 |
| Caching | Thủ công | Thủ công qua header | Built-in (CDN) |
| User-specific | Có thể | Có thể | Không thể |
| Dùng Node.js APIs | Phức tạp | Thoải mái | Thoải mái |
| Static Optimization | Bị block bởi _app | Không block | Tận dụng hoàn toàn |

## 11. Interview Questions

**Junior Level:**

**Q: `getStaticProps` chạy ở đâu và khi nào?**

<details>
<summary>Đáp án</summary>

`getStaticProps` chạy **trên server, tại build time** — khi developer chạy lệnh `next build` trước khi deploy.

Không chạy trên client. Không chạy mỗi khi user request. Chỉ chạy một lần khi build (hoặc định kỳ nếu có ISR với `revalidate`).

Hệ quả: code trong `getStaticProps` không bao giờ được ship xuống browser, nên có thể dùng thoải mái các server-side logic như đọc database, file system, API keys private. Nhưng cũng không có `req`, `res`, hay bất kỳ thông tin nào về user đang xem trang.

</details>

---

**Q: Tại sao `getStaticProps` không có `req` và `res` trong context?**

<details>
<summary>Đáp án</summary>

Vì `getStaticProps` chạy tại **build time**, không phải khi user request.

`req` (HTTP request object) và `res` (HTTP response object) chỉ tồn tại trong ngữ cảnh của một HTTP request đang diễn ra. Khi `getStaticProps` chạy lúc build, không có user nào đang gửi request — không có gì để gắn `req` và `res` vào.

Đây cũng là lý do `getStaticProps` không thể đọc cookies hay headers của user, không biết ai đang xem trang, và không thể làm auth redirect dựa trên session. Những tác vụ đó thuộc về request time, không phải build time.

</details>

---

**Mid Level:**

**Q: Sự khác biệt giữa `fallback: false`, `fallback: true`, và `fallback: 'blocking'` trong `getStaticPaths`? Khi nào dùng cái nào?**

<details>
<summary>Đáp án</summary>

Ba giá trị `fallback` quyết định hành vi khi user request đến một path **không có trong danh sách được pre-built**:

`fallback: false` — Path không có trong list → **404 ngay lập tức**. Dùng khi số lượng paths là cố định và đã biết trước hết (ví dụ: danh sách 10 categories). Đơn giản nhất, nhưng mọi path mới cần rebuild.

`fallback: 'blocking'` — Path chưa được build → **server build page on-demand**, user chờ đến khi xong, sau đó page được cache. Request tiếp theo nhận page đã cache. Không có loading state. Dùng cho sites có nhiều pages (blog, e-commerce) không thể build tất cả tại deploy. Đây là lựa chọn phổ biến nhất trong production.

`fallback: true` — Path chưa được build → **ngay lập tức trả về page với props rỗng**, build ngầm, rồi client nhận data mới qua re-render. Component phải handle loading state bằng `router.isFallback`. Phức tạp hơn, ít dùng hơn, nhưng cho UX tốt hơn nếu implement đúng (user thấy layout ngay, data điền vào sau).

Trong thực tế: `fallback: 'blocking'` là lựa chọn mặc định an toàn cho hầu hết cases. `fallback: false` khi số pages nhỏ và cố định. Tránh `fallback: true` trừ khi có lý do UX cụ thể.

</details>

---

**Senior Level:**

**Q: Một trang product listing cần hiển thị giá (cập nhật mỗi giờ) và stock (cập nhật real-time). Bạn sẽ thiết kế data fetching như thế nào?**

<details>
<summary>Đáp án</summary>

Đây là bài toán **hybrid fetching** — không phải mọi data đều có cùng requirements về freshness.

Phân tích từng loại data:

Thông tin sản phẩm (tên, mô tả, hình ảnh): thay đổi hiếm → `getStaticProps` + `revalidate` dài (ví dụ: 24 giờ)

Giá: cập nhật mỗi giờ → `getStaticProps` + `revalidate: 3600`

Stock: real-time → **không thể static**, phải client-side fetch

```javascript
// pages/products/[id].js

function ProductPage({ product, price }) {
  // Stock fetch ở client — real-time
  const { data: stock } = useSWR(
    `/api/products/${product.id}/stock`,
    fetcher,
    { refreshInterval: 30000 } // Refresh mỗi 30 giây
  );

  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <Price amount={price.current} currency={price.currency} />
      <StockIndicator
        count={stock?.available ?? null}
        isLoading={!stock}
      />
      <AddToCartButton disabled={stock?.available === 0} />
    </div>
  );
}

export async function getStaticProps({ params }) {
  // Fetch static + semi-static data tại build time
  const [product, price] = await Promise.all([
    db.products.findById(params.id),
    pricingService.getCurrentPrice(params.id),
  ]);

  if (!product) return { notFound: true };

  return {
    props: { product, price },
    revalidate: 3600, // Giá rebuild sau 1 giờ
  };
}
```

Kết quả: Page load cực nhanh từ CDN (product info + giá từ last build), stock cập nhật real-time ở client. Không cần SSR, không cần hit server cho mỗi page view.

Đây là pattern "static shell + dynamic islands" — phần lớn page là static, chỉ phần thực sự cần dynamic mới fetch ở client.

</details>

**Bài 1:** Giải thích output của đoạn code sau. `getStaticProps` sẽ nhận được gì trong `context`? HTML được tạo khi nào?

```javascript
// pages/docs/[...slug].js
export async function getStaticProps({ params }) {
  console.log('params:', params);
  return { props: { params } };

export async function getStaticPaths() {
  return {

      { params: { slug: ['getting-started'] } },
```

<details>
<summary>Đáp án</summary>

Route `pages/docs/[...slug].js` là **catch-all route** — `[...slug]` bắt nhiều segments.

`getStaticPaths` định nghĩa 3 paths sẽ được build:
- `/docs/getting-started`
- `/docs/api/reference`
- `/docs/api/reference/hooks`

`getStaticProps` sẽ được gọi **3 lần** tại build time, mỗi lần với một path khác nhau.

Output của `console.log('params:', params)`:

Lần 1: `params: { slug: ['getting-started'] }`

Lần 2: `params: { slug: ['api', 'reference'] }`

Lần 3: `params: { slug: ['api', 'reference', 'hooks'] }`

`params.slug` luôn là **array** với catch-all routes, ngay cả khi chỉ có một segment. Đây là khác biệt với dynamic route thường `[slug]` — nơi `params.slug` là string.

HTML được tạo khi nào: **3 file HTML được tạo tại build time** (`next build`). Không chạy lại khi user request.

Vì `fallback: false`, bất kỳ URL nào không có trong 3 paths trên (ví dụ: `/docs/unknown`) sẽ trả về 404.

</details>

**Bài 2:** Code sau có vấn đề gì? Phân tích và fix:

```javascript
// pages/profile/[username].js

function ProfilePage({ user, posts }) {
  return (
    <div>
      <h1>{user.name}</h1>
      <p>Joined: {user.createdAt.toLocaleDateString()}</p>
      <PostList posts={posts} />
    </div>
  );
}

export async function getStaticProps({ params }) {
  const user = await db.users.findByUsername(params.username);

  if (!user) return { notFound: true };

  const posts = await db.posts.findByUserId(user.id, {
    where: { isPublic: true },
    orderBy: { publishedAt: 'desc' },
  });

  return {
    props: { user, posts },
  };
}

export async function getStaticPaths() {
  const users = await db.users.findAll();
  return {
    paths: users.map(u => ({ params: { username: u.username } })),
    fallback: false,
  };
}

export default ProfilePage;
```

<details>
<summary>Đáp án</summary>

Có 3 vấn đề:

**Vấn đề 1: `user.createdAt.toLocaleDateString()` sẽ crash**

`getStaticProps` serialize props thành JSON. Đối tượng `Date` không serialize thành `Date` — nó thành string khi JSON.stringify. Khi component nhận được `user`, `user.createdAt` là một **string**, không phải `Date` object. Gọi `.toLocaleDateString()` trên string sẽ throw `TypeError`.

Fix: Serialize date trước khi return, và xử lý trong component:

```javascript
// Trong getStaticProps:
return {
  props: {
    user: {
      ...user,
      createdAt: user.createdAt.toISOString(), // String rõ ràng
    },
    posts,
  },
};

// Trong component:
<p>Joined: {new Date(user.createdAt).toLocaleDateString('vi-VN')}</p>
```

**Vấn đề 2: `getStaticPaths` build tất cả users — không scalable**

`db.users.findAll()` có thể trả về hàng triệu users. Build tất cả trang profiles tại deploy time là không thực tế — build time sẽ vô cùng lâu.

Fix: Chỉ pre-build profiles của users active hoặc phổ biến, dùng `fallback: 'blocking'` cho phần còn lại:

```javascript
export async function getStaticPaths() {
  const activeUsers = await db.users.findMostActive(500); // Top 500
  return {
    paths: activeUsers.map(u => ({ params: { username: u.username } })),
    fallback: 'blocking', // Build on-demand cho phần còn lại
  };
}
```

**Vấn đề 3: Không có `revalidate` — profile tĩnh mãi mãi**

User đăng bài mới, thêm bio, update avatar → trang profile không bao giờ cập nhật cho đến khi deploy lại.

Fix: Thêm `revalidate` phù hợp:

```javascript
return {
  props: { user, posts },
  revalidate: 3600, // Rebuild sau 1 giờ nếu có request mới
};
```

</details>

**Bài 3 (Nâng cao):** Bạn đang xây dựng một documentation site với 500 trang Markdown. CMS team muốn có khả năng preview draft pages trước khi publish. Thiết kế đầy đủ solution bao gồm: cấu trúc `getStaticProps`, Preview Mode API routes, và cách CMS gọi preview. Giải thích trade-off của từng quyết định thiết kế.

<details>
<summary>Đáp án</summary>

**Kiến trúc tổng thể:**

```
CMS (Sanity/Contentful)
  │
  ├── Publish content → Webhook → /api/revalidate → Rebuild page
  │
  └── Preview draft → /api/preview?secret=xxx&slug=/docs/intro → Set cookies → Redirect
```

**Bước 1: `getStaticProps` với Preview Mode support**

```javascript
// pages/docs/[...slug].js

export async function getStaticProps({ params, preview, previewData }) {
  const slug = params.slug.join('/');

  const doc = await cmsClient.getDoc(slug, {
    // preview: true → fetch draft, false → fetch published only
    useDraft: preview,
    // previewData chứa thông tin về ai đang preview (nếu cần audit)
    userId: previewData?.userId,
  });

  if (!doc) return { notFound: true };

  return {
    props: {
      doc,
      isPreview: !!preview,
    },
    // preview mode bypass revalidate — luôn render fresh
    // non-preview: rebuild sau 1 giờ
    revalidate: preview ? undefined : 3600,
  };
}
```

**Bước 2: Preview API Route**

```javascript
// pages/api/preview.js

export default function handler(req, res) {
  // Verify secret — tránh bất kỳ ai kích hoạt preview
  if (req.query.secret !== process.env.PREVIEW_SECRET || !req.query.slug) {
    return res.status(401).json({ message: 'Invalid token or missing slug' });
  }

  // Validate slug format để tránh open redirect
  const slug = req.query.slug;
  if (!slug.startsWith('/docs/')) {
    return res.status(400).json({ message: 'Invalid slug' });
  }

  // Set preview mode cookies
  // previewData được truyền vào context.previewData trong getStaticProps
  res.setPreviewData({
    userId: req.query.userId || 'anonymous',
  });

  // Redirect đến trang cần preview
  res.redirect(slug);
}
```

```javascript
// pages/api/exit-preview.js

export default function handler(req, res) {
  res.clearPreviewData();
  res.redirect(req.query.returnTo || '/');
}
```

**Bước 3: On-demand Revalidation khi publish**

```javascript
// pages/api/revalidate.js — CMS gọi webhook này khi publish

export default async function handler(req, res) {
  if (req.method !== 'POST') {
    return res.status(405).json({ message: 'Method not allowed' });
  }

  if (req.headers['x-webhook-secret'] !== process.env.REVALIDATION_SECRET) {
    return res.status(401).json({ message: 'Invalid secret' });
  }

  const { slug } = req.body;

  if (!slug) {
    return res.status(400).json({ message: 'Missing slug' });
  }

  try {
    await res.revalidate(`/docs/${slug}`);
    return res.json({ revalidated: true, slug });
  } catch (err) {
    return res.status(500).json({ message: 'Error revalidating', error: err.message });
  }
}
```

**Cách CMS gọi preview (từ phía CMS):**

```javascript
// CMS webhook config (ví dụ trong Sanity Studio)
const previewUrl = `https://your-site.com/api/preview`
  + `?secret=${PREVIEW_SECRET}`
  + `&slug=/docs/${document.slug.current}`
  + `&userId=${currentUser.id}`;

// CMS gọi URL này khi editor click "Preview"
```

**Trade-off của từng quyết định:**

`revalidate: 3600` — Trade-off giữa freshness và server cost. 1 giờ là hợp lý cho documentation (không thay đổi liên tục), nhưng nếu có lỗi typo được fix, user có thể thấy bản cũ đến 1 giờ. On-demand revalidation giải quyết vấn đề này khi kết hợp với CMS webhook.

`fallback: 'blocking'` (nên dùng cho 500 trang) — Build tất cả 500 trang tại deploy mất thời gian nhưng không phải vấn đề lớn. Tuy nhiên nếu sau này site có thêm nhiều trang, `fallback: 'blocking'` cho phép scale tốt hơn mà không cần thay đổi config.

Validate slug trong preview API — Không validate slug có thể gây **open redirect vulnerability**: attacker gọi `/api/preview?secret=xxx&slug=https://evil.com` → redirect đến site độc hại. Validation `slug.startsWith('/docs/')` ngăn chặn điều này.

Preview secret trong env var — Secret phải được rotate định kỳ. Nếu bị lộ, attacker có thể xem draft content. Không nên hardcode trong code.

</details>

## 13. Summary

`getStaticProps` đại diện cho một tư duy hoàn toàn khác với `getServerSideProps`: thay vì làm việc tại request time, hãy làm việc **một lần tại build time** và phục vụ kết quả cho mọi request.

**Hai câu hỏi để quyết định có dùng `getStaticProps` không:**

```
1. Output của page này có thay đổi theo từng user không?
   → Có: getServerSideProps hoặc client-side fetch
   → Không: getStaticProps là lựa chọn đúng

2. Data thay đổi nhanh đến mức nào?
   → Real-time: client-side fetch
   → Vài phút/giờ: getStaticProps + revalidate (ISR)
   → Vài ngày/tháng: getStaticProps thuần
```

**Ba điều cần nhớ về context object:**

Không có `req`, `res` — không có HTTP request đang diễn ra. Có `params` — biết page nào đang build. Có `preview` — cho editorial workflow với draft content.

**Một nguyên tắc tổng quát:**

`getStaticProps` và `getServerSideProps` không phải hai lựa chọn về "cái nào đúng". Chúng giải quyết hai bài toán khác nhau. Dùng sai công cụ — dùng `getServerSideProps` cho content không cần per-request freshness — là nguồn gốc của vô số performance issue trong Next.js codebase thực tế.

Bài tiếp theo **M2.6** sẽ đào sâu vào `getStaticPaths` — cơ chế để `getStaticProps` biết cần build những trang nào, và ba chiến lược `fallback` với implications thực tế của từng cái.
