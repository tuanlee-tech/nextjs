# M2.1 — Pages Router: File-based Routing — Định tuyến dựa trên file kế thừa

## 1. Lesson Goal

Module M2 là module 🔴 quan trọng nhất cho mục tiêu **maintain legacy** của bạn. Bài đầu tiên này phải giúp bạn:

- Đọc một thư mục `pages/` bất kỳ và **suy ra ngay** toàn bộ sitemap của ứng dụng, không cần mở file routing config nào cả.
- Phân biệt chính xác vai trò của `_app.js`, `_document.js`, `_error.js` — ba file dễ bị nhầm lẫn nhất với người mới đọc codebase legacy.
- Nhận ra **rủi ro kiến trúc gốc** của quy ước "mọi file trong `pages/` đều là route" — một lỗi rất phổ biến trong các dự án Next 9-12 thực tế.
- Hiểu tại sao quy ước này, dù là một bước tiến lớn so với routing config tập trung, lại **không đủ** để xử lý layout phức tạp — tạo tiền đề cho App Router (M3) ra đời.

## 2. Evolution Context

**Pain Point gốc (trước Next.js, dùng React Router thuần):** Mọi route phải được khai báo thủ công trong một file config trung tâm:

```jsx
// routes.js — React Router thuần, KHÔNG phải Next.js
<Switch>
  <Route
    exact
    path='/'
    component={Home}
  />
  <Route
    path='/about'
    component={About}
  />
  <Route
    path='/blog/:slug'
    component={BlogPost}
  />
  <Route
    path='/dashboard/settings'
    component={Settings}
  />
  {/* ... hàng chục route khác, một file phình to liên tục */}
</Switch>
```

**Tại sao thất bại ở quy mô lớn:**

- File `routes.js` trở thành **điểm xung đột merge** kinh điển — mọi người thêm trang mới đều phải sửa cùng một file.
- Không có liên hệ trực tiếp giữa cấu trúc thư mục và cấu trúc URL — muốn biết `/dashboard/settings` render component nào, phải tra cứu qua file config, không thể đoán từ cấu trúc folder.
- Code-splitting theo route phải tự thiết lập tay bằng `React.lazy()` + `Suspense` cho từng route — dễ quên, dễ làm sai.

**Cách mới (Next.js, từ phiên bản đầu tiên, hoàn thiện dần tới Next 9-12):** Routing dựa trên file (file-based routing) — cấu trúc thư mục `pages/` **chính là** sơ đồ route, không cần file config nào khác:

```
pages/
  index.js          → /
  about.js          → /about
  blog/
    index.js        → /blog
    [slug].js       → /blog/:slug
```

Code-splitting theo từng file là **tự động**, do framework đảm nhiệm — mỗi file trong `pages/` tự động trở thành một JS bundle riêng, chỉ tải khi người dùng truy cập đúng route đó (hoặc được `<Link>` prefetch trước).

## 3. Mental Model

Hãy coi thư mục `pages/` là **sơ đồ trang web viết bằng hệ thống file**, theo đúng nghĩa đen — không phải một thư mục chứa code thuần túy như mọi thư mục khác trong dự án.

Hệ quả tất yếu của mental model này, và đây là điểm cốt lõi cần khắc sâu:

> **Next.js Pages Router không phân biệt "file code" và "file route" trong `pages/`. Với framework, mọi file hợp lệ (.js/.jsx/.ts/.tsx) nằm trong `pages/` đều là một route công khai, không có ngoại lệ trừ các tên đặc biệt được dự trữ.**

Đây không phải là một chi tiết nhỏ — nó là nguồn gốc của một trong những lỗi kiến trúc phổ biến nhất ở các dự án Next.js legacy, sẽ phân tích kỹ ở mục 7 và mục 8.

## 4. Core Concepts

**4.1. Quy tắc mapping cơ bản**

| File                                  | URL                           |
| ------------------------------------- | ----------------------------- |
| `pages/index.js`                      | `/`                           |
| `pages/about.js`                      | `/about`                      |
| `pages/blog/index.js`                 | `/blog`                       |
| `pages/blog/first-post.js`            | `/blog/first-post`            |
| `pages/dashboard/settings/profile.js` | `/dashboard/settings/profile` |

Lồng folder = lồng URL path, **trực tiếp 1:1**, không có cơ chế "route group" để gom file mà không ảnh hưởng URL (khác với App Router, xem mục 6).

**4.2. Dynamic Routes — Định tuyến động**

```
pages/blog/[slug].js        → khớp /blog/hello-world
                               useRouter().query.slug === 'hello-world'

pages/blog/[...slug].js     → catch-all, khớp /blog/a, /blog/a/b, /blog/a/b/c
                               KHÔNG khớp /blog (thiếu segment)
                               query.slug === ['a', 'b', 'c']

pages/blog/[[...slug]].js   → optional catch-all, khớp CẢ /blog
                               VÀ /blog/a/b/c
                               query.slug === undefined khi ở /blog
```

```tsx
// pages/blog/[slug].tsx
import { useRouter } from 'next/router';

export default function BlogPost() {
  const router = useRouter();
  const { slug } = router.query; // lấy giá trị segment động từ URL
  return <article>Đang xem bài viết: {slug}</article>;
}
```

**4.3. Các file đặc biệt (Special Files) — dự trữ theo tên, không theo quy ước chung**

Khác với việc đoán quy ước, đây là **danh sách tên file cụ thể** mà framework dành riêng:

```tsx
// pages/_app.js — bọc MỌI page, nơi đặt global CSS, context provider, layout chung
function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
export default MyApp;
```

```tsx
// pages/_document.js — tùy biến khung <html>/<body>, CHỈ chạy ở server
// Không thể dùng useState/useEffect/event handler ở đây — đây không phải
// một "page" bình thường, nó định nghĩa SHELL HTML cho mọi request.
import Document, { Html, Head, Main, NextScript } from 'next/document';

class MyDocument extends Document {
  render() {
    return (
      <Html lang='vi'>
        <Head />
        <body>
          <Main />
          <NextScript />
        </body>
      </Html>
    );
  }
}
export default MyDocument;
```

```tsx
// pages/_error.js — trang lỗi fallback dùng chung cho cả 404 và 500
// nếu KHÔNG có pages/404.js hoặc pages/500.js riêng
function Error({ statusCode }) {
  return (
    <p>
      {statusCode
        ? `Đã xảy ra lỗi ${statusCode} trên server`
        : 'Đã xảy ra lỗi trên client'}
    </p>
  );
}
// Custom _error.js phải định nghĩa `getInitialProps` để lấy statusCode từ `res` hoặc `err`
Error.getInitialProps = ({ res, err }) => {
  const statusCode = res ? res.statusCode : err ? err.statusCode : 404;
  return { statusCode };
};

export default Error;
```

`pages/404.js` và `pages/500.js` là hai file tùy chọn, override riêng cho từng mã lỗi cụ thể, ưu tiên cao hơn `_error.js`.

**Lưu ý quan trọng về `_error.js`:** File này chỉ được sử dụng trong **production**.
Ở môi trường development, Next.js sẽ hiển thị error overlay với stack trace thay vì
render `_error.js`. Điều này có nghĩa là bạn không thể test UI của trang lỗi custom
bằng cách chạy `next dev` — cần build và start production (`next build && next start`)
hoặc dùng `next dev` với production error pages (không được hỗ trợ trực tiếp).

**4.4. `pages/api/*` — không phải page, là serverless function**

```
pages/api/users.js   → endpoint API tại /api/users, KHÔNG render UI,
                         export default function handler(req, res) {...}
```

Đây sẽ là chủ đề riêng ở M2.7 — ở đây chỉ cần nhớ: framework nhận diện prefix `api/` để xử lý khác hẳn route UI thông thường (không bundle JS client, không render React).

## 5. Legacy Recognition

Dấu hiệu nhận diện một dự án đang ở "Pages Router Era" thực thụ, không chỉ qua tên thư mục:

- Có `pages/` (hoặc `src/pages/`) ở gốc dự án, **không có** `app/` — hoặc có cả hai cùng lúc (dấu hiệu dự án **đang trong quá trình migrate dần**, vì Next.js cho phép `pages/` và `app/` tồn tại song song để migrate từng phần, sẽ học kỹ ở M2.10/M17).
- Có `pages/_app.js` với một pattern rất đặc trưng của thời kỳ chưa có layout component gốc:

```tsx
// pages/_app.js — pattern "Per-Page Layout" rất phổ biến trong code Next 10-12
// Đây là cách CỘNG ĐỒNG tự giải quyết vấn đề "không có layout lồng nhau"
// trước khi App Router ra đời (xem mục 6 và mục 9)
function MyApp({ Component, pageProps }) {
  const getLayout = Component.getLayout ?? ((page) => page);
  return getLayout(<Component {...pageProps} />);
}
export default MyApp;
```

- Nếu thấy pattern `Component.getLayout = ...` rải rác ở nhiều file `pages/*.js` → đây là dấu hiệu rõ ràng dự án đã từng cố gắng xử lý layout phức tạp bằng tay, **không phải** bug, mà là một workaround có chủ đích — hiểu được lý do tồn tại của nó (mục 9) sẽ giúp bạn không xóa nhầm khi maintain.
- **Dấu hiệu rủi ro cần đặc biệt chú ý:** một file component thuần túy (ví dụ `Button.jsx`, `utils.js`) bị đặt **trực tiếp trong** `pages/` (không phải trong `pages/api/` hay một thư mục `components/` riêng). Vì quy tắc ở mục 3, file này **sẽ trở thành một route công khai thực sự** tại `/Button`, dù không ai có ý định đó.

**Caveat khi coexist `pages/` và `app/`:** Mặc dù Next.js cho phép cả hai thư mục
tồn tại song song (từ Next.js 13.4+), việc navigate giữa một page ở `pages/` và một
page ở `app/` thường gây ra **full page reload** hoặc loading state "nhấp nháy" —
giống như navigate giữa hai ứng dụng độc lập. Điều này là do hai router có kiến trúc
rendering tree khác nhau và không share client cache/state. Chiến lược migration
incremental vẫn khả thi, nhưng cần lưu ý UX breakpoint tại ranh giới hai router. Nhiều team tưởng rằng coexist là "trong suốt" nhưng thực tế có UX penalty khi user navigate qua lại giữa hai router.

## 6. Modern Approach

Đây chỉ là cái nhìn trước (preview) — chi tiết đầy đủ thuộc về Module M3, nhưng cần nắm ngay để hiểu **động lực** ra đời của App Router:

- **Private Folders (`_folderName`):** App Router cho phép đặt một folder với tiền tố `_` để **loại trừ hoàn toàn** nó (và mọi folder con) khỏi routing — bạn có thể colocate component, helper ngay cạnh route mà không lo trở thành route công khai. Đây là điều Pages Router **chưa từng có**: trong `pages/`, mọi file luôn là route, không có cơ chế "private" nào, đây chính là nguồn gốc rủi ro đã nói ở mục 5.
- **Chỉ tên file đặc biệt mới là route (`page.tsx`, `route.tsx`)**: trong `app/`, một folder không tự động là route — nó phải chứa file `page.tsx` mới "công khai hóa" route đó. Component phụ trợ đặt cùng cấp folder **không** bị coi là route, khác hẳn nguyên tắc "mọi file = route" của Pages Router.
- **Route Groups (`(groupName)`):** gom các route liên quan vào một folder mà **không ảnh hưởng URL** — giải quyết đúng hạn chế "lồng folder = lồng URL cứng nhắc" của Pages Router.
- **Nested Layouts gốc (`layout.tsx`)**: thay thế hoàn toàn pattern `getLayout` thủ công ở mục 5 — layout giờ là một phần tự nhiên của cấu trúc route, không cần workaround ở `_app.js`.

## 7. Production Scenario

**Bối cảnh:** Dự án Next.js 10, thư mục `pages/` có hơn 70 file sau 2 năm phát triển, không có quy ước rõ ràng về tổ chức. Một kỹ sư mới được giao thêm 3 trang con cho khu vực `/dashboard/settings`: `profile.js`, `billing.js`, `team.js` — cả 3 cần chung một sidebar điều hướng và phải **giữ trạng thái sidebar (mục đang mở, vị trí scroll)** khi chuyển giữa 3 trang.

**Vấn đề nếu làm "ngây thơ":** Wrap `<DashboardLayout>` trực tiếp bên trong từng file page:

```tsx
// pages/dashboard/settings/profile.js — ❌ cách làm ngây thơ
import DashboardLayout from '../../../components/DashboardLayout';

export default function ProfilePage() {
  return (
    <DashboardLayout>
      <ProfileForm />
    </DashboardLayout>
  );
}
```

Vì `_app.js` render `<Component {...pageProps} />`, khi chuyển từ `profile.js` sang `billing.js`, React nhận diện đây là **hai Component khác nhau** ở vị trí gốc của cây — toàn bộ cây, bao gồm `<DashboardLayout>` nằm bên trong mỗi file, **bị unmount và mount lại từ đầu**. Hệ quả thực tế: sidebar bị đóng lại, vị trí scroll reset về đầu trang, mọi state cục bộ trong layout (ví dụ trạng thái "đang hover", animation) bị mất — đúng là bug mà kỹ sư mới gặp phải khi QA test.

**Giải pháp đúng (per-page layout pattern, mục 5):** Đưa `DashboardLayout` lên `_app.js`, gắn vào `Component` qua `getLayout` — vì lúc này `DashboardLayout` là **cùng một component, cùng một vị trí trong cây**, qua mỗi lần chuyển trang chỉ có `children` của nó thay đổi (`profile.js` → `billing.js`), nên React **giữ nguyên instance** của `DashboardLayout`, bảo toàn toàn bộ state nội bộ và vị trí scroll. Chi tiết refactor đầy đủ ở mục 9.

## 8. Debugging Scenario

**Hiện tượng thực tế (có thật, từng được report nhiều lần trong cộng đồng Next.js thời Pages Router):** Google Search Console báo cáo một URL lạ được index: `https://yourapp.com/Button`, trả về một trang trống hoặc lỗi runtime khi truy cập trực tiếp.

**Quy trình debug:**

1. Mở `pages/Button.jsx` (hoặc tên tương tự) — xác nhận đây thực chất là một **component UI tái sử dụng** (`<Button onClick={...}>...</Button>`), được đặt nhầm trực tiếp trong `pages/` thay vì `components/`.
2. Hiểu nguyên nhân gốc: theo mục 3, framework không có khái niệm "file private" trong `pages/` — bất kỳ ai (kể cả bot crawl của Google) truy cập `/Button` đều nhận được kết quả render của component đó **như một page độc lập**, không có props được truyền vào (`onClick` không tồn tại, dữ liệu cần thiết không có) → thường gây lỗi runtime hoặc render rỗng/sai.
3. Kiểm tra mức độ ảnh hưởng: route rác này có bị crawl/index không (ảnh hưởng SEO), có lộ thông tin nội bộ không (nếu component vô tình render dữ liệu nhạy cảm để debug).
4. **Khắc phục:** di chuyển toàn bộ file component thuần túy ra khỏi `pages/`, vào một thư mục riêng ở cấp gốc dự án (ví dụ `components/`), chỉ giữ trong `pages/` đúng những file thực sự là route. Vì Pages Router không có cơ chế private folder, đây là **quy ước team phải tự kỷ luật tuân theo**, không phải thứ framework tự bảo vệ bạn.

## 9. Refactoring Scenario

**Trước (lặp lại layout ở từng file, gây mất state khi chuyển trang — bug ở mục 7):**

```tsx
// pages/dashboard/settings/profile.js
import DashboardLayout from '../../../components/DashboardLayout';
export default function ProfilePage() {
  return (
    <DashboardLayout>
      <ProfileForm />
    </DashboardLayout>
  );
}

// pages/dashboard/settings/billing.js
import DashboardLayout from '../../../components/DashboardLayout';
export default function BillingPage() {
  return (
    <DashboardLayout>
      <BillingForm />
    </DashboardLayout>
  );
}

// pages/dashboard/settings/team.js
import DashboardLayout from '../../../components/DashboardLayout';
export default function TeamPage() {
  return (
    <DashboardLayout>
      <TeamSettings />
    </DashboardLayout>
  );
}
```

**Sau (per-page layout pattern, layout giữ nguyên instance qua mỗi lần chuyển trang):**

```tsx
// pages/_app.js
function MyApp({ Component, pageProps }) {
  // Nếu page không định nghĩa getLayout, fallback: hiển thị page không cần layout
  const getLayout = Component.getLayout ?? ((page) => page);
  return getLayout(<Component {...pageProps} />);
}
export default MyApp;
```

```tsx
// pages/dashboard/settings/profile.js
import DashboardLayout from '../../../components/DashboardLayout';

function ProfilePage() {
  return <ProfileForm />;
}

// Gắn layout NGAY TẠI ĐÂY, nhưng việc RENDER layout diễn ra ở _app.js
ProfilePage.getLayout = function getLayout(page) {
  return <DashboardLayout>{page}</DashboardLayout>;
};

export default ProfilePage;
```

```tsx
// pages/dashboard/settings/billing.js — lặp lại đúng cùng một pattern
import DashboardLayout from '../../../components/DashboardLayout';

function BillingPage() {
  return <BillingForm />;
}

BillingPage.getLayout = function getLayout(page) {
  return <DashboardLayout>{page}</DashboardLayout>;
};

export default BillingPage;
```

**Điểm khác biệt cốt lõi:** Ở phiên bản trước, `DashboardLayout` được tạo **bên trong** từng `Component` khác nhau ở mỗi page — với React, đổi `Component` đồng nghĩa đổi luôn cây con bên trong nó. Ở phiên bản sau, `DashboardLayout` luôn được render **ở đúng cùng một vị trí trong cây của `_app.js`**, chỉ phần `{page}` (children) thay đổi theo route — React reconciliation (đã học M1.2) nhận diện đây là cùng một component instance, nên **không unmount/remount** `DashboardLayout`, bảo toàn state sidebar, scroll position, animation đang chạy.

## 10. Architecture Notes

- **Khi nào quy ước "1 file = 1 route, không có ngoại lệ" vẫn ổn:** các site nhỏ, marketing landing page, blog đơn giản — không có layout phức tạp lồng nhau, không có nhiều người maintain cùng lúc. Việc thiếu route group/private folder không gây đau ở quy mô này.
- **Khi nào nó trở thành nợ kiến trúc thực sự:** ứng dụng dashboard nhiều cấp, nhiều team cùng làm trên `pages/`, cần tổ chức file theo feature mà không muốn lộ ra URL — đây chính là lúc `getLayout` pattern, quy ước team về vị trí file, và cuối cùng là **động lực thật để migrate sang App Router** (sẽ học chi tiết chiến lược ở M2.10).
- **`next.config.js` rewrites/redirects** là một escape hatch để tách URL công khai khỏi cấu trúc file vật lý (ví dụ file `pages/internal/v2/profile.js` nhưng muốn URL công khai là `/profile`) — nhưng đây là giải pháp ở tầng cấu hình, tăng thêm một lớp gián tiếp cần nhớ khi đọc codebase, không giải quyết gốc vấn đề "không thể private hóa file" như mục 6 đã nêu.
- **File-based routing và code-splitting tự động đi cùng nhau theo thiết kế** — đây vẫn là điểm Pages Router làm đúng và App Router giữ nguyên triết lý: không cần nhớ wrap `React.lazy()` cho mỗi route, framework tự đảm nhiệm hoàn toàn ở tầng build.

## 11. Interview Questions

**[Junior] `pages/blog/[slug].js` và `pages/blog/[...slug].js` khác nhau thế nào?**

<details>
<summary>Đáp án</summary>

`[slug].js` khớp với đúng một segment URL (ví dụ `/blog/hello-world`, `query.slug` là một string). `[...slug].js` là catch-all, khớp với một hoặc nhiều segment (`/blog/a`, `/blog/a/b/c`), `query.slug` là một array các segment, và **không** khớp với `/blog` khi không có segment nào phía sau. Nếu cần khớp cả `/blog` (không có segment) lẫn các segment phía sau, phải dùng optional catch-all `[[...slug]].js`.

</details>

**[Junior/Mid] `_app.js` và `_document.js` khác nhau ở điểm cốt lõi nào?**

<details>
<summary>Đáp án</summary>

`_app.js` bọc quanh **component của page** (`<Component {...pageProps}/>`), chạy cả ở server và client như một component React bình thường — nơi phù hợp để đặt global state, context provider, global CSS. `_document.js` định nghĩa khung HTML gốc (`<html>`, `<body>`) cho toàn bộ ứng dụng, chỉ chạy ở **server**, không được dùng hook hay event handler — nó không re-render theo navigation client-side, vì nó không phải một phần của cây React tương tác thông thường mà là một phần của quá trình tạo HTML shell ban đầu.

</details>

**[Mid/Senior] Vì sao việc đặt một file component thuần túy (không phải page) trực tiếp trong `pages/` được xem là một rủi ro kiến trúc, không chỉ là "code không gọn"?**

<details>
<summary>Đáp án</summary>

Vì trong Pages Router, framework không có khái niệm "file private trong `pages/`" — mọi file hợp lệ ở đó đều được công khai như một route thực sự, có thể truy cập trực tiếp qua URL tương ứng với đường dẫn file. Một component không được thiết kế để đứng độc lập (thiếu props cần thiết, phụ thuộc dữ liệu từ component cha) khi bị truy cập như vậy có thể gây lỗi runtime, lộ nội dung không mong muốn, hoặc tệ hơn là bị các bot tìm kiếm crawl và index như một trang thật, ảnh hưởng SEO của toàn site.

</details>

**[Senior/Staff] Pattern `Component.getLayout` ở `_app.js` giải quyết vấn đề gì mà việc wrap layout trực tiếp trong từng page không giải quyết được?**

<details>
<summary>Đáp án</summary>

Khi layout được wrap trực tiếp bên trong mỗi file page, mỗi lần chuyển route, `_app.js` nhận một `Component` khác (khác page khác), khiến toàn bộ cây con — bao gồm layout nằm bên trong — bị unmount và mount lại từ đầu, làm mất mọi state cục bộ của layout (vị trí scroll, trạng thái mở/đóng sidebar, animation). Pattern `getLayout` đưa việc render layout lên `_app.js`, ở một vị trí cố định trong cây component qua mọi lần chuyển route — chỉ phần children thay đổi theo page. Theo cơ chế reconciliation của React, layout được giữ nguyên instance, không unmount, nên toàn bộ state của nó được bảo toàn xuyên suốt quá trình điều hướng.

</details>

**[Staff/Architect] Next.js xử lý việc một dự án có cả `pages/` và `app/` cùng lúc như thế nào, và vì sao điều này quan trọng với chiến lược migration?**

<details>
<summary>Đáp án</summary>

Next.js cho phép `pages/` và `app/` tồn tại song song trong cùng một dự án — các route được định nghĩa ở `app/` sẽ ưu tiên hơn route tương ứng ở `pages/` nếu trùng đường dẫn, cho phép từng route được di chuyển một cách độc lập, không cần "đập đi xây lại" toàn bộ ứng dụng trong một lần release. Đây là nền tảng cho chiến lược Incremental Migration (Strangler Fig pattern) sẽ học chi tiết ở M2.10 và M17 — di chuyển dần từng route sang App Router trong khi phần còn lại của ứng dụng vẫn chạy ổn định trên Pages Router.

</details>

## 12. Exercises

**Bài 1 (Junior/Mid):** Cho cấu trúc file sau trong `pages/`, hãy viết ra URL tương ứng của từng file, và chỉ rõ file nào **không** phải là route công khai theo đúng nghĩa UI page.

```
pages/
  index.js
  contact.js
  products/
    index.js
    [id].js
    [...filters].js
  api/
    products.js
  _app.js
```

<details>
<summary>Đáp án</summary>

`pages/index.js` → `/`

`pages/contact.js` → `/contact`

`pages/products/index.js` → `/products`

`pages/products/[id].js` → `/products/:id` (ví dụ `/products/42`)

`pages/products/[...filters].js` → catch-all, khớp `/products/a`, `/products/a/b`, ... (không khớp `/products` trống vì thiếu segment, route đó đã thuộc về `index.js`)

`pages/api/products.js` → không phải page UI, là API endpoint tại `/api/products`, trả JSON/response thuần, không render React

`pages/_app.js` → không phải route, là file đặc biệt bọc toàn bộ page, không có URL tương ứng

</details>

**Bài 2 (Senior):** Bạn vừa nhận một dự án Pages Router với 8 trang trong `pages/account/*.js`, mỗi trang tự wrap `<AccountLayout>` riêng (giống pattern lỗi ở mục 7). QA report rằng mỗi lần chuyển giữa các trang trong khu vực `/account`, một banner thông báo (hiển thị trong `AccountLayout`, có animation fade-in) bị **chạy lại animation từ đầu** dù người dùng chỉ chuyển tab, không reload trang. Hãy giải thích nguyên nhân và đưa ra hướng refactor.

<details>
<summary>Đáp án</summary>

Nguyên nhân giống chính xác bug đã phân tích ở mục 7: vì `AccountLayout` được tạo **bên trong** từng file page riêng biệt, mỗi lần chuyển route, `_app.js` nhận diện đây là các `Component` khác nhau, nên toàn bộ cây con — bao gồm `AccountLayout` và banner animation nằm trong nó — bị unmount rồi mount lại hoàn toàn mới. Animation fade-in chạy lại vì component banner thực sự được **tạo mới từ đầu**, không phải đang tồn tại và bị "kích hoạt lại".

Hướng refactor: áp dụng đúng pattern ở mục 9 — chuyển việc render `AccountLayout` lên `pages/_app.js` qua `Component.getLayout`, để mỗi lần chuyển route trong khu vực `/account`, `AccountLayout` (và banner bên trong nó) được giữ nguyên instance, chỉ phần nội dung con thay đổi. Banner sẽ không bị mount lại, animation fade-in chỉ chạy đúng một lần khi người dùng lần đầu vào khu vực `/account`, đúng như hành vi UX mong đợi.

</details>

## 13. Summary

- Pages Router dùng quy ước file-based routing: cấu trúc thư mục `pages/` ánh xạ trực tiếp 1:1 sang cấu trúc URL, không cần file routing config tập trung, code-splitting theo route là tự động.
- Dynamic routes (`[slug].js`), catch-all (`[...slug].js`), và optional catch-all (`[[...slug]].js`) đã tồn tại từ Pages Router, không phải tính năng riêng của App Router.
- `_app.js`, `_document.js`, `_error.js`, `404.js`, `500.js` là các file đặc biệt được dự trữ **theo đúng tên**, mỗi file có vai trò và giới hạn runtime (server-only với `_document.js`) khác nhau.
- Rủi ro kiến trúc gốc: Pages Router **không có khái niệm private folder** — mọi file hợp lệ trong `pages/` đều là route công khai, kể cả component/helper bị đặt nhầm vị trí — đây là nguồn gốc của nhiều bug SEO/runtime thực tế.
- Pattern `Component.getLayout` ở `_app.js` là giải pháp cộng đồng cho việc thiếu nested layout gốc, đồng thời giải quyết một vấn đề kỹ thuật cụ thể: giữ nguyên state của layout (scroll, sidebar, animation) qua các lần chuyển route, nhờ vào cách React reconciliation nhận diện component instance.
- App Router (M3) giải quyết triệt để các hạn chế này bằng Private Folders, quy ước "chỉ `page.tsx` mới là route", Route Groups, và Nested Layouts gốc.
