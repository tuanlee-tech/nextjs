# M2.4 — `getServerSideProps`: SSR Per Request

## 1. Lesson Goal

Sau bài này bạn phải:

- Giải thích được tại sao `getServerSideProps` tốt hơn `getInitialProps` về mặt thiết kế
- Hiểu đầy đủ context object và ba kiểu return value
- Nhận ra khi nào `getServerSideProps` là lựa chọn sai — đây là kỹ năng quan trọng nhất
- Biết các production patterns: auth guard, redirect, header forwarding
- Hiểu trade-off giữa `getServerSideProps` và `getStaticProps`

---

## 2. Evolution Context

Ở bài M2.3, bạn đã thấy vấn đề cốt lõi của `getInitialProps`: **universal execution** — cùng một code chạy trên cả server lẫn client, gây ra class bug khó debug và buộc developer phải viết guard code cho cả hai môi trường.

Next.js 9.3 (March 2020) ra đời với một quyết định thiết kế rõ ràng:

> Tách biệt hoàn toàn hai loại data fetching: **build-time** và **per-request**.

Và tạo ra hai API riêng biệt thay thế `getInitialProps`:

- `getServerSideProps` — chạy trên server, mỗi request
- `getStaticProps` — chạy tại build time (sẽ học ở M2.5)

**Contract của `getServerSideProps` rất đơn giản:**

```
Mỗi khi user request đến page này → Next.js chạy getServerSideProps trên server
→ Lấy data → Render HTML → Gửi về browser
```

Không có "chạy ở đâu?". Không có environment detection. Không có absolute URL hack. **Luôn luôn là server.**

---

## 3. Mental Model

Hãy tưởng tượng `getServerSideProps` như một **middleware** nằm giữa HTTP request và React component:

```
Browser                    Next.js Server
  │                             │
  │  GET /dashboard             │
  │ ──────────────────────────► │
  │                             │
  │                    [getServerSideProps chạy]
  │                    - Đọc cookies từ req
  │                    - Query database
  │                    - Validate session
  │                    - Return { props } hoặc { redirect } hoặc { notFound }
  │                             │
  │                    [React render với props]
  │                    → HTML hoàn chỉnh
  │                             │
  │  200 OK + HTML              │
  │ ◄────────────────────────── │
  │                             │
```

Điểm khác biệt quan trọng so với `getInitialProps`: khi user **navigate bằng `<Link>`** đến page có `getServerSideProps`, Next.js vẫn gọi server. Không phải client tự fetch — browser gửi request đến một Next.js API endpoint đặc biệt, server chạy `getServerSideProps`, trả về JSON, rồi client render.

```
Soft navigation (click <Link>):
  Browser → GET /_next/data/[buildId]/dashboard.json → Server chạy getServerSideProps
  → Server trả về JSON → Client render (không tải lại trang)
```

Đây là lý do `getServerSideProps` **không bao giờ** chạy trên client — nhưng mỗi navigation vẫn tốn một network round-trip đến server.

## 4. Core Concepts

### 4.1 Cú pháp và cấu trúc

```javascript
// pages/dashboard.js

// ✅ Named export, không phải static method
// ✅ Tách biệt hoàn toàn khỏi component
export async function getServerSideProps(context) {
  // ... fetch data
  return {
    props: {
      /* data cho component */
    },
  };
}

function DashboardPage({ user, stats }) {
  return (
    <div>
      <h1>Xin chào, {user.name}</h1>
      <Stats data={stats} />
    </div>
  );
}

export default DashboardPage;
```

**Quan sát kiến trúc:** `getServerSideProps` là **named export** (`export async function`), hoàn toàn tách biệt khỏi component. Không phải static method gắn vào component như `getInitialProps`. Điều này cho phép:

- Tree-shaking: code `getServerSideProps` không bao giờ được bundle vào client
- Tách biệt concerns: data fetching logic và UI logic ở hai nơi rõ ràng
- Dễ test độc lập

### 4.2 Context Object

```javascript
export async function getServerSideProps(context) {
  const {
    // Route params — cho dynamic routes như /products/[id]
    params, // { id: '123' } với route /products/[id]

    // Query string — phần sau dấu ?
    query, // { page: '2', sort: 'price' } với URL /products?page=2&sort=price

    // HTTP Objects — luôn có, không bao giờ undefined
    req, // IncomingMessage — đọc cookies, headers, body
    res, // ServerResponse — set headers, cookies

    // URL info
    resolvedUrl, // URL thực tế, ví dụ: "/products/123?ref=home"

    // Locale (nếu i18n được config)
    locale, // 'vi', 'en', ...
    locales, // ['vi', 'en', ...]
    defaultLocale, // 'en'
  } = context;

  // ...
}
```

**`params` vs `query` — điểm dễ nhầm:**

```javascript
// Route file: pages/blog/[slug].js
// URL: /blog/my-first-post?preview=true

export async function getServerSideProps({ params, query }) {
  console.log(params); // { slug: 'my-first-post' } — từ file system routing
  console.log(query); // { slug: 'my-first-post', preview: 'true' } — toàn bộ query string
  // query chứa cả params! params chỉ chứa dynamic route segments
}
```

Trong thực tế, với dynamic routes luôn dùng `params` để lấy route segment, tránh nhầm lẫn.

**Lưu ý về `params` trong `getServerSideProps`:** Khác với `getInitialProps` (M2.3)
nơi dynamic route parameters được gom chung vào `query` object, `getServerSideProps`
tách riêng `params` chỉ chứa route parameters. Tuy nhiên, `query` object trong
`getServerSideProps` vẫn **chứa cả** dynamic params lẫn query string parameters.
Ví dụ với route `/blog/[slug]` và URL `/blog/hello?ref=home`:

```javascript
// getServerSideProps
params = { slug: 'hello' }; // chỉ route params
query = { slug: 'hello', ref: 'home' }; // cả route params + query string
```

Điều này khác với `getInitialProps` nơi `query` chứa tất cả (và không có `params` riêng).
Khi migrate từ `getInitialProps` sang `getServerSideProps`, cần đổi từ `query.id` sang `params.id`.

### 4.3 Ba kiểu Return Value

Đây là phần quan trọng nhất. `getServerSideProps` có thể return ba dạng khác nhau, mỗi dạng kích hoạt một behavior khác nhau của Next.js:

**Return 1: `{ props }` — Render bình thường**

```javascript
export async function getServerSideProps({ params }) {
  const product = await db.products.findById(params.id);

  return {
    props: {
      product,
      // Props phải serializable (JSON-safe)
      // Không được pass functions, class instances, circular references
    },
  };
}
```

**Return 2: `{ notFound: true }` — Render trang 404**

```javascript
export async function getServerSideProps({ params }) {
  const product = await db.products.findById(params.id);

  if (!product) {
    return { notFound: true };
    // Next.js render pages/404.js
    // HTTP status code: 404
    // Không cần tạo custom error component trong page này
  }

  return { props: { product } };
}
```

**Return 3: `{ redirect }` — Redirect đến URL khác**

```javascript
export async function getServerSideProps({ req }) {
  const session = await getSession(req);

  if (!session) {
    return {
      redirect: {
        destination: '/login?returnTo=/dashboard',
        permanent: false, // true = 308, false = 307
      },
    };
  }

  return { props: { user: session.user } };
}
```

**Phân biệt `permanent: true` vs `permanent: false`:**

```
permanent: false → HTTP 307 Temporary Redirect
  - Browser KHÔNG cache redirect này
  - Dùng cho auth redirect (user có thể login rồi access được)
  - Dùng cho conditional redirect (điều kiện có thể thay đổi)

permanent: true → HTTP 308 Permanent Redirect
  - Browser CÓ THỂ cache và tự redirect mà không hỏi server
  - Dùng khi URL thay đổi vĩnh viễn (refactor routing)
  - NGUY HIỂM: một khi browser đã cache, rất khó undo
```

Trong thực tế, 99% các trường hợp auth redirect nên dùng `permanent: false`.

### 4.4 Đọc Cookies và Headers

```javascript
export async function getServerSideProps({ req, res }) {
  // Đọc cookie — req.cookies là object đã được parse
  const sessionToken = req.cookies['session-token'];
  const preference = req.cookies['theme'] || 'light';

  // Đọc header
  const userAgent = req.headers['user-agent'];
  const acceptLanguage = req.headers['accept-language'];
  const forwardedFor = req.headers['x-forwarded-for']; // IP thực khi đứng sau proxy

  // Set response header (trước khi return)
  res.setHeader('Cache-Control', 'no-store'); // Không cache page này
  res.setHeader('X-Custom-Header', 'value');

  // Set cookie
  res.setHeader('Set-Cookie', [
    `theme=${preference}; Path=/; HttpOnly; Secure; SameSite=Strict`,
  ]);

  return { props: {} };
}
```

**Lưu ý về `req.cookies`:** Next.js tự parse `Cookie` header thành object. Nhưng nếu bạn cần đọc cookie với tên có ký tự đặc biệt, hãy dùng library như `cookie`:

```javascript
import { parse } from 'cookie';

export async function getServerSideProps({ req }) {
  const cookies = parse(req.headers.cookie || '');
  const token = cookies['auth.session']; // Tên có dấu chấm
  // ...
}
```

### 4.5 TypeScript Support

```typescript
import type { GetServerSideProps, InferGetServerSidePropsType } from 'next';

type Product = {
  id: string;
  name: string;
  price: number;
};

type Props = {
  product: Product;
};

// Typed getServerSideProps
export const getServerSideProps: GetServerSideProps<Props> = async ({ params }) => {
  const product = await db.products.findById(params!.id as string);

  if (!product) {
    return { notFound: true };
  }

  return { props: { product } };
};

// InferGetServerSidePropsType giúp type component props từ return type của getServerSideProps
function ProductPage({ product }: InferGetServerSidePropsType<typeof getServerSideProps>) {
  return <div>{product.name}</div>;
}

export default ProductPage;
```

## 5. Legacy Recognition

Khi bạn đọc codebase, đây là cách phân biệt `getServerSideProps` với các patterns khác:

**Signal 1: Named export `getServerSideProps`**

```javascript
// Luôn là named export — không bao giờ là default, không bao giờ là static method
export async function getServerSideProps(context) { ... }
// hoặc
export const getServerSideProps = async (context) => { ... }
```

**Signal 2: Return luôn có wrapper object**

```javascript
// Khác với getInitialProps return plain object
return { props: { ... } }       // Render bình thường
return { notFound: true }       // 404
return { redirect: { ... } }    // Redirect
```

**Signal 3: Có thể dùng Node.js APIs tự do**

```javascript
// Thấy pattern này → code biết mình chạy trên server
import fs from 'fs';
import path from 'path';

export async function getServerSideProps() {
  const content = fs.readFileSync(
    path.join(process.cwd(), 'data.json'),
    'utf8',
  );
  return { props: { content } };
}
```

**Signal 4: Không có environment detection**

```javascript
// Không có typeof window === 'undefined'
// Không có if (req) checks
// Code đơn giản, thẳng thắn
export async function getServerSideProps({ req }) {
  const token = req.cookies.token; // Không cần check req tồn tại không
  // ...
}
```

## 6. Modern Approach

Trong App Router (Next.js 13+), `getServerSideProps` không tồn tại. Thay vào đó, bạn dùng Server Components với `fetch` trực tiếp:

```javascript
// app/dashboard/page.js — App Router equivalent
// Server Component — chạy trên server mỗi request (khi không cache)

async function DashboardPage() {
  // Không cần getServerSideProps wrapper
  // Fetch trực tiếp trong component body
  const user = await getCurrentUser(); // Server-only function
  const stats = await fetchStats(user.id);

  if (!user) {
    redirect('/login'); // Import từ 'next/navigation'
  }

  return (
    <div>
      <h1>Xin chào, {user.name}</h1>
      <Stats data={stats} />
    </div>
  );
}

export default DashboardPage;
```

Conceptually, `getServerSideProps` và Server Component đều là "code chạy trên server trước khi render". Sự khác biệt chính là trong App Router, server logic nằm **trong** component thay vì một function riêng biệt ở ngoài.

## 7. Production Scenario

**Scenario: E-commerce product page với user-specific pricing**

Một platform B2B: giá sản phẩm thay đổi theo từng khách hàng (enterprise contract, discount tier). Không thể static render vì mỗi user thấy giá khác nhau. Nhưng cần SEO tốt.

```javascript
// pages/products/[slug].js

export async function getServerSideProps({ params, req, res }) {
  const { slug } = params;

  // 1. Lấy session của user
  const session = await getIronSession(req, res, sessionOptions);

  // 2. Fetch product info (public data — có thể cache)
  const product = await db.products.findBySlug(slug);

  if (!product || !product.isActive) {
    return { notFound: true };
  }

  // 3. Fetch pricing theo user tier
  // Nếu chưa login → giá public. Nếu đã login → giá theo contract
  const pricing = session.userId
    ? await pricingService.getPricing(product.id, session.userId)
    : await pricingService.getPublicPricing(product.id);

  // 4. Set cache headers có điều kiện
  if (!session.userId) {
    // User chưa login → có thể cache ở CDN ngắn hạn
    res.setHeader('Cache-Control', 's-maxage=60, stale-while-revalidate=300');
  } else {
    // User đã login → không cache (giá cá nhân hóa)
    res.setHeader('Cache-Control', 'private, no-store');
  }

  return {
    props: {
      product: serializeProduct(product), // Helper để đảm bảo JSON-safe
      pricing,
      isAuthenticated: !!session.userId,
    },
  };
}
```

**Điểm đáng chú ý trong scenario này:**

`getServerSideProps` có thể set response headers. Đây là cách để implement **conditional caching**: public user thì CDN cache được, logged-in user thì không. Đây là pattern thực tế mà static generation không thể làm được.

## 8. Debugging Scenario

**Bug: Page load rất chậm, TTFB lên đến 3-4 giây**

```javascript
// pages/home.js — code có vấn đề
export async function getServerSideProps() {
  // Waterfall requests — mỗi request chờ cái trước
  const featuredProducts = await fetch('/api/featured').then((r) => r.json());
  const banners = await fetch('/api/banners').then((r) => r.json());
  const categories = await fetch('/api/categories').then((r) => r.json());
  const testimonials = await fetch('/api/testimonials').then((r) => r.json());

  return {
    props: { featuredProducts, banners, categories, testimonials },
  };
}
```

**Chẩn đoán:**

4 requests chạy tuần tự (sequential/waterfall). Nếu mỗi request mất 800ms → tổng cộng 3.2 giây TTFB chỉ từ data fetching.

Thêm vào đó, đây là **trang home** — không có user-specific data. Không cần `getServerSideProps` chút nào!

**Fix 1: Parallel fetching cho trường hợp vẫn cần SSR**

```javascript
export async function getServerSideProps() {
  // Promise.all — tất cả request chạy song song
  const [featuredProducts, banners, categories, testimonials] =
    await Promise.all([
      fetch('http://localhost:3000/api/featured').then((r) => r.json()),
      fetch('http://localhost:3000/api/banners').then((r) => r.json()),
      fetch('http://localhost:3000/api/categories').then((r) => r.json()),
      fetch('http://localhost:3000/api/testimonials').then((r) => r.json()),
    ]);
  // Thời gian = request chậm nhất, không phải tổng tất cả

  return { props: { featuredProducts, banners, categories, testimonials } };
}
```

**Fix 2: Đổi sang `getStaticProps` + ISR (đúng hơn)**

```javascript
// Trang home không có user-specific data → không cần SSR!
export async function getStaticProps() {
  const [featuredProducts, banners, categories, testimonials] =
    await Promise.all([
      // Gọi thẳng service layer, không qua HTTP
      productService.getFeatured(),
      bannerService.getActive(),
      categoryService.getAll(),
      testimonialService.getPublished(),
    ]);

  return {
    props: { featuredProducts, banners, categories, testimonials },
    revalidate: 60, // Rebuild trang này mỗi 60 giây nếu có request mới
  };
}
// Kết quả: TTFB từ 3-4 giây → < 50ms (file tĩnh từ CDN)
```

Đây là bug cực kỳ phổ biến: dùng `getServerSideProps` khi không cần thiết chỉ vì "nó đơn giản hơn".

## 9. Refactoring Scenario

**Nhiệm vụ: Migrate auth flow từ client-side sang `getServerSideProps`**

**Trước — Pattern phổ biến nhưng có vấn đề:**

```javascript
// pages/dashboard.js — client-side auth check
import { useEffect } from 'react';
import { useRouter } from 'next/router';
import { useAuth } from '../hooks/useAuth';

function DashboardPage() {
  const { user, isLoading } = useAuth();
  const router = useRouter();

  useEffect(() => {
    if (!isLoading && !user) {
      router.replace('/login');
    }
  }, [user, isLoading, router]);

  // Vấn đề: user thấy flash của dashboard trước khi bị redirect
  if (isLoading || !user) {
    return <div>Loading...</div>;
  }

  return <Dashboard user={user} />;
}
```

**Vấn đề với pattern này:**

```
1. HTML ban đầu → skeleton/loading state (browser nhận HTML rỗng)
2. JS tải → useAuth hook chạy → fetch /api/me
3. Nếu không có session → router.replace('/login')

Trong khoảng thời gian giữa bước 1 và 3:
- User thấy loading spinner hoặc layout trống
- URL vẫn là /dashboard dù user không được phép
- Search engines có thể index trang loading state
```

**Sau — Server-side auth redirect:**

```javascript
// pages/dashboard.js — server-side auth check
import { getIronSession } from 'iron-session';
import { sessionOptions } from '../lib/session';

function DashboardPage({ user }) {
  // User LUÔN tồn tại ở đây — đã được verify trên server
  // Không cần loading state, không cần guard clause
  return <Dashboard user={user} />;
}

export async function getServerSideProps({ req, res }) {
  const session = await getIronSession(req, res, sessionOptions);

  // Redirect ngay trên server — user không bao giờ thấy dashboard nếu chưa login
  if (!session.userId) {
    return {
      redirect: {
        destination: `/login?returnTo=/dashboard`,
        permanent: false,
      },
    };
  }

  // Fetch user data — đã biết userId hợp lệ
  const user = await db.users.findById(session.userId);

  return {
    props: {
      user: {
        id: user.id,
        name: user.name,
        email: user.email,
        // KHÔNG pass password hash hay sensitive fields
      },
    },
  };
}

export default DashboardPage;
```

**Kết quả:**

```
Trước:
  → Browser nhận HTML (loading state)
  → JS tải → auth check → redirect
  → 2-3 roundtrips, user thấy flash

Sau:
  → Server check session → nếu không có → 307 redirect ngay
  → Browser nhận redirect, không bao giờ render dashboard HTML
  → 0 flash, secure, SEO-safe
```

## 10. Architecture Notes

### Khi nào KHÔNG dùng `getServerSideProps`

Đây là câu hỏi quan trọng hơn "khi nào dùng". Nhiều developer mặc định dùng `getServerSideProps` vì "nó safe" — đây là sai lầm về architecture.

**Không dùng khi data không thay đổi theo từng request:**

```javascript
// ❌ Lãng phí — blog post không thay đổi từng giây
export async function getServerSideProps({ params }) {
  const post = await db.posts.findBySlug(params.slug);
  return { props: { post } };
}
// Mỗi user vào đọc post → server query DB → render → gửi về
// 1000 users đọc cùng post = 1000 DB queries

// ✅ Đúng — dùng getStaticProps + ISR
export async function getStaticProps({ params }) {
  const post = await db.posts.findBySlug(params.slug);
  return {
    props: { post },
    revalidate: 3600, // Rebuild sau 1 giờ nếu có request mới
  };
}
// 1000 users đọc cùng post = 1 file HTML tĩnh từ CDN
```

**Không dùng khi data có thể fetch ở client:**

```javascript
// ❌ Không cần SSR cho data thay đổi real-time
export async function getServerSideProps() {
  const livePrice = await fetchBitcoinPrice(); // Thay đổi mỗi giây
  return { props: { livePrice } };
}
// Data đã cũ ngay khi HTML đến browser

// ✅ Fetch ở client để có data mới nhất
function PricePage() {
  const { data: livePrice } = useSWR('/api/price', fetcher, {
    refreshInterval: 1000, // Refresh mỗi giây
  });
  return <div>{livePrice}</div>;
}
// getServerSideProps không cần thiết ở đây
```

**Một anti-pattern phổ biến khác: Dùng `getServerSideProps` để fetch data public,
không đổi theo user, rồi set `Cache-Control: public` để "giả lập" static generation.**

```javascript
// ❌ Anti-pattern: dùng getServerSideProps + cache header thay vì getStaticProps
export async function getServerSideProps({ res }) {
  const posts = await db.posts.findAll(); // Data giống nhau cho mọi user
  res.setHeader('Cache-Control', 'public, s-maxage=60');
  return { props: { posts } };
}
```

**Vấn đề:** Dù có cache header, mỗi request vẫn chạy qua Next.js server (hoặc serverless
function) để kiểm tra cache. Với `getStaticProps` + ISR, file HTML được pre-render tại
build time và phục vụ trực tiếp từ CDN/edge — không cần chạy code JavaScript nào.
`getServerSideProps` + cache là "static generation expensive", không phải "static generation
proper". Chi phí compute vẫn cao hơn đáng kể so với true static generation.

**Ma trận quyết định:**

```
Data thay đổi theo từng user?
  ├── Có → Data thay đổi theo từng REQUEST?
  │         ├── Có (session, permissions) → getServerSideProps ✅
  │         └── Không → getStaticProps + ISR hoặc client-side fetch
  └── Không → Data thay đổi thường xuyên?
              ├── Rất thường (< 1 phút) → Client-side fetch (SWR/React Query)
              ├── Thỉnh thoảng (phút - giờ) → getStaticProps + ISR
              └── Hiếm (ngày - tháng) → getStaticProps không revalidate
```

### `getServerSideProps` và Caching

`getServerSideProps` **không có built-in caching**. Mỗi request là một lần thực thi mới. Nhưng bạn có thể implement caching theo nhiều cách:

**Cache response với `Cache-Control` header (CDN caching):**

```javascript
export async function getServerSideProps({ req, res }) {
  // Nội dung public, không có data cá nhân hóa
  // CDN cache 60 giây, stale-while-revalidate cho phép phục vụ stale trong khi rebuild
  res.setHeader(
    'Cache-Control',
    'public, s-maxage=60, stale-while-revalidate=600',
  );

  const data = await fetchPublicData();
  return { props: { data } };
}
```

**In-memory cache ở server (cho data không đổi thường xuyên):**

```javascript
// lib/cache.js
const cache = new Map();

function withCache(key, ttl, fn) {
  const cached = cache.get(key);
  if (cached && cached.expires > Date.now()) {
    return cached.value;
  }
  const value = fn();
  cache.set(key, { value, expires: Date.now() + ttl });
  return value;
}

// pages/products.js
export async function getServerSideProps() {
  const categories = await withCache(
    'categories',
    5 * 60 * 1000, // 5 phút
    () => db.categories.findAll(),
  );

  const products = await db.products.findAll(); // Không cache — thay đổi thường
  return { props: { products, categories } };
}
```

**Lưu ý quan trọng:** In-memory cache chỉ tồn tại trong một process. Với serverless deployment (Vercel), mỗi instance là một process riêng — cache không chia sẻ giữa các instance. Dùng Redis/Memcached cho shared cache trong môi trường distributed.

### `getServerSideProps` và Error Handling

```javascript
export async function getServerSideProps({ params }) {
  try {
    const data = await fetchData(params.id);
    return { props: { data } };
  } catch (error) {
    // Cách 1: Throw để Next.js render error page (500)
    // throw error;

    // Cách 2: Log và return notFound nếu là "not found" error
    if (error.code === 'NOT_FOUND') {
      return { notFound: true };
    }

    // Cách 3: Return error state trong props (graceful degradation)
    console.error('Failed to fetch data:', error);
    return {
      props: {
        data: null,
        error: 'Không thể tải dữ liệu. Vui lòng thử lại.',
      },
    };
  }
}
```

Không có một cách đúng duy nhất — phụ thuộc vào UX requirements. Tuy nhiên, **không bao giờ để error không được catch** trong `getServerSideProps` — Next.js sẽ render 500 page mà không có context nào về nguyên nhân.

## 11. Interview Questions

**Junior Level:**

**Q: `getServerSideProps` khác `getInitialProps` ở điểm gì quan trọng nhất?**

<details>
<summary>Đáp án</summary>

Điểm khác biệt quan trọng nhất là **execution context**:

`getInitialProps` chạy trên server (hard navigation) VÀ trên client (soft navigation khi user click Link). Developer phải tự handle cả hai môi trường bằng cách check `req` có tồn tại không.

`getServerSideProps` **chỉ chạy trên server**, không bao giờ chạy trên client. Khi user click Link đến page có `getServerSideProps`, Next.js tự động gửi request đến server để lấy data, rồi hydrate phía client. Developer không cần xử lý hai execution context.

Điều này dẫn đến code đơn giản hơn, an toàn hơn, và dễ debug hơn.

Ngoài ra, `getServerSideProps` là named export thay vì static method, và return value phải wrap trong `{ props: {...} }` thay vì return plain object trực tiếp.

</details>

---

**Q: Khi nào dùng `return { notFound: true }` thay vì `return { redirect: { destination: '/404' } }`?**

<details>
<summary>Đáp án</summary>

Luôn dùng `return { notFound: true }` khi resource không tồn tại, không bao giờ redirect đến `/404`.

Lý do:

`notFound: true` khiến Next.js render trang 404 với **HTTP status code 404**. Đây là điều search engines và các HTTP client mong đợi khi một resource không tồn tại.

Nếu bạn `redirect` đến `/404`, browser sẽ thấy HTTP 307/308 (redirect) rồi HTTP 200 (trang 404 render). Search engines sẽ không hiểu đây là content không tồn tại — có thể index trang 404 như một trang bình thường, hoặc không xóa trang gốc khỏi index.

```javascript
// ❌ Sai — search engines thấy 307 → 200, không phải 404
return { redirect: { destination: '/404', permanent: false } };

// ✅ Đúng — HTTP 404, Next.js render pages/404.js
return { notFound: true };
```

</details>

**Mid Level:**

**Q: Giải thích tại sao `getServerSideProps` không phải lựa chọn tốt cho trang blog post, và bạn sẽ dùng gì thay thế?**

<details>
<summary>Đáp án</summary>

Blog post là nội dung **không thay đổi theo từng user và không thay đổi liên tục**. Dùng `getServerSideProps` nghĩa là mỗi user request đến bài viết đó đều phải:

1. Server nhận request
2. Query database để lấy bài viết
3. Render HTML
4. Gửi về browser

Với 10,000 lượt đọc một bài phổ biến trong ngày → 10,000 database queries và server rendering operations cho cùng một nội dung không đổi.

**Giải pháp tốt hơn: `getStaticProps` với ISR (Incremental Static Regeneration)**

```javascript
export async function getStaticProps({ params }) {
  const post = await db.posts.findBySlug(params.slug);
  if (!post) return { notFound: true };

  return {
    props: { post },
    revalidate: 3600, // Cho phép rebuild sau 1 giờ nếu có request mới
  };
}

export async function getStaticPaths() {
  const popularPosts = await db.posts.findPopular(100);
  return {
    paths: popularPosts.map((p) => ({ params: { slug: p.slug } })),
    fallback: 'blocking', // Các bài khác generate on-demand
  };
}
```

Kết quả: bài viết được build thành file HTML tĩnh, phục vụ từ CDN. 10,000 lượt đọc = phục vụ từ CDN cache, gần như zero server cost và TTFB dưới 50ms.

ISR đảm bảo khi bài được cập nhật, trang sẽ được rebuild (sau khoảng `revalidate` giây kể từ lần có request mới tiếp theo).

</details>

**Senior Level:**

**Q: Trong một Next.js app lớn, bạn nhận ra nhiều pages đều có `getServerSideProps` để check session. Code bị lặp lại khắp nơi. Bạn sẽ refactor như thế nào mà không break existing behavior?**

<details>
<summary>Đáp án</summary>

Đây là bài toán **higher-order function** hoặc **wrapper pattern** để tái sử dụng auth logic.

**Approach 1: Higher-Order Function (HOF)**

```javascript
// lib/with-session.js
export function withSession(handler) {
  return async function wrappedHandler(context) {
    const session = await getIronSession(
      context.req,
      context.res,
      sessionOptions,
    );

    if (!session.userId) {
      return {
        redirect: {
          destination: `/login?returnTo=${context.resolvedUrl}`,
          permanent: false,
        },
      };
    }

    // Inject session vào context để handler dùng
    context.session = session;
    return handler(context);
  };
}
```

```javascript
// pages/dashboard.js
import { withSession } from '../lib/with-session';

export const getServerSideProps = withSession(async ({ session }) => {
  const user = await db.users.findById(session.userId);
  return { props: { user } };
});
```

```javascript
// pages/settings.js
export const getServerSideProps = withSession(async ({ session, query }) => {
  const settings = await db.settings.findByUserId(session.userId);
  return { props: { settings } };
});
```

**Approach 2: Next.js Middleware (Next.js 12+, tốt hơn cho auth)**

Với Middleware, auth check xảy ra trước khi `getServerSideProps` chạy, ở edge network — nhanh hơn và centralized hơn:

```javascript
// middleware.ts
import { NextResponse } from 'next/server';

export function middleware(request) {
  const session = request.cookies.get('session')?.value;

  if (!session && request.nextUrl.pathname.startsWith('/dashboard')) {
    return NextResponse.redirect(
      new URL(`/login?returnTo=${request.nextUrl.pathname}`, request.url),
    );
  }
}

export const config = {
  matcher: ['/dashboard/:path*', '/settings/:path*', '/admin/:path*'],
};
```

Với middleware, không cần lặp session check trong mỗi `getServerSideProps` nữa. HOF pattern vẫn hữu ích cho các logic phức tạp hơn (role check, feature flag) nhưng basic auth nên để middleware xử lý.

Tôi sẽ recommend dùng Middleware cho auth redirect (Next.js 12+), và HOF pattern cho logic phức tạp hơn cần data từ DB.

</details>

## 12. Exercises

**Bài 1:** Đoạn code sau có bao nhiêu vấn đề? Hãy phân tích và đề xuất fix:

```javascript
// pages/orders/[id].js
export async function getServerSideProps({ params, req }) {
  const orderId = params.id;
  const userId = req.cookies.user_id;

  const order = await fetch(`/api/orders/${orderId}`).then((r) => r.json());

  if (order.userId !== userId) {
    return {
      redirect: {
        destination: '/403',
        permanent: true,
      },
    };
  }

  return {
    props: {
      order,
      loadedAt: new Date(),
    },
  };
}
```

<details>
<summary>Đáp án</summary>

Có 4 vấn đề:

**Vấn đề 1: Relative URL trong server-side fetch**

`fetch('/api/orders/${orderId}')` dùng relative URL. Trên server không có base URL context, request sẽ fail với `TypeError: Only absolute URLs are supported`.

Fix: Gọi thẳng service layer hoặc dùng absolute URL. Tốt nhất là bỏ API route trung gian:

```javascript
const order = await db.orders.findById(orderId);
```

**Vấn đề 2: `permanent: true` cho access denied redirect**

Redirect đến `/403` với `permanent: true` (HTTP 308) sẽ được browser cache. Nếu user sau đó được cấp quyền truy cập order này, browser vẫn tự động redirect sang `/403` mà không hỏi server.

Fix: Dùng `permanent: false` cho conditional redirect:

```javascript
return { redirect: { destination: '/403', permanent: false } };
```

Hoặc tốt hơn, dùng `notFound: true` nếu muốn ẩn sự tồn tại của order khỏi unauthorized user.

**Vấn đề 3: `new Date()` không serialize an toàn thành Date object**

`loadedAt: new Date()` sẽ bị JSON.stringify thành string `"2024-01-01T..."`. Component nhận được `loadedAt` là string, không phải Date. Nếu component gọi `loadedAt.getFullYear()` sẽ crash.

Fix: Return timestamp number hoặc ISO string với ý định rõ ràng:

```javascript
loadedAt: Date.now(); // number — explicit
loadedAt: new Date().toISOString(); // string — explicit
```

**Vấn đề 4: Không có error handling**

Nếu DB query fail, exception không được catch → Next.js render 500 error page không có context. Cần wrap trong try/catch.

**Code sau khi fix:**

```javascript
export async function getServerSideProps({ params, req }) {
  try {
    const orderId = params.id;
    const userId = req.cookies.user_id;

    if (!userId) {
      return { redirect: { destination: '/login', permanent: false } };
    }

    const order = await db.orders.findById(orderId);

    if (!order) {
      return { notFound: true };
    }

    if (order.userId !== userId) {
      return { notFound: true }; // Ẩn sự tồn tại khỏi unauthorized user
    }

    return {
      props: {
        order,
        loadedAt: Date.now(),
      },
    };
  } catch (error) {
    console.error('Failed to load order:', error);
    throw error;
  }
}
```

</details>

**Bài 2:** Thiết kế `getServerSideProps` cho một trang `/feed` của mạng xã hội. Yêu cầu:

- User chưa login → redirect về `/login`
- User đã login → hiển thị 20 bài post mới nhất của những người họ follow
- Có pagination qua query string `?page=1`
- Nếu `page` không hợp lệ (không phải số, âm, quá lớn) → redirect về `?page=1`

<details>
<summary>Đáp án</summary>

```javascript
// pages/feed.js

const PAGE_SIZE = 20;
const MAX_PAGE = 100; // Giới hạn để tránh query quá sâu

export async function getServerSideProps({ req, res, query, resolvedUrl }) {
  // 1. Auth check
  const session = await getSession(req);

  if (!session) {
    return {
      redirect: {
        destination: `/login?returnTo=${encodeURIComponent(resolvedUrl)}`,
        permanent: false,
      },
    };
  }

  // 2. Validate và parse page param
  const rawPage = query.page;
  const parsedPage = parseInt(rawPage, 10);

  const isValidPage =
    rawPage !== undefined && // Có query param
    !isNaN(parsedPage) && // Là số
    parsedPage >= 1 && // Không âm hoặc 0
    parsedPage <= MAX_PAGE; // Không quá lớn

  // Nếu không có page param, default là 1 (không redirect)
  const page = rawPage === undefined ? 1 : isValidPage ? parsedPage : null;

  if (page === null) {
    // Page không hợp lệ → redirect về page 1
    return {
      redirect: {
        destination: '/feed?page=1',
        permanent: false,
      },
    };
  }

  // 3. Fetch data
  try {
    const [posts, totalCount] = await Promise.all([
      db.posts.findFeedForUser(session.userId, {
        skip: (page - 1) * PAGE_SIZE,
        take: PAGE_SIZE,
        orderBy: { createdAt: 'desc' },
      }),
      db.posts.countFeedForUser(session.userId),
    ]);

    const totalPages = Math.min(Math.ceil(totalCount / PAGE_SIZE), MAX_PAGE);

    // 4. Handle page vượt quá tổng số page
    if (page > totalPages && totalPages > 0) {
      return {
        redirect: {
          destination: `/feed?page=${totalPages}`,
          permanent: false,
        },
      };
    }

    // 5. Cache headers — feed cá nhân hóa, không cache
    res.setHeader('Cache-Control', 'private, no-store');

    return {
      props: {
        posts,
        pagination: {
          currentPage: page,
          totalPages,
          totalCount,
          hasNextPage: page < totalPages,
          hasPrevPage: page > 1,
        },
      },
    };
  } catch (error) {
    console.error('Failed to load feed:', error);
    throw error;
  }
}
```

**Lý giải các quyết định thiết kế:**

`resolvedUrl` được encode trước khi đưa vào redirect destination để tránh XSS qua query parameter.

`Promise.all` để fetch posts và count song song — không có lý do để chạy tuần tự.

`MAX_PAGE` để tránh bị abuse với `/feed?page=999999` dẫn đến query với `OFFSET` cực lớn (slow query).

Cache-Control `private, no-store` vì feed là dữ liệu cá nhân hóa — không được phép cache ở CDN.

Redirect khi page vượt quá totalPages để tránh user bị stuck ở trang trống.

</details>

## 13. Summary

`getServerSideProps` giải quyết vấn đề cốt lõi của `getInitialProps` bằng một contract đơn giản: **chỉ chạy trên server, mỗi request, không có ngoại lệ**.

**Ba return type cần nhớ:**

```
{ props: { ... } }                              → Render component với data
{ notFound: true }                              → HTTP 404, render pages/404.js
{ redirect: { destination, permanent } }        → HTTP 307/308 redirect
```

**Quyết định quan trọng nhất khi gặp `getServerSideProps`:**

Không phải "làm sao dùng đúng" mà là **"có thực sự cần SSR không?"**

```
Data thay đổi theo từng user, mỗi request? → getServerSideProps ✅
Data không đổi theo user?                  → getStaticProps + ISR tốt hơn
Data thay đổi real-time?                   → Client-side fetch tốt hơn
```

`getServerSideProps` có chi phí thực: mỗi request tốn server CPU, memory, và network. Dùng đúng chỗ, nó là công cụ mạnh mẽ. Dùng sai chỗ, nó là bottleneck của cả hệ thống.

Bài tiếp theo **M2.5** sẽ cover `getStaticProps` — phần bổ sung còn lại trong bộ đôi data fetching của Next.js 9.3, và là câu trả lời cho bài toán "data không đổi theo từng user".
