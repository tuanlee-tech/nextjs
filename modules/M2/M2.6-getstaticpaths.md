# M2.6 — getStaticPaths: Dynamic SSG

## 1. Lesson Goal

Sau bài này bạn phải trả lời được 3 câu hỏi cốt lõi:

- Tại sao Next.js cần `getStaticPaths`? Thiếu nó thì điều gì xảy ra?
- Ba chiến lược `fallback` khác nhau ở điểm gì — chọn sai thì hậu quả là gì?
- Một site 100,000 sản phẩm nên pre-render bao nhiêu trang lúc build?

## 2. Evolution Context

Ở M2.5 bạn đã hiểu: `getStaticProps` cho phép Next.js fetch data **tại build time** → tạo ra HTML tĩnh → serve cực nhanh.

Nhưng có một vấn đề chưa giải quyết:

```
pages/
  blog/
    [slug].tsx   ← dynamic route
```

Trang này có thể là `/blog/hello-world`, `/blog/nextjs-guide`, `/blog/anything-here`...

**Next.js không có khả năng đọc suy nghĩ bạn.** Nó không biết có bao nhiêu bài blog, tên slug là gì. Không có thông tin này → không thể pre-render.

`getStaticPaths` ra đời để giải quyết đúng vấn đề đó: **"Hãy nói với tôi tất cả các path cần pre-render."**

Timeline liên quan:

| Version | Sự kiện |
|---------|---------|
| Next.js 9.3 | `getStaticPaths` ra mắt cùng `getStaticProps` |
| Next.js 9.5 | Thêm `fallback: 'blocking'` mode |
| Next.js 13 | App Router: `generateStaticParams` thay thế |
| Next.js 14+ | `dynamicParams` config thay thế `fallback` |

## 3. Mental Model

Hãy hình dung nhà in sách:

- **`getStaticProps`** = công thức in một cuốn sách (cần gì, lấy từ đâu)
- **`getStaticPaths`** = danh sách đặt hàng (in những cuốn nào, ISBN nào)

Không có danh sách đặt hàng → nhà in không biết phải in bao nhiêu cuốn → không thể in sẵn.

```
getStaticPaths         getStaticProps
─────────────────      ─────────────────────────────────────
"Tôi cần pre-render   "OK, với path /blog/hello-world,
/blog/hello-world,     tôi sẽ fetch data và tạo HTML này"
/blog/nextjs-guide,
/blog/react-hooks"
```

**Quy tắc tuyệt đối:**

> `getStaticPaths` **chỉ được dùng** cùng với `getStaticProps` trên dynamic route pages.  
> Dùng với `getServerSideProps` → compile error.  
> Dùng trên static route (không có `[param]`) → compile error.

## 4. Core Concepts

### 4.1 Anatomy của getStaticPaths

```typescript
// pages/blog/[slug].tsx

import { GetStaticPaths, GetStaticProps } from 'next'

// Luôn phải export cùng file với getStaticProps
export const getStaticPaths: GetStaticPaths = async () => {
  // Fetch danh sách tất cả slugs từ DB/CMS/API
  const posts = await fetchAllPosts()

  return {
    // paths: array of param objects
    paths: posts.map((post) => ({
      params: { slug: post.slug }, // key phải match tên file [slug].tsx
    })),

    // fallback: hành vi với path KHÔNG có trong paths trên
    fallback: false, // | true | 'blocking'
  }
}

export const getStaticProps: GetStaticProps = async ({ params }) => {
  const post = await fetchPost(params!.slug as string)

  return {
    props: { post },
  }
}
```

**`paths` array format — quan trọng:**

```typescript
// Single dynamic segment: pages/blog/[slug].tsx
paths: [
  { params: { slug: 'hello-world' } },
  { params: { slug: 'nextjs-guide' } },
]

// Nested dynamic: pages/[category]/[id].tsx
paths: [
  { params: { category: 'tech', id: '1' } },
  { params: { category: 'life', id: '2' } },
]

// Catch-all: pages/blog/[...slug].tsx
paths: [
  { params: { slug: ['2024', '01', 'my-post'] } }, // /blog/2024/01/my-post
  { params: { slug: ['about'] } },                  // /blog/about
]
```

**Lỗi phổ biến với `params` type:**

```typescript
// ❌ SAI: id là number
paths: posts.map((p) => ({ params: { id: p.id } }))

// ✅ ĐÚNG: id phải là string
paths: posts.map((p) => ({ params: { id: String(p.id) } }))
// Hoặc
paths: posts.map((p) => ({ params: { id: p.id.toString() } }))
```

> `params` values **phải là string**. Next.js không tự convert. Truyền number → build thành công nhưng trang không match khi navigate.

### 4.2 Ba Fallback Strategies — Cốt lõi của bài này

Đây là phần **quan trọng nhất** và cũng là nơi gây bug nhiều nhất.

`fallback` quyết định: **Next.js làm gì khi user visit một path KHÔNG có trong `paths` array?**

---

#### fallback: false — Danh sách đóng

```typescript
return {
  paths: [
    { params: { slug: 'page-a' } },
    { params: { slug: 'page-b' } },
  ],
  fallback: false,
}
```

**Cơ chế:**

```
User visit /blog/page-a  → ✅ Pre-rendered HTML
User visit /blog/page-b  → ✅ Pre-rendered HTML
User visit /blog/page-c  → 404 (Next.js không biết page này)
User visit /blog/xyz     → 404
```

**Khi nào dùng:**
- Tập hợp paths là **hữu hạn và biết trước** — không bao giờ thêm mới
- Ví dụ: danh sách quốc gia, danh mục cố định, tài liệu static

```typescript
// Ví dụ thực tế: trang country
export const getStaticPaths: GetStaticPaths = async () => {
  // 249 quốc gia — cố định, không đổi
  const countries = await getCountries()

  return {
    paths: countries.map((c) => ({ params: { code: c.code } })),
    fallback: false, // Không quốc gia nào được thêm vào giữa 2 build
  }
}
```

**Trade-off:**
- ✅ Simple, không cần handle loading state
- ✅ Mọi page đều được pre-render hoàn toàn
- ❌ Thêm content mới → phải rebuild toàn bộ site

---

#### fallback: true — Sinh page theo yêu cầu (có skeleton)

```typescript
return {
  paths: topPosts.map((p) => ({ params: { slug: p.slug } })),
  fallback: true, // Các post khác sẽ được generate khi được visit
}
```

**Cơ chế chi tiết:**

```
Build time:
  → Pre-render 100 top posts

User visit /blog/popular-post → ✅ HTML tĩnh, instant

User visit /blog/old-post-2021 (chưa pre-render):
  → Request 1 (first visitor):
      1. Next.js trả về HTML "skeleton/fallback" ngay lập tức (không có data)
      2. Client-side: Next.js trigger getStaticProps trong background
      3. Khi data về → hydrate page với real data
      4. Page được cache vào .next/cache

  → Request 2+ (mọi visitor sau):
      → Serve từ cache, instant như pre-rendered page
```

**Vấn đề quan trọng: phải handle `isFallback`**

```typescript
// pages/blog/[slug].tsx
import { useRouter } from 'next/router'

interface Props {
  post: Post
}

export default function BlogPost({ post }: Props) {
  const router = useRouter()

  // ⚠️ BẮT BUỘC phải check trước khi dùng data
  if (router.isFallback) {
    return (
      <div>
        <Skeleton height="2rem" width="60%" /> {/* title skeleton */}
        <Skeleton count={8} />                 {/* content skeleton */}
      </div>
    )
  }

  // Từ đây trở xuống: post guaranteed không phải undefined
  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </article>
  )
}
```

**Bug cực kỳ phổ biến — không check `isFallback`:**

```typescript
// ❌ CỰC KỲ NGUY HIỂM
export default function BlogPost({ post }: Props) {
  // Lần đầu render (fallback state), post = undefined
  // → post.title → TypeError: Cannot read properties of undefined
  return (
    <article>
      <h1>{post.title}</h1> {/* 💥 CRASH ở đây */}
    </article>
  )
}
```

> `fallback: true` render trang **client-side** trước với props rỗng, sau đó mới có data. Không check `isFallback` → crash không tránh khỏi.

**Khi nào dùng:**
- Blog, news site với hàng nghìn bài, muốn build nhanh
- Content cũ ít được đọc nhưng không muốn 404

---

#### fallback: 'blocking' — Sinh page theo yêu cầu (không có skeleton)

```typescript
return {
  paths: topProducts.slice(0, 1000).map((p) => ({ params: { id: p.id.toString() } })),
  fallback: 'blocking',
}
```

**Cơ chế chi tiết:**

```
User visit /product/99999 (chưa pre-render):
  → Request 1 (first visitor):
      1. Browser gửi request → Next.js server nhận
      2. Next.js gọi getStaticProps({ params: { id: '99999' } })
      3. Browser CHỜ (blocking) cho đến khi HTML được tạo xong
      4. Browser nhận HTML đầy đủ → render
      5. Page được cache

  → Request 2+:
      → Serve từ cache, instant
```

**Không cần check `isFallback`:**

```typescript
// ✅ Clean — không cần xử lý fallback state
export default function ProductPage({ product }: Props) {
  // product luôn có data khi component này render
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.price}</p>
    </div>
  )
}
```

**So sánh fallback: true vs 'blocking':**

| | `fallback: true` | `fallback: 'blocking'` |
|---|---|---|
| User experience (first visit) | Thấy skeleton ngay, data load sau | Chờ hoàn toàn, nhận trang đầy đủ |
| TTFB (Time to First Byte) | Cực nhanh (skeleton HTML nhỏ) | Chậm hơn (phải chờ getStaticProps) |
| Code phức tạp | Cao (phải handle isFallback) | Thấp (không cần xử lý gì thêm) |
| SEO | ⚠️ Kém hơn (crawler thấy skeleton) | ✅ Tốt (crawler thấy HTML đầy đủ) |
| Cảm giác UX | Skeleton rõ ràng | Loading hoàn toàn |

**Khi nào dùng `'blocking'`:**
- Trang quan trọng SEO: product detail, article
- Không muốn phức tạp thêm state management cho fallback
- Dùng khi user chấp nhận "chờ một lần, nhanh mãi mãi sau đó"

---

#### So sánh 3 modes — Decision Tree

```
Tập paths có cố định không?
├── Có → fallback: false
└── Không (thêm content liên tục)
    │
    SEO có quan trọng không?
    ├── Không quan trọng / UX skeleton tốt → fallback: true
    └── Quan trọng / muốn simple code → fallback: 'blocking'
```

### 4.3 getStaticPaths chỉ chạy lúc nào?

```
                    getStaticPaths runs
                    ─────────────────
npm run build  ──→  Lần 1: Enumerate all paths

ISR revalidate ──→  Lần 2: Chỉ chạy lại khi có path mới cần check
                    (không phải mọi ISR trigger đều chạy lại getStaticPaths)

Production     ──→  Không chạy theo mỗi request
```

> `getStaticPaths` **không chạy theo từng request**. Nó chỉ chạy ở build time. Đây là điểm khác biệt cơ bản với `getServerSideProps`.

## 5. Legacy Recognition

Khi đọc codebase cũ, nhận ra ngay pattern này:

```typescript
// Pattern 1: Đơn giản nhất — fetch all, pre-render all
export const getStaticPaths: GetStaticPaths = async () => {
  const { data } = await fetch('/api/posts').then(r => r.json())
  return {
    paths: data.map((p: Post) => ({ params: { id: p.id } })), // ⚠️ id có thể là number
    fallback: false
  }
}

// Pattern 2: getStaticPaths không fetch — hardcode (dấu hiệu enum cố định)
export const getStaticPaths: GetStaticPaths = async () => {
  return {
    paths: ['vi', 'en', 'ja'].map(locale => ({ params: { lang: locale } })),
    fallback: false
  }
}

// Pattern 3: Selective pre-render — pattern production phổ biến
export const getStaticPaths: GetStaticPaths = async () => {
  // Chỉ lấy top 100, phần còn lại để fallback
  const featured = await fetchFeaturedProducts(100)
  return {
    paths: featured.map(p => ({ params: { slug: p.slug } })),
    fallback: 'blocking'
  }
}

// Pattern 4: Empty paths — dấu hiệu nguy hiểm
export const getStaticPaths: GetStaticPaths = async () => {
  return {
    paths: [],         // ← không pre-render gì hết
    fallback: true     // ← mọi trang đều generate on-demand
  }
}
// → Vấn đề: build nhanh nhưng first visitor của mọi trang đều thấy skeleton
```

**Dấu hiệu codebase dùng `fallback: true` không đúng cách:**

```typescript
// ❌ Thiếu isFallback check — bug tiềm ẩn
export default function Post({ post }: { post: Post }) {
  return <h1>{post.title}</h1> // crash nếu chưa load xong
}

// Nhìn vào log Sentry cũ thường thấy:
// TypeError: Cannot read properties of undefined (reading 'title')
// Chỉ xảy ra ở first visit của uncached pages
```

## 6. Modern Approach

### generateStaticParams (App Router)

`getStaticPaths` → `generateStaticParams` trong App Router:

```typescript
// app/blog/[slug]/page.tsx

// Tương đương getStaticPaths
export async function generateStaticParams() {
  const posts = await fetchAllPosts()

  return posts.map((post) => ({
    slug: post.slug,
    // Không cần wrap trong { params: { ... } }
    // Trả về trực tiếp object params
  }))
}

// Tương đương fallback behavior → dùng route segment config
export const dynamicParams = true  // fallback: 'blocking' (default)
// export const dynamicParams = false // fallback: false → 404 nếu không có trong params
```

**Không có tương đương trực tiếp cho `fallback: true`** (skeleton mode) trong App Router. Thay vào đó dùng Suspense:

```typescript
// app/blog/[slug]/page.tsx
import { Suspense } from 'react'

// generateStaticParams cho top posts
export async function generateStaticParams() {
  const topPosts = await getTopPosts(100)
  return topPosts.map(p => ({ slug: p.slug }))
}

export const dynamicParams = true // sinh thêm on-demand với SSR

export default async function BlogPage({ params }: { params: { slug: string } }) {
  // Với App Router, component này là Server Component
  // Không cần check isFallback — Suspense xử lý loading state
  const post = await fetchPost(params.slug)

  if (!post) notFound()

  return <article><h1>{post.title}</h1></article>
}
```

**So sánh tổng quan:**

| Pages Router | App Router |
|---|---|
| `getStaticPaths` | `generateStaticParams` |
| `fallback: false` | `dynamicParams = false` |
| `fallback: 'blocking'` | `dynamicParams = true` (default) |
| `fallback: true` + `isFallback` | Suspense boundaries |
| `params` wrapped object | Flat object |

## 7. Production Scenario

### Bài toán: E-commerce với 500,000 sản phẩm

```typescript
// pages/products/[id].tsx

export const getStaticPaths: GetStaticPaths = async () => {
  // Không thể pre-render 500,000 sản phẩm → build mất 10+ tiếng
  // Chiến lược: chỉ pre-render top 2,000 theo doanh thu

  const topProducts = await db.product.findMany({
    orderBy: { totalRevenue: 'desc' },
    take: 2000,
    select: { id: true },
  })

  return {
    paths: topProducts.map((p) => ({
      params: { id: p.id.toString() },
    })),
    fallback: 'blocking',
    // 498,000 sản phẩm còn lại sẽ được generate on-demand
    // Sau lần đầu được visit → cached → nhanh như pre-rendered
  }
}

export const getStaticProps: GetStaticProps = async ({ params }) => {
  const product = await db.product.findUnique({
    where: { id: parseInt(params!.id as string) },
  })

  // Sản phẩm không tồn tại → 404
  if (!product) {
    return { notFound: true }
  }

  return {
    props: { product },
    // ISR: revalidate mỗi 30 phút (giá, tồn kho có thể thay đổi)
    revalidate: 60 * 30,
  }
}
```

**Tại sao pattern này hoạt động tốt:**

```
2,000 products được pre-render:
  → Covers ~80% traffic (law of 80/20)
  → Build time: ~10 minutes (vs hours if all 500K)

498,000 products còn lại:
  → First visitor: chờ 200-500ms (getStaticProps runs)
  → Sau đó: cached, serve từ disk
  → Chấp nhận được vì long-tail products ít traffic

Kết quả:
  → Build nhanh
  → 80% users: instant response
  → 20% users: chờ một lần, nhanh mãi
```

### Thêm monitoring cho fallback generation

```typescript
export const getStaticProps: GetStaticProps = async ({ params }) => {
  const start = Date.now()
  const product = await db.product.findUnique({ where: { id: parseInt(params!.id as string) } })

  if (!product) return { notFound: true }

  // Log khi fallback generation xảy ra (giúp theo dõi pages nào hay được visit)
  const duration = Date.now() - start
  if (duration > 1000) {
    // Slow query → cảnh báo
    logger.warn('Slow static generation', { id: params!.id, duration })
  }

  return {
    props: { product },
    revalidate: 1800,
  }
}
```

## 8. Debugging Scenario

### Bug 1: isFallback crash — phổ biến nhất

**Symptom:** Production error chỉ xảy ra ở first visit của uncached pages, không tái hiện khi test local (vì local đã build và cache toàn bộ).

```typescript
// Bug
export default function PostPage({ post }: { post: Post | undefined }) {
  return (
    <div>
      <h1>{post.title}</h1> {/* TypeError khi post = undefined */}
    </div>
  )
}

// Sentry log:
// TypeError: Cannot read properties of undefined (reading 'title')
// at PostPage (pages/blog/[slug].tsx:8)
// Chỉ xảy ra với users là first visitor
```

**Fix:**

```typescript
import { useRouter } from 'next/router'

export default function PostPage({ post }: { post: Post }) {
  const router = useRouter()

  // Bắt buộc guard ở đây
  if (router.isFallback) {
    return <PostSkeleton />
  }

  return (
    <div>
      <h1>{post.title}</h1>
    </div>
  )
}
```

### Bug 2: Build thành công nhưng dynamic pages trả 404

```typescript
// getStaticPaths trả về empty paths với fallback: false
export const getStaticPaths: GetStaticPaths = async () => {
  try {
    const posts = await fetchPosts()
    return {
      paths: posts.map(p => ({ params: { slug: p.slug } })),
      fallback: false,
    }
  } catch (error) {
    // ❌ Khi API lỗi lúc build → trả về empty paths thay vì throw
    console.error('Failed to fetch posts')
    return { paths: [], fallback: false } // ← Mọi trang đều 404!
  }
}
```

**Fix:** Không bao giờ swallow error trong getStaticPaths:

```typescript
export const getStaticPaths: GetStaticPaths = async () => {
  // ✅ Để error throw ra → build fail → developer biết ngay
  const posts = await fetchPosts() // nếu fail → build fail
  return {
    paths: posts.map(p => ({ params: { slug: p.slug } })),
    fallback: 'blocking', // dùng blocking thay vì false để an toàn hơn
  }
}
```

### Bug 3: Params type mismatch

```typescript
// File: pages/products/[id].tsx
// id trong DB là integer: 1, 2, 3, ...

// ❌ BUG
export const getStaticPaths: GetStaticPaths = async () => {
  const products = await getProducts()
  return {
    paths: products.map(p => ({
      params: { id: p.id } // id = number (1, 2, 3...)
    })),
    fallback: false,
  }
}

// Build warning: "A required parameter (id) was not provided as a string"
// Pages sẽ không match đúng URL

// ✅ FIX
paths: products.map(p => ({
  params: { id: String(p.id) } // Luôn convert sang string
})),
```

## 9. Refactoring Scenario

### Migrate getStaticPaths → generateStaticParams

**Trước (Pages Router):**

```typescript
// pages/blog/[slug].tsx
import { GetStaticPaths, GetStaticProps, InferGetStaticPropsType } from 'next'

type Post = { slug: string; title: string; content: string }

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetchTopPosts(200)

  return {
    paths: posts.map((p) => ({ params: { slug: p.slug } })),
    fallback: 'blocking',
  }
}

export const getStaticProps: GetStaticProps<{ post: Post }> = async ({ params }) => {
  const post = await fetchPost(params!.slug as string)

  if (!post) return { notFound: true }

  return {
    props: { post },
    revalidate: 3600,
  }
}

export default function BlogPost({ post }: InferGetStaticPropsType<typeof getStaticProps>) {
  return <article>{post.title}</article>
}
```

**Sau (App Router):**

```typescript
// app/blog/[slug]/page.tsx
import { notFound } from 'next/navigation'

// Tương đương getStaticPaths
export async function generateStaticParams() {
  const posts = await fetchTopPosts(200)
  return posts.map((p) => ({ slug: p.slug }))
  // Trả về flat object, không wrap trong { params: {} }
}

// Tương đương fallback: 'blocking'
export const dynamicParams = true

// Tương đương revalidate: 3600
export const revalidate = 3600

// Server Component thay thế cho default export + getStaticProps
export default async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetchPost(params.slug)

  if (!post) notFound()

  return <article>{post.title}</article>
}
```

**Migration checklist:**

```
□ getStaticPaths → generateStaticParams
  □ Bỏ wrapper { params: { key: value } } → flat { key: value }
  □ Bỏ fallback property

□ fallback: false  → dynamicParams = false (thêm vào page.tsx)
□ fallback: 'blocking' → dynamicParams = true (hoặc không cần thêm, đây là default)
□ fallback: true → xóa isFallback logic → dùng Suspense nếu cần skeleton

□ getStaticProps → async Server Component
  □ Fetch data trực tiếp trong component
  □ notFound() thay cho return { notFound: true }

□ revalidate: N → export const revalidate = N
```

## 10. Architecture Notes

### Khi nào dùng getStaticPaths vs không dùng

```
Dynamic route có cần SSG không?
├── Không cần SSG (data thay đổi theo từng user, cần auth)
│   → Không dùng getStaticPaths + getStaticProps
│   → Dùng getServerSideProps hoặc client-side fetch
│
└── Cần SSG (content public, cần SEO, performance)
    │
    ├── Paths hữu hạn và biết trước (< 10,000)?
    │   └── Có → Có thể pre-render tất cả
    │       ├── Số lượng nhỏ → getStaticPaths tất cả + fallback: false
    │       └── Số lượng lớn → top N + fallback: 'blocking'
    │
    └── Paths vô hạn hoặc không biết trước?
        → Top N by relevance + fallback: 'blocking'
```

### getStaticPaths và ISR kết hợp

```typescript
// Combination mạnh nhất trong Pages Router
export const getStaticPaths: GetStaticPaths = async () => ({
  paths: await getTopPaths(),
  fallback: 'blocking', // sinh on-demand cho paths khác
})

export const getStaticProps: GetStaticProps = async ({ params }) => ({
  props: { data: await fetchData(params!.id as string) },
  revalidate: 3600, // ISR: tái tạo sau 1 giờ nếu có request mới
})
```

```
Result:
  Pre-render top paths tại build time
    → Serve instantly
  New paths: generate on first request, cache
    → Serve from cache sau đó
  Stale pages: revalidate sau 1 giờ nếu có request
    → Always reasonably fresh
```

### Build time estimation

```typescript
// Rule of thumb:
// Mỗi page trong getStaticPaths tốn ~100-500ms để generate
// (tùy thuộc DB query, network latency, complexity)

// 100 pages   → ~10-50 seconds
// 1,000 pages → ~1-8 minutes
// 10,000 pages → ~17-83 minutes (không chấp nhận được)

// Best practice: không quá 1,000-5,000 pages trong getStaticPaths
// Phần còn lại để fallback: 'blocking' hoặc ISR xử lý
```

## 11. Interview Questions

**Junior level:**

**Q1: `getStaticPaths` dùng để làm gì? Tại sao không thể chỉ dùng `getStaticProps` cho dynamic routes?**

<details>
<summary>Đáp án</summary>

`getStaticProps` chỉ biết **cách** fetch data cho một page, nhưng với dynamic routes như `/blog/[slug]`, Next.js không biết **những slug nào** tồn tại để pre-render.

`getStaticPaths` giải quyết vấn đề này bằng cách cung cấp danh sách tất cả các paths cần pre-render tại build time. Không có `getStaticPaths` → Next.js không thể biết cần tạo ra những trang nào → không thể SSG cho dynamic routes.

Hai API này phải đi cùng nhau: `getStaticPaths` cung cấp **danh sách paths**, `getStaticProps` cung cấp **data cho từng path**.

</details>

**Mid level:**

**Q2: Giải thích sự khác biệt giữa `fallback: true` và `fallback: 'blocking'`. Khi nào chọn cái nào?**

<details>
<summary>Đáp án</summary>

**`fallback: true`:**
- Khi user visit path chưa được pre-render, Next.js trả về HTML "skeleton" ngay lập tức (không có data)
- Client-side: Next.js chạy `getStaticProps` trong background, khi xong thì hydrate page với real data
- Developer **phải** check `router.isFallback` để render skeleton UI, nếu không component crash vì props là `undefined`
- TTFB cực nhanh (skeleton nhỏ), nhưng UX không đồng nhất (user thấy skeleton flash)
- SEO kém hơn vì crawler có thể thấy skeleton thay vì real content

**`fallback: 'blocking'`:**
- Browser gửi request → Next.js server chạy `getStaticProps` → trả về HTML đầy đủ
- Browser phải **chờ** cho đến khi generation xong (giống SSR cho lần đầu)
- Không cần check `isFallback`, code đơn giản hơn
- SEO tốt hơn (crawler nhận full HTML)
- TTFB chậm hơn cho first visitor của uncached pages

**Chọn `'blocking'` khi:** trang quan trọng SEO, muốn code đơn giản, user chấp nhận loading  
**Chọn `true` khi:** muốn UI responsive ngay, có skeleton design sẵn, SEO không quan trọng bằng perceived performance

</details>

**Senior level:**

**Q3: Với một e-commerce site có 500,000 sản phẩm, bạn thiết kế getStaticPaths như thế nào? Trade-off là gì?**

<details>
<summary>Đáp án</summary>

Không pre-render toàn bộ 500,000 sản phẩm vì build time sẽ mất hàng giờ và không khả thi.

**Chiến lược selective pre-render:**

```typescript
export const getStaticPaths: GetStaticPaths = async () => {
  // Chỉ pre-render top ~1,000-2,000 sản phẩm theo doanh thu/lượt xem
  // Đây là ~80% traffic thực tế (Pareto principle)
  const topProducts = await db.product.findMany({
    orderBy: { pageViews: 'desc' },
    take: 2000,
    select: { id: true },
  })

  return {
    paths: topProducts.map(p => ({ params: { id: String(p.id) } })),
    fallback: 'blocking', // 498,000 sản phẩm còn lại: generate on-demand
  }
}

export const getStaticProps: GetStaticProps = async ({ params }) => {
  const product = await getProduct(params!.id as string)
  if (!product) return { notFound: true }
  return { props: { product }, revalidate: 1800 } // ISR 30 phút
}
```

**Trade-offs:**
- Build time: ~3-5 phút thay vì 8+ tiếng → ✅ Deploy thường xuyên được
- 80% users nhận pre-rendered page → instant response → ✅
- 20% users (long-tail): chờ ~200-500ms lần đầu → cached sau đó → chấp nhận được
- ISR `revalidate: 1800` → giá, tồn kho được cập nhật mỗi 30 phút → đủ fresh

**Risk cần xử lý:**
- Sản phẩm bị xóa → `getStaticProps` trả `notFound: true` → cache nhận 404
- Thêm sản phẩm mới hot → cần mechanism để invalidate/revalidate cache (on-demand revalidation)

</details>

**Staff level:**

**Q4: `getStaticPaths` chạy lúc nào trong vòng đời của application? Nó có chạy lại khi ISR trigger không?**

<details>
<summary>Đáp án</summary>

`getStaticPaths` chạy **duy nhất tại build time** (khi chạy `next build`). Nó không chạy lại theo từng request.

**Với ISR:** Khi một page ISR được revalidate (do `revalidate` period expired + có new request), Next.js chỉ chạy lại `getStaticProps` cho page đó — **không** chạy lại `getStaticPaths`.

Điều này có nghĩa:
- Paths được enumerate một lần duy nhất lúc build
- New paths (không có trong `paths`) chỉ được xử lý theo `fallback` strategy
- Nếu `fallback: false`, new content tạo sau build sẽ mãi là 404 cho đến khi rebuild

**Implication cho architecture:**
- Với `fallback: 'blocking'` + ISR: new paths được generate on-demand và revalidate theo schedule → không cần rebuild
- Với `fallback: false`: phải trigger rebuild khi có content mới → thích hợp cho CMS với webhook trigger builds
- Webhook-triggered rebuild (e.g., Vercel deploy hook từ CMS) + `fallback: false` là pattern phổ biến cho editorial content

</details>

**Bài 1 — Cơ bản:** Bạn có một trang `/categories/[slug]` với 5 category cố định: `tech`, `life`, `travel`, `food`, `finance`. Viết `getStaticPaths` phù hợp. Chọn fallback mode nào và tại sao?

<details>
<summary>Đáp án</summary>

```typescript
// pages/categories/[slug].tsx
import { GetStaticPaths, GetStaticProps } from 'next'

const CATEGORIES = ['tech', 'life', 'travel', 'food', 'finance']

export const getStaticPaths: GetStaticPaths = async () => {
  return {
    paths: CATEGORIES.map((slug) => ({
      params: { slug },
    })),
    fallback: false,
    // Lý do chọn false:
    // - Chỉ có 5 category, tập hợp hoàn toàn cố định
    // - Không bao giờ thêm category mới giữa 2 deploy
    // - User visit /categories/invalid → 404 là đúng behavior
    // - Không cần handle fallback state → code đơn giản nhất
  }
}

export const getStaticProps: GetStaticProps = async ({ params }) => {
  const slug = params!.slug as string
  const categoryData = await fetchCategoryData(slug)

  if (!categoryData) return { notFound: true }

  return {
    props: { category: categoryData },
    revalidate: 86400, // revalidate 1 ngày vì category data ít thay đổi
  }
}
```

Lý do không chọn `'blocking'` hay `true`: đây là tập hợp đóng, hoàn toàn biết trước, không có lý do gì để cho phép dynamic generation. `fallback: false` đơn giản và đúng nhất.

</details>

**Bài 2 — Trung bình:** Một news site có 50,000 bài báo trong DB. Mỗi bài có `id`, `slug`, `publishedAt`, `pageViews`. Thiết kế `getStaticPaths` sao cho build time dưới 5 phút, trong khi vẫn đảm bảo bài báo mới nhất và hot nhất được serve nhanh nhất.

<details>
<summary>Đáp án</summary>

Chiến lược: kết hợp pre-render bài **hot** (nhiều views) + bài **mới nhất** (24h gần đây), phần còn lại dùng `fallback: 'blocking'`.

```typescript
export const getStaticPaths: GetStaticPaths = async () => {
  // Lấy 500 bài hot nhất (phần lớn traffic)
  const hotArticles = await db.article.findMany({
    orderBy: { pageViews: 'desc' },
    take: 500,
    select: { slug: true },
  })

  // Lấy 200 bài mới nhất trong 24h (trending content)
  const recentArticles = await db.article.findMany({
    where: {
      publishedAt: {
        gte: new Date(Date.now() - 24 * 60 * 60 * 1000),
      },
    },
    orderBy: { publishedAt: 'desc' },
    take: 200,
    select: { slug: true },
  })

  // Merge và deduplicate
  const allSlugs = [
    ...hotArticles.map((a) => a.slug),
    ...recentArticles.map((a) => a.slug),
  ]
  const uniqueSlugs = [...new Set(allSlugs)] // tối đa ~700 slugs

  return {
    paths: uniqueSlugs.map((slug) => ({ params: { slug } })),
    fallback: 'blocking',
    // fallback: 'blocking' vì:
    // - Bài mới publish sau build cần được serve ngay
    // - SEO quan trọng với news (Google crawler cần full content)
    // - Không muốn phức tạp isFallback cho newsroom
  }
}

export const getStaticProps: GetStaticProps = async ({ params }) => {
  const article = await db.article.findUnique({
    where: { slug: params!.slug as string },
  })

  if (!article) return { notFound: true }

  return {
    props: { article },
    revalidate: 300, // ISR 5 phút — news cập nhật liên tục
  }
}
```

Ước tính build time: ~700 pages × 200ms/page = ~140 giây (~2-3 phút), đáp ứng yêu cầu dưới 5 phút.

</details>

## 13. Summary

`getStaticPaths` giải quyết một vấn đề cụ thể: **Next.js cần biết trước các dynamic paths để pre-render tại build time**.

Ba fallback strategies phục vụ ba use case khác nhau:

- `false` — Danh sách đóng, paths hữu hạn, đơn giản nhất
- `true` — Sinh page on-demand với skeleton UI, cần check `isFallback`, TTFB thấp
- `'blocking'` — Sinh page on-demand như SSR, không cần skeleton, SEO tốt

**Production pattern quan trọng nhất:** Không pre-render toàn bộ → chọn lọc top N paths quan trọng + `fallback: 'blocking'` cho phần còn lại + ISR để keep data fresh.

**Migration sang App Router:**
- `getStaticPaths` → `generateStaticParams` (flat object, không wrap `{ params }`)
- `fallback: false` → `dynamicParams = false`
- `fallback: 'blocking'` → `dynamicParams = true` (default)
- `fallback: true` → Suspense boundaries

Bài tiếp theo **M2.7 — API Routes** sẽ đi vào cách xây dựng backend endpoints ngay trong Next.js, không cần server riêng.
