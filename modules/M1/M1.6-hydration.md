# M1.6 — Hydration: Đồng bộ trạng thái Client-Server

## 0. Tóm Tắt

<details>
<summary>Thời gian đọc: 5 phút</summary>

Hydration = React **tái sử dụng DOM server-render có sẵn**, build Virtual DOM trong memory để so khớp (diff Virtual DOM vs DOM thật), rồi **gắn event listeners** vào đúng node — không tạo lại DOM từ đầu. Hydration Mismatch xảy ra khi server render và client render ra output khác nhau. React 18 mang Selective Hydration (chia theo Suspense boundary, ưu tiên phần user tương tác). App Router thay đổi đơn vị hydrate: **chỉ Client Components (`'use client'`) mới hydrate**, Server Components không bao giờ tham gia — giảm Hydration Cost mang tính kiến trúc.

**1. Lesson Goal (Học xong phải nhớ gì?)**

- Hydration thực sự làm gì ở cấp độ DOM — không phải chỉ "gắn JS vào HTML"

- Tại sao chuyển từ Pages Router sang App Router giảm "Hydration Cost" triệt để — nhờ **thay đổi đơn vị hydrate**, không phải tối ưu code

**2. Evolution Context (Bài này xuất hiện để giải quyết vấn đề gì?)**

**Trước đó**

- Server-rendered thuần (PHP, Rails): HTML hiển thị tức thì, nhưng tương tác phải viết lại logic client bằng tay → trùng lặp, hai nguồn sự thật
- CSR thuần (React SPA): `index.html` rỗng, JS tải xong mới render → blank screen, SEO kém, FCP tệ
- React 15: `ReactDOM.render()` trên server-rendered HTML — không có hydration chuyên dụng, có thể double render
- React 16–17: `ReactDOM.hydrate()` — hydrate toàn bộ tree, đồng bộ (blocking), không dừng được
- Pages Router (Next 9–12): toàn bộ page là một đơn vị hydrate duy nhất, không có khái niệm "miễn hydrate" cho phần tĩnh

↓

**Vấn đề**

- Hydration blocking: toàn bộ cây hydrate đồng bộ → Main Thread block, input "treo" khi cây lớn
- Hydration Mismatch: server render và client render output khác nhau → React phải discard & re-render
- Bundle JS lớn: phần tĩnh (text, hình ảnh) vẫn phải gửi JS và hydrate — không tối ưu được
- Workaround phổ biến: `typeof window` rải khắp codebase, `next/dynamic` `{ ssr: false }` tràn lan
- TTI chậm: FCP nhanh (0.8s) nhưng TTI chậm (4.2s) → "vùng chết" người dùng nhìn thấy nhưng không tương tác được

↓

**Giải pháp**

- React 18: `ReactDOM.hydrateRoot()` → **Selective Hydration**, concurrent features, có thể ưu tiên và tạm dừng
- Suspense boundaries → chia cây thành "đảo" hydrate độc lập
- App Router: **Server Components không hydrate** — chỉ Client Components (`'use client'`) tham gia
- RSC Payload: Server Components serialize thành binary format nhẹ, tham gia reconciliation qua payload thay vì hydration nặng
- Nguyên tắc: đẩy boundary `'use client'` sát lá nhất, hoãn non-determinism bằng `useEffect`

**3. Mental Model (Ý tưởng cốt lõi)**

> **Hydration = Cấy hệ thần kinh vào bức tượng thạch cao** — HTML server-render là bức tượng đã tạo hình sẵn (đẹp, nhìn thấy ngay, nhưng không cử động). React chạy qua từng "khớp xương" (DOM node) đã có, gắn "dây thần kinh" (event listeners, state, refs) vào đúng vị trí — không đập vỡ, không tạo lại.

Hệ quả:

- Hình dáng JS tính ra (Virtual DOM lần đầu) **khác** hình dáng thạch cao (DOM server) → không biết khớp xương nào ứng với cái nào → **đập và đổ lại từ đầu** (Hydration Mismatch)
- Cấy toàn bộ bức tượng cùng lúc (Pages Router) tốn hơn cấy từng phần theo nhu cầu (App Router, Selective Hydration)

**4. Core Concepts (20% kiến thức quan trọng nhất)**

**Concept 1: Luồng đầy đủ của Hydration**

1. Server render → HTML string (có marker ẩn cho Suspense boundaries)
2. Browser parse & paint → user thấy nội dung (FCP) nhưng **chưa tương tác được** → "Uncanny Valley" của SSR
3. JS bundle tải & execute
4. `hydrateRoot()` chạy: build Virtual DOM, **diff với DOM thật** (không phải diff hai Virtual DOM như Reconciliation thông thường)
5. Khớp → gắn event listener, fiber instance, state → app interactive (TTI)
6. Không khớp → Hydration Mismatch warning, React patch hoặc discard & re-render

**Concept 2: Hydration Mismatch — Nguồn gốc thực sự**

Bản chất: server render và client render (lần đầu) output khác nhau → React không khớp được DOM.

Nguồn phổ biến:

- **Non-deterministic trong render**: `new Date()`, `Math.random()`, `crypto.randomUUID()` — mỗi lần gọi ra giá trị khác
- **Phụ thuộc môi trường**: `typeof window` trong JSX render → server không có `window`, client có → nhánh khác nhau
- **HTML không hợp lệ**: `<p>` lồng `<div>`, `<table>` thiếu `<tbody>` → browser tự re-parse, cấu trúc DOM khác React mong đợi
- **Browser extension**: Grammarly, Dark Reader, ad-blockers chèn attribute/DOM trước khi hydrate
- **Locale/timezone**: server UTC vs client local → `toLocaleDateString()` khác nhau
- **Non-deterministic ID**: `Math.random()` hoặc `id++` trong render → server và client khác nhau

**Concept 3: Selective Hydration (React 18)**

Trước React 18: `ReactDOM.hydrate()` — đồng bộ, blocking, một khi bắt đầu không dừng được, không ưu tiên được → Main Thread block khi cây lớn.

React 18 + Suspense:

- Chia cây thành các "đảo" (island) độc lập theo ranh giới `<Suspense>`
- **Ưu tiên hydrate phần user đang tương tác** — vừa click vào phần nào thì hydrate phần đó trước
- **Không block Main Thread liên tục** — hydration chia thành work units, nhường thời gian cho browser xử lý input (ứng dụng trực tiếp của Concurrent Rendering / Time Slicing từ M1.5)

**Concept 4: App Router — Thay đổi đơn vị hydrate**

Pages Router: toàn bộ page là một cây hydrate duy nhất — dù 90% nội dung tĩnh, JS vẫn tính trên 100% cây.

App Router:

- **Server Components không bao giờ hydrate** — render HTML ở server, không gửi JS, không có fiber cần "sống lại"
- Chỉ cây con từ `'use client'` mới tham gia hydrate
- RSC Payload: Server Components serialize thành binary format nhẹ → tham gia reconciliation qua payload (nhẹ) thay vì hydration (nặng)
- Kết quả: bundle JS giảm 60–80%, TTI giảm về dưới 1s — vì đơn vị hydrate co lại từ "cả trang" xuống "widget nhỏ"

**Concept 5: RSC Payload**

Server Components không "biến mất" sau server render — chúng được serialize thành RSC Payload (compact binary format):

- Reconcile cây component giữa server và client
- Xác định vị trí Client Components (placeholders)
- Truyền props từ Server Component → Client Component

Đây là lý do RSC giảm Hydration Cost — tham gia reconciliation qua payload nhẹ thay vì hydration nặng.

👉 Công thức / sơ đồ

```
Server render → HTML string
    ↓
Browser parse & paint → FCP (nhìn thấy nhưng chưa tương tác)
    ↓
JS bundle tải xong → hydrateRoot() chạy
    ↓
Build Virtual DOM → diff với DOM thật
    ↓
Khớp → gắn event listeners, state, refs → TTI (tương tác được)
    ↓
Không khớp → Hydration Mismatch → discard & re-render
```

```
Pages Router:
Server Component + Client Component
    ↓
Toàn bộ cây hydrate chung
    ↓
Hydration Cost tuyến tính với kích thước trang

App Router:
Server Component (không hydrate) → HTML + RSC Payload
    ↓
Client Component ('use client') → hydrate độc lập
    ↓
Hydration Cost tuyến tính với số lượng Client Components
```

**5. Legacy Recognition (Nhìn code biết gì?)**

Dấu hiệu nhận biết code Pages Router (Next 9–12) có rủi ro hydration cao

✅ **Nhận ra ngay:**

- `getServerSideProps`/`getStaticProps` fetch data, nhưng toàn bộ component cây vẫn bị gửi JS và hydrate — không có cách "miễn hydrate" phần tĩnh
- `typeof window` rải khắp codebase để né `window is not defined` khi SSR — triệu chứng, không phải giải pháp
- `next/dynamic` với `{ ssr: false }` tràn lan như "miếng dán" cho component dùng `window`
- `ReactDOM.render()` hoặc `ReactDOM.hydrate()` thay vì `hydrateRoot()`
- Không có khái niệm Server Component / Client Component — toàn bộ là Client Component ngầm định

❌ **Không nên viết:**

- `typeof window` trong JSX render trực tiếp → gây Hydration Mismatch
- `Math.random()` / `crypto.randomUUID()` trong render → non-deterministic ID
- `new Date()` trong render → server và client khác thời điểm
- `suppressHydrationWarning` cho mismatch cấu trúc (thêm/xóa/đổi element) — chỉ dùng cho khác biệt chấp nhận được (timestamp vài ms)
- `'use client'` ở component cha — kéo toàn bộ cây con vào hydrate

**7. Production Scenario (Thực tế)**

**Context:** E-commerce product page (Pages Router, Next 11) — Lighthouse TTI 4.2s, FCP 0.8s. Khoảng cách 3.4s là "vùng chết": user nhìn thấy trang nhưng bấm không phản hồi.

**Problem:** Toàn bộ trang — mô tả sản phẩm (text tĩnh dài), reviews (list tĩnh), 1 nút "Add to Cart" — là **một component tree hydrate duy nhất**. Bundle 480KB vì mô tả dùng MDX runtime được bundle vào client, dù không cần tương tác.

**Root Cause:** Không có cơ chế tách phần tĩnh khỏi phần động — toàn bộ phải hydrate.

**Solution (migrate App Router):**

- Mô tả sản phẩm + rich-text render → Server Component, render HTML ở server, **không gửi MDX runtime xuống client**
- Reviews list (chỉ hiển thị) → Server Component
- Chỉ "Add to Cart" + bộ chọn size/màu → Client Component, đặt trong `<Suspense>` riêng để hydrate độc lập

**Kết quả:** Bundle JS giảm 60–80%, TTI < 1s. Đơn vị hydrate co lại từ "cả trang" xuống "1 widget nhỏ".

👉 Rule

> Nếu gặp **TTI chậm trong khi FCP nhanh** hãy nghĩ ngay đến **tách phần tĩnh ra Server Component, chỉ hydrate phần thực sự cần tương tác**.

**8. Debugging Scenario (Debug nhanh)**

Checklist

- □ Mở Incognito/disable extension → lỗi biến mất? → Extension chèn DOM (Grammarly, Dark Reader) → không phải bug code
- □ Grep codebase: `Date.now()`, `new Date()`, `Math.random()`, `crypto.randomUUID()` trong render (không phải `useEffect`)
- □ Tìm `typeof window`, `typeof document`, `typeof localStorage` trong JSX hoặc logic render (không phải effect/event handler)
- □ Kiểm tra HTML hợp lệ: `<table>` thiếu `<tbody>`, `<a>` lồng `<a>`, `<p>` chứa block element → browser tự re-parse
- □ Kiểm tra locale/timezone: server UTC vs client local → `toLocaleDateString()` khác nhau
- □ Kiểm tra `useId()` vs `Math.random()`/`id++` → non-deterministic ID gây mismatch
- □ React DevTools: đọc kỹ diff trong thông báo lỗi — chỉ thẳng node bị lệch
- □ `suppressHydrationWarning` có dùng đúng chỗ? → chỉ cho khác biệt chấp nhận được, không cho mismatch cấu trúc

**9. Refactoring Scenario (Nhìn code cũ sẽ làm gì?)**

Checklist migrate từ Pages Router → App Router (giảm Hydration Cost)

1. **Xác định** phần nào thực sự cần tương tác (state, effect, event handler) → đánh dấu `'use client'`
2. **Tách** phần tĩnh (text, hình ảnh, list hiển thị) ra Server Component — không hydrate
3. **Đẩy** boundary `'use client'` sát lá nhất có thể — không kéo cây con không cần tương tác vào hydrate
4. **Hoãn** non-determinism (`window`, `localStorage`, `Date`) bằng `useEffect` — render đầu cùng giá trị mặc định
5. **Thay** `Math.random()`/`id++` bằng `useId()` — deterministic ID
6. **Dùng** `suppressHydrationWarning` chỉ cho khác biệt chủ ý và chấp nhận được
7. **Thêm** Suspense boundaries theo "đơn vị tương tác" — phần nào cần sớm nhất
8. **Test** với Incognito + disable extension → verify không phải nhiễu môi trường

Không sửa khi

- Component không có server render (pure CSR) → không cần hydrate, không có mismatch
- Dùng `suppressHydrationWarning` cho mismatch cấu trúc → phải fix root cause
- Đặt `'use client'` ở component cha khi chỉ lá cần tương tác → kéo cây con vào hydrate vô ích

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng**

- SSR/SSG với React — cần HTML nhanh + tương tác đầy đủ
- App Router — tách phần tĩnh (Server Component) và phần động (Client Component)
- Selective Hydration với Suspense — chia "đảo" hydrate độc lập
- `useId()` cho deterministic ID trong forms, labels, ARIA
- `useEffect` để hoãn non-determinism (`window`, `localStorage`, `Date`)

**Khi không nên dùng**

- Pure CSR (Create React App) → không có hydration, không cần lo mismatch
- `suppressHydrationWarning` cho mismatch cấu trúc → phải fix root cause
- `'use client'` ở component cha khi chỉ lá cần tương tác → tăng Hydration Cost vô ích
- Chia quá nhỏ Client Component → overhead serialization, nhiều request module nhỏ lẻ

Trade-off

Pros

- FCP nhanh (HTML server-render) + TTI tốt (hydrate chỉ phần cần)
- Server Components giảm bundle JS đáng kể
- Selective Hydration không block Main Thread
- Suspense boundary cho phép ưu tiên phần user tương tác

Cons

- Hydration Mismatch là rủi ro kiến trúc mặc định — cần awareness liên tục
- `typeof window` trong render → dễ gây mismatch nếu không cẩn thận
- Browser extension có thể chèn DOM → nhiễu không kiểm soát được
- Learning curve: phải hiểu Server/Client Component boundary, RSC Payload
- Chia quá nhỏ → overhead serialization và module requests

**Golden Rules**

- ⭐ Hydration = **diff Virtual DOM vs DOM thật** + gắn event listeners — không phải "gắn JS vào HTML"
- ⭐ Hydration Mismatch là **rủi ro kiến trúc mặc định** — không phải lỗi hiếm, cần awareness liên tục
- ⭐ Hoãn non-determinism (`window`, `Date`, `random`) bằng `useEffect` — render đầu cùng giá trị mặc định
- ⭐ Dùng `useId()` cho deterministic ID — không dùng `Math.random()`/`id++` trong render
- ⭐ Đẩy `'use client'` **sát lá nhất** — không kéo cây con không cần tương tác vào hydrate
- ⭐ `suppressHydrationWarning` chỉ cho khác biệt **chủ ý và chấp nhận được** — không cho mismatch cấu trúc
- ⭐ Suspense boundary không chỉ là UX — là **đơn vị Selective Hydration**

## 2. Evolution Context

**Pain Point gốc:** Web có hai cực, cả hai đều tệ.

Cực 1 — Server-rendered thuần (PHP, Rails, jQuery thời kỳ đầu): HTML hiển thị tức thì, nhưng muốn tương tác (toggle menu, validate form) phải viết lại logic ở client bằng tay → trùng lặp logic, hai nguồn sự thật (server logic và client logic lệch nhau).

Cực 2 — CSR thuần (React SPA cổ điển, Create React App): Một file `index.html` rỗng, JS tải xong mới render → màn hình trắng (blank screen), SEO kém, Time To First Byte tốt nhưng First Contentful Paint rất tệ.

**Cách cũ (trước React 16):**

Trước React 16, không có API hydration chuyên dụng. Người ta dùng `ReactDOM.render()`
trên server-rendered HTML. React vẫn cố gắng reuse DOM nodes, nhưng không có
cơ chế so khớp chuyên dụng như `hydrate()`.

Kết quả: có thể gây double render,
flash nội dung, và không tận dụng được DOM nodes đã có hiệu quả.

React 16 (September 2017) giới thiệu `ReactDOM.hydrate()` — API chuyên dụng
để "thổi sự sống" vào DOM server-rendered mà không tạo lại từ đầu.

**Tại sao thất bại:** Ném DOM cũ đi đồng nghĩa với việc HTML server render chỉ có giá trị "trang trí" cho lần paint đầu — không tiết kiệm được công việc thực sự (React vẫn phải tính toán lại toàn bộ Virtual DOM và áp dụo lại vào DOM thật).

**Cách mới:** Hydration đúng nghĩa — React **tái sử dụng DOM nodes đã tồn tại**, chỉ build Virtual DOM tree trong memory để so khớp (reconcile) với DOM thật, rồi **gắn (attach) event listeners** vào đúng node đó. Không xóa, không tạo lại — chỉ "thổi sự sống" vào.

Dòng tiến hóa API:

| Giai đoạn                | API                                       | Đặc điểm                                                                               |
| ------------------------ | ----------------------------------------- | -------------------------------------------------------------------------------------- |
| React 15 trở về trước    | `ReactDOM.render()`                       | Không có hydration rõ ràng — render lên HTML server-rendered, có thể gây double render |
| React 16–17              | `ReactDOM.hydrate(<App/>, container)`     | Hydrate toàn bộ tree, đồng bộ (blocking), một khi bắt đầu không dừng được              |
| React 18+                | `ReactDOM.hydrateRoot(container, <App/>)` | Bật Concurrent Features → **Selective Hydration**, có thể ưu tiên, có thể tạm dừng     |
| Next.js App Router (13+) | RSC + `hydrateRoot` nội bộ                | Chỉ Client Components mới được đưa vào cây hydrate — Server Components không tham gia  |

## 3. Mental Model

Hãy tưởng tượng HTML server-render giống một **bức tượng thạch cao đã tạo hình sẵn**: đẹp, đúng hình dáng, nhìn thấy ngay — nhưng không cử động được.

Hydration giống như **cấy hệ thần kinh vào bức tượng đó** mà không đập vỡ nó: React chạy qua từng "khớp xương" (DOM node) đã có, gắn các "dây thần kinh" (event listeners, state, refs) vào đúng vị trí, để bức tượng có thể bắt đầu cử động khi bị chạm vào (click, input...).

Hệ quả tất yếu của mental model này:

- Nếu hình dáng JS tính ra ở client (Virtual DOM lần đầu) **khác** hình dáng thạch cao đã đổ ở server → React không biết "khớp xương" nào ứng với cái nào → buộc phải **đập và đổ lại từ đầu** (discard & re-render). Đây chính là Hydration Mismatch.
- Cấy thần kinh cho **toàn bộ bức tượng cùng lúc** (Pages Router, hydrate cả cây) tốn thời gian hơn cấy cho **từng phần nhỏ theo nhu cầu** (App Router, chỉ Client Components, có thể chọn ưu tiên phần người dùng sờ vào trước — Selective Hydration).

## 4. Core Concepts

**4.1. Luồng đầy đủ của Hydration**

1. Server chạy `renderToString` / `renderToPipeableStream` → xuất ra HTML string (đã bao gồm các marker ẩn để React nhận diện ranh giới Suspense, component boundary).
2. Browser nhận HTML, **parse và paint ngay** → người dùng thấy nội dung (First Contentful Paint) nhưng **chưa thể tương tác** — đây là "Uncanny Valley" của SSR: nhìn như xong nhưng bấm không có phản hồi.
3. JS bundle tải và execute.
4. `hydrateRoot()` chạy: React build Virtual DOM tree trong memory, **diff với DOM thật hiện có** (không phải diff hai Virtual DOM như Reconciliation thông thường ở M1.2 — đây là diff Virtual DOM vs DOM thật).
5. Nếu khớp → gắn event listener, gắn fiber instance vào DOM node, set up state. App trở thành interactive (Time To Interactive).
6. Nếu không khớp → cảnh báo Hydration Mismatch, React patch hoặc discard tùy mức độ lệch.

**4.2. Hydration Mismatch — nguồn gốc thực sự**

Bản chất: **server render một lần, client render một lần (độc lập)** — bất kỳ thứ gì làm cho hai lần render đó ra **output khác nhau** đều gây mismatch. Các nguồn phổ biến:

```tsx
// ❌ Non-deterministic: server render giờ A, client render (hydrate) giờ B
function Clock() {
  // new Date() chạy lúc server xử lý request, KHÁC thời điểm
  // browser hydrate (chênh vài trăm ms đến vài giây do network)
  return <span>{new Date().toLocaleTimeString()}</span>;
}
```

```tsx
// ❌ Phụ thuộc môi trường: window chỉ tồn tại ở client
function Banner() {
  // Server: typeof window === 'undefined' → render branch A
  // Client (hydrate lần đầu, trước khi effect chạy): branch A vẫn đúng
  // NHƯNG nếu logic check window nằm ngay trong JSX render (không qua effect)
  // → server trả về branch A, client (browser có window) trả về branch B
  const isClient = typeof window !== 'undefined';
  return <div>{isClient ? 'Client View' : 'Server View'}</div>;
}
```

```tsx
// ❌ Random ID không seed: Math.random(), crypto.randomUUID() trong render
function Item() {
  const id = Math.random(); // mỗi lần gọi (server vs client) ra giá trị khác
  return <li key={id}>...</li>;
}
```

```tsx
// ❌ HTML không hợp lệ bị browser tự "sửa"
function Bad() {
  // <p> không được lồng <div> bên trong — browser sẽ tự re-parse,
  // sinh ra cấu trúc DOM khác hoàn toàn so với cây React mong đợi
  return (
    <p>
      <div>Invalid nesting</div>
    </p>
  );
}
```

Một nguồn rất phổ biến trong thực tế production **không nằm trong code của bạn**: **browser extension** (Grammarly, Dark Reader, ad-blockers) chèn thêm attribute vào DOM ngay sau khi HTML được parse, trước khi React hydrate → React thấy DOM "lạ" so với HTML server trả về.

**4.3. Selective Hydration (React 18)**

Trước React 18, hydration là một block đồng bộ — bắt đầu là chạy hết toàn bộ cây, không dừng được, không ưu tiên được. Nếu cây lớn, Main Thread bị block, mọi click trong lúc đó bị "treo".

React 18 + Suspense cho phép:

- **Chia cây thành các "đảo" (island) độc lập** theo ranh giới `<Suspense>`.
- **Hydrate theo độ ưu tiên**: nếu người dùng click vào một phần chưa hydrate, React **ưu tiên hydrate ngay phần đó trước**, các phần khác chờ.
- **Không block Main Thread liên tục**: hydration được chia nhỏ thành các đơn vị công việc (work units), nhường thời gian cho browser xử lý input — đây chính là ứng dụng trực tiếp của Concurrent Rendering / Time Slicing đã học ở M1.5.

```tsx
// ✅ Selective Hydration: phần Comments nặng, không chặn phần Sidebar
// có thể tương tác trước
import { Suspense } from 'react';

export default function ProductPage() {
  return (
    <main>
      <Sidebar /> {/* hydrate ngay, nhẹ */}
      <Suspense fallback={<CommentsSkeleton />}>
        <Comments /> {/* hydrate độc lập, không chặn Sidebar */}
      </Suspense>
    </main>
  );
}
```

**4.4. Cú twist của App Router: không phải mọi thứ đều hydrate**

Đây là khác biệt **kiến trúc**, không phải tối ưu nhỏ: trong App Router, **Server Components không bao giờ hydrate** — chúng không gửi JS xuống client, không có fiber tương ứng cần "sống lại". Chỉ cây con bắt đầu từ `'use client'` mới tham gia quá trình hydrate.

```tsx
// app/page.tsx — Server Component (mặc định, KHÔNG có 'use client')
// → Render ra HTML, KHÔNG gửi JS, KHÔNG hydrate, KHÔNG có khả năng mismatch
// vì nó chỉ render đúng MỘT LẦN (ở server) và không bao giờ chạy lại ở client
import LikeButton from './LikeButton';

export default async function ProductPage() {
  const product = await getProduct(); // chạy ở server, không lộ logic ra client
  return (
    <article>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <LikeButton productId={product.id} /> {/* chỉ phần này hydrate */}
    </article>
  );
}
```

```tsx
// app/LikeButton.tsx — Client Component
'use client';
import { useState } from 'react';

export default function LikeButton({ productId }: { productId: string }) {
  // Component này CẦN hydrate vì cần useState, onClick
  const [liked, setLiked] = useState(false);
  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? '❤️ Đã thích' : '🤍 Yêu thích'}
    </button>
  );
}
```

→ Toàn bộ `<h1>`, `<p>` không bao giờ tốn JS, không bao giờ hydrate, không bao giờ mismatch. Chỉ `<LikeButton>` tham gia.

**RSC Payload — Cơ chế đồng bộ Server-Client**

Trong App Router, Server Components không chỉ "không hydrate" — chúng còn
được serialize thành **RSC Payload** (compact binary format) để gửi xuống client.
Client dùng RSC Payload để:

1. Reconcile cây component giữa server và client
2. Xác định vị trí của Client Components (placeholders)
3. Truyền props từ Server Component sang Client Component

Điều này có nghĩa: Server Components **không hoàn toàn "biến mất"** sau khi
server render xong. Chúng vẫn tham gia vào quá trình reconciliation trên client
thông qua RSC Payload, nhưng không cần hydration (không có event listeners,
không có state, không cần JS bundle).

```typescript
// Ví dụ RSC Payload (đơn giản hóa)
// Server Component trả về:
<div>
  <h1>Product Name</h1>
  <LikeButton productId="123" />  {/* Client Component placeholder */}
</div>

// RSC Payload serialize thành:
["$","div",null,{"children":[
  ["$","h1",null,{"children":"Product Name"}],
  ["$","$L1",null,{"productId":"123"}]  // placeholder cho LikeButton
]}]
// $L1 reference: {id: "path/to/LikeButton.js", chunks: ["chunk-abc.js"]}
```

Đây là lý do tại sao App Router có thể "giảm hydration cost triệt để" —
không phải vì Server Components "không làm gì cả", mà vì chúng tham gia
reconciliation qua RSC Payload (nhẹ) thay vì hydration (nặng).

## 5. Legacy Recognition

Nhận diện code Pages Router (Next 9–12) liên quan đến hydration:

```tsx
// pages/_app.tsx — Pages Router
// TOÀN BỘ cây <Component {...pageProps} /> là MỘT đơn vị hydrate duy nhất.
// Next.js gọi ReactDOM.hydrate (≤17) hoặc hydrateRoot (18) cho CẢ TRANG,
// không có khái niệm "phần này không cần hydrate".
function MyApp({ Component, pageProps }) {
  return <Component {...pageProps} />;
}
export default MyApp;
```

Dấu hiệu nhận biết file Legacy có rủi ro hydration cao:

- Dùng `getServerSideProps`/`getStaticProps` để fetch data, nhưng **toàn bộ component cây**, kể cả phần tĩnh thuần (banner, footer, text mô tả) **vẫn bị gửi JS và hydrate** — không có cách "miễn hydrate" cho phần tĩnh.
- Pattern `if (typeof window !== 'undefined')` rải khắp codebase để né lỗi `window is not defined` khi SSR — đây là **triệu chứng**, không phải **giải pháp**: nó che được lỗi server crash, nhưng nếu đặt sai vị trí (trong JSX render trực tiếp) sẽ tạo Hydration Mismatch.
- Dùng `next/dynamic` với `{ ssr: false }` tràn lan như một "miếng dán" cho mọi component dùng `window`, thay vì thiết kế lại boundary.

```tsx
// pages/dashboard.tsx — pattern né lỗi kiểu cũ, ẨN CHỨA rủi ro mismatch
function Dashboard() {
  // ❌ Đây là pattern phổ biến trong code Next 9-12, ĐÚNG để né server crash
  // nhưng SAI nếu nghĩ nó "an toàn 100% cho hydration"
  const width = typeof window !== 'undefined' ? window.innerWidth : 0;
  // server trả 0, client (sau hydrate) trả về width thật → nếu render
  // trực tiếp giá trị này ra UI ngay lần render đầu → mismatch
  return <div>Width: {width}</div>;
}
```

## 6. Modern Approach

**Nguyên tắc 1 — Không che giấu non-determinism, hoãn nó lại bằng `useEffect`:**

```tsx
'use client';
import { useEffect, useState } from 'react';

export default function ViewportWidth() {
  // Lần render đầu (cả server và client) đều ra null → KHỚP NHAU, không mismatch
  const [width, setWidth] = useState<number | null>(null);

  useEffect(() => {
    // Chỉ chạy ở client, SAU khi hydrate xong → an toàn cập nhật lại UI
    setWidth(window.innerWidth);
  }, []);

  return <div>Width: {width ?? '...'}</div>;
}
```

**Nguyên tắc 2 — Đẩy boundary Client Component xuống tận lá (leaf), không bọc cả cây:**

```tsx
// ❌ Sai: bọc 'use client' ở component cha, kéo theo TOÀN BỘ con cháu
// phải hydrate, dù phần lớn chỉ hiển thị text tĩnh
'use client';
export default function ProductCard({ product }) {
  const [liked, setLiked] = useState(false);
  return (
    <div>
      <h2>{product.name}</h2>
      <p>{product.longDescription}</p> {/* tĩnh, không cần hydrate */}
      <button onClick={() => setLiked(!liked)}>Like</button>
    </div>
  );
}
```

```tsx
// ✅ Đúng: tách phần tương tác ra component riêng, cha vẫn là Server Component
// app/ProductCard.tsx (Server Component)
import LikeButton from './LikeButton';
export default function ProductCard({ product }) {
  return (
    <div>
      <h2>{product.name}</h2>
      <p>{product.longDescription}</p> {/* render ở server, KHÔNG hydrate */}
      <LikeButton productId={product.id} />
    </div>
  );
}
```

**Nguyên tắc 3 — Dùng `suppressHydrationWarning` chỉ khi sự khác biệt là CHỦ Ý và CHẤP NHẬN ĐƯỢC** (ví dụ timestamp hiển thị giờ local, sai khác vài mili-giây không quan trọng):

```tsx
// Chấp nhận được vì sai khác không ảnh hưởng UX, chỉ là 1 con số timestamp
<time suppressHydrationWarning>{new Date().toLocaleTimeString()}</time>
```

> ⚠️ `suppressHydrationWarning` chỉ tắt cảnh báo ở **node đó**, KHÔNG sửa được mismatch cấu trúc (thêm/xóa/đổi loại element). Dùng sai chỗ này là dấu hiệu Senior sẽ bắt được ngay trong code review.

**Nguyên tắc 4 — Tận dụng Suspense để Selective Hydration hoạt động đúng nghĩa**, không chỉ để có loading spinner đẹp:

```tsx
<Suspense fallback={<Skeleton />}>
  <HeavyInteractiveWidget />
</Suspense>
```

## 7. Production Scenario

**Bối cảnh:** Một trang sản phẩm e-commerce (Pages Router, Next 11) có Lighthouse TTI rất tệ (4.2s) trong khi FCP chỉ 0.8s — khoảng cách 3.4s là "vùng chết" người dùng nhìn thấy trang nhưng bấm vào không phản hồi.

**Phân tích:** Toàn bộ trang — bao gồm mô tả sản phẩm (text tĩnh, dài), reviews (list tĩnh), và 1 nút "Add to Cart" — là **một component tree duy nhất**, hydrate đồng bộ. Bundle JS của trang nặng 480KB vì mô tả sản phẩm dùng một rich-text renderer (MDX runtime) được bundle vào client, dù nội dung không cần tương tác.

**Hướng xử lý khi migrate lên App Router:**

- Mô tả sản phẩm + render rich-text → chuyển thành Server Component, render HTML ở server, **không gửi MDX runtime xuống client nữa**.
- Reviews list (chỉ hiển thị) → Server Component.
- Chỉ "Add to Cart" + bộ chọn size/màu → Client Component, đặt trong `<Suspense>` riêng để hydrate độc lập, không chờ phần reviews tải xong.

**Kết quả thực tế kiểu này thường thấy:** bundle JS giảm 60–80%, TTI giảm về dưới 1s, vì đơn vị hydrate đã co lại từ "cả trang" xuống "1 widget nhỏ".

## 8. Debugging Scenario

**Lỗi console:**

```
Error: Hydration failed because the server rendered HTML didn't match the client.
As a result this tree will be regenerated on the client.
```

**Quy trình debug có hệ thống (không đoán mò):**

1.  **Loại trừ browser extension trước tiên** — mở tab Ẩn danh (Incognito) không có extension, hoặc disable hết extension. Nếu lỗi biến mất → nguồn gốc là extension chèn DOM (Grammarly, Dark Reader...), không phải bug code. Xử lý: thêm `suppressHydrationWarning` ở `<html>`/`<body>` nếu không kiểm soát được, vì đây là nhiễu từ môi trường người dùng.
2.  **Grep codebase tìm các nguồn non-determinism kinh điển**: `Date.now()`, `new Date()`, `Math.random()`, `crypto.randomUUID()` nằm **trực tiếp trong phần render** (không phải trong `useEffect`).
3.  **Tìm `typeof window`, `typeof document`, `typeof localStorage`** xuất hiện ngay trong JSX hoặc logic tính giá trị render — không phải trong effect/event handler.
4.  **Kiểm tra HTML hợp lệ**: thẻ `<table>` thiếu `<tbody>`, `<a>` lồng `<a>`, `<p>` chứa block element — browser tự "sửa" DOM khi parse, gây lệch cấu trúc so với React mong đợi.
5.  **Kiểm tra dữ liệu phụ thuộc locale/timezone**: server chạy ở UTC, client chạy ở giờ địa phương người dùng — `toLocaleDateString()` không kèm timezone cố định sẽ ra hai kết quả khác nhau.
6.  React DevTools (bản mới) thường chỉ thẳng ra node bị lệch trong thông báo lỗi (kèm diff) — luôn đọc kỹ phần diff trước khi grep mù.

7.  Kiểm tra `useId()` và deterministic IDs

        React 18 giới thiệu `useId()` hook để tạo IDs deterministic giữa server và client.
        Nếu bạn dùng `Math.random()`, `crypto.randomUUID()`, hoặc incrementing counter
        (`id++`) trong render, sẽ gây hydration mismatch vì server và client tạo ra
        IDs khác nhau.

        ```tsx
        // ❌ Sai — non-deterministic ID
        function FormField() {
          const id = Math.random().toString(36)  // Server và client khác nhau!
          return <input id={id} />
        }

        // ✅ Đúng — deterministic ID với useId()
        import { useId } from 'react'
        function FormField() {
          const id = useId()  // Same ID on server and client
          return <input id={id} />
        }
        ```
        Đây là một trong những nguyên nhân phổ biến nhất gây hydration mismatch

    trong production, đặc biệt với forms, labels, và ARIA attributes.

## 9. Refactoring Scenario

**Trước (Pages Router, Next 10, pattern "né lỗi" kinh điển):**

```tsx
// components/UserGreeting.tsx
function UserGreeting() {
  // ❌ Đọc localStorage trực tiếp trong render để né lỗi "window undefined"
  // khi server-render, nhưng gây mismatch vì server luôn trả "Guest"
  // còn client (nếu đã có user) trả ra tên thật ngay từ lần render đầu
  const username =
    typeof window !== 'undefined'
      ? (localStorage.getItem('username') ?? 'Guest')
      : 'Guest';

  return <p>Xin chào, {username}!</p>;
}
```

**Sau (App Router, tách rõ Server/Client, hoãn non-determinism đúng cách):**

```tsx
// app/UserGreeting.tsx
'use client';
import { useEffect, useState } from 'react';

export default function UserGreeting() {
  // Lần render đầu ở CẢ server và client đều ra 'Guest' → khớp nhau
  const [username, setUsername] = useState('Guest');

  useEffect(() => {
    // Chạy SAU hydrate, an toàn truy cập localStorage
    const stored = localStorage.getItem('username');
    if (stored) setUsername(stored);
  }, []);

  return <p>Xin chào, {username}!</p>;
}
```

**Điểm khác biệt cốt lõi sau refactor:**

- Không còn nhánh `typeof window` ảnh hưởng tới **output của lần render đầu** — cả hai phía luôn xuất phát từ cùng giá trị `'Guest'`.
- Cập nhật giá trị thật diễn ra **sau** khi cây đã hydrate ổn định, qua một re-render bình thường (không phải hydration), nên không có rủi ro mismatch.
- Nếu phần này không cần ở Server Component cha, có thể tách nhỏ hơn nữa để không kéo theo các phần tĩnh xung quanh vào diện hydrate.

## 10. Architecture Notes

- **Quy tắc đặt boundary `'use client'`:** đặt **càng sát lá (leaf) càng tốt**. Một `'use client'` ở component cha sẽ kéo **toàn bộ cây con** vào diện phải hydrate, bất kể con cháu có cần tương tác hay không — đây là lỗi kiến trúc phổ biến nhất khi mới migrate sang App Router ("copy nguyên cụm code cũ, dán `'use client'` lên đầu file để code chạy được").
- **Trade-off của việc chia quá nhỏ:** mỗi boundary Client Component là một "đảo" cần serialize props qua ranh giới Server→Client (xem M6.4 sắp tới) — chia quá mức tạo overhead serialization và tăng số lượng request module nhỏ lẻ. Không có công thức cố định; nguyên tắc thực dụng: **gom theo "đơn vị tương tác"**, không gom theo "đơn vị hiển thị".
- **Suspense không chỉ là UX (loading state) — nó là đơn vị Selective Hydration.** Thiết kế Suspense boundary nên đi cùng câu hỏi: "phần nào người dùng cần tương tác được SỚM NHẤT?" — đặt boundary để phần đó không bị các phần nặng/chậm khác chặn.
- **Hydration Cost không tuyến tính với kích thước trang** — nó tuyến tính với **số lượng và độ phức tạp của Client Components**. Một trang dài nhưng 100% Server Component có Hydration Cost ≈ 0. Đây là lý do RSC được xem là thay đổi kiến trúc, không phải tối ưu hiệu năng thông thường.
- Liên kết tới M7 (Caching) và M4.6 (PPR/Cache Components): việc tách HTML tĩnh (cache được) và phần cần hydrate động là cùng một tư duy thiết kế — ranh giới Server/Client Component thường trùng với ranh giới cacheable/non-cacheable.

## 11. Interview Questions

**[Junior/Mid] Hydration là gì? Tại sao Next.js cần nó?**

<details>
<summary>Đáp án</summary>

Hydration là quá trình React gắn (attach) state, event listeners và logic tương tác vào HTML đã được render sẵn ở server, bằng cách build Virtual DOM trong memory rồi so khớp với DOM thật hiện có — không tạo lại DOM từ đầu. Next.js cần nó để vừa có First Paint nhanh (nhờ HTML server-render) vừa có khả năng tương tác đầy đủ của một ứng dụng React (nhờ JS chạy ở client).

</details>

**[Mid] Hydration Mismatch là gì, và liệt kê ít nhất 3 nguyên nhân phổ biến.**

<details>
<summary>Đáp án</summary>

Hydration Mismatch xảy ra khi HTML mà server render ra khác với cây mà React tính toán ở lần render đầu trên client, khiến React không thể so khớp DOM hiện có và phải bỏ phần đó để render lại. Ba nguyên nhân phổ biến: (1) giá trị non-deterministic trong render như `new Date()` hoặc `Math.random()`; (2) nhánh điều kiện dựa trên `typeof window` đặt trực tiếp trong JSX (server không có window, client có); (3) browser extension chèn thêm attribute/DOM ngay sau khi parse HTML, trước khi React hydrate.

</details>

**[Senior] Selective Hydration của React 18 giải quyết vấn đề gì so với hydration kiểu cũ?**

<details>
<summary>Đáp án</summary>

Trước React 18, `ReactDOM.hydrate` chạy đồng bộ trên toàn bộ cây — một khi bắt đầu thì không dừng được, không ưu tiên được, và có thể block Main Thread khiến input của người dùng bị "treo" nếu cây lớn. Selective Hydration (dựa trên `hydrateRoot` + Suspense boundaries) cho phép chia cây thành các đơn vị hydrate độc lập, ưu tiên hydrate phần người dùng đang tương tác (ví dụ vừa click vào), và nhường thời gian xử lý input cho browser giữa các đơn vị công việc — tận dụng Concurrent Rendering / Time Slicing.

</details>

**[Senior/Staff] Tại sao Server Components trong App Router lại giảm "Hydration Cost" một cách triệt để, thay vì chỉ là tối ưu nhỏ?**

<details>
<summary>Đáp án</summary>

Vì Server Components thay đổi **đơn vị tham gia hydrate**: chúng render ra HTML ở server và không bao giờ chạy lại ở client — không có fiber tương ứng cần "sống lại", không gửi JS xuống, nên không tốn bất kỳ chi phí hydration nào. Chỉ cây con bắt đầu từ ranh giới `'use client'` mới tham gia hydrate. Ở Pages Router, ngược lại, toàn bộ page là một cây hydrate duy nhất — dù 90% nội dung là text tĩnh, JS và công việc hydrate vẫn tính trên 100% cây đó. Vì vậy mức giảm Hydration Cost khi chuyển sang App Router tỉ lệ với tỉ lệ nội dung tĩnh trong trang, không phải một con số cố định.

</details>

**[Staff/Architect] Đặt boundary `'use client'` ở component cha thay vì component lá gây ra hệ quả gì về kiến trúc?**

<details>
<summary>Đáp án</summary>

Đặt `'use client'` ở component cha sẽ kéo toàn bộ cây con vào diện Client Component — mọi component con, kể cả những phần thuần hiển thị (text, hình ảnh tĩnh) đều bị gửi JS xuống và phải hydrate, dù chúng không cần tương tác. Điều này làm tăng bundle size, tăng Hydration Cost, và đi ngược lại lợi ích chính của RSC. Nguyên tắc đúng là đẩy boundary xuống lá càng sát phần thực sự cần `useState`/`useEffect`/event handler càng tốt, đồng thời truyền dữ liệu cần thiết qua props từ Server Component cha.

</details>

## 12. Exercises

**Bài 1 (Mid):** Đoạn code dưới đây gây Hydration Mismatch. Tìm nguyên nhân và sửa lại theo nguyên tắc "hoãn non-determinism bằng effect".

```tsx
function PromoBanner() {
  const isWeekend = [0, 6].includes(new Date().getDay());
  return isWeekend ? <p>Giảm giá cuối tuần!</p> : <p>Khám phá ngay</p>;
}
```

<details>
<summary>Đáp án</summary>

Nguyên nhân: `new Date().getDay()` được tính ngay trong render, tại hai thời điểm khác nhau (server xử lý request vs client hydrate vài trăm ms-vài giây sau), và có thể rơi vào hai phía của ranh giới cuối tuần nếu request chạy đúng lúc giao thời (ví dụ 23:59:59 Chủ Nhật ở server, 00:00:01 Thứ Hai ở client theo múi giờ khác) — quan trọng hơn, ngay cả khi không lệch ngày, giá trị này vẫn phụ thuộc đồng hồ máy chủ và máy khách, là nguồn rủi ro mismatch không kiểm soát được.

```tsx
'use client';
import { useEffect, useState } from 'react';

export default function PromoBanner() {
  // Render đầu tiên (server và client) luôn ra cùng giá trị mặc định
  const [isWeekend, setIsWeekend] = useState(false);

  useEffect(() => {
    const day = new Date().getDay();
    setIsWeekend(day === 0 || day === 6);
  }, []);

  return isWeekend ? <p>Giảm giá cuối tuần!</p> : <p>Khám phá ngay</p>;
}
```

Lưu ý: nếu trạng thái cuối tuần cần đúng ngay từ lần render đầu (ví dụ ảnh hưởng SEO), giải pháp đúng hơn là tính `isWeekend` ở server dựa trên giờ server, chấp nhận đó là "sự thật" duy nhất, và không có nhánh client tính lại khác đi.

</details>

```tsx
// pages/article/[id].tsx
function ArticlePage({ article, relatedArticles }) {
```

<details>
<summary>Đáp án</summary>

Phân tích: toàn bộ trang hiện là một Client Component ngầm định (vì dùng `useState` ở component cấp trang) — kéo theo `RelatedList` và phần nội dung bài viết (text tĩnh, có khi rất dài) vào diện hydrate, dù chúng không cần tương tác gì.

```tsx
// app/article/[id]/page.tsx — Server Component, KHÔNG hydrate
import ShareButton from './ShareButton';
import RelatedList from './RelatedList';

export default async function ArticlePage({
  params,
}: {
  params: { id: string };
}) {
  const article = await getArticle(params.id);
  const relatedArticles = await getRelatedArticles(params.id);

  return (
    <div>
      <h1>{article.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: article.contentHtml }} />
      <ShareButton url={article.url} /> {/* chỉ phần này hydrate */}
      <RelatedList items={relatedArticles} />{' '}
      {/* thuần hiển thị, render ở server */}
    </div>
  );
}
```

```tsx
// app/article/[id]/ShareButton.tsx — Client Component, đơn vị hydrate duy nhất
'use client';
import { useState } from 'react';
import ShareMenu from './ShareMenu';

export default function ShareButton({ url }: { url: string }) {
  const [showShareMenu, setShowShareMenu] = useState(false);
```

## 13. Summary

- Hydration = React tái sử dụng DOM server-render có sẵn, build Virtual DOM trong memory, so khớp rồi gắn event listener — không tạo lại DOM từ đầu.
- Hydration Mismatch xảy ra khi output của lần render server và lần render client đầu tiên khác nhau — nguồn gốc kinh điển: `Date`/`Math.random`, `typeof window` trong render, HTML không hợp lệ, browser extension.
- React 18 mang đến Selective Hydration: chia cây theo Suspense boundary, ưu tiên hydrate phần người dùng tương tác, không block Main Thread liên tục — đây là ứng dụng trực tiếp của Concurrent Rendering (M1.5).
- App Router thay đổi đơn vị hydrate: chỉ Client Components (`'use client'`) mới hydrate; Server Components không bao giờ tham gia — đây là lý do RSC giảm Hydration Cost mang tính kiến trúc, không phải tối ưu thông thường.
- Nguyên tắc thực hành: đẩy boundary `'use client'` xuống sát lá nhất có thể, hoãn mọi giá trị non-deterministic bằng `useEffect`, chỉ dùng `suppressHydrationWarning` cho khác biệt chấp nhận được, và thiết kế Suspense boundary theo "đơn vị tương tác" thực sự cần sớm.
