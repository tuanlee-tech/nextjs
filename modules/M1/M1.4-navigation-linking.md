# M1.4: Navigation & Linking

> 🎯 **Lesson Goal:** Sử dụng `next/link` với prefetch understanding, programmatic navigation với `useRouter` từ `next/navigation`, handle search params với `useSearchParams`, và phân biệt `next/navigation` vs `next/router`.

> ⚠️ **Đọc trước khi code:** Nếu trước đây bạn từng đụng Pages Router, tay bạn sẽ tự động gõ `import { useRouter } from 'next/router'`. Đó là phản xạ sai đầu tiên 9/10 người mới chuyển qua App Router gặp phải — và nó không fail âm thầm, nó crash. Giữ trong đầu: **App Router = `next/navigation`**, không có ngoại lệ.

---

## 1. Evolution Context

Multi-page app (MPA) navigate bằng cách request lại nguyên trang HTML từ server. Mỗi lần click link, browser tải lại toàn bộ document — flash trắng, layout re-mount, state trong React (giỏ hàng, filter đang chọn, scroll position) bay sạch.

Single-page app (SPA) giải quyết vấn đề này bằng client-side routing: React Router intercept click, swap component tree, không reload trang. Vấn đề là React Router không biết gì về Server Component tree — nó chỉ chạy ở client, nên sau mỗi navigate phải fetch lại toàn bộ data ở client (waterfall: render trước, fetch sau, render lại).

Next.js Pages Router cải thiện bằng prefetch, nhưng vẫn prefetch **toàn bộ page** — cả JS bundle và JSON data — bất kể bạn chỉ đổi 1 segment nhỏ trong layout.

App Router hiện tại (`next/navigation`) là bước tiếp theo trong cùng đường hướng đó: RSC-aware, biết phân biệt segment nào đã có trong cache, segment nào cần fetch mới.

**Timeline đã xác nhận:**
`<a>` tag (full reload) → React Router (client SPA routing) → Next.js `<Link>` Pages Router (prefetch toàn page) → Next.js `<Link>` App Router / `next/navigation` (RSC-aware, incremental prefetch).

---

## 2. Mental Model

> 💡 **[MODEL]** Router Cache trong App Router là segment-based — lưu RSC payload trong memory, hoàn toàn khác Pages Router (lưu toàn bộ page JSON theo từng route).
>
> **[ANALOGY]** Tưởng tượng một thư viện: Pages Router giống việc photocopy lại nguyên cuốn sách mỗi lần bạn lật sang chương mới. App Router giống việc bạn chỉ photocopy đúng những trang chưa có sẵn trên bàn — các trang chung như layout, sidebar đã nằm đó từ trước, không ai đi photocopy lại chúng.

Áp dụng vào Rental Management SaaS: sidebar (logo, menu, thông tin landlord đang login) là phần dùng chung giữa `/dashboard`, `/dashboard/properties`, `/dashboard/revenue`. Khi navigate giữa các route này, App Router chỉ fetch RSC payload của phần `children` thay đổi — sidebar không bị fetch lại, không bị re-mount.

## 3. Core Concepts

### 3.1 `next/link` — Client-Side Navigation & Prefetch

**[FACT]** `<Link>` từ `next/link` thực hiện client-side navigation. Nó không trigger full page reload và preserve state của layout cha (sidebar, header, modal đang mở ở layout trên). Source: `nextjs.org/docs/app/api-reference/components/link`.

**[FACT]** Next.js 16 dùng incremental prefetch: khi `<Link>` enter viewport, chỉ RSC payload của segment **chưa có** trong Router Cache mới được fetch — không phải toàn bộ route. Source: Next.js 16 Release Notes / `next/link` docs.

Trong sidebar của Rental Management SaaS, một số route (báo cáo doanh thu) trả về data nặng và đổi liên tục. Prefetch tự động cho những route này là lãng phí — đó là lý do `prefetch={false}` tồn tại.

```tsx [src/components/navigation/sidebar-nav.tsx]
import Link from 'next/link';

// [FACT] <Link> thực hiện client-side navigation, preserve state của layout cha.
// [FACT] Next.js 16 incremental prefetch — chỉ fetch segment chưa có trong Router Cache.
// Source: nextjs.org/docs/app/api-reference/components/link
interface PropertyNavItem {
  id: string;
  label: string;
  href: string;
  // Route trả data nặng / đổi liên tục -> không nên prefetch tự động khi vào viewport
  isHeavyDynamicRoute?: boolean;
}

const NAV_ITEMS: PropertyNavItem[] = [
  { id: 'overview', label: 'Tổng quan', href: '/dashboard' },
  { id: 'properties', label: 'Bất động sản', href: '/dashboard/properties' },
  { id: 'revenue', label: 'Báo cáo doanh thu', href: '/dashboard/revenue', isHeavyDynamicRoute: true },
];

export function SidebarNav({ items = NAV_ITEMS }: { items?: PropertyNavItem[] }) {
  return (
    <nav aria-label="Main navigation">
      <ul>
        {items.map((item) => (
          <li key={item.id}>
            <Link href={item.href} prefetch={item.isHeavyDynamicRoute ? false : undefined}>
              {item.label}
            </Link>
          </li>
        ))}
      </ul>
    </nav>
  );
}
```

### 3.2 `useRouter` từ `next/navigation` — Programmatic Navigation

**[FACT]** `useRouter` trong App Router phải import từ `next/navigation`. Import từ `next/router` sẽ throw lỗi hoặc dùng sai instance, vì `next/router` phụ thuộc Pages Router context không tồn tại trong App Router tree. Source: `nextjs.org/docs/app/api-reference/functions/use-router`.

Dùng khi nào: bất kỳ navigation nào xảy ra **sau một hành động**, không phải do user click trực tiếp vào link — ví dụ sau khi submit form tạo hợp đồng thuê, redirect sang trang chi tiết hợp đồng vừa tạo.

```tsx [src/components/contracts/contract-create-form.tsx]
'use client';

import { useRouter } from 'next/navigation';
// [FACT] App Router -> next/navigation. Import từ next/router sẽ throw
// "Invariant: Invalid hook call" vì context Pages Router không tồn tại ở đây.
import { useState } from 'react';
import { captureException } from '@/lib/telemetry'; // client-side error tracking (Sentry/Datadog RUM tương đương)

interface CreateContractPayload {
  propertyId: string;
  tenantId: string;
  startDate: string;
}

async function createContract(payload: CreateContractPayload): Promise<{ id: string }> {
  const res = await fetch('/api/contracts', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(payload),
  });

  if (!res.ok) {
    // Không swallow lỗi — throw để caller log đầy đủ context và hiển thị UI phù hợp
    throw new Error(`Failed to create contract: ${res.status} ${res.statusText}`);
  }

  return res.json();
}

export function ContractCreateForm({ propertyId, tenantId }: { propertyId: string; tenantId: string }) {
  const router = useRouter();
  const [isSubmitting, setIsSubmitting] = useState(false);
  const [error, setError] = useState<string | null>(null);

  async function handleSubmit(startDate: string) {
    setIsSubmitting(true);
    setError(null);

    try {
      const contract = await createContract({ propertyId, tenantId, startDate });

      // [FACT] router.push() là programmatic navigation — không có viewport-based
      // prefetch trigger như <Link>, nhưng đúng công cụ cho redirect sau hành động.
      router.push(`/dashboard/contracts/${contract.id}`);
    } catch (err) {
      captureException(err, {
        tags: { feature: 'contract-create' },
        extra: { propertyId, tenantId },
      });
      setError('Không thể tạo hợp đồng. Vui lòng thử lại.');
    } finally {
      setIsSubmitting(false);
    }
  }

  return (
    <form
      onSubmit={(e) => {
        e.preventDefault();
        const startDate = new FormData(e.currentTarget).get('startDate') as string;
        void handleSubmit(startDate);
      }}
    >
      <input name="startDate" type="date" required />
      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Đang tạo...' : 'Tạo hợp đồng'}
      </button>
      {error && <p role="alert">{error}</p>}
    </form>
  );
}
```

### 3.3 `useSearchParams` — Query String Handling

**[FACT]** `useSearchParams()` trả về `URLSearchParams` read-only. Next.js 16 bắt buộc component dùng hook này phải nằm trong `<Suspense>` — nếu không, toàn bộ route segment bị deopt sang client-side rendering, mất static rendering. Source: `nextjs.org/docs/app/api-reference/functions/use-search-params`.

Cách đúng: tách phần dùng `useSearchParams` ra component riêng, chỉ component đó bị deopt CSR — phần shell còn lại của page vẫn static.

```tsx [src/app/dashboard/properties/page.tsx]
import { Suspense } from 'react';
import { PropertyFilterResults } from './property-filter-results';

// page.tsx KHÔNG gọi useSearchParams trực tiếp. Phần dynamic được tách
// ra component riêng để giữ shell (heading, layout) ở dạng static render.
export default function PropertiesPage() {
  return (
    <div>
      <h1>Danh sách bất động sản</h1>
      <Suspense fallback={<PropertyListSkeleton />}>
        <PropertyFilterResults />
      </Suspense>
    </div>
  );
}

function PropertyListSkeleton() {
  return <div aria-busy="true">Đang tải danh sách...</div>;
}
```

```tsx [src/app/dashboard/properties/property-filter-results.tsx]
'use client';

import { useSearchParams } from 'next/navigation';

// Component duy nhất dùng useSearchParams trong page này.
// Chỉ component này bị deopt CSR — không kéo theo phần shell của page.tsx.
export function PropertyFilterResults() {
  const searchParams = useSearchParams();
  const status = searchParams.get('status') ?? 'all';
  const city = searchParams.get('city');

  // Production: fetch data theo status/city ở đây, render danh sách thật.
  return (
    <ul>
      <li>Trạng thái lọc: {status}</li>
      {city && <li>Thành phố: {city}</li>}
    </ul>
  );
}
```

### 3.4 `usePathname` — Current Path

`usePathname()` từ `next/navigation` lấy current path. Khác `useSearchParams`, giá trị này luôn có sẵn ngay tại lần render đầu tiên — không phụ thuộc query string động, nên không cần `<Suspense>`.

```tsx [src/components/navigation/active-nav-link.tsx]
'use client';

import Link from 'next/link';
import { usePathname } from 'next/navigation';

export function ActiveNavLink({ href, children }: { href: string; children: React.ReactNode }) {
  const pathname = usePathname();
  const isActive = pathname === href;

  return (
    <Link href={href} aria-current={isActive ? 'page' : undefined} data-active={isActive}>
      {children}
    </Link>
  );
}
```

### 3.5 `<a>` vs `<Link>` — When to Use Which

**[FACT]** Dùng `<a>` thuần cho internal route sẽ bypass Router Cache, trigger full browser navigation, xóa sạch React state. Source: Next.js Link docs.

Quy tắc đơn giản: route nội bộ (cùng domain, do app của bạn handle) → `<Link>`. URL bên ngoài → `<a>`.

```tsx [src/components/property-detail/external-map-link.tsx]
import Link from 'next/link';

export function PropertyExternalLinks({ propertyId, mapUrl }: { propertyId: string; mapUrl: string }) {
  return (
    <div>
      {/* Internal route — luôn <Link> để giữ Router Cache + state */}
      <Link href={`/dashboard/properties/${propertyId}/edit`}>Sửa thông tin</Link>

      {/* External URL — target + rel bắt buộc, tránh reverse tabnabbing */}
      <a href={mapUrl} target="_blank" rel="noopener noreferrer">
        Xem trên Google Maps
      </a>
    </div>
  );
}
```

## 4. Anti-Patterns in the Wild

### AP1: Dùng `<a>` thay vì `<Link>`

❌ **Code Smell:**
```tsx [src/components/navigation/sidebar-nav.tsx]
// Code junior thường viết — dùng <a> cho route nội bộ
export function SidebarNavBad() {
  return (
    <nav>
      <a href="/dashboard">Tổng quan</a>
      <a href="/dashboard/properties">Bất động sản</a>
    </nav>
  );
}
```

✅ **Staff Refactor:**
```tsx [src/components/navigation/sidebar-nav.tsx]
import Link from 'next/link';

// Trade-off: không có trade-off tiêu cực. <Link> luôn tốt hơn cho internal route.
export function SidebarNavGood() {
  return (
    <nav>
      <Link href="/dashboard">Tổng quan</Link>
      <Link href="/dashboard/properties">Bất động sản</Link>
    </nav>
  );
}
```

🧠 **Why:** Không có trade-off positive khi giữ `<a>`. `<Link>` luôn tốt hơn cho internal route.

🔥 **Incident Risk:** User click menu, thấy flash trắng, layout re-mount, giỏ hàng / filter / scroll position biến mất khi chuyển trang.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **`<Link>` (Default)** | Auto incremental prefetch, RSC aware, preserve state | Không | Internal Next.js routes | External URLs | Nhỏ (memory cache) |
| **`<a>` tag** | Native HTML, đơn giản | Full page reload, mất state, không prefetch | External URLs (`https://...`) | Internal Next.js routes | Cao (full network + parse) |
| **`useRouter().push()`** | Programmatic control (sau submit form) | Mất viewport-based prefetch trigger | Post-login redirect, form submit | Standard navigation UI | Bằng `<Link>` |
| **`useSearchParams` (No Suspense)** | Code nhanh, không tách component | Phá hỏng static rendering toàn page | Không bao giờ dùng | Always | Rất cao (mất SSG) |

## 6. Legacy Recognition

- **Legacy:** `next/router` (Pages Router) — import path và API khác hoàn toàn với `next/navigation`.
- **Legacy:** `router.query` — object này bị loại bỏ trong `next/navigation`.
- **Legacy:** Pages Router `<Link>` — prefetch toàn bộ page JSON thay vì incremental RSC payload.
- **Legacy:** `window.location.href` để navigate programmatically trong React component — bỏ qua hoàn toàn Router Cache.

---

## 7. Modern Approach

- **Modern:** `next/navigation` (App Router) — `useRouter`, `usePathname`, `useSearchParams`, `useParams`.
- **Migration import path:** Find & replace `from 'next/router'` thành `from 'next/navigation'` — làm theo route, không làm toàn repo trong 1 PR.
- **Migration `router.query`:** Thay `const { id } = router.query` bằng `const { id } = useParams()` (từ `next/navigation`).
- **Migration `useSearchParams` gây crash static page:** Tách component chứa `useSearchParams` ra Client Component riêng, inject qua `<Suspense>` — pattern giống 3.3.

**Strategy cho codebase lớn:** đây là Strangler Fig pattern áp dụng vào routing. Next.js cho phép `pages/` và `app/` chạy song song trong cùng project (hybrid mode) — bạn migrate route theo từng nhóm nhỏ (vd: hết toàn bộ flow "Quản lý hợp đồng" trước, để "Quản lý thanh toán" lại Pages Router), không bao giờ rewrite toàn bộ app cùng lúc.

**Rollback plan:** Nếu sau khi migrate 1 route sang `app/` phát hiện bug nghiêm trọng ở production, xoá file route đó trong `app/` — Next.js tự động fallback phục vụ lại route tương ứng trong `pages/` nếu file đó còn tồn tại. Không cần revert toàn bộ migration, không cần deploy emergency rollback cho cả app.

## 8. Production Scenario

### Scenario 1: Unnecessary Full Page Reloads

- **Symptom:** Layout bị unmount/remount, global state (badge số hợp đồng sắp hết hạn, session UI của landlord) bị reset khi click menu.
- **Root cause:** Dùng `<a>` thay vì `<Link>` cho route nội bộ.
- **Detect:** Mở Network tab, thấy request trả về `text/html` document đầy đủ thay vì RSC payload (Flight data).
- **Fix:** Đổi thành `<Link>`.

### Scenario 2: Page bị Deopt sang CSR Bất Thường

- **Symptom:** View Source không thấy HTML của dynamic segment, Lighthouse báo "Does not serve static HTML".
- **Root cause:** `useSearchParams` dùng không có `<Suspense>` ở parent.
- **Detect:** Check terminal output lúc build, warning "Page could not be statically rendered".
- **Fix:** Thêm `<Suspense>`, tách component theo pattern ở 3.3.

### Scenario 3: Router Cache Phục Vụ Data Cũ

- **Symptom:** Landlord cập nhật giá thuê property, navigate sang tab khác rồi quay lại — vẫn thấy giá cũ.
- **Root cause:** Router Cache lưu RSC payload cũ, không tự revalidate khi data server thay đổi do hành động khác (vd: action ở 1 tab khác, hoặc API call trực tiếp).
- **Detect:** React DevTools kiểm tra component tree state, so sánh với data thật trong database.
- **Fix:** Gọi `router.refresh()` để clear segment cache và re-fetch RSC payload mới nhất.

## 9. Performance & Overhead

Raw material hiện tại chưa có benchmark số cụ thể (bandwidth saving của incremental prefetch, memory footprint của Router Cache) — đừng tự bịa số liệu. Thay vào đó, đây là cách bạn tự đo trên codebase thật:

- **Đo bandwidth tiết kiệm của incremental prefetch:** Mở DevTools → Network → filter theo response header `Next-Router-State-Tree` hoặc filter type `fetch`/`other` chứa RSC payload (định dạng text dạng flight, không phải JSON thuần). So sánh tổng `transferred size` khi navigate giữa 2 route share layout vs 2 route không share layout.
- **Xác nhận route nào static, route nào dynamic:** Đọc kỹ output của `next build` — ký hiệu `○ (Static)` vs `ƒ (Dynamic)` cạnh mỗi route. Nếu route bạn nghĩ là static lại hiện `ƒ`, khả năng cao là do `useSearchParams` thiếu `<Suspense>` (AP2).
- **Đo tác động lên TTFB/CLS:** Chạy Lighthouse (tab Performance, throttling "Slow 4G") trước và sau khi thêm `prefetch={false}` cho route nặng — nếu page có 50+ `<Link>` trong viewport (vd: danh sách property), prefetch đồng thời 50 route có thể tạo network congestion đo được rõ trong waterfall chart.
- **Đo Router Cache:** Dùng React DevTools Profiler kết hợp Network tab — click qua lại giữa các route đã visit, nếu không thấy request RSC mới fire, nghĩa là cache đang serve từ memory.

## 10. Architecture Notes

- **Router Cache Strategy:** Cache trong App Router là in-memory, segment-based ở client, lưu RSC payload của route đã visit hoặc được prefetch. Vì segment-based, shared layout (sidebar, header) không bị fetch lại khi navigate giữa các children của layout đó.
- **Prefetch Strategy:** Default `<Link>` prefetch khi vào viewport. Dùng `prefetch={false}` cho route dynamic nặng hoặc route private (dashboard nội bộ) để tránh lãng phí bandwidth và tránh leak request không cần thiết. Giữ default cho route public, content tĩnh.
- **Pitfall — `useRouter().query` is undefined:** Object `query` không tồn tại trong `next/navigation`. Dùng `useParams()` thay thế. (Source: Next.js App Router Migration Guide).
- **Pitfall — Scroll to top bị thiếu:** `<Link>` chỉ auto scroll to top khi layout segment thay đổi; nếu chỉ đổi search params, scroll giữ nguyên vị trí cũ. `[THEORY — NEEDS VERIFICATION]` cơ chế fallback chính xác khi chỉ query thay đổi — cần test trực tiếp trên version Next.js bạn đang dùng trước khi assume hành vi.
- **Pitfall — `window.location` trong Server Component:** Server Component không có `window`. Đọc path qua `headers().get('x-next-url')` ở server, hoặc đẩy logic xuống Client Component dùng `usePathname()`.
- **Middleware Redirects (chỉ nhắc qua — chi tiết thuộc M7):** Redirect set ở middleware không cần `useRouter` — Next.js tự handle navigation ở network layer trước khi React render.

## 11. Interview Questions

### Q1: `next/navigation` khác `next/router` thế nào? (Junior)
<details>
<summary>Đáp án</summary>

`next/navigation` là API cho App Router — RSC-aware, không có object `query`. `next/router` là API legacy cho Pages Router. Dùng sai package trong sai router model sẽ throw lỗi hook invalid ngay lập tức.

</details>

### Q2: Tại sao `useSearchParams` cần Suspense boundary? (Mid)
<details>
<summary>Đáp án</summary>

Search params có thể thay đổi tùy ý theo URL người dùng gõ, nên component dùng nó không thể static render được. `<Suspense>` cho phép Next.js render phần tĩnh của page ngay lập tức, hoãn phần phụ thuộc search params đến khi client hydrate xong — không kéo cả page xuống CSR.

</details>

### Q3: Khi nào nên dùng `prefetch={false}` trên `<Link>`? (Mid)
<details>
<summary>Đáp án</summary>

Khi link dẫn tới trang có data nặng hoặc dashboard private, mà việc prefetch tự động lúc hover/viewport sẽ tốn bandwidth hoặc fire request không cần thiết tới resource nhạy cảm.

</details>

### Q4: Dùng `<a>` cho external link có ổn không? (Junior)
<details>
<summary>Đáp án</summary>

Có, và bắt buộc. `<Link>` chỉ dành cho internal route. External link phải dùng `<a>` kèm `target="_blank"` và `rel="noopener noreferrer"` để tránh lỗi bảo mật reverse tabnabbing.

</details>

### Q5: Incremental prefetching trong Next 16 giải quyết vấn đề gì của Pages Router? (Senior)
<details>
<summary>Đáp án</summary>

Pages Router prefetch toàn bộ JSON data của 1 page mỗi lần. Next 16 chỉ fetch RSC payload của subtree chưa có trong Router Cache — shared layout không bị fetch lại — giảm đáng kể data transfer khi user navigate trong cùng layout.

</details>

## 13. Summary

- `<Link>` là default cho internal route, `<a>` chỉ dùng cho external URL — không bao giờ dùng `<a>` cho navigation nội bộ.
- `useSearchParams` bắt buộc nằm trong `<Suspense>` — nếu không, toàn page mất static rendering. Tách ra component riêng.
- `useRouter` import từ `next/navigation` — `next/router` là Pages Router, dùng sai sẽ crash ngay khi mount.
- Incremental prefetch (Next.js 16) chỉ fetch segment chưa có — shared layout không bị fetch lại, tiết kiệm bandwidth.

## 14. Terminology Evolution

### `next/navigation`
```
┌─────────────┬─────────────────┬─────────────────────────────┐
│ Era         │ Status          │ Dùng khi nào                │
├─────────────┼─────────────────┼─────────────────────────────┤
│ Next.js 13+ │ Active          │ App Router navigation hooks │
│ Next.js <13 │ N/A             │ Không tồn tại               │
│ Next.js 12  │ Legacy          │ next/router (Pages Router)  │
└─────────────┴─────────────────┴─────────────────────────────┘
```
**Lịch sử:** App Router navigation hooks — bao gồm `useRouter`, `usePathname`, `useSearchParams`, `useParams`. Khác hoàn toàn `next/router`.
**Codebase legacy:** `import { useRouter } from 'next/router'` — phải đổi thành `next/navigation` khi route đó chuyển sang App Router.

### `Incremental prefetch`
```
┌──────────────┬─────────────────┬──────────────────────────────────┐
│ Era          │ Status          │ Dùng khi nào                     │
├──────────────┼─────────────────┼──────────────────────────────────┤
│ Next.js 16+  │ Active          │ Chỉ prefetch segment chưa có     │
│              │                 │ trong Router Cache               │
│ Next.js 13-15│ Legacy          │ Prefetch toàn bộ route RSC payload│
│ Pages Router │ Legacy          │ Prefetch toàn bộ page JSON       │
└──────────────┴─────────────────┴──────────────────────────────────┘
```
**Lịch sử:** Next.js 16 chỉ prefetch phần chưa có trong Router Cache, không prefetch toàn bộ route — giảm data transfer.
**Codebase legacy:** Pages Router prefetch toàn bộ page data (JS + JSON) bất kể đã có sẵn hay chưa.

### `Router Cache`
```
┌──────────────┬─────────────────┬──────────────────────────────────┐
│ Era          │ Status          │ Dùng khi nào                     │
├──────────────┼─────────────────┼──────────────────────────────────┤
│ Next.js 13+  │ Active          │ In-memory client cache lưu RSC   │
│              │                 │ payload theo segment             │
│ Pages Router │ Legacy          │ Page-level JSON cache             │
└──────────────┴─────────────────┴──────────────────────────────────┘
```
**Lịch sử:** In-memory cache phía client lưu RSC payload của route đã visit hoặc được prefetch. Segment-based — đây là khái niệm không nằm sẵn trong Bible, được bổ sung từ raw material.
**Codebase legacy:** Pages Router lưu toàn bộ page JSON, không phân biệt theo segment.
