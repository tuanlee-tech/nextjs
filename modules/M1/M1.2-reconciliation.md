# M1.2 — Reconciliation: Cách React Diff Hoạt Động

## 1. Lesson Goal

Sau bài này, bạn phải trả lời được chính xác — không phải đại khái — các câu hỏi sau:

- Khi `setState` được gọi, React quyết định "update tại chỗ" hay "unmount + mount lại" dựa trên tiêu chí nào?
- Tại sao `key={index}` lại nguy hiểm, và nguy hiểm trong trường hợp nào thì không?
- Stack Reconciler và Fiber Reconciler khác nhau ở **cái gì**, không khác nhau ở **cái gì**?
- "Bailout" trong reconciliation nghĩa là gì, và khi nào React tận dụng được nó?

Đây không phải bài học về syntax `key`. Đây là bài học về mô hình tư duy đứng sau toàn bộ rendering pipeline mà bạn đã chạm ở M1.1.

## 2. Evolution Context

### Pain Point

Bài toán gốc: cho hai cây (cây UI cũ và cây UI mới sau khi state thay đổi), tìm tập hợp tối thiểu các phép biến đổi (insert/delete/move) để biến cây cũ thành cây mới.

Đây chính là bài toán **tree edit distance** trong khoa học máy tính — thuật toán tổng quát tối ưu cho nó có độ phức tạp **O(n³)**. Với một cây UI thực tế có vài nghìn node, chạy lại thuật toán này trên mỗi lần render là bất khả thi. Một thao tác gõ phím trong input cũng có thể trigger render — không thể chấp nhận một thuật toán cubic chạy nhiều lần mỗi giây.

Đây là pain point thật của team React năm 2013: họ không cần một thuật toán diff *đúng nhất*, họ cần một thuật toán diff *đủ nhanh* để chạy real-time.

**Ví dụ cây component:**
```
        App
       /   \
    Header  Main
            |
          Button
```
### Cách cũ: Stack Reconciler

Giải pháp đầu tiên (React 0.x → 15) là **Stack Reconciler**: một thuật toán đệ quy, đồng bộ, dùng trực tiếp call stack của JavaScript để đi qua cây Element. Mỗi lần có update, React đi từ root, đệ quy xuống từng component, gọi `render()`, so sánh, áp dụng thay đổi — tất cả trong một lần chạy không thể dừng.

```
React cũ đi xuống bằng đệ quy       Trong JS engine:
                                    Call Stack
render(App)                         ┌─────────────┐
   |                                │ Button()    │  ← đang chạy
   ├── render(Header)               ├─────────────┤
   |                                │ Main()      │
   └── render(Main)                 ├─────────────┤
          |                         │ App()       │
          └── render(Button)        └─────────────┘
```

### Tại sao thất bại

Stack Reconciler hoạt động tốt khi cây nhỏ. Nhưng nó có ba giới hạn không thể vượt qua khi ứng dụng lớn lên:

- **Không thể tạm dừng.** Vì dùng call stack thật, JS engine phải chạy hết một lần gọi đệ quy mới được nhường quyền điều khiển. Cây lớn → block main thread → drop frame → giật, lag khi người dùng đang gõ hoặc scroll.
- **Không thể ưu tiên.** Một update "khẩn cấp" (gõ phím) và một update "không khẩn cấp" (render lại danh sách 5000 dòng dữ liệu real-time) được xử lý với độ ưu tiên ngang nhau — theo đúng thứ tự gọi.
- **Không thể abort hoặc tái sử dụng work đã làm.** Nếu một update mới đến giữa lúc đang xử lý update cũ, Stack Reconciler không có cách nào hủy phần đang làm — phải chạy xong rồi mới xử lý tiếp.

```
Nếu cây lớn:              React gọi:
App                       App()
 ├─ A                         A()
 │   ├─ A1                      A1()
 │   ├─ A2                      A2()
 │   └─ A3                      A3()
 └─ B                         B()
     ├─ B1                      B1()
     └─ B2                      B2()

Một khi bắt đầu render() thì JS phải chạy hết.

Không thể:
render()
...
⏸ pause
vì vị trí nằm trong call stack.

```

Đây chính là rào cản khiến team React không thể build Suspense, Concurrent Rendering, hay Time Slicing trên nền Stack Reconciler — vốn dĩ kiến trúc dựa trên call stack không có khái niệm "tạm dừng".

### Cách mới: Fiber Reconciler

React 16 thay toàn bộ reconciler bằng **Fiber** — không phải một feature, mà là một bản viết lại hoàn toàn cách React đi qua cây.

Ý tưởng cốt lõi: thay vì dùng call stack của JS, React tự dựng cấu trúc dữ liệu của riêng nó — một fiber node cho mỗi component/element, nối với nhau bằng con trỏ `child`, `sibling`, `return` (cha). Đây là một linked list có thể đi qua bằng loop thông thường, không phải bằng đệ quy — nghĩa là React có thể dừng giữa đường, lưu lại vị trí, nhường quyền điều khiển cho browser xử lý input/paint, rồi quay lại tiếp tục đúng vị trí đó.

```text
FiberNode {
  type        // component type hoặc tag DOM (vd: 'div', MyComponent)
  key         // identity hint từ developer
  pendingProps / memoizedProps
  child       // con đầu tiên
  sibling     // anh/em kế tiếp
  return      // fiber cha
  alternate   // con trỏ sang fiber tương ứng ở cây khác (current <-> work-in-progress)
  flags       // effect tag: Placement, Update, Deletion...
}
```
**Fiber links:**

```
          App
          |
        child
          ↓
       Header ───── sibling ────> Main
                                    |
                                  child
                                    ↓
                                 Button
React đi như sau:

1. App
   |
   v
2. Header
   |
   v
3. Main
   |
   v
4. Button

Sau Button: 
return → Main
return → App

```

---

**Fiber có thể pause**: Khi đang render mà Browser có việc, React có thể pause tại fiber node đang làm việc, tiến hành  lưu lại con trỏ `currentFiber`  biết đang ở node nào ─> Trả quyền cho Browser làm việc (xử lý input, paint UI, animation,scroll,...). Sau khi xong, Browser trả lại control cho React resume lại từ đúng node đang dở (`currentFiber`)

```
Fiber traversal:

App
 |
 ↓
Header
 |
 ↓
Main
 |
 ↓
Button


Có thể:

Button
  |
  |  ⏸ pause
  |
Browser


resume:

Button
  |
  ↓
Main
  |
  ↓
next work

```

Điều này giúp cho việc:
- Time Slicing: Chia nhỏ công việc render thành từng chunk ~5ms
- Priority Scheduling: Update từ user interaction (click, type) được ưu tiên cao hơn data fetch
- Concurrent Mode: 	React có thể "bắt đầu" một update, thấy update quan trọng hơn, "bỏ dở" và restart

`Fiber` không chỉ là refactor — nó là nền tảng cho mọi tính năng "concurrent" của `React` hiện đại.

Quan trọng: Fiber **không thay đổi thuật toán diff** (hai heuristic ở phần Core Concepts vẫn giữ nguyên). Fiber chỉ thay đổi **cách thực thi** thuật toán đó — biến nó từ một lần chạy đệ quy không thể ngắt, thành một work loop có thể ngắt, có ưu tiên, có thể hủy.



### Trade-offs

Fiber không miễn phí:

- Mỗi lần render giữ **hai cây** trong memory cùng lúc: cây `current` (đang hiển thị) và cây `work-in-progress` (đang dựng) — gấp đôi cấu trúc dữ liệu so với Stack Reconciler.
- Vì render phase có thể bị tạm dừng và chạy lại từ đầu, code trong component (và một số lifecycle cũ) có thể bị gọi nhiều hơn một lần cho cùng một update logic — đây là lý do `componentWillMount`, `componentWillReceiveProps`, `componentWillUpdate` bị đổi tên thành `UNSAFE_*`: chúng vốn được viết với giả định "chỉ chạy đúng một lần", giả định đó không còn đúng dưới Fiber.
- Độ phức tạp implementation tăng đáng kể — đây là lý do bạn hiếm khi thấy team tự viết reconciler riêng.

### Production Reality

Trong codebase thực tế bạn maintain, dấu hiệu code được viết cho/trong thời Stack Reconciler:

- Lifecycle method không có prefix `UNSAFE_` nhưng dùng để fetch data hoặc set state dựa trên props cũ (giả định chạy một lần).
- Comment kiểu "chạy 1 lần khi mount" gắn trên những API mà dưới Concurrent Mode không còn đảm bảo điều đó.

### Migration

Khi nâng cấp một codebase React 15 → 16+, reconciler tự động đổi — bạn không "migrate" Fiber theo nghĩa chỉnh code thủ công. Nhưng bạn phải audit mọi lifecycle method không an toàn, vì hành vi "đảm bảo chạy một lần" mà code cũ giả định không còn giữ nguyên 100%.

### Interview Perspective

Câu hỏi kinh điển: "Fiber là gì, và nó giải quyết vấn đề gì mà Stack Reconciler không giải quyết được?" 

Senior trả lời đúng phải tách rõ hai lớp: 
1. Thuật toán diff (heuristic, không đổi) 
2. Cơ chế thực thi (đổi hoàn toàn, từ call stack sang linked list ngắt được). 

Junior thường trả lời sai vì nhập nhằng hai lớp này.

## 3. Mental Model

Nắm chắc hai mô hình tư duy sau, mọi hành vi reconciliation đều suy ra được từ đây.

**Mô hình 1 — "Đủ nhanh, không phải tối ưu nhất".**
React không tìm cách biến đổi cây cũ → cây mới với số bước ít nhất tuyệt đối. React tìm cách trả lời câu hỏi đó trong O(n) bằng cách đặt giả định hợp lý, chấp nhận thỉnh thoảng làm dư một chút việc (ví dụ: move thừa một DOM node) để đổi lấy tốc độ tính toán.

**Mô hình 2 — "Identity = Type + Key + vị trí trong cây cha", không phải nội dung.**
React không so sánh "hai element này có giống nhau không" bằng deep-equal nội dung. Nó so sánh danh tính: cùng `type`, cùng `key`, cùng vị trí cấu trúc → React coi đó là "cùng một thứ, chỉ cập nhật props". Khác một trong ba yếu tố trên (đặc biệt là `type`) → React coi đó là "thứ khác hoàn toàn" và unmount/mount lại, bất kể nội dung props có giống nhau 99% hay không.

Đây là lý do một component ẩn đi (`condition && <A />` → `false`) rồi hiện lại không "nhớ" state cũ — với React, đó là hai identity khác nhau theo thời gian, dù code trông như "component A vẫn ở đó".

## 4. Core Concepts

### Hai giả định Heuristic làm nền cho toàn bộ thuật toán

1. **Hai element có `type` khác nhau sẽ tạo ra hai cây con khác nhau.** React không cố gắng diff bên trong — nó unmount toàn bộ cây con cũ, mount toàn bộ cây con mới. Không có ngoại lệ, bất kể props có giống nhau bao nhiêu.
2. **Developer có thể đánh dấu identity ổn định qua các lần render bằng `key`.** Không có key, React fallback về so sánh theo vị trí (index) trong danh sách con.

Hai giả định này biến bài toán O(n³) thành O(n) — đánh đổi: kết quả không phải lúc nào cũng là tập thay đổi tối thiểu tuyệt đối, nhưng đủ tốt cho gần như mọi UI thực tế.

### Diff một phần tử đơn (single child)

So sánh fiber cũ và element mới theo thứ tự:

```text
key cũ !== key mới        -> tạo fiber mới, fiber cũ bị đánh dấu Deletion
key giống, type khác      -> tạo fiber mới, fiber cũ bị đánh dấu Deletion
key giống, type giống     -> reuse fiber, chỉ clone + cập nhật props
```

### Diff danh sách (array reconciliation)

Đây là phần hầu hết bug `key` xảy ra. Thuật toán rút gọn:

1. Build một map `key -> fiber cũ` từ danh sách con cũ.
2. Lặp qua danh sách con mới theo thứ tự. Với mỗi phần tử, tra map bằng key (hoặc index nếu không có key).
3. Nếu tìm thấy fiber cũ cùng `key` và cùng `type` → reuse, update props.
4. React theo dõi một biến gọi là `lastPlacedIndex`. Nếu fiber được reuse nhưng vị trí trong danh sách mới nhỏ hơn `lastPlacedIndex` đã từng thấy, fiber đó bị đánh dấu `Placement` (cần di chuyển DOM node thật), dù state/props bên trong vẫn được giữ nguyên.
5. Phần tử mới không match được fiber cũ nào → tạo mới (`Placement`). Fiber cũ không có trong danh sách mới → `Deletion`.

Hệ quả thực tế quan trọng nhất: match-bằng-key chỉ quyết định "có giữ state/instance hay không" — nó **không** đảm bảo zero DOM operation khi thứ tự đổi. Một list bị sắp xếp lại hoàn toàn vẫn có thể sinh ra nhiều lệnh "move" DOM, dù mọi key đều đúng.

### Fiber Flags (Effect Tags)

Mỗi fiber sau khi qua render phase được gắn một hoặc nhiều flag mô tả việc cần làm ở commit phase: `Placement` (insert/move), `Update` (cập nhật thuộc tính/text), `Deletion` (gỡ khỏi DOM + chạy cleanup effect). Toàn bộ các flag này được gom thành một effect list, và commit phase chỉ việc đi qua list này một lượt, áp dụng tuần tự — đây là lý do commit phase nhanh và đồng bộ.

### Bailout — khi React không cần diff gì cả

Nếu cùng `type`, cùng `key`, và không có update nào được lên lịch cho fiber đó hoặc bất kỳ con của nó (`childLanes` rỗng) — React có thể bỏ qua hoàn toàn việc gọi lại function component, tái sử dụng luôn fiber con cũ. Đây gọi là `bailoutOnAlreadyFinishedWork`.

`React.memo`/`PureComponent` mở rộng thêm một lớp bailout: nếu props shallow-equal (so sánh tham chiếu từng prop), React bỏ qua re-render dù component cha đã re-render. Đây là lý do `useCallback`/`useMemo` "vô dụng" nếu bạn không bọc component con trong `memo` — không có memo, không có gì để bailout.

### Render Phase vs Commit Phase

Toàn bộ reconciliation (diff, tạo/tái sử dụng fiber, gắn flag) xảy ra ở render phase — phase này pure, không chạm DOM, có thể bị Scheduler tạm dừng/hủy/chạy lại. Commit phase đọc effect list đã build xong, áp dụng vào DOM thật một lần, đồng bộ, không thể ngắt — vì DOM không cho phép trạng thái "nửa cập nhật" hiển thị cho người dùng.

## 5. Legacy Recognition

Đọc một codebase và nhận ra ngay tư duy "thời chưa hiểu reconciliation":

- `items.map((item, index) => <Row key={index} ... />)` trong danh sách có thể filter/sort/thêm-xoá ở giữa — luôn là cờ đỏ.
- Hoàn toàn không có `key` (React sẽ warning console, nhưng warning bị ignore trong nhiều codebase legacy).
- Định nghĩa component bên trong function/render của component khác:

```jsx
function Page() {
  // Anti-pattern: type của Header() là một function reference MỚI mỗi lần Page render
  function Header() {
    return <h1>Dashboard</h1>;
  }
  return <Header />;
}
```

Mỗi lần `Page` render lại, `Header` là một reference khác về mặt JS → React áp dụng giả định #1 ("type khác nhau = cây khác nhau") → unmount + mount lại toàn bộ `Header`, mất state, mất CSS transition, chạy lại effect mount.

- Lifecycle `componentWillReceiveProps`/`componentWillUpdate` không có prefix `UNSAFE_` dùng để side-effect dựa trên giả định "chỉ chạy một lần mỗi update" — dấu hiệu code viết cho Stack Reconciler, có thể vỡ dưới Concurrent features.

## 6. Modern Approach

- **Key = identity ổn định của dữ liệu**, không phải vị trí hiển thị. Dùng id thật từ database/API, không dùng index — đặc biệt khi danh sách có thể sort, filter, thêm/xoá giữa danh sách.
- **Tách component item ra top-level module scope**, không định nghĩa inline trong component cha — giữ `type` ổn định qua các lần render để React tận dụng được bailout.
- **`React.memo` đi kèm props ổn định** (hằng số đưa ra ngoài render, callback bọc `useCallback` nếu truyền xuống component đã `memo`) — nếu không, memo không có tác dụng vì props luôn "mới" về tham chiếu.
- **Dùng `key` có chủ đích để force remount** khi bạn thực sự muốn reset toàn bộ state của một subtree (ví dụ đổi `key={selectedRecordId}` cho một form chỉnh sửa khi người dùng chuyển sang record khác) — đây là kỹ thuật hợp lệ, không phải anti-pattern, miễn là bạn hiểu chi phí đi kèm (xem mục Architecture Notes).

## 7. Production Scenario

Bối cảnh: bảng danh sách hợp đồng thuê trong dashboard quản lý — mỗi dòng có một ô input để nhập "ghi chú nội bộ" (inline-edit), bảng có thể sắp xếp theo trạng thái thanh toán.

```jsx
function ContractTable({ contracts }) {
  return (
    <table>
      <tbody>
        {contracts.map((contract, index) => (
          // Bug ẩn: key={index} trong bảng có thể bị sort lại
          <ContractRow key={index} contract={contract} />
        ))}
      </tbody>
    </table>
  );
}

function ContractRow({ contract }) {
  return (
    <tr>
      <td>{contract.tenantName}</td>
      <td>
        <input defaultValue={contract.note} />
      </td>
    </tr>
  );
}
```

Người dùng gõ ghi chú vào dòng của hợp đồng A (đang ở vị trí số 2), sau đó bấm "Sắp xếp theo trạng thái" — hợp đồng A bị đẩy xuống vị trí số 5. Ghi chú họ vừa gõ xuất hiện ở dòng của hợp đồng đang giờ nằm ở vị trí số 2 (một hợp đồng khác), không đi theo hợp đồng A.

Nguyên nhân: vì `key={index}`, React match fiber theo vị trí, không theo bản ghi. DOM `<input>` ở vị trí số 2 được reuse (cùng key, cùng type) thay vì bị huỷ/tạo mới — nhưng `defaultValue` là uncontrolled, giá trị nội tại của input nằm trong DOM node, không nằm trong React state. React reuse đúng DOM node, sai dữ liệu logic.

## 8. Debugging Scenario

Bug report từ QA: "Ghi chú tôi gõ vào hợp đồng bị nhảy sang dòng khác sau khi sort."

Quy trình debug đúng:

1. Mở React DevTools → tab Components → bật "Highlight updates when components render". Sort lại bảng, quan sát: toàn bộ `ContractRow` highlight là Update, không có row nào highlight Placement/mount mới — dấu hiệu React đang reuse fiber sai chỗ.
2. Mở tab Profiler, record một lần sort, xem flame graph: số lượng `ContractRow` re-render đúng bằng số dòng, nhưng không có row nào unmount dù dữ liệu đã đổi vị trí hoàn toàn — củng cố nghi vấn key sai.
3. Kiểm tra source: thấy `key={index}`. Xác nhận giả thuyết.
4. Sửa: `key={contract.id}`, chuyển `<input defaultValue>` thành controlled với state lưu theo `contract.id`, hoặc giữ uncontrolled nhưng đảm bảo key đúng để React mount/unmount đúng input khi vị trí đổi.

## 9. Refactoring Scenario

**Trước:**

```jsx
function ContractTable({ contracts }) {
  return (
    <table>
      <tbody>
        {contracts.map((contract, index) => (
          <tr key={index}>
            <td>{contract.tenantName}</td>
            <td><input defaultValue={contract.note} /></td>
          </tr>
        ))}
      </tbody>
    </table>
  );
}
```

**Sau:**

```jsx
const ContractRow = memo(function ContractRow({ contract, onNoteBlur }) {
  return (
    <tr>
      <td>{contract.tenantName}</td>
      <td>
        <input
          defaultValue={contract.note}
          onBlur={(e) => onNoteBlur(contract.id, e.target.value)}
        />
      </td>
    </tr>
  );
});

function ContractTable({ contracts, onNoteBlur }) {
  return (
    <table>
      <tbody>
        {contracts.map((contract) => (
          // Key ổn định theo identity dữ liệu, không theo vị trí render
          <ContractRow key={contract.id} contract={contract} onNoteBlur={onNoteBlur} />
        ))}
      </tbody>
    </table>
  );
}
```

Thay đổi thực hiện: (1) key đổi từ index sang id thật — sửa đúng root cause; (2) tách `ContractRow` ra top-level — `type` ổn định, đủ điều kiện bailout qua `memo`; (3) callback `onNoteBlur` nên được `useCallback` ở component cha nếu muốn `memo` phát huy tác dụng đầy đủ.

## 10. Architecture Notes

- Reconciliation, dù là O(n), vẫn tỉ lệ với số node thực sự được render — không phải số node trong dữ liệu. Đây là lý do danh sách hàng chục nghìn dòng cần virtualization (chỉ render node trong viewport): giảm n trong O(n), không phải tối ưu thuật toán diff.
- Đặt state ở đúng tầng thấp nhất cần nó. Lift state lên quá cao khiến mỗi update kéo theo reconciliation của cả subtree lớn hơn cần thiết — đây là vấn đề kiến trúc, không phải vấn đề thuật toán diff.
- Trick "đổi `key` để force remount" hữu ích nhưng không miễn phí: nó hủy toàn bộ DOM node cũ (mất focus, mất scroll position, mất CSS transition đang chạy, chạy lại mọi effect mount) — chỉ dùng khi reset hoàn toàn là hành vi mong muốn, không dùng như cách "tắt nhanh" một bug state.
- Hiểu rõ ranh giới render phase (interruptible) vs commit phase (atomic) là nền tảng bắt buộc trước khi học Suspense (M1.4), Concurrent Rendering (M1.5) và đặc biệt Hydration (M1.6) — nơi cây fiber phía client phải match với markup tĩnh do server gửi xuống, và sai lệch `type`/`key` giữa hai bên chính là nguồn gốc phổ biến nhất của hydration mismatch.

## 11. Interview Questions

**[Junior]** Tại sao React cần `key` khi render danh sách? Điều gì xảy ra nếu bỏ key hoặc dùng index?

<details>
<summary>Đáp án</summary>
React cần một cách xác định "identity" của từng phần tử trong danh sách qua các lần render, để biết phần tử nào giữ nguyên, phần tử nào mới, phần tử nào bị xoá.

Không có `key`, React fallback về so sánh theo vị trí (index) — nếu danh sách chỉ append/giữ thứ tự cố định thì không vấn đề, nhưng nếu có sort/filter/insert giữa danh sách, React sẽ match sai phần tử theo vị trí, dẫn đến tái sử dụng nhầm DOM node/state giữa các bản ghi khác nhau, đặc biệt nguy hiểm với uncontrolled input, animation, hoặc component có state nội bộ.
</details>

---

**[Mid]** Phân biệt rõ: thuật toán diff của React thay đổi gì, và không thay đổi gì khi chuyển từ Stack Reconciler sang Fiber Reconciler?

<details>
<summary>Đáp án</summary>
Hai heuristic diff cốt lõi (type khác nhau = cây khác nhau; key xác định identity ổn định) không đổi — đây là phần thuộc về "thuật toán". 

Cái thay đổi là cơ chế thực thi: Stack Reconciler dùng call stack JS, đệ quy, đồng bộ, không thể dừng giữa đường; 

Fiber dùng cấu trúc linked-list tự quản lý, đi qua bằng loop, có thể dừng/nhường quyền điều khiển/tiếp tục đúng vị trí, làm nền cho việc ưu tiên update và các tính năng concurrent sau này.
</details>

---

**[Senior]** Giải thích cơ chế "bailout" trong reconciliation. Khi nào React bỏ qua hoàn toàn việc gọi lại function component?

<details>
<summary>Đáp án</summary>

Bailout xảy ra khi fiber giữ nguyên type và key, và không có bất kỳ update nào được lên lịch cho fiber đó hoặc toàn bộ cây con của nó (childLanes rỗng) — React tái sử dụng trực tiếp con trỏ child cũ, không gọi lại function component, không diff lại children. 

`React.memo/PureComponent` mở thêm một lớp bailout dựa trên so sánh shallow props: nếu props không đổi tham chiếu (hoặc comparator trả về true), component bị skip re-render dù component cha đã render lại — điều kiện tiên quyết là props thực sự ổn định về tham chiếu, nếu không memo không có tác dụng.
</details>

---

**[Senior/Staff]** Tại sao thay đổi `key` của một subtree được dùng như kỹ thuật force-remount? Trade-off của kỹ thuật này so với việc viết logic reset thủ công là gì?

<details>
<summary>Đáp án</summary>

Theo giả định #2, key là tín hiệu identity do developer cung cấp — đổi key nghĩa là báo cho React "đây là một identity khác", bất kể type và props bên trong giống nhau đến đâu.

React áp dụng đúng quy tắc diff phần tử đơn: 

>key khác → unmount fiber cũ (state, ref, DOM node, effect cleanup chạy) → mount fiber mới hoàn toàn (effect mount chạy lại từ đầu). 

**Trade-off**: cách này đơn giản, ít code, nhưng phá huỷ hoàn toàn DOM cũ — mất scroll position, mất focus, ngắt CSS transition đang chạy, và chạy lại toàn bộ side-effect mount (có thể tốn kém nếu effect đó gọi API). Reset thủ công (set lại từng field trong state) giữ được DOM node và các giá trị không cần reset, nhưng tốn nhiều code hơn và dễ sót field nếu state phức tạp.
</details>

---

**[Architect]** Thuật toán diff O(n) heuristic của React đôi khi tạo ra nhiều DOM operation hơn một thuật toán tối ưu lý thuyết O(n³). Đây có phải vấn đề thực tế trong production không?

<details>
<summary>Đáp án</summary>
Về lý thuyết đúng — heuristic đổi tính tối ưu tuyệt đối để lấy tốc độ tính toán O(n), nên đôi khi sinh ra nhiều lệnh "move" DOM hơn mức tối thiểu lý thuyết.

Trong thực tế đây hầu như không phải vấn đề, vì hai lý do: 
- (1) chi phí tính toán thuật toán O(n³) trên cây vài nghìn node sẽ chậm hơn nhiều lần so với phần DOM operation "dư" mà heuristic O(n) tạo ra; 
- (2) chi phí thật của một DOM mutation (reflow, repaint) thường lớn hơn nhiều so với chi phí của chính lệnh move đó — vài lệnh move dư không đáng kể so với việc tránh được một thuật toán cubic chạy trên mọi keystroke.

Vấn đề thực sự trong production gần như luôn nằm ở key sai (gây mount/unmount thừa) hơn là ở việc heuristic không tối ưu tuyệt đối.
</details>

**Bài 1.** Đoạn code dưới đây dùng cho danh sách task có thể kéo-thả (drag & drop) để đổi thứ tự. Xác định bug liên quan đến reconciliation và sửa lại.

```jsx
function TaskList({ tasks }) {
  return (
    <ul>
      {tasks.map((task, index) => (
        <li key={index}>
          <input type="checkbox" defaultChecked={task.done} />
          {task.title}
        </li>
      ))}
    </ul>
  );
}
```

<details>
<summary>Đáp án</summary>

Bug: `key={index}` kết hợp với checkbox uncontrolled (defaultChecked) trong danh sách có thể đổi thứ tự (drag & drop). 

Sau khi kéo thả, React match checkbox theo vị trí, không theo task — trạng thái checked hiển thị có thể thuộc về task khác với task thực sự đang ở vị trí đó.   

Sửa bằng cách dùng `key={task.id}` (id ổn định từ dữ liệu, không phụ thuộc thứ tự hiển thị), đảm bảo mỗi li/input luôn gắn đúng với task của nó bất kể vị trí trong danh sách thay đổi thế nào.
</details>

**Bài 2.** Với mỗi cặp render dưới đây, cho biết React sẽ "update tại chỗ" hay "unmount + mount lại", và giải thích bằng đúng giả định heuristic liên quan.

- a) `<UserCard key="u1" />` → `<UserCard key="u1" />` (props khác nhau)
- b) `<UserCard key="u1" />` → `<AdminCard key="u1" />`
- c) `{showCard && <UserCard />}` chuyển từ `true` sang `false` rồi lại `true`

<details>
<summary>Đáp án</summary>

- a) Update tại chỗ. Cùng type (`UserCard`), cùng key (`u1`) — theo giả định #1 và #2, React coi đây là cùng một identity, chỉ cập nhật props, fiber/state/DOM node được giữ nguyên.
- b) Unmount + mount lại. Type đổi từ `UserCard` sang `AdminCard` — bất kể key giống nhau, giả định #1 áp dụng trước: type khác nhau nghĩa là cây con khác nhau, không có nỗ lực diff bên trong, fiber cũ bị huỷ hoàn toàn, fiber mới được tạo từ đầu.
- c) Unmount khi chuyển `false`, mount mới hoàn toàn khi chuyển lại `true`. Khi điều kiện `false`, không có element nào được render ở vị trí đó — về mặt cây, `UserCard` biến mất hoàn toàn (không phải "ẩn"). Khi điều kiện `true` trở lại, React tạo một fiber hoàn toàn mới ở vị trí đó — không có cách nào "nhớ" lại state cũ từ identity đã bị huỷ trước đó.
</details>

**Bài 3.** Component dưới đây bị remount không mong muốn mỗi khi `Dashboard` re-render (mất state form, mất focus input). Xác định nguyên nhân và refactor.

```jsx
function Dashboard({ user }) {
  const Sidebar = () => <nav>{user.role}</nav>;

  return (
    <div>
      <Sidebar />
      <SettingsForm />
    </div>
  );
}
```

<details>
<summary>Đáp án</summary>

- **Nguyên nhân**: `Sidebar` được định nghĩa lại bên trong thân hàm `Dashboard`, nên mỗi lần `Dashboard` render, `Sidebar` là một function reference hoàn toàn mới về mặt JS — với React, đó là một type khác so với lần render trước, dù tên và nội dung giống nhau. 

Theo giả định #1, type khác nhau buộc React unmount fiber `Sidebar` cũ và mount fiber mới, kéo theo mọi state/effect bên trong `Sidebar` và tham chiếu DOM bị tạo lại. 

- **Refactor**: đưa `Sidebar` ra ngoài, định nghĩa ở top-level module scope, để type của nó ổn định qua mọi lần `Dashboard` render.

```jsx
function Sidebar({ role }) {
  return <nav>{role}</nav>;
}

function Dashboard({ user }) {
  return (
    <div>
      <Sidebar role={user.role} />
      <SettingsForm />
    </div>
  );
}
```
</details>

## 13. Summary

- Reconciliation giải bài toán tree-diff bằng heuristic O(n), đánh đổi tính tối ưu tuyệt đối để đạt tốc độ thực thi được trong thời gian thực.
- Hai giả định nền: type khác nhau = cây khác nhau (không ngoại lệ); key là tín hiệu identity do developer khai báo, fallback về index nếu thiếu.
- Fiber không đổi thuật toán diff — nó đổi cơ chế thực thi, từ call stack đồng bộ không thể ngắt sang linked-list có thể dừng/ưu tiên/abort.
- Bailout cho phép React skip hoàn toàn việc re-render một subtree khi type/key không đổi và không có update nào được lên lịch — điều kiện để memo/useCallback/useMemo thực sự có tác dụng.
- Phần lớn bug "key" trong production không phải lỗi cú pháp, mà là lỗi mô hình tư duy: nhầm "vị trí hiển thị" với "identity dữ liệu".
- Hiểu rõ render phase (interruptible, pure) vs commit phase (atomic, áp dụng DOM) là nền tảng bắt buộc cho Suspense, Concurrent Rendering, và Hydration ở các bài tiếp theo.
