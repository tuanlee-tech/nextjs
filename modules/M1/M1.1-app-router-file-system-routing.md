# M1.1: App Router & File-System Routing

> 🎯 **Lesson Goal:** Tạo routes với file-based routing, phân biệt `page.tsx` vs `layout.tsx`, implement dynamic segments, và giải thích tại sao App Router không cần `_app.tsx`.

---

## 1. Evolution Context

Tôi từng inherit một codebase Pages Router với 40+ routes, `_app.tsx` dài 380 dòng, lồng 6 HOC (`withAuth`, `withTheme`, `withAnalytics`, `withFeatureFlag`, `withErrorBoundary`, `withLayout`). Thêm 1 page mới, bạn phải nhớ page đó cần wrap bởi HOC nào, theo thứ tự nào — sai thứ tự không throw error, chỉ là behavior sai âm thầm. Đó là lý do team quyết định migrate.

**Root problem:** Khi app phình to, việc chia sẻ UI (nav, sidebar) và maintain state giữa các routes trong Pages Router trở thành "Wrapper Hell".

**Cách cũ:** `pages/_app.tsx` kết hợp hàng loạt HOCs (Higher-Order Components) để inject layout.

**Tại sao thất bại:**
- Không support nested layouts tự nhiên — muốn `/dashboard/*` có layout khác `/marketing/*`, bạn phải tự `if (router.pathname.startsWith('/dashboard'))` ngay trong `_app.tsx`. Số nhánh if/else tăng theo số route group.
- Mỗi lần chuyển route, toàn bộ tree dưới `_app.tsx` re-mount. Sidebar unmount/remount, state mất sạch (scroll position, form input đang gõ, websocket connection).
- `_app.tsx` trở thành god-file — single point of contention trong PR review, hầu hết feature đều phải touch file này, conflict merge liên tục.

**Timeline:** Pages Router (flat routing, `_app` wrapper) → Next.js 13 App Router (Nested folders, `layout.tsx` native, Server Components default) → Next.js 16 (`params` trở thành async, chuẩn hóa Request API).

Bài này tập trung vào cách App Router giải quyết "Wrapper Hell" bằng nested layout theo file system — không cần conditional logic, không cần HOC, layout tự "biết" áp dụng cho route nào dựa trên vị trí folder.

## 2. Mental Model

> 💡 **[MODEL]** `page.tsx` phải được coi là "route entry point" (thin wrapper), KHÔNG phải nơi định nghĩa component logic chính.

> 🏠 **[ANALOGY]** `layout.tsx` giống như khung nhà — khi bạn di chuyển giữa các phòng (routes), khung nhà vẫn đứng yên, chỉ nội thất thay đổi. `template.tsx` giống như dỡ khung nhà và dựng lại mỗi lần vào phòng mới.

Tại sao mental model này quan trọng: nếu bạn coi `page.tsx` là nơi "viết code chính", bạn sẽ rơi vào AP1 (mục 4) — 500 dòng dồn vào 1 file, không tách được để re-use ở `loading.tsx`, không test độc lập được. Nếu bạn coi `layout.tsx` re-mount mỗi lần navigate (giống `template.tsx`), bạn sẽ expect sai behavior và debug sai hướng khi gặp Scenario 1 (mục 8) — tưởng app bug, thật ra đó là feature.

Ghi nhớ ngắn: `page.tsx` trả lời "URL này render gì?". `layout.tsx` trả lời "UI nào persistent qua nhiều URL?". Hai câu hỏi khác nhau — đừng trộn logic của câu hỏi này vào file trả lời câu hỏi kia.

## 3. Core Concepts

### 3.1 Reserved Filenames & Role

**[FACT]** App Router dùng convention-based file naming — Next.js reserve một danh sách filename cố định trong `app/`, mỗi file giữ đúng 1 role, không dùng lẫn cho nhau. (Source: `nextjs.org/docs/app/building-your-application/routing`)

| Filename | Role | Chi tiết |
| :--- | :--- | :--- |
| `page.tsx` | UI riêng cho route | Bắt buộc để folder trở thành 1 route accessible. |
| `layout.tsx` | UI dùng chung, persistent | Wrap `page.tsx` và nested routes con, không re-render khi navigate. |
| `loading.tsx` | Suspense fallback | Hiện khi route đang fetch/stream. |
| `error.tsx` | Error Boundary | Cho route segment đó — mechanics chi tiết ở **M1.3**, bài này chỉ cần biết file này tồn tại. |
| `not-found.tsx` | UI 404 | Hiện khi gọi `notFound()` hoặc route không match. |
| `route.ts` | Route Handler (API) | Chi tiết ở **M2.3** — bài này chỉ cần biết nó conflict với `page.tsx` nếu đặt cùng level. |
| `template.tsx` | Persistent UI, re-mount | Giống `layout.tsx` nhưng re-mount mỗi navigation. |
| `default.tsx` | Fallback cho Parallel Routes slot | Khi không có route con match slot đó. |

```ts [src/app/dashboard/page.tsx]
// [FACT] page.tsx phải export default 1 React component — đây là route
// entry point, KHÔNG phải nơi viết logic chính (xem Mental Model, mục 2).
// Source: nextjs.org/docs/app/building-your-application/routing/pages
import { DashboardOverview } from "@/components/dashboard/dashboard-overview";

export default function DashboardPage() {
  // page.tsx ngắn gọn như thế này giúp: (1) reviewer đọc PR nhanh,
  // (2) DashboardOverview re-use được trong loading.tsx skeleton,
  // (3) test component độc lập không cần mount route tree.
  return <DashboardOverview />;
}
```

Quên `export default` không throw error rõ ràng ngay — route render ra trang trắng hoặc 404 tùy version. Đây là một trong root cause của Scenario 3 (mục 8).

### 3.2 Nested Layouts & State Preservation

**[FACT]** `layout.tsx` không re-render khi navigate giữa các children. State và DOM được preserve. (Source: `nextjs.org/docs/app/building-your-application/routing/layouts-and-templates#layouts-do-not-re-render`)

**[FACT]** Root layout (`app/layout.tsx`) bắt buộc phải định nghĩa `<html>` và `<body>`. Nested layout KHÔNG được chứa 2 tag này. (Source: `nextjs.org/docs/app/building-your-application/routing/layouts-and-templates#root-layout-required`)

Đây là khác biệt lớn nhất so với Pages Router. `app/layout.tsx` (root) và `app/dashboard/layout.tsx` (nested) tạo thành 1 cây — khi user navigate `/dashboard/orders` → `/dashboard/profile`, chỉ phần `page.tsx` bên trong unmount/mount, `app/dashboard/layout.tsx` (sidebar, header) giữ nguyên DOM, giữ nguyên React state.

```ts [src/app/dashboard/layout.tsx]
import { DashboardSidebar } from "@/components/dashboard/dashboard-sidebar";
import { DashboardHeader } from "@/components/dashboard/dashboard-header";

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  // Sidebar và Header ở đây KHÔNG re-mount khi user click giữa
  // /dashboard/orders <-> /dashboard/profile. Nếu bạn fetch data ở đây
  // (VD: unread notification count), nó CHỈ fetch 1 lần khi user vào
  // /dashboard/* lần đầu — không tự re-fetch khi đổi tab con.
  // Đây là điểm production engineer hay miss (xem Scenario 1, mục 8).
  return (
    <div className="dashboard-shell">
      <DashboardSidebar />
      <div className="dashboard-main">
        <DashboardHeader />
        {children}
      </div>
    </div>
  );
}
```

### 3.3 Dynamic Routes

**[FACT]** Cú pháp dynamic routes: `[slug]` (1 segment), `[...slug]` (catch-all, bắt buộc ≥1 segment), `[[...slug]]` (optional catch-all, chấp nhận 0 segment). (Source: `nextjs.org/docs/app/building-your-application/routing/dynamic-routes`)

**[FACT]** Next.js 16 thay đổi `params` từ object sync sang Promise. Bắt buộc phải `await params` trong `page.tsx` hoặc `layout.tsx`. (Source: Next.js 16 Upgrade Guide / Async Request APIs)

```ts [src/app/posts/[id]/page.tsx]
// params là Promise trong Next.js 16 — component phải là async function
// để await nó. Đây là breaking change lớn nhất khi upgrade từ v15.
import { getPostById } from "@/lib/posts";
import { logger } from "@/lib/logger";
import { notFound } from "next/navigation";

interface PostPageProps {
  params: Promise<{ id: string }>;
}

export default async function PostPage({ params }: PostPageProps) {
  // KHÔNG destructure sync (const { id } = params) — sẽ throw runtime
  // error trong Next.js 16.
  const { id } = await params;

  try {
    const post = await getPostById(id);
    if (!post) {
      // notFound() throw 1 internal "exception" được Next.js catch —
      // không phải lỗi thật, đừng log nó như error.
      notFound();
    }
    return <article>{post.title}</article>;
  } catch (error) {
    // Structured logging — production cần biết postId nào gây lỗi,
    // không chỉ "something went wrong".
    logger.error({ err: error, postId: id }, "Failed to load post detail");
    throw error; // re-throw để error.tsx (chi tiết ở M1.3) xử lý UI, không swallow.
  }
}
```

```text
app/
├── posts/[id]/page.tsx           // /posts/123       -> { id: "123" }
├── docs/[...slug]/page.tsx       // /docs/a/b/c      -> { slug: ["a","b","c"] }
│                                  // /docs            -> 404 (catch-all cần >=1 segment)
└── shop/[[...filters]]/page.tsx  // /shop            -> { filters: undefined }
                                   // /shop/men/shoes  -> { filters: ["men","shoes"] }
```

### 3.4 Route Groups & Private Folders

**[FACT]** Route Groups `(groupName)` loại bỏ folder khỏi URL path nhưng vẫn giữ cấu trúc file system. (Source: `nextjs.org/docs/app/building-your-application/routing/route-groups`)

**[FACT]** Private folders `_folderName` bị loại bỏ hoàn toàn khỏi hệ thống routing, kể cả khi bên trong có `page.tsx`. (Source: `nextjs.org/docs/app/building-your-application/routing/colocation`)

```text
app/
├── (marketing)/
│   ├── layout.tsx        // layout riêng cho marketing pages
│   ├── about/page.tsx    // URL: /about (KHÔNG phải /marketing/about)
│   └── contact/page.tsx
├── (shop)/
│   ├── layout.tsx        // layout khác, riêng cho shop pages
│   └── products/page.tsx
└── dashboard/
    ├── _components/
    │   └── widgets/page.tsx   // PRIVATE — Next.js bỏ qua hoàn toàn,
    │                           // dù bên trong có page.tsx, route
    │                           // /dashboard/widgets KHÔNG được tạo.
    └── page.tsx
```

Dùng Route Groups khi bạn cần 2 layout khác nhau cho 2 nhóm route nhưng URL không có prefix tương ứng. Dùng Private folders để colocate component/hook/util gần route mà không sợ Next.js biến nó thành route ngoài ý muốn.

### 3.5 Parallel Routes & Intercepting Routes (Conceptual Only)

Bài này chỉ cần bạn biết 2 khái niệm này tồn tại — Next.js cho phép render nhiều `page.tsx` song song trong cùng layout qua slot `@folder`, và "chặn" route để hiện modal qua convention `(.)folder`. Chi tiết implementation, edge cases, và behavior đầy đủ của `default.tsx` cho slot thuộc về **M8.1** — bài này không đi sâu hơn ví dụ cấu trúc dưới đây.

```ts [src/app/dashboard/@notifications/default.tsx]
// [NOTE] Chỉ minh họa cấu trúc tồn tại, KHÔNG đi sâu — chi tiết ở M8.1.
// default.tsx là fallback khi Next.js không tìm route con match
// cho slot @notifications (VD: lần đầu load, hoặc soft navigation
// không update slot này).
export default function DefaultNotificationSlot() {
  return null;
}
```

## 4. Anti-Patterns in the Wild

### AP1: Đặt component vào `page.tsx` trực tiếp thay vì tách file

- ❌ **Code Smell:** File `page.tsx` dài 500 dòng chứa toàn bộ UI và logic.

  ```ts [src/app/dashboard/page.tsx]
  "use client";
  import { useEffect, useState } from "react";

  export default function DashboardPage() {
    const [orders, setOrders] = useState([]);
    const [profile, setProfile] = useState(null);

    useEffect(() => {
      // ... 50 dòng fetch logic, validate, sort, filter dồn ở đây
    }, []);

    // ... 400 dòng JSX cho sidebar, header, chart, table
    // Khi cần skeleton cho loading.tsx, bạn phải copy-paste lại gần
    // như toàn bộ structure này — không có gì để re-use.
    return <div>{/* ... */}</div>;
  }
  ```

- ✅ **Staff Refactor:** `app/dashboard/page.tsx` chỉ export default thin wrapper. Logic nằm ở `components/dashboard/`.

  ```ts [src/app/dashboard/page.tsx]
  // Trade-off: Tăng số lượng file đổi lại dễ re-use component ở
  // loading.tsx (skeleton) hoặc test độc lập (render DashboardOverview
  // trong Jest/RTL mà không cần mount cả route tree).
  import { DashboardOverview } from "@/components/dashboard/dashboard-overview";

  export default function DashboardPage() {
    return <DashboardOverview />;
  }
  ```

- 🧠 **Why:** Tăng số lượng file đổi lại dễ re-use component ở `loading.tsx` (skeleton) hoặc test độc lập.
- 🔥 **Incident Risk:** Trùng lặp code khi cần dựng skeleton UI cho `loading.tsx`.

### AP2: Quên `'use client'` rồi dùng `useState` trong Server Component

- ❌ **Code Smell:** Thêm `useState` vào `page.tsx` (Server Component default) mà không khai báo directive.

  ```ts [src/app/settings/page.tsx]
  // Server Component KHÔNG support React hooks — build fail hoặc
  // hydration crash, tùy việc bundler optimize ra sao.
  import { useState } from "react";

  export default function SettingsPage() {
    const [tab, setTab] = useState("profile");
    return (
      <div>
        <button onClick={() => setTab("profile")}>Profile</button>
        <button onClick={() => setTab("security")}>Security</button>
      </div>
    );
  }
  ```

- ✅ **Staff Refactor:** Tách phần có `useState` ra một Client Component riêng, import vào `page.tsx`.

  ```ts [src/app/settings/page.tsx]
  import { SettingsTabs } from "@/components/settings/settings-tabs";

  export default function SettingsPage() {
    // page.tsx vẫn là Server Component — chỉ phần cần interactivity
    // (tab switching) mới xuống client.
    return <SettingsTabs />;
  }
  ```

  ```ts [src/components/settings/settings-tabs.tsx]
  "use client";
  // Trade-off: Mất Server Component benefits (auto streaming) ở
  // component này đổi lại chạy được onClick/useState trên browser.
  import { useState } from "react";

  export function SettingsTabs() {
    const [tab, setTab] = useState<"profile" | "security">("profile");
    return (
      <div>
        <button onClick={() => setTab("profile")}>Profile</button>
        <button onClick={() => setTab("security")}>Security</button>
        {tab === "profile" ? <div>Profile form</div> : <div>Security form</div>}
      </div>
    );
  }
  ```

- 🧠 **Why:** Mất Server Component benefits (auto streaming) ở component đó đổi lại chạy được trên client.
- 🔥 **Incident Risk:** Build error `useState is not exported from 'react'` hoặc runtime hydration crash.

### AP3: Dùng `template.tsx` khi chỉ cần `layout.tsx`

- ❌ **Code Smell:** Copy toàn bộ code từ `layout.tsx` sang `template.tsx` nghĩ rằng nó giống nhau.

  ```ts [src/app/dashboard/template.tsx]
  // Junior nghĩ "template" với "layout" chỉ khác tên file, copy nguyên
  // logic sang đây — không biết rằng file này re-mount mỗi navigation.
  import { DashboardSidebar } from "@/components/dashboard/dashboard-sidebar";

  export default function DashboardTemplate({
    children,
  }: {
    children: React.ReactNode;
  }) {
    return (
      <div>
        <DashboardSidebar />
        {children}
      </div>
    );
  }
  ```

- ✅ **Staff Refactor:** Xóa `template.tsx`, giữ `layout.tsx`.

  ```ts [src/app/dashboard/layout.tsx]
  // Trade-off: Không có trade-off positive ở case này — file
  // template.tsx xóa hoàn toàn, dùng layout.tsx duy nhất.
  import { DashboardSidebar } from "@/components/dashboard/dashboard-sidebar";

  export default function DashboardLayout({
    children,
  }: {
    children: React.ReactNode;
  }) {
    return (
      <div>
        <DashboardSidebar />
        {children}
      </div>
    );
  }
  ```

- 🧠 **Why:** `template.tsx` re-mount mỗi navigation — layout thì không. Chỉ dùng `template.tsx` khi bạn thật sự cần re-mount effects (enter/exit animation, reset 1 form theo URL change).
- 🔥 **Incident Risk:** State bị reset (re-mount) mỗi lần click navigate, UX tệ (form input bị clear, video bị reload).

### AP4: `params` sync access trong Next.js 16

- ❌ **Code Smell:** `const { id } = params` khi upgrade lên Next 16.

  ```ts [src/app/posts/[id]/page.tsx]
  // Đúng ở Next.js 15, crash ở Next.js 16 vì params giờ là Promise.
  export default function PostPage({ params }: { params: { id: string } }) {
    const { id } = params;
    return <div>Post {id}</div>;
  }
  ```

- ✅ **Staff Refactor:** Chạy codemod `@next/codemod@canary upgrade` hoặc manual sửa thành `const { id } = await params`. Component phải là `async`.

  ```ts [src/app/posts/[id]/page.tsx]
  // Trade-off: Sinh ra nhiều async component, trace code phức tạp hơn
  // (phải await ở mọi nơi dùng params), nhưng đây là direction chính
  // thức của framework — không có lựa chọn "giữ sync" lâu dài.
  interface PostPageProps {
    params: Promise<{ id: string }>;
  }

  export default async function PostPage({ params }: PostPageProps) {
    const { id } = await params;
    return <div>Post {id}</div>;
  }
  ```

- 🧠 **Why:** Chuẩn hóa Async Request API (đồng nhất với `searchParams`), hỗ trợ streaming dưới mức route segment, tương thích Edge Runtime.
- 🔥 **Incident Risk:** App crash 500 ngay lập tức với lỗi `TypeError: Cannot destructure property 'id' of 'params' as it is undefined`.

### AP5: Thêm `'use client'` ở layout

- ❌ **Code Smell:** Thêm `'use client'` vào `app/layout.tsx` để dùng `useState` cho theme toggle.

  ```ts [src/app/layout.tsx]
  "use client"; // Junior thêm vào đây để dùng useState — kéo TOÀN BỘ
  // children xuống client bundle theo, dù children không cần.
  import { useState } from "react";

  export default function RootLayout({ children }: { children: React.ReactNode }) {
    const [theme, setTheme] = useState("light");
    return (
      <html lang="en" data-theme={theme}>
        <body>
          <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
            Toggle theme
          </button>
          {children}
        </body>
      </html>
    );
  }
  ```

- ✅ **Staff Refactor:** Tạo `<ThemeToggleProvider>` là Client Component, bọc nó BÊN TRONG `layout.tsx` (Server Component).

  ```ts [src/app/layout.tsx]
  // Trade-off: Tốn thêm 1 file wrapper, nhưng root layout giữ được
  // Server Component — children KHÔNG bị force client bundle.
  import { ThemeToggleProvider } from "@/components/theme/theme-toggle-provider";

  export default function RootLayout({ children }: { children: React.ReactNode }) {
    return (
      <html lang="en">
        <body>
          <ThemeToggleProvider>{children}</ThemeToggleProvider>
        </body>
      </html>
    );
  }
  ```

  ```ts [src/components/theme/theme-toggle-provider.tsx]
  "use client";
  import { useState } from "react";

  export function ThemeToggleProvider({ children }: { children: React.ReactNode }) {
    const [theme, setTheme] = useState<"light" | "dark">("light");
    return (
      <div data-theme={theme}>
        <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
          Toggle theme
        </button>
        {children}
      </div>
    );
  }
  ```

- 🧠 **Why:** Layout nên là Server Component. Để client logic ở component con — chỉ subtree thật sự cần interactivity mới xuống client bundle.
- 🔥 **Incident Risk:** Mất toàn bộ lợi ích của Server Components cho subtree, tất cả children bị force thành client bundle, tăng JS size đáng kể.

### AP6: Dùng `pages/` và `app/` cho cùng route

- ❌ **Code Smell:** Dùng hybrid mode, giữ `pages/api` nhưng tạo `app/dashboard` trùng path mà không có quy ước rõ.

  ```text
  pages/
  └── api/orders.ts        // GET /api/orders (Pages API Route)
  app/
  └── dashboard/page.tsx   // route mới ở App Router
  // Không vấn đề ngay lập tức vì 2 path khác nhau — nhưng team không
  // có quy ước, sau vài sprint có người tạo app/api/orders/route.ts
  // trùng path với pages/api/orders.ts.
  ```

- ✅ **Staff Refactor:** Chọn 1 router. Chuyển `pages/api` sang `app/api/route.ts`. Không duplicate.

  ```text
  // Trade-off: Tốn effort migrate toàn bộ pages/api/* sang
  // app/api/*/route.ts, đổi lại giảm rủi ro routing precedence
  // conflict về 0.
  app/
  ├── api/orders/route.ts   // GET/POST /api/orders — duy nhất 1 nguồn sự thật
  └── dashboard/page.tsx
  ```

- 🧠 **Why:** App Router takes precedence khi conflict, nhưng không nên duplicate — plan migration rõ ràng, không để 2 router tồn tại song song vô thời hạn.
- 🔥 **Incident Risk:** Routing precedence conflicts, khó debug xem request đi về đâu, build cache không tối ưu.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Nested `layout.tsx`** | Share UI, preserve state, zero config | Dễ block rendering nếu để data fetch nặng | Mọi lúc cần share nav/sidebar | Khi cần re-mount DOM/state | Rất thấp |
| **Route Groups `(folder)`** | Gom chung layout mà không thêm URL path | Cần hiểu rõ convention dấu `()` | Phân tách UI auth vs public cùng 1 URL | Nếu URL path thực sự cần đổi | Không có |
| **`template.tsx`** | Re-mount component, reset state | Mất state, worse performance, DOM thrash | Onboarding flow, enter/exit animations | 99% use cases thông thường | Re-render DOM toàn bộ subtree |
| **Flat Routing (Pages Legacy)** | Đơn giản 1-1 map URL-file | Wrapper hell cho layouts, không colocate | Maintain legacy codebase | Projects mới | Wrapper component hell |

## 6. Legacy Recognition

Nếu bạn mở 1 codebase và thấy những pattern dưới đây — bạn đang đứng trước Pages Router. Nhận diện được nó là bước đầu để biết phải migrate gì:

- **Legacy:** `pages/index.tsx` → file-based routing trong thư mục `pages/`, mỗi file map trực tiếp 1 route, không có concept "layout" native.
- **Legacy:** `pages/_app.tsx` → god-file wrapper, dùng HOCs để inject global layout, theme, auth check.
- **Legacy:** `getServerSideProps` → data fetching pattern cũ, `params`/`query` truy cập sync, không có concept streaming theo segment.
- **Legacy:** `pages/_document.tsx` → custom document để override `<html>`/`<body>`, thay thế bởi root layout trong App Router.

---

## 7. Modern Approach

- **Modern:** `app/page.tsx` → route entry point trong App Router, thin wrapper (xem mục 3.1).
- **Modern:** `app/layout.tsx` → thay thế cả `_app.tsx` và `_document.tsx`, bắt buộc chứa `<html>` và `<body>` (chỉ ở root).
- **Migration `pages/_app.tsx` → `app/layout.tsx`:** Chuyển wrapper logic (VD: `SessionProvider`) vào `app/layout.tsx`. Chú ý: Global CSS (`globals.css`) chỉ được import ở Root Layout.
- **Migration `pages/index.tsx` → `app/page.tsx`:** Đổi tên file, đổi cấu trúc folder theo route segment, xóa return type không cần thiết khi dùng Server Components.
- **Migration `getServerSideProps` params sync → Async params:** Upgrade Next 16, chạy codemod chuyển toàn bộ page component sang `async function`, `await params`.
- **Strategy:** Đây là ứng cử viên kinh điển cho **Strangler Fig** — Next.js hỗ trợ chạy song song `pages/` và `app/` trong cùng project. Migrate từng route group 1, không rewrite toàn bộ trong 1 PR. Bắt đầu từ route ít traffic nhất, dùng feature flag hoặc canary deploy để theo dõi error rate trước khi migrate route quan trọng.
- **Rollback plan:** Giữ nguyên folder `pages/` trong suốt quá trình migrate, không xóa sớm. Nếu 1 route mới ở `app/` có vấn đề production, xóa riêng folder route đó trong `app/`, route tương ứng ở `pages/` (nếu chưa xóa) tự động phục vụ lại — không cần redeploy toàn bộ app.

## 8. Production Scenario

### Scenario 1: Navigate giữa `/dashboard/orders` và `/dashboard/profile` — layout không re-mount

- **Symptom:** Data fetch cũ ở layout (user profile avatar, unread notifications) vẫn hiển thị mà không gọi lại API.
- **Root cause:** Đây là behavior đúng của `layout.tsx` (preserve state) — không phải bug.
- **Detect:** Dùng React DevTools, thấy component tree của layout không bị unmount khi navigate.
- **Fix đúng:** Dùng `router.refresh()` từ client để ép re-fetch data ở Server Component, KHÔNG dùng `template.tsx` chỉ để "fix" việc này — đó là dùng sai tool, kéo theo toàn bộ side-effect của AP3.

### Scenario 2: `params` sync access breaking khi upgrade 15→16

- **Symptom:** Crash 500 ngay khi truy cập dynamic route `/posts/[id]`.
- **Root cause:** `params` là Promise trong v16, code cũ truy cập sync.
- **Detect:** Check stack trace, tìm `TypeError: Cannot destructure property ... of 'params' as it is undefined`.
- **Fix:** Chạy `npx @next/codemod@canary upgrade`. Nếu codemod không cover hết (custom wrapper quanh page component), sửa manual theo pattern ở AP4.

### Scenario 3: "Layout không apply"

- **Symptom:** Vào route con nhưng mất header/footer của layout cha.
- **Root cause:** Đặt `layout.tsx` sai segment level (đặt sâu quá trong folder con) hoặc quên `export default`.
- **Detect:** Vẽ routing tree ra giấy. `layout.tsx` chỉ ảnh hưởng đến các `page.tsx` nằm cùng cấp hoặc nằm bên trong folder đó — không áp dụng ngược lên route group cha.

## 9. Performance & Overhead

Raw material không cung cấp benchmark number chính thức cho case này — đừng bịa số. Đây là cách tự đo trong codebase của bạn trước khi đưa ra quyết định kiến trúc:

**Bundle size: Server Components vs Pages Router**
- Dùng `next build` (output mặc định đã in First Load JS theo route) hoặc `@next/bundle-analyzer` để so sánh client JS bundle giữa 1 page Pages Router (toàn bộ component là client by default) và page tương đương ở App Router (Server Component default).
- Metric cần nhìn: cột **First Load JS** (kB) trong output `next build`. Server Component không serialize logic xuống client — chỉ gửi HTML + script hydrate cho phần `'use client'`.
- Quy tắc thực dụng: nếu sau khi convert sang App Router mà First Load JS không giảm, kiểm tra ngay xem có đang để `'use client'` quá cao trong tree không — đó chính là root cause của AP5.

**Re-render cost: `template.tsx` vs `layout.tsx`**
- Dùng React DevTools Profiler, record 1 lần navigate. Với `layout.tsx`, component layout KHÔNG xuất hiện trong flame graph của lần render đó. Với `template.tsx`, nó xuất hiện mỗi lần — cộng thêm full unmount/remount cost của toàn bộ subtree bên trong.
- Nếu layout chứa heavy component (chart, video player, websocket), chi phí re-mount này là jank thật — đo bằng Chrome Performance tab, tìm Long Task >50ms quanh thời điểm navigate.

**Nested layout blocking behavior**
- Nếu `layout.tsx` có `await fetch(...)` không bọc Suspense, toàn bộ subtree (kể cả `page.tsx` con) chờ layout fetch xong mới render — TTFB của page con bị cộng thêm latency của layout fetch.
- Đo bằng Network tab: so sánh thời điểm response trả về của layout's data call với thời điểm content của `page.tsx` xuất hiện trên screen (marker Largest Contentful Paint trong Performance tab).
- Quy tắc thực dụng: nếu layout fetch >100ms và không critical cho above-the-fold UI, bọc nó trong `<Suspense>` riêng với `loading.tsx`, đừng để nó block toàn route.

Số liệu trên phụ thuộc network, server location, payload size cụ thể của bạn — việc của Staff Engineer là đo trên traffic production thật, không copy số từ blog người khác.

## 10. Architecture Notes

- **Nested Layouts & Data Fetching:** Không để data fetch nặng trong layout nếu nó block rendering toàn bộ subtree. Dùng Suspense boundary hoặc streaming để tách phần chậm ra khỏi critical path.
- **Route Groups cho Multi-Tenant / Multi-Role:** Có thể dùng route groups để phân tách layout cho các tenant hoặc user role khác nhau mà không thay đổi URL structure — ví dụ `(admin)` và `(customer)` cùng phục vụ domain gốc.
- **Private Folders (`_`) cho Colocation:** Tận dụng `_components/` để colocate component gần route mà không tạo thêm route. Đây là pattern được khuyến khích thay vì để mọi component dùng chung ở `components/` xa route.
- **Dynamic Routes & Cache:** Catch-all routes cần cân nhắc với ISR/SSG strategy — `[...slug]` có thể tạo ra số lượng page variant lớn, ảnh hưởng trực tiếp đến build time nếu bạn pre-render tất cả.
- **Known Pitfall — Multiple root layouts:** Chỉ được phép có đúng 1 cặp `<html><body>` ở `app/layout.tsx`. Nested layouts KHÔNG được chứa 2 tag này — tạo thêm sẽ throw build error.
- **Known Pitfall — Catch-all route precedence conflict:** Next.js không cho phép dynamic segments conflict (VD: `[slug]` và `[...slug]` cùng level) — throw error ngay lúc build, không phải runtime.
- **Known Pitfall — Route group breaking relative imports:** `[THEORY — NEEDS VERIFICATION]` Folder `(dashboard)` là 1 level thực tế trên file system nên relative import phải tính đến nó — cần xác nhận lại strict behavior này có khác nhau giữa Turbopack và Webpack hay không trước khi đưa vào core khẳng định chắc chắn.

## 11. Interview Questions

### Q1: Sự khác biệt giữa `layout.tsx` và `template.tsx`? (Junior)
<details>
<summary>Đáp án</summary>

`layout.tsx` preserve state, không re-mount khi navigate giữa các route con cùng cây. `template.tsx` re-mount — tạo instance React mới — trên mỗi lần navigation, kể cả khi nội dung giống nhau. Mặc định luôn chọn `layout.tsx`; chỉ dùng `template.tsx` khi bạn cố ý cần reset state hoặc replay animation.

</details>

### Q2: Giải thích tại sao `params` phải là async trong Next.js 16? (Mid)
<details>
<summary>Đáp án</summary>

Để chuẩn hóa Async Request API — đồng nhất cách truy cập với `searchParams`, hỗ trợ streaming dưới mức route segment (không phải chờ toàn bộ params resolve mới bắt đầu render phần không phụ thuộc), và tương thích tốt hơn với Edge Runtime, nơi các request API vốn đã thiên về async.

</details>

### Q3: Tại sao App Router không cần `_app.tsx`? (Junior)
<details>
<summary>Đáp án</summary>

Vì `app/layout.tsx` đã thay thế hoàn toàn vai trò của `_app.tsx` — wrap toàn bộ app, giữ global state/provider, import global CSS — và còn gánh thêm vai trò của `_document.tsx` (định nghĩa `<html>`/`<body>`), nên không cần file riêng nữa.

</details>

### Q4: Layout và page khác nhau như thế nào? (Fresher)
<details>
<summary>Đáp án</summary>

`page.tsx` là UI độc nhất cho 1 URL cụ thể — bắt buộc phải có để folder trở thành route. `layout.tsx` là UI dùng chung cho nhiều route con — không bắt buộc, không tự tạo route riêng, và không re-render khi chuyển giữa các route con đó.

</details>

### Q5: Nếu tôi muốn reset state của form khi user chuyển tab trong dashboard, nên dùng gì? (Mid)
<details>
<summary>Đáp án</summary>

KHÔNG dùng `template.tsx` cho cả route — nó reset luôn cả phần bạn không muốn reset (sidebar, header). Gắn `key` prop của React vào component form, lấy giá trị từ URL segment hiện tại, để force re-mount cục bộ đúng phần cần reset mà không đụng tới layout xung quanh.

</details>

## 13. Summary

- `layout.tsx` là persistent UI wrapper — tận dụng để giữ state chung, đừng dùng `template.tsx` thay thế vô cớ.
- `page.tsx` phải là thin wrapper — tách logic ra component riêng để re-use và test.
- `params` trong Next.js 16 là Promise — luôn `await` trước khi dùng, đặc biệt khi upgrade từ v15.
- App Router loại bỏ `_app.tsx` — mọi global setup chuyển vào root `layout.tsx`, và nhớ chỉ 1 cặp `<html><body>` tồn tại trong toàn app.

## 14. Terminology Evolution

### Segment
```
┌───────────────┬──────────┬───────────────────────────────┐
│ Era           │ Status   │ Dùng khi nào                   │
├───────────────┼──────────┼───────────────────────────────┤
│ v13+          │ Active   │ App Router directory           │
│ v12 and below │ Legacy   │ Pages Router (implicit)        │
└───────────────┴──────────┴───────────────────────────────┘
```
Lịch sử: Trước v13, segment là khái niệm implicit trong URL parsing — không có đại diện trực tiếp trên file system. Từ v13, segment được ánh xạ 1-1 với folder trong `app/`.
Codebase legacy: `pages/dashboard/settings.tsx` — vẫn là 2 segments về mặt URL, nhưng không thể hiện qua cấu trúc folder lồng nhau như App Router.

### layout.tsx
```
┌───────────────┬──────────┬───────────────────────────────┐
│ Era           │ Status   │ Dùng khi nào                   │
├───────────────┼──────────┼───────────────────────────────┤
│ v13+          │ Active   │ App Router — persistent UI     │
│ v12 and below │ Legacy   │ pages/_app.tsx (partial)       │
└───────────────┴──────────┴───────────────────────────────┘
```
Lịch sử: `_app.tsx` là god-file wrapper, dùng HOC để mô phỏng layout. Từ v13, `layout.tsx` tách biệt concern và hỗ trợ nested layout native theo file system.
Codebase legacy: `_app.tsx` kết hợp nhiều HOC để inject layout cho toàn app, không phân biệt được theo route group.

### template.tsx
```
┌───────────────┬──────────┬───────────────────────────────┐
│ Era           │ Status   │ Dùng khi nào                   │
├───────────────┼──────────┼───────────────────────────────┤
│ v13+          │ Active   │ App Router — re-mount UI       │
│ N/A           │ N/A      │ Không có tương đương           │
└───────────────┴──────────┴───────────────────────────────┘
```
Lịch sử: Được giới thiệu trong App Router từ v13 để xử lý use case cần re-mount (animation enter/exit, reset state theo URL).
Codebase legacy: Không có tương đương trong Pages Router — muốn re-mount, bạn phải tự gắn `key` thủ công ở `_app.tsx`.

### params (async)
```
┌───────────────┬──────────┬───────────────────────────────┐
│ Era           │ Status   │ Dùng khi nào                   │
├───────────────┼──────────┼───────────────────────────────┤
│ v16+          │ Active   │ App Router — async Promise     │
│ v13–v15       │ Legacy   │ App Router — sync object       │
│ v12 and below │ Legacy   │ pages/ — getServerSideProps    │
└───────────────┴──────────┴───────────────────────────────┘
```
Lịch sử: v13–v15, `params` là object sync truyền trực tiếp vào page component. v16+, `params` là Promise để chuẩn hóa Async Request API.
Codebase legacy: `const { slug } = params` (v15) → `const { slug } = await params` (v16).

### app/
```
┌───────────────┬──────────┬───────────────────────────────┐
│ Era           │ Status   │ Dùng khi nào                   │
├───────────────┼──────────┼───────────────────────────────┤
│ v13+          │ Active   │ App Router directory           │
│ v12 and below │ Legacy   │ pages/ directory                │
└───────────────┴──────────┴───────────────────────────────┘
```
Lịch sử: v13 giới thiệu `app/` song song với `pages/` (opt-in). Từ v14+, `app/` trở thành recommended default cho project mới.
Codebase legacy: `pages/index.tsx`, `pages/_app.tsx` — toàn bộ routing dồn vào 1 folder phẳng.

### page.tsx
```
┌───────────────┬──────────┬───────────────────────────────┐
│ Era           │ Status   │ Dùng khi nào                   │
├───────────────┼──────────┼───────────────────────────────┤
│ v13+          │ Active   │ App Router — route UI          │
│ v12 and below │ Legacy   │ pages/*.tsx                    │
└───────────────┴──────────┴───────────────────────────────┘
```
Lịch sử: `pages/index.tsx` → `app/page.tsx`. Khác biệt quan trọng: `page.tsx` chỉ render khi route match — nó không đóng vai trò layout wrapper như mọi file trong `pages/` từng làm.
Codebase legacy: Mọi file trong `pages/` tự động là 1 route. Trong `app/`, chỉ riêng `page.tsx` mới tạo route — file khác trong cùng folder (layout, loading...) không tạo route.

├───────────────┼──────────┼───────────────────────────────┤
