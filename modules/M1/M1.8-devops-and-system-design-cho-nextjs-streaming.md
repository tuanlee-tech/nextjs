**4. Core Concepts**

**Concept 1: Chunked Transfer Encoding**

- `Transfer-Encoding: chunked` = server gửi HTML từng phần
- Nếu thấy `Content-Length: 12345` = buffer đang bật, streaming chết

**Concept 2: Các layer buffer**

| Layer      | Buffer khi nào                | Fix                           |
| ---------- | ----------------------------- | ----------------------------- |
| Nginx      | `proxy_buffering on` mặc định | `proxy_buffering off`         |
| Cloudflare | Auto Minify HTML              | Tắt Auto Minify               |
| AWS Lambda | Mặc định không stream         | `InvokeMode: RESPONSE_STREAM` |
| Gzip       | Nén cần đủ data               | `compress: false`             |

**Concept 3: PPR + Streaming**

- Shell prerender tại build → cache ở edge
- Hole stream tại request → TTFB ~0ms
- Nếu buffer → shell + hole gửi cùng lúc, mất lợi PPR

**Concept 4: Standalone Output**

- `output: 'standalone'` tạo minimal image
- Phải tự copy `public/` và `.next/static/`

**5. Legacy Recognition**

Dấu hiệu nhận biết

✅

- `getServerSideProps` đợi tất cả data → TTFB chậm
- Không có `Suspense` → không có streaming
- `next export` → static HTML, không stream

❌

- `Suspense` nhưng proxy buffer → "fake streaming"
- PPR bật nhưng CDN cache HTML động → stale shell

| Legacy                               | Modern                                |
| ------------------------------------ | ------------------------------------- |
| `getServerSideProps` + đợi full data | `Suspense` + stream từng phần         |
| `next export` static                 | PPR: shell static + hole dynamic      |

| Lambda mặc định                      | Lambda Function URL `RESPONSE_STREAM` |

**7. Production Scenario**

**Context:** E-commerce product page, deploy VPS + Nginx + Cloudflare

**Problem:** User báo trang load chậm, skeleton không hiện

**Root Cause:** Nginx `proxy_buffering on`, Cloudflare Auto Minify bật

**Solution:**

```nginx
location / {
    proxy_pass http://nextjs:3000;
    proxy_buffering off;
    proxy_cache off;
}
```

```
Cloudflare Page Rule:
  Auto Minify HTML: OFF
  Cache Level: Bypass (cho /product/*)
```

**Lesson:** Streaming chỉ có ý nghĩa khi **mọi layer** forward chunk ngay.

👉 Rule:

> Nếu gặp **TTFB chậm dù đã dùng Suspense** hãy nghĩ ngay đến **proxy buffer**.

## 1. Lesson Goal

Hiểu tại sao streaming của Next.js App Router **không tự nhiên chạy được** ở mọi hạ tầng. Biết cách cấu hình từng loại hạ tầng (reverse proxy, CDN, serverless, container) để tận dụng trọn vẹn Suspense, PPR, và TTFB tối ưu. Đồng thời nhận ra trade-offs giữa "zero-config" (Vercel) và "full-control" (self-host).

## 2. Evolution Context

| Era                               | Cách deploy                     | Vấn đề streaming                                                                                                |
| --------------------------------- | ------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| Next.js 9–12 (Pages Router)       | `next start` hoặc `next export` | Không có streaming. SSR trả full HTML một lần.                                                                  |
| Next.js 13–14 (App Router)        | Vercel hoặc self-host           | Streaming có nhưng proxy/CDN hay buffer.                                                                        |
| Next.js 15+                       | `htmlLimitedBots` config        | Bot detection tùy chỉnh.                                                                                        |
| Next.js 16 (PPR stable, Oct 2025) | `ppr: true`, `cacheComponents`  | Shell prerender + hole stream. Theo Next.js blog chính thức, PPR stable release là Next.js 16 vào October 2025. |

## 3. Mental Model

**Streaming giống như giao đồ ăn từng món:**

- Bếp (Next.js) nấu xong món nào giao món đó.
- Nhưng nếu người giao hàng (proxy/CDN) đợi đủ cả bữa mới giao → khách đói.
- **Mỗi layer giữa server và client có thể "đợi đủ rồi mới giao" → streaming bị phá hỏng.**

Theo Next.js docs chính thức, "Any layer between your server and the client that buffers the response can diminish the benefits of streaming."

## 4. Core Concepts

### 4.1 HTTP Chunked Transfer Encoding

Streaming dựa trên `Transfer-Encoding: chunked`. Server gửi HTML từng phần, không cần biết tổng kích thước trước.

```bash
# Kết quả mong muốn
curl -i -N http://localhost:3000/product/1

HTTP/1.1 200 OK
Transfer-Encoding: chunked
Content-Type: text/html; charset=utf-8

<html>...          ← chunk 1 (shell)
<script>...        ← chunk 2 (metadata)
<div id="S:1">...  ← chunk 3 (hole data)
```

Nếu thấy `Content-Length: 12345` thay vì `Transfer-Encoding: chunked`, streaming đang bị chặn.

### 4.2 Các layer có thể buffer

| Layer                        | Cách buffer                   | Cách phát hiện                 |
| ---------------------------- | ----------------------------- | ------------------------------ |
| Reverse proxy (Nginx)        | `proxy_buffering on` mặc định | `curl -I` không thấy `chunked` |
| CDN (Cloudflare)             | Auto Minify HTML              | TTFB ≈ Total time              |
| Load balancer (ALB)          | WAF/Lambda@Edge transform     | Response delay đều đặn         |
| Compression (Gzip)           | Nén cần đủ data               | Chunk đầu delay bất thường     |
| Serverless (Lambda mặc định) | Trả toàn bộ response một lần  | Không có `chunked`             |

Theo Next.js docs self-hosting, "Nginx and similar reverse proxies buffer responses by default."

## 5. Legacy Recognition

**Anti-pattern từ Pages Router era:**

```js
// pages/product/[id].js — Pages Router, không có streaming
export async function getServerSideProps() {
  const data = await fetchSlowAPI(); // Đợi 3 giây
  return { props: { data } };
}

// User đợi 3 giây nhìn trang trắng
// Không có Suspense, không có fallback
```

**App Router đúng:**

```tsx
// app/product/[id]/page.tsx — Streaming
export default function ProductPage() {
  return (
    <main>
      <h1>Product</h1> {/* Shell: render ngay */}
      <Suspense fallback={<Skeleton />}>
        <SlowData /> {/* Hole: stream sau */}
      </Suspense>
    </main>
  );
}
```

**Nhưng** nếu proxy buffer, App Router cũng trở thành "getServerSideProps 2.0" — user vẫn đợi 3 giây.

## 6. Modern Approach

### 6.1 Reverse Proxy — Nginx

**Vấn đề: Nginx buffer response by default**

Nginx mặc định đợi nhận đủ response từ upstream (Next.js) rồi mới gửi về client. Điều này phá vỡ streaming của PPR — HTML tĩnh và dynamic content đều bị delay.

```
Client ◀────── [Nginx buffer đầy] ──────◀ Next.js streaming chunks
                │
                ▼
         ❌ TTFB chậm, PPR vô dụng
```

---

**Giải pháp 1: Header từ Next.js (gợi ý cho Nginx)**

Khi **không kiểm soát được config Nginx** (ví dụ: khách hàng tự quản lý infrastructure, hoặc dùng shared hosting), bạn có thể gửi header để "đề nghị" Nginx tắt buffering:

```js
// next.config.js
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*{/}?',
        headers: [{ key: 'X-Accel-Buffering', value: 'no' }],
      },
    ];
  },
};
```

**Cơ chế:** Nginx đọc header này từ upstream response và tự động tắt buffering **nếu**:

- Module `ngx_http_proxy_module` có sẵn (gần như luôn có)
- `proxy_buffering` **chưa được set explicitly** trong config

```nginx
# Nginx mặc định: proxy_buffering on;

location / {
    proxy_pass http://nextjs:3000;
    proxy_http_version 1.1;
    # proxy_buffering on;  ← default, có thể có hoặc không trong file

    # Nếu config KHÔNG có proxy_buffering → header có tác dụng
    # Nếu config CÓ proxy_buffering on/off → header bị ignore
}
```

---

**Vấn đề của Giải pháp 1**

Header `X-Accel-Buffering` là **extension của Nginx**, không phải chuẩn HTTP. Việc nó có tác dụng hay không phụ thuộc hoàn toàn vào config Nginx — điều bạn không kiểm soát được.

| Trường hợp                                                     | Kết quả                                                |
| -------------------------------------------------------------- | ------------------------------------------------------ |
| Nginx config **không set** `proxy_buffering` + header có       | Buffering **tắt**, streaming hoạt động ✅              |
| Nginx config **không set** `proxy_buffering` + header không có | Buffering **bật** (default), streaming bị phá ❌       |
| Nginx config đã set `proxy_buffering on`                       | Header bị **ignore**, streaming bị phá ❌              |
| Nginx config đã set `proxy_buffering off`                      | Header bị **ignore**, nhưng streaming vẫn hoạt động ✅ |

> **Tóm lại:** Giải pháp 1 là **"hy vọng"** — không đảm bảo.

---

**Giải pháp 2: Config trực tiếp trên Nginx (khuyến nghị)**

Khi **kiểm soát được Nginx**, tắt buffering explicitly:

```nginx
server {
    listen 443 ssl http2;
    server_name app.example.com;

    location / {
        proxy_pass http://nextjs:3000;
        proxy_http_version 1.1;
        proxy_buffering off;      # ← Tắt buffering, chắc chắn
        proxy_cache off;          # ← Không cache HTML động
        proxy_read_timeout 60s;
    }

    # Static assets vẫn cache bình thường
    location /_next/static {
        proxy_pass http://nextjs:3000;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

|               | Giải pháp 1 (Header)     | Giải pháp 2 (Config)       |
| ------------- | ------------------------ | -------------------------- |
| **Kiểm soát** | Phụ thuộc upstream admin | Bạn tự quyết định          |
| **Đảm bảo**   | Không                    | Có                         |
| **Use case**  | Không sửa được Nginx     | Self-host, có quyền config |

---

**Best Practice**

```
┌─────────────────────────────────────────┐
│  Luôn dùng Giải pháp 2 khi có thể       │
│                                         │
│  proxy_buffering off;                   │
│                                         │
│  Giải pháp 1 (header) chỉ là fallback   │
│  cho trường hợp không kiểm soát được    │
│  infrastructure                         │
└─────────────────────────────────────────┘
```

Nếu muốn "double safety", dùng cả 2 — header không gây hại khi Nginx đã tắt buffering:

```nginx
location / {
    proxy_pass http://nextjs:3000;
    proxy_http_version 1.1;
    proxy_buffering off;  # ← Quyết định cuối cùng
    # Header X-Accel-Buffering: no vẫn được gửi, nhưng bị ignore
}
```

**Kiểm chứng:**

```bash
curl -i -N http://your-site.com/product/1 | grep "Transfer-Encoding"
# Kết quả đúng: Transfer-Encoding: chunked
```

### 6.2 CDN — Cloudflare

**Vấn đề:** "Auto Minify HTML" phải đọc toàn bộ HTML để minify → buffer.

**Giải pháp:**

```
Page Rules:
  URL: *example.com/product/*
  Settings:
    Cache Level: Bypass        ← Không cache HTML động
    Auto Minify HTML: OFF       ← Tắt minify
```

### 6.3 Serverless — AWS Lambda

**Vấn đề:** Lambda mặc định trả toàn bộ response một lần. Không stream.

**Giải pháp:** Lambda Function URL với `InvokeMode: RESPONSE_STREAM`.

```yaml
# SAM template
Resources:
  NextJSFunction:
    Type: AWS::Lambda::Function
    FunctionUrlConfig:
      AuthType: NONE
      InvokeMode: RESPONSE_STREAM # ← BẮT BUỘC
```

**Hoặc:** Dùng ECS/Fargate thay vì Lambda nếu cần streaming đơn giản.

Theo Vercel blog (March 2023), "Vercel's serverless architecture is largely powered by AWS Lambda, which... does not provide native support for HTTP streaming." Vercel đã phát triển custom solution để enable streaming trên nền Lambda.

### 6.4 Docker — Standalone Output

Theo Next.js docs chính thức, `output: 'standalone'` tạo minimal image chỉ chứa file cần thiết runtime.

```dockerfile
# Multi-stage build
FROM node:20-alpine AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build

FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production

# Standalone output chỉ cần 3 thư mục
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static

EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"
CMD ["node", "server.js"]
```

**Lưu ý:** Standalone mode không tự động copy `public` và `.next/static`. Phải copy thủ công.

## 7. Production Scenario

```nginx
# nginx.conf
server {

    }
```

### Scenario 2: SaaS Dashboard trên Vercel

**Yêu cầu:**

- Dashboard personalized theo user
- Không thể cache HTML (vì personalized)
- Vẫn muốn TTFB tốt

**Giải pháp:**

```tsx
// app/dashboard/page.tsx
export const dynamic = 'force-dynamic'; // Không cache HTML

export default function Dashboard() {
```

## 8. Debugging Scenario

### Problem: Streaming không hoạt động sau deploy

**Symptom:** User báo trang load chậm, skeleton không hiện.

**Debug steps:**

```bash
# Step 1: Kiểm tra chunked encoding
curl -i -N https://site.com/product/1 | head -5

# Mong đợi: Transfer-Encoding: chunked
# Thực tế: Content-Length: 15420  ← Buffer đang bật!

# Step 2: Kiểm tra TTFB
curl -o /dev/null -s -w "TTFB: %{time_starttransfer}s\n" \
  https://site.com/product/1

# TTFB ≈ Total time → streaming bị chặn

# Step 3: Kiểm tra từng layer
# Test trực tiếp Next.js (bypass Nginx)
curl -i -N http://localhost:3000/product/1 | grep "Transfer-Encoding"
# Nếu có chunked → lỗi ở Nginx
# Nếu không có → lỗi ở Next.js config
```

**Root cause:** Nginx `proxy_buffering on` mặc định.

**Fix:**

```nginx
location / {
    proxy_buffering off;
    proxy_cache off;
}
```

## 9. Refactoring Scenario

### Từ: SSR thuần (buffer toàn bộ)

```tsx
// Trước: Không có Suspense, đợi tất cả data
export default async function Page() {
  const product = await fetchProduct(); // 1 giây
  const reviews = await fetchReviews(); // 3 giây
  const related = await fetchRelated(); // 2 giây

  return (
    <main>
      <h1>{product.name}</h1>
      <Reviews data={reviews} />
      <RelatedProducts data={related} />
    </main>
  );
}
// TTFB = 6 giây, user nhìn trang trắng
```

### Sang: Streaming với Suspense

```tsx
// Sau: Shell render ngay, hole stream dần
export default function Page() {
  return (
    <main>
      <h1>Product</h1> {/* Shell: ngay lập tức */}
      <Suspense fallback={<ProductSkeleton />}>
        <ProductInfo /> {/* Stream sau 1 giây */}
      </Suspense>
      <Suspense fallback={<ReviewSkeleton />}>
        <ReviewList /> {/* Stream sau 3 giây */}
      </Suspense>
      <Suspense fallback={<RelatedSkeleton />}>
        <RelatedProducts /> {/* Stream sau 2 giây */}
      </Suspense>
    </main>
  );
}
// TTFB = 50ms, user thấy layout ngay
```

**Nhưng** nếu Nginx buffer, cả hai đoạn code trên đều cho TTFB = 6 giây. **Streaming chỉ có ý nghĩa khi hạ tầng hỗ trợ.**

### 10.2 Multi-instance Cache Coordination

Theo Next.js docs self-hosting:

> **"App Router deployments with multiple instances need cache tag coordination."**

Điều này chỉ trở thành vấn đề khi ứng dụng được deploy trên **nhiều Next.js instances** (Kubernetes, Docker Swarm, ECS, nhiều VM phía sau Load Balancer,...).

---

#### Tại sao cần coordination?

Với một instance duy nhất:

```
User
  │
  ▼
Next.js Instance
 ├── Data Cache
 ├── Full Route Cache
 └── Router Cache
```

Giả sử:

```ts
// Cache được lưu trong memory/disk của instance này.
fetch('/api/products', {
  next: {
    tags: ['products'],
  },
});
```

Khi gọi:

```ts
revalidateTag('products');
```

instance chỉ cần xóa cache nội bộ của chính nó.

Lần request tiếp theo sẽ cache miss và dữ liệu mới được fetch lại.

Không có vấn đề gì.

---

#### Vấn đề với nhiều instances

Khi deploy lên production thường không chỉ có 1 server:

```
                  User
                    │
                    ▼
              Load Balancer
             /      |      \
            ▼       ▼       ▼
      Instance A  Instance B  Instance C
```

Mỗi instance đều có cache riêng.

Ví dụ:

```
Instance A
products -> cache v1

Instance B
products -> cache v1

Instance C
products -> cache v1
```

Admin cập nhật sản phẩm.

Request được xử lý bởi **Instance A**: `revalidateTag('products');`

Lúc này:

```
Instance A
products -> deleted

Instance B
products -> cache v1

Instance C
products -> cache v1
```

Nếu request tiếp theo được Load Balancer chuyển sang **Instance B**, người dùng vẫn nhận dữ liệu cũ.

=> Cache giữa các instances bị **inconsistent**.

---

#### Giải pháp: Shared Cache Coordination

Các instances cần một nơi lưu trạng thái invalidate chung (thường là Redis).

```
                     Redis
              Tag State / Versions
          ▲             ▲         ▲
          │             │         │
      Instance A   Instance B    Instance C
```

Khi Instance A gọi: `revalidateTag('products');`. Redis sẽ ghi nhận rằng tag `products` đã bị invalidate.

Trước mỗi request, Next.js gọi: `refreshTags();`

Instance sẽ đọc trạng thái tag từ Redis.

Nếu phát hiện tag đã bị invalidate: `local cache version != Redis version` thì instance sẽ:

- Xóa cache local tương ứng
- Rebuild cache ở request tiếp theo

Nhờ đó tất cả instances đều dần đồng bộ trạng thái cache.

---

#### Ví dụ Custom Cache Handler

```js
// next.config.js
module.exports = {
  cacheHandler: require.resolve('./cache-handler.js'),
};
```

---

#### Pseudo-code: Cache Handler

```js
// cache-handler.js
const { CacheHandler } = require('@neshca/cache-handler');
const createRedisHandler = require('@neshca/cache-handler/redis-stack');

module.exports = class extends CacheHandler {
  constructor(options) {
    super(options);
    this.handlers = [createRedisHandler({ client: redisClient })];

    // Lưu version tag mà instance này đang biết
    // Ví dụ: { products: 18, users: 5, posts: 11 }
    this.localTagVersions = new Map();
  }

  async refreshTags() {
    // 1. Lấy version mới nhất từ Redis
    //    Redis key "next:tag-versions":
    //    ┌──────────┬─────────┐
    //    │ tag      │ version │
    //    │────────────────────│
    //    │ products │   19    │
    //    │ users    │    5    │
    //    │ posts    │   11    │
    //    └──────────┴─────────┘
    const redisTagVersions = await redis.hGetAll('next:tag-versions');

    // 2. So sánh từng tag
    for (const [tag, version] of Object.entries(redisTagVersions)) {
      const redisVersion = Number(version);
      const localVersion = this.localTagVersions.get(tag);

      // Nếu version khác nhau → tag đã bị revalidate ở instance khác
      // Redis: products -> 19
      // Local: products -> 18  ❌ stale!
      if (localVersion !== redisVersion) {
        console.log(`[refreshTags] ${tag}: ${localVersion} → ${redisVersion}`);

        // Cập nhật version local
        this.localTagVersions.set(tag, redisVersion);

        // Next.js tự động invalidate tất cả
        // cache entries gắn tag ["products"] này
        //
        // Trước invalidate:
        //   /products?page=1   tags: ["products"]  ✅ valid
        //   /products?page=2   tags: ["products"]  ✅ valid
        //   /products/123      tags: ["products"]  ✅ valid
        //
        // Sau invalidate:
        //   /products?page=1   tags: ["products"]  ❌ invalid
        //   /products?page=2   tags: ["products"]  ❌ invalid
        //   /products/123      tags: ["products"]  ❌ invalid
      }
    }

    // 3. Request tiếp tục xử lý với cache đã đồng bộ
    //
    // fetch(..., { next: { tags: ["products"] } })
    //
    // ┌─────────────────┐     ┌─────────────┐
    // │  Cache INVALID  │────▶│  Cache MISS │
    // └─────────────────┘     └──────┬──────┘
    //                                │
    //    ┌───────────────────────────┘
    //    ▼
    // ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
    // │ Fetch DB/API│───▶│ Ghi cache   │───▶│ Trả response│
    // └─────────────┘    │ mới với     │    └─────────────┘
    //                    │ version 19  │
    //                    └─────────────┘
    //
    // Hoặc:
    //
    // ┌─────────────────┐     ┌─────────────┐
    // │  Cache VALID    │────▶│  Cache HIT  │
    // └─────────────────┘     └──────┬──────┘
    //                                │
    //                                ▼
    //                         ┌─────────────┐
    //                         │ Trả cache   │
    //                         │ ngay lập tức│
    //                         └─────────────┘
  }
};
```

---

> **Lưu ý**
>
> `refreshTags()` **không phải** là quá trình chạy liên tục (background polling).
>
> Next.js gọi phương thức này trước khi xử lý request để đồng bộ trạng thái invalidate giữa local cache và shared cache backend.
>
> Cơ chế này đảm bảo rằng dù request được phân phối tới bất kỳ instance nào, tất cả đều có thể biết những tag nào đã bị `revalidate`.

### 10.3 Build ID cho PPR Cross-deploy

#### Vấn đề

Khi deploy ứng dụng Next.js với **Partial Prerendering (PPR)**, mỗi lần build tạo ra một **shell HTML** khác nhau. Shell này chứa:

- HTML tĩnh được prerender
- Các placeholder cho dynamic content
- **Build ID** để React nhận diện phiên bản

Vấn đề xảy ra khi **nhiều instances chạy song song** với build khác nhau:

```
┌─────────────────────────────────────────────────────────────┐
│                      Load Balancer                          │
│                                                             │
│   Request 1 ──────▶  Instance A (Build ID #1, Shell v1)     │
│                                                             │
│   Request 2 ──────▶  Instance B (Build ID #2, Shell v2)     │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                                                             |
│  (Load Balancer) Request 1: User nhận Shell v1 từ Instance A|
│       │                                                     |
│       ▼                                                     |
│  React bắt đầu hydrate trên browser                         |
│       │                                                     |
│       ▼                                                     |
│  User tương tác, trigger request 2                          |
│       │                                                     |
│       ▼                                                     |
│  (Load Balancer) Request 2: API call đến Instance B         |
│       │                                                     |
│       ▼                                                     |
│  Instance B trả về Shell v2 (khác v1)                       |
│       │                                                     |
```

## 11. Interview Questions

**Q1: Tại sao streaming của Next.js có thể không hoạt động sau khi deploy lên VPS?**

<details>
<summary>Đáp án</summary>

Vì reverse proxy (Nginx) buffer response by default. Nginx đợi nhận đủ response từ upstream rồi mới gửi xuống client. Cần tắt buffering bằng `proxy_buffering off` hoặc header `X-Accel-Buffering: no`. Theo Next.js docs, "Nginx and similar reverse proxies buffer responses by default."

</details>

**Q2: AWS Lambda mặc định có hỗ trợ streaming không? Nếu không, giải pháp là gì?**

<details>
<summary>Đáp án</summary>

**Không** — theo mặc định.

```
┌─────────────────────────────────────────┐
│  PPR cần streaming                      │
│                                         │
│  Chunk 1: Shell (gửi ngay)              │
│       ↓                                 │
│  Chunk 2: Dynamic content (gửi sau)     │
│       ↓                                 │
│  Chunk 3: More content (gửi sau nữa)    │
│                                         │
│  → TTFB nhanh, UX mượt                  │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│  Lambda mặc định (buffered)             │
│                                         │
│  [đợi tất cả chunk]                     │
│       ↓                                 │
│  [gửi 1 lần duy nhất]                   │
│                                         │
│  → TTFB chậm, PPR vô dụng               │
└─────────────────────────────────────────┘
```

AWS Lambda trả về toàn bộ response một lần (buffered mode). Điều này không tương thích với PPR vì PPR yêu cầu stream HTML từng phần khi Suspense boundary resolve.

Giải pháp: Dùng Lambda Function URL với `InvokeMode: RESPONSE_STREAM`, hoặc chuyển sang ECS/Fargate.

**1. Bật Response Streaming trên Lambda Function URL**

```yaml
# serverless.yml (ví dụ)
Resources:
  MyFunction:
    Type: AWS::Lambda::Function
    Properties:
      # ...

  MyFunctionUrl:
    Type: AWS::Lambda::Url
    Properties:
      TargetFunctionArn: !Ref MyFunction
      InvokeMode: RESPONSE_STREAM # ← Bật streaming
      AuthType: NONE
```

Hoặc qua AWS CLI:

```bash
aws lambda create-function-url-config \
  --function-name my-nextjs-app \
  --invoke-mode RESPONSE_STREAM \
  --auth-type NONE
```

**Lưu ý:**

- Chỉ Lambda Function URL hỗ trợ `RESPONSE_STREAM`
- API Gateway **không** hỗ trợ streaming response
- Cần runtime Node.js 18+ hoặc custom runtime

---

**2. Chuyển sang ECS/Fargate**

Nếu streaming phức tạp hoặc cần full Node.js server:

```
┌─────────────────────────────────────────┐
│  ECS/Fargate                            │
│                                         │
│  Container chạy Next.js server          │
│  → Hỗ trợ streaming native              │
│  → Không giới hạn response size         │
│  → Phù hợp PPR, SSR, API routes         │
│                                         │
│  Trade-off: Cần quản lý container,      │
│  cost cao hơn Lambda nhẹ                │
└─────────────────────────────────────────┘
```

---

**So sánh các option**

|                | Lambda Buffered     | Lambda Response Stream | ECS/Fargate       |
| -------------- | ------------------- | ---------------------- | ----------------- |
| **Streaming**  | ❌ Không            | ✅ Có                  | ✅ Có             |
| **Setup**      | Đơn giản            | Cần Function URL       | Cần VPC, ALB      |
| **Cold start** | Cao                 | Cao                    | Thấp (keep-alive) |
| **Cost**       | Pay per request     | Pay per request        | Pay per uptime    |
| **Giới hạn**   | 6MB response        | 20MB stream            | Không giới hạn    |
| **Phù hợp**    | Static/API đơn giản | PPR + streaming        | Heavy SSR, WS     |

---

**Tóm tắt**

```
PPR + AWS?
    │
    ├── Lambda Function URL + RESPONSE_STREAM ──▶ ✅ Streaming hoạt động
    │
    ├── Lambda mặc định (buffered) ────────────▶ ❌ Không dùng được PPR
    │
    └── ECS/Fargate ────────────────────────────▶ ✅ Streaming hoạt động,
                                                     linh hoạt hơn
```

> **Theo Next.js docs:** "AWS Lambda, for example, requires response streaming mode to be explicitly enabled (it is not the default)."

</details>

**Q3: Tại sao PPR cần streaming support ở hạ tầng?**

<details>
<summary>Đáp án</summary>

PPR gửi shell prerender trước, sau đó stream hole vào. Nếu hạ tầng buffer, shell và hole được gửi cùng lúc sau khi tất cả resolve xong. Theo Next.js docs, _"Without it, the static shell and dynamic content are delivered together after the full render completes, eliminating PPR's time-to-first-byte advantage."_

</details>

**Q4: Khi nào nên dùng Vercel thay vì self-host?**

<details>
<summary>Đáp án</summary>

- Vercel khi: team không có DevOps, cần preview deploy, traffic biến động, muốn zero-config.
- Self-host khi: cần full control, cost-sensitive, có team DevOps, hoặc cần feature không support trên Vercel (WebSocket, long-running process).

Theo Divotion (một agency chuyên về headless commerce) chỉ ra một vấn đề quan trọng về **vendor lock-in** trong hệ sinh thái Next.js/Vercel.

`"The line between Next.js and Vercel is paper thin, and that is a problem if you ever want to leave."`

| Tính năng                                 | Trên Vercel                           | Self-host/AWS/GCP                               |
| ----------------------------------------- | ------------------------------------- | ----------------------------------------------- |
| **Edge Functions**                        | 1-click, global                       | Cần setup phức tạp, không hoàn toàn tương đương |
| **ISR (Incremental Static Regeneration)** | On-demand revalidation hoạt động ngay | Cần cache handler tùy chỉnh, không native       |
| **Image Optimization**                    | Tự động, global CDN                   | Cần cấu hình riêng hoặc không hỗ trợ            |
| **Analytics / Speed Insights**            | Tích hợp sẵn                          | Không có                                        |
| **Preview Deployments**                   | Tự động mỗi PR                        | Cần CI/CD tự xây                                |
| **Server Actions**                        | Hoạt động ổn định                     | Cần config thêm, có thể gặp edge case           |

```
Bạn xây app trên Vercel
        │
        ▼
┌─────────────────────────────────┐
│  Dùng đủ tính năng "Next.js"    │
│  nhưng thực ra phụ thuộc Vercel │
│                                 │
│  • Edge config                  │
│  • KV / Postgres (Vercel)       │
│  • Blob storage                 │
│  • Analytics                    │
└─────────────────────────────────┘
        │
        ▼
Muốn chuyển sang AWS/GCP/self-host
        │
        ▼
┌─────────────────────────────────┐
│  ❌ Phát hiện nhiều thứ không   │
│     hoạt động như cũ            │
│                                 │
│  • ISR revalidation hỏng        │
│  • Image optimization mất       │
│  • Edge Functions không có      │
│  • Preview deploy tự viết lại   │
│  • Database phải migrate        │
└─────────────────────────────────┘
        │
        ▼
Chi phí chuyển đổi rất cao → "Bị khóa" vào Vercel
```

**So sánh với các framework khác**

| Framework     | Platform "chính chủ" | Self-host dễ dàng?                 |

| **Nuxt**      | —                    | ✅ Dễ, không phụ thuộc platform    |
| **SvelteKit** | —                    | ✅ Dễ, adapter pattern rõ ràng     |

**Exercise 2:** Cấu hình Nginx để hỗ trợ streaming cho Next.js App Router.

<details>
<summary>Đáp án</summary>

**Giải pháp 1: Tắt buffering trực tiếp trên Nginx (khuyến nghị)**

```nginx
server {
    listen 443 ssl http2;
    server_name app.example.com;

    location / {
        proxy_pass http://nextjs:3000;
        proxy_http_version 1.1;
        proxy_buffering off;      # ← Tắt buffering
        proxy_cache off;          # ← Không cache HTML động
        proxy_read_timeout 60s;
    }

    location /_next/static {
        proxy_pass http://nextjs:3000;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

| Directive                | Ý nghĩa                                                |
| ------------------------ | ------------------------------------------------------ |
| `proxy_buffering off`    | Nginx gửi chunk về client ngay khi nhận, không đợi đủ  |
| `proxy_cache off`        | Không cache HTML động (static vẫn cache ở block riêng) |
| `proxy_http_version 1.1` | HTTP/1.1 hỗ trợ keep-alive cho streaming               |

---

**Giải pháp 2: Gửi header từ Next.js (fallback)**

Khi **không kiểm soát được config Nginx**, gửi header để gợi ý:

```js
// next.config.js
module.exports = {
  async headers() {
    return [
      {
        source: '/:path*{/}?',
        headers: [{ key: 'X-Accel-Buffering', value: 'no' }],
      },
    ];
  },
};
```

**Cơ chế trong Nginx:**

```nginx
# Logic đơn giản hóa của Nginx
if (proxy_buffering == NGX_CONF_UNSET) {      # Chưa set trong config
    if (upstream_header "X-Accel-Buffering" == "no") {
        buffering = off;                        # Header thắng
    } else {
        buffering = on;                         # Default
    }
}
# Nếu đã set proxy_buffering explicitly trong config
# → Config luôn thắng, header bị ignore
```

---

**So sánh 2 giải pháp**

|               | Giải pháp 1: Config Nginx  | Giải pháp 2: Header            |
| ------------- | -------------------------- | ------------------------------ |
| **Ưu tiên**   | Cao (override header)      | Thấp (chỉ khi config chưa set) |
| **Kiểm soát** | Bạn tự quyết định          | Phụ thuộc admin Nginx          |
| **Đảm bảo**   | ✅ Chắc chắn               | ⚠️ "Hy vọng"                   |
| **Use case**  | Self-host, có quyền config | Khách hàng tự quản lý Nginx    |

---

**Best Practice**

```
┌─────────────────────────────────────────┐
│  1. LUÔN dùng Giải pháp 1 khi có thể    │
│                                         │
│     proxy_buffering off;                │
│                                         │
│  2. Thêm Giải pháp 2 như "double safety"│
│     (không gây hại nếu Giải pháp 1 đã   │
│     có, header chỉ bị ignore)           │
│                                         │
│  3. KHÔNG dùng mỗi Giải pháp 2 —        │
│     không đảm bảo streaming hoạt động   │
└─────────────────────────────────────────┘
```

---

**Tóm tắt**

| Trường hợp                                       | Kết quả                                   |
| ------------------------------------------------ | ----------------------------------------- |
| `proxy_buffering off` trong Nginx                | ✅ Streaming hoạt động (header bị ignore) |
| Config chưa set + header `X-Accel-Buffering: no` | ✅ Streaming hoạt động                    |
| Config chưa set + không có header                | ❌ Buffering bật, streaming bị phá        |
| `proxy_buffering on` trong Nginx + có header     | ❌ Header bị ignore, streaming bị phá     |

</details>

## 13. Summary

- Streaming dựa trên HTTP Chunked Transfer Encoding. Một layer buffer là đủ phá hỏng.
- Nginx: `proxy_buffering off` hoặc `X-Accel-Buffering: no`.
- Cloudflare: Tắt Auto Minify HTML, bypass cache cho route động.
- AWS Lambda: Dùng Function URL với `RESPONSE_STREAM` hoặc chuyển ECS.
- Docker: `output: 'standalone'`, nhớ copy `public` và `.next/static`.
- PPR cần streaming support — nếu không, mất lợi ích TTFB.
- Vercel vs self-host: trade-off giữa zero-config và full control.
