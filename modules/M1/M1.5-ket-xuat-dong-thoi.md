# M1.5 — Concurrent Rendering: Time Slicing & Priority System

## 0. Tóm Tắt

<details>
<summary>Thời gian đọc: 5 phút</summary>

Concurrent Rendering là cơ chế **interruptible rendering** của React 18 — chia render thành units of work nhỏ, yield cho browser giữa các unit, đánh ưu tiên theo Lanes. Từ synchronous (block main thread) → concurrent (responsive, ưu tiên đúng). `startTransition` đánh dấu non-urgent updates, `useDeferredValue` tạo deferred copy từ prop. Là **prerequisite** cho Suspense, Streaming SSR, và Next.js App Router navigation.

⚠️ **Điểm quan trọng: React 18 không tự động bật Concurrent Rendering**. Chỉ upgrade package hoặc dùng `createRoot()` chưa đủ. React 18 có ba chế độ render: Legacy synchronous (`ReactDOM.render` — giống React 17), Synchronous với batching (`createRoot()` nhưng không dùng concurrent features — render vẫn uninterruptible), và Concurrent thực sự (`createRoot()` + `startTransition`/`useDeferredValue` — mới có time slicing và interruptible). Theo React Working Group: "_There is no concurrent mode, there are only concurrent features._" `createRoot()` mở cửa, startTransition mới bước vào.

**3. Mental Model (Ý tưởng cốt lõi)**

> **Concurrent React = Ông quản lý thông minh với task queue** — chia công việc thành từng phần nhỏ, sau mỗi phần kiểm tra "có việc khẩn hơn không?", nếu có thì tạm dừng xử lý việc khẩn rồi quay lại. Concurrent ≠ parallel (multi-thread), mà là **interleaved** (chuyển đổi giữa tasks trong một thread). Nhưng ông quản lý này chỉ làm việc khi được giao đúng công cụ (`startTransition`), không tự động.

Analogy: Old React = ông bếp không thể bị interrupt — phải nấu xong mới làm việc khác. Concurrent React = quản lý nhà hàng — có thể chuyển đổi giữa các công việc, ưu tiên khách gọi nước trước món ăn phức tạp. Nhưng quản lý chỉ bắt đầu làm việc khi được giao đúng nhiệm vụ — không tự động quản lý từ khi bước vào nhà hàng.

**4. Core Concepts (20% kiến thức quan trọng nhất)**

**Concept 1: Synchronous Rendering Bottleneck**

- React 15: reconciliation đệ quy đơn thuần — không có "unit of work", không thể yield
- React 16: Fiber rewrite — mỗi component là Fiber node, có thể pause/resume
- React 18: bật concurrent mode — render interruptible
- Ví dụ: filter 5000 items (~50ms) block input keystroke → user cảm thấy lag

**Concept 2: Fiber Architecture + Time Slicing**

- Fiber = unit of work — JavaScript object với `return`, `child`, `sibling`, `alternate`, `lanes`
- Work loop: xử lý MỘT fiber mỗi lần, check `timeRemaining()`, yield nếu < 1ms
- React dùng `MessageChannel.postMessage()` (latency ~0ms) thay vì `requestIdleCallback` (latency ~50ms)
- Time slicing target: ~5ms chunks, browser không bị block quá 5ms → 60fps giữ được
- Kết quả: total render time có thể lâu hơn chút, nhưng **perceived performance** tốt hơn nhiều

**Concept 3: Priority System (Lanes)**

- Lanes là internal implementation — không tương tác trực tiếp, dùng public APIs
- Priority từ cao xuống thấp: `SyncLane` → `InputContinuousLane` → `DefaultLane` → `TransitionLane` → `RetryLane` → `IdleLane`
- React xử lý high priority trước, có thể interrupt và discard low priority work đã stale
- Ví dụ: user gõ "phone" rồi "phones" → transition cho "phone" bị discard, chỉ render "phones"

**Concept 4: startTransition / useTransition**

- `startTransition`: module-level function — dùng khi không cần `isPending`
- `useTransition`: hook — có `isPending` để track loading state
- Chỉ mark **synchronous** `setState` bên trong callback — `async` trong callback không được support
- Khi nào dùng: search/filter large dataset, tab switching, navigation, sort/reorder
- Khi nào **KHÔNG** dùng: form input, button click, error states, checkout, real-time data
- Đây là API duy nhất "bật" concurrent rendering cho update đó — không dùng = render vẫn sync
- React 19: thử nghiệm async transitions (chưa stable)

**Concept 5: useDeferredValue**

- "Phiên bản component" của startTransition — dùng khi nhận value từ prop/context
- `value` cập nhật ngay (high priority), `deferredValue` lags behind (low priority)
- Không có `isPending` — dùng comparison (`value !== deferredValue`) để check stale
- Dùng khi: component nhận prop, không kiểm soát event handler, cần defer expensive computation

**Concept 6: Automatic Batching**

- React 17: chỉ batch trong React event handlers
- React 18: batch **mọi nơi** — setTimeout, promises, native events (chỉ với `createRoot()`)
- `flushSync()` để opt-out — force immediate render (hiếm gặp, dùng khi cần đọc DOM giữa 2 updates)
- Batching có ngay khi dùng `createRoot()`, nhưng không đồng nghĩa với concurrent

**5. Legacy Recognition (Nhìn code biết gì?)**

Dấu hiệu nhận biết code "pre-Concurrent"

✅ **Nhận ra ngay:**

- `debounce` / `throttle` trong search handlers (lodash)
- `setTimeout` để "delay" state updates
- `shouldComponentUpdate` / `React.memo` dùng rất aggressively
- Comment "// Workaround: delay render to avoid jank"
- Không có `startTransition` hay `useDeferredValue` trong codebase
- `ReactDOM.render()` thay vì `createRoot()`
- Nhầm lẫn: tưởng `createRoot()` = concurrent ngay — thực tế chưa đủ

❌ **Không nên viết:**

- `async` bên trong `startTransition` callback → setState sau `await` không phải transition
- Wrap urgent updates (input, error, form submit) trong transition
- `useDeferredValue` cho real-time data (stock price, live chat) → user thấy stale quá lâu
- `flushSync()` không cần thiết — chỉ dùng khi thực sự cần đọc DOM giữa 2 updates
- Tưởng upgrade React 18 = app concurrent hết — không cần dùng `startTransition`

**7. Production Scenario (Thực tế)**

**Context:** Dashboard analytics với multiple heavy components — chart (~30ms), data table 500 rows (~80ms), filters panel. User báo cáo "lag kinh khủng khi filter" trên Chromebook cũ.

**Problem:** Filter change trigger re-render toàn bộ dashboard đồng bộ → slider bị block, không mượt.

**Root Cause:** Không có cơ chế ưu tiên — filter UI update và heavy chart/table re-render cùng priority.

**Solution:**

- Slider dùng **local state** — di chuyển mượt (urgent, không đợi dashboard)
- `handleRevenueCommit` (thả slider) mới trigger `startTransition` → dashboard update non-urgent
- `isPending` overlay nhẹ thay vì full spinner
- Chart và table trong `Suspense` — stream khi ready

**Kết quả:** Slider 60fps, dashboard re-render với low priority, Chromebook cũ vẫn responsive.

👉 Rule

> Nếu gặp **input lag khi heavy component re-render cùng lúc** hãy nghĩ ngay đến **tách urgent (input state) và non-urgent (heavy computation) với `startTransition`**.

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng**

- Search/filter với large dataset (> 500 items)
- Tab switching với heavy component tree
- Sort/reorder expensive list
- Navigation (App Router tự làm, custom navigation cần `startTransition`)
- Lazy loading với `React.lazy` (tránh spinner flash)
- Heavy derived computation từ prop (dùng `useDeferredValue`)

**Khi không nên dùng**

- Form input state (phải instant)
- Button click feedback (user expect immediate)
- Error messages (phải hiện ngay)
- Animation triggers (phải đúng timing)
- Checkout / payment flow (urgent UX)
- Real-time data (stock price, live chat)
- Component đồng bộ, không có heavy computation

Trade-off

Pros

- Input luôn responsive dù render work nặng
- Không cần debounce/throttle — React tự discard stale transitions
- Navigation mượt mà, không flash spinner
- Automatic batching giảm số lần render
- `isPending` tự động track, không cần manual state

Cons

- Learning curve: phải hiểu priority system, không chỉ "wrap mọi thứ trong transition"
- `async` trong transition không được support (limitation hiện tại)
- `useDeferredValue` có thể show stale data — cần visual indicator
- Concurrent rendering không fix slow network hay expensive computation ngoài render
- Strict Mode double-invoke effects trong dev — cần verify cleanup đúng

```
└── Rewrite từ đầu — mỗi component là một "fiber" (unit of work)
└── Cơ sở hạ tầng cho concurrency — nhưng chưa bật
└── Render vẫn synchronous ở React 16, nhưng Fiber cho phép pause/resume

React 16.3–16.x (2018–2020)
└── Concurrent Mode experimental — opt-in với `ReactDOM.unstable_createRoot()`
└── Chủ yếu dùng cho testing và research; một số công ty lớn (Meta) dùng
    trong production với hỗ trợ từ React team

React 17 (October 2020)
└── "No New Features" release — stepping stone cho gradual upgrade
└── Không có Concurrent Mode; chuẩn bị nền tảng cho React 18
└── Event delegation changes, JSX Transform, easier gradual upgrades

React 18.0 (Tháng 3/2022) ← Bước ngoặt
└── Concurrent Rendering STABLE — opt-in qua `createRoot()`, chỉ kích hoạt
    khi sử dụng concurrent features (startTransition, useDeferredValue, Suspense transitions)
└── Automatic batching — chỉ với `createRoot()`, batch ở mọi context
    (promises, setTimeout, native events), không chỉ event handlers
    ⚠️ Legacy `ReactDOM.render()` giữ behavior cũ (React 17-style)
└── startTransition — API chính thức để đánh dấu update "không khẩn cấp"
└── useDeferredValue — deferred derived state
└── Suspense trong transitions — không flash fallback khi navigation nhanh

Next.js 13+ (2022 đến nay)
└── Pages Router: Vẫn dùng legacy rendering (React 17-style), không có concurrent features
    → Đây là lý do chính để migrate từ Pages Router sang App Router
└── App Router: Dùng React 18, `createRoot()` tự động, concurrent features enabled
└── Streaming SSR tận dụng concurrent rendering để stream HTML chunks
└── loading.js với Suspense hoạt động nhờ concurrent model
```

**Lưu ý quan trọng:** Concurrent Rendering trong Next.js App Router là **always on**. Bạn không opt-in, không cấu hình. Hiểu nó giúp bạn lý giải tại sao app behave theo cách nó đang behave — đặc biệt khi debug navigation, loading states, và performance.

## 3. Mental Model

Phải build mental model đúng trước khi đụng vào code.

**Synchronous React = Ông bếp không thể bị interrupt**

Hình dung React như một người đầu bếp (single thread). Trong React cũ, khi đầu bếp bắt đầu nấu một món (render component tree), anh ta **không thể dừng lại** dù bất cứ chuyện gì xảy ra. Khách gọi nước ư? Đợi đi. Có điện thoại khẩn ư? Đợi đi. Phải nấu xong mới làm việc khác.

Hậu quả: nếu món ăn phức tạp (component tree lớn), mọi thứ khác phải chờ → UI đơ.

**Concurrent React = Ông quản lý thông minh với task queue**

React 18 biến đầu bếp thành quản lý. Quản lý có thể:

1. Chia công việc thành từng phần nhỏ (time slicing)
2. Sau mỗi phần, kiểm tra: "Có việc khẩn hơn không?"
3. Nếu có: tạm dừng việc đang làm, xử lý việc khẩn, rồi quay lại
4. Đánh nhãn mức độ ưu tiên cho từng công việc (priority lanes)

```
Old React (synchronous):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Render 500 items...............................................done
                           ↑
                   User typed here — phải đợi

New React (concurrent):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Render 500 items...pause...User input handled...resume...done
                       ↑              ↑
               React yields    Urgent work done
               to browser      instantly
```

**Keyword để nhớ:** Concurrent không có nghĩa là parallel (chạy nhiều thread cùng lúc). JavaScript vẫn single-threaded. Concurrent nghĩa là **interleaved** — React có thể chuyển đổi giữa các task trong một thread.

## 4. Core Concepts

### 4.1 Tại sao Synchronous Rendering Thất bại

Để hiểu solution, phải hiểu rõ pain point.

```tsx
// Scenario thực tế: Search input + Filter 5000 items
// React 17 — synchronous render

function ProductSearch() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(allProducts); // 5000 items

  function handleInput(e: React.ChangeEvent<HTMLInputElement>) {
    const value = e.target.value;
    setQuery(value);

    // Lọc 5000 items — tốn ~50ms trên mid-range device
    const filtered = allProducts.filter((p) =>
      p.name.toLowerCase().includes(value.toLowerCase()),
    );
    setResults(filtered);
  }

  return (
    <>
      <input
        value={query}
        onChange={handleInput}
        placeholder='Tìm sản phẩm...'
      />
      <ProductGrid products={results} /> {/* Render 5000 rows */}
    </>
  );
}
```

**Điều xảy ra khi user gõ:**

```
User nhấn phím "a"
    ↓
onChange trigger → setQuery + setResults
    ↓
React bắt đầu re-render (đồng bộ, không thể interrupt)
    ↓
Re-render input (nhanh) → Re-render 5000 product rows (~50ms)
    ↓
Browser BLOCKED trong 50ms — không thể xử lý gì khác
    ↓
Browser paint lần đầu (user thấy kết quả)
    ↓
User nhấn phím "b" — browser nhận event
    ↓
Lặp lại từ đầu
```

**Cảm giác của user:** input có độ trễ, animation bị giật, không mượt. FPS giảm từ 60 xuống ~30 hoặc thấp hơn. Cảm giác "lag".

**Tại sao thất bại ở architectural level:** Trong React 15 và trước đó, reconciliation là đệ quy đơn thuần. Không có khái niệm "unit of work", không thể yield. Khi bắt đầu render, phải đi đến cuối stack call. React 16 viết lại toàn bộ thành Fiber để chuẩn bị cho concurrent rendering — nhưng phải mất đến React 18 mới stable.

### 4.2 Fiber Architecture — Nền Tảng Của Concurrency

Bạn đã học về Fiber ở M1.1. Đây là phần kết nối trực tiếp với Concurrent Rendering.

**Fiber = Unit of Work**

Mỗi component trong tree là một Fiber node. Fiber là một JavaScript object bình thường với đầy đủ thông tin để pause và resume:

```javascript
// Cấu trúc Fiber (đơn giản hóa để minh họa)
const fiber = {
  type: UserCard, // Component function/class
  props: { userId: '123' }, // Current props
  stateNode: domNode, // DOM node tương ứng

  // Liên kết trong tree — để duyệt
  return: parentFiber, // Parent
  child: firstChildFiber, // First child
  sibling: nextSibling, // Next sibling

  // Kết quả của render lần trước
  alternate: oldFiber, // "Work in progress" pair

  // ← Đây là chìa khóa cho concurrency
  lanes: TransitionLane, // Mức độ ưu tiên của update này
  pendingProps: newProps, // Props mới chờ được apply
};
```

**Work Loop — Vòng lặp có thể bị interrupt**

```javascript
// Pseudocode — cách React work loop hoạt động
function workLoop(deadline) {
  let shouldYield = false;

  while (nextUnitOfWork && !shouldYield) {
    // Xử lý một fiber (một component)
    nextUnitOfWork = performUnitOfWork(nextUnitOfWork);

    // Sau MỖI fiber, kiểm tra: "Có nên yield không?"
    shouldYield = deadline.timeRemaining() < 1;
    //            ↑ Còn < 1ms là yield cho browser
  }

  if (!nextUnitOfWork && workInProgress) {
    // Toàn bộ tree đã render xong, commit
    commitRoot();
  } else {
    // Còn việc, schedule lại để tiếp tục
    requestIdleCallback(workLoop);
  }
}
```

**Điểm then chốt:** `performUnitOfWork` xử lý đúng MỘT fiber mỗi lần. Sau đó React check thời gian còn lại. Nếu ít hơn 1ms, React dừng và yield cho browser xử lý user events, animations, etc. Sau đó schedule lại để tiếp tục.

Thực tế React dùng `MessageChannel.postMessage()` thay vì `requestIdleCallback` vì `requestIdleCallback` có latency quá cao (~50ms). `MessageChannel` cho latency ~0.

### 4.3 Time Slicing — Render Không Block Browser

Time slicing là kết quả tự nhiên của Fiber work loop. React "slice" thời gian render thành những đơn vị nhỏ (~5ms mỗi slice) và yield về browser giữa các slice.

```
Frame 1 (16ms = 60fps):
├── Browser: handle user events (4ms)
├── React: render 10 fibers (5ms)
└── Browser: paint (7ms)

Frame 2 (16ms):
├── Browser: handle user events (2ms)
├── React: render 12 fibers (5ms)
└── Browser: paint, animations (9ms)

Frame 3 (16ms):
├── Browser: handle user events (1ms)
├── React: render 8 fibers — DONE (4ms)
└── React: commit to DOM (2ms), Browser: paint (9ms)
```

**Điều thay đổi với user:** Browser không bao giờ bị block quá ~5ms. Animations mượt, input responsive, scroll không giật.

**Điều KHÔNG thay đổi:** Tổng thời gian render có thể còn lâu hơn một chút (vì overhead của yielding). Nhưng **perceived performance** tốt hơn nhiều. Người dùng cảm thấy app nhanh dù total render time tương đương.

**Khi nào React yield vs không yield?**

```
Synchronous (không yield):
- Discrete event handlers (click, keydown) — React xử lý xong ngay
- flushSync() — force synchronous render

Concurrent (có thể yield):
- startTransition() — low priority, interruptible
- useDeferredValue — lowest priority
- Suspense retry renders — interruptible
```

### 4.4 Priority System — React Quyết Định Cái Gì Quan Trọng Hơn

React 18 sử dụng hệ thống "Lanes" — mỗi update được gán vào một lane với mức ưu tiên tương ứng.

**Lưu ý quan trọng về Lanes:**

Lanes là internal implementation detail của React. Bạn không cần (và không thể)
tương tác trực tiếp với Lanes trong code ứng dụng. Thay vào đó, React cung cấp
các public APIs (`startTransition`, `useDeferredValue`) để bạn đánh dấu priority
một cách declarative.

Tuy nhiên, việc hiểu Lanes giúp bạn:

- Debug tại sao một update bị delay hoặc interrupt
- Hiểu React DevTools Profiler output
- Thiết kế component architecture phù hợp với priority model

> ⚠️ React team có thể thay đổi internal Lanes implementation bất cứ lúc nào.
> Không viết code phụ thuộc vào Lanes behavior cụ thể.

```
Lanes (từ cao xuống thấp):

SyncLane                  — Đồng bộ (flushSync, legacy code)
InputContinuousLane       — Continuous input (hover, scroll, drag)
DefaultLane               — Event handlers bình thường (click, keydown)
TransitionLane            — startTransition updates
RetryLane                 — Suspense retry (khi promise resolve)
IdleLane                  — Deferred work, useDeferredValue
```

**Ví dụ Priority System trong thực tế:**

```
1. User đang type → setQuery → DefaultLane (ưu tiên cao)
2. Đồng thời: setFilterResults trong startTransition → TransitionLane (ưu tiên thấp)

React scheduler:
- Thấy DefaultLane → xử lý NGAY (re-render input)
- Thấy TransitionLane → xử lý sau, có thể interrupt nếu DefaultLane update lại

Kết quả:
- Input cập nhật tức thì (mỗi keystroke)
- Filter results cập nhật sau (~100-200ms trễ hơn)
- User không cảm thấy lag dù đang filter 5000 items
```

**Điều developer cần biết:** Bạn không interact trực tiếp với Lanes. Thay vào đó bạn dùng APIs như `startTransition` và `useDeferredValue` — React nội bộ map chúng sang đúng Lane.

### 4.5 startTransition và useTransition

Đây là API quan trọng nhất mà developers dùng để interact với priority system.

**startTransition (module-level function):**

```typescript
import { startTransition } from 'react';

// Dùng khi không cần tracking isPending
startTransition(() => {
  // State update bên trong được đánh dấu "non-urgent"
  setSearchResults(filter(data, query));
});
```

**useTransition (hook — có isPending):**

```typescript
import { useTransition } from 'react'

function SearchPage() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState<Product[]>(allProducts)
  const [isPending, startTransition] = useTransition()

  function handleSearch(e: React.ChangeEvent<HTMLInputElement>) {
    const value = e.target.value

    // Urgent: update input ngay lập tức
    // DefaultLane — high priority
    setQuery(value)

    // Non-urgent: update filter results
    // TransitionLane — low priority, interruptible
    startTransition(() => {
      const filtered = filterProducts(allProducts, value)
      setResults(filtered)
    })
  }

  return (
    <div>
      <input
        value={query}
        onChange={handleSearch}
        placeholder="Tìm kiếm..."
      />

      {/* isPending = true khi transition đang xử lý */}
      {/* Dùng để show visual feedback nhẹ — không phải full page spinner */}
      <div style={{ opacity: isPending ? 0.6 : 1, transition: 'opacity 0.2s' }}>
        <ProductGrid products={results} />
      </div>

      {isPending && (
        <span className="search-hint">Đang lọc kết quả...</span>
      )}
    </div>
  )
}
```

**Điều gì thực sự xảy ra bên dưới:**

```
User gõ "phone":
  1. setQuery("phone") → DefaultLane → React re-render input NGAY
  2. startTransition(() => setResults(...)) → TransitionLane → schedule sau

User tiếp tục gõ "phones":
  3. setQuery("phones") → DefaultLane → Re-render input NGAY
  4. React nhận ra có TransitionLane update từ bước 2 chưa done
  5. React DISCARD bước 2 (stale) — không commit intermediate state
  6. startTransition(() => setResults(filter("phones"))) → TransitionLane mới

Kết quả:
  - Input luôn responsive (DefaultLane không bị block)
  - Results chỉ render 1 lần cho "phones", không render trung gian "phon", "phone"
  - isPending = true từ bước 2 đến khi results xong
```

**Đây là "interruptible rendering"**: React có thể discard work-in-progress của transition nếu state đã stale.

**Giới hạn của startTransition — PHẢI BIẾT:**

```typescript
// 🔴 KHÔNG HOẠT ĐỘNG — async bên trong startTransition KHÔNG được mark là transition
function handleSearch(query: string) {
  startTransition(async () => {
    // Phần async này KHÔNG phải transition
    const results = await fetchSearchResults(query); // ← async!
    setResults(results); // ← Đây là DefaultLane, không phải TransitionLane!
  });
}

// Lý do: startTransition chỉ mark những setState ĐỒNG BỘ bên trong callback
// async/await thoát khỏi synchronous execution context của startTransition

// ✅ ĐÚNG — fetch trước, transition sau
async function handleSearch(query: string) {
  const results = await fetchSearchResults(query); // Fetch bên ngoài

  startTransition(() => {
    setResults(results); // ← setState đồng bộ BÊN TRONG transition
  });
}
```

**Khi nào KHÔNG dùng startTransition:**

```typescript
// 🔴 Sai — form submission phải urgent
function handleSubmitOrder() {
  startTransition(() => {
    setOrderStatus('processing'); // User expect IMMEDIATE feedback
    submitOrder(cartItems);
  });
}
// User click "Đặt hàng" → không thấy feedback ngay → confusing

// 🔴 Sai — error states phải urgent
function handleError(error: Error) {
  startTransition(() => {
    setErrorMessage(error.message); // Error phải hiện ngay!
  });
}

// ✅ Đúng — dùng startTransition cho UI không critical
function handleTabChange(tab: string) {
  startTransition(() => {
    setActiveTab(tab); // Tab content load, không urgent
  });
}
```

**Rule of thumb:** Dùng `startTransition` khi update là **non-critical response** — user không expect immeditate feedback, và độ trễ nhỏ là acceptable. Không dùng cho form submissions, error states, hoặc bất kỳ thứ gì user expect "instant".

**React 19 — Async Transitions (Experimental)**

React 19 đang thử nghiệm "async transitions" — cho phép `await` bên trong
`startTransition` callback. Điều này giải quyết limitation hiện tại:

```typescript
// React 19 (experimental) — async transitions
startTransition(async () => {
  const data = await fetchData(); // ← await được support!
  setData(data); // Vẫn là transition
});
```

Tuy nhiên, đây vẫn là experimental feature. Trong production hiện tại,
vẫn phải fetch bên ngoài rồi wrap setState trong startTransition.

### 4.6 useDeferredValue

`useDeferredValue` là "phiên bản component" của startTransition — dùng khi bạn nhận value từ prop hoặc muốn defer một derived computation mà không cần control event handler.

```typescript
import { useDeferredValue, useMemo } from 'react'

function ProductSearch({ searchQuery }: { searchQuery: string }) {
  // searchQuery cập nhật immediate (high priority)
  // deferredQuery lags behind khi system busy (low priority)
  const deferredQuery = useDeferredValue(searchQuery)

  // Computation này chỉ chạy khi deferredQuery thay đổi
  // → chạy ít hơn, với priority thấp
  const filteredProducts = useMemo(
    () => filterProducts(allProducts, deferredQuery),
    [deferredQuery]
  )

  // Khi searchQuery !== deferredQuery: đang "pending"
  const isStale = searchQuery !== deferredQuery

  return (
    <div>
      {/* Visual indicator: kết quả hơi stale */}
      <div style={{ opacity: isStale ? 0.7 : 1 }}>
        <ProductGrid products={filteredProducts} />
      </div>
    </div>
  )
}
```

**So sánh startTransition vs useDeferredValue:**

```
startTransition:
  - Bạn KIỂM SOÁT khi nào update xảy ra (wrap setState)
  - Cần access vào event handler hoặc điểm update
  - Có isPending để track loading state
  - Dùng khi: update được trigger từ event handler bạn kiểm soát

useDeferredValue:
  - Bạn NHẬN value từ bên ngoài (prop hoặc context)
  - Không cần access event handler
  - Không có isPending — dùng comparison (value !== deferredValue)
  - Dùng khi: value đến từ parent/prop, bạn không kiểm soát khi nào nó thay đổi

Ví dụ điển hình:
  useTransition → Component A kiểm soát search query
  useDeferredValue → Component B nhận query như prop, chỉ cần slow down filter
```

**Một điều tinh tế:**

```typescript
function ExpensiveComponent({ value }: { value: string }) {
  const deferred = useDeferredValue(value)

  // Khi value thay đổi:
  // 1. React render component này với value MỚI (high priority) → isPending sense
  // 2. React đồng thời render với deferred value CŨ (backup)
  // 3. Nếu đang busy: user thấy deferred (cũ) trong khi mới đang process
  // 4. Khi xong: swap sang mới
  // Điều này có nghĩa component có thể render 2 lần với 2 values khác nhau

  return <HeavyList items={filterItems(bigData, deferred)} />
}
```

### 4.7 Automatic Batching

Bonus của React 18 — không phải concurrent feature thuần túy nhưng liên quan đến scheduler và thường được hỏi trong interview.

```typescript
// React 17: chỉ batch trong React event handlers
function handleClick() {
  setCount((c) => c + 1); // \
  setFlag((f) => !f); // → 1 render (batched — trong event handler)
}

setTimeout(() => {
  setCount((c) => c + 1); // render 1 (NOT batched — outside React)
  setFlag((f) => !f); // render 2
}, 1000);

fetchData().then(() => {
  setCount((c) => c + 1); // render 1 (NOT batched — in promise)
  setData(data); // render 2
});
```

```typescript
// React 18: batch
// MỌI NƠI — event handlers, setTimeout, promises, native events
function handleClick() {
  setCount((c) => c + 1); // \
  setFlag((f) => !f); // → 1 render
}

setTimeout(() => {
  setCount((c) => c + 1); // \
  setFlag((f) => !f); // → 1 render (tự động batch!)
}, 1000);

fetchData().then((data) => {
  setCount((c) => c + 1); // \
  setData(data); // → 1 render (tự động batch!)
});
```

**Khi cần opt out khỏi automatic batching:**

```typescript
import { flushSync } from 'react-dom';

// Trường hợp hiếm gặp: cần DOM update ngay giữa 2 setState
function handleSpecialCase() {
  flushSync(() => {
    setCount((c) => c + 1); // Force immediate render
  });
  // DOM đã được update ở đây

  flushSync(() => {
    setFlag((f) => !f); // Another immediate render
  });
}
// Dùng khi cần đọc layout từ DOM giữa hai updates (ví dụ: scroll position)
```

### 4.8 React 18 Không Tự Động Bật Concurrent Rendering

Đây là điểm **quan trọng nhất** mà phần lớn developer nhầm lẫn khi upgrade lên React 18. Việc upgrade package hoặc dùng `createRoot()` chưa đủ để app chạy concurrent.

#### 4.8.1 Ba "Chế Độ" Render Trong React 18

React 18 internal có **ba thuật toán render** khác nhau, không phải chỉ một:

| Chế độ                    | API kích hoạt                                                              | Automatic Batching | Interruptible | Time Slicing |
| ------------------------- | -------------------------------------------------------------------------- | ------------------ | ------------- | ------------ |
| **Legacy synchronous**    | `ReactDOM.render()` (deprecated)                                           | ❌ Không           | ❌ Không      | ❌ Không     |
| **Synchronous (default)** | `createRoot()` nhưng không dùng concurrent features                        | ✅ Có              | ❌ Không      | ❌ Không     |
| **Concurrent**            | `createRoot()` + `startTransition`/`useDeferredValue`/Suspense transitions | ✅ Có              | ✅ Có         | ✅ Có        |

**Sự khác biệt cốt lõi:**

- **Legacy synchronous** (`ReactDOM.render`): Giống hệt React 17. Không có gì mới.
- **Synchronous với `createRoot`**: Có automatic batching ở mọi context (setTimeout, promise), nhưng render vẫn **uninterruptible** — một khi bắt đầu render phải chạy đến cuối.
- **Concurrent**: Chỉ bật khi dùng các concurrent features. Render **interruptible**, có time slicing và priority lanes.

#### 4.8.2 "Không Có Concurrent Mode, Chỉ Có Concurrent Features"

Theo React Working Group, React 18 đã **bỏ khái niệm "Concurrent Mode" toàn app** :

> _"There is no concurrent mode, there are only concurrent features."_

Điều này có nghĩa:

- Không còn "bật/tắt" concurrent cho toàn bộ app
- Concurrent rendering chỉ active trong **phạm vi của concurrent feature** đang dùng
- Phần còn lại của app vẫn render synchronously

#### 4.8.4 Các Concurrent Features "Bật" Concurrent Rendering

Chỉ khi bạn dùng một trong những API này, React mới thực sự render concurrently cho update đó:

| Feature                                        | Lane được gán    | Behavior                                       |
| ---------------------------------------------- | ---------------- | ---------------------------------------------- |
| `startTransition` / `useTransition`            | `TransitionLane` | Interruptible, có thể discard stale work       |
| `useDeferredValue`                             | `IdleLane`       | Lowest priority, luôn yield cho urgent work    |
| Suspense trong transitions                     | `RetryLane`      | Giữ UI cũ trong khi load, không flash fallback |
| `useSyncExternalStore` với concurrent features | Tùy context      | Tương thích với concurrent rendering           |

**Mọi `setState` khác** — dù đã `createRoot()` — vẫn là **synchronous render** (`DefaultLane`), không thể interrupt.

#### 4.8.5 Tại Sao React Team Làm Vậy?

Lý do chính là **gradual adoption** và **không breaking change** :

1. **Không cần rewrite toàn bộ app** — code cũ vẫn chạy y chang
2. **Không cần fix hết StrictMode warnings trước** — chỉ cần fix phần dùng concurrent features
3. **Concurrent rendering chỉ có lợi khi dùng đúng** — dùng sai còn tệ hơn (ví dụ: wrap form submit trong transition)
4. **Debugging dễ hơn** — developer biết chính xác phần nào đang concurrent

## 8. Debugging Scenario

**Bug 1: "startTransition không hoạt động — vẫn còn lag"**

```typescript
// ✅ Option 1: Fetch bên ngoài transition, setState bên trong
async function handleSearch(query: string) {

  // setState đồng bộ bên trong transition

    setResults(data)  // ← Đây là transition
```

**Task:** Migrate component search từ debounce pattern sang React 18 concurrent approach.

**Before — React 17 style với debounce:**

```tsx
// components/ProductSearch.tsx — codebase cũ

  id: string;
  name: string;
  category: string;
  price: number;
```

**After — React 18 concurrent approach:**

```typescript
// components/ProductSearch.tsx — React 18
import { useState, useTransition, useMemo } from 'react'

interface Product {
  id: string
  name: string
  category: string
  price: number
}

function ProductSearch({ products }: { products: Product[] }) {
  const [query, setQuery] = useState('')
  const [filteredProducts, setFilteredProducts] = useState(products)
  const [isPending, startTransition] = useTransition()

  function handleSearch(e: React.ChangeEvent<HTMLInputElement>) {
    const value = e.target.value

    // Urgent: input phải responsive tức thì
    setQuery(value)

    // Non-urgent: filter operation — có thể delay
    startTransition(() => {
      const result = products.filter(p =>
        p.name.toLowerCase().includes(value.toLowerCase())
      )
      setFilteredProducts(result)
    })
  }

  return (
    <div>
      <input
        value={query}
        onChange={handleSearch}
        placeholder="Tìm sản phẩm..."
      />

      {/* isPending thay thế isSearching — không cần debounce state */}
      {isPending && <span>Đang lọc...</span>}

      <div style={{ opacity: isPending ? 0.7 : 1 }}>
        <ProductList items={filteredProducts} />
      </div>
    </div>
  )
}
```

**So sánh trực tiếp:**

```
Debounce approach:
  - Artificial 300ms delay dù user ngừng gõ rồi
  - isSearching state quản lý thủ công
  - Nếu user gõ rất nhanh < 300ms: kết quả sai timing
  - Cần cleanup debounce (memory leak nếu quên)
  - Filter vẫn chạy trên main thread đồng bộ → có thể vẫn jank

startTransition approach:
  - Input update NGAY (0ms delay)
  - isPending tự động track
  - React tự discard stale transitions
  - Không cần cleanup
  - Filter được schedule với low priority — yielding cho input events
```

**Migration checklist:**

- ✅ Xóa `debounce` / `throttle` wrapper
- ✅ Tách setState thành urgent (input) và non-urgent (filter/heavy computation)
- ✅ Wrap non-urgent setState trong `startTransition`
- ✅ Replace `isLoading` manual state bằng `isPending` từ `useTransition`
- ✅ Verify không có async trong startTransition callback
- ✅ Test trên slow device (Chrome DevTools → CPU 4x slowdown)

## 10. Architecture Notes

**Khi nào NÊN dùng startTransition:**

```
✅ Dùng startTransition cho:
  - Search/filter với large dataset (> 500 items)
  - Tab switching với heavy component tree
  - Sort/reorder expensive list
  - Toggle phức tạp ảnh hưởng nhiều components
  - Navigation (App Router tự làm, nhưng nếu custom navigation)
  - Lazy loading với React.lazy (dùng startTransition để avoid spinner flash)
```

**Khi nào KHÔNG NÊN dùng:**

```
🔴 Không dùng startTransition cho:
  - Form input state (phải instant)
  - Button click feedback (user expect immediate)
  - Error messages (phải hiện ngay)
  - Animation triggers (phải đúng timing)
  - Checkout / payment flow (urgent UX)
  - Real-time data (stock price, live chat)
```

**startTransition vs useDeferredValue — Decision tree:**

```
Bạn kiểm soát điểm update (event handler, button click)?
  ├── Có → startTransition / useTransition
  └── Không (nhận từ prop, context)?
        ├── Value đến từ prop → useDeferredValue
        └── Cần đọc context → useDeferredValue với context value
```

**Concurrent Rendering + Suspense — Tương tác quan trọng:**

```typescript
// startTransition ngăn Suspense fallback flash khi navigate nhanh
function NavLink({ href, children }: { href: string; children: React.ReactNode }) {
  const router = useRouter()
  const [isPending, startTransition] = useTransition()

  return (
    <Link
      href={href}
      onClick={(e) => {
        e.preventDefault()
        startTransition(() => {
          router.push(href)
        })
      }}
      aria-busy={isPending}
    >
      {children}
    </Link>
  )
}

// Không có startTransition:
//   Click → trang mới suspend → loading.tsx hiện ngay → data xong → trang mới
//   Nếu load nhanh: loading.tsx flash rồi biến mất → UX xấu

// Với startTransition:
//   Click → trang mới suspend → GIỮ trang cũ → data xong → trang mới
//   Loading.tsx chỉ hiện nếu quá chậm → UX mượt
```

**Concurrent Rendering không phải silver bullet:**

```
Concurrent Rendering GIẢI QUYẾT:
  ✅ Input lag khi render heavy component cùng lúc
  ✅ Navigation flash spinner
  ✅ Jank khi update multiple states

Concurrent Rendering KHÔNG GIẢI QUYẾT:
  🔴 Slow network fetch (vẫn phải đợi server)
  🔴 Expensive computation không phải render (Web Worker cần thiết)
  🔴 Memory leak (vẫn phải cleanup effects, subscriptions)
  🔴 Truly slow component tree (vẫn cần optimization: memoization, virtualization)
```

## 11. Interview Questions

**Junior Level**

**Q1: React 18 Concurrent Rendering là gì? Tại sao nó quan trọng?**

<details>
<summary>Đáp án</summary>

Concurrent Rendering là khả năng của React 18 cho phép "interruptible rendering" — React có thể tạm dừng việc render một component tree để xử lý update có mức ưu tiên cao hơn.

Trước React 18, rendering là synchronous và uninterruptible: một khi React bắt đầu render, nó phải hoàn thành toàn bộ trước khi browser có thể làm bất cứ điều gì khác. Với component tree lớn, điều này gây ra UI freeze và input lag.

React 18 với Concurrent Rendering:

- Chia render work thành units nhỏ (Fiber)
- Sau mỗi unit, yield cho browser xử lý user events
- Đánh ưu tiên: user input (high) > data updates (normal) > background work (low)
- Kết quả: UI luôn responsive dù có render work nặng

Quan trọng với Next.js vì App Router navigation, Suspense streaming, và `loading.tsx` behavior đều dựa trên concurrent rendering.

</details>

**Q2: `startTransition` là gì và khi nào nên dùng?**

<details>
<summary>Đáp án</summary>

`startTransition` là API để đánh dấu một state update là "non-urgent". React sẽ xử lý update đó với priority thấp hơn, có thể delay hoặc interrupt nếu cần xử lý việc khẩn hơn.

Dùng khi:

- Search/filter large dataset
- Tab switching với heavy content
- Navigation (Next.js `<Link>` tự dùng internally)
- Bất kỳ update nào mà một chút delay là acceptable

Không dùng khi:

- Form input (phải instant)
- Button click response ngay
- Error states
- Payment/checkout flow

Ví dụ cơ bản:

```typescript
const [isPending, startTransition] = useTransition();

function handleSearch(query: string) {
  setQuery(query); // Urgent: input update ngay

  startTransition(() => {
    setResults(filter(data, query)); // Non-urgent: có thể trễ
  });
}
```

`isPending` cho biết transition đang xử lý, dùng để show subtle loading indicator.

</details>

**Mid Level**

**Q3: Sự khác biệt giữa `startTransition` và `useDeferredValue` là gì? Khi nào dùng cái nào?**

<details>
<summary>Đáp án</summary>

Cả hai đều đánh dấu work là "low priority" nhưng từ góc độ khác nhau:

`startTransition` / `useTransition` — bạn kiểm soát event:

```typescript
const [isPending, startTransition] = useTransition();

function handleClick() {
  startTransition(() => {
    setExpensiveState(newValue); // Bạn quyết định KHI NÀO mark là transition
  });
}
```

`useDeferredValue` — bạn nhận value từ bên ngoài:

```typescript
function ChildComponent({ value }) {
  const deferredValue = useDeferredValue(value);
  // value cập nhật ngay (high priority)
  // deferredValue cập nhật sau (low priority)
  const result = useMemo(() => expensiveCalc(deferredValue), [deferredValue]);
}
```

Chọn theo:

- Có access vào event handler / điểm trigger → `startTransition`
- Chỉ nhận prop/context, không kiểm soát khi nào update → `useDeferredValue`
- Cần `isPending` để show loading indicator → `startTransition`

Ví dụ quyết định: parent search page kiểm soát query → dùng `useTransition`. Child component nhận query như prop và filter data → dùng `useDeferredValue`.

</details>

**Q4: Tại sao không được dùng async/await bên trong startTransition callback?**

<details>
<summary>Đáp án</summary>

`startTransition` chỉ mark những `setState` được gọi **synchronously** trong callback của nó.

Khi có `await`, code sau `await` chạy trong một microtask/callback khác — không còn trong synchronous execution context của `startTransition`. React không còn "biết" những setState đó là transition.

```typescript
startTransition(async () => {
  // Phần này là transition ✅
  setLoading(true)

  const data = await fetch(...)  // ← Sau đây thoát khỏi transition context

  // Phần này KHÔNG phải transition 🔴 — chạy như DefaultLane
  setResults(data)
})
```

Fix: fetch bên ngoài, setState đồng bộ bên trong:

```typescript
const data = await fetch(...)  // Fetch bên ngoài
startTransition(() => {
  setResults(data)  // setState đồng bộ bên trong ← transition ✅
})
```

Lưu ý: React 19 đang thử nghiệm async transitions nhưng chưa stable rộng rãi tại thời điểm này.

</details>

**Senior Level**

**Q5: Giải thích cơ chế "time slicing" hoạt động như thế nào trong React. React yield cho browser bằng cơ chế nào?**

<details>
<summary>Đáp án</summary>

Time slicing là kết quả của Fiber architecture kết hợp với React Scheduler.

Fiber chia render tree thành từng "units of work". React Scheduler có một work loop duyệt qua từng fiber:

```javascript
// Đơn giản hóa
function workLoop() {
  while (nextFiber && hasTimeRemaining()) {
    nextFiber = processOneFiber(nextFiber);
  }

  if (nextFiber) {
    // Chưa xong, schedule tiếp
    scheduleCallback(workLoop);
  }
}
```

`hasTimeRemaining()` check xem còn thời gian trong "frame" hiện tại không. React target 5ms chunks. Nếu hết budget, nó yield và schedule tiếp.

React dùng `MessageChannel.postMessage()` để yield, không phải `requestIdleCallback`:

- `requestIdleCallback`: latency ~50ms, chỉ gọi khi browser idle → quá chậm cho UI
- `MessageChannel`: latency ~0ms, gọi ngay sau current task queue → tốt hơn nhiều

Flow:

1. React bắt đầu work loop qua `MessageChannel` message
2. Xử lý fibers trong 5ms
3. Post message mới để schedule tiếp
4. Browser xử lý user events trong khoảng giữa hai messages
5. Nếu có high-priority update → React interrupt low-priority work
6. Xử lý high-priority → resume (hoặc restart) low-priority

Điều quan trọng: time slicing chỉ ảnh hưởng đến **concurrent renders** (transitions, deferred). Synchronous renders vẫn chạy đến cuối.

</details>

**Q6: Trong Next.js App Router, "concurrent rendering" ảnh hưởng đến behavior của navigation như thế nào? Tại sao `loading.tsx` không phải lúc nào cũng hiện ngay khi click Link?**

<details>
<summary>Đáp án</summary>

Next.js App Router `<Link>` component tự động wrap navigation trong `startTransition`. Đây là lý do tại sao `loading.tsx` không flash ngay khi click.

Flow khi click `<Link>`:

1. Click → Next.js gọi `startTransition(() => router.push(href))`
2. React đánh dấu đây là "transition" (low priority)
3. Trang mới bắt đầu render, gặp async Server Component → suspend
4. **Vì là transition**: React GIỮ trang hiện tại thay vì show `loading.tsx` ngay
5. React đợi trang mới resolve

Nếu trang mới resolve nhanh (< threshold, thường ~1-2 giây):

- React swap trang cũ → mới trực tiếp, không flash `loading.tsx`

Nếu trang mới chậm:

- React eventually show `loading.tsx` fallback (timeout mechanism)
- User thấy loading state sau một chút, không ngay lập tức

Hệ quả cho developers:

- `loading.tsx` không nhất thiết hiện với fast connections
- Nếu muốn luôn show loading: dùng `router.push` mà không wrap trong transition
- Nếu muốn custom progress bar: dùng `useTransition` + `router.push` + `isPending`
- Trên slow connections: `loading.tsx` vẫn hiện sau timeout → đảm bảo UX

</details>

**Vấn đề 1: `async` trong `startTransition` callback — `handleSort`**

Ironically, trong trường hợp này code vẫn "hoạt động" vì không có `await` thực sự — tất cả setState đều đồng bộ. Nhưng đây là pattern nguy hiểm vì nếu sau này ai đó thêm `await` vào, setState sau `await` sẽ không còn là transition. Nên xóa `async` đi.

```typescript
    setSortOrder(newOrder);
```

**Vấn đề 2: Filter input bị wrap trong transition — `handleFilter`**

```typescript
function handleFilter(e: React.ChangeEvent<HTMLInputElement>) {
  startTransition(() => {
    setFilterText(e.target.value); // ← BUG: Input state bị defer!
    setPage(1);
  });
}
```

`setFilterText` là input state — phải update ngay lập tức để input field phản hồi ngay. Khi bọc trong `startTransition`, React có thể delay update này, input sẽ "lag" — thấy chữ không xuất hiện ngay khi gõ.

Fix: tách urgent update ra khỏi transition:

```typescript
function handleFilter(e: React.ChangeEvent<HTMLInputElement>) {
  const value = e.target.value;

  setFilterText(value); // Urgent: input phải responsive ngay

  startTransition(() => {
    setPage(1); // Non-urgent: reset page có thể delay
    // (Nếu filter logic heavy, tách ra component riêng dùng useDeferredValue)
  });
}
```

Tóm lại: `handlePageChange` là pattern đúng. `handleSort` đúng logic nhưng có async không cần thiết. `handleFilter` sai vì mix urgent với non-urgent state trong transition.

**Exercise 2 — Thiết kế**

Bạn đang build một IDE-like editor trong browser với các features:

- Code editor (input — user gõ code)
- Syntax highlighting (re-process khi code thay đổi)
- Error linting (chạy language server, expensive)
- Preview pane (render output, rất expensive — ~200ms)
- File tree sidebar (update khi save, nhanh)

Thiết kế state management và concurrent strategy cho các features này. Giải thích lý do cho từng quyết định.

<details>
<summary>Đáp án</summary>

```typescript
'use client'

import { useState, useTransition, useDeferredValue, useMemo } from 'react'

function CodeEditor() {
  // STATE ARCHITECTURE:

  // 1. Code content — urgent, phải update ngay khi gõ
  const [code, setCode] = useState('')

  // 2. Transitions cho những thứ expensive
  const [isPending, startTransition] = useTransition()

  // 3. Preview state — update chậm nhất, transition
  const [previewKey, setPreviewKey] = useState(0)

  // 4. File tree — riêng biệt, không liên quan concurrent strategy của editor
  const [fileTree, setFileTree] = useState(initialFileTree)

  // DERIVED STATE với useDeferredValue:

  // Syntax highlighting — re-process khi code thay đổi
  // code update ngay (input responsive)
  // deferredCode lags → highlight không block typing
  const deferredCode = useDeferredValue(code)

  const highlightedCode = useMemo(
    () => applySyntaxHighlighting(deferredCode),
    [deferredCode]
  )

  // Linting — expensive, dùng deferred
  const lintResults = useMemo(
    () => runLinter(deferredCode),
    [deferredCode]
  )

  function handleCodeChange(newCode: string) {
    // URGENT: update code state ngay — input field responsive
    setCode(newCode)

    // NON-URGENT: trigger preview re-render
    // Preview ~200ms render, không cần ngay khi gõ
    startTransition(() => {
      setPreviewKey(k => k + 1)  // Force preview re-render
    })
  }

  function handleSave() {
    // Save là urgent action — user expects immediate feedback
    // KHÔNG dùng startTransition
    saveFile(code)
    updateFileTree(fileTree, currentFile)
    setFileTree(updated => updated)  // Sync update
  }

  return (
    <div className="ide">
      {/* Input — luôn responsive vì code state update ngay */}
      <CodeInput
        value={code}
        onChange={handleCodeChange}
        highlightedCode={highlightedCode}  // deferredCode derived
      />

      {/* Lint errors — dùng deferredCode, có thể hơi trễ khi typing nhanh */}
      <LintPanel results={lintResults} />

      {/* Preview — expensive, update với transition */}
      <PreviewPane
        key={previewKey}
        code={deferredCode}  // Dùng deferred để không block typing
        isPending={isPending}
      />

      {/* File tree — update riêng biệt, không cần concurrent strategy */}
      <FileTree files={fileTree} />
    </div>
  )
}
```

**Giải thích từng quyết định:**

`code` state update synchronously vì đây là input state — mọi keystroke phải phản hồi ngay. Nếu delay, user thấy cursor lag.

`useDeferredValue(code)` cho syntax highlighting và linting vì: cả hai là expensive computation (đặc biệt linting với large files), và hơi trễ khi typing nhanh là acceptable. User không cần highlight cập nhật cho TỪNG keystroke — cập nhật mỗi vài ms là đủ.

`startTransition` cho preview vì: preview ~200ms render, nếu không có transition, mỗi keystroke sẽ trigger 200ms render block. Với transition, React chỉ commit preview khi không có input pending. Dùng `isPending` để show "Đang render..." overlay trong preview pane.

File tree KHÔNG cần concurrent strategy vì: chỉ update khi save (không phải real-time), ít data, render nhanh.

`handleSave` KHÔNG dùng startTransition vì: save là intentional user action, user expect immediate feedback (file saved indicator, tree update). Delay sẽ confuse user.

</details>

Production bug: "Sau khi chúng tôi upgrade lên React 18, một số component bắt đầu render 2 lần trong development mode. Đây có phải bug không?"

```typescript
'use client'

function OrderCounter() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    // Trong dev mode, đây log 2 lần: "Effect ran"
    console.log('Effect ran')

    const subscription = subscribeToOrders((order) => {
      setCount(c => c + 1)
    })

    return () => {
      subscription.unsubscribe()
    }
  }, [])

  return <div>Orders: {count}</div>
}
```

Giải thích hiện tượng, đây có phải bug không, và cần sửa gì.

<details>
<summary>Đáp án</summary>

**Không phải bug — đây là intentional behavior của React 18 Strict Mode.**

React 18 Strict Mode (chỉ trong development) double-invoke effects để phát hiện side effects không được cleanup đúng cách:

```
Render → Mount → Effect runs → Cleanup → Effect runs lại
```

Đây là cách React 18 "stress test" components để tìm bugs tiềm ẩn trước production.

**Tại sao React 18 thêm behavior này?**

Concurrent Rendering có thể mount/unmount components nhiều hơn trước (ví dụ: khi interrupt và retry renders). Nếu cleanup không đúng, app sẽ có memory leaks hoặc bugs kỳ lạ trong production.

**Code hiện tại có vấn đề không?**

Nhìn vào code: cleanup có `subscription.unsubscribe()`. Điều này có vẻ đúng.

Nhưng vấn đề tiềm ẩn: nếu `subscribeToOrders` không idempotent — nếu subscribe 2 lần mà chỉ unsubscribe 1 lần thì có memory leak. Cần verify.

**Nếu code đúng (cleanup hoàn chỉnh): không cần sửa gì**, double invoke trong dev chỉ verify cleanup hoạt động.

**Nếu thấy behavior khác nhau giữa lần 1 và lần 2**: đó là signal có bug thật cần fix:

```typescript
useEffect(() => {
  console.log('Effect ran');

  // Đảm bảo subscription là truly idempotent
  const subscription = subscribeToOrders((order) => {
    setCount((c) => c + 1);
  });

  // Cleanup phải undo TẤT CẢ những gì effect làm
  return () => {
    subscription.unsubscribe();
    console.log('Cleanup ran'); // Verify cleanup chạy
  };
}, []);
```

**Rule to remember:** Nếu thấy effect chạy 2 lần trong dev và cleanup → run → re-run là bình thường, không phải bug. Đây là React 18 verify code của bạn đúng. Behavior này chỉ có trong development, không có trong production.

Nếu double invoke gây ra UI sai hoặc lỗi trong dev: đó là bug thật cần fix trước khi production. React đang giúp bạn phát hiện sớm.

## 13. Summary

**Concurrent Rendering = React học cách không độc chiếm main thread.**

Những điều cốt lõi cần nhớ:

**Cơ chế:** Fiber architecture chia render thành units of work. Work loop xử lý từng unit, check thời gian còn lại, yield cho browser nếu budget hết. Dùng `MessageChannel` để schedule tiếp sau khi yield. Kết quả: browser không bao giờ bị block > ~5ms.

**Priority system (Lanes):** Mỗi update có priority. User events (DefaultLane) > Transitions (TransitionLane) > Deferred (IdleLane). React xử lý high priority trước, có thể interrupt và discard low priority work đã cũ.

**APIs:**

- `startTransition` / `useTransition`: wrap non-urgent setState, cho phép interrupt, có `isPending` để track
- `useDeferredValue`: nhận value từ prop/context, tạo deferred copy với low priority
- Automatic batching: React 18 batch mọi setState không kể context

**Giới hạn bắt buộc nhớ:**

- `startTransition` callback phải synchronous — async không được mark là transition
- Không wrap urgent updates (input state, errors, form submit) trong transition

**Trong Next.js App Router:**

- `<Link>` tự wrap navigation trong `startTransition`
- `loading.tsx` không flash ngay vì navigation là transition
- `isPending` từ custom `useTransition` + `router.push` cho phép custom progress indicator

**Concurrent Rendering không fix:** slow network, expensive computation ngoài render (dùng Web Worker), hay truly massive component tree (vẫn cần memoization + virtualization). Nó chỉ cải thiện **perceived performance** bằng cách ưu tiên đúng work.

Bài tiếp theo M1.6 sẽ đi vào **Hydration** — khi HTML từ server gặp React trên client, chuyện gì xảy ra, tại sao mismatch errors xuất hiện, và tại sao Concurrent Rendering + Suspense thay đổi cách hydration hoạt động từ "all-at-once" sang "selective". Đây là bài ghép nối giữa server rendering và client rendering — nền tảng để hiểu App Router architecture.
