# M1.2: Server Components vs Client Components

> 🎯 **Lesson Goal:** Phân biệt Server Component và Client Component về execution model, xác định đúng chỗ đặt `'use client'`, giải thích composition pattern, và nhận diện `'use client'` là React feature không phải Next.js.

---

## 1. Evolution Context

Tôi từng review 1 PR mà reviewer trước tôi approve: page chi tiết sản phẩm chỉ render tên, giá, ảnh — không 1 dòng interactivity nào — nhưng route đó kéo theo 340KB JS xuống client. Lý do: cả file được viết kiểu component class cũ rồi convert function component, không ai nghĩ tới việc tách phần "chỉ hiển thị" ra khỏi phần "cần tương tác". Đó chính xác là Root Problem bài này giải quyết.

**Root problem:** Gửi quá nhiều JavaScript không cần thiết xuống browser. Các component chỉ có nhiệm vụ fetch data và render tĩnh vẫn bị bundle thành JS gửi cho client, gây chậm initial load.

**Cách cũ (SSR/CSR truyền thống):** Dùng `getServerSideProps` hoặc `useEffect` ở client.

**Tại sao thất bại:** Dù server có render HTML, browser vẫn phải download + parse toàn bộ JS logic của component đó để "hydrate" — quá trình gắn lại event listener và state vào HTML đã render sẵn. Việc này triệt tiêu lợi ích SSR: bạn trả tiền cho cả 2 — server render HTML, VÀ client tải/parse JS để làm lại gần như cùng việc.

**Timeline:** Pages Router (SSR = Server render HTML + Client hydrate toàn bộ JS) → React 18 RFC (Tách biệt Server logic khỏi Client bundle) → Next.js 13+ (Stable implementation của RSC).

Bài này, team sẽ học cách quyết định: phần nào của component tree thực sự cần "sống" trên browser (state, event, browser API), phần nào chỉ cần render 1 lần trên server rồi gửi HTML đi — không cần JS theo kèm.

---

## 2. Mental Model

> 💡 **[MODEL]** "Server Component First": Mặc định mọi file là SC. Chỉ thêm `'use client'` ở leaf nodes (node lá) khi tuyệt đối cần interactivity (onclick, state, hooks).

> 🍽️ **[ANALOGY]** Hãy tưởng tượng 1 nhà hàng: Server Component là đầu bếp ở bếp (server) — nấu xong đưa món ra, không đi theo khách về nhà. Client Component là bếp mini trên bàn ăn — khách tự nấu tiếp (interactivity), nhưng món khởi đầu vẫn được phục vụ sẵn (pre-render). `'use client'` là ranh giới giữa bếp chính và bếp mini.

Tại sao mental model này quan trọng: nếu bạn coi `'use client'` như 1 "tính năng để bật" mỗi khi không chắc, bạn sẽ rơi vào Scenario 1 (mục 8) — bundle phình to mà không ai biết tại sao. Nếu bạn nghĩ "Client Component nghĩa là nó không chạy ở server", bạn sẽ rơi vào AP4 (mục 4) và debug sai hướng khi gặp hydration mismatch.

Ghi nhớ ngắn: câu hỏi không phải "component này có dùng React không?" — mọi component đều dùng React. Câu hỏi đúng là "component này có cần sống tiếp trên browser sau khi HTML đã render chưa?". Nếu không — để nó là SC, đừng đụng vào.

## 3. Core Concepts

### 3.1 Server Component (SC)

**[FACT]** Server Components không có lifecycle hooks (`useState`, `useEffect`), không truy cập Browser APIs (`window`, `document`), nhưng có thể là `async` component. (Source: React RSC Docs)

**[FACT]** SC render server-side, không gửi JS bundle cho component đó xuống client. (Source: `nextjs.org/docs/app/building-your-application/rendering/server-components`)

```ts [src/app/dashboard/user-profile.tsx]
// SC mặc định — không cần directive nào cả, đây CHÍNH LÀ default
// behavior của mọi file trong App Router. Bạn không "chọn" SC, bạn
// chỉ "không chọn" CC.
import { getUserProfile } from "@/lib/users";
import { logger } from "@/lib/logger";

interface UserProfileProps {
  userId: string;
}

export async function UserProfile({ userId }: UserProfileProps) {
  try {
    // Direct DB/filesystem access — SC chạy trên server nên có thể
    // query DB trực tiếp, không cần qua 1 REST endpoint trung gian.
    const profile = await getUserProfile(userId);
    return (
      <div>
        <h2>{profile.displayName}</h2>
        <p>{profile.bio}</p>
      </div>
    );
  } catch (error) {
    logger.error({ err: error, userId }, "Failed to load user profile");
    throw error; // error.tsx (chi tiết ở M1.3) xử lý UI fallback — không swallow ở đây.
  }
}
```

### 3.2 Client Component (CC)

**[FACT]** Client Components vẫn được pre-render trên server để tạo HTML ban đầu, sau đó mới hydrate trên browser. (Source: `nextjs.org/docs/app/building-your-application/rendering/server-components`)

```ts [src/components/profile/like-button.tsx]
"use client";
// "Client" ở đây không có nghĩa "chỉ chạy ở browser" — component này
// chạy 2 lần: 1 lần trên server để render HTML, 1 lần trên browser
// để hydrate (gắn lại event listener, state).
import { useState } from "react";
import { logger } from "@/lib/logger";

interface LikeButtonProps {
  postId: string;
  initialLiked: boolean;
}

export function LikeButton({ postId, initialLiked }: LikeButtonProps) {
  const [liked, setLiked] = useState(initialLiked);
  const [isPending, setIsPending] = useState(false);

  async function handleClick() {
    setIsPending(true);
    try {
      const response = await fetch(`/api/posts/${postId}/like`, {
        method: "POST",
      });
      if (!response.ok) {
        throw new Error(`Like request failed with status ${response.status}`);
      }
      setLiked((prev) => !prev);
    } catch (error) {
      // KHÔNG swallow error — log để debug, rollback UI nếu request fail
      // để tránh optimistic UI lệch khỏi trạng thái thật trên server.
      logger.error({ err: error, postId }, "Failed to toggle like");
    } finally {
      setIsPending(false);
    }
  }

  return (
    <button onClick={handleClick} disabled={isPending}>
      {liked ? "Liked" : "Like"}
    </button>
  );
}
```

### 3.3 `'use client'` Directive — Boundary Marker

**[FACT]** `'use client'` là một React boundary marker, KHÔNG phải Next.js invention. Nó đánh dấu ranh giới phía trên cùng của subtree đó thuộc về client bundle. (Source: `react.dev/reference/rsc/server-components`)

```ts [src/components/theme/theme-toggle.tsx]
"use client";
// Boundary marker nghĩa là: MỌI module được import từ file này trở
// đi đều thuộc về client bundle — dù module đó tự thân không cần
// interactivity gì cả.
import { useState } from "react";
import { ThemeIcon } from "./theme-icon"; // ThemeIcon trở thành CC vì
// nằm dưới boundary này, dù bản thân nó chỉ render 1 SVG tĩnh.

export function ThemeToggle() {
  const [theme, setTheme] = useState<"light" | "dark">("light");
  return (
    <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
      <ThemeIcon theme={theme} />
    </button>
  );
}
```

### 3.4 Composition Pattern — SC pass CC as children

**[FACT]** CC không thể import SC trực tiếp (SC sẽ bị biến thành CC). SC có thể truyền CC thông qua `children` prop (Composition Pattern). (Source: Next.js Docs: Composition Patterns)

```ts [src/app/dashboard/layout.tsx]
// SC (không có 'use client') truyền CC làm children — đây là pattern
// để giữ Server-rendered content "lồng vào trong" 1 Client Component
// mà không phá vỡ boundary.
import { DashboardSidebar } from "@/components/dashboard/dashboard-sidebar";
import { NotificationProvider } from "@/components/dashboard/notification-provider"; // CC

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  // DashboardSidebar (SC) render xong trên server, kết quả render
  // được "rót" vào slot children của NotificationProvider (CC) —
  // DashboardSidebar KHÔNG bị kéo vào client bundle dù nằm "bên trong" CC.
  return (
    <NotificationProvider>
      <DashboardSidebar />
      {children}
    </NotificationProvider>
  );
}
```

```ts [src/components/dashboard/notification-provider.tsx]
"use client";
// Nếu file này viết `import { DashboardSidebar } from "..."` rồi
// gọi <DashboardSidebar /> trực tiếp ở đây — DashboardSidebar sẽ bị
// kéo vào client bundle, mất hoàn toàn lợi ích SC của nó. Đây là rule
// quan trọng nhất của bài này: CC nhận SC qua props (children), KHÔNG
// import SC.
import { useState } from "react";

export function NotificationProvider({
  children,
}: {
  children: React.ReactNode;
}) {
  const [unreadCount, setUnreadCount] = useState(0);
  return <div data-unread={unreadCount}>{children}</div>;
}
```

### 3.5 Serialization Constraint

**[FACT]** Props truyền từ SC sang CC phải là serializable (chuỗi JSON hợp lệ). Không được truyền functions, Classes, hay Symbols. (Source: React RSC Docs: Passing Server Components to Client Components as props)

```ts [src/app/posts/[id]/page.tsx]
import { LikeButton } from "@/components/profile/like-button";
import { getPostById } from "@/lib/posts";

interface PostPageProps {
  params: Promise<{ id: string }>;
}

export default async function PostPage({ params }: PostPageProps) {
  const { id } = await params;
  const post = await getPostById(id);

  return (
    <article>
      <h1>{post.title}</h1>
      {/* postId là string — serializable, qua boundary an toàn. */}
      <LikeButton postId={post.id} initialLiked={post.likedByCurrentUser} />
      {/* post.publishedAt là Date object — PHẢI .toISOString() trước
          khi đưa qua boundary, không bao giờ pass Date object thô. */}
      <time dateTime={post.publishedAt.toISOString()}>
        {post.publishedAt.toISOString()}
      </time>
    </article>
  );
}
```

### 3.6 React Server Components (RSC) Concept

**[FACT]** RSC là React 18+ feature, không phải Next.js invention. Next.js là framework đầu tiên adopt RSC stable. (Source: React RSC Docs)

```ts [src/lib/rsc-note.ts]
// [NOTE] Đây là ghi chú concept, không phải code chạy thật.
//
// React team tạo RSC để giải quyết đúng Root Problem ở mục 1: một
// component chỉ fetch data và render tĩnh không cần gửi JS logic của
// nó xuống client. Next.js implement RSC trong App Router — nhưng
// bản thân RSC là 1 React primitive (định nghĩa ở cấp React, không
// phải cấp framework), nên khi bạn học RSC, bạn học React, không chỉ
// học "1 feature của Next.js".
```

## 4. Anti-Patterns in the Wild

### AP1: Đặt `'use client'` ở root layout

- ❌ **Code Smell:** Thêm `'use client'` vào `app/layout.tsx` chỉ để dùng 1 hook (VD: theme toggle).

  ```ts [src/app/layout.tsx]
  "use client"; // Chỉ để dùng useState cho theme toggle — nhưng kéo
  // TOÀN BỘ app xuống client bundle, mất sạch RSC benefit.
  import { useState } from "react";

  export default function RootLayout({ children }: { children: React.ReactNode }) {
    const [theme, setTheme] = useState("light");
    return (
      <html lang="en" data-theme={theme}>
        <body>
          <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
            Toggle theme
          </button>
          {children}
        </body>
      </html>
    );
  }
  ```

- ✅ **Staff Refactor:** Giữ layout là SC. Tách hook ra `<ThemeToggleProvider>` component, đánh dấu `'use client'` ở component đó, import vào layout.

  ```ts [src/app/layout.tsx]
  // Trade-off: Tăng 1 file wrapper đổi lại giữ được toàn bộ RSC
  // benefits cho app — mọi Server Component con vẫn render server-side
  // mà không bị kéo vào client bundle.
  import { ThemeToggleProvider } from "@/components/theme/theme-toggle-provider";

  export default function RootLayout({ children }: { children: React.ReactNode }) {
    return (
      <html lang="en">
        <body>
          <ThemeToggleProvider>{children}</ThemeToggleProvider>
        </body>
      </html>
    );
  }
  ```

  ```ts [src/components/theme/theme-toggle-provider.tsx]
  "use client";
  import { useState } from "react";

  export function ThemeToggleProvider({ children }: { children: React.ReactNode }) {
    const [theme, setTheme] = useState<"light" | "dark">("light");
    return (
      <div data-theme={theme}>
        <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
          Toggle theme
        </button>
        {children}
      </div>
    );
  }
  ```

- 🧠 **Why:** Tăng 1 file wrapper đổi lại giữ được toàn bộ RSC benefits cho app.
- 🔥 **Incident Risk:** Toàn bộ app bị chuyển thành Client Component. JS bundle phình to (từ 200KB lên 800KB+), mất hoàn toàn lợi ích RSC.

### AP2: Pass non-serializable props từ SC sang CC

- ❌ **Code Smell:** `<UserCard onClick={handleClick} date={new Date()} />`

  ```ts [src/app/users/[id]/page.tsx]
  import { UserCard } from "@/components/users/user-card";

  export default async function UserPage() {
    function handleClick() {
      console.log("clicked");
    }
    // date là Date object, handleClick là function — cả 2 đều KHÔNG
    // serializable, React throw runtime error ngay khi render SC này.
    return <UserCard onClick={handleClick} date={new Date()} />;
  }
  ```

- ✅ **Staff Refactor:** Truyền primitive values: `<UserCard dateString={new Date().toISOString()} />`. Định nghĩa `handleClick` bên trong CC.

  ```ts [src/app/users/[id]/page.tsx]
  import { UserCard } from "@/components/users/user-card";

  export default async function UserPage() {
    // Trade-off: Mất tính tiện lợi của closure (handleClick phải tự
    // chứa logic riêng bên trong CC) đổi lại tuân thủ serialization boundary.
    return <UserCard dateString={new Date().toISOString()} />;
  }
  ```

  ```ts [src/components/users/user-card.tsx]
  "use client";
  import { logger } from "@/lib/logger";

  interface UserCardProps {
    dateString: string;
  }

  export function UserCard({ dateString }: UserCardProps) {
    function handleClick() {
      try {
        // Logic định nghĩa NGAY tại đây, trong CC — không nhận từ SC qua props.
        console.log("clicked at", dateString);
      } catch (error) {
        logger.error({ err: error }, "UserCard click handler failed");
      }
    }
    return <button onClick={handleClick}>{dateString}</button>;
  }
  ```

- 🧠 **Why:** Mất tính tiện lợi của closure (handleClick phải tự fetch data lại) đổi lại tuân thủ serialization boundary.
- 🔥 **Incident Risk:** Runtime crash: `Error: Only plain objects can be passed to Client Components as props`.

### AP3: Fetching data trong CC với `useEffect`

- ❌ **Code Smell:** `useEffect(() => { fetch('/api/data') }, [])` trong component.

  ```ts [src/components/dashboard/recent-orders.tsx]
  "use client";
  import { useEffect, useState } from "react";

  export function RecentOrders() {
    const [orders, setOrders] = useState([]);
    useEffect(() => {
      // Waterfall: browser phải tải JS bundle, hydrate, RỒI MỚI bắt
      // đầu fetch — user thấy loading spinner dù data có thể có sẵn
      // ngay từ lần render đầu nếu fetch ở server.
      fetch("/api/orders/recent")
        .then((res) => res.json())
        .then(setOrders);
    }, []);
    return <ul>{orders.map((o) => <li key={o.id}>{o.title}</li>)}</ul>;
  }
  ```

- ✅ **Staff Refactor:** Fetch data ở SC parent, truyền xuống CC qua props.

  ```ts [src/app/dashboard/page.tsx]
  import { RecentOrdersList } from "@/components/dashboard/recent-orders-list";
  import { getRecentOrders } from "@/lib/orders";

  export default async function DashboardPage() {
    // Trade-off: Component mất tính "tự chủ" (co-location) — danh
    // sách order không tự fetch được nữa, phụ thuộc page cha — đổi
    // lại tránh waterfall request, tốt cho SEO và UX.
    const orders = await getRecentOrders();
    return <RecentOrdersList orders={orders} />;
  }
  ```

  ```ts [src/components/dashboard/recent-orders-list.tsx]
  interface Order {
    id: string;
    title: string;
  }

  export function RecentOrdersList({ orders }: { orders: Order[] }) {
    // KHÔNG cần 'use client' — chỉ render danh sách tĩnh, mặc định
    // là SC, không tốn JS bundle nào cho component này.
    return (
      <ul>
        {orders.map((order) => (
          <li key={order.id}>{order.title}</li>
        ))}
      </ul>
    );
  }
  ```

- 🧠 **Why:** Component mất tính "tự chủ" (co-location) đổi lại tránh waterfall requests, tốt cho SEO và UX (không thấy loading spinner nhấp nháy).
- 🔥 **Incident Risk:** Server trả về layout rỗng/bố cục bị nhảy (layout shift) khi client bắt đầu fetch data sau khi hydrate.

### AP4: Nhầm "Client Component không chạy trên server"

- ❌ **Code Smell:** Viết code `if (typeof window !== 'undefined')` bừa bãi trong CC vì nghĩ nó không chạy trên server.

  ```ts [src/components/analytics/page-view-tracker.tsx]
  "use client";
  // Junior nghĩ CC "không chạy trên server" nên guard window ngay ở
  // phần render chính — nhưng code này chạy server-render trước,
  // RỒI MỚI hydrate trên browser. screenWidth sẽ khác nhau giữa 2 lần.
  export function PageViewTracker({ pageName }: { pageName: string }) {
    const screenWidth = typeof window !== "undefined" ? window.innerWidth : 0;
    return <div data-page={pageName} data-width={screenWidth} />;
  }
  ```

- ✅ **Staff Refactor:** Hiểu CC chạy 2 lần (Server render HTML → Client hydrate). Chỉ guard browser APIs ở nơi thực sự gọi (VD: trong `useEffect`).

  ```ts [src/components/analytics/page-view-tracker.tsx]
  "use client";
  // Trade-off: Không có trade-off, chỉ là sửa sai mental model.
  import { useEffect, useState } from "react";

  export function PageViewTracker({ pageName }: { pageName: string }) {
    const [screenWidth, setScreenWidth] = useState<number | null>(null);

    useEffect(() => {
      // useEffect CHỈ chạy trên browser, sau khi hydrate xong — đây
      // là nơi đúng để đọc browser API, không phải ở thân hàm render.
      setScreenWidth(window.innerWidth);
    }, []);

    return <div data-page={pageName} data-width={screenWidth ?? "unknown"} />;
  }
  ```

- 🧠 **Why:** Không có trade-off, chỉ là sửa sai mental model.
- 🔥 **Incident Risk:** Lỗi hydration mismatch nếu logic phân nhánh `window` nằm ở level render chính (server render ra `0`, client render ra `1920` → React phát hiện mismatch, log warning, có thể discard hydration của subtree đó).

### AP5: "Server Component = SSR"

- ❌ **Code Smell:** Coi SC chỉ là `getServerSideProps` được rename, nghĩ rằng logic nặng "rồi cũng hydrate trên client".

  ```ts [src/app/reports/page.tsx]
  // SAI hoàn toàn: SC không hydrate, code này chạy 100% trên SERVER,
  // mỗi request — "đẩy cho client" là điều KHÔNG xảy ra với SC.
  import { generateHeavyReportImage } from "@/lib/reports";

  export default async function ReportsPage() {
    const imageBuffer = await generateHeavyReportImage();
    return <img src={`data:image/png;base64,${imageBuffer.toString("base64")}`} />;
  }
  ```

- ✅ **Staff Refactor:** Hiểu RSC output KHÔNG chứa JS bundle cho component đó. SSR output chứa HTML + JS bundle để hydrate.

  ```ts [src/app/reports/page.tsx]
  // Trade-off: N/A (Conceptual correction) — đây không phải lựa chọn
  // kiến trúc, mà là sửa hiểu sai. Việc nặng vẫn phải chạy ở đâu đó:
  // tối ưu thuật toán, cache kết quả, hoặc offload sang background
  // job — KHÔNG có cách nào "đẩy" nó cho client vì SC không bao giờ
  // chạy lại ở client.
  import { getCachedReportImageUrl } from "@/lib/reports";

  export default async function ReportsPage() {
    // Pre-generate và cache ảnh report ở background job, SC chỉ đọc
    // URL đã cache — không tự chạy lại CPU-bound task mỗi request.
    const imageUrl = await getCachedReportImageUrl();
    return <img src={imageUrl} alt="Report" />;
  }
  ```

- 🧠 **Why:** N/A (Conceptual correction) — RSC output không chứa JS bundle cho component đó, nhưng server-side execution cost vẫn 100% tính vào server, không "biến mất" hay "chuyển cho client" như mô hình hydrate của SSR truyền thống.
- 🔥 **Incident Risk:** Đặt logic nặng (image processing, crypto) vào SC nghĩ rằng "nó sẽ hydrate trên client anyway" → tốn tài nguyên server vô ích, mỗi request đều gánh full computation.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |

## 6. Legacy Recognition

Nhận diện 1 codebase đang ở mô hình "tiền-RSC" qua những dấu hiệu sau:

- **Legacy:** `_app.tsx` wrapping toàn bộ app bằng JS function — mọi page đều chạy qua đây, mọi page đều là client bundle theo định nghĩa CC ở bài này.
- **Legacy:** `getServerSideProps` → data fetching pattern cũ — server có render HTML, nhưng component vẫn gửi kèm JS bundle để hydrate, khác hoàn toàn với SC (không gửi JS bundle).
- **Legacy:** `useEffect(() => fetch(...), [])` → client-side data fetching trong Pages Router, tạo waterfall giống AP3.
- **Legacy:** Toàn bộ component tree là Client Component — Pages Router không có khái niệm SC, mọi thứ mặc định hydrate.

---

## 7. Modern Approach

- **Modern:** `layout.tsx` (Server) + Client leaf components — SC làm "khung" và "nguồn data", CC chỉ ở đúng những node lá cần interactivity.
- **Migration `_app.tsx` bọc toàn bộ app (VD: Redux Provider):** Tạo `<Providers>` component đánh dấu `'use client'`, import nó vào `app/layout.tsx` (SC). Giữ `layout.tsx` là SC — đây là Strangler Fig ở cấp component: bọc dần phần cần client logic vào 1 lớp wrapper mỏng, không rewrite cả layout.
- **Migration `getServerSideProps` chuyển thành `useEffect` ở page:** Page trong App Router là SC theo default. Đưa logic fetch từ `useEffect` lên scope của SC page (gọi trực tiếp trong thân `async function`), truyền data xuống CC dạng props.
- **Migration Third-party lib dùng `window` ở module scope, import vào SC:** Dùng dynamic import (`const Lib = dynamic(() => import('lib'))`) bên trong 1 CC wrapper — không import trực tiếp lib đó vào SC, vì module scope của lib chạy ngay lúc import, sẽ throw lỗi `window is not defined` trên server.
- **Rollback plan:** Nếu architecture RSC gây crash hydration phức tạp không fix kịp trong 1 release window, thêm `'use client'` tạm lên từng page bị lỗi cụ thể để rollback về CC behavior — không rollback toàn bộ app về Pages Router. Tối ưu lại boundary sau khi đã ổn định production.

## 8. Production Scenario

### Scenario 1: "Component không có state nhưng vẫn thêm 'use client'"

- **Symptom:** Lighthouse báo JS bundle size quá lớn, Total Blocking Time (TBT) cao.
- **Root cause:** Component không dùng hooks nhưng bị đánh dấu `'use client'`, kéo theo toàn bộ dependency tree của nó vào client bundle.
- **Detect:** Dùng `@next/bundle-analyzer`, check có unexpected packages trong client chunk.
- **Fix:** Audit lại component, bỏ `'use client'` nếu nó không có hook/event handler thật — để nó trở về SC mặc định, và kiểm tra cả dependency mà nó import có tự ý kéo `'use client'` theo không (xem Pitfall ở mục 10).

### Scenario 2: "Hydration mismatch error"

- **Symptom:** Console báo `Text content did not match. Server: "..." Client: "..."`, UI bị flash.
- **Root cause:** SC hoặc CC render khác nhau giữa server và client (VD: dùng `Date.now()`, `Math.random()` trực tiếp trong render).
- **Detect:** Đọc stack trace React, tìm đúng component.
- **Fix:** Move logic generate random/date vào `useEffect` (nếu là CC) hoặc generate ở server truyền xuống dưới dạng giá trị cố định.

### Scenario 3: Lỗi `Error: Functions cannot be passed directly to Client Components`

- **Symptom:** App crash 500 tại build time hoặc runtime khi render SC.
- **Root cause:** SC truyền 1 function handler xuống CC.
- **Detect:** Tìm dòng code `<ClientComp onClick={fn} />`.
- **Fix:** Truyền primitive data xuống, để CC tự định nghĩa function gọi API endpoint riêng của nó — không pass function handler trực tiếp qua serialization boundary.

## 9. Performance & Overhead

Raw material xác nhận rõ: chưa có benchmark number chính thức cho bundle size reduction, hydration time, hay server render latency giữa CC-only và SC+CC composition — đừng bịa số, tự đo trong codebase của bạn theo hướng sau:

**Bundle size: CC-only vs SC + CC Composition**
- Dùng `@next/bundle-analyzer` (`ANALYZE=true next build`), so sánh cột **First Load JS** giữa 1 route hoàn toàn CC (mọi file `'use client'`) và route tương đương đã refactor theo SC-first.
- Quy tắc thực dụng: nếu route không có form/interactivity phức tạp mà First Load JS vượt 100kB, nghi ngờ ngay có `'use client'` đặt sai chỗ — tìm xem có component nào bị kéo vào client bundle "oan" không.

**Hydration time: SSR truyền thống vs RSC**
- Đo bằng React DevTools Profiler hoặc Performance tab > User Timing marks (React tự emit marks này ở development build) để xem khoảng thời gian từ "HTML received" đến "hydration complete".
- Quy tắc thực dụng: route có nhiều SC con, vùng cần hydrate càng nhỏ — Time to Interactive (TTI) giảm tương ứng vì browser parse ít JS hơn để đạt trạng thái tương tác được.

**Server render latency: async SC**
- Đo bằng structured log timestamp trước/sau `await` trong SC (qua logger có sẵn, KHÔNG console.log thô ở production), theo dõi qua APM (Datadog, New Relic, OpenTelemetry).
- Quy tắc thực dụng: nếu 1 SC có `await` tới DB/API chậm (>200ms) mà phần đó không cần block toàn route, tách nó thành component riêng — cơ chế Suspense/streaming để tách phần chậm ra khỏi critical path thuộc về M1.3/M4.1, ở đây bạn chỉ cần nhớ nguyên tắc: đừng để 1 fetch chậm trong SC kéo theo cả page phải chờ.

Số liệu cụ thể phụ thuộc hạ tầng, payload, network thật của bạn — Staff Engineer đo trên production traffic, không tin benchmark generic copy từ blog khác.

## 10. Architecture Notes

- **SC + CC Composition là default architecture:** Mọi project App Router nên bắt đầu với SC-first. Chỉ đánh dấu `'use client'` ở leaf nodes thực sự cần interactivity — không đặt ở giữa cây để "cho chắc".
- **Third-party Libraries & Client Boundaries:** Lib có `"use client"` khai báo trong package export tạo ra 1 Client Boundary. SC import lib đó vẫn là SC, nhưng bundle của lib bị đẩy vào client chunk. (Source: React RFC 0189 - Package Exports)
- **Known Pitfall — `Date.now()` trong SC gây cache issue:** `[THEORY — NEEDS VERIFICATION]` SC render tĩnh có thể bị cache; nếu dùng `Date.now()` trong render, thời gian hiển thị có thể bị đóng băng khi cache hit. Cần xác nhận lại cơ chế cache chính xác của Next.js 16 với dynamic functions trong SC trước khi coi đây là fact chắc chắn.
- **Known Pitfall — Truyền SC qua props không phải `children`:** `[THEORY — NEEDS VERIFICATION]` React hiện chỉ cho phép truyền SC qua prop `children` một cách chắc chắn; truyền qua tên prop tùy ý (VD: `header={<ServerHeader />}`) cần confirm lại chính xác behavior ở React 19/Next 16 trước khi dạy như rule tuyệt đối.
- **Data Flow Design:** SC nên là data fetcher và context provider. CC nên là interactive UI leaf. Không để CC tự fetch data qua `useEffect` nếu có thể fetch ở SC parent rồi truyền xuống.
- **Bundle Analysis định kỳ:** Dùng `@next/bundle-analyzer` như 1 bước trong CI, không chỉ chạy 1 lần lúc setup — mục tiêu: client bundle chỉ chứa interactivity logic, không chứa data-fetching/presentation logic.

## 11. Interview Questions

### Q1: Client Component có thể wrap Server Component không? (Junior)
<details>
<summary>Đáp án</summary>

Có, thông qua prop `children`. SC được render trên server, kết quả HTML được "rót" vào slot `children` của CC, CC chỉ hydrate phần của riêng nó, không hydrate lại nội dung SC bên trong.

</details>

### Q2: Tại sao `'use client'` lại gọi là "boundary" chứ không phải "directive" trên component? (Mid)
<details>
<summary>Đáp án</summary>

Vì nó không đổi bản chất của riêng file hiện tại, mà nó báo cho bundler biết: "từ dòng này trở đi, mọi module được import đều thuộc về client bundle". Nó đánh dấu 1 ranh giới trong cây import, không phải 1 thuộc tính gắn lên 1 component đơn lẻ.

</details>

### Q3: React Server Components khác SSR như thế nào? (Mid)
<details>
<summary>Đáp án</summary>

SSR gửi cả HTML và JS bundle của component để hydrate trên client. RSC chỉ gửi HTML, không gửi JS bundle của component đó xuống client — component đó không bao giờ chạy lại trên browser.

</details>

### Q4: Tại sao `useEffect` không chạy trong Server Component? (Junior)
<details>
<summary>Đáp án</summary>

Vì SC chỉ chạy 1 lần trên server để tạo HTML rồi kết thúc, nó không có lifecycle trên browser — `useEffect` gắn vào lifecycle mount/update/unmount của browser, nên trong ngữ cảnh SC nó hoàn toàn vô nghĩa.

</details>

### Q5: Nếu SC có dữ liệu user, và CC là nút "Follow", làm sao kết nối 2 cái mà không leak sensitive data? (Junior/Mid)
<details>
<summary>Đáp án</summary>

SC chỉ truyền `userId` (giá trị serializable) xuống CC qua props — không truyền nguyên object user đầy đủ nếu nó chứa data nhạy cảm không cần ở client. CC định nghĩa hàm xử lý click ngay trong chính nó, gọi tới 1 API endpoint riêng với `userId` đó. Toàn bộ logic xử lý dữ liệu nhạy cảm nằm trên server, CC chỉ là cái nút gửi ID lên.

</details>

### 🟢 Exercise 1: Phân loại SC vs CC

**Đề bài:** Cho 1 component tree: `Header` (chỉ render logo + nav links), `SearchBar` (có `useState` cho input), `UserProfile` (fetch data từ DB, render tĩnh), `LikeButton` (có `onClick` + `useState`).

**Acceptance Criteria:**
- [ ] Mỗi component được đánh dấu SC hoặc CC, kèm 1 câu lý do (không state/event = SC, có state/event/browser API = CC).
- [ ] Vẽ ra (text hoặc diagram) component tree với ký hiệu phân biệt SC/CC.
- [ ] Xác định vị trí đặt `'use client'` tối ưu nhất — chỉ ở đúng những leaf node cần, không cao hơn mức cần thiết.

<details>
<summary>Hint</summary>

SC = không state, không event handler, có thể `async`. CC = có interactivity thật (state, event, browser API). `'use client'` chỉ đặt ở leaf node cần nhất — nếu `SearchBar` có `useState` nhưng `Header` chỉ render tĩnh bên cạnh nó, đừng đặt `'use client'` ở cấp cha chung của cả hai.

</details>

### 🟡 Exercise 2: Fix Serialization Error

**Đề bài:** Cho đoạn code SC truyền props xuống CC bị lỗi: `<UserCard user={userObject} onFollow={handleFollow} createdAt={new Date()} />`, trong đó `userObject` chứa nested data và `handleFollow` là async function.

**Acceptance Criteria:**
- [ ] Không còn lỗi build/runtime liên quan serialization.
- [ ] `createdAt` được serialize đúng cách trước khi qua boundary.
- [ ] `handleFollow` không còn được pass trực tiếp từ SC — logic tương ứng được định nghĩa lại bên trong CC.
- [ ] CC vẫn hiển thị đúng toàn bộ data cần thiết và vẫn xử lý được hành động follow.

<details>
<summary>Hint</summary>

Serialize `Date` thành string (`.toISOString()`) trước khi pass. Với `userObject` chứa nested data, chỉ pass đúng những field CC thực sự cần hiển thị — đừng pass nguyên object nếu không cần, càng ít field qua boundary càng ít rủi ro. `handleFollow` định nghĩa lại bên trong CC, gọi tới 1 API endpoint riêng.

</details>

### 🔴 Exercise 3: Biến 1 Form Phức tạp thành SC + CC Leaf Architecture

**Đề bài:** Cho sẵn 1 form đăng ký (toàn bộ là Client Component) gồm: validation real-time, dropdown tỉnh/thành phố fetch từ API, file upload preview, captcha. Refactor thành kiến trúc SC + CC leaf hợp lý.

**Acceptance Criteria:**
- [ ] SC chịu trách nhiệm fetch danh sách tỉnh/thành phố và render form shell.
- [ ] CC chỉ còn ở đúng những phần cần: input field cần validation real-time, file upload preview, captcha widget.
- [ ] Không còn `'use client'` ở component nào không thật sự cần (form container không nhất thiết phải là CC).
- [ ] `@next/bundle-analyzer` cho thấy First Load JS của route này giảm rõ rệt so với bản gốc toàn CC.
- [ ] Form vẫn hoạt động đầy đủ chức năng sau refactor.

<details>
<summary>Hint</summary>

Danh sách tỉnh/thành phố fetch ở SC, truyền xuống CC dạng array các giá trị primitive (string/number), không truyền nguyên response object thô nếu không cần hết field. File upload và captcha bắt buộc là CC vì cần browser API (File, canvas...). Validation real-time có thể tách thành 1 CC riêng nhỏ cho từng input, không bọc cả form trong `'use client'`.

</details>

## 13. Summary

- Mặc định mọi component là Server Component — chỉ thêm `'use client'` ở leaf nodes thực sự cần interactivity.
- `'use client'` là boundary marker cho bundler, không phải decorator — mọi import bên dưới nó đều thành client bundle.
- CC vẫn pre-render trên server rồi mới hydrate — đừng nhầm "client" với "chỉ chạy ở browser".
- Props từ SC sang CC phải serializable — không pass function, class instance, hay symbol qua boundary.
