# M1.3: Loading, Error, và Streaming UI

> 🎯 **Lesson Goal:** Implement `loading.tsx` để hiển thị skeleton trong khi async Server Component fetch data, implement `error.tsx` với `reset()` để handle lỗi runtime, và giải thích `Suspense` boundary auto-wrap vs manual wrap.

---

## 1. Evolution Context

Tôi từng debug 1 dashboard mà document HTML mất 4.8 giây mới bắt đầu download — không phải vì server chậm, mà vì 1 API "lịch sử đơn hàng" treo 4 giây trong khi 2 widget còn lại của page đã sẵn sàng từ giây đầu tiên. User nhìn màn hình trắng suốt 4.8 giây, dù 80% nội dung page không liên quan gì đến API chậm đó.

**Root problem:** Users phải nhìn vào một màn hình trống (hoặc toàn bộ spinner) trong khi server chờ *tất cả* data load xong mới trả về HTML.

**Cách cũ (Blocking SSR / Client Fetching):** Dùng `getServerSideProps` hoặc `useEffect` + `isLoading` state.

**Tại sao thất bại:** Tính chất "all-or-nothing". Nếu 1 API trong page chậm 5 giây, toàn bộ page bị block 5 giây, dù các phần khác đã sẵn sàng. Server không có cách nào nói "tôi xong phần này rồi, gửi trước đã" — phải chờ tất cả mới gửi gì cả.

**Timeline:** Blocking SSR (Next.js < 13) → React ra mắt Suspense concept → Next.js 13+ App Router áp dụng Streaming SSR (gửi HTML từng phần qua `loading.tsx` và manual `<Suspense>`).

Bài này, team học cách "cắt" 1 page thành nhiều phần độc lập về tốc độ load, để phần nhanh không phải chờ phần chậm — và học cách xử lý khi 1 phần đó throw lỗi giữa đường.

---

## 2. Mental Model

> 🏭 **[ANALOGY]** `<Suspense>` giống như "khe cắm placeholder" trong dây chuyền lắp ráp. Phần còn lại của sản phẩm vẫn xuất xưởng (stream) trong khi linh kiện đó đang được gia công.

Tại sao mental model này quan trọng: nếu bạn coi cả page là 1 khối duy nhất phải hoàn thiện 100% mới được "xuất xưởng" (gửi response), bạn sẽ rơi vào AP2 (mục 4) — page chờ widget chậm nhất dù phần còn lại đã sẵn sàng. `<Suspense>` cho phép bạn đánh dấu "khe trống" — phần đó cứ để placeholder chạy trước, dây chuyền không dừng lại chờ nó.

Ghi nhớ ngắn: `loading.tsx` là khe trống cho NGUYÊN cả route segment. Manual `<Suspense>` là khe trống cho TỪNG phần nhỏ bên trong 1 page. Chọn cái nào tùy vào việc bạn muốn "chờ cả dây chuyền" hay "chỉ chờ đúng linh kiện đang chậm".

## 3. Core Concepts

### 3.1 `loading.tsx` — Auto-Suspense

**[FACT]** `loading.tsx` được Next.js tự động wrap trong `<Suspense>`. Nó trigger khi route segment đang load (async Server Components chưa resolve xong). (Source: `nextjs.org/docs/app/building-your-application/routing/loading-ui-and-streaming`)

```ts [src/app/dashboard/loading.tsx]
// loading.tsx KHÔNG cần 'use client' — nó chỉ render UI tĩnh, mặc
// định là Server Component giống mọi file khác trong App Router.
// Next.js tự wrap toàn bộ route segment này trong <Suspense>, bạn
// không cần tự viết <Suspense> cho trường hợp route-level này.
export default function DashboardLoading() {
  return (
    <div className="dashboard-skeleton" aria-busy="true" aria-label="Loading dashboard">
      <div className="skeleton-line" style={{ width: "60%", height: "24px" }} />
      <div className="skeleton-line" style={{ width: "100%", height: "200px" }} />
    </div>
  );
}
```

### 3.2 `error.tsx` — Segment Error Boundary

**[FACT]** `error.tsx` bắt buộc phải là Client Component (`'use client'`). Nó nhận 2 props: `error` (Error instance) và `reset` (function để retry rendering). (Source: `nextjs.org/docs/app/api-reference/file-conventions/error`)

```ts [src/app/dashboard/error.tsx]
"use client";
import { useEffect } from "react";
import { reportError } from "@/lib/monitoring";

interface DashboardErrorProps {
  error: Error & { digest?: string };
  reset: () => void;
}

export default function DashboardError({ error, reset }: DashboardErrorProps) {
  useEffect(() => {
    // Gửi tới monitoring service, KHÔNG console.error thẳng (xem AP3).
    // error.digest là ID Next.js gắn cho lỗi xảy ra ở server-side —
    // dùng nó để tra log thật trên server, không leak stack trace ra UI.
    reportError(error);
  }, [error]);

  return (
    <div role="alert">
      <h2>Đã có lỗi xảy ra khi tải dashboard.</h2>
      <p>Mã lỗi tham chiếu: {error.digest ?? "unknown"}</p>
      <button onClick={reset}>Thử lại</button>
    </div>
  );
}
```

### 3.3 `global-error.tsx` — Fatal Error Catch

**[FACT]** `global-error.tsx` catch lỗi bị bỏ qua bởi `error.tsx` hoặc lỗi xảy ra ở root `layout.tsx`. File này bắt buộc phải tự định nghĩa thẻ `<html>` và `<body>` vì root layout đã fail. (Source: Next.js Docs global-error)

```ts [src/app/global-error.tsx]
"use client";
// Root layout fail nghĩa là KHÔNG còn layout nào bao quanh để cung
// cấp <html>/<body> nữa — global-error.tsx phải tự đứng độc lập,
// tự render lại toàn bộ document shell từ đầu.
import { useEffect } from "react";
import { reportError } from "@/lib/monitoring";

export default function GlobalError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    reportError(error);
  }, [error]);

  return (
    <html lang="en">
      <body>
        <h1>Hệ thống gặp lỗi nghiêm trọng.</h1>
        <p>Mã lỗi tham chiếu: {error.digest ?? "unknown"}</p>
        <button onClick={reset}>Thử lại</button>
      </body>
    </html>
  );
}
```

### 3.4 Manual `<Suspense>` Wrapping

**[FACT]** Manual `<Suspense>` wrapping được dùng bên trong `page.tsx` để tạo granular loading states (chỉ show skeleton ở 1 widget cụ thể, không block cả route). (Source: Next.js Docs)

```ts [src/app/dashboard/page.tsx]
import { Suspense } from "react";
import { OrderHistory } from "@/components/dashboard/order-history";
import { WidgetSkeleton } from "@/components/dashboard/widget-skeleton";

export default function DashboardPage() {
  return (
    <div>
      <h1>Dashboard</h1>
      {/* OrderHistory fetch chậm — wrap riêng để phần còn lại của
          page không phải chờ nó. Phần này render trước, OrderHistory
          stream vào sau khi sẵn sàng. */}
      <Suspense fallback={<WidgetSkeleton />}>
        <OrderHistory />
      </Suspense>
    </div>
  );
}
```

### 3.5 Streaming SSR

**[FACT]** Streaming SSR sử dụng cơ chế HTTP chunked transfer encoding. Server gửi layout HTML trước, giữ kết nối mở, và flush thêm HTML chunks khi từng `<Suspense>` boundary resolve. (Source: React Docs on Streaming)

```ts [src/lib/streaming-note.ts]
// [NOTE] Ghi chú concept, không phải code chạy thật.
//
// Khi server gặp 1 <Suspense> boundary chưa resolve, nó KHÔNG dừng
// lại chờ — nó gửi fallback UI của boundary đó trong response HTML
// đầu tiên, giữ connection mở (Transfer-Encoding: chunked), rồi
// flush thêm 1 chunk HTML + script nhỏ để "swap" fallback thành
// nội dung thật ngay khi data sẵn sàng.
//
// Bài này chỉ giới thiệu concept ở mức này — tối ưu waterfall (parallel
// fetch, preload pattern, request deduplication) thuộc về M9.x.
```

## 4. Anti-Patterns in the Wild

### AP1: `error.tsx` là Server Component

- ❌ **Code Smell:** Quên `'use client'` ở đầu file `error.tsx`.

  ```ts [src/app/dashboard/error.tsx]
  // Quên 'use client'. Error Boundary dựa trên cơ chế client-side
  // (React class lifecycle componentDidCatch) — không tồn tại ở
  // server, nên file này bắt buộc phải là CC. Ở đây còn dùng useEffect
  // trong 1 file Server Component, build sẽ fail vì hook không hợp lệ.
  import { useEffect } from "react";

  export default function DashboardError({
    error,
    reset,
  }: {
    error: Error;
    reset: () => void;
  }) {
    useEffect(() => {
      console.error(error);
    }, [error]);

    return (
      <div>
        <p>{error.message}</p>
        <button onClick={reset}>Thử lại</button>
      </div>
    );
  }
  ```

- ✅ **Staff Refactor:** Thêm `'use client'` ngay khi tạo file (thậm chí dùng snippet/template).

  ```ts [src/app/dashboard/error.tsx]
  "use client";
  // Trade-off: Mất khả năng dùng trực tiếp server-only logic (DB
  // query, fs access) trong file này — nhưng đây không phải mất mát
  // thật, vì error.tsx không nên chứa logic đó ngay từ đầu.
  import { useEffect } from "react";
  import { reportError } from "@/lib/monitoring";

  export default function DashboardError({
    error,
    reset,
  }: {
    error: Error & { digest?: string };
    reset: () => void;
  }) {
    useEffect(() => {
      reportError(error);
    }, [error]);

    return (
      <div role="alert">
        <p>Đã có lỗi xảy ra. Mã tham chiếu: {error.digest ?? "unknown"}</p>
        <button onClick={reset}>Thử lại</button>
      </div>
    );
  }
  ```

- 🧠 **Why:** Mất khả năng dùng trực tiếp server-only logic trong file này (nhưng nên tách logic ra file khác anyway).
- 🔥 **Incident Risk:** Next.js throw lỗi build/runtime. Error boundary hoàn toàn vô dụng, app crash trắng trang thay vì hiện fallback UI.

### AP2: Không có `Suspense` boundary cho slow components

- ❌ **Code Smell:** Chỉ dùng `loading.tsx` ở root route cho 1 page có 3 widget, 1 trong số đó fetch mất 5s.

  ```ts [src/app/dashboard/page.tsx]
  // Page có 3 widget, OrderHistory fetch mất 3s. Cả route phải chờ
  // widget chậm nhất xong mới render BẤT KỲ thứ gì — dù UserProfile
  // và Notifications đã sẵn sàng từ rất sớm.
  import { UserProfile } from "@/components/dashboard/user-profile";
  import { OrderHistory } from "@/components/dashboard/order-history";
  import { Notifications } from "@/components/dashboard/notifications";

  export default async function DashboardPage() {
    return (
      <div>
        <UserProfile />
        <OrderHistory />
        <Notifications />
      </div>
    );
  }
  ```

- ✅ **Staff Refactor:** Trong `page.tsx`, wrap riêng widget chậm đó bằng `<Suspense fallback={<WidgetSkeleton />}>`.

  ```ts [src/app/dashboard/page.tsx]
  import { Suspense } from "react";
  import { UserProfile } from "@/components/dashboard/user-profile";
  import { OrderHistory } from "@/components/dashboard/order-history";
  import { Notifications } from "@/components/dashboard/notifications";
  import { WidgetSkeleton } from "@/components/dashboard/widget-skeleton";

  export default function DashboardPage() {
    // Trade-off: Tăng complexity của component tree (quản lý nhiều
    // fallback riêng) đổi lại TTFB cực nhanh cho UI visible —
    // UserProfile và Notifications hiện ngay, OrderHistory stream
    // vào sau khi sẵn sàng.
    return (
      <div>
        <UserProfile />
        <Suspense fallback={<WidgetSkeleton />}>
          <OrderHistory />
        </Suspense>
        <Notifications />
      </div>
    );
  }
  ```

- 🧠 **Why:** Tăng complexity của component tree (quản lý nhiều fallback) đổi lại TTFB (Time to First Byte) cực nhanh cho UI visible.
- 🔥 **Incident Risk:** User bị nhìn toàn bộ route spinner trong 5s thay vì đọc được nội dung 2 widget kia ngay lập tức.

### AP3: Log lỗi production trong `error.tsx` với `console.error`

- ❌ **Code Smell:** Viết `console.error(error)` trong `error.tsx`.

  ```ts [src/app/dashboard/error.tsx]
  "use client";
  import { useEffect } from "react";

  export default function DashboardError({
    error,
    reset,
  }: {
    error: Error;
    reset: () => void;
  }) {
    useEffect(() => {
      // console.error xuất hiện NGAY trên browser console của
      // end-user — bất kỳ ai mở DevTools đều thấy full stack trace,
      // có thể chứa đường dẫn DB query, internal API URL...
      console.error(error);
    }, [error]);

    return (
      <div>
        {/* Render thẳng error.message ra UI — có thể leak chi tiết
            implementation cho end-user. */}
        <p>{error.message}</p>
        <button onClick={reset}>Thử lại</button>
      </div>
    );
  }
  ```

- ✅ **Staff Refactor:** Dùng monitoring service (VD: `Sentry.captureException(error)`). Tuyệt đối không render `error.message` thẳng ra UI nếu không sanitize.

  ```ts [src/app/dashboard/error.tsx]
  "use client";
  // Trade-off: Phải setup 3rd party service (Sentry/Datadog...) —
  // tốn cost/effort ban đầu đổi lại log được tập trung, alert
  // real-time, và an toàn cho end-user.
  import { useEffect } from "react";
  import { reportError } from "@/lib/monitoring";

  export default function DashboardError({
    error,
    reset,
  }: {
    error: Error & { digest?: string };
    reset: () => void;
  }) {
    useEffect(() => {
      reportError(error); // gửi full error tới monitoring service,
      // KHÔNG log ra browser console.
    }, [error]);

    return (
      <div role="alert">
        {/* Chỉ hiện message generic + digest để tra log — không bao
            giờ render error.message thô nếu chưa sanitize. */}
        <p>Đã có lỗi xảy ra. Mã tham chiếu: {error.digest ?? "unknown"}</p>
        <button onClick={reset}>Thử lại</button>
      </div>
    );
  }
  ```

- 🧠 **Why:** Phải setup 3rd party service (tốn cost/effort) đổi lại log được tập trung, alert real-time, an toàn.
- 🔥 **Incident Risk:** Internal stack traces, database query paths, hoặc secret keys bị leak ra browser console của end-user.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **`loading.tsx` (Auto-Suspense)** | Zero-config, route-level consistency | Coarse-grained (block cả route) | Simple pages, layout transitions | Pages có widgets load tốc độ khác nhau | Rất thấp |
| **Manual `<Suspense>`** | Granular streaming, TTFB tối ưu | Có thể gây Layout Shift nếu fallback size != real size | Dashboards, trang mix fast/slow data | Tiny static pages (overkill) | UI engineering effort |
| **`error.tsx` (Segment)** | Giữ nguyên layout cha, context-aware | Không catch lỗi ở layout cùng cấp | Bất kỳ feature route nào | Catch lỗi global/root layout | Phải viết CC |
| **`global-error.tsx`** | Catch được fatal errors (root layout crash) | Mất hoàn toàn styling/layout gốc, phải rewrite HTML | Kéo dài mạng lưới an toàn cuối cùng | Handle standard data fetch errors | Phải duplicate HTML boilerplate |

## 6. Legacy Recognition

Bible ghi Legacy Bridge là N/A, nhưng raw material có đủ data cụ thể để nhận diện code cũ — dùng nó:

- **Legacy:** Pages Router `_error.tsx` — custom error page ở cấp page, không phân biệt được lỗi theo từng route segment như `error.tsx`.
- **Legacy:** Client-side `isLoading` state toàn trang — quản lý loading bằng global state hoặc context, mọi widget dùng chung 1 boolean.
- **Legacy:** `getServerSideProps` blocking — toàn bộ page chờ data trước khi render, không có khái niệm "stream từng phần".

## 7. Modern Approach

- **Modern:** `app/error.tsx` với `'use client'` và `reset()` để self-healing — retry render mà không cần full page reload.
- **Modern:** `app/loading.tsx` cho route-level skeleton, `<Suspense>` cho widget-level skeleton.
- **Migration `_error.tsx`:** Tạo `app/error.tsx`. Thêm `'use client'`. Chuyển logic sang dùng `reset()` thay vì hack `router.reload()` — đây là incremental, route-by-route, không rewrite toàn bộ error handling 1 lần.
- **Migration `isLoading` state toàn trang:** Xóa global state `isLoading`. Đẩy data fetching lên Server Component parent, wrap phần UI cần chờ bằng `<Suspense>`.
- **Migration `getServerSideProps` blocking:** Chuyển page thành async SC. Tách các fetch call độc lập vào các component con và wrap chúng bằng `<Suspense>` — bắt đầu với widget chậm nhất, đo lại TTFB, rồi mới quyết định có cần tách thêm không.
- **Rollback plan:** Nếu manual `<Suspense>` gây Layout Shift quá nặng hoặc CDN lỗi chunked transfer, tạm gộp các component con lại vào 1 async SC parent và xóa manual boundaries, dựa vào `loading.tsx` cho đến khi fix được root cause ở tầng infra.

## 8. Production Scenario

### Scenario 1: Toàn bộ page bị block dù chỉ 1 API chậm

- **Symptom:** Network tab cho thấy document HTML chờ 4-5 giây mới start download.
- **Root cause:** Không có granular `<Suspense>`, page là 1 async block lớn.
- **Detect:** Check DevTools Network → Timing → TTFB cao bất thường.
- **Fix:** Thêm manual `<Suspense>` xung quanh component gọi API chậm.

### Scenario 2: Streaming bị broken bởi Reverse Proxy

- **Symptom:** Locally stream ngon, lên production (sau Nginx/Cloudflare) lại hiện spinner chờ toàn bộ.
- **Root cause:** Proxy buffer response lại thay vì flush ngay.
- **Detect:** Check response headers thiếu `Transfer-Encoding: chunked`.
- **Fix:** Config proxy (VD: Nginx: `X-Accel-Buffering: no`).

### Scenario 3: Error Boundary không catch lỗi

- **Symptom:** App crash trắng trang thay vì hiện UI fallback.
- **Root cause:** `error.tsx` thiếu `'use client'` hoặc lỗi xảy ra trong `layout.tsx` cùng cấp mà không có `global-error.tsx`.
- **Detect:** Check terminal/runtime console logs — tìm lỗi build liên quan hook trong Server Component, hoặc xác nhận lỗi xảy ra ở chính layout (không phải children của nó).

## 9. Performance & Overhead

Raw material xác nhận chưa có benchmark cụ thể về TTFB improvement, chunked transfer overhead, hay memory khi giữ nhiều connection mở — đừng bịa số, đo theo hướng sau:

**TTFB: Manual `<Suspense>` vs Blocking SSR**
- Đo bằng Chrome DevTools Network tab → chọn document request → tab Timing, so sánh giá trị TTFB giữa version blocking (chờ tất cả async resolve) và version có manual `<Suspense>` (layout HTML gửi ngay, chunk sau).
- Quy tắc thực dụng: nếu page có ≥1 fetch call >1s và bạn không thấy cải thiện TTFB sau khi thêm `<Suspense>`, kiểm tra ngay proxy/CDN có buffer response không (xem Scenario 2, mục 8).

**Chunked transfer encoding overhead**
- Mỗi chunk HTTP có thêm vài byte overhead cho chunk-size header — với page có nhiều `<Suspense>` boundary (10+), tổng overhead này vẫn nhỏ hơn nhiều so với latency tiết kiệm được từ việc không phải chờ phần chậm nhất.
- Đo bằng cách so sánh tổng response size (tab Network, cột Size) giữa response chunked và response buffer toàn bộ.

**Memory: giữ nhiều TCP connection mở khi streaming**
- Mỗi request đang stream giữ connection mở lâu hơn so với response trả 1 lần — theo dõi qua APM hoặc profiling tool của runtime để xem connection pool/event loop có bị nghẽn khi traffic cao không.
- Quy tắc thực dụng: nếu nhiều route stream đồng thời và CPU/memory tăng bất thường dưới load test, cân nhắc giới hạn timeout cho fetch bên trong `<Suspense>` — không để 1 fetch treo vô thời hạn giữ connection mở mãi.

Số liệu cụ thể phụ thuộc hạ tầng, proxy, traffic pattern thật của bạn — đo trên staging/production traffic thật trước khi rollout rộng.

## 10. Architecture Notes

- **Granular Suspense Strategy:** Trong dashboard nhiều widget, mỗi widget fetch data độc lập nên có `<Suspense>` riêng. Tránh "nested waterfall" bằng cách fetch song song ở SC parent rồi truyền xuống, hoặc dùng `Promise.all` khi các fetch không phụ thuộc nhau.
- **Error Boundary Hierarchy:** `error.tsx` ở cấp feature route → `global-error.tsx` ở root. Không dùng `error.tsx` để catch lỗi layout cùng cấp (nó chỉ catch children).
- **Known Pitfall — `error.tsx` không catch lỗi trong chính `layout.tsx` cùng cấp:** `error.tsx` chỉ catch lỗi của component con bên trong layout. Lỗi ở chính layout đó phải dùng `global-error.tsx`. (Source: Next.js Docs Error Handling)
- **Known Pitfall — `reset()` gây infinite loop:** Xảy ra khi lỗi do logic sai trong render (VD: map qua undefined) chứ không phải do data fetch lỗi tạm thời. `reset()` re-render với cùng faulty state/props, nên lỗi lập tức xuất hiện lại — `reset()` không phải nút "fix mọi thứ", nó chỉ hữu ích cho lỗi thoáng qua.
- **Known Pitfall — Layout Shift từ Skeleton:** `[THEORY — NEEDS VERIFICATION]` Fallback component trong `<Suspense>` có height/kích thước nhỏ hơn component thực tế khi data load xong, gây nhảy UI. Cần xác nhận cơ chế CSS intrinsic sizing có tự giải quyết vấn đề này trong Next.js 15+/16 hay không trước khi coi đây là behavior chắc chắn — hiện tại vẫn nên tự set `min-height`/`aspect-ratio` khớp với content thật.
- **CDN / Proxy Config:** Streaming yêu cầu proxy không buffer response. Nếu dùng Cloudflare, Nginx, AWS ALB — cần verify chunked transfer config trước khi tin tưởng streaming hoạt động đúng trên production.

## 11. Interview Questions

### Q1: Khi nào dùng `loading.tsx` và khi nào dùng `<Suspense>` thủ công? (Junior)
<details>
<summary>Đáp án</summary>

`loading.tsx` cho loading toàn bộ route segment — đơn giản, zero-config. `<Suspense>` thủ công khi cần granular streaming, chỉ skeleton 1 phần cụ thể bên trong page, để tối ưu TTFB cho phần còn lại.

</details>

### Q2: Tại sao `error.tsx` bắt buộc phải là Client Component? (Mid)
<details>
<summary>Đáp án</summary>

Vì Error Boundary là cơ chế client-side của React, dựa trên class component lifecycle `componentDidCatch`. Server Components không có lifecycle nên không có cơ chế nào để catch lỗi runtime theo cách này trên client.

</details>

### Q3: Streaming SSR hoạt động dưới nền tảng HTTP như thế nào? (Mid)
<details>
<summary>Đáp án</summary>

Dùng `Transfer-Encoding: chunked`. Server flush layout HTML ngay, giữ TCP connection mở, và gửi thêm các chunk HTML khi từng `<Suspense>` boundary resolve data xong.

</details>

### Q4: Sự khác biệt giữa `error.tsx` và `global-error.tsx`? (Junior)
<details>
<summary>Đáp án</summary>

`error.tsx` catch lỗi ở children, giữ nguyên layout cha. `global-error.tsx` catch lỗi ở chính root layout (fatal case) và bắt buộc tự render thẻ `<html><body>` vì layout gốc đã chết, không còn gì bao quanh nó nữa.

</details>

### Q5: Cài đặt `reset()` trong `error.tsx` giải quyết được lỗi gì, và không giải quyết được lỗi gì? (Senior)
<details>
<summary>Đáp án</summary>

Giải quyết lỗi tạm thời — network timeout, 500 từ API, lỗi do trạng thái server đổi giữa lúc đó. Không giải quyết được lỗi logic client-side persist, ví dụ bug khiến render luôn fail với cùng state/props — `reset()` chỉ re-render lại với chính state đang gây lỗi, nên nó sẽ crash lại ngay, tạo cảm giác "nút Try again vô dụng".

</details>

<details>

### 🟡 Exercise 2: Manual Suspense với Multiple Boundaries

**Đề bài:** Tạo page có 3 widget: `UserProfile` (fast, ~100ms), `OrderHistory` (slow, ~3s), `Notifications` (medium, ~1s). Chỉ dùng `loading.tsx` cho toàn bộ route là KHÔNG đủ ở bài này.

- [ ] So sánh TTFB giữa bài này và bài 1 — TTFB ở bài 2 phải thấp hơn rõ rệt.

<details>
<summary>Hint</summary>

Wrap riêng widget chậm nhất trước, đo lại, rồi mới quyết định có cần `<Suspense>` cho cả `Notifications` hay không. Dùng DevTools Network → chọn request document → tab Timing để đọc TTFB, so sánh trực tiếp với giá trị ở Exercise 1.

### 🔴 Exercise 3: Debug Streaming bị Broken và Layout Shift

**Đề bài:** Cho sẵn 1 app streaming bị lỗi 2 vấn đề: (1) stream đúng ở local nhưng lên production qua Nginx lại bị block toàn bộ; (2) skeleton có height 100px nhưng content thực tế cao 400px, gây layout shift khi stream xong.

**Acceptance Criteria:**
- [ ] Xác định đúng nguyên nhân vấn đề (1) là do proxy buffer response — không phải do code Next.js.
- [ ] Config proxy để preserve chunked transfer (không tự ý đổi sang giải pháp khác như tắt hẳn streaming).
- [ ] Sửa skeleton để khớp kích thước thực tế của content (không chỉ tăng height tùy ý — phải có lý do đo được).

<details>
<summary>Hint</summary>

Với Nginx, tìm header liên quan đến buffering (gợi ý: tên header có chữ "Accel-Buffering"). Với skeleton, đo chính xác kích thước content thật (DevTools → Inspect → Computed) rồi áp cho fallback bằng `min-height` hoặc cấu trúc DOM placeholder tương tự, không đoán số tùy ý.

</details>

## 13. Summary

- `loading.tsx` là auto-Suspense ở route level — dùng cho simple pages, nhưng đừng để 1 slow widget block cả trang.
- `error.tsx` bắt buộc là Client Component — nó catch lỗi children, không catch lỗi layout cùng cấp. `global-error.tsx` là safety net cuối cùng.
- Manual `<Suspense>` là công cụ tối ưu TTFB — wrap từng slow component riêng biệt để stream layout ngay lập tức.
- Streaming cần proxy/CDN config đúng — `Transfer-Encoding: chunked` phải được preserve qua từng layer hạ tầng.

## 14. Terminology Evolution

### Suspense
```
┌─────────────┬──────────┬─────────────────────────────────┐
│ Era          │ Status   │ Dùng khi nào                       │
├─────────────┼──────────┼─────────────────────────────────┤
│ React 16.6+  │ Active   │ Code-splitting (lazy loading)      │
│ React 18+    │ Active   │ Data fetching + Streaming           │
│ Next.js 13+  │ Active   │ loading.tsx auto-wrap                │
└─────────────┴──────────┴─────────────────────────────────┘
```
Lịch sử: React primitive để declare loading state — không phải Next.js feature. Ban đầu chỉ phục vụ code-splitting (`React.lazy`), mở rộng sang data fetching/streaming từ React 18.
Codebase legacy: `React.lazy()` + `<Suspense>` trong SPA thuần, trước khi có khái niệm streaming SSR.

### Streaming SSR
```
┌─────────────┬──────────┬─────────────────────────────────┐
│ Era          │ Status   │ Dùng khi nào                       │
├─────────────┼──────────┼─────────────────────────────────┤
│ React 18+    │ Active   │ Server render HTML từng chunk        │
│ Next.js 13+  │ Active   │ App Router default behavior          │
│ Next.js <13  │ Legacy   │ Blocking SSR — chờ toàn bộ tree        │
└─────────────┴──────────┴─────────────────────────────────┘
```
Lịch sử: Server render HTML theo từng chunk, không chờ toàn bộ component tree ready — enabled bởi Suspense, dùng HTTP chunked transfer encoding.
Codebase legacy: `getServerSideProps` — blocking cho đến khi tất cả data resolve mới trả response.

### Error Boundary
```
┌─────────────┬──────────┬─────────────────────────────────────────┐
│ Era          │ Status   │ Dùng khi nào                                 │
├─────────────┼──────────┼─────────────────────────────────────────┤
│ React 16+    │ Active   │ Catch render errors (class component)         │
│ Next.js 13+  │ Active   │ error.tsx convention                            │
│ Next.js <13  │ Legacy   │ pages/_error.tsx                                │
└─────────────┴──────────┴─────────────────────────────────────────┘
```
Lịch sử: React pattern để catch render errors. `error.tsx` là Next.js convention cho Error Boundary trong App Router, tự động wire vào đúng route segment.
Codebase legacy: `pages/_error.tsx` hoặc custom `_error.js` — catch ở cấp page, không phân biệt theo segment.

### `reset()`
```
┌─────────────┬──────────┬─────────────────────────────────────────┐
│ Era          │ Status   │ Dùng khi nào                                 │
├─────────────┼──────────┼─────────────────────────────────────────┤
│ Next.js 13+  │ Active   │ error.tsx — self-healing retry                 │
│ N/A          │ N/A      │ Không có tương đương trong Pages Router        │
└─────────────┴──────────┴─────────────────────────────────────────┘
```
Lịch sử: Function prop do Next.js inject vào `error.tsx`, dùng để clear error state và re-render lại đúng segment đó (self-healing UI) — không phải full page reload. Được bổ sung từ raw material, chưa nằm trong Terminology gốc của Bible.
Codebase legacy: `router.reload()` hoặc manual page refresh — cách duy nhất để "thử lại" trong Pages Router, tốn nhiều hơn vì reload cả page.

### `global-error.tsx`
```
┌─────────────┬──────────┬─────────────────────────────────────────┐
│ Era          │ Status   │ Dùng khi nào                                 │
├─────────────┼──────────┼─────────────────────────────────────────┤
│ Next.js 13+  │ Active   │ Catch fatal root layout errors                 │
│ N/A          │ N/A      │ Không có tương đương trong Pages Router        │
└─────────────┴──────────┴─────────────────────────────────────────┘
```
Lịch sử: File convention catch lỗi ở chính root layout, yêu cầu tự render `<html><body>` — là safety net cuối cùng khi mọi error boundary khác đã không catch được. Được bổ sung từ raw material, chưa nằm trong Terminology gốc của Bible.
Codebase legacy: Không có tương đương trực tiếp. `pages/_error.tsx` catch ở page level, không phân biệt riêng trường hợp root layout fatal.
