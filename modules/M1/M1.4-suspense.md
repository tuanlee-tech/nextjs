# M1.4 — Suspense: Ngữ nghĩa Ranh giới & Thời điểm Fallback

## 0. Tóm Tắt

<details>
<summary>Thời gian đọc: 5 phút</summary>

Suspense là **async boundary** của React — component "suspend" bằng cách throw Promise trong lúc render, React bắt Promise từ ancestor Suspense gần nhất, show fallback, rồi retry khi Promise resolve. Không phải "cách hiện spinner đẹp hơn", mà là **infrastructure cốt lõi** cho toàn bộ hệ thống async của React hiện đại (Streaming SSR, Next.js App Router, PPR).

**3. Mental Model (Ý tưởng cốt lõi)**

> **Suspense = Async Error Boundary** — cùng cơ chế bắt exception, nhưng bắt Promise thay vì Error. Khi Promise resolve, React **retry render** component đó (pause and resume), không phải "catch và bỏ qua".

Analogy: Nhà hàng đặt bình hoa tạm (fallback) khi món ăn (data) chưa sẵn. Khi món ra, dọn bình hoa đi — khách không cần gọi lại, nhà hàng tự xử lý.

**4. Core Concepts (20% kiến thức quan trọng nhất)**

**Concept 1: Throw Promise — Trái tim của Suspense**

- Component throw thenable (có `.then()`) trong render phase → React bắt → Suspense kích hoạt
- `throw new Error()` → Error Boundary; `throw promise` → Suspense
- React 19: `use(promise)` là API chính thức — tạo promise **bên ngoài** component, truyền vào như prop (render-as-you-fetch)
- Next.js App Router: async Server Component tự động suspend khi `await`

**Concept 2: Boundary Semantics — 3 Rules**

- **Nearest Ancestor**: chỉ Suspense gần nhất bị trigger, không phải tất cả ancestors
- **All-or-Nothing**: toàn bộ subtree trong boundary cùng số phận — dù component khác đã ready cũng bị ẩn theo
- **Atomic Reveal**: khi Promise resolve, toàn bộ subtree re-render cùng lúc, không pop in từng cái

**Concept 3: Fallback Reveal Timing**

- **Initial render**: fallback hiện ngay lập tức khi suspend
- **startTransition**: React giữ UI cũ, fetch data mới, chỉ show fallback khi thực sự chậm → tránh "flash of loading"
- Next.js `<Link>` tự động dùng `startTransition` → navigation mượt mà
- React 18: có thể skip fallback nếu Promise resolve đủ nhanh
- React 19: "pre-warming" cho nested Suspense boundaries

**Concept 4: Nested Suspense — Chiến lược Granularity**

- Nhiều boundaries → progressive loading, UX tốt hơn, nhưng có thể "popcorn effect"
- Ít boundaries → đơn giản, nhưng một phần chậm block toàn trang
- Nguyên tắc: nhóm cùng tốc độ + cùng visual group; tách khác tốc độ hoặc độc lập

**Concept 5: Suspense + Error Boundary — Combo Bắt Buộc**

- Suspense = pending state; Error Boundary = error state
- Không thể thay thế nhau
- Next.js: `loading.tsx` = Suspense, `error.tsx` = Error Boundary (phải là Client Component)

👉 Công thức / sơ đồ

```
Component render
    │
    ├── Data sẵn sàng → Render bình thường → Commit DOM
    │
    └── Data chưa sẵn sàng → throw Promise (thenable)
                │
        React bắt Promise
                │
        Tìm Suspense ancestor GẦN NHẤT
                │
        Mount fallback UI thay subtree
                │
        Promise resolves
                │
        UNMOUNT fallback → RE-RENDER subtree → Commit DOM
```

**5. Legacy Recognition (Nhìn code biết gì?)**

Dấu hiệu nhận biết code "pre-Suspense"

✅ **Nhận ra ngay:**

- `useState(true)` cho `isLoading`, `useState(false)` cho `isError`
- `useEffect` để fetch data
- Conditional render `{isLoading ? <Spinner /> : <Content />}` khắp nơi
- Multiple loading states cho multiple data sources trong cùng component
- Waterfall fetching (fetch B sau khi A xong)
- Race condition tiềm ẩn khi prop thay đổi trong lúc fetch đang chạy

❌ **Không nên viết:**

- Tạo promise bên trong component body → vòng lặp vô hạn khi retry
- `throw new Promise(() => {})` thủ công trong Client Component → never resolves, spinner xoay mãi
- Wrap component đồng bộ trong Suspense → vô nghĩa

**6. Modern Approach (Ngày nay làm thế nào?)**

| Legacy                               | Modern                                                   |
| ------------------------------------ | -------------------------------------------------------- |
| `useEffect` + `setState` fetch       | Async Server Component `await` hoặc `use(promise)`       |
| `isLoading` / `isError` state        | Suspense fallback + `error.tsx`                          |
| Conditional render loading khắp nơi  | Suspense boundary ở "async entry points"                 |
| Waterfall: child mount rồi mới fetch | Parallel fetch: tất cả bắt đầu ngay khi React encounter  |
| Race condition thủ công              | Mỗi request là server render riêng, không race           |
| SSR đợi toàn bộ data → TTFB chậm     | Streaming SSR: stream từng phần theo Suspense resolution |
| Manual loading state management      | `loading.tsx` tự động, zero boilerplate                  |

**7. Production Scenario (Thực tế)**

**Context:** E-commerce product detail page

**Problem:** 4 data sources với tốc độ khác nhau (50ms → 1.2s). Trước Suspense: phải đợi 1.2s mới hiện gì.

**Root Cause:** Không có cơ chế stream riêng từng phần — toàn trang blocked bởi data chậm nhất.

**Solution:** 4 nested Suspense boundaries, mỗi section fetch song song, stream độc lập:

```
Server nhận request
    ↓
Stream HTML skeleton xuống ngay (~50ms TTFB)
    ↓
ProductHero ready (~50ms) → stream thay skeleton
StockBadge ready (~200ms) → stream tiếp
ProductReviews ready (~700ms) → stream tiếp
RelatedProducts ready (~1.2s) → stream cuối
```

**Lesson:** Streaming SSR giảm TTFB từ ~1.2s xuống ~50ms. User thấy nội dung ngay, trang "sống dậy" từng phần.

👉 Rule

> Nếu gặp **data sources có tốc độ khác nhau** hãy nghĩ ngay đến **nested Suspense boundaries + parallel fetch**.

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng**

- Data fetching async trong React (Server Component hoặc `use()`)
- Code splitting với `React.lazy()`
- Streaming SSR / Next.js App Router
- Cần progressive loading — từng phần hiện khi sẵn sàng
- Cần isolate failure — một section fail không crash toàn trang

**Khi không nên dùng**

- Component đồng bộ, không có async operation
- Real-time data (WebSocket, live updates) — không phải "one-time fetch"
- Data có dependency tuần tự bắt buộc — Suspense không fix waterfall
- Muốn hiện partial content trong boundary — Suspense là all-or-nothing

Trade-off

Pros

- Tách biệt loading logic khỏi component business logic
- Parallel fetching tự nhiên với Server Components
- Streaming SSR giảm TTFB đáng kể
- UX tốt hơn: progressive loading, không flash spinner với `startTransition`
- Không race condition, không manual state management

Cons

- All-or-nothing: component ready cũng bị ẩn nếu sibling suspend
- "Popcorn effect" nếu quá nhiều boundaries pop in lần lượt
- Cần thiết kế nhiều skeleton states
- Learning curve: phải hiểu cơ chế throw Promise, boundary semantics
- Không fix waterfall ở data layer — chỉ quyết định UI khi chờ

## 3. Mental Model

Trước khi đi vào code, phải build đúng mental model. Code chỉ là implementation của mental model này.

**Suspense = Async Error Boundary**

Bạn đã biết `Error Boundary` bắt lỗi khi component `throw new Error(...)`. Suspense dùng **cùng cơ chế đó**, nhưng bắt **Promise** thay vì Error.

```
// Error Boundary: catch errors
try { dangerousRender() } catch (error) { showErrorUI() }

// Suspense: catch promises (conceptually)
try { asyncRender() } catch (promise) { showFallback(); promise.then(retry) }
```

**Luồng hoạt động đầy đủ:**

```
Component đang render
        │
        ├── Data đã sẵn sàng → Render bình thường → Commit DOM
        │
        └── Data chưa sẵn sàng → throw Promise (thenable)
                    │
            React bắt cái Promise đó
                    │
            Tìm Suspense ancestor GẦN NHẤT
                    │
            Mount fallback UI thay cho toàn bộ subtree
                    │
            Promise resolves (data đã có)
                    │
            React schedule một lần render mới (retry)
                    │
            Render lại subtree
                    │
            Nếu lần này không còn suspend → Unmount fallback
                    │
            Commit DOM với content thật
```

**Điểm khác biệt then chốt với try-catch:** Suspense không "catch và bỏ qua" như try-catch. Khi Promise resolve, React **retry render** component đó. Đây là cơ chế "pause and resume".

**Analogy dễ nhớ:** Suspense giống như một nhà hàng (React) đang phục vụ khách. Khi món ăn (data) chưa sẵn sàng, họ đặt một cái bình hoa tạm thời (fallback) lên bàn. Khi món ăn ra, họ dọn bình hoa đi và đặt món ăn thật vào — không phải khách phải gọi lại, nhà hàng tự xử lý.

## 4. Core Concepts

### 4.1 Cơ chế Throw Promise — Trái tim của Suspense

Đây là thứ thực sự xảy ra bên dưới. Bạn không viết cái này trong code hàng ngày, nhưng **bắt buộc phải hiểu** để debug và hiểu tại sao các thứ khác hoạt động.

```javascript
// Pattern "resource" mà các thư viện dùng nội bộ
// (Relay, SWR Suspense mode, React Query Suspense mode đều làm tương tự)

function createResource(promise) {
  let status = 'pending';
  let result;

  // Bắt đầu execute promise NGAY LẬP TỨC khi resource được tạo
  const suspender = promise.then(
    (data) => {
      status = 'success';
      result = data;
    },
    (error) => {
      status = 'error';
      result = error;
    },
  );

  return {
    read() {
      if (status === 'pending') {
        throw suspender; // ← ĐÂY là "suspend" — throw một thenable!
      }
      if (status === 'error') {
        throw result; // ← Error Boundary bắt cái này
      }
      return result; // ← Trả data khi đã ready
    },
  };
}

// Component dùng resource:
const userResource = createResource(fetchUser(id));

function UserProfile() {
  const user = userResource.read(); // suspend nếu chưa ready, return data nếu ready
  return <div>{user.name}</div>;
}
```

**Key insight:** React kiểm tra xem thứ bị throw có phải là "thenable" (có `.then()` method) không. Nếu có → Suspense. Nếu không → Error. Đây là cách React phân biệt hai loại exception.

```javascript
throw new Error("something broke")   // → Error Boundary bắt
throw new Promise(resolve => ...)    // → Suspense bắt (thenable!)
throw { then: fn }                   // → Cũng là Suspense (thenable object)
```

### 4.2 React 18+ — use() Hook: API Chính Thức

Từ React 19, `use()` là cách chính thức để trigger Suspense từ client code:

```typescript
import { use } from 'react'

// Promise phải được tạo BÊN NGOÀI component
// Đây là "render-as-you-fetch" — bắt đầu fetch trước khi render
function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  // use() suspend component nếu promise chưa resolve
  // Khi resolve, return value ngay lập tức
  const user = use(userPromise)
  return <div>{user.name}</div>
}

// Parent: tạo promise sớm, trước khi render child
function Page({ userId }: { userId: string }) {
  // Fetch bắt đầu ngay khi Page render — không đợi UserProfile mount
  const userPromise = fetchUser(userId)

  return (
    <Suspense fallback={<UserSkeleton />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  )
}
```

**Tại sao phải tạo promise bên ngoài component?**

Trong `Client Component`, mỗi lần render là hàm component chạy lại.

Mỗi lần render lại tạo Promise mới nên React không bao giờ "đuổi kịp" Promise đã resolve.

Vì vậy, Promise phải được tạo bên ngoài component hoặc được cache.

```typescript
// 🔴 SAI — tạo promise BÊN TRONG component
function UserProfile({ userId }: { userId: string }) {
  // Mỗi lần render tạo promise mới
  // ❌ React retry → tạo promise mới → retry → tạo promise mới → vòng lặp!
  const userPromise = fetchUser(userId)  // ← Bug tiềm ẩn
  const user = use(userPromise)
  return <div>{user.name}</div>
}

// ✅ ĐÚNG — tạo promise bên ngoài, truyền vào như prop
function Page({ userId }: { userId: string }) {
  const userPromise = useMemo(() => fetchUser(userId), [userId])
  return (
    <Suspense fallback={<UserSkeleton />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  )
}
```

> `Client Component:` _Render phải là pure function_ → đừng tạo side effect hoặc Promise mới trong quá trình render.

### 4.4 Boundary Semantics — Ai Bị Ảnh Hưởng?

Đây là phần kỹ thuật quan trọng nhất của bài.

**Rule 1: Nearest Ancestor — Chỉ Suspense gần nhất bị kích hoạt**

```tsx
<Suspense fallback={<PageSpinner />}>
  {' '}
  {/* Boundary A — xa hơn */}
  <Header /> {/* Không suspend */}
  <Suspense fallback={<ContentSkeleton />}>
    {' '}
    {/* Boundary B — gần hơn */}
    <MainContent /> {/* SUSPENDS */}
  </Suspense>{' '}
  {/* ← B bắt, không phải A */}
  <Footer /> {/* Không suspend */}
</Suspense>
```

Kết quả khi `MainContent` suspend:

- `ContentSkeleton` hiện (Boundary B kích hoạt)
- `Header` và `Footer` vẫn render bình thường
- `PageSpinner` KHÔNG xuất hiện (Boundary A không bị trigger)

**Rule 2: All-or-Nothing — Toàn bộ subtree trong boundary cùng số phận**

```tsx
<Suspense fallback={<Spinner />}>
  <ComponentA /> {/* ← đã có data, sẵn sàng render */}
  <ComponentB /> {/* ← suspend! */}
  <ComponentC /> {/* ← đã có data, sẵn sàng render */}
</Suspense>
```

Kết quả: `<Spinner />` hiện. `ComponentA` và `ComponentC` (dù đã ready) cũng bị ẩn. Chỉ khi `ComponentB` xong, cả ba mới xuất hiện **cùng nhau**.

**Lý do design này:** Đảm bảo UI consistency. Nếu `ComponentA` hiện trước, layout có thể bị "nhảy" khi `ComponentB` pop in. All-or-nothing đảm bảo UI xuất hiện như một khối hoàn chỉnh.

**Rule 3: Atomic Reveal — Subtree hiện ra cùng lúc**

Khi Promise resolve, React không chỉ show component bị suspend. React re-render **toàn bộ subtree** bên trong boundary cùng một lần. Layout transitions sẽ được áp dụng nhất quán.

### 4.5 Fallback Reveal Timing — Khi Nào Fallback Xuất Hiện?

Đây là phần bị hiểu sai nhiều nhất, đặc biệt khi làm việc với Next.js navigation.

**Case 1: Initial Render (trang load lần đầu)**

```tsx
// Người dùng vào /dashboard lần đầu
// Fallback hiện NGAY LẬP TỨC khi component suspend
<Suspense fallback={<DashboardSkeleton />}>
  <DashboardContent /> {/* suspend → <DashboardSkeleton /> hiện ngay */}
</Suspense>
```

Không có delay, không có debounce — fallback xuất hiện ngay khi React gặp suspension trong initial render.

**Case 2: Navigation với startTransition (React 18+)**

```tsx
// 🔴 Không dùng startTransition — flash spinner
function NavLink({ href }: { href: string }) {
  const router = useRouter();

  return (
    <button onClick={() => router.push(href)}>
      Go {/* Nhấn → Spinner flash ngay lập tức dù trang load nhanh */}
    </button>
  );
}

// ✅ Dùng startTransition — UX mượt mà hơn
function NavLink({ href }: { href: string }) {
  const [isPending, startTransition] = useTransition();
  const router = useRouter();

  function handleClick() {
    startTransition(() => {
      router.push(href); // ← wrap trong transition
    });
  }

  return (
    // isPending = true trong khi trang mới đang chuẩn bị
    // Có thể dùng isPending để show progress bar nhỏ
    <button
      onClick={handleClick}
      aria-busy={isPending}
    >
      Go {isPending && <SmallSpinner />}
    </button>
  );
}
```

Với `startTransition`, hành vi của React thay đổi:

```
Không startTransition:          Với startTransition:
Nhấn Link                       Nhấn Link
    ↓                               ↓
Trang mới suspend               React GIỮ trang hiện tại
    ↓                               ↓
Fallback xuất hiện NGAY         Fetch data trang mới
    ↓                               ↓
Data load xong                  Data xong → swap sang trang mới
    ↓                           HOẶC timeout → show fallback
Trang mới hiện
```

**Quan trọng với Next.js:** `<Link>` trong App Router tự động dùng `startTransition` nội bộ. Đó là lý do navigation trong App Router cảm thấy mượt mà hơn Pages Router, đặc biệt với fast connections.

> ⚠️ **Lưu ý:** Đây là implementation detail của Next.js, không phải behavior mặc định của React.
> Trong React thuần (không dùng framework), bạn phải tự gọi `startTransition`.

**Case 3: React 18 — Không Flash Fallback Cho Fast Loads**

React 18 có cơ chế tránh hiện fallback khi data load quá nhanh. Nếu Promise resolve đủ nhanh (thường dưới một render cycle), React có thể skip fallback hoàn toàn và show content trực tiếp. Điều này tránh "flash of loading state" khó chịu.

**Case 4: React 19 — Pre-warming cho Suspended Trees**

React 19 giới thiệu cải tiến "pre-warming" cho các cây component bị suspend.
Khi một component suspend, React 19 sẽ "pre-warm" (chuẩn bị trước) các Suspense boundaries
lồng nhau để giảm thời gian chờ khi data cascade. Điều này đặc biệt hữu ích cho
nested Suspense boundaries trong Server Components.

Đây là một trong những improvements to Suspense được đề cập trong React 19 release notes.

### 4.6 Nested Suspense — Chiến lược Granularity

```tsx
// Ví dụ: Dashboard với nhiều sections có tốc độ load khác nhau

function Dashboard() {
  return (
    <main>
      {/* Không cần Suspense — data từ cache, cực nhanh */}
      <DashboardHeader />

      {/* Section 1: Stats — DB query nhanh */}
      <Suspense fallback={<StatsSkeleton />}>
        <StatsPanel />
      </Suspense>

      {/* Section 2: Chart — aggregate query, chậm hơn */}
      <Suspense fallback={<ChartSkeleton />}>
        <RevenueChart />
      </Suspense>

      {/* Section 3: Feed — external service, chậm nhất */}
      <Suspense fallback={<FeedSkeleton />}>
        <ActivityFeed />
      </Suspense>
    </main>
  );
}
```

**Điều xảy ra:** `StatsPanel`, `RevenueChart`, và `ActivityFeed` fetch **song song** (parallel). Mỗi section stream xuống browser độc lập khi data của nó sẵn sàng. Không có waterfall, không có "toàn trang phải chờ section chậm nhất".

**Trade-off của granularity:**

```
Ít Suspense boundaries:
+ Đơn giản hơn
+ Ít skeleton states
- Một phần chậm block toàn bộ trang
- UX tệ hơn cho sections độc lập

Nhiều Suspense boundaries:
+ Progressive loading — từng phần hiện khi sẵn sàng
+ UX tốt hơn cho non-critical content
- Nhiều skeleton states cần design
- Có thể bị "popcorn effect" (sections pop in lần lượt, giật)
- Cần quản lý nhiều loading states hơn
```

**Nguyên tắc chọn granularity:** Nhóm những thứ cùng tốc độ và cùng "visual group" vào một boundary. Tách những thứ khác tốc độ hoặc độc lập về mặt UI.

### 4.7 Suspense + Error Boundary — Combo Bắt Buộc

Suspense chỉ xử lý "pending" state (Promise chưa resolve). Khi Promise reject (throw error), Suspense **không bắt được** — Error Boundary mới là người bắt.

```
Component render
    ↓
throw Promise → Suspense bắt → show fallback → Promise resolve → retry render
throw Error   → Error Boundary bắt → show error UI (permanent)
```

```tsx
// Pattern chuẩn production — luôn có cả hai
class ErrorBoundary extends React.Component
  { fallback: React.ComponentType; children: React.ReactNode },
  { hasError: boolean }
> {
  state = { hasError: false }

  static getDerivedStateFromError() {
    return { hasError: true }
  }

  render() {
    if (this.state.hasError) {
      return <this.props.fallback />
    }
    return this.props.children
  }
}

// Dùng cùng nhau:
<ErrorBoundary fallback={DataErrorUI}>
  <Suspense fallback={<DataSkeleton />}>
    <AsyncDataComponent />
  </Suspense>
</ErrorBoundary>
```

**Trong Next.js App Router — `error.tsx` đảm nhiệm vai trò ErrorBoundary:**

```
app/
├── dashboard/
│   ├── error.tsx     ← ErrorBoundary tự động (phải là Client Component)
│   ├── loading.tsx   ← Suspense boundary tự động
│   └── page.tsx      ← Component được bảo vệ bởi cả hai
```

```tsx
// app/dashboard/error.tsx
'use client'; // Error boundaries phải là Client Component

// Next.js tự truyền `error` và `reset` props
export default function DashboardError({
  error,
  reset, // Hàm để retry
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  return (
    <div>
      <h2>Có lỗi xảy ra</h2>
      <button onClick={reset}>Thử lại</button>
    </div>
  );
}
```

## 5. Legacy Recognition

**Nhận ra code viết trước khi hiểu Suspense:**

```tsx
// 🔴 Pattern cũ — "fetch-on-render" với manual loading state
// Rất phổ biến trong codebase Next.js 9–12, Pages Router

function UserDashboard({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [posts, setPosts] = useState<Post[] | null>(null);
  const [isUserLoading, setIsUserLoading] = useState(true);
  const [isPostsLoading, setIsPostsLoading] = useState(true);
  const [userError, setUserError] = useState<string | null>(null);

  // Fetch user — mỗi component tự quản lý fetch lifecycle của mình
  useEffect(() => {
    setIsUserLoading(true);
    setUserError(null);
    fetchUser(userId)
      .then((data) => {
        setUser(data);
        setIsUserLoading(false);
      })
      .catch((err) => {
        setUserError(err.message);
        setIsUserLoading(false);
      });
  }, [userId]);

  // Fetch posts — WATERFALL tiềm ẩn: posts không fetch cho đến khi user xong
  useEffect(() => {
    if (!user) return; // ← Đây là waterfall! Đợi user rồi mới fetch posts
    setIsPostsLoading(true);
    fetchPosts(user.id).then((data) => {
      setPosts(data);
      setIsPostsLoading(false);
    });
  }, [user]);

  // Loading state phân tán — phức tạp và dễ bị race condition
  return (
    <div>
      {isUserLoading ? (
        <UserSkeleton />
      ) : userError ? (
        <ErrorMsg />
      ) : (
        <UserCard user={user!} />
      )}
      {isPostsLoading ? <PostsSkeleton /> : <PostsList posts={posts ?? []} />}
    </div>
  );
}
```

**Dấu hiệu nhận ra code "pre-Suspense thinking":**

- `useState(true)` cho `isLoading`, `useState(false)` cho `isError`
- `useEffect` để fetch data
- Conditional render `{isLoading ? <Spinner /> : <Content />}` khắp nơi
- Multiple loading states cho multiple data sources trong cùng component
- Waterfall fetching (fetch B sau khi A xong)
- Race condition tiềm ẩn khi `userId` thay đổi trong lúc fetch đang chạy

## 6. Modern Approach

```typescript
// ✅ Modern approach — Suspense-first thinking

// Mỗi Server Component tự chịu trách nhiệm lấy data của mình
// Không có loading state, không có useEffect, không có race condition
async function UserCard({ userId }: { userId: string }) {
  const user = await db.user.findUnique({ where: { id: userId } })
  if (!user) notFound()  // Next.js built-in — trigger not-found.tsx

  return (
    <div className="user-card">
      <img src={user.avatar} alt={user.name} />
      <h2>{user.name}</h2>
    </div>
  )
}

// Posts fetch độc lập — KHÔNG phụ thuộc vào user
// Tức là cả 2 có thể fetch SONG SONG
async function PostsList({ userId }: { userId: string }) {
  const posts = await db.post.findMany({ where: { authorId: userId } })

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}

// Parent: thuần layout + Suspense orchestration
// Không có state, không có effect — chỉ khai báo structure
function UserDashboard({ userId }: { userId: string }) {
  return (
    <div>
      {/* UserCard và PostsList fetch SONG SONG
          Không có waterfall — cả 2 bắt đầu ngay khi React encounter chúng */}
      <Suspense fallback={<UserSkeleton />}>
        <UserCard userId={userId} />
      </Suspense>

      <Suspense fallback={<PostsSkeleton />}>
        <PostsList userId={userId} />
      </Suspense>
    </div>
  )
}
```

**So sánh trực tiếp — "Fetch patterns":**

```
Fetch-then-render (rất cũ):
  Fetch TẤT CẢ data trước → Rồi mới render
  Vấn đề: Render block cho đến khi data cuối cùng xong

  Ví dụ:
  async function Page() {
  // ⛔ BLOCKING: phải chờ xong hết
  const user = await fetchUser();
  const posts = await fetchPosts();
  /*
  *
  *  0ms    → start fetchUser
  *  1000ms → user xong
  *
  *  1000ms → start fetchPosts
  *  3000ms → posts xong
  *
  *  3000ms → render UI
  *
  */
  return renderUI(user, posts);
  }

Fetch-on-render (phổ biến hiện nay, nhưng không optimal):
  Render → Mount → useEffect → Fetch
  Vấn đề: Mỗi child phải mount trước rồi mới bắt đầu fetch → waterfall

Render-as-you-fetch (Suspense way):
  Fetch bắt đầu → Render song song → Suspend → Data ready → Commit
  Ưu điểm: Không waterfall, progressive display
```

## 7. Production Scenario

**Scenario:** Trang product detail cho một e-commerce platform. Các requirements thực tế:

- Product name/price: từ DB có cache, ~50ms
- Stock status: real-time API (kho thực), không cache, ~200ms
- Product reviews: external review service, ~700ms
- Related products: recommendation engine, ~1.2s

```typescript
// app/products/[id]/page.tsx

// Fetch functions với caching strategy khác nhau
async function getProductDetail(id: string) {
  return fetch(`/api/products/${id}`, {
    next: { revalidate: 3600 }  // Cache 1 giờ
  }).then(r => r.json())
}

async function getStockStatus(id: string) {
  return fetch(`/api/stock/${id}`, {
    cache: 'no-store'  // Không cache — luôn fresh
  }).then(r => r.json())
}

// Server Components — mỗi cái tự fetch, tự suspend
async function ProductHero({ productId }: { productId: string }) {
  const product = await getProductDetail(productId)
  return (
    <section>
      <h1>{product.name}</h1>
      <p className="price">{product.price}</p>
      <p>{product.description}</p>
    </section>
  )
}

async function StockBadge({ productId }: { productId: string }) {
  const stock = await getStockStatus(productId)
  return (
    <span className={stock.available ? 'in-stock' : 'out-of-stock'}>
      {stock.available ? `Còn ${stock.quantity} sản phẩm` : 'Hết hàng'}
    </span>
  )
}

async function ProductReviews({ productId }: { productId: string }) {
  const reviews = await fetch(`https://reviews.external.com/products/${productId}`)
    .then(r => r.json())
  return <ReviewsList reviews={reviews} />
}

async function RelatedProducts({ productId }: { productId: string }) {
  const related = await fetch(`/api/recommendations/${productId}`).then(r => r.json())
  return <ProductGrid products={related} />
}

// Page component: chỉ là orchestration
// Không có state, không có effect, không có loading logic
export default function ProductPage({ params }: { params: { id: string } }) {
  return (
    <main>
      {/* Critical content — hiện sớm nhất, data từ cache */}
      <Suspense fallback={<ProductHeroSkeleton />}>
        <ProductHero productId={params.id} />
      </Suspense>

      {/* Stock: riêng biệt vì no-cache, cần fresh nhất */}
      <Suspense fallback={<StockSkeleton />}>
        <StockBadge productId={params.id} />
      </Suspense>

      {/* Reviews: slow external service, không block hero */}
      <Suspense fallback={<ReviewsSkeleton />}>
        <ProductReviews productId={params.id} />
      </Suspense>

      {/* Recommendations: slowest, non-critical */}
      <Suspense fallback={<RelatedSkeleton />}>
        <RelatedProducts productId={params.id} />
      </Suspense>
    </main>
  )
}
```

**Điều xảy ra từng bước:**

```
1. Server nhận GET /products/123
2. React bắt đầu render ProductPage
3. Encounter ProductHeroSkeleton → stream HTML skeleton xuống ngay
4. Encounter StockSkeleton, ReviewsSkeleton, RelatedSkeleton → stream tiếp
5. Browser nhận HTML — thấy page skeleton, có thể bắt đầu parse/paint

6. Server: ProductHero fetch xong (~50ms) → React stream HTML thật thay skeleton
7. Server: StockBadge fetch xong (~200ms) → stream tiếp
8. Server: Reviews fetch xong (~700ms) → stream tiếp
9. Server: Related fetch xong (~1.2s) → stream cuối cùng

Kết quả:
- TTFB (Time To First Byte): ~50ms (page shell)
- User thấy nội dung đầu tiên sau ~50ms, không phải ~1.2s
- Progressive enhancement — trang "sống dậy" từng phần
```

**Đây chính xác là Streaming SSR — và toàn bộ built trên Suspense.**

## 8. Debugging Scenario

**Bug 1: "Suspense fallback không bao giờ hiện"**

```tsx
// 🔴 Bug: Suspense không triggered vì component không suspend
function ProductList({ categoryId }: { categoryId: string }) {
  const [products, setProducts] = useState<Product[]>([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // useEffect chạy SAU render — component đã render xong
    // Không throw Promise trong lúc render → Suspense không bao giờ biết
    fetch(`/api/products?category=${categoryId}`)
      .then((r) => r.json())
      .then((data) => {
        setProducts(data);
        setLoading(false);
      });
  }, [categoryId]);

  // Component tự xử lý loading state — Suspense wrapper vô nghĩa
  return (
    <Suspense fallback={<FancySkeleton />}>
      {' '}
      {/* ← Không bao giờ trigger */}
      {loading ? <SimpleSkeleton /> : <Grid products={products} />}
    </Suspense>
  );
}
```

**Cách chẩn đoán:** Đặt `console.log` trong component xem nó có render với `products = []` không, hoặc check React DevTools — nếu không thấy Suspense boundary bị activated, component không đang suspend.

**Fix — chuyển sang Server Component:**

```typescript

  const products = await db.product.findMany({ where: { categoryId } })

}

// Bây giờ Suspense mới có tác dụng

```

**Bug 2: Spinner xoay mãi không dừng**

```tsx
// 🔴 Bug: Client Component cố dùng throw Promise thủ công — sai cách

'use client';
function DashboardWidget() {
  const [data, setData] = useState<DashboardData | null>(null);

  useEffect(() => {
    fetchDashboardData().then(setData);
  }, []);

  if (!data) {
    // Lỗi nghiêm trọng: throw Promise không bao giờ resolve
    // VÀ useEffect không chạy vì component chưa bao giờ được mount!
    throw new Promise(() => {}); // ← Empty executor = never resolves

  return <Widget data={data} />;
}
```

**Tại sao bug xảy ra:**

1. Component render lần đầu → `data = null` → throw `new Promise(() => {})`
2. React suspend → show fallback
3. React đăng ký listener trên Promise đó để biết khi nào retry
4. `useEffect` **không chạy** vì component chưa bao giờ commit vào DOM
5. `fetchDashboardData()` **không bao giờ được gọi**
6. Promise không bao giờ resolve
7. Fallback hiện mãi mãi

**Fix options:**

```tsx
// Option 1: Đơn giản — dùng manual loading state (Client Component)
'use client';
function DashboardWidget() {
  const [data, setData] = useState<DashboardData | null>(null);

  useEffect(() => {
    fetchDashboardData().then(setData);
  }, []);

  if (!data) return <WidgetSkeleton />; // ← Không throw, just conditional render
  return <Widget data={data} />;
}

// Option 2: Tốt hơn — dùng use() hook đúng cách
('use client');
function DashboardWidget({
  dataPromise,
}: {
  dataPromise: Promise<DashboardData>;
}) {
  const data = use(dataPromise); // ← use() throw Promise đúng cách nội bộ
  return <Widget data={data} />;
}

// Option 3: Tốt nhất — Server Component
async function DashboardWidget() {
  const data = await fetchDashboardData();
  return <Widget data={data} />;
}
```

## 10. Architecture Notes

**Suspense Placement Strategy — Đặt Suspense ở đâu?**

```
Câu hỏi để quyết định:

1. Content này có phụ thuộc data async không?
   Không → Không cần Suspense

2. Data load ở đây nhanh hay chậm (tương đối)?
   Nhanh (< 200ms, cached) → Có thể nhóm chung với section khác
   Chậm (> 300ms) → Nên có Suspense riêng

3. Content này có thể hiện độc lập không?
   Phụ thuộc vào content khác về mặt layout → Cùng Suspense boundary
   Độc lập → Suspense riêng

4. Đây là critical path (user cần ngay) hay secondary?
   Critical → Ưu tiên load sớm, minimal nesting
   Secondary → Có thể defer với Suspense riêng

5. Nếu section này fail, user còn dùng được trang không?
   Có → Suspense + Error Boundary riêng để isolate failure
   Không → Có thể dùng page-level error.tsx
```

**Anti-pattern: Suspense Wrapping component không async**

```tsx
// 🔴 Anti-pattern — Suspense vô nghĩa
function ProductCard({ product }: { product: Product }) {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      {/* Component này không async — không bao giờ suspend */}
      <div className='card'>{product.name}</div>
    </Suspense>
  );
}

// ✅ Đúng — không cần Suspense cho component đồng bộ
function ProductCard({ product }: { product: Product }) {
  return <div className='card'>{product.name}</div>;
}
```

**Pattern chuẩn: Suspense ở "async entry points"**

```
Rule: Đặt Suspense ngay trên component có await (Server Component)
      hoặc ngay trên component dùng use() hook (Client Component)

Không cần:
<Suspense>
  <SyncWrapper>        ← không async
    <SyncChild>        ← không async
      <AsyncLeaf />    ← async ở đây
    </SyncChild>
  </SyncWrapper>
</Suspense>

Đặt đây:
<SyncWrapper>
  <SyncChild>
    <Suspense>
      <AsyncLeaf />    ← Suspense ngay trên component async
    </Suspense>
  </SyncChild>
</SyncWrapper>
```

**Về performance — Suspense và parallel fetch:**

Khi có nhiều async Server Components trong cùng trang, React render chúng **không theo thứ tự chờ nhau**. Tất cả bắt đầu fetch ngay khi React encounter chúng trong render tree. Đây là default behavior — không cần làm gì thêm để đạt parallel fetching với Server Components.

Nguy cơ waterfall chỉ xảy ra khi:

```typescript
// 🔴 Waterfall: Component B chỉ render sau khi A xong
async function ComponentA() {
  const a = await fetchA()
  return <ComponentB aId={a.id} />  // B phụ thuộc data của A
}

async function ComponentB({ aId }: { aId: string }) {
  const b = await fetchB(aId)       // Chỉ bắt đầu sau A
  return <div>{b.name}</div>
}
```

Cách tránh waterfall khi data có dependency:

```typescript
// ✅ Fetch cả 2 từ parent, pass xuống như props
async function ParentPage({ id }: { id: string }) {
  // Fetch song song với Promise.all
  const [a, b] = await Promise.all([fetchA(id), fetchB(id)])
  return (
    <div>
      <ComponentA data={a} />
      <ComponentB data={b} />
    </div>
  )
}
```

## 11. Interview Questions

**Junior Level**

**Q1: Suspense trong React là gì và bạn dùng nó khi nào?**

<details>
<summary>Đáp án</summary>

React Suspense là cơ chế cho phép component "tạm dừng" (suspend) quá trình render khi data hoặc code chưa sẵn sàng. Khi một component suspend, React sẽ hiện UI fallback từ Suspense boundary gần nhất cho đến khi sẵn sàng.

Dùng khi:

- Lazy loading component với `React.lazy()` — delay load code bundle
- Data fetching với async Server Components (Next.js App Router)
- Đọc Promise với `use()` hook (React 18+)

Ví dụ thực tế nhất trong Next.js:

```tsx
// loading.tsx tự động wrap page trong Suspense
export default function Loading() {
  return <DashboardSkeleton />;
}

// Hoặc viết thủ công cho granular control
<Suspense fallback={<ReviewsSkeleton />}>
  <ProductReviews productId={id} />
</Suspense>;
```

Điểm quan trọng: Suspense không phải chỉ là "cách hiện spinner đẹp hơn". Nó là cơ chế async boundary giúp React biết phần nào của UI đang chờ data và phần nào đã sẵn sàng để stream.

</details>

---

**Q2: Sự khác biệt giữa `React.lazy()` và async Server Component trong Next.js là gì?**

<details>
<summary>Đáp án</summary>

Cả hai đều trigger Suspense nhưng dùng cho mục đích khác nhau:

`React.lazy()` là code splitting — delay việc download JavaScript bundle:

```tsx
// Chỉ download bundle khi component này được render lần đầu
const HeavyChart = React.lazy(() => import('./HeavyChart'))

// Cần Suspense để handle lúc bundle đang tải về
<Suspense fallback={<ChartPlaceholder />}>
  <HeavyChart data={data} />
</Suspense>
```

Async Server Component là data fetching — delay vì đang chờ data từ server:

```typescript
// Component này render trên server, suspend khi đợi DB/API
async function ProductReviews({ productId }: { productId: string }) {
  const reviews = await db.review.findMany({ where: { productId } })
  return <ReviewsList reviews={reviews} />
}
```

Key differences:

- `React.lazy()`: client-side, về code splitting
- Async RSC: server-side, về data fetching
- `React.lazy()` gửi JS xuống client sau khi bundle download xong
- Async RSC không gửi JS (zero client-side JavaScript cho Server Component)

</details>

**Mid Level**

**Q3: Nếu có 3 component bên trong một Suspense boundary và chỉ 1 component suspend, điều gì xảy ra? Làm sao để chỉ 1 component đó bị affect?**

<details>
<summary>Đáp án</summary>

Khi một trong 3 component suspend, toàn bộ nội dung bên trong Suspense boundary bị thay thế bởi fallback — kể cả 2 component đã ready. Đây là "all-or-nothing semantics" của Suspense.

```tsx
<Suspense fallback={<Loading />}>
  <ComponentA /> {/* đã ready */}
  <ComponentB /> {/* SUSPEND */}
  <ComponentC /> {/* đã ready */}
</Suspense>
// Kết quả: <Loading /> hiện, ComponentA và ComponentC bị ẩn
```

Để chỉ ComponentB bị affected, dùng nested Suspense:

```tsx
<ComponentA />          {/* render ngay */}
<Suspense fallback={<LoadingB />}>
  <ComponentB />        {/* chỉ B bị suspend */}
</Suspense>
<ComponentC />          {/* render ngay */}
```

Quyết định dùng shared hay separate boundary phụ thuộc vào UX:

- Dùng shared boundary khi các component cần xuất hiện cùng nhau (tránh layout jump)
- Dùng separate boundary khi các component độc lập về visual, user có thể dùng A và C trong khi đợi B

</details>

---

**Q4: Tại sao `useEffect` + `setState` không trigger Suspense nhưng `use()` hook lại trigger được?**

<details>
<summary>Đáp án</summary>

Nguyên nhân nằm ở **thời điểm** code chạy trong React lifecycle:

`useEffect` chạy SAU khi component đã render và commit vào DOM. Tại thời điểm đó, React đã "xong" với render phase. Không có cơ chế nào để "quay ngược" và kích hoạt Suspense. Loading state phải được quản lý thủ công.

```tsx
function Component() {
  const [data, setData] = useState(null);

  // Chạy sau render — không thể trigger Suspense
  useEffect(() => {
    fetch('/api/data').then(setData);
  }, []);

  if (!data) return <Skeleton />; // Manual handling
  return <Content data={data} />;
}
```

`use()` hook thực thi TRONG lúc render. Khi React đang render component và gặp `use(promise)` với promise chưa resolve, `use()` throw một thenable ngay trong render phase. React bắt thenable này → kích hoạt Suspense.

```tsx
function Component({ dataPromise }: { dataPromise: Promise<Data> }) {
  // Chạy trong render phase — CÓ THỂ throw thenable → trigger Suspense
  const data = use(dataPromise);
  return <Content data={data} />;
}
```

Tóm lại: Suspense chỉ hoạt động với thứ xảy ra trong **render phase**. `useEffect` là post-render, `use()` là in-render.

</details>

**Senior Level**

**Q5: `startTransition` thay đổi hành vi Suspense như thế nào? Trong Next.js, cơ chế này ảnh hưởng đến navigation ra sao?**

<details>
<summary>Đáp án</summary>

Không có `startTransition`, khi state update khiến component mới suspend, React lập tức show fallback — dù component chỉ cần 50ms để load xong. Điều này tạo "flash of loading" khó chịu.

Với `startTransition`, React đánh dấu update là "non-urgent transition":

- React GIỮ UI hiện tại trong khi xử lý transition
- Nếu new content load đủ nhanh, React swap trực tiếp — không show fallback
- Nếu chậm (user đợi lâu), React eventually show fallback

```tsx
const [isPending, startTransition] = useTransition();

function navigate(href: string) {
  startTransition(() => {
    router.push(href); // Navigation là "non-urgent"
  });
}
// isPending = true trong khi navigate đang xử lý
// Có thể dùng isPending cho progress indicator nhẹ thay vì full page spinner
```

Trong Next.js App Router:

- `<Link>` component tự động wrap navigation trong `startTransition`
- Kết quả: click Link → trang hiện giữ nguyên → trang mới load → swap (nếu nhanh)
- `loading.tsx` chỉ hiện khi load thực sự chậm (React timeout)
- Điều này giải thích tại sao App Router navigation cảm thấy "instant" hơn Pages Router với fast connections

Trade-off: `startTransition` có thể gây cảm giác "button không respond" nếu không có visual feedback. Dùng `isPending` để show progress bar hoặc disable button trong khi pending.

</details>

**Q6: Giải thích cách Next.js `loading.tsx` và Suspense tự viết khác nhau và khi nào nên dùng cái nào.**

<details>
<summary>Đáp án</summary>

`loading.tsx` là abstraction của Next.js — tự động tạo Suspense wrapper cho toàn bộ route segment:

```
# Về mặt code, Next.js làm điều này nội bộ:
import Loading from './loading'
import Page from './page'

export default function SegmentWrapper(props) {
  return (
    <Suspense fallback={<Loading />}>
      <Page {...props} />

```

Hạn chế của `loading.tsx`:

- Chỉ wrap toàn bộ page — all-or-nothing cho toàn trang
- Không thể có granularity trong page
- Không kiểm soát được placement

Dùng `loading.tsx` khi:

- Muốn page-level loading (toàn trang load cùng lúc)
- Đơn giản, không cần granular control
- Skeleton cho toàn bộ page layout

Dùng Suspense tự viết khi:

- Cần granular loading: từng section load riêng
- Muốn một số section hiện ngay (từ cache) trong khi section khác đang load
- Cần nested Suspense cho streaming
- Muốn khác nhau skeleton cho từng component

Ví dụ kết hợp:

```typescript
// app/products/page.tsx
```

```tsx
    fetch(`/api/products?category=${categoryId}`)
      .then((r) => r.json())
      .then((data) => {
        setProducts(data);
        setLoading(false);
      });
  }, [categoryId]);

  return (
    <Suspense fallback={<FancyGridSkeleton />}>
      {loading ? <SimpleSpinner /> : <ProductGrid products={products} />}
    </Suspense>
  );
}
```

<details>
<summary>Đáp án</summary>

**(a) Suspense không hoạt động đúng.**

`useEffect` chạy sau render, không phải trong render. Component không bao giờ throw Promise trong lúc render, vì vậy Suspense boundary không bao giờ được trigger. `<FancyGridSkeleton />` sẽ không bao giờ hiện.

**(b) `<SimpleSpinner />` hiện khi data đang load.**

Đó là fallback thủ công bên trong conditional render, không phải Suspense fallback. Hai loading states tồn tại song song nhưng chỉ cái trong conditional render thực sự hoạt động.

**(c) Có race condition.**

Nếu `categoryId` thay đổi nhanh (user click nhiều filter), có thể có nhiều fetch đang chạy cùng lúc. Fetch nào về sau sẽ set state cuối cùng, dù đó không phải request mới nhất. Kết quả: hiện data của category cũ.

**Fix đầy đủ:**

```typescript
// Chuyển sang Server Component — giải quyết cả 3 vấn đề
async function ProductList({ categoryId }: { categoryId: string }) {
  const products = await db.product.findMany({
    where: { categoryId }
  })
  return <ProductGrid products={products} />
}

// Trong parent:
<Suspense fallback={<FancyGridSkeleton />}>
  <ProductList categoryId={categoryId} />
</Suspense>
```

Server Components không có race condition (mỗi request là một server render riêng), Suspense hoạt động đúng, và không cần state/effect.

**Exercise 2 — Thiết kế Suspense boundaries**

Bạn đang build trang profile người dùng cho một social network. Trang gồm:

- **Avatar + tên**: từ cache CDN, cực nhanh (~30ms)

Thiết kế Suspense boundaries và giải thích từng quyết định.

**Exercise 3 — Debug Production Bug**

Bug report từ team: "Trang `/checkout` hiện loading spinner mãi mãi khi user ở trên mobile connection chậm. Trên WiFi thì ok. Code đây:"

```tsx
// app/checkout/page.tsx (App Router)

  const cart = await getCart(cartId);
  const shipping = await calculateShipping(cart); // Phụ thuộc cart
  const tax = await calculateTax(cart, shipping); // Phụ thuộc shipping
  return (
    <Summary
      cart={cart}
      shipping={shipping}
      tax={tax}
    />
  );
}

export default function CheckoutPage() {
  const cartId = getCartIdFromCookie();

  return (
    <Suspense fallback={<CheckoutSkeleton />}>
      <OrderSummary cartId={cartId} />
    </Suspense>
  );
}
```

Xác định vấn đề performance và đề xuất fix.

<details>
<summary>Đáp án</summary>

Có 2 vấn đề:

**Vấn đề 1: Sequential waterfall — 3 requests chạy tuần tự**

```
getCart()  → xong → calculateShipping(cart) → xong → calculateTax(cart, shipping)
   ~300ms                          ~400ms                          ~200ms
                                                              Tổng: ~900ms
```

`calculateShipping` không bắt đầu cho đến khi `getCart` xong. `calculateTax` không bắt đầu cho đến khi `calculateShipping` xong. Trên mobile connection chậm, mỗi request thêm latency overhead → tổng thời gian nhân 3.

**Vấn đề 2: Artificial dependency**

`calculateShipping` có thể chỉ cần `cart.items` và `cart.address`, không cần toàn bộ cart. `calculateTax` có thể tính được từ `cart.total` và `shipping.amount` — có thể bắt đầu sau khi có đủ thông tin, không nhất thiết phải tuần tự.

**Fix — Parallel fetch khi có thể, pipeline khi phải sequential:**

```typescript
async function OrderSummary({ cartId }: { cartId: string }) {
  // Bước 1: Fetch cart
  const cart = await getCart(cartId)

  // Bước 2: Fetch shipping và các thứ không phụ thuộc nhau — SONG SONG
  const [shipping, discounts] = await Promise.all([
    calculateShipping(cart),
    getApplicableDiscounts(cart)  // Ví dụ: fetch thêm data không phụ thuộc shipping
  ])

  // Bước 3: Tax phụ thuộc shipping — phải sequential
  const tax = await calculateTax(cart, shipping)

  return <Summary cart={cart} shipping={shipping} tax={tax} discounts={discounts} />
}
```

Hoặc tốt hơn — tách thành API endpoint gom tất cả logic:

```typescript
async function OrderSummary({ cartId }: { cartId: string }) {
  // Một request thay vì 3 — latency giảm đáng kể trên mobile
  const { cart, shipping, tax } = await getCheckoutSummary(cartId)
  return <Summary cart={cart} shipping={shipping} tax={tax} />
}
```

## 13. Summary

**Suspense là async boundary — không phải loading UI helper.**

Những điều cốt lõi cần nắm vững:

**Cơ chế:** Component "suspend" bằng cách throw một thenable (Promise) trong lúc render. React bắt thenable từ Suspense ancestor gần nhất, show fallback, và retry khi Promise resolve. Đây là extension của Error Boundary mechanism.

**Boundary semantics — 3 rules:**

- Nearest ancestor: chỉ Suspense gần nhất bị trigger, không phải tất cả ancestors
- All-or-nothing: toàn bộ subtree trong một boundary show fallback cùng nhau
- Atomic reveal: khi Promise resolve, toàn bộ subtree hiện ra cùng lúc

**Fallback timing:**

- Initial render: fallback hiện ngay lập tức khi suspend
- `startTransition`: React giữ UI cũ, chỉ show fallback khi load thực sự chậm
- Next.js `<Link>` dùng `startTransition` nội bộ → navigation mượt mà hơn

**Trong Next.js — 4 cấp độ:**

- `loading.tsx`: page-level Suspense tự động
- Suspense thủ công trong page: component-level granular
- Async Server Components: suspend tự động khi `await`
- Streaming SSR: HTML stream từng phần dựa trên Suspense resolution

**Luôn nhớ combo:** Suspense xử lý "pending", Error Boundary xử lý "error". Hai cái không thay thế nhau. Production code cần cả hai.

**Và quan trọng nhất:** Suspense không fix waterfall. Performance issue trong data fetching phải được giải quyết ở fetching layer. Suspense chỉ quyết định _khi nào và cách nào_ React hiện UI trong khi chờ data.

Bài tiếp theo M1.5 sẽ đi sâu vào Concurrent Rendering — cơ sở cho `startTransition`, `useDeferredValue`, và cách React 18 prioritize work. Những concept đó sẽ giải thích rõ hơn tại sao Suspense behave khác nhau trong các contexts.
