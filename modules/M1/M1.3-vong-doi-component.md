**2. Evolution Context (Bài này xuất hiện để giải quyết vấn đề gì?)**

**Trước đó**

- jQuery: dev tự quản lý DOM
- Class Component (React 0.x–16): lifecycle theo thời gian (mount/update/unmount)
- Stack Reconciler → Fiber Reconciler (M1.2)

↓

**Vấn đề**

- Class lifecycle: một concern bị **xé thành 3 nơi** (`componentDidMount` + `componentDidUpdate` + `componentWillUnmount`)
- Dễ quên đồng bộ: thêm subscribe trong mount nhưng quên xử lý props đổi trong `componentDidUpdate`
- Tái sử dụng logic khó: phải dùng HOC/render props → "wrapper hell", cây component sâu hơn
- `useEffect` bị hiểu nhầm = `componentDidMount` về timing

↓

**Giải pháp**

- Hooks (React 16.8, 2019): gom concern vào **một closure duy nhất** với dependency array
- `useEffect` theo trục **đồng bộ hoá** (sync với giá trị nào) thay vì trục **thời gian** (xảy ra khi nào)
- Custom hook thay thế HOC/render props để tái sử dụng logic

**3. Mental Model (Ý tưởng cốt lõi)**

> **Class lifecycle = trục thời gian** (việc này xảy ra khi nào: mount/update/unmount). **Hooks = trục đồng bộ hoá** (effect này phụ thuộc giá trị nào, chạy lại khi giá trị đó đổi). Dependency array không phải "danh sách trigger" — nó là khai báo "effect phụ thuộc vào đúng các giá trị này, không hơn không kém".

> **Function component không có "instance" sống xuyên suốt.** Mỗi lần render = lần gọi hàm mới. State nằm trên **Fiber node** (`memoizedState` linked list), mỗi hook chiếm một node theo đúng **thứ tự gọi**.

**4. Core Concepts (20% kiến thức quan trọng nhất)**

**Concept 1 — Timing: `useLayoutEffect` vs `useEffect`**

```
MOUNT/UPDATE:
  render() chạy
    ↓
  React commit DOM
    ↓
  useLayoutEffect — ĐỒNG BỘ, TRƯỚC browser paint
    ↓
  Browser paint
    ↓
  useEffect — BẤT ĐỒNG BỘ, SAU paint (React 17)
              PHỤ THUỘC CONTEXT (React 18/19 — interaction-driven có thể trước paint)
```

|                 | `componentDidMount/Update` | `useLayoutEffect`         | `useEffect`                                     |
| --------------- | -------------------------- | ------------------------- | ----------------------------------------------- |
| **Timing**      | Đồng bộ, trước paint       | Đồng bộ, trước paint      | Bất đồng bộ, sau paint (hoặc phụ thuộc context) |
| **Tương đương** | —                          | ✅ Chính xác              | ❌ Khác                                         |
| **Dùng khi**    | Đo/sửa layout trước paint  | Đo/sửa layout trước paint | Side effect không cần block paint               |

> `useLayoutEffect` mới là timing tương đương `componentDidMount` thật. `useEffect` khác về timing.

**Concept 2 — Dependency Array**

| Pattern                 | Tương đương class                            | Lưu ý                                |
| ----------------------- | -------------------------------------------- | ------------------------------------ |
| `useEffect(fn)`         | Không có tương đương                         | Chạy sau MỌI render                  |
| `useEffect(fn, [])`     | `componentDidMount` + `componentWillUnmount` | Timing thật = `useLayoutEffect`      |
| `useEffect(fn, [a, b])` | `componentDidUpdate` tự so sánh `prevProps`  | React tự so sánh, không cần thủ công |

**Concept 3 — Cleanup Function**

- Chạy ở **hai thời điểm**: (1) trước khi effect chạy lại (dependency đổi), (2) khi unmount
- Class: phải lặp code ở `componentDidUpdate` **và** `componentWillUnmount`
- Hooks: một cleanup function duy nhất, React tự gọi đúng lúc

**Concept 4 — Rules of Hooks**

| Quy tắc                                           | Lý do (từ Fiber)                                                                                                                     |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| Chỉ gọi ở top level, không trong điều kiện/loop   | React match state bằng **vị trí trong linked list `memoizedState`**, không bằng tên. Gọi có điều kiện → thứ tự lệch → đọc nhầm state |
| Chỉ gọi trong function component hoặc custom hook | Đảm bảo gọi trong context có Fiber node                                                                                              |

**Concept 5 — Stale Closure**

- Effect "đóng" (closure) lấy giá trị tại thời điểm tạo
- Thiếu dependency → effect không chạy lại → dùng giá trị cũ
- Fix: `eslint-plugin-react-hooks/exhaustive-deps` — không phải gợi ý, là điều kiện correctness

**Concept 6 — Class component chưa biến mất**

- **Error Boundary** (`getDerivedStateFromError`, `componentDidCatch`): vẫn chỉ tồn tại ở class component, chưa có hook chính thức thay thế
- Codebase hooks-first vẫn giữ class `ErrorBoundary` nhỏ, viết một lần, dùng nhiều nơi

**5. Legacy Recognition (Nhìn code biết gì?)**

Dấu hiệu nhận biết

✅

- `componentWillMount`/`componentWillReceiveProps`/`componentWillUpdate` không prefix `UNSAFE_` — giả định "chạy 1 lần" không còn đúng dưới Fiber
- Logic subscribe rải ra 3 method (`componentDidMount` + `componentDidUpdate` + `componentWillUnmount`) với so sánh `prevProps` thủ công
- HOC/render props chỉ để chia sẻ stateful logic — dấu hiệu cần custom hook
- `this.setState(newState, callback)` để "chờ" state cập nhật
- Constructor đầy `this.handleX = this.handleX.bind(this)`

❌

- `useEffect` copy nguyên logic từ `componentDidMount` + `componentDidUpdate` + `componentWillUnmount` vào 3 `useEffect` riêng — migrate sai
- Disable `eslint-plugin-react-hooks/exhaustive-deps` "tạm thời" rồi không sửa

---

**6. Modern Approach (Ngày nay làm thế nào?)**

| Legacy                                                              | Modern                                                                    |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| Class lifecycle (time-based)                                        | Hooks (synchronization-based)                                             |
| Concern xé ra 3 method                                              | Một `useEffect` per concern                                               |
| HOC/render props để share logic                                     | Custom hook (`useDocumentTitle`, `useSubscription`)                       |
| `componentDidMount` + `componentDidUpdate` + `componentWillUnmount` | `useEffect(fn, [deps])` với cleanup function                              |
| `this.setState` + callback                                          | `useEffect` theo dõi state                                                |
| `componentDidMount` timing                                          | `useLayoutEffect` (nếu cần trước paint), `useEffect` (mặc định)           |
| `useEffect` cho mọi thứ                                             | `use` hook (React 19) cho declarative data fetching, suspend trong render |

**7. Production Scenario (Thực tế)**

**Case: Badge trạng thái thanh toán real-time — "hiển thị sai hợp đồng"**

- **Context:** Badge nhận dữ liệu qua WebSocket subscription theo `contractId`
- **Problem:** User chọn hợp đồng khác trong danh sách → badge vẫn hiển thị trạng thái hợp đồng cũ
- **Root Cause:** Class component, thiếu `componentDidUpdate` — chỉ subscribe trong `componentDidMount`, quên resubscribe khi `contractId` đổi
- **Solution:** `useEffect` với dependency `[contractId]` — React tự gọi cleanup cũ + subscribe mới khi `contractId` đổi
- **Lesson:** Class: 3 nơi cho 1 concern, dễ quên đồng bộ. Hooks: 1 closure, React lo phần còn lại

👉 Rule:

> Nếu gặp **component vẫn "hoạt động" lúc mount nhưng sai khi props đổi** hãy nghĩ ngay đến **thiếu `componentDidUpdate` (class) hoặc thiếu dependency trong `useEffect` (hooks)**.

**8. Debugging Scenario (Debug nhanh)**

**Bug: "Badge gọi callback với dữ liệu hợp đồng trước đó"**

```jsx
useEffect(() => {
  const subscription = subscribeToPaymentStatus(contractId, (newStatus) => {
    setStatus(newStatus);
    onStatusChange(newStatus); // dùng onStatusChange từ closure
  });
  return () => subscription.unsubscribe();
}, [contractId]); // thiếu onStatusChange
```

Checklist:

- □ Bật `eslint-plugin-react-hooks/exhaustive-deps` — cảnh báo: missing dependency `onStatusChange`
- □ Hiểu: effect tạo ở lần render đầu, closure lấy `onStatusChange` đời cũ. `contractId` không đổi → effect không chạy lại → callback cũ tiếp tục được dùng
- □ Fix: thêm `onStatusChange` vào dependency — nhưng nếu cha tạo callback mới mỗi render (không `useCallback`), effect resubscribe liên tục
- □ Fix đúng: cha bọp `onStatusChange` trong `useCallback`, hoặc lưu callback mới nhất vào `ref` để đọc trong effect mà không cần đưa vào dependency

**9. Refactoring Scenario (Nhìn code cũ sẽ làm gì?)**

**Trước — Class component, 3 concern gom chung, lặp code:**

```jsx
class ContractDetailPage extends React.Component {
  componentDidMount() {
    document.title = `Hợp đồng #${this.props.contractId}`;
    this.subscription = subscribeToPaymentStatus(
      this.props.contractId,
      this.handleStatusChange,
    );
    logAnalyticsEvent('view_contract_detail', {
      contractId: this.props.contractId,
    });
  }

  componentDidUpdate(prevProps) {
    if (prevProps.contractId !== this.props.contractId) {
      document.title = `Hợp đồng #${this.props.contractId}`;
      this.subscription.unsubscribe();
      this.subscription = subscribeToPaymentStatus(
        this.props.contractId,
        this.handleStatusChange,
      );
      logAnalyticsEvent('view_contract_detail', {
        contractId: this.props.contractId,
      });
    }
  }

  componentWillUnmount() {
    this.subscription.unsubscribe();
  }
  // ...
}
```

**Sau — Hooks, mỗi concern = một custom hook, tái sử dụng được:**

```jsx
function useDocumentTitle(title) {
  useEffect(() => {
    document.title = title;
  }, [title]);
}

function usePaymentStatusSubscription(contractId, onChange) {
  useEffect(() => {
    const subscription = subscribeToPaymentStatus(contractId, onChange);
    return () => subscription.unsubscribe();
  }, [contractId, onChange]);
}

function useAnalyticsView(eventName, payload) {
  useEffect(() => {
    logAnalyticsEvent(eventName, payload);
  }, [eventName, payload]);
}

function ContractDetailPage({ contractId }) {
  const [status, setStatus] = useState(null);
  useDocumentTitle(`Hợp đồng #${contractId}`);
  usePaymentStatusSubscription(contractId, setStatus);
  useAnalyticsView('view_contract_detail', { contractId });
  return <span>{status}</span>;
}
```

Không sửa khi:

- Chưa audit lifecycle methods không an toàn (`UNSAFE_`)
- Chưa có test cho logic sau khi tách

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng**

- `useEffect`: mặc định cho side effects không cần block paint
- `useLayoutEffect`: đo/sửa layout trước paint (measure DOM, scroll position)
- Custom hook: logic có thể tái sử dụng, thay cho HOC/render props
- `useReducer`: state phức tạp, nhiều field liên quan, transition rõ ràng

**Khi không nên dùng**

- `useLayoutEffect` cho mọi thứ: block paint, có thể gây jank
- `useEffect` copy nguyên 3 lifecycle method vào 3 effect riêng — migrate sai
- Disable `exhaustive-deps`: gây stale closure, bug khó trace

Trade-off

| Pros                                               | Cons                                                  |
| -------------------------------------------------- | ----------------------------------------------------- |
| Hooks: concern gom một chỗ, dễ đọc, dễ test        | Dependency array: dễ thiếu/thừa, cần hiểu closure     |
| Custom hook: tái sử dụng logic, không wrapper hell | Rules of Hooks: constraint về thứ tự gọi              |
| Cleanup tự động: không lặp code                    | `useEffect` timing khác `componentDidMount` — dễ nhầm |
| `use` hook (React 19): declarative data fetching   | Mới, cần hiểu suspend trong render                    |

## 2. Evolution Context

### Pain Point

Một component thực tế thường có nhiều "concern" độc lập cần đồng bộ với thế giới bên ngoài: subscribe một WebSocket, set `document.title`, log analytics, đặt một interval. Với class lifecycle, các concern này **không có chỗ riêng** — chúng buộc phải gom chung vào `componentDidMount`, rồi lặp lại tương tự trong `componentDidUpdate` (nếu cần re-run khi props đổi), rồi lại lặp lại lần nữa trong `componentWillUnmount` (để cleanup). Một concern logic duy nhất bị **xé thành 2-3 nơi khác nhau**, mỗi nơi phải tự tay giữ đồng bộ với hai nơi còn lại.

### Cách cũ: Class Lifecycle

React tổ chức vòng đời component theo 3 phase, mỗi phase có method riêng:

```text
MOUNTING:
  constructor()
  static getDerivedStateFromProps(props, state)
  render()
  componentDidMount()

UPDATING (khi props/state đổi):
  static getDerivedStateFromProps(props, state)
  shouldComponentUpdate(nextProps, nextState)
  render()
  getSnapshotBeforeUpdate(prevProps, prevState)
  componentDidUpdate(prevProps, prevState)

UNMOUNTING:
  componentWillUnmount()
```

Side effect (subscribe, timer, fetch) phải đặt trong `componentDidMount`. Nếu effect đó phụ thuộc vào props, bạn phải tự viết logic so sánh `prevProps` trong `componentDidUpdate` để biết có nên re-run không. Cleanup luôn đặt trong `componentWillUnmount` — tách biệt hoàn toàn về vị trí code so với nơi effect được tạo ra.

### Tại sao thất bại

- **Logic theo "thời điểm", không theo "mục đích".** Một concern (ví dụ: subscription) bị chia theo câu hỏi "việc này xảy ra ở khoảnh khắc nào trong đời component" thay vì "việc này đồng bộ với điều gì". Đọc `componentDidMount` của một component lớn, bạn thấy 5 concern không liên quan nằm cạnh nhau.
- **Dễ quên đồng bộ.** Thêm subscribe trong `componentDidMount` nhưng quên xử lý case props đổi trong `componentDidUpdate` là lỗi phổ biến nhất trong codebase class component thực tế — component vẫn "hoạt động" lúc mount, bug chỉ lộ ra khi props đổi sau đó.
- **Tái sử dụng logic khó.** Muốn dùng lại logic "subscribe theo props" ở hai component khác nhau, bạn phải dùng Higher-Order Component hoặc render props — cả hai đều tạo thêm một layer component bọc ngoài ("wrapper hell"), làm cây component sâu hơn và khó debug hơn (nhớ lại M1.2: mỗi layer bọc thêm là một layer reconciliation thêm).

### Cách mới: Hooks

`useEffect` gom toàn bộ vòng đời của MỘT concern — khởi tạo, re-run khi dependency đổi, cleanup khi unmount hoặc trước lần re-run kế tiếp — vào **một closure duy nhất**:

```jsx
useEffect(() => {
  const subscription = subscribeToSomething(dependency);
  return () => subscription.unsubscribe(); // cleanup
}, [dependency]);
```

React tự động: gọi effect sau mount; khi `dependency` đổi, tự gọi cleanup cũ rồi chạy lại effect mới; khi component unmount, tự gọi cleanup. Bạn không tự so sánh `prevProps` — React làm việc đó qua dependency array.

### Trade-offs

- Tư duy "theo concern" buộc bạn phải hiểu rõ **dependency array** — liệt kê thiếu một giá trị reactive dẫn đến stale closure (effect dùng giá trị cũ); liệt kê thừa dẫn đến effect chạy lại nhiều hơn cần thiết.
- Một component giờ có thể có nhiều `useEffect` riêng biệt — tốt cho separation of concern, nhưng cũng có nghĩa bạn phải tự suy luận thứ tự chạy giữa nhiều effect (chạy theo thứ tự khai báo trong code).
- `useEffect` và `componentDidMount` **không cùng timing thực thi** (xem Core Concepts) — đây là nguồn nhầm lẫn rất phổ biến khi migrate.

### Production Reality

Bug kinh điển nhất khi chuyển từ class sang hooks không nằm ở cú pháp, mà ở việc thiếu dependency trong array — đội ngũ React giải quyết bằng eslint rule `exhaustive-deps`, cảnh báo ngay khi build, nhưng codebase legacy thường disable rule này "tạm thời" rồi không bao giờ sửa lại.

### Migration

Migrate một class component sang hooks **không phải** chuyển 1-1 từng lifecycle method sang một hook tương đương — đó là cách migrate sai phổ biến nhất (ví dụ máy móc copy nguyên logic từ `componentDidMount` + `componentDidUpdate` + `componentWillUnmount` vào 3 `useEffect` riêng). Migrate đúng là **xác định lại concern**, rồi viết một `useEffect` cho mỗi concern với dependency array phản ánh đúng giá trị mà concern đó phụ thuộc.

### Interview Perspective

Câu hỏi senior thường dùng để lọc: _"Bạn nói `useEffect` tương đương `componentDidMount` — vậy chúng có chạy ở cùng thời điểm không?"_ — câu trả lời đúng phải nói "không", và chỉ ra `useLayoutEffect` mới là timing tương đương thật.

## 3. Mental Model

**Mô hình 1 — Class lifecycle là trục thời gian; Hooks là trục đồng bộ hoá.**
Class lifecycle trả lời câu hỏi "việc này xảy ra khi nào" (mount/update/unmount). `useEffect` trả lời câu hỏi khác hẳn: "effect này đồng bộ với những giá trị nào, và khi giá trị đó đổi, effect chạy lại". Dependency array không phải "danh sách trigger" theo nghĩa imperative — nó là khai báo "effect này phụ thuộc vào đúng các giá trị này, không hơn không kém".

**Mô hình 2 — Function component không có "instance" sống xuyên suốt nhiều lần render.**
Class component có `this`, một instance JS thật tồn tại liên tục, state là property gắn trên instance đó. Function component thì mỗi lần render là một **lần gọi hàm mới** — không có gì "tồn tại" giữa hai lần gọi theo nghĩa JS thông thường. State của hooks thực ra nằm trên **Fiber node** (đã học ở M1.2): `memoizedState` là một linked list, mỗi hook chiếm một node trong list đó theo đúng **thứ tự gọi**. Hàm component "mượn" lại đúng node tương ứng mỗi lần render — đây là lý do Rules of Hooks tồn tại (xem Core Concepts).

## 4. Core Concepts

### Timing: useLayoutEffect vs useEffect

Đây là điểm gây nhầm lẫn nhiều nhất khi so sánh với class lifecycle:

```text
MOUNT:
  render() chạy (function body thực thi)
  -> React commit DOM
  -> useLayoutEffect chạy — ĐỒNG BỘ, TRƯỚC khi browser paint
  -> Browser paint lên màn hình
  -> useEffect chạy — BẤT ĐỒNG BỘ, SAU khi paint

UPDATE (dependency đổi):
  render() chạy lại
  -> React commit DOM mới
  -> cleanup useLayoutEffect cũ chạy, rồi useLayoutEffect mới chạy (đồng bộ, trước paint)
  -> Browser paint
  -> cleanup useEffect cũ chạy, rồi useEffect mới chạy (bất đồng bộ, sau paint)

UNMOUNT:
  -> cleanup useLayoutEffect chạy (đồng bộ)
  -> cleanup useEffect chạy
```

**`componentDidMount`/`componentDidUpdate` có timing khớp với `useLayoutEffect`** — cả hai đều chạy đồng bộ, sau DOM commit, trước browser paint. `useEffect` truyền thống chạy bất đồng bộ, sau paint — nhưng từ React 18+, behavior này có thể thay đổi tùy context (interaction-driven updates có thể chạy `useEffect` trước paint). Điều này làm cho sự khác biệt giữa useEffect và useLayoutEffect mờ nhạt hơn — nhưng useLayoutEffect vẫn đảm bảo chạy sync trước paint trong MỌI trường hợp, không phụ thuộc vào trigger. Để đảm bảo timing chặt chẽ nhất, vẫn nên dùng useLayoutEffect cho DOM measurements.

### Dependency Array — bản đồ quy đổi từ class

```text
useEffect(fn)             -> chạy sau MỌI lần render (không có lifecycle class tương đương trực tiếp)
useEffect(fn, [])         -> chạy 1 lần sau mount + cleanup khi unmount
                              (gần với componentDidMount + componentWillUnmount,
                               nhưng timing thật khớp với useLayoutEffect nếu đổi sang đó)
useEffect(fn, [a, b])     -> chạy lại khi a hoặc b đổi
                              (gần với componentDidUpdate tự so sánh prevProps.a !== a)
```

### Cleanup Function — giải quyết tận gốc bug "quên đồng bộ"

Hàm return từ effect chạy ở **hai thời điểm**: (1) ngay trước khi effect chạy lại lần kế tiếp (do dependency đổi), và (2) khi component unmount. Class component không có khái niệm tương đương cho (1) — `componentWillUnmount` CHỈ chạy khi unmount thật, nên muốn "cleanup trước khi resubscribe" bạn phải tự gọi cleanup logic đó cả trong `componentDidUpdate` **và** `componentWillUnmount` (trùng code). Hooks gộp hai trường hợp đó thành một cleanup function duy nhất.

### Rules of Hooks — và vì sao nó tồn tại

Hai quy tắc: (1) chỉ gọi hook ở top level, không trong điều kiện/loop/hàm lồng; (2) chỉ gọi hook trong function component hoặc custom hook.

Lý do (nối tiếp Mental Model 2): React không match state với hook bằng _tên_, mà bằng **vị trí trong linked list `memoizedState`** trên Fiber node — hook đầu tiên gọi luôn lấy node đầu tiên, hook thứ hai luôn lấy node thứ hai, bất kể tên biến là gì. Nếu một hook bị gọi có điều kiện (`if (x) useState(...)`), thứ tự gọi giữa hai lần render có thể lệch nhau — hook thứ 3 ở lần render này có thể vô tình đọc nhầm state lưu cho hook thứ 2 ở lần render trước, gây bug khó trace vì không có exception rõ ràng (có exception nếu số lượng hook đổi giữa hai lần render, nhưng nếu số lượng không đổi mà chỉ thứ tự đổi thì bug âm thầm hơn).

## 5. Legacy Recognition

- `componentWillMount`, `componentWillReceiveProps`, `componentWillUpdate` (không có prefix `UNSAFE_`) — đã học ở M1.2 rằng các method này được viết với giả định "chỉ chạy một lần", giả định không còn đúng dưới Fiber.
- Logic subscribe/unsubscribe bị rải ra `componentDidMount` + `componentDidUpdate` + `componentWillUnmount`, có so sánh `prevProps` thủ công.
- HOC hoặc render props chỉ dùng để chia sẻ **stateful logic** (không liên quan UI) — dấu hiệu rõ ràng rằng một custom hook sẽ đơn giản hơn nhiều.
- `this.setState(newState, callback)` dùng callback để "chờ" state cập nhật xong rồi làm gì đó tiếp — thường có thể thay bằng một `useEffect` theo dõi giá trị state đó.
- Constructor đầy `this.handleX = this.handleX.bind(this)` — boilerplate đặc trưng của thời chưa có class field arrow function hoặc hooks.

## 6. Modern Approach

- Tổ chức effect theo **concern**, mỗi `useEffect` chỉ làm một việc — không gộp nhiều side effect không liên quan vào một effect.
- Dependency array phải **đầy đủ** (tuân theo eslint `exhaustive-deps`) — đây không phải gợi ý, đây là điều kiện để effect không bị stale closure.
- Đóng gói logic side-effect có thể tái sử dụng thành **custom hook** (`useDocumentTitle`, `usePaymentStatusSubscription`...) thay cho HOC/render props.
- Dùng `useLayoutEffect` chỉ khi thực sự cần đo/sửa layout trước paint — mặc định nên dùng `useEffect`.
- Với state phức tạp (nhiều field liên quan, transition rõ ràng), `useReducer` thường rõ ràng hơn nhiều `useState` rời rạc — đây là phiên bản hiện đại của pattern "state machine" mà nhiều class component cũ tự chế bằng cách so sánh cờ boolean thủ công.
- `use` Hook : React 19 giới thiệu use hook — một cách mới để read resources (Promises, Context) trong render phase. Điều này thay đổi cách chúng ta nghĩ về "effects" và "data fetching". `use` có thể được gọi có điều kiện (khác với các hooks khác) — nhưng vẫn phải tuân theo rules trong một số context. `use` cho phép "suspend" component trong render phase — không cần `useEffect` để fetch data. Đây là bước tiến hóa tiếp theo của React từ "effects for everything" sang "declarative data fetching".

## 7. Production Scenario

Bối cảnh: một badge hiển thị trạng thái thanh toán real-time của hợp đồng thuê, nhận dữ liệu qua subscription (WebSocket).

```jsx
class PaymentStatusBadge extends React.Component {
  state = { status: null };

  componentDidMount() {
    this.subscription = subscribeToPaymentStatus(
      this.props.contractId,
      (status) => this.setState({ status }),
    );
  }

  // Bug thật: nếu thiếu hàm này, component KHÔNG bao giờ resubscribe
  // khi contractId đổi — vẫn lắng nghe trạng thái của hợp đồng CŨ.
  componentDidUpdate(prevProps) {
    if (prevProps.contractId !== this.props.contractId) {
      this.subscription.unsubscribe();
      this.subscription = subscribeToPaymentStatus(
        this.props.contractId,
        (status) => this.setState({ status }),
      );
    }
  }

  componentWillUnmount() {
    this.subscription.unsubscribe();
  }

  render() {
    return <span>{this.state.status}</span>;
  }
}
```

Đây chính là pain point ở mục 2: concern "subscription theo `contractId`" bị tách thành 3 method. Bug phổ biến nhất trong thực tế: `componentDidUpdate` bị bỏ quên hoàn toàn (chỉ viết `componentDidMount` + `componentWillUnmount`) — component hoạt động đúng lúc mount, nhưng khi cha đổi `contractId` qua prop (ví dụ người dùng chọn hợp đồng khác trong danh sách mà không unmount badge), badge tiếp tục hiển thị trạng thái của hợp đồng cũ.

## 8. Debugging Scenario

Bug report khác: "Badge trạng thái đôi lúc gọi callback với dữ liệu của hợp đồng trước đó, dù đã chuyển sang hợp đồng mới." — lần này ở bản hooks:

```jsx
function PaymentStatusBadge({ contractId, onStatusChange }) {
  const [status, setStatus] = useState(null);

  useEffect(() => {
    const subscription = subscribeToPaymentStatus(contractId, (newStatus) => {
      setStatus(newStatus);
      onStatusChange(newStatus); // dùng onStatusChange từ closure
    });
    return () => subscription.unsubscribe();
  }, [contractId]); // thiếu onStatusChange trong dependency array

  return <span>{status}</span>;
}
```

Quy trình debug đúng:

1. Bật eslint `react-hooks/exhaustive-deps` (nếu chưa bật) — rule sẽ cảnh báo ngay: "React Hook useEffect has a missing dependency: 'onStatusChange'".
2. Hiểu vì sao cảnh báo đúng: effect được tạo ra ở lần render đầu, đóng (closure) lấy đúng phiên bản `onStatusChange` tại thời điểm đó. Vì `contractId` không đổi, effect không chạy lại — closure cũ tiếp tục được dùng, gọi `onStatusChange` đời cũ dù component cha đã truyền xuống một `onStatusChange` mới (ví dụ cha re-tạo callback mỗi lần render vì không bọc `useCallback`).
3. Fix tối thiểu: thêm `onStatusChange` vào dependency array — nhưng nếu cha tạo callback mới mỗi lần render (không `useCallback`), effect sẽ resubscribe liên tục không cần thiết. Fix đúng hơn: cha bọc `onStatusChange` trong `useCallback` với dependency ổn định, hoặc dùng pattern lưu callback mới nhất vào một `ref` để đọc trong effect mà không cần đưa vào dependency array.

## 9. Refactoring Scenario

**Trước** — một `componentDidMount` gánh 3 concern không liên quan, lặp lại logic ở `componentDidUpdate`:

```jsx
class ContractDetailPage extends React.Component {
  componentDidMount() {
    document.title = `Hợp đồng #${this.props.contractId}`;
    this.subscription = subscribeToPaymentStatus(
      this.props.contractId,
      this.handleStatusChange,
    );
    logAnalyticsEvent('view_contract_detail', {
      contractId: this.props.contractId,
    });
  }

  componentDidUpdate(prevProps) {
    if (prevProps.contractId !== this.props.contractId) {
      document.title = `Hợp đồng #${this.props.contractId}`;
      this.subscription.unsubscribe();
      this.subscription = subscribeToPaymentStatus(
        this.props.contractId,
        this.handleStatusChange,
      );
      logAnalyticsEvent('view_contract_detail', {
        contractId: this.props.contractId,
      });
    }
  }

  componentWillUnmount() {
    this.subscription.unsubscribe();
  }

  handleStatusChange = (status) => this.setState({ status });

  render() {
    return <span>{this.state?.status}</span>;
  }
}
```

**Sau** — mỗi concern tách thành một custom hook riêng, tái sử dụng được ở component khác:

```jsx
function useDocumentTitle(title) {
  useEffect(() => {
    document.title = title;
  }, [title]);
}

function usePaymentStatusSubscription(contractId, onChange) {
  useEffect(() => {
    const subscription = subscribeToPaymentStatus(contractId, onChange);
    return () => subscription.unsubscribe();
  }, [contractId, onChange]);
}

function useAnalyticsView(eventName, payload) {
  useEffect(() => {
    logAnalyticsEvent(eventName, payload);
  }, [eventName, payload]);
}

function ContractDetailPage({ contractId }) {
  const [status, setStatus] = useState(null);

  useDocumentTitle(`Hợp đồng #${contractId}`);
  usePaymentStatusSubscription(contractId, setStatus);
  useAnalyticsView('view_contract_detail', { contractId });

  return <span>{status}</span>;
}
```

Lợi ích thực: mỗi concern đọc riêng, sửa riêng, test riêng; `useDocumentTitle`/`useAnalyticsView` dùng lại được ở bất kỳ trang nào khác — điều không thể làm gọn gàng với class lifecycle vì logic gắn chặt vào method cố định của class.

## 10. Architecture Notes

- Custom hook là **đơn vị tái sử dụng logic** chính trong codebase hooks-first — thay cho HOC/render props. Khi review PR, một custom hook tên rõ ràng (`usePaymentStatusSubscription`) tự nó đã là documentation, còn một HOC bọc nhiều lớp thường che giấu luồng dữ liệu.
- Rules of Hooks không phải convention tuỳ chọn — nó phản ánh trực tiếp cấu trúc `memoizedState` linked list trên Fiber (M1.2). Một linter vi phạm rule này nên được coi là lỗi nghiêm trọng, không phải warning có thể bỏ qua.
- **Class component chưa biến mất hoàn toàn**: Error Boundary (`static getDerivedStateFromError`, `componentDidCatch`) hiện vẫn chỉ tồn tại ở class component — chưa có hook chính thức để catch lỗi render xảy ra ở component con. Đây là lý do bạn vẫn thấy một class `ErrorBoundary` nhỏ trong codebase hooks-first hiện đại, thường được viết một lần và tái sử dụng ở nhiều nơi.
- Hướng tới M1.4 (Suspense): Suspense đẩy mô hình tư duy đi xa hơn nữa khỏi "lifecycle như chuỗi sự kiện" — một component có thể "suspend" giữa lúc render, hoàn toàn không có khái niệm tương đương trong lifecycle method cổ điển.
- React Compiler tự động memoize effects và dependencies — giảm nhu cầu thêm useMemo/useCallback thủ công. Tuy nhiên: Compiler không thay thế việc hiểu dependency array. `eslint-plugin-react-hooks/exhaustive-deps` vẫn quan trọng để đảm bảo correctness. Một số edge cases (dynamic dependencies, conditional effects) vẫn cần manual handling. Hiểu rõ cơ chế effect lifecycle vẫn là bắt buộc để debug khi compiler không hoạt động như mong đợi.

<summary>Đáp án</summary>

State của các hook không được lưu theo tên biến, mà được lưu trên Fiber node dưới dạng một linked list (`memoizedState`) — mỗi lần gọi hook sẽ lấy đúng node tiếp theo trong list đó theo thứ tự gọi, không theo tên.

Nếu một hook bị gọi có điều kiện, thứ tự gọi giữa hai lần render có thể lệch nhau, khiến hook ở vị trí thứ N của lần render này vô tình đọc nhầm state đã lưu cho hook ở vị trí thứ N của lần render trước — gây bug sai lệch state mà không phải lúc nào cũng có exception rõ ràng để phát hiện ngay.

</details>

---

**[Senior/Staff]** Có tính năng nào của class component mà hooks hiện vẫn chưa thay thế được không?

<details>
<summary>Đáp án</summary>

`Error Boundary`. Catch lỗi xảy ra trong quá trình render của component con — thông qua static `getDerivedStateFromError` và `componentDidCatch` — hiện vẫn chỉ tồn tại ở class component, chưa có hook chính thức tương đương.

Vì lý do này, codebase hooks-first hiện đại thường vẫn giữ một class `Error Boundary` nhỏ, viết một lần, dùng lại nhiều nơi, thay vì cố tránh class hoàn toàn.

</details>

**[Architect]** Thiết kế một custom hook bọc logic subscription (ví dụ WebSocket) sao cho tránh được cả hai lỗi: duplicate subscription khi dependency không đổi, và stale closure trong callback khi dependency liên quan đổi nhưng không nằm trong dependency array.

<details>
<summary>Đáp án</summary>

**Nguyên tắc thiết kế:** dependency array của effect phải liệt kê đầy đủ mọi giá trị reactive được dùng bên trong effect (tuân theo `eslint-plugin-react-hooks/exhaustive-deps`) để tránh **stale closure** — không cố tình bỏ sót dependency để "tránh **resubscribe** nhiều lần".

Nếu một callback truyền vào hook thay đổi reference mỗi lần render của component cha (gây **resubscribe** không cần thiết dù logic không thực sự đổi), giải pháp đúng là ổn định reference đó ở phía gọi (`useCallback` với dependency hợp lý), hoặc lưu callback mới nhất vào một ref bên trong custom hook và đọc từ `ref` đó trong effect — cách này giữ effect chỉ phụ thuộc vào dependency thực sự cần **resubscribe** (như `contractId`), trong khi vẫn luôn gọi callback mới nhất, không bị stale.

</details>

## 12. Exercises

**Bài 1.** Class component dưới đây hiển thị đồng hồ đếm ngược "thời gian còn lại để thanh toán". Có một bug rò bộ nhớ (memory leak). Tìm và sửa, viết lại bằng hooks.

```jsx
class PaymentCountdown extends React.Component {
  state = { secondsLeft: this.props.initialSeconds };

  componentDidMount() {
    setInterval(() => {
      this.setState((prev) => ({ secondsLeft: prev.secondsLeft - 1 }));
    }, 1000);
  }

  render() {
    return <span>{this.state.secondsLeft}s</span>;
  }
}
```

<details>
<summary>Đáp án</summary>

Bug: `componentDidMount` tạo `setInterval` nhưng không lưu lại id để clear, và hoàn toàn không có `componentWillUnmount` — interval tiếp tục chạy và gọi `setState` trên một component đã unmount, gây **memory leak** và warning trong console. Bản hooks tương đương, đúng và an toàn:

```jsx
function PaymentCountdown({ initialSeconds }) {
  const [secondsLeft, setSecondsLeft] = useState(initialSeconds);

  useEffect(() => {
    const intervalId = setInterval(() => {
      setSecondsLeft((prev) => prev - 1);
    }, 1000);
    return () => clearInterval(intervalId);
  }, []);

  return <span>{secondsLeft}s</span>;
}
```

</details>

**Bài 2.** Đoạn hook dưới đây có bug stale closure. Tìm dependency bị thiếu và giải thích hậu quả.

```jsx
function ContractSearchBox({ contracts }) {
  const [keyword, setKeyword] = useState('');

  useEffect(() => {
    const filtered = contracts.filter((c) => c.tenantName.includes(keyword));
    renderResultsToAnalytics(filtered.length);
  }, [contracts]);

  return (
    <input
      value={keyword}
      onChange={(e) => setKeyword(e.target.value)}
    />
  );
}
```

<details>
<summary>Đáp án</summary>

Dependency array thiếu keyword. Effect chỉ chạy lại khi `contracts` đổi, không chạy lại khi người dùng gõ vào input — bên trong effect, keyword luôn là giá trị tại thời điểm effect được tạo (closure cũ), nên `renderResultsToAnalytics` luôn nhận số liệu filter theo keyword cũ (thường là chuỗi rỗng ban đầu), không phản ánh đúng những gì người dùng đang gõ. Sửa: thêm keyword vào dependency array — [contracts, keyword].

</details>

**Bài 3.** Cho `Parent` chứa `Child`, cả hai đều có `useLayoutEffect` và `useEffect`. Sắp xếp đúng thứ tự thực thi khi component mount lần đầu.

```jsx
function Child() {
  useLayoutEffect(() => console.log('Child layout effect'), []);
  useEffect(() => console.log('Child effect'), []);
  return <span>child</span>;
}

function Parent() {
  useLayoutEffect(() => console.log('Parent layout effect'), []);
  useEffect(() => console.log('Parent effect'), []);
  return <Child />;
}
```

<details>
<summary>Đáp án</summary>

Thứ tự đúng: function body của **Parent** chạy trước (render top-down, vì **Parent** gọi tạo ra element Child), sau đó function body của **Child** chạy.

Nhưng effect lại chạy theo thứ tự ngược — bottom-up: con phải "sẵn sàng" trước khi cha coi như đã mount xong.

Vì vậy thứ tự log chính xác là: Child layout effect, Parent layout effect, (browser paint), Child effect, Parent effect.

</details>

## 13. Summary

- Class lifecycle tổ chức code theo trục thời gian (mount/update/unmount); hooks tổ chức theo trục đồng bộ hoá (effect phụ thuộc giá trị nào, chạy lại khi giá trị đó đổi).
- `useEffect` không cùng timing với `componentDidMount`/`componentDidUpdate` — timing tương đương thật là `useLayoutEffect` (đồng bộ, trước paint); `useEffect` chạy bất đồng bộ, sau paint.
- Cleanup function của hook xử lý cả hai trường hợp (trước lần effect kế tiếp, và khi unmount) trong một chỗ duy nhất — class phải lặp code ở `componentDidUpdate` và `componentWillUnmount`.
- Rules of Hooks không phải convention tuỳ chọn — nó phản ánh đúng cấu trúc linked list `memoizedState` trên Fiber node đã học ở M1.2; gọi hook có điều kiện làm lệch thứ tự, gây đọc nhầm state.
- Bug phổ biến nhất khi dùng hooks không phải cú pháp, mà là dependency array thiếu/thừa — gây stale closure hoặc effect chạy lại không cần thiết.
- Class component chưa hoàn toàn lỗi thời: Error Boundary vẫn là tính năng class-only, chưa có hook chính thức thay thế.
