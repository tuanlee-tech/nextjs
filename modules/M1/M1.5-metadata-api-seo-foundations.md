# M1.5: Metadata API & SEO Foundations

> 🎯 **Lesson Goal:** Implement `generateMetadata` cho static và dynamic routes, phân biệt `metadata` object vs `generateMetadata` function, và giải thích tại sao Metadata API thay thế `next/head`.

> ⚠️ **Phạm vi bài này:** Bạn sẽ học cách khai báo field `openGraph.images` bằng 1 URL string. Bạn sẽ **không** học cách tự render ảnh đó (Vercel OG, `ImageResponse`, canvas) — đó là kỹ thuật riêng nằm ở M8.3. Nếu interview hỏi sâu phần generate ảnh, trả lời đúng API, không tự bịa implementation.

---

## 1. Evolution Context

React render thẳng vào `<body>`. Việc đẩy `<meta>` tag lên `<head>` để SEO bot đọc được luôn là một dạng "hack" — vì `<head>` nằm ngoài cây component mà React quản lý trực tiếp.

Cách cũ: `next/head` (Pages Router) hoặc `react-helmet` (SPA era trước đó). Cả hai chạy **trong** React lifecycle — nghĩa là chúng tồn tại như 1 component bình thường, render ra, rồi mới được portal lên `<head>`. Hệ quả: dễ gây hydration mismatch (server render 1 set tag, client render set khác do timing), block streaming (HTML body phải chờ), và không có cơ chế de-duplicate khi nhiều component cùng set 1 meta key (2 component cùng set `title` → 2 thẻ `<title>` cùng tồn tại, browser chọn 1 cách không đoán trước được).

**Timeline đã xác nhận:** `react-helmet` (SPA era) → `next/head` (Pages Router) → Metadata API (App Router v13.2+, định nghĩa metadata bên ngoài React render tree, streaming-compatible).

---

## 2. Mental Model

> 💡 **[MODEL]** Metadata được Next.js extract **trước** khi React render UI, cho phép stream `<head>` ngay lập tức mà không cần chờ toàn bộ component tree render xong.
>
> **[ANALOGY]** Tưởng tượng 1 cuốn sách: `next/head` giống như viết tiêu đề chương **sau khi** đã in xong toàn bộ nội dung — phải mở sách ra mới biết chương đó tên gì. Metadata API giống như in tiêu đề ngay trên bìa sách — search engine và social bot đọc được ngay, không cần "mở sách" (không cần render xong React tree).

## 3. Core Concepts

### 3.1 `metadata` Object — Static Metadata

**[FACT]** `metadata` object là static export trực tiếp từ `page.tsx` hoặc `layout.tsx`, được Next.js eval lúc build — zero runtime cost. Source: `nextjs.org/docs/app/api-reference/functions/generate-metadata#metadata-object`.

```tsx [src/app/layout.tsx]
import type { Metadata } from 'next';

// title.template cho phép mọi page con chỉ cần khai báo tên trang —
// site name tự động append, tránh hardcode "| RentalSaaS" lặp lại trong từng file.
export const metadata: Metadata = {
  title: {
    template: '%s | RentalSaaS',
    default: 'RentalSaaS — Quản lý cho thuê bất động sản',
  },
  description: 'Nền tảng quản lý hợp đồng, thanh toán và vận hành cho thuê bất động sản.',
};

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="vi">
      <body>{children}</body>
    </html>
  );
}
```

```tsx [src/app/about/page.tsx]
import type { Metadata } from 'next';

export const metadata: Metadata = {
  title: 'Giới thiệu',
  description: 'RentalSaaS là nền tảng quản lý cho thuê bất động sản cho landlord Việt Nam.',
};

export default function AboutPage() {
  return <main>{/* nội dung trang giới thiệu */}</main>;
}
```

### 3.2 `generateMetadata` — Dynamic Metadata

**[FACT]** `generateMetadata` là async function, nhận `params` (Promise trong Next.js 16) và `searchParams`, dùng khi metadata phụ thuộc dữ liệu động (DB, fetch). Source: `nextjs.org/docs/app/api-reference/functions/generate-metadata`.

```tsx [src/app/properties/[id]/page.tsx]
import type { Metadata } from 'next';
import { notFound } from 'next/navigation';
import { logger } from '@/lib/logger';

interface PropertyPageProps {
  params: Promise<{ id: string }>;
}

interface PropertySummary {
  id: string;
  title: string;
  shortDescription: string;
}

async function fetchPropertySummary(id: string): Promise<PropertySummary | null> {
  try {
    const res = await fetch(`${process.env.INTERNAL_API_URL}/properties/${id}/summary`, {
      // Cache 5 phút — generateMetadata và page component cùng gọi function này,
      // Next.js dedupe request fetch giống nhau trong cùng 1 render pass.
      next: { revalidate: 300 },
    });

    if (res.status === 404) return null;
    if (!res.ok) {
      throw new Error(`Failed to fetch property summary: ${res.status}`);
    }

    return res.json();
  } catch (err) {
    // Không swallow lỗi — log đủ context để trace được property nào gây lỗi
    logger.error({ err, propertyId: id }, 'metadata.property_summary.fetch_failed');
    throw err;
  }
}

export async function generateMetadata({ params }: PropertyPageProps): Promise<Metadata> {
  // [FACT] params là Promise trong Next.js 16 — phải await trước khi destructure.
  // Source: Next.js 16 Upgrade Guide
  const { id } = await params;
  const property = await fetchPropertySummary(id);

  if (!property) {
    // Property không tồn tại -> metadata generic, để page tự gọi notFound()
    return { title: 'Không tìm thấy bất động sản' };
  }

  return {
    title: property.title,
    description: property.shortDescription,
  };
}

export default async function PropertyPage({ params }: PropertyPageProps) {
  const { id } = await params;
  const property = await fetchPropertySummary(id);

  if (!property) {
    notFound();
  }

  return <main>{/* render property detail */}</main>;
}
```

### 3.3 `metadataBase` & Social Cards

**[FACT]** `metadataBase` set base URL tuyệt đối, dùng để resolve relative URL trong `openGraph.images` hoặc `twitter.images`. Source: Metadata API docs.

```tsx [src/app/layout.tsx]
import type { Metadata } from 'next';

export const metadata: Metadata = {
  metadataBase: new URL(process.env.NEXT_PUBLIC_SITE_URL ?? 'https://rentalsaas.example.com'),
  title: {
    template: '%s | RentalSaaS',
    default: 'RentalSaaS',
  },
  openGraph: {
    siteName: 'RentalSaaS',
    type: 'website',
    // URL relative — Next.js tự resolve thành absolute nhờ metadataBase phía trên.
    // Ảnh /og-default.png là static asset có sẵn trong /public — generate ảnh
    // động cho từng property KHÔNG thuộc phạm vi bài này, xem M8.3.
    images: ['/og-default.png'],
  },
  twitter: {
    card: 'summary_large_image',
  },
};
```

### 3.4 SEO-Specific Metadata — `robots`, `alternates`

`robots` và `alternates` là 2 field SEO foundational — sâu hơn (structured data, JSON-LD) thuộc M8.3, bài này chỉ dừng ở mức field cơ bản.

```tsx [src/app/dashboard/contracts/[id]/page.tsx]
import type { Metadata } from 'next';

// robots: noindex cho trang dashboard nội bộ — không nên xuất hiện trên Google
// dù URL có lỡ bị crawl tới (vd: bị share nhầm trong group chat công khai).
export const metadata: Metadata = {
  robots: {
    index: false,
    follow: false,
  },
};
```

```tsx [src/app/properties/[id]/page.tsx]
import type { Metadata } from 'next';

interface PropertyPageProps {
  params: Promise<{ id: string }>;
}

// alternates.canonical cho property public — tránh duplicate content nếu
// cùng 1 property có thể truy cập qua nhiều query string (vd: ?ref=email-campaign).
export async function generateMetadata({ params }: PropertyPageProps): Promise<Metadata> {
  const { id } = await params;
  return {
    alternates: {
      canonical: `/properties/${id}`,
    },
  };
}
```

### 3.5 Metadata API vs `next/head`

**[FACT]** `next/head` không support trong App Router — import sẽ bị ignore hoặc throw. Source: Next.js App Router Migration Docs.

**[FACT]** Metadata API tự động de-duplicate tags. Nếu layout set `title: "RentalSaaS"` và page set `title: "Bất động sản"`, kết quả cuối cùng là 1 thẻ `<title>` duy nhất — merge theo `title.template` nếu layout có khai báo, hoặc override hoàn toàn nếu không. Source: Metadata API docs.

```tsx [src/app/properties/page.tsx]
import type { Metadata } from 'next';

// Không có thẻ <Head> nào ở đây. Next.js extract object này, merge với
// title.template ở root layout, rồi stream <head> trước khi React render UI.
export const metadata: Metadata = {
  title: 'Bất động sản',
};

export default function PropertiesPage() {
  return <main>{/* danh sách property */}</main>;
}
```

## 4. Anti-Patterns in the Wild

### AP1: Dùng `next/head` trong App Router

❌ **Code Smell:**
```tsx [src/app/properties/page.tsx]
// Code junior thường viết — copy nguyên pattern từ Pages Router
import Head from 'next/head';

export default function PropertiesPageBad() {
  return (
    <>
      <Head>
        <title>Bất động sản</title>
      </Head>
      <main>...</main>
    </>
  );
}
```

✅ **Staff Refactor:**
```tsx [src/app/properties/page.tsx]
import type { Metadata } from 'next';

// Trade-off: không có. next/head hoàn toàn broken trong App Router.
export const metadata: Metadata = {
  title: 'Bất động sản',
};

export default function PropertiesPage() {
  return <main>...</main>;
}
```

🧠 **Why:** Không có trade-off để bảo vệ code cũ — `next/head` không hoạt động trong App Router, không phải vấn đề style.

🔥 **Incident Risk:** Meta tags không xuất hiện trong HTML production, SEO index rỗng — Google Search Console không thấy gì để index.

---

### AP2: Hardcode metadata trong layout cho dynamic route

❌ **Code Smell:**
```tsx [src/app/properties/[id]/layout.tsx]
import type { Metadata } from 'next';

// Code junior thường viết — static title cho route phục vụ hàng nghìn property khác nhau
export const metadata: Metadata = {
  title: 'Bất động sản',
};

export default function PropertyLayout({ children }: { children: React.ReactNode }) {
  return <div>{children}</div>;
}
```

✅ **Staff Refactor:** Xóa static `metadata` ở layout. Dùng `generateMetadata` trong `page.tsx` như ở mục 3.2 — fetch tên property thật theo từng `id`.

```tsx [src/app/properties/[id]/layout.tsx]
// Layout không export metadata nữa — để page.tsx tự quyết định qua generateMetadata.
// Trade-off: thêm 1 async call lúc render, đổi lại mỗi property có title/description đúng tên thật.
export default function PropertyLayout({ children }: { children: React.ReactNode }) {
  return <div>{children}</div>;
}
```

🧠 **Why:** Tăng 1 async call lúc render đổi lại SEO accuracy tuyệt đối cho từng property.

🔥 **Incident Risk:** Google Search Console báo "Duplicate title tags" hàng nghìn lần (1 lần / property), đánh tụt ranking toàn bộ section bất động sản.

## 5. Trade-offs

| Approach | Pros | Cons | Khi nào dùng | Khi nào KHÔNG dùng | Overhead / Cost |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Static `metadata` object** | Zero runtime cost, eval tại build time | Không nhận dynamic data | Static pages (About, Contact) | Dynamic routes (`[id]`) | Không có |
| **`generateMetadata`** | Dynamic SEO chính xác 100% | Block TTFB nếu query chậm | Dynamic routes, DB-driven pages | Static pages (overkill) | Network/delay overhead |
| **`next/head` (Legacy)** | N/A | Broken ở App Router, block streaming | Không bao giờ | Always | Crash/error |
| **Title Templates** | DRY title structure cho toàn app | Cần hiểu rõ quy luật merge layout/page | Root/layout level | Leaf page (định nghĩa thẳng) | Rất thấp |

## 6. Legacy Recognition

- **Legacy:** `next/head` (Pages Router) — import `Head` component, render `<Head><title>...</title></Head>` trong JSX.
- **Legacy:** `react-helmet` — SPA-era head management, dùng trong create-react-app.
- **Legacy:** Props drilling meta data từ `getServerSideProps` xuống `<Head>` component.
- **Legacy:** Dynamic `<Head>` render trong loop `.map()` — set meta tag động ngay trong component tree.

---

## 7. Modern Approach

- **Modern:** Metadata API (App Router) — `export const metadata = {...}` hoặc `export async function generateMetadata(...)`.
- **Migration — xóa `<Head>` component:** Find/replace `next/head` theo từng route, chuyển thẻ `<title>`/`<meta>` thành object `metadata` export.
- **Migration — props drilling từ `getServerSideProps`:** Move fetch logic thẳng vào `generateMetadata` của page tương ứng, bỏ bước truyền `meta` qua props.
- **Migration — dynamic `<Head>` trong loop:** Metadata API không hỗ trợ render nhiều meta tag động trong loop. Nếu cần list nhiều biến thể URL (vd: multi-language), chuyển vào `alternates.languages` thay vì cố nhồi vào `<Head>` cũ.

**Strategy cho codebase lớn:** Strangler Fig — `pages/` và `app/` chạy song song. Migrate route theo nhóm tính năng (vd: toàn bộ flow "Property listing" trước, "Tenant portal" để sau), không bao giờ rewrite metadata toàn site trong 1 PR.

**Rollback plan:** Giữ `pages/` route tương ứng còn sống trong giai đoạn migrate. Nếu Metadata API gây TTFB tăng đột biến ở 1 route cụ thể (vd: do `generateMetadata` fetch nặng), xóa route đó khỏi `app/` để Next.js fallback phục vụ lại bằng `pages/` cũ — không cần rollback toàn bộ migration.

## 8. Production Scenario

### Scenario 1: Social Cards Không Hiện Ảnh

- **Symptom:** Share link property lên Zalo/Facebook không hiển thị ảnh preview.
- **Root cause:** Thiếu `metadataBase` trong root layout. Relative URL `/og-default.png` bị resolve sai thành domain của platform đang crawl (vd: `facebook.com/og-default.png`).
- **Detect:** Dùng Facebook Sharing Debugger hoặc xem trực tiếp request của bot trong server log.
- **Fix:** Thêm `metadataBase: new URL('https://rentalsaas.example.com')` ở root layout.

### Scenario 2: View Source Không Thấy Dynamic Title

- **Symptom:** `curl` HTML của `/properties/123` trả về title generic ("Bất động sản"), không phải tên property thật.
- **Root cause:** Dùng static `metadata` ở layout thay vì `generateMetadata` ở page cho route `[id]`.
- **Detect:** View source trực tiếp qua `curl` (không dùng DevTools — DevTools có thể hiển thị title đã update sau hydration, đánh lừa bạn rằng mọi thứ ổn).
- **Fix:** Chuyển sang `generateMetadata` async như mục 3.2.

### Scenario 3: `generateMetadata` Làm Chậm TTFB

- **Symptom:** TTFB của trang property tăng từ 200ms lên 2s sau khi thêm dynamic metadata.
- **Root cause:** Fetch DB lấy tên property bên trong `generateMetadata` không có cache, mỗi request lại query lại từ đầu.
- **Detect:** Trace log (OpenTelemetry span hoặc Next.js build trace) thấy latency tập trung ở phase metadata resolution, trước khi page component render.
- **Fix:** Cache fetch (`next: { revalidate: ... }` như mục 3.2) hoặc chuyển route đó sang ISR.

## 9. Performance & Overhead

Raw material hiện tại chưa có số benchmark cụ thể (TTFB delta, build time cho hàng nghìn dynamic route) — không tự bịa số. Cách bạn tự đo trên codebase thật:

- **Đo TTFB thực:** `curl -o /dev/null -s -w "TTFB: %{time_starttransfer}s\n" https://yourapp.com/properties/123` — chạy nhiều lần, so sánh route dùng static `metadata` vs route dùng `generateMetadata` có fetch DB.
- **Cách ly chi phí của riêng `generateMetadata`:** Wrap fetch trong function đó bằng 1 OpenTelemetry span riêng (`tracer.startSpan('generateMetadata.fetchProperty')`) — nhìn trace để biết bao nhiêu % TTFB nằm ở đây so với phần render page chính.
- **Build time cho route động số lượng lớn:** Nếu dùng `generateStaticParams` cho hàng nghìn property, chạy `next build --debug` để xem breakdown thời gian build từng route — nếu metadata fetch lặp lại data đã fetch ở page component, đảm bảo cùng dùng 1 cache key (`fetch` với cùng URL + options) để Next.js dedupe, tránh build chậm gấp đôi vì gọi API 2 lần cho mỗi property.
- **Kiểm tra social bot crawl thực tế:** Giả lập user-agent bot — `curl -A "facebookexternalhit/1.1" https://yourapp.com/properties/123` — đọc thẳng HTML trả về để confirm `og:image`, `og:title` đã resolve đúng absolute URL, không phải đoán qua browser.

## 10. Architecture Notes

- **Metadata Resolution Order:** Layout metadata → page metadata. Page override layout, nhưng nếu layout dùng `title.template` (vd: `%s | RentalSaaS`), title của page được inject vào `%s` thay vì bị ghi đè hoàn toàn — hiểu nhầm chỗ này dễ gây title sai mà code "nhìn đúng".
- **Pitfall — export `metadata` trong Client Component:** Metadata chỉ hoạt động trong Server Component. Thêm `'use client'` rồi export `metadata` sẽ bị Next.js throw build error hoặc ignore hoàn toàn. (Source: Next.js Docs)
- **Pitfall — `generateMetadata` trả về non-serializable data:** Metadata phải là plain JSON object. Trả `Date` instance trực tiếp sẽ lỗi — phải convert `.toISOString()` trước khi return.
- **Pitfall — quên `await params` trong Next.js 16:** `params` là Promise. Truy cập `params.id` trực tiếp (không await) throw `TypeError`. (Source: Next.js 16 Upgrade Guide)
- **Dynamic OG Image Strategy (chỉ nhắc qua — chi tiết thuộc M8.3):** Bạn có thể truyền URL động vào `openGraph.images` bên trong `generateMetadata` (vd: trỏ tới `/api/og?propertyId=123`), nhưng việc endpoint đó render ảnh ra sao (canvas, `ImageResponse`, satori) không thuộc phạm vi M1.5.
- **SEO & Core Web Vitals:** `generateMetadata` chậm = TTFB chậm, vì server phải resolve xong metadata mới bắt đầu stream `<head>`. Query nặng bắt buộc phải cache hoặc dùng ISR — nếu không sẽ kéo cả LCP xuống theo.
- **Title Template Pattern:** `title: { template: '%s | RentalSaaS', default: 'RentalSaaS' }` ở root layout. Mọi page con chỉ cần `title: 'PropertyName'`, site name tự động append — DRY, tránh hardcode lặp lại ở 39+ lesson page khác nhau trong app thật.

## 11. Interview Questions

### Q1: Metadata API khác `next/head` thế nào? (Junior)
<details>
<summary>Đáp án</summary>

Metadata API khai báo tĩnh hoặc async bên ngoài React render tree, không block streaming HTML body và tự động de-duplicate tag trùng key. `next/head` chạy trong component tree, dễ gây hydration mismatch và không support trong App Router.

</details>

### Q2: Làm sao generate OG image động cho từng property? (Mid)
<details>
<summary>Đáp án</summary>

Truyền URL động vào mảng `openGraph.images` bên trong `generateMetadata` (vd: `url: /api/og?propertyId=${id}`) trỏ tới endpoint xử lý render ảnh. Implementation chi tiết của endpoint đó (canvas, `ImageResponse`) không thuộc API này — đó là phần riêng ở M8.3.

</details>

### Q3: `metadataBase` làm gì và tại sao quan trọng? (Mid)
<details>
<summary>Đáp án</summary>

Nó là base URL tuyệt đối để resolve các relative path trong social card metadata. Thiếu nó, relative path sẽ resolve sai thành domain của platform đang crawl link (vd: thành domain của Facebook) — social card vỡ ảnh.

</details>

### Q4: Layout và Page đều export `metadata`, cái nào thắng? (Junior)
<details>
<summary>Đáp án</summary>

Page override layout. Nhưng nếu layout dùng `title.template` (vd: `%s | RentalSaaS`), title của page được inject vào `%s` thay vì bị ghi đè hoàn toàn — kết quả là merge, không phải replace.

</details>

### Q5: `generateMetadata` chậm ảnh hưởng thế nào đến Core Web Vitals? (Senior)
<details>
<summary>Đáp án</summary>

Nó làm tăng TTFB vì server phải resolve xong toàn bộ metadata trước khi stream `<head>`. TTFB tăng kéo theo LCP tăng — nếu query bên trong nặng, bắt buộc phải cache data hoặc chuyển route sang ISR để không phá Core Web Vitals.

</details>

- [ ] OG image URL trong HTML trả về là absolute URL (nhờ `metadataBase` ở layout).
- [ ] Xử lý đúng `await params` (Next.js 16) — không truy cập `params.id` trực tiếp.
- [ ] Property không tồn tại → metadata fallback hợp lý, không throw lỗi 500.

### 🔴 Exercise 3: Migration từ `next/head` sang Metadata API

Cho sẵn 1 Pages Router codebase dùng `next/head` để set title/description cho trang property detail, fetch data qua `getServerSideProps`.

## 14. Terminology Evolution

### Metadata API
```
┌─────────────┬─────────────────┬─────────────────────────────┐
│ Era         │ Status          │ Dùng khi nào                │
├─────────────┼─────────────────┼─────────────────────────────┤
│ v13.2+      │ Active          │ App Router — declarative    │
│             │                 │ metadata                    │
│ v13.0-13.1  │ N/A             │ Không tồn tại               │
│ Pages Router│ Legacy          │ next/head                   │
└─────────────┴─────────────────┴─────────────────────────────┘
```
**Lịch sử:** v13.2+, App Router only. Replacement cho `next/head`. Định nghĩa metadata bên ngoài React render tree, streaming-compatible.
**Codebase legacy:** `import Head from 'next/head'` — render `<Head>` ngay trong JSX tree.

### `generateMetadata`
```
┌─────────────┬─────────────────┬─────────────────────────────┐
│ Era         │ Status          │ Dùng khi nào                │
├─────────────┼─────────────────┼─────────────────────────────┤
│ v13.2+      │ Active          │ App Router — dynamic        │
│             │                 │ metadata từ data fetch      │
│ Pages Router│ Legacy          │ getServerSideProps + props  │
│             │                 │ truyền vào <Head>           │
└─────────────┴─────────────────┴─────────────────────────────┘
```
**Lịch sử:** Async function tạo metadata từ dynamic data. Nhận `params` (Promise trong Next.js 16) và `searchParams`.
**Codebase legacy:** Fetch data trong `getServerSideProps`, truyền `meta` props xuống component để render `<Head>`.

### `metadataBase`
```
┌─────────────┬─────────────────┬─────────────────────────────┐
│ Era         │ Status          │ Dùng khi nào                │
├─────────────┼─────────────────┼─────────────────────────────┤
│ v13.2+      │ Active          │ App Router — base URL cho   │
│             │                 │ social card                 │
│ Pages Router│ Legacy          │ Không có tương đương trực   │
│             │                 │ tiếp (hardcode absolute URL)│
└─────────────┴─────────────────┴─────────────────────────────┘
```
**Lịch sử:** Required field để giải quyết relative URL cho social card. Thiếu nó, relative path resolve sai domain. *(Bổ sung từ raw material — chưa nằm trong Bible Terminology gốc.)*
**Codebase legacy:** Hardcode absolute URL trong `<Head>`, hoặc tự nối `process.env.NEXT_PUBLIC_URL` thủ công trước mỗi path.

### Title Template
```
┌─────────────┬─────────────────┬─────────────────────────────┐
│ Era         │ Status          │ Dùng khi nào                │
├─────────────┼─────────────────┼─────────────────────────────┤
│ v13.2+      │ Active          │ App Router — DRY title      │
│             │                 │ structure                   │
```
