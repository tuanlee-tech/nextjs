**2. Evolution Context (Bài này xuất hiện để giải quyết vấn đề gì?)**

**Trước đó**

- jQuery era: dev tự quản lý DOM → state rải rác, dễ out-of-sync, khó scale
- React 0–15: Stack Reconciler — đệ quy đồng bộ, không thể interrupt → UI freeze khi render lớn

**Vấn đề**

- Dev hiểu "render" = "DOM update" → debug sai tầng, không hiểu tại sao UI không update, tại sao component "render quá nhiều"
- Không hiểu Hydration Error, không hiểu tại sao Server Component không có `useState`
- Không hiểu tại sao `key` quan trọng, tại sao `React.memo` đôi khi vô dụng

↓

**Giải phú**

- React 16 (2017): Fiber Architecture — rewrite engine, async-capable, work units có thể pause/resume
- React 18 (2022): Concurrent Rendering stable → Suspense, Streaming → Next.js 13 App Router khả thi
- Next.js 16 (2025+): Cache Components, PPR stable

**3. Mental Model (Ý tưởng cốt lõi)**

> **React Rendering = Chụp ảnh mô tả UI + So sánh với ảnh cũ trong cùng một lượt. DOM Update = Chỉ in lại phần khác (đã được đánh dấu từ trước).**

```
┌─────────────────────────────────────────────┐
│  RENDER PHASE (có thể interrupt)              │
│  ─────────────────────────────                │
│  1. Gọi component functions → React Elements  │
│  2. ĐỒNG THỜI so sánh với tree cũ             │
│     → Đánh dấu: Placement | Update | Deletion │
│  ⚠️ KHÔNG có DOM update! KHÔNG có side effects!│
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│  COMMIT PHASE (không thể interrupt)           │
│  ──────────────────────────────────           │
│  1. Đọc flags → apply DOM mutations           │
│  2. useLayoutEffect (sync, trước paint)       │
│  3. Atomic swap: work-in-progress → current   │
│  4. Browser paint                             │
│  5. useEffect (async, sau paint)              │
│  ✅ DOM update CHỈ xảy ra ở đây               │
└─────────────────────────────────────────────┘
```

**4. Core Concepts (20% kiến thức quan trọng nhất)**

**Concept 1 — React Element: "Ảnh chụp" UI**

- JSX compile thành `React.createElement()` → trả về **plain JavaScript object** (KHÔNG phải DOM node)
- Rất nhẹ, cheap to create/discard → React có thể throw away và tạo lại mỗi render
- DOM operations mới là thứ cần minimize

**Concept 2 — Ba Giai đoạn Kết xuất**

| Giai đoạn        | Đặc điểm                                                        | Có thể interrupt? |
| ---------------- | --------------------------------------------------------------- | ----------------- |
| **Trigger**      | `setState`, props change, context change                        | —                 |
| **Render Phase** | Gọi function → React Elements + Reconciliation (đánh dấu flags) | ✅ Có (React 18+) |
| **Commit Phase** | Apply DOM mutations + effects                                   | ❌ Không          |

- Render Phase phải **PURE** — không side effects, không fetch, không mutate
- Strict Mode dev: mount → unmount → remount để phát hiện side effects không cleanup

**Concept 3 — Reconciliation Algorithm (O(n) thay vì O(n³))**

- **Heuristic 1:** Khác type → Destroy & Rebuild hoàn toàn (kể cả state bên trong!)
- **Heuristic 2:** `key` prop = stable identity. Không có `key` → dùng index → state lệch khi list thay đổi

**Concept 4 — Fiber Architecture**

- Mỗi component = **Fiber node** (linked list: child → sibling → return)
- **Double buffering:** 2 trees song song (current + work-in-progress) → atomic swap khi render xong
- Enables: pause/resume, prioritize urgent updates, Concurrent Mode, Suspense, Streaming

**Concept 5 — Điều gì trigger re-render (và KHÔNG)**

✅ Trigger: `setState`, `dispatch` (useReducer), context value change
❌ KHÔNG trigger: mutate object trực tiếp, thay đổi `ref.current`, biến local

👉 Cascade re-render: state update ở ancestor → toàn bộ subtree re-render (dù descendant không dùng state đó) → cần `React.memo` để cắt cascade

**5. Legacy Recognition (Nhìn code biết gì?)**

Dấu hiệu nhận biết

✅

- Class Components (`componentDidMount`, `componentWillUnmount`)
- `UNSAFE_componentWillMount/ReceiveProps/Update` (deprecated từ React 16.3)
- `ReactDOM.render` (React 17) thay vì `createRoot` (React 18)
- `getServerSideProps`/`getStaticProps` trong Pages Router

❌

- Inline objects/functions trong props khi dùng `React.memo` (phá memo)
- `key={index}` trong list (state lệch)
- `new Date()`, `Math.random()` trong render (hydration mismatch)

**7. Production Scenario (Thực tế)**

**"Tại sao component re-render quá nhiều lần?"**

- **Context:** `ProductPage` với `FilterPanel` + `ProductList`
- **Problem:** Mỗi lần filter thay đổi → `ProductList` re-render dù không dùng filter state
- **Root Cause:** Cascade re-render + inline objects/functions phá `React.memo`
- **Solution:** `useCallback` + `useMemo` + `React.memo` + tách state input ra component riêng
- **Lesson:** `React.memo` chỉ hữu ích khi props thực sự stable. Thêm memo mà không fix reference instability = thêm overhead không lợi ích

👉 Rule:

> Nếu gặp **component re-render "quá nhiều"** hãy nghĩ ngay đến **cascade re-render + inline props phá memo, không phải "React chậm"**.

---

**8. Debugging Scenario (Debug nhanh)**

**Hydration Error — từ góc độ Reconciliation**

- Hydration = Reconciliation đặc biệt: so sánh React Element tree với **existing HTML DOM** (do server render)
- Nếu không match → React không biết tin DOM hay code

Checklist:

- □ `new Date()`, `Math.random()` trong render → dùng `useState('')` + `useEffect` để cập nhật sau hydrate
- □ `typeof window`, `navigator`, `localStorage` trong render → dùng `useState` + `useEffect` hoặc `dynamic({ ssr: false })`
- □ Conditional rendering dựa trên browser APIs
- □ Thư viện bên ngoài dùng `window`
- □ Browser extensions inject HTML

**Fix:**

- `suppressHydrationWarning` — cho content dynamic nhưng acceptable (timestamps)
- `dynamic({ ssr: false })` — component hoàn toàn chỉ chạy client (maps, editors)

**10. Architecture Notes (Decision Framework)**

**Khi nên dùng**

- Server Component: chỉ render data, không cần state/effects/browser API
- Client Component: cần `useState`, `useEffect`, event handler, browser API
- `React.memo`: component nhận props stable, nằm trên đường cascade re-render
- `useCallback`/`useMemo`: props là functions/objects, truyền xuống component đã memo

**Khi không nên dùng**

- `React.memo` khi props là inline objects/functions (memo vô hiệu)
- `useMemo` cho calculation rẻ (overhead > lợi ích)
- `suppressHydrationWarning` khi có thể fix root cause

Trade-off

| Pros                                               | Cons                                        |
| -------------------------------------------------- | ------------------------------------------- |
| Fiber: interruptible, concurrent, Suspense         | Complexity cao hơn Stack                    |
| Server Component: zero JS bundle, direct DB access | Không có state, không tương tác tức thì     |
| `React.memo`: cắt cascade re-render                | Shallow compare, dễ bị phá bởi inline props |
| `useEffect` cho client-only content                | Thêm render cycle, phức tạp hóa flow        |

**13. Summary (One Page)**

**Những gì phải nhớ**

- Rendering ≠ DOM update. 3 giai đoạn tách biệt: Trigger → Render Phase (pure, no DOM) → Commit Phase (DOM + effects)
- React Elements = plain objects, cheap to create. DOM operations mới cần minimize
- Reconciliation: O(n) nhờ 2 heuristics — khác type = destroy & rebuild, `key` = stable identity
- Fiber: linked list work units, double buffering, enable Concurrent Mode → Suspense → Streaming → App Router
- Server Component: reconcile (merge RSC Payload) nhưng không re-render (không state)
- Client Component: full lifecycle, re-render → reconcile → DOM update
- Hydration = reconcile Element tree với existing HTML. Lần render đầu client phải byte-identical với server

## 1. Lesson Goal

Sau bài học này, bạn sẽ:

- **Hiểu đúng** "rendering" trong React nghĩa là gì — và không phải là gì
- **Nắm vững** ba giai đoạn kết xuất: Trigger → Render Phase → Commit Phase
- **Giải thích** tại sao Reconciliation Algorithm dùng heuristics thay vì exact tree diffing
- **Hiểu** Fiber Architecture và tại sao nó là nền tảng của Concurrent Mode
- **Kết nối** tất cả với Next.js: tại sao đây là điều kiện tiên quyết để hiểu Hydration, Server Components và Streaming

**Tại sao bài này quan trọng hơn bạn nghĩ:**

Hầu hết React Developer sử dụng `useState`, `useEffect` hàng ngày mà không hiểu React thực sự làm gì bên dưới. Điều này ổn với ứng dụng nhỏ. Nhưng khi vào Next.js App Router với Server Components và Streaming — mọi thứ trở nên kỳ lạ nếu bạn không có mental model đúng. Hydration Error, performance vấn đề không giải thích được, Server Component "tại sao lại không có state" — tất cả có câu trả lời trong bài này.

## 2. Evolution Context

Trong M0, bạn đã hiểu tại sao Next.js tồn tại và nó tiến hóa thế nào. Một chi tiết quan trọng từ M0.2: **React 18 là bước ngoặt để App Router khả thi**. App Router cần Concurrent Rendering, Streaming, và Server Components. Tất cả xây trên nền tảng của bài học này.

**Timeline tương quan trực tiếp với bài học hôm nay:**

```
2013  → React ra đời.
        Stack Reconciler — đệ quy đồng bộ, không thể interrupt.

2017  → React 16: Fiber Reconciler.
        Rewrite hoàn toàn engine bên dưới.
        Async-capable architecture, work units có thể pause/resume.

2020  → Concurrent Mode (experimental, React 17)
        Lần đầu tiên React có thể "ưu tiên" renders.

2022  → React 18: Concurrent Rendering stable (default với createRoot)
        useTransition, Suspense stable, createRoot thay ReactDOM.render.

2022  → Next.js 13: App Router (experimental).
        Được xây hoàn toàn trên React 18 Concurrent.

05/2023  → Next.js 13.4: App Router stable

10/2023  → Next.js 14: Server Actions stable, Turbopack stable, Partial Prerendering (experimental)

2024 → Next.js 15: async Request APIs (params/cookies/headers as Promise), fetch default no-store, refined PPR (experimental)

2025 → Next.js 16: Cache Components stable ('use cache'), PPR stable, Server Functions đã stable từ v14, Compiler improvements

06/2026 → Next.js 16+ (current, stable)
```

**Insight then chốt:** Nếu React không rewrite sang Fiber năm 2017, chúng ta sẽ không có App Router ngày hôm nay. Fiber không phải optimization — đó là cuộc cách mạng kiến trúc thầm lặng nhất trong lịch sử React.

## 3. Mental Model

### Pain Point: Sai lầm phổ biến nhất của React Developer

Hỏi bất kỳ React Developer nào: _"Khi bạn gọi `setState`, điều gì xảy ra?"_

Câu trả lời phổ biến nhất: _"Component re-render, UI được cập nhật."_

Câu trả lời này **đúng một nửa và sai một nửa theo cách nguy hiểm nhất**.

Vấn đề là từ "re-render" bị hiểu nhầm thành "DOM được vẽ lại". Hiểu nhầm này dẫn đến:

- Không hiểu tại sao đôi khi UI không update như mong đợi
- Không biết tại sao component render "quá nhiều lần"
- Không hiểu Hydration Error là gì
- Không hiểu tại sao Server Component không có `useState`

### Cách cũ: jQuery — Tự mình quản lý DOM

Trước React, để update UI, developer phải **tự tay** thao tác DOM:

```javascript
// jQuery era — Developer là người phụ trách đồng bộ state ↔ UI
function updateUserProfile(user) {
  $('#user-name').text(user.name);
  $('#user-avatar').attr('src', user.avatar);
  $('#user-bio').text(user.bio);

  if (user.isPremium) {
    $('#premium-badge').show();
    $('#upgrade-btn').hide();
  } else {
    $('#premium-badge').hide();
    $('#upgrade-btn').show();
  }

  // ... 20 dòng nữa để sync mọi thứ
  // Và nếu quên update một chỗ → bug
}
```

### Tại sao cách cũ thất bại

- **Không có nguồn sự thật duy nhất**: State nằm rải rác trong DOM. DOM là state.
- **Dễ out-of-sync**: Quên update một element → UI sai, không có error.
- **Khó scale**: 10 developers làm cùng một page → không ai biết ai update gì.
- **Performance không predictable**: Ai cũng tự update DOM theo cách riêng.

### Cách mới: React — Mô tả, không ra lệnh

React đảo ngược mô hình: thay vì **ra lệnh** "update DOM như thế này", bạn **mô tả** "UI trông như thế này với state này":

```jsx
// React — Developer mô tả UI, React lo phần DOM
function UserProfile({ user }) {
  return (
    <div>
      <h1>{user.name}</h1>
      <img src={user.avatar} />
      <p>{user.bio}</p>
      {user.isPremium ? <PremiumBadge /> : <UpgradeButton />}
    </div>
  );
}
// React tự tính toán DOM nào cần thay đổi
// Developer không cần biết "cách" — chỉ cần biết "kết quả"
```

**Mental Model đúng:**

```
React Rendering = Chụp ảnh mô tả UI + So sánh với ảnh cũ trong cùng một lượt
DOM Update      = Chỉ in lại phần khác (đã được đánh dấu từ trước)
```

```
┌─────────────────────────────────────────────────────────────┐
│  RENDER PHASE (có thể interrupt)                            │
│  ─────────────────────────────                              │
│  1. Gọi component functions → tạo React Element tree mới    │
│  2. ĐỒNG THỜI so sánh với tree cũ (Reconciliation)          │

│  3. Output: Work-in-progress tree với "hướng dẫn" DOM       │
│                                                             │
│  ⚠️ KHÔNG có DOM update ở đây!                              │
│  ⚠️ KHÔNG có side effects ở đây!                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
```

## 4. Core Concepts

### 4.1 React Element — "Ảnh chụp" UI

Khi bạn viết JSX, Babel biên dịch nó thành `React.createElement()` calls:

```jsx
// Bạn viết:
const element = (
  <div className='card'>
    <h1>Xin chào</h1>
    <p>Nội dung bài học</p>
  </div>
);
```

```javascript
// Trình duyệt thấy sau khi Babel compile:
const element = React.createElement(
  'div',
  { className: 'card' },
  React.createElement('h1', null, 'Xin chào'),
  React.createElement('p', null, 'Nội dung bài học'),
);
```

```javascript
// Kết quả trả về — plain JavaScript object, KHÔNG phải DOM node:
{
  $$typeof: Symbol(react.element), // Bảo vệ chống XSS injection
  type: 'div',
  key: null,
  ref: null,
  props: {
    className: 'card',
    children: [
      {
        type: 'h1',
        props: { children: 'Xin chào' },
        key: null,
        ref: null
      },
      {
        type: 'p',
        props: { children: 'Nội dung bài học' },
        key: null,
        ref: null
      }
    ]
  }
}
```

**Insight quan trọng:** React Elements rất nhẹ — chỉ là plain object. Tạo hàng nghìn React Elements mỗi frame là bình thường. Tạo hàng nghìn **DOM nodes** mỗi frame là thảm họa performance.

Đây là lý do React có thể "throw away" và tạo lại React Element tree mỗi render mà không lo performance — vì những objects này cực kỳ cheap. DOM operations mới là thứ React cần minimize.

### 4.2 Ba Giai đoạn Kết xuất

React rendering có **3 giai đoạn** tách biệt hoàn toàn:

```
┌─────────────┐      ┌──────────────────────┐      ┌────────────────────┐
│   TRIGGER   │ ──▶  │    RENDER PHASE      │ ──▶  │   COMMIT PHASE     │
│             │      │  (Gọi function,      │      │  (Cập nhật DOM     │
│ setState    │      │   trả về Elements)   │      │   thực sự)         │
│ props thay  │      │   KHÔNG có DOM!      │      │                    │
└─────────────┘      └──────────────────────┘      └────────────────────┘
```

#### Giai đoạn 1: Trigger — Điều gì khởi động quá trình?

Có đúng 2 nguyên nhân:

**Nguyên nhân 1 — Initial render:**

```javascript
// App lần đầu khởi động
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<App />);
// → React render toàn bộ component tree từ đầu
```

**Nguyên nhân 2 — State update:**

```javascript
function Counter() {
  const [count, setCount] = useState(0);

  // setCount trigger re-render
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

**Những thứ KHÔNG trigger re-render (sai lầm thường gặp):**

```javascript
function Component() {
  // ❌ Thay đổi biến local: KHÔNG re-render
  let localVar = 0;
  localVar = 1; // Vô nghĩa với UI

  // ❌ Mutate object trực tiếp: KHÔNG re-render (đây là bug!)
  const [user, setUser] = useState({ name: 'Tuân' });
  user.name = 'Changed'; // React không biết gì về điều này
  // Phải dùng: setUser({ ...user, name: 'Changed' })

  // ❌ Thay đổi ref: Không re-render (by design — dùng cho side effects)
  const ref = useRef(0);
  ref.current = 1; // React không quan tâm
}
```

#### Giai đoạn 2: Render Phase — Gọi function để lấy mô tả

Trong **Render Phase**, React vừa gọi component functions để tạo Element tree mới, vừa đồng thời thực hiện **Reconciliation** (thông qua `beginWork()` và `completeWork()`) — so sánh với current tree qua Fiber nodes và đánh dấu những gì cần thay đổi `(Placement | Update | Deletion flags)`.

```javascript
function UserCard({ userId }) {
  const [user, setUser] = useState(null);

  // ← React gọi function này mỗi lần render
  // Nhận về: React Element tree (objects mô tả UI)
  // KHÔNG phải DOM, KHÔNG phải update gì cả

  return (
    <div className='card'>
      <Avatar user={user} />
      <UserInfo user={user} />
    </div>
  );
}
```

**Đặc điểm của Render Phase — bắt buộc phải nhớ:**

**1. Pure — không được có side effects:**

```javascript
// ❌ SAI: fetch trong render phase
function BadProductPage({ productId }) {
  fetch(`/api/product/${productId}`); // Chạy mỗi render → thảm họa!
  return <div>...</div>;
}

// ✅ ĐÚNG: fetch trong effect (sau khi render hoàn tất)
function GoodProductPage({ productId }) {
  const [product, setProduct] = useState(null);

  useEffect(() => {
    fetch(`/api/product/${productId}`)
      .then((res) => res.json())
      .then(setProduct);
  }, [productId]);

  return <div>{product?.name ?? 'Loading...'}</div>;
}
```

**2. Có thể bị interrupt (React 18 Concurrent Mode):**

Đây là thứ Fiber Architecture mang lại. React có thể dừng giữa chừng Render Phase nếu có việc ưu tiên hơn (user đang gõ bàn phím, animation đang chạy). Đây là lý do Render Phase phải pure — nếu có side effects, việc interrupt/restart sẽ chạy side effect đó nhiều lần.

**3. Strict Mode**

**Development + Strict Mode:**
React cố ý `mount → unmount → remount` components để phát hiện **side effects** **không được cleanup** đúng cách. Điều này có nghĩa component function của bạn sẽ chạy 2 lần, và useEffect cleanup + setup cũng chạy 2 lần.

> Nếu `console.log` trong render bị in 2 lần → đây là lý do, không phải bug

**Production**: chỉ gọi 1 lần

---

#### Giai đoạn 3: Commit Phase — DOM update thực sự

Sau Render Phase, `Commit Phase` đọc các flags được đánh dấu ở Fiber nodes và apply DOM mutations đã được tính toán trước (chỉ update DOM những gì thực sự thay đổi)

```
Render Phase output (mới):      Commit Phase — React quyết định:
┌─────────────────────┐         ┌──────────────────────────────────────┐
│ <div>               │         │ <div> → Giữ nguyên DOM node          │
│   <span>42</span>   │  ──▶    │ <span> → Update text "0" → "42"      │
│   <button>+</button>│         │ <button> → Giữ nguyên                │
│ </div>              │         │                                      │
└─────────────────────┘         └──────────────────────────────────────┘
                                           ↓
                                  Sau khi DOM update:
                                  1. useLayoutEffect (sync, trước paint)
                                  2. Atomic swap: work-in-progress → current
                                  3. Browser vẽ ra màn hình
                                  4. useEffect (async, sau paint)
```

`Commit Phase` gồm **3 bước**:

- **(1) DOM mutations**
- **(2) useLayoutEffect sync**
- **(3) Atomic swap** — work-in-progress trở thành current. Sau đó browser paint, rồi useEffect chạy async.

**Trade-off quan trọng:**

`Commit Phase` là **synchronous** và **không thể interrupt**. Một khi React bắt đầu apply DOM changes, nó phải hoàn thành. Đây là lý do bạn không nên làm công việc nặng trong `useLayoutEffect`.

### 4.3 Reconciliation Algorithm — Tại sao O(n) thay vì O(n³)?

**Vấn đề lý thuyết:** So sánh 2 cây node tổng quát là O(n³). Với 1.000 nodes → 1 tỷ operations. Với 10.000 nodes → 1 nghìn tỷ. Không thể dùng cho UI.

**Giải pháp của React:**

React đánh đổi tính tối ưu tuyệt đối của thuật toán tree diff có độ phức tạp O(n³) để lấy hiệu năng.

Bằng việc áp dụng hai heuristic (quy tắc kinh nghiệm) dựa trên các đặc điểm phổ biến trong việc xây dựng và cập nhật của giao diện người dùng (UI).

Những giả định này cho phép React bỏ qua phần lớn các khả năng cần xem xét và thực hiện quá trình reconciliation chỉ với một lần duyệt cây, đạt độ phức tạp O(n).

#### Heuristic 1: Khác type → Destroy và Rebuild hoàn toàn

```jsx
// Trường hợp A: Cùng type → React UPDATE props, giữ DOM node
// Before: <div className="old">
// After:  <div className="new">
// → React chỉ update className attribute
// → DOM node được REUSE

// Trường hợp B: Khác type → Hủy (Tear down) toàn bộ subtree
// Before: <div>...</div>
// After:  <section>...</section>
// → React UNMOUNT <div> và remove DOM nodes MỌI con cháu của nó
// → React MOUNT <section> và toàn bộ subtree mới từ đầu
// → KHÔNG cố tái sử dụng bất kỳ thứ gì bên trong
```

**Hệ quả thực tế — State reset:**

```jsx
function AuthGate({ isLoggedIn }) {
  return (
    <div>
      {
        isLoggedIn ? (
          <Dashboard /> // Type: Dashboard
        ) : (
          <LoginForm />
        ) // Type: LoginForm
      }
    </div>
  );
}
// Mỗi lần isLoggedIn đổi:
// → Unmount component type cũ (toàn bộ state, refs, effects bị destroy)
// → Mount component type mới (state bắt đầu từ initial values)
```

```jsx
// Ví dụ thực tế gây bug:
function Tabs({ activeTab }) {

      {activeTab === 'settings' && <SettingsForm />}
    </div>

// Hoặc lift state lên cha
```

#### Heuristic 2: `key` prop — Identity cho Reconciliation

Khi reconcile danh sách, React cần biết "item này ở render cũ tương ứng với item nào ở render mới". Không có `key`, React dùng **vị trí (index)** làm identity.

```jsx
// ❌ Không có key — React dùng index

// Render 1: [A(0), B(1), C(2)]
// Render 2 (xóa A): [B(0), C(1)]

// React nghĩ:
// "Index 0: vẫn tồn tại, thay đổi props từ A → B" → UPDATE
// "Index 1: vẫn tồn tại, thay đổi props từ B → C" → UPDATE
// "Index 2: đã bị xóa" → UNMOUNT

// Vấn đề: React REUSE DOM node của A để render B
// Nếu A có state bên trong (vd: text trong <input>) → state đó bị dính vào B!

function BadList({ items }) {
  return (
    <ul>
      {items.map((item) => (
        <ItemWithInput item={item} /> // ← Không có key
      ))}
    </ul>
  );
}
```

```jsx
// ✅ Có key — React dùng stable identity

// Render 1: [A(key="a"), B(key="b"), C(key="c")]
// Render 2 (xóa A): [B(key="b"), C(key="c")]

// React track bằng key:
// "key='a': đã biến mất" → UNMOUNT A (state bị destroy đúng chỗ)
// "key='b': vẫn còn, vị trí thay đổi" → MOVE + UPDATE props nếu cần
// "key='c': vẫn còn, vị trí thay đổi" → MOVE

function GoodList({ items }) {
  return (
    <ul>
      {items.map((item) => (
        <ItemWithInput
          key={item.id}
          item={item}
        /> // ← Stable ID từ data
      ))}
    </ul>
  );
}
```

**`key` không chỉ dùng cho list — dùng để force remount:**

```jsx
// Kỹ thuật quan trọng: thay đổi key → force unmount + remount
// Reset TOÀN BỘ state bên trong component

function ProfilePage({ userId }) {
  return (
    // Mỗi khi userId thay đổi:
    // → React thấy key khác → UNMOUNT UserProfile cũ
    // → MOUNT UserProfile mới với state hoàn toàn mới
    // → Không cần cleanup effect, không cần derived state logic
    <UserProfile
      key={userId}
      userId={userId}
    />
  );
}
```

**Trade-off:** Force remount bằng `key` là "nuclear option" — mọi thứ bên trong bị destroy. Dùng khi bạn thực sự muốn "bắt đầu lại" — không phải khi chỉ muốn sync state với props.

### 4.4 Fiber Architecture — Nền tảng của mọi thứ hiện đại

**Pain Point của Stack Reconciler (React 0–15):**

```
User click button → setCount() → React bắt đầu render

React chiếm JavaScript call stack
    ↓
Render component A
    ↓ (đang ở đây, không thể dừng)
Render component B
    ↓ (đang ở đây)
Render component C (có 500 children)
    ↓ (mắc kẹt ở đây 300ms)
Browser muốn vẽ animation → PHẢI ĐỢI
User đang gõ bàn phím → PHẢI ĐỢI
UI bị jank (lag, freeze)
```

**Fiber giải quyết bằng cách chia work thành đơn vị nhỏ có thể pause:**

Mỗi component trong tree được biểu diễn bằng một **Fiber node** — một object chứa đầy đủ thông tin để React có thể:

- Dừng giữa chừng
- Resume từ đúng chỗ đó
- Abort và restart nếu cần

```javascript
// Fiber node — đơn vị work nhỏ nhất (simplified)
type Fiber = {
  // Identity
  type: string | Function,      // 'div', 'span', hoặc MyComponent
  key: string | null,

  // Tree structure — Linked List, không phải đệ quy
  child: Fiber | null,          // Node con đầu tiên
  sibling: Fiber | null,        // Node anh em tiếp theo
  return: Fiber | null,         // Node cha (không phải parent trong DOM sense)

  // State
  pendingProps: Props,           // Props đang được apply trong render này
  memoizedProps: Props,          // Props của lần render trước
  memoizedState: any,            // State/hooks của lần render trước

  // DOM
  stateNode: DOMNode | null,    // DOM node thực tế tương ứng

  // Work flags
  flags: number,                 // Placement | Update | Deletion | ...

  // Double buffering
  alternate: Fiber | null,       // Work-in-progress twin
};
```

**Tại sao Linked List thay vì Recursion?**

Với Stack Reconciler: Render là đệ quy → không thể pause giữa đệ quy → all-or-nothing.

Với Fiber Linked List: React có thể traverse theo từng bước (child → sibling → return) → có thể yield control sau mỗi bước → browser có thể xử lý animation/input → React tiếp tục.

**Double Buffering — Cơ chế an toàn cho Concurrent Rendering:**

React duy trì **2 Fiber trees song song**:

```
CURRENT TREE                        WORK-IN-PROGRESS TREE
(Đang hiển thị trên màn hình)      (Đang được tính toán)

     App ────────────────────────────────── App*
    /   \                                 /   \
  Nav   Page                           Nav*  Page*
       /   \                                 /   \
   Header  Content                       Header  Content*
             |                                     |
          [Post A]                              [Post A]
          [Post B]                              [Post B]
                                                [Post C]  ← State update mới

*alternate: trỏ về current tree
```

```
Khi render hoàn tất:
→ Commit Phase apply DOM changes
→ useLayoutEffect sync (vẫn thấy alternate trỏ đến old tree)
→ Atomic swap: work-in-progress TRỞ THÀNH current
→ Current cũ TRỞ THÀNH work-in-progress cho lần render sau
→ User KHÔNG BAO GIỜ thấy half-rendered state
→ Browser vẽ ra màn hình
→ useEffect (async, sau paint — thấy current tree mới)
```

**Fiber Architecture làm nền tảng cho chuỗi mắt xích sau đây — quan trọng với Next.js:**

```
Fiber Architecture
        ↓ làm nền tảng cho
Concurrent Mode (React 18)
        ↓ tạo điều kiện cho
┌───────────────────────────────────────────────────────┐
│ • Interrupt long renders để xử lý user input          │
│ • Prioritize urgent updates (typing > background sync)│
│ • Time slicing (chia render thành nhiều frames)       │
│ • Suspense (pause render, chờ data/code)              │
└───────────────────────────────────────────────────────┘
        ↓ Suspense mở đường cho
Streaming HTML (Next.js App Router)
        ↓ giúp hiện thực hóa
Server Components được gửi dần (progressive)
        ↓ dẫn tới
App Router architecture như chúng ta biết ngày nay
```

### 4.5 Điều gì trigger Re-render (và điều gì KHÔNG)

Hiểu rõ điều này giúp debug performance issues:

```javascript
function ReRenderExplainer() {
  // ✅ Trigger re-render:
  const [count, setCount] = useState(0);
  const [user, setUser] = useReducer(userReducer, initialUser);
  const theme = useContext(ThemeContext); // ← Re-render khi context value đổi

  // ❌ KHÔNG trigger re-render:
  const ref = useRef(0); // Thay đổi ref.current không re-render
  let localVar = 0; // Biến local reset mỗi render

  return <Child />;
  // Quan trọng: <Child> RE-RENDER mỗi khi ReRenderExplainer re-render
  // Dù <Child> không nhận props nào!
  // → Đây là nguồn gốc của performance vấn đề phổ biến nhất
}
```

**Re-render lan truyền (Cascade Re-render) — Vấn đề thực tế:**

```jsx
function Parent() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <button onClick={() => setCount((c) => c + 1)}>Click</button>
      <ExpensiveChild />
      {/* ↑ Re-render MỖI LẦN Parent re-render!
          Dù ExpensiveChild không liên quan đến count */}
    </div>
  );
}

// Fix: Memo hoá component
const ExpensiveChild = React.memo(function ExpensiveChild() {
  // Chỉ re-render khi props thực sự thay đổi
  return <div>Expensive work...</div>;
});
```

**Trap phổ biến — Inline objects/functions phá vỡ memo:**

```jsx
function Parent() {
  const [count, setCount] = useState(0);

  return (
    <MemoizedChild
      // ❌ Mỗi render tạo object mới → shallow compare thấy "khác"
      // → React.memo vô dụng!
      config={{ theme: 'dark', size: 'lg' }}
      // ❌ Mỗi render tạo function mới → tương tự
      onClick={() => console.log('clicked')}
    />
  );
}

// Fix:
// Tạo ngoài component → stable reference
const CONFIG = { theme: 'dark', size: 'lg' };

function Parent() {
  const [count, setCount] = useState(0);

  // Sử dụng useCallback để tạo function stable reference
  const handleClick = useCallback(() => console.log('clicked'), []);

  return (
    <MemoizedChild
      config={CONFIG}
      onClick={handleClick}
    />
  );
}
```

## 6. Modern Approach

Next.js 13+ App Router với React 18 thay đổi hoàn toàn cách data fetching hoạt động — trực tiếp nhờ Fiber + Concurrent Mode:

```typescript
// app/dashboard/page.tsx
// Server Component — KHÔNG có reconciliation trên client
// Chạy một lần trên server, return RSC Payload
async function DashboardPage() {
  // Fetch data TRỰC TIẾP trong component — không cần useEffect!
  // Chạy trên server → không block browser
```

**Reconciliation xảy ra với toàn bộ tree, nhưng "re-render" chỉ xảy ra với Client Components:**

```
Next.js App Router — Request lifecycle:

SERVER                                    CLIENT
───────────────────────────────           ──────────────────────────────
1. Server Components render               4. Nhận HTML (cho FCP)
   → Chạy một lần trên server                + RSC Payload (JSON-like)
   → Không có state, effects, events
   → Output: RSC Payload (serialized)     5. Hydration:
                                               React reconcile RSC Payload
2. HTML streaming bắt đầu (FCP)              với HTML đã có sẵn
   (React DOM Server tạo HTML từ RSC         → Merge Server Components
    Payload)                                    vào React tree
                                          6. Client Components "thức dậy":
3. RSC Payload stream cùng lúc               → Attach event handlers
                                               → Có full lifecycle từ đây

                                             có thể re-render (useState,
                                             useEffect). Server Components
                                             KHÔNG re-render — nhưng vẫn
                                             được reconcile khi RSC Payload
                                             mới đến (Soft Navigation).
```

| _"Reconciliation chỉ xảy ra với Client Components"_ | **Sai**. React vẫn reconcile RSC Payload của Server Components khi merge vào tree. | Reconciliation xảy ra với **toàn bộ tree** — cả Server và Client Components.                     |

| -------------------------------- | ------------------------------------ | ---------------------------------- |

## 8. Debugging Scenario

### Hydration Error — Diagnose từ Reconciliation Model

```
Error: Hydration failed because the initial UI does not match
what was rendered on the server.

Warning: Expected server HTML to contain a matching <div> in <div>.
```

Đây là lỗi **Reconciliation thất bại trong quá trình Hydration**.

**Cơ chế Hydration:**

```
Server render: React tạo HTML string và gửi xuống browser
Client: Browser hiển thị HTML (user thấy nội dung ngay)
Client: React "hydrate" — gọi lại component functions,
        tạo React Element tree, so sánh (reconcile) với DOM đang có
Nếu match → OK: React attach event handlers, app interactive
Nếu NOT match → Hydration Error: React không biết nên tin DOM hay code
```

**Nguyên nhân phổ biến và cách fix:**

```typescript
// ❌ Bug 1: new Date() trong render
// app/components/timestamp.tsx
'use client';
function Timestamp() {
  // Server render: "10:00:00"
  // Client render lúc hydrate: "10:00:02" → MISMATCH!
  return <time>{new Date().toLocaleTimeString()}</time>;
}

// ✅ Fix: Client-only rendering cho dynamic time
'use client';
import { useState, useEffect } from 'react';

function Timestamp() {
  // Server render: '' (empty) → Client render initial: '' → MATCH!
  const [time, setTime] = useState('');

  useEffect(() => {
    // Chỉ chạy trên client, sau khi hydration hoàn tất
    // Lúc này React đã "control" DOM → safe để update
    setTime(new Date().toLocaleTimeString());
    const interval = setInterval(
      () => setTime(new Date().toLocaleTimeString()),
      1000
    );
    return () => clearInterval(interval);
  }, []);

  return <time suppressHydrationWarning>{time}</time>;
  // suppressHydrationWarning: báo React "tôi biết cái này dynamic,
  // nhưng hãy bỏ qua warning message trong console, không fix mismatch."

  // React vẫn reconcile và có thể chọn client hoặc server output.
  // Dùng khi bạn chấp nhận mismatch và muốn giữ DOM từ server.
}
```

```typescript
  const [relatedProducts, setRelatedProducts] = useState<Product[]>([]);

  const [error, setError] = useState<Error | null>(null);
```

## 10. Architecture Notes

### Server Components phá vỡ mental model Reconciliation truyền thống

Đây là insight quan trọng nhất của bài học này đối với Next.js:

```
TRUYỀN THỐNG (Pages Router + CSR):
─────────────────────────────────
Tất cả components đều là Client Components
Reconciliation xảy ra với MỌI component khi state thay đổi
Data fetching → useEffect → setState → re-render → reconcile DOM

APP ROUTER (Server + Client Components):
─────────────────────────────────────────
Server Components:
  • Chạy trên server (Node.js, không phải browser)
  • Không có Reconciliation vì không có client lifecycle
  • Không có useState, useEffect, event handlers
  • Output: RSC Payload (serialized component description)
  • Được gửi xuống client như "data", không như "code"

Client Components:
  • Có full Reconciliation lifecycle
  • useState, useEffect, event handlers
  • Được hydrated → interactive
  • Re-renders → Reconciliation → DOM updates

Ranh giới: 'use client' directive
  • Từ directive này trở xuống → Client Component subtree
  • Reconciliation bắt đầu từ đây
```

**Hệ quả kiến trúc:**

```
Tại sao Server Component không dùng được useState?
→ useState tồn tại để trigger re-render → trigger Reconciliation
→ Server Components không có state/effects
→ "State" không có ý nghĩa

Tại sao Server Component không dùng được onClick?
→ Event handlers chạy trên client khi user interact
→ Server Component không tồn tại trên client để respond
→ Không có cơ chế để attach handlers

Tại sao Server Component có thể async?
→ Chạy trên server → không block browser UI thread
→ Không có Concurrent Mode concerns (không interrupt-able render trên client)
→ Chỉ là Node.js async/await bình thường

Tại sao truyền Server Component qua props sang Client Component không được?
- Server Component chỉ chạy trên server, không tồn tại trong bundle client.
- React render Server Component trên server và serialize kết quả thành RSC
  Payload.
- Client Component trông chờ (expect) nhận React Elements hoặc primitive values
- RSC Payload không phải React Element theo nghĩa thông thường
- Client Component không nhận được Server Component function để tự render, mà
  chỉ nhận được reference đến UI đã được render thông qua RSC Payload.

→ Vì vậy, không thể truyền Server Component function/class để Client tự render

Server Component đã được render trên server thành RSC Payload (serialized UI description). Bạn có thể truyền kết quả render (<ServerComponent /> as children) vì đây là React Element đã serialized. Bạn không thể truyền component function (component={ServerComponent}) vì function không serializable qua RSC Payload.

Ví dụ:
❌ <Client component={ServerComponent} />
như một prop để Client gọi ra bằng :
const ComponentViaServer = props.component;
return <ComponenViatServer />; → Bảo Client tự render một Server Component

thay vào đó hãy truyền element (`<ServerComponent />`) bằng children

Lúc này <ServerComponent /> đã được render ở server và truyền xuống
Client một reference tới phần UI đã render trong RSC payload
Khi hydrate, React biết chèn đoạn UI đó vào đúng vị trí reference.

Ví dụ:
// ✅ Children pattern
<Client>
  <ServerComponent />
</Client>

hoặc các composition/slot props để React ghép UI vào đúng vị trí
// ✅ composition/slot pattern
<Client slot={<ServerComponent />} /


```

**Pattern: Server wraps Client (đúng cách):**

## 11. Interview Questions

### Junior Level

**Q1: "Rendering" trong React có nghĩa là gì? Nó khác DOM update như thế nào?**

<details>
<summary>Đáp án</summary>

"Rendering" trong React có nghĩa là React **gọi component function của bạn** để lấy mô tả UI dưới dạng React Elements (plain JavaScript objects). Đây là **Render Phase** — chưa có DOM update.

Quá trình đầy đủ gồm 3 bước tách biệt:

1. **Trigger**: `setState`, props change, hoặc context change khởi động quá trình.

2. **Render Phase**: React gọi component function, nhận về React Element tree mới. **Đồng thời trong phase này**, React thực hiện **Reconciliation** — so sánh Element tree mới với tree cũ (thông qua Fiber nodes) và đánh dấu những gì cần thay đổi (Placement, Update, Deletion flags). Phase này phải pure, không có side effects, và có thể bị interrupt (React 18+).

3. **Commit Phase**: React đọc các flags đã đánh dấu từ Render Phase và **apply DOM mutations** thực sự. Đây mới là lúc DOM được thay đổi. Commit Phase là đồng bộ, không thể interrupt. Sau đó React chạy effects (`useLayoutEffect` trước paint, `useEffect` sau paint).

**Tóm lại:**

- **Render** = Tính toán "UI sẽ trông như thế nào" + so sánh với UI cũ (Reconciliation)
- **DOM update** = Thực hiện thay đổi đã tính toán (Commit Phase)

React hiệu quả vì nó batches và minimizes DOM operations — thứ tốn kém nhất trong browser — chứ không phải vì "Virtual DOM" là magic.

</details>

**Q2: `key` prop trong React dùng để làm gì? Tại sao không nên dùng index làm key?**

<details>
<summary>Đáp án</summary>

`key` cho React biết **identity** của từng item trong list qua các renders. Reconciliation Algorithm dùng `key` để xác định "item này ở render cũ là item nào ở render mới".

**Tại sao không dùng index:**
Khi list thay đổi (xóa item đầu), index bị shift. Item B từ index 1 trở thành index 0. React nghĩ "index 0 vẫn tồn tại, chỉ thay đổi props" nên UPDATE props thay vì UNMOUNT A và reuse B đúng cách. Kết quả: nếu A có internal state (vd: input value đang gõ), state đó "dính" vào B — đây là bug nghiêm trọng.

**Nên dùng:** stable ID từ data (database ID, UUID).

**Bonus:** `key` không chỉ dùng cho list. Thay đổi `key` của bất kỳ component nào buộc React unmount component cũ và mount component hoàn toàn mới — cách đáng tin cậy nhất để reset toàn bộ state.

</details>

### Mid Level

**Q3: Fiber Architecture là gì và tại sao nó là bước ngoặt? Giải thích double buffering.**

<details>
<summary>Đáp án</summary>

**Fiber** là kiến trúc reconciliation của React từ v16 (2017), thay thế Stack Reconciler. Vấn đề của Stack Reconciler: rendering là đệ quy đồng bộ, không thể interrupt — render lớn chiếm JavaScript call stack → browser không xử lý được input/animation → UI freeze (jank).

**Fiber giải quyết:** Biểu diễn component tree dưới dạng linked list của **Fiber nodes** (work units nhỏ). React có thể traverse từng node, yield control về browser giữa các nodes, và resume sau. Điều này enables ( dẫn tới các khả năng ):

- Interrupt long renders để xử lý urgent updates (như user typing)
- Prioritize renders (user input > background sync)
- Concurrent Mode, Suspense, Streaming

**Double buffering:** React duy trì 2 Fiber trees song song:

- **current tree**: Tree đang hiển thị trên màn hình
- **work-in-progress tree**: Tree đang được tính toán cho render mới

Khi render hoàn tất, React thực hiện atomic swap — work-in-progress trở thành current, current cũ trở thành work-in-progress cho lần sau. Đảm bảo user không bao giờ thấy half-rendered state, và React có thể abort work-in-progress nếu có update ưu tiên cao hơn đến.

Fiber là nền tảng cho toàn bộ Next.js App Router architecture.

</details>

---

**Q4: Tại sao `React.memo` đôi khi không có tác dụng dù component trông "không đổi"?**

<details>
<summary>Đáp án</summary>

`React.memo` dùng **shallow comparison** (so sánh nông) để so sánh props. Nó so sánh reference (===), không so sánh deep value.

Vấn đề xảy ra khi props là objects hoặc functions được tạo **inline trong parent component**:

```jsx
// Mỗi lần Parent render, `config` là object MỚI (reference mới)
// Dù value giống hệt → shallow compare thấy "khác" → memo vô hiệu
<MemoizedChild
  config={{ theme: 'dark' }}
  onClick={() => doSomething()}
/>
```

**Fix:**

- Objects: dùng `useMemo` hoặc khai báo constant ngoài component
- Functions: dùng `useCallback`
- Primitive values (string, number, boolean): không có vấn đề này

Insight quan trọng: `React.memo` chỉ hữu ích khi props thực sự stable qua renders. Thêm memo mà không fix reference instability = thêm overhead mà không có lợi ích.

</details>

### Senior Level

**Q5: Giải thích Hydration Error trong Next.js từ góc độ Reconciliation. Khi nào dùng `suppressHydrationWarning` vs `dynamic({ ssr: false })`?**

<details>
<summary>Đáp án</summary>

**Hydration là Reconciliation đặc biệt:**
Thay vì so sánh 2 React Element trees, React reconcile React Element tree (từ component functions) với **existing HTML DOM** (do server render). Nếu không match → React không biết nên tin DOM hay code → Hydration Error.

**Nguyên nhân gốc rễ:** Một số thứ cho kết quả khác nhau khi chạy trên server (Node.js) vs client (browser):

- `new Date()` → thời điểm khác nhau
- `Math.random()` → giá trị khác nhau
- `typeof window` → `'undefined'` trên server, `'object'` trên client
- `localStorage`, `navigator` → không tồn tại trên server

**Khi nào dùng gì:**

`suppressHydrationWarning` — cho content dynamic nhưng không ảnh hưởng UX:

- Timestamps, counters real-time
- Content thực sự khác nhau giữa server/client nhưng acceptable
- Vẫn server-render HTML placeholder → tốt cho SEO/FCP

`dynamic({ ssr: false })` — component hoàn toàn chỉ chạy được trên client:

- Maps (Leaflet, Google Maps) — cần browser APIs
- Rich text editors (draft.js, etc.)
- Components dùng canvas, WebGL
- Trade-off: không có gì trên server → không có SEO cho content đó, blank khi FCP

Nguyên tắc: ưu tiên fix hydration mismatch thay vì suppress. Suppress và skip-ssr là last resort khi component thực sự không thể server-render.

</details>

---

**Q6: Server Components không có Reconciliation — vậy khi props của Server Component thay đổi, điều gì xảy ra?**

<details>
<summary>Đáp án</summary>

Khi props của một Server Component thay đổi (ví dụ: navigation sang URL mới qua `<Link>`), Next.js thực hiện **Soft Navigation**:

1. Next.js gửi request lên server với params mới (không reload trang)
2. Server Component chạy lại trên server như một function mới — đây là **server re-run**, không phải "re-render" theo nghĩa React client-side
3. Server trả về RSC Payload mới (serialized UI description)
4. Client nhận RSC Payload mới và **reconcile** nó vào existing React tree
5. Client Components bên trong **không bị remount** nếu vẫn tồn tại trong tree mới — state và refs được giữ nguyên

**Soft Navigation** là client-side routing: giống như SPA, browser không reload trang, chỉ fetch dữ liệu cần thiết cho route mới.

**Reconciliation ở đây có nghĩa là gì?**

React ở client đọc RSC Payload mới và so sánh với React Element tree hiện tại để quyết định DOM node nào cần insert, update, hay delete. Tuy nhiên, đối với Server Components, đây không phải "reconciliation lifecycle" (không có `useState`, `useEffect`, `shouldComponentUpdate`) — đơn giản là merge output đã render sẵn ở server vào tree.

**Phân biệt quan trọng:**

|                           | Server Component                            | Client Component                   |
| ------------------------- | ------------------------------------------- | ---------------------------------- |
| **Thay đổi khi navigate** | Re-run trên server                          | Reconcile ở client, giữ state      |
| **Có state/effects?**     | Không                                       | Có                                 |
| **Re-render?**            | Không — chỉ re-run ở server                 | Có — nếu props hoặc state thay đổi |
| **Remount?**              | Luôn "mount mới" vì là output mới từ server | Không, nếu key và type giữ nguyên  |

---

</details>

### Exercise 1 — Quan sát Render Cascade

Tạo ứng dụng nhỏ sau và dùng React DevTools Profiler để quan sát:

```jsx
function App() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <button onClick={() => setCount((c) => c + 1)}>Count: {count}</button>
      <Level2 />
    </div>
  );
}

function Level2() {
  console.log('Level2 render');
  return <Level3 />;
}

function Level3() {
  console.log('Level3 render');
  return <Level4 />;
}

function Level4() {
  console.log('Level4 render');
  return <div>Leaf component</div>;
}
```

**Câu hỏi:**

1. Click button nhiều lần. Level2, Level3, Level4 có re-render không? Tại sao?
2. Dùng `React.memo` để ngăn Level2 re-render. Điều gì xảy ra với Level3 và Level4?
3. Nếu bạn dùng `React.memo` cho Level4 nhưng không cho Level2 và Level3, kết quả như thế nào?

**Mục tiêu:** Hiểu cascade re-render và cách memo interrupt cascade.

<details>
<summary>Đáp án</summary>

**Câu hỏi 1: Click button nhiều lần. Level2, Level3, Level4 có re-render không? Tại sao?**

Có, cả ba đều re-render.

Khi `App` re-render do `setCount`, React tạo lại React Element tree. `<Level2 />` xuất hiện ở vị trí cũ, cùng type (`Level2`), không có props. Theo heuristic Reconciliation, React giữ DOM node và tiếp tục reconcile subtree. Vì `Level2` không được memo hoá, React gọi function `Level2` → return `<Level3 />` → gọi `Level3` → return `<Level4 />` → gọi `Level4`.

Đây là **cascade re-render**: state update ở ancestor kéo theo toàn bộ subtree re-render, dù descendant không dùng state đó. React không "biết" `Level2` không cần `count`; React chỉ thấy `App` return tree mới và reconcile toàn bộ.

---

**Câu hỏi 2: Dùng `React.memo` để ngăn Level2 re-render. Điều gì xảy ra với Level3 và Level4?**

`Level3` và `Level4` cũng **không re-render**.

```jsx
const Level2 = React.memo(function Level2() {
  console.log('Level2 render');
  return <Level3 />;
});
```

Khi `App` re-render, React tạo Element `<Level2 />` mới. `React.memo` thực hiện shallow compare props. `Level2` không nhận props nào → props object rỗng so với rỗng → không thay đổi. React **skip** gọi function `Level2` hoàn toàn, đồng thời **skip reconcile toàn bộ subtree** bên dưới.

Kết quả: `Level3` và `Level4` không bị gọi, không bị reconcile, không re-render.

**Insight:** `React.memo` không chỉ bảo vệ component được memo, mà còn **cắt cascade** xuống toàn bộ subtree. Đây là lý do memo nên đặt ở điểm cao nhất có thể.

---

**Câu hỏi 3: Nếu bạn dùng `React.memo` cho Level4 nhưng không cho Level2 và Level3, kết quả như thế nào?**

`Level2` và `Level3` vẫn re-render. `Level4` sẽ **không re-render** (trong trường hợp code gốc không truyền props), nhưng việc này ít giá trị thực tế.

Quá trình:

1. `App` re-render → gọi `Level2` (không memo) → return `<Level3 />`
2. React reconcile `Level3` → gọi `Level3` → return `<Level4 />`
3. React reconcile `Level4`. `Level4` có `React.memo`, không nhận props → shallow compare thấy không đổi → skip gọi `Level4`

Tuy nhiên, nếu `Level3` truyền props cho `Level4` dưới dạng inline object/function (ví dụ `<Level4 data={{}} />`), memo sẽ vô hiệu vì reference mới mỗi render.

**Bài học:** Memo ở leaf node không cắt được cascade từ ancestor. Để tối ưu hiệu quả, memo phải đặt ở vị trí mà props thực sự stable — thường là gần root của subtree không liên quan đến state thay đổi.

</details>

### Exercise 2 — Key Identity Experiment

```jsx
function ItemWithState({ label }) {
  const [inputValue, setInputValue] = useState('');
  const [clickCount, setClickCount] = useState(0);

  return (
    <div style={{ border: '1px solid #ccc', margin: '8px', padding: '8px' }}>
      <strong>{label}</strong>
      <input
        value={inputValue}
        onChange={(e) => setInputValue(e.target.value)}
        placeholder='Gõ gì đó...'
      />
      <button onClick={() => setClickCount((c) => c + 1)}>
        Clicks: {clickCount}
      </button>
    </div>
  );
}

function App() {
  const [items, setItems] = useState([
    { id: 'a', label: 'Item A' },
    { id: 'b', label: 'Item B' },
    { id: 'c', label: 'Item C' },
  ]);

  const removeFirst = () => setItems(items.slice(1));
  const shuffleItems = () => setItems([...items].reverse());

  return (
    <div>
      <button onClick={removeFirst}>Xóa item đầu tiên</button>
      <button onClick={shuffleItems}>Đảo thứ tự</button>

      <h3>Dùng index làm key (BUGGY):</h3>
      {items.map((item, index) => (
        <ItemWithState
          key={index}
          label={item.label}
        />
      ))}

      <h3>Dùng id làm key (CORRECT):</h3>
      {items.map((item) => (
        <ItemWithState
          key={item.id}
          label={item.label}
        />
      ))}
    </div>
  );
}
```

**Thực hành:**

1. Gõ text và click button ở tất cả items trong cả 2 sections
2. Click "Xóa item đầu tiên" → quan sát state có bị "lệch" không ở section index-key
3. Click "Đảo thứ tự" → quan sát state ở cả 2 sections

**Mục tiêu:** Trực quan hóa tại sao key quan trọng với mắt của mình.

<details>
<summary>Đáp án</summary>

**Thực hành 1 & 2: Gõ text, click button, rồi xóa item đầu tiên**

**Section dùng `index` làm key (BUGGY):**

Khi click "Xóa item đầu tiên", item A (index 0) biến mất. React reconcile list:

- Index 0 trước là A, nay là B → React nghĩ "cùng vị trí, update props từ A sang B" → **reuse DOM node của A** cho B
- Index 1 trước là B, nay là C → reuse DOM node của B cho C
- Index 2 trước là C, nay không còn → unmount C

Hệ quả: state bên trong (text input, click count) của A bị "dính" vào B, của B dính vào C. Nếu bạn đã gõ "hello" vào input của Item A, sau khi xóa, input của Item B sẽ hiển thị "hello". State bị **lệch identity**.

**Section dùng `id` làm key (CORRECT):**

Khi click "Xóa item đầu tiên":

- key="a" biến mất → React unmount A hoàn toàn (DOM node + state bên trong bị destroy đúng chỗ)
- key="b" vẫn còn, vị trí thay đổi → React move DOM node của B + giữ nguyên state
- key="c" vẫn còn, vị trí thay đổi → React move DOM node của C + giữ nguyên state

State (input, click count) đi theo đúng item. Không có hiện tượng lệch.

---

**Thực hành 3: Click "Đảo thứ tự"**

**Section `index` key:**

- Render mới: [C(0), B(1), A(2)]
- React nghĩ: index 0 cập nhật từ A→C, index 1 từ B→B (giữ), index 2 từ C→A
- Kết quả: reuse DOM node ở vị trí 0 cho C, giữ DOM node 1 cho B, reuse DOM node 2 cho A
- State bị hoán đổi theo vị trí, không theo item. Input của A sẽ xuất hiện ở vị trí cuối cùng.

**Section `id` key:**

- React track: key="c" chuyển lên đầu, key="b" giữa, key="a" xuống cuối
- React **move** DOM nodes theo đúng key, giữ nguyên state bên trong mỗi node
- Item A vẫn giữ input của A dù đã chuyển xuống cuối

---

**Tóm tắt mental model:**

`key` là **identity** của component instance qua các renders. Không có `key`, React dùng vị trí (index) làm identity — khi list thay đổi, identity bị lệch so với data. `key` ổn định (stable ID) đảm bảo React biết "đây là cùng một item" dù vị trí thay đổi, giúp state, focus, animation đi theo đúng item.

</details>

### Exercise 3 — Reproduce và Fix Hydration Error trong Next.js

Trong Next.js App Router project:

```typescript
// app/components/buggy-greeting.tsx
'use client';

export function BuggyGreeting() {
  // Tạo ra Hydration Error có chủ đích
  const hour = new Date().getHours();
  const greeting = hour < 12 ? 'Chào buổi sáng' : 'Chào buổi chiều';

  return (
    <div>
      <p>{greeting}, {Math.random().toFixed(4)}!</p>
    </div>
  );
}
```

**Nhiệm vụ:**

1. Import component này vào một page và reproduce Hydration Error
2. Đọc error message — Next.js hint element nào bị mismatch
3. Fix `Math.random()` — đảm bảo server và client render cùng giá trị
4. Fix time-based greeting mà không suppress warning
5. Giải thích bằng văn bản: tại sao cách fix của bạn hoạt động theo Reconciliation model

<details>
<summary>Đáp án</summary>

**Bước 1: Reproduce Hydration Error**

Import `BuggyGreeting` vào một Server Component (page):

```tsx
// app/page.tsx
import { BuggyGreeting } from './components/buggy-greeting';

export default function Page() {
  return <BuggyGreeting />;
}
```

Khi refresh trang:

- Server render lúc 10:00:00: `hour = 10`, `Math.random()` = `0.1234` → HTML chứa "Chào buổi sáng, 0.1234!"
- Client hydrate lúc 10:00:02: `hour = 14`, `Math.random()` = `0.5678` → React Element tree chứa "Chào buổi chiều, 0.5678!"
- React reconcile Element tree với DOM đang có → mismatch → **Hydration Error**

---

**Bước 2: Đọc error message**

Next.js 13+ sẽ chỉ ra element cụ thể bị mismatch, ví dụ:

```

Warning: Expected server HTML to contain a matching <p> in <div>
```

---

**Bước 3: Fix `Math.random()`**

`Math.random()` không thể render cùng giá trị ở server và client. Cách fix: chỉ hiển thị random sau khi hydration hoàn tất.

```tsx
'use client';

import { useState, useEffect } from 'react';

export function FixedGreeting() {
  const [randomId, setRandomId] = useState('');

  useEffect(() => {
    // Chỉ chạy trên client, sau khi hydration xong
    setRandomId(Math.random().toFixed(4));
  }, []);

  return (
    <div>
      <p>ID: {randomId || 'loading...'}</p>
    </div>
  );
}
```

Server render: `randomId = ''` → HTML chứa "ID: loading..."
Client initial render (hydrate): `randomId = ''` → match với server HTML
Client effect chạy: `setRandomId('0.5678')` → trigger re-render → update DOM → an toàn vì React đã control DOM

---

**Bước 4: Fix time-based greeting mà không suppress warning**

Dùng `useState` + `useEffect` để đảm bảo server và client render cùng giá trị ban đầu:

```tsx
'use client';

import { useState, useEffect } from 'react';

export function FixedGreeting() {
  const [greeting, setGreeting] = useState('Xin chào');

  useEffect(() => {
    const hour = new Date().getHours();
    setGreeting(hour < 12 ? 'Chào buổi sáng' : 'Chào buổi chiều');
  }, []);

  return (
    <div>
      <p>{greeting}</p>
    </div>
  );
}
```

Server render: "Xin chào"
Client hydrate: "Xin chào" → match
Effect chạy sau hydrate: cập nhật theo giờ thực → re-render an toàn

---

**Bước 5: Giải thích theo Reconciliation model**

Hydration là quá trình Reconciliation đặc biệt: React so sánh React Element tree (từ component functions chạy trên client) với **existing HTML DOM** (do server tạo ra). Nếu hai cây không match, React không biết nên tin DOM hay tin code.

Fix hoạt động vì:

1. **Render Phase đầu tiên trên client** tạo Element tree giống hệt server output → Reconciliation thành công → Hydration hoàn tất
2. **Commit Phase sau đó** (do `useEffect`) mới thay đổi DOM → lúc này React đã "own" DOM, việc update là bình thường, không còn so sánh với server HTML nữa

Nguyên tắc: đảm bảo **lần render đầu tiên trên client** (trong quá trình hydrate) cho kết quả **byte-identical** với server HTML. Mọi dynamic content chỉ được cập nhật sau khi hydration hoàn tất (trong `useEffect` hoặc user interaction).

</details>

### Exercise 4 (Bonus) — Profile và Tối ưu Re-renders

```typescript
// Cho component này, dùng React DevTools Profiler để:
// 1. Xác định component nào re-render khi thêm một todo
// 2. Xác định component nào re-render khi toggle một todo
// 3. Tối ưu để minimize re-renders

interface Todo {
  id: string;
  text: string;
  completed: boolean;
}

function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [input, setInput] = useState('');

  const addTodo = () => {
    if (!input.trim()) return;
    setTodos([...todos, { id: crypto.randomUUID(), text: input, completed: false }]);
    setInput('');
  };

  const toggleTodo = (id: string) => {
    setTodos(todos.map(t => t.id === id ? { ...t, completed: !t.completed } : t));
  };

  return (
    <div>
      <input value={input} onChange={e => setInput(e.target.value)} />
      <button onClick={addTodo}>Thêm</button>
      <ul>
        {todos.map(todo => (
          <TodoItem
            key={todo.id}
            todo={todo}
            onToggle={toggleTodo}
          />
        ))}
      </ul>
    </div>
  );
}

function TodoItem({ todo, onToggle }: { todo: Todo; onToggle: (id: string) => void }) {
  return (
    <li
      onClick={() => onToggle(todo.id)}
      style={{ textDecoration: todo.completed ? 'line-through' : 'none' }}
    >
      {todo.text}
    </li>
  );
}
```

<details>
<summary>Đáp án</summary>

**Phân tích re-render hiện tại**

Dùng React DevTools Profiler:

1. **Khi thêm một todo:** `TodoApp` re-render (do `setTodos` + `setInput`). `TodoItem` cho todo mới được mount. Các `TodoItem` cũ cũng re-render vì `TodoApp` return tree mới và `toggleTodo` là function mới mỗi render.
2. **Khi toggle một todo:** `TodoApp` re-render. Tất cả `TodoItem` re-render vì `todos` array mới và `toggleTodo` function mới.

Vấn đề chính:

- `toggleTodo` tạo mới mỗi render → truyền xuống mọi `TodoItem` → memo vô dụng
- `todos` array mới mỗi lần add/toggle → nếu dùng `React.memo` cho `TodoItem`, props `todo` object mới (spread) vẫn khiến re-render

---

**Tối ưu — 3 bước**

**Bước 1: Tách state input ra khỏi `TodoApp`**

```tsx
function TodoInput({ onAdd }: { onAdd: (text: string) => void }) {
  const [input, setInput] = useState('');

  const handleAdd = () => {
    if (!input.trim()) return;
    onAdd(input);
    setInput('');
  };

  return (
    <div>
      <input
        value={input}
        onChange={(e) => setInput(e.target.value)}
      />
      <button onClick={handleAdd}>Thêm</button>
    </div>
  );
}
```

Khi gõ input, chỉ `TodoInput` re-render. `TodoApp` không bị ảnh hưởng.

---

**Bước 2: Ổn định callbacks và memo hoá list item**

```tsx
function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);

  const addTodo = useCallback((text: string) => {
    setTodos((prev) => [
      ...prev,
      { id: crypto.randomUUID(), text, completed: false },
    ]);
  }, []);

  const toggleTodo = useCallback((id: string) => {
    setTodos((prev) =>
      prev.map((t) => (t.id === id ? { ...t, completed: !t.completed } : t)),
    );
  }, []);

  return (
    <div>
      <TodoInput onAdd={addTodo} />
      <ul>
        {todos.map((todo) => (
          <TodoItem
            key={todo.id}
            todo={todo}
            onToggle={toggleTodo}
          />
        ))}
      </ul>
    </div>
  );
}
```

- `addTodo` và `toggleTodo` dùng `useCallback` với empty deps → stable references
- `setTodos` dùng functional update (`prev => ...`) → không cần `todos` trong deps

---

**Bước 3: Memo hoá `TodoItem`**

```tsx
const TodoItem = React.memo(function TodoItem({
  todo,
  onToggle,
}: {
  todo: Todo;
  onToggle: (id: string) => void;
}) {
  console.log('TodoItem render', todo.id);
  return (
    <li
      onClick={() => onToggle(todo.id)}
      style={{ textDecoration: todo.completed ? 'line-through' : 'none' }}
    >
      {todo.text}
    </li>
  );
});
```

Với `React.memo`:

- Khi thêm todo: chỉ `TodoItem` mới được mount. Các `TodoItem` cũ thấy `todo` object cũ (reference giữ nguyên trong array map) và `onToggle` stable → **không re-render**.
- Khi toggle todo A: chỉ `TodoItem` A thấy `todo` object mới (vì spread tạo object mới cho item đó). Các `TodoItem` khác thấy `todo` cũ + `onToggle` cũ → **không re-render**.

---

**Kết quả sau tối ưu**

| Thao tác    | Trước tối ưu                            | Sau tối ưu                         |
| ----------- | --------------------------------------- | ---------------------------------- |
| Gõ input    | `TodoApp` + tất cả `TodoItem` re-render | Chỉ `TodoInput` re-render          |
| Thêm todo   | Tất cả `TodoItem` re-render             | Chỉ `TodoItem` mới mount           |
| Toggle todo | Tất cả `TodoItem` re-render             | Chỉ `TodoItem` bị toggle re-render |

**Lưu ý production:** Nếu list < 100 items, việc re-render toàn bộ có thể không phải bottleneck. Chỉ tối ưu sau khi profiling xác nhận đây là vấn đề thực sự.
