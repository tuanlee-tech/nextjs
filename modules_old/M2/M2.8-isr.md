# M2.8 — ISR: Incremental Static Regeneration

## 1. Lesson Goal

Sau bài này bạn phải:

- Giải thích được **tại sao** ISR ra đời — không phải "ISR là gì"
- Hiểu cơ chế **stale-while-revalidate** ở mức vận hành thực tế
- Sử dụng thành thạo `revalidate` (time-based) và `res.revalidate()` (on-demand)
- Biết khi nào **không nên dùng** ISR và tại sao
- Nhận ra ISR code trong legacy codebase và đánh giá được health của nó
- Map chính xác ISR Pages Router sang tương đương App Router

## 2. Evolution Context

Để hiểu ISR, cần quay lại bức tranh lớn.

**Next.js 9.0 (2019):** Bạn chỉ có hai lựa chọn:
- `getStaticProps` → trang tĩnh, fast, nhưng bị stale khi data thay đổi
- `getServerSideProps` → luôn fresh, nhưng slow vì tính toán mỗi request

Đây là một **false dichotomy** — thực tế không phải mọi trang đều cần fresh data theo từng millisecond, nhưng cũng không thể chờ rebuild toàn bộ site mỗi khi có thay đổi.

**Pain point điển hình:** Một trang e-commerce có 80,000 SKU. Giá sản phẩm thay đổi vài lần mỗi ngày. Nếu dùng SSG:
- Build time: 40 phút
- Trigger rebuild mỗi lần giá đổi → CI/CD bị nghẽn
- User vào lúc build đang chạy → thấy giá cũ 40 phút

Nếu dùng SSR:
- Mỗi request → gọi API pricing → 200ms latency thêm vào
- 80,000 concurrent users = 80,000 DB queries/s
- Chi phí server tăng theo traffic

**Next.js 9.5 (tháng 7/2020):** ISR ra đời với proposal đơn giản:
> "Serve trang tĩnh đã cache, nhưng cho phép regenerate trong background mà không cần rebuild toàn site."

**Timeline ISR:**
```
Next.js 9.5 (2020)   → ISR ra mắt, revalidate trong getStaticProps
Next.js 12.1 (2022)  → On-demand ISR beta (res.revalidate())
Next.js 12.2 (2022)  → On-demand ISR stable
Next.js 13.0 (2022)  → App Router: fetch({ next: { revalidate } })
Next.js 14.0 (2023)  → revalidateTag, revalidatePath stable
Next.js 15.0 (2024)  → "use cache" directive (experimental)
```

ISR không phải công nghệ mới — nó implement **RFC 5861** (HTTP stale-while-revalidate), một chuẩn HTTP header từ 2010. Next.js chỉ đưa nó lên server-side rendering.

## 3. Mental Model

### Stale-While-Revalidate là gì?

Hãy tưởng tượng một **sạp báo bán báo giấy**:

- Buổi sáng, sạp nhận tờ báo mới (build time → static page)
- Cả ngày, khách đến mua và nhận tờ báo đó (serve from cache → fast)
- Sau một khoảng thời gian nhất định (revalidate period), tòa soạn gửi báo cập nhật
- Khách mua **đúng lúc** báo cũ hết hạn vẫn nhận tờ báo cũ, nhưng bạn đặt in tờ mới ngay lập tức
- Khách **tiếp theo** nhận tờ báo mới

Cơ chế này trong Next.js ISR:

```
Timeline: revalidate = 60 giây

t=0s     Request 1 → Cache MISS → Generate page → Store in cache
t=30s    Request 2 → Cache HIT  → Serve ngay (30ms)
t=90s    Request 3 → Cache STALE → Serve trang cũ + Trigger background regen
t=91s    [Background] Fetch mới → Render → Ghi đè cache
t=92s    Request 4 → Cache HIT  → Serve trang mới (30ms)
```

**Điểm then chốt:** Không có request nào phải đợi server regenerate. Request 3 nhận trang cũ ngay lập tức — chỉ triggering regeneration như một side effect.

### ISR vs SSG vs SSR (bảng tư duy)

```
                   Build Time    Request Time   Freshness      Failure Mode
SSG                Dài           ~0ms           Stale forever  Build fails
ISR (time)         Ngắn          ~0ms           Eventually     Serve stale  
ISR (on-demand)    Ngắn          ~0ms (post-rev) Near-real-time Serve stale
SSR                Ngắn          200-500ms      Always fresh   Request fails
```

ISR nằm ở vùng giao thoa: **performance của SSG + freshness của SSR** — nhưng không hoàn hảo ở cả hai chiều. Đây là trade-off cần hiểu rõ trước khi chọn.

## 4. Core Concepts

### 4.1 Time-Based ISR

Chỉ cần thêm `revalidate` vào return value của `getStaticProps`:

```javascript
// pages/products/[id].js

export async function getStaticPaths() {
  // Pre-generate top 100 sản phẩm bán chạy nhất
  const topProducts = await fetchTopProducts({ limit: 100 });

  return {
    paths: topProducts.map((product) => ({
      params: { id: String(product.id) },
    })),
    fallback: 'blocking', // Xem giải thích fallback ở phần 4.2
  };
}

export async function getStaticProps({ params }) {
  const product = await fetchProductById(params.id);

  // Trả về 404 nếu không tìm thấy sản phẩm
  if (!product) {
    return { notFound: true };
  }

  return {
    props: {
      product,
    },
    revalidate: 300, // Regenerate tối đa 1 lần mỗi 5 phút
  };
}

export default function ProductPage({ product }) {
  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.price.toLocaleString('vi-VN')} VND</p>
    </div>
  );
}
```

**Giải mã `revalidate: 300`:**
- `300` là số giây — không phải "regenerate mỗi 300 giây"
- Mà là "sau 300 giây, trang này được coi là stale — request tiếp theo sẽ trigger regeneration"
- Nếu không có request nào đến sau 300 giây, Next.js **không tự động regenerate**
- ISR là **demand-driven**, không phải **time-driven**

Đây là lỗi tư duy rất phổ biến khi mới học ISR.

### 4.2 fallback Options với Dynamic Routes

Khi dùng ISR với dynamic routes, `fallback` trong `getStaticPaths` quyết định điều gì xảy ra với các path **chưa được pre-generate**:

**`fallback: false`**

```javascript
export async function getStaticPaths() {
  return {
    paths: [{ params: { id: '1' } }, { params: { id: '2' } }],
    fallback: false,
    // Path /products/999 → 404
    // Dùng khi: tập path cố định, ít, không thay đổi
  };
}
```

**`fallback: true`**

```javascript
// pages/products/[id].js
export async function getStaticPaths() {
  return {
    paths: [{ params: { id: '1' } }],
    fallback: true,
    // Path /products/999 → Show loading UI, generate ngầm, redirect về trang mới
  };
}

export default function ProductPage({ product }) {
  const router = useRouter();

  // Phải handle trạng thái loading khi fallback: true
  if (router.isFallback) {
    return <LoadingSkeleton />;
  }

  return <div>{product.name}</div>;
}
```

**`fallback: 'blocking'`**

```javascript
export async function getStaticPaths() {
  return {
    paths: [{ params: { id: '1' } }],
    fallback: 'blocking',
    // Path /products/999 → Đợi generate xong (như SSR), không cần xử lý loading
    // Dùng khi: không muốn layout shift, SEO quan trọng
  };
}
```

**So sánh fallback:**

```
fallback: false      → 404 cho path lạ. Dùng khi path list cố định.
fallback: true       → Loading UI ngay, generate background. UX nhanh nhưng phải code loading state.
fallback: 'blocking' → Đợi generate (SSR one-time), không cần loading state. SEO-safe.
```

**Pattern thực tế:** Hầu hết production site với ISR dùng `fallback: 'blocking'` cho content pages (blog, product) và `fallback: false` cho admin/utility pages.

### 4.3 On-Demand ISR

Time-based ISR có giới hạn: nếu `revalidate: 3600` (1 giờ) nhưng bạn vừa publish bài viết mới — user phải đợi tối đa 1 giờ mới thấy nội dung cập nhật. On-demand ISR giải quyết điều này.

**Bước 1: Tạo API Route revalidation**

```javascript
// pages/api/revalidate.js

export default async function handler(req, res) {
  // Kiểm tra secret token để ngăn revalidation trái phép
  if (req.query.secret !== process.env.REVALIDATION_SECRET) {
    return res.status(401).json({
      message: 'Invalid revalidation token',
    });
  }

  // Lấy path cần revalidate từ query string
  const path = req.query.path;

  if (!path) {
    return res.status(400).json({ message: 'Path is required' });
  }

  try {
    // res.revalidate() trigger regeneration cho path cụ thể
    await res.revalidate(path);

    return res.status(200).json({
      revalidated: true,
      path,
      timestamp: new Date().toISOString(),
    });
  } catch (err) {
    // Next.js throw error nếu path không tồn tại trong app
    return res.status(500).json({
      message: 'Error revalidating',
      error: err.message,
    });
  }
}
```

**Bước 2: Gọi từ CMS Webhook**

```javascript
// Ví dụ: Webhook handler khi content update trên Sanity CMS
// pages/api/cms-webhook.js

export default async function handler(req, res) {
  if (req.method !== 'POST') {
    return res.status(405).end();
  }

  const { _type, slug } = req.body;

  // Map content type → Next.js path
  const pathMap = {
    post: `/blog/${slug.current}`,
    product: `/products/${slug.current}`,
    page: `/${slug.current}`,
  };

  const pathToRevalidate = pathMap[_type];

  if (!pathToRevalidate) {
    return res.status(200).json({ message: 'Unknown content type, skipping' });
  }

  // Trigger revalidation
  await res.revalidate(pathToRevalidate);

  // Cũng revalidate listing page nếu content mới được tạo
  if (_type === 'post') {
    await res.revalidate('/blog');
  }

  return res.status(200).json({
    revalidated: [pathToRevalidate, '/blog'],
  });
}
```

**Pattern quan trọng:** On-demand ISR thường đi kèm với Webhook từ headless CMS (Contentful, Sanity, Strapi). Khi editor bấm "Publish" → CMS gọi webhook → Next.js revalidate ngay.

### 4.4 Graceful Degradation khi Lỗi

Đây là một trong những tính năng quan trọng nhất của ISR mà ít người để ý:

```javascript
export async function getStaticProps({ params }) {
  try {
    const product = await fetchProductById(params.id);

    return {
      props: { product, error: null },
      revalidate: 300,
    };
  } catch (error) {
    // Nếu background regeneration fail:
    // → Next.js GIỮ NGUYÊN trang cũ đang serve
    // → Không throw error ra cho user
    // → Log error ở server side

    // Nhưng nếu đây là lần đầu tiên build (không có cache):
    // → Error sẽ propagate và build fail

    console.error(`Failed to fetch product ${params.id}:`, error);

    // Cách 1: Throw để Next.js giữ trang cũ (trong background regen)
    throw error;

    // Cách 2: Return props rỗng với error state
    // return {
    //   props: { product: null, error: 'Failed to load' },
    //   revalidate: 30, // Retry sớm hơn
    // };
  }
}
```

**Graceful degradation flow:**

```
Background regeneration thất bại:
→ Next.js log lỗi server-side
→ Cache CŨ vẫn được giữ nguyên
→ User tiếp tục nhận trang cũ (stale nhưng không broken)
→ Lần request tiếp theo sẽ retry revalidation

So sánh với SSR:
→ API fail = request fail = user thấy error page
→ Không có safety net
```

Đây là lý do ISR phù hợp hơn SSR cho nhiều use case: **failure mode tốt hơn**.

## 5. Legacy Recognition

Khi đọc codebase cũ, đây là các pattern nhận dạng ISR:

**Pattern 1: `revalidate` trong getStaticProps**

```javascript
// ✅ Đây là ISR
export async function getStaticProps() {
  return {
    props: { data },
    revalidate: 60, // Số này tồn tại = ISR
  };
}

// ❌ Đây là thuần SSG (không có revalidate)
export async function getStaticProps() {
  return {
    props: { data },
    // không có revalidate
  };
}
```

**Pattern 2: `fallback: 'blocking'` hoặc `fallback: true` trong getStaticPaths**

```javascript
// ✅ ISR với dynamic routes
export async function getStaticPaths() {
  return {
    paths: [...],
    fallback: 'blocking', // Dấu hiệu của ISR pattern
  };
}
```

**Pattern 3: API Route `/api/revalidate`**

```javascript
// ✅ On-demand ISR
// Tìm file pages/api/revalidate.js hoặc pages/api/webhook.js
// Tìm res.revalidate() call
await res.revalidate('/some-path');
```

**Đánh giá health khi review legacy ISR code:**

```javascript
// 🔴 Code smell #1: revalidate quá thấp
export async function getStaticProps() {
  return {
    props: { data },
    revalidate: 1, // 1 giây → gần như SSR nhưng tệ hơn (lag 1 request)
  };
}

// 🔴 Code smell #2: fallback: true không có loading state
export default function Page({ data }) {
  // Thiếu: if (router.isFallback) return <Loading />;
  return <div>{data.name}</div>; // Crash nếu data là undefined
}

// 🔴 Code smell #3: revalidate: 0
// Không hợp lệ — sẽ bị Next.js ignore hoặc hành xử không dự đoán được

// 🟡 Code smell #4: revalidate rất lớn không có on-demand ISR
export async function getStaticProps() {
  return {
    props: { data },
    revalidate: 86400, // 24 giờ — OK nếu có webhook, tệ nếu không
  };
}
```

## 6. Modern Approach

ISR trong App Router không còn gắn với `getStaticProps` mà được tích hợp trực tiếp vào `fetch()` và các cache primitives mới.

### Time-based (Pages Router → App Router)

```javascript
// Pages Router
// pages/products/[id].js
export async function getStaticProps({ params }) {
  const product = await fetch(`/api/products/${params.id}`).then(r => r.json());

  return {
    props: { product },
    revalidate: 300,
  };
}

// App Router
// app/products/[id]/page.js
export default async function ProductPage({ params }) {
  // revalidate được đặt trực tiếp trên fetch call
  const product = await fetch(`/api/products/${params.id}`, {
    next: { revalidate: 300 },
  }).then(r => r.json());

  return <div>{product.name}</div>;
}
```

### On-demand (Pages Router → App Router)

```javascript
// Pages Router: res.revalidate()
await res.revalidate('/products/123');

// App Router: revalidatePath() hoặc revalidateTag()
import { revalidatePath, revalidateTag } from 'next/cache';

// Revalidate theo đường dẫn cụ thể
revalidatePath('/products/123');

// Revalidate theo tag (linh hoạt hơn nhiều)
// Fetch với tag:
const product = await fetch('/api/products/123', {
  next: { tags: ['products', 'products-123'] },
}).then(r => r.json());

// Revalidate tất cả fetch có tag 'products-123'
revalidateTag('products-123');
```

**App Router mạnh hơn Pages Router ISR ở điểm nào:**

```
Pages Router ISR            App Router Cache
─────────────────────────── ──────────────────────────────
revalidate ở page level     revalidate ở từng fetch call
on-demand theo path         on-demand theo path HOẶC tag
Không có granular control   fetch-level granularity
res.revalidate() clunky     revalidatePath/Tag clean API
```

### fallback trong App Router

`fallback` không còn tồn tại trong App Router. Thay vào đó, dynamic routes luôn được generate on-demand (tương đương `fallback: 'blocking'`), và bạn control build-time generation bằng `generateStaticParams`:

```javascript
// App Router equivalent của getStaticPaths + fallback: 'blocking'
// app/products/[id]/page.js

export async function generateStaticParams() {
  // Pre-generate top 100 sản phẩm
  const topProducts = await fetchTopProducts({ limit: 100 });

  return topProducts.map((product) => ({
    id: String(product.id),
  }));
}

// Các path không có trong generateStaticParams
// sẽ tự động generate on-demand (blocking) hoặc 404
// Tùy vào config dynamicParams:
export const dynamicParams = true; // default: true (generate on-demand)
// export const dynamicParams = false; // → 404 cho unknown paths
```

## 7. Production Scenario

### Bài toán: E-commerce Product Catalog

**Context:** Site bán lẻ với 50,000 SKU. Yêu cầu:
- Trang product load < 100ms (performance SLA)
- Giá cập nhật tối đa 3 lần/ngày
- Flash sale có thể trigger price change ngay lập tức
- Không rebuild toàn site

**Giải pháp ISR kết hợp time-based và on-demand:**

```javascript
// pages/products/[slug].js

export async function getStaticPaths() {
  // Chỉ pre-generate top 500 sản phẩm hot nhất
  // 49,500 sản phẩm còn lại sẽ được generate on-demand (fallback: 'blocking')
  const hotProducts = await db.product.findMany({
    orderBy: { viewCount: 'desc' },
    take: 500,
    select: { slug: true },
  });

  return {
    paths: hotProducts.map((p) => ({ params: { slug: p.slug } })),
    fallback: 'blocking',
  };
}

export async function getStaticProps({ params }) {
  const [product, relatedProducts, reviews] = await Promise.all([
    fetchProduct(params.slug),
    fetchRelatedProducts(params.slug, { limit: 6 }),
    fetchReviews(params.slug, { limit: 10 }),
  ]);

  if (!product) {
    return { notFound: true };
  }

  return {
    props: {
      product,
      relatedProducts,
      reviews,
      generatedAt: new Date().toISOString(),
    },
    // Time-based: tự động refresh sau 1 giờ nếu không có webhook
    revalidate: 3600,
  };
}
```

```javascript
// pages/api/webhooks/price-update.js
// Được gọi bởi pricing service khi có thay đổi giá

import crypto from 'crypto';

export default async function handler(req, res) {
  // Verify HMAC signature từ pricing service
  const signature = req.headers['x-webhook-signature'];
  const expectedSignature = crypto
    .createHmac('sha256', process.env.WEBHOOK_SECRET)
    .update(JSON.stringify(req.body))
    .digest('hex');

  if (signature !== expectedSignature) {
    return res.status(401).json({ message: 'Invalid signature' });
  }

  const { updatedSlugs } = req.body;

  // Batch revalidate tất cả sản phẩm vừa đổi giá
  const revalidationPromises = updatedSlugs.map((slug) =>
    res.revalidate(`/products/${slug}`)
  );

  await Promise.allSettled(revalidationPromises);
  // Promise.allSettled → không bị block nếu 1 slug fail

  return res.status(200).json({
    revalidated: updatedSlugs.length,
    timestamp: new Date().toISOString(),
  });
}
```

**Kết quả đạt được:**
- 500 sản phẩm hot: load < 50ms (cached at CDN edge)
- 49,500 sản phẩm còn lại: lần đầu ~300ms (generate), sau đó < 50ms
- Flash sale: trigger webhook → tất cả trang liên quan regenerate trong 30 giây
- Chi phí server: giảm ~90% so với toàn SSR

## 8. Debugging Scenario

### "Trang không update dù đã qua thời gian revalidate"

Đây là bug report phổ biến nhất khi làm việc với ISR.

**Bước 1: Hiểu cơ chế trước khi debug**

```
ISR không tự cron job chạy. Revalidation CHỈ xảy ra khi có REQUEST đến.
Nếu không có request → không regenerate → user không thấy thay đổi.

Đây không phải bug. Đây là thiết kế có chủ ý.
```

**Bước 2: Kiểm tra cache header**

```bash
# Gửi request và xem cache headers
curl -I https://yoursite.com/products/abc

# Header quan trọng cần xem:
# x-nextjs-cache: HIT   → Đang serve từ cache
# x-nextjs-cache: MISS  → Cache chưa có, generate ngay
# x-nextjs-cache: STALE → Đang serve stale, revalidation đang chạy ngầm
# x-nextjs-cache: REVALIDATED → Vừa được revalidate thành công
```

**Bước 3: Phân biệt CDN cache vs Next.js ISR cache**

```
[User] → [CDN Edge] → [Next.js Server] → [ISR Cache]

Vấn đề: CDN có cache RIÊNG!
Nếu CDN cache duration > revalidate duration:
→ User nhận CDN cache cũ
→ Request không đến Next.js server
→ ISR không được trigger
→ Trang không bao giờ cập nhật
```

```javascript
// Fix: Đặt Cache-Control header phù hợp trong API route
// hoặc cấu hình CDN để respect x-nextjs-cache

// next.config.js
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'Cache-Control',
            // s-maxage: thời gian CDN cache
            // stale-while-revalidate: cho phép serve stale trong khi revalidate
            value: 'public, s-maxage=300, stale-while-revalidate=600',
          },
        ],
      },
    ];
  },
};
```

```bash
# Vercel cung cấp UI để xem ISR status

# Hoặc dùng Vercel CLI
vercel logs --filter="revalidat"
```

**Bước 5: Bug điển hình — `notFound: true` block ISR**

```javascript
// 🔴 Bug ẩn
export async function getStaticProps({ params }) {
  const product = await fetchProduct(params.id);

  if (!product) {
    // notFound: true sẽ cache trang 404!
    // Nếu sau đó product được tạo → trang vẫn là 404 cho đến khi revalidate
    return { notFound: true };
  }

  return {
    props: { product },
    revalidate: 3600,
  };
}

// ✅ Fix: Thêm revalidate cho notFound case
export async function getStaticProps({ params }) {
  const product = await fetchProduct(params.id);

  if (!product) {
    return {
      notFound: true,
      revalidate: 60, // Retry sau 60s — sản phẩm có thể được tạo sau
    };
  }

  return {
    props: { product },
    revalidate: 3600,
  };
}
```

## 9. Refactoring Scenario

### Chuyển SSR Performance-bottleneck sang ISR

**Trước (SSR):** Trang blog nhận 50,000 lượt xem/ngày. `getServerSideProps` gọi API mất 300ms mỗi request. Không cần real-time data — bài viết cập nhật vài lần/tuần.

```javascript
// TRƯỚC: pages/blog/[slug].js (SSR)
// Chi phí: 50,000 req/ngày × 300ms = 15,000 giây CPU/ngày

export async function getServerSideProps({ params }) {
  // Mỗi request đều gọi API này
  const post = await fetch(`${API_URL}/posts/${params.slug}`).then(r => r.json());
  const author = await fetch(`${API_URL}/users/${post.authorId}`).then(r => r.json());
  const relatedPosts = await fetch(`${API_URL}/posts/related/${post.id}`).then(r => r.json());

  return {
    props: { post, author, relatedPosts },
  };
}
```

**Sau (ISR + On-demand):**

```javascript
// SAU: pages/blog/[slug].js (ISR)
// Chi phí: ~100 req/ngày (chỉ revalidation requests) × 300ms = 30 giây CPU/ngày

export async function getStaticPaths() {
  // Pre-generate 50 bài viết phổ biến nhất
  const popularPosts = await fetchPopularPosts({ limit: 50 });

  return {
    paths: popularPosts.map((post) => ({ params: { slug: post.slug } })),
    fallback: 'blocking',
  };
}

export async function getStaticProps({ params }) {
  const post = await fetch(`${API_URL}/posts/${params.slug}`).then(r => r.json());

  if (!post) {
    return { notFound: true, revalidate: 300 };
  }

  // Parallel fetch thay vì sequential
  const [author, relatedPosts] = await Promise.all([
    fetch(`${API_URL}/users/${post.authorId}`).then(r => r.json()),
    fetch(`${API_URL}/posts/related/${post.id}`).then(r => r.json()),
  ]);

  return {
    props: { post, author, relatedPosts },
    revalidate: 3600, // 1 giờ time-based
    // Editor publish bài mới → webhook → revalidate ngay
  };
}
```

```javascript
// pages/api/cms-revalidate.js — Webhook từ CMS
export default async function handler(req, res) {
  if (req.headers['x-cms-secret'] !== process.env.CMS_WEBHOOK_SECRET) {
    return res.status(401).end();
  }

  const { slug, action } = req.body;

  if (action === 'publish' || action === 'update') {
    await res.revalidate(`/blog/${slug}`);
    // Revalidate listing page nếu bài mới
    if (action === 'publish') {
      await res.revalidate('/blog');
    }
  }

  return res.status(200).json({ ok: true });
}
```

- Latency: 300ms → 15ms (95th percentile)
- Server CPU: giảm 99.8%
- TTFB (Time to First Byte): cải thiện 20x
- SEO score: giữ nguyên (server-rendered HTML)

## 10. Architecture Notes

### Khi nào chọn ISR?

```
Câu hỏi quyết định:

1. Data có thay đổi không?
   → Không → SSG thuần (không cần revalidate)
   → Có → tiếp tục

2. User có cần thấy data cập nhật NGAY khi request không?
   → Có → SSR (hoặc client-side fetch)
   → Không bắt buộc → ISR

3. Data có thay đổi theo từng user không?
   → Có → SSR + auth (không thể cache)
   → Không → ISR

4. Tần suất update như thế nào?
   → Vài lần/ngày → ISR time-based (revalidate: 3600)
   → Không thể đoán → ISR + on-demand webhook
   → Mỗi phút → ISR revalidate: 60 (hoặc xem xét lại)
   → Real-time → SSR hoặc WebSocket
```

### ISR Anti-patterns

**Anti-pattern 1: ISR thay thế SSR cho user-specific data**

```javascript
// 🔴 KHÔNG ĐƯỢC — Dashboard cá nhân không phù hợp với ISR
// Tất cả user sẽ thấy cùng một trang!
export async function getStaticProps({ params }) {
  const userDashboard = await fetchUserDashboard(params.userId);
  return {
    props: { userDashboard },
    revalidate: 60,
  };
}
```

**Anti-pattern 2: revalidate rất thấp không có lý do**

```javascript
// 🔴 Hành vi gần SSR nhưng luôn trả về data lag 1 request
export async function getStaticProps() {
  return {
    props: { data },
    revalidate: 1, // 1 giây → thundering herd risk cao
  };
}
```

**Anti-pattern 3: Không xử lý fallback state**

```javascript
// 🔴 Crash khi fallback: true
export default function Page({ product }) {
  return <h1>{product.name}</h1>; // product = undefined khi isFallback
}

// ✅ Đúng
export default function Page({ product }) {
  const router = useRouter();
  if (router.isFallback) return <Skeleton />;
  return <h1>{product.name}</h1>;
}
```

### Self-hosted vs Vercel: ISR khác nhau thế nào?

```
Vercel:
→ ISR cache tại CDN edge (global)
→ Revalidation trigger CDN purge tự động
→ Zero config

Self-hosted (Node.js):
→ ISR cache tại filesystem: .next/server/pages/
→ Không có CDN purge tự động
→ Cần dùng custom cache handler hoặc Redis

Self-hosted với custom cache handler:
```

```javascript
// next.config.js — Custom ISR cache handler (self-hosted)
module.exports = {
  experimental: {
    incrementalCacheHandlerPath: require.resolve('./cache-handler.js'),
  },
};

// cache-handler.js — Lưu ISR cache vào Redis thay vì filesystem
const { createClient } = require('redis');

module.exports = class RedisCacheHandler {
  constructor(options) {
    this.redis = createClient({ url: process.env.REDIS_URL });
    this.redis.connect();
  }

  async get(key) {
    const data = await this.redis.get(key);
    return data ? JSON.parse(data) : null;
  }

  async set(key, data, ctx) {
    const ttl = ctx.revalidate || 3600;
    await this.redis.setEx(key, ttl, JSON.stringify(data));
  }
};
```

## 11. Interview Questions

**Junior:**

**Q1: ISR là gì và khác gì với SSG?**

<details>
<summary>Đáp án</summary>

SSG (Static Site Generation) tạo ra trang HTML tại build time và không bao giờ thay đổi cho đến lần build tiếp theo. Nhược điểm là khi data thay đổi, phải rebuild toàn bộ site.

ISR (Incremental Static Regeneration) cho phép trang tĩnh được regenerate trong background sau khi được deploy, mà không cần rebuild toàn bộ site. Bằng cách thêm `revalidate` vào `getStaticProps`, Next.js sẽ tái tạo trang đó trong nền khi có request đến sau khoảng thời gian revalidate.

Điểm khác biệt chính: SSG = "tĩnh vĩnh viễn", ISR = "tĩnh nhưng có thể tự cập nhật theo cơ chế stale-while-revalidate".

</details>

**Mid:**

**Q2: Giải thích cơ chế stale-while-revalidate trong ISR. Điều gì xảy ra khi request đến sau thời gian revalidate?**

<details>
<summary>Đáp án</summary>

Khi một request đến sau thời gian revalidate đã qua:

Bước 1 — Next.js serve ngay trang cached cũ (stale) cho request đó. User không phải đợi.

Bước 2 — Đồng thời, Next.js trigger background regeneration: gọi lại `getStaticProps`, fetch data mới, render HTML mới.

Bước 3 — Khi background generation hoàn tất, cache mới ghi đè cache cũ.

Bước 4 — Request tiếp theo nhận trang mới.

Điểm quan trọng: Revalidation là demand-driven (chỉ xảy ra khi có request), không phải cron job tự động chạy sau mỗi `revalidate` giây. Nếu không có request nào đến, trang sẽ không được regenerate dù đã quá thời gian revalidate.

</details>

**Q3: Sự khác biệt giữa `fallback: true`, `fallback: false`, và `fallback: 'blocking'` trong `getStaticPaths`?**

<details>
<summary>Đáp án</summary>

`fallback: false` — Các path không có trong `getStaticPaths` sẽ trả về 404. Dùng khi số lượng path cố định và ít, không có path mới được tạo sau build.

`fallback: true` — Path không được pre-generate sẽ nhận được một phiên bản "fallback" của trang ngay lập tức (có thể show loading skeleton), trong khi Next.js generate trang thật trong background. Component cần kiểm tra `router.isFallback` để xử lý trạng thái loading. Sau khi generate xong, trang được cache và các request sau sẽ nhận trang đầy đủ.

`fallback: 'blocking'` — Tương tự `fallback: true` nhưng thay vì show fallback UI, request sẽ đợi cho đến khi server generate xong (giống SSR cho lần đầu). Không cần xử lý `isFallback`. SEO-safe hơn vì HTML đầy đủ được trả về ngay. Đây là lựa chọn phổ biến nhất trong production.

</details>

**Senior:**

**Q4: Khi nào KHÔNG nên dùng ISR? Và đâu là rủi ro của ISR mà ít người để ý?**

<details>
<summary>Đáp án</summary>

Không nên dùng ISR khi:

Thứ nhất, data là user-specific. ISR cache toàn cầu — mọi user thấy cùng một trang. Dashboard cá nhân, giỏ hàng, profile page không thể dùng ISR.

Thứ hai, cần real-time data. ISR có độ trễ ít nhất bằng revalidate period. Chat, stock prices, live sports scores cần SSR hoặc WebSocket.

Thứ ba, data thay đổi cực kỳ thường xuyên. `revalidate: 1` gần như SSR nhưng tệ hơn vì luôn trả về data lag 1 request.

Rủi ro ít được để ý:

Thundering herd: Khi revalidate period hết và có 1000 request đến cùng lúc, tất cả đều trigger revalidation. Next.js có deduplication built-in nhưng không hoàn hảo trong mọi setup.

CDN caching layer: Nhiều team quên rằng CDN phía trước có cache riêng. ISR revalidate xong nhưng CDN vẫn serve trang cũ vì CDN TTL chưa hết.

notFound cache: `return { notFound: true }` cũng bị cache. Nếu một sản phẩm bị xóa rồi được tạo lại, trang 404 vẫn được serve cho đến khi revalidate — trừ khi bạn thêm `revalidate` vào `notFound` response.

Self-hosted ISR: Trên Vercel ISR hoạt động out-of-the-box với CDN invalidation. Tự host cần configure custom cache handler, otherwise ISR cache chỉ nằm trên 1 server instance — không work với multi-instance deployment.

</details>

**Q5: Làm sao implement on-demand ISR và tại sao nó quan trọng hơn time-based ISR trong nhiều use case?**

<details>
<summary>Đáp án</summary>

On-demand ISR được implement bằng cách tạo một API Route gọi `res.revalidate(path)`, thường được trigger bởi webhook từ CMS hoặc backend service khi có data thay đổi.

```
// Ví dụ flow:
Editor publish bài viết trên Sanity CMS
→ Sanity gọi webhook đến /api/cms-webhook
→ Handler gọi res.revalidate('/blog/ten-bai-viet')
→ Next.js regenerate trang ngay lập tức
→ User thấy nội dung mới trong vài giây
```

Quan trọng hơn time-based vì:

Time-based ISR có "best case" freshness bằng revalidate period. Nếu `revalidate: 3600` và editor vừa publish bài quan trọng, user phải chờ tối đa 1 giờ. Với breaking news hay flash sale, đây không chấp nhận được.

On-demand ISR cho phép freshness gần real-time với chi phí của SSG. Trang vẫn được serve từ cache (nhanh), nhưng regenerate ngay khi có thay đổi thay vì đợi theo lịch.

Trong production, best practice là kết hợp cả hai: `revalidate` dài (safety net) + on-demand webhook (freshness chính).

</details>

<summary>Đáp án</summary>

Vấn đề 1 — Pre-generate 50,000 sản phẩm tại build time: Đây là anti-pattern nghiêm trọng. Build sẽ mất hàng giờ vì Next.js phải call `getStaticProps` 50,000 lần. Giải pháp: Chỉ pre-generate top N sản phẩm (ví dụ 100-500), còn lại dùng `fallback: 'blocking'` để generate on-demand.

Vấn đề 2 — `revalidate: 2` quá thấp: 2 giây revalidate với 50,000 sản phẩm và traffic cao sẽ tạo thundering herd problem. Gần như mọi request đều trigger background regeneration, gây tải nặng cho server. Nên cân nhắc tăng revalidate lên hợp lý (60-3600s) hoặc dùng on-demand ISR nếu cần freshness cao.

Vấn đề 3 — `fallback: true` nhưng không handle `isFallback`: Component render `product.name` trực tiếp. Khi `fallback: true` và path chưa được generate, `product` sẽ là `undefined` → crash. Cần thêm: `if (router.isFallback) return <Loading />;`

Vấn đề 4 — Không handle `product` null/undefined: Nếu `fetchProduct` trả về null (sản phẩm không tồn tại), component sẽ crash. Cần kiểm tra và return `{ notFound: true }`.

Vấn đề 5 — Không có error handling trong `getStaticProps`: Nếu `fetchProduct` throw error trong lần build đầu tiên, build sẽ fail. Nên wrap trong try/catch.

Phiên bản đã fix:

```javascript
    const product = await fetchProduct(params.id);
    if (!product) {
      return { notFound: true, revalidate: 60 };
    }
    return { props: { product }, revalidate: 300 };
  } catch (err) {
    throw err; // Giữ cache cũ khi background regen fail
  }
}

  const router = useRouter();
  if (router.isFallback) return <Loading />;
  return <div>{product.name}</div>;
```

## 13. Summary

ISR là câu trả lời của Next.js cho bài toán: **"Làm sao có performance của static site nhưng data vẫn được cập nhật?"**

Cơ chế cốt lõi là **stale-while-revalidate** — serve trang cached ngay lập tức, đồng thời trigger regeneration trong background. Không có request nào phải đợi server compute, nhưng data vẫn được refresh định kỳ.

**Hai chế độ ISR:**

Time-based (`revalidate: N`) phù hợp khi bạn có thể chấp nhận data lag tối đa N giây. On-demand (`res.revalidate(path)`) phù hợp khi cần cập nhật ngay khi có thay đổi — thường được trigger bởi CMS webhook.

**Ba quyết định quan trọng khi dùng ISR:**

Quyết định một — `fallback` strategy: `'blocking'` là lựa chọn mặc định an toàn nhất cho production; chỉ dùng `true` khi UX cần loading state ngay.

Quyết định hai — `revalidate` value: Cân nhắc giữa freshness và server load. Không đặt quá thấp nếu không cần thiết.

Quyết định ba — Có thêm on-demand webhook không: Nếu data có thể thay đổi đột ngột (flash sale, breaking news), time-based ISR một mình không đủ.

**Khi KHÔNG dùng ISR:** Data user-specific (dashboard, cart), real-time requirements (chat, live scores), hoặc data thay đổi sub-second.

**Migration sang App Router:** `revalidate` trong `getStaticProps` → `next: { revalidate }` trong `fetch()`; `res.revalidate()` → `revalidatePath()` / `revalidateTag()`. Logic tương đương nhưng App Router cho phép granularity ở mức từng fetch call thay vì toàn bộ page.

ISR là một trong những tính năng khiến Next.js trở thành lựa chọn hàng đầu cho content sites — nó giải quyết được bài toán mà SSG và SSR riêng lẻ không làm được.
