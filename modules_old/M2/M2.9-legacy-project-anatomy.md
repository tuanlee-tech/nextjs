# M2.9 — Legacy Project Anatomy

## 1. Lesson Goal

Sau bài này bạn phải:

- Mở một legacy Next.js project và hiểu được **cấu trúc trong 30 phút** thay vì 3 ngày
- Đọc được `_app.js`, `_document.js`, `next.config.js` như đọc hồ sơ bệnh nhân
- Nhận ra project này dùng **Next.js era nào** từ các clue trong code
- Vẽ được **bản đồ data flow** của project mà không cần ai giải thích
- Xác định được **technical debt** và biết cái nào nguy hiểm, cái nào có thể sống chung
- Biết cách tiếp cận khi được assign vào một legacy codebase thực tế

## 2. Evolution Context

Phần lớn công việc của một engineer không phải là xây dựng từ đầu — mà là **maintain, extend, và cải thiện** hệ thống người khác đã viết.

Thực tế của các công ty đang dùng Next.js:

```
2019-2020: Công ty build app mới với Next.js 9-10
           → Dùng getInitialProps vì chưa có getServerSideProps
           → Custom server.js phổ biến

2020-2021: App phát triển, team thêm người
           → Mix getInitialProps + getServerSideProps
           → Bundle phình to, performance suy giảm

2021-2022: Next.js 12, team mới thêm code
           → Một số trang mới dùng ISR, trang cũ vẫn SSR thuần
           → next.config.js chứa đủ thứ accumulated qua các năm

2022-2024: Next.js 13-14 ra đời
           → Công ty chưa kịp migrate
           → Pages Router đang chạy production, App Router chỉ thí nghiệm
           → Engineering debt ngày càng cao
```

Bạn bước vào một trong những công ty đó. Đây là bài học để bạn không bị lạc.

## 3. Mental Model

### Đọc codebase như một nhà khảo cổ

Một legacy codebase là **kết quả của nhiều lớp quyết định** theo thời gian. Giống khảo cổ học — mỗi lớp phản ánh một thời kỳ, một mindset, một áp lực deadline khác nhau.

```
Lớp 2022: App Router thử nghiệm (app/)
           ↓
Lớp 2021: ISR, next/script, next/image v2 (pages/blog/)
           ↓
Lớp 2020: getServerSideProps, NextAuth v3 (pages/dashboard/)
           ↓
Lớp 2019: getInitialProps, custom server (pages/legacy-feature/)
           ↓
Nền tảng:  _app.js, _document.js, next.config.js (chứa tất cả các lớp)
```

Chiến lược đọc: **từ ngoài vào trong, từ config đến code, từ tổng quan đến chi tiết.**

```
Bước 1: package.json → Biết công nghệ và era
Bước 2: next.config.js → Biết kiến trúc và config đặc biệt
Bước 3: pages/_app.js → Biết global state và providers
Bước 4: pages/_document.js → Biết HTML structure choices
Bước 5: Cấu trúc thư mục pages/ → Hiểu domain model
Bước 6: Một page cụ thể → Hiểu data fetching pattern
Bước 7: pages/api/ → Hiểu backend structure
```

**Những gì CẦN chú ý khi scan lần đầu:**
- Có thư mục `app/` không? Nếu có: project đang migrate sang App Router
- Có file `server.js` hoặc `server.ts` không? Nếu có: custom server, ISR có thể bị ảnh hưởng
- Thư mục `store/` dùng gì? Redux = codebase nặng và cũ hơn; Zustand/Jotai = tương đối mới
- Có `middleware.js` ở root không? Nếu có: Next.js 12+, có edge logic

### 4.2 Đọc `_app.js` — DNA của Project

`_app.js` là file quan trọng nhất để đọc đầu tiên. Nó tiết lộ toàn bộ kiến trúc frontend.

```javascript
// pages/_app.js — Đây là một ví dụ điển hình từ 2021-2022

// 🔍 CLUE #1: Import thứ tự tiết lộ dependencies
import '../styles/globals.css';
// CSS-in-JS? Không thấy → Đang dùng CSS Modules hoặc global CSS

import { SessionProvider } from 'next-auth/react';
// 🔍 CLUE #2: next-auth/react (không phải next-auth/client) → NextAuth v4+ (2022)

import { Provider } from 'react-redux';
// 🔍 CLUE #3: Redux Provider → State management là Redux

import { store } from '../store';
// Tìm file store/index.js tiếp theo để hiểu store shape

import { QueryClient, QueryClientProvider } from 'react-query';
// 🔍 CLUE #4: react-query → Server state riêng biệt với Redux
// react-query (không phải @tanstack/react-query) → v3 hoặc cũ hơn

import Layout from '../components/Layout';
// 🔍 CLUE #5: Layout wrapper ở _app → Toàn bộ trang share cùng layout
// Tìm components/Layout/index.js để hiểu global UI structure

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000, // 5 phút
      retry: 1,
    },
  },
});

function MyApp({ Component, pageProps: { session, ...pageProps } }) {
  // 🔍 CLUE #6: Destructure session từ pageProps → NextAuth session được pass từ SSR

  return (
    <SessionProvider session={session}>
      {/* Auth context bao ngoài cùng */}
      <Provider store={store}>
        {/* Redux bao trong Auth */}
        <QueryClientProvider client={queryClient}>
          {/* React Query bao trong Redux */}
          <Layout>
            {/* Layout global cho tất cả trang */}
            <Component {...pageProps} />
          </Layout>
        </QueryClientProvider>
      </Provider>
    </SessionProvider>
  );
}

// 🔍 CLUE #7: Không có getInitialProps ở đây → Tốt!
// Nếu có getInitialProps trong _app.js → Tắt Automatic Static Optimization
// cho TẤT CẢ các trang

export default MyApp;
```

**Checklist đọc `_app.js`:**

```
□ Authentication provider nào? (SessionProvider, Auth0Provider, custom Context...)
□ State management nào? (Redux, Zustand, Jotai, Context API...)
□ Server state nào? (react-query, SWR, Apollo...)
□ CSS solution nào? (CSS Modules, styled-components, emotion, tailwind...)
□ Analytics/Monitoring? (GA, Segment, Sentry, Datadog...)
□ CÓ getInitialProps trong _app.js không? → Nếu có: performance vấn đề nghiêm trọng
□ CÓ PageTransition nào không? → Animation library
□ CÓ ThemeProvider không? → Design system
```

**Code smell nguy hiểm nhất trong `_app.js`:**

```javascript
// 🔴 NGUY HIỂM: getInitialProps trong _app.js
// Điều này disable Automatic Static Optimization cho TẤT CẢ trang trong project
// Kể cả những trang không cần data fetching

MyApp.getInitialProps = async (appContext) => {
  // Khi có dòng này, KHÔNG CÓ trang nào trong project được tối ưu static
  // Mọi trang đều bị force thành SSR
  const appProps = await App.getInitialProps(appContext);
  const globalData = await fetchGlobalData();

  return {
    ...appProps,
    pageProps: {
      ...appProps.pageProps,
      globalData,
    },
  };
};

// ✅ Giải pháp thay thế: Dùng SWR/react-query để fetch global data
// hoặc Context API với lazy initialization
```

### 4.3 Đọc `_document.js` — HTML Architecture Choices

`_document.js` ít được đụng vào nhưng tiết lộ nhiều về styling choices:

```javascript
// pages/_document.js

import Document, { Html, Head, Main, NextScript } from 'next/document';

// 🔍 CLUE #1: Import ServerStyleSheet → Project dùng styled-components
import { ServerStyleSheet } from 'styled-components';

class MyDocument extends Document {
  // 🔍 CLUE #2: Static getInitialProps trong Document → SSR style collection
  static async getInitialProps(ctx) {
    const sheet = new ServerStyleSheet();
    const originalRenderPage = ctx.renderPage;

    try {
      ctx.renderPage = () =>
        originalRenderPage({
          enhanceApp: (App) => (props) =>
            sheet.collectStyles(<App {...props} />),
        });

      const initialProps = await Document.getInitialProps(ctx);

      return {
        ...initialProps,
        styles: (
          <>
            {initialProps.styles}
            {sheet.getStyleElement()}
          </>
        ),
      };
    } finally {
      sheet.seal();
    }
  }

  render() {
    return (
      // 🔍 CLUE #3: lang="vi" → Vietnamese site
      <Html lang="vi">
        <Head>
          {/* 🔍 CLUE #4: Google Fonts import → Font không tối ưu (Next.js 13 dùng next/font) */}
          <link rel="preconnect" href="https://fonts.googleapis.com" />
          <link
            href="https://fonts.googleapis.com/css2?family=Be+Vietnam+Pro:wght@400;600;700&display=swap"
            rel="stylesheet"
          />
          {/* 🔍 CLUE #5: Meta theme-color → PWA hoặc mobile consideration */}
          <meta name="theme-color" content="#1a1a2e" />
        </Head>
        <body>
          <Main />
          <NextScript />
        </body>
      </Html>
    );
  }
}

export default MyDocument;
```

**Nếu `_document.js` đơn giản hoặc không tồn tại:** Project dùng CSS Modules hoặc Tailwind — không cần style collection.

**Nếu `_document.js` phức tạp với emotion:** Dùng `@emotion/server` — tương tự styled-components nhưng khác syntax.

### 4.4 Đọc `next.config.js` — Bảo tàng Kiến trúc

`next.config.js` là file tích lũy mọi quyết định qua các năm. Đọc nó để hiểu lịch sử project:

```javascript
// next.config.js — Phân tích từng dòng

/** @type {import('next').NextConfig} */
const nextConfig = {

  // 🔍 ERA CLUE #1: reactStrictMode: false → Team đã tắt vì có side-effects kép
  // (common workaround khi upgrade lên React 18)
  reactStrictMode: false,

  // 🔍 ERA CLUE #2: target: 'serverless' → Next.js 9-11 ONLY, đã bị remove trong 12
  // Nếu thấy config này → Project cũ, có thể bị break khi upgrade
  // target: 'serverless', // ← Đã comment out khi upgrade lên Next.js 12

  // 🔍 ERA CLUE #3: images.domains (old API)
  // remotePatterns (new API, Next.js 12.3+) → Nếu vẫn dùng domains: project chưa update
  images: {
    domains: [
      'cdn.example.com',
      'images.unsplash.com',
      'lh3.googleusercontent.com', // Google OAuth avatar
    ],
    // Không có formats: ['image/avif', 'image/webp'] → Chưa optimize image format
  },

  // 🔍 ERA CLUE #4: env (old pattern)
  // Pattern cũ: expose qua env config
  // Pattern mới: NEXT_PUBLIC_ prefix trong .env file
  env: {
    SITE_URL: process.env.SITE_URL, // Lẽ ra phải là NEXT_PUBLIC_SITE_URL
  },

  // 🔍 ERA CLUE #5: webpack config override → Custom bundling
  webpack: (config, { isServer }) => {
    // Thêm SVG support — Next.js 13+ có built-in SVG handling
    config.module.rules.push({
      test: /\.svg$/,
      use: ['@svgr/webpack'],
    });

    // 🔴 RED FLAG: Nếu có nhiều webpack overrides → Build process phức tạp,
    // upgrade Next.js version sẽ khó khăn

    return config;
  },

  // 🔍 ERA CLUE #6: experimental config
  experimental: {
    // 'appDir: true' → Team đã thử App Router (Next.js 13+)
    // appDir: true, // ← Đã tắt, chưa sẵn sàng production

    // 'scrollRestoration: true' → Experimental feature từ Next.js 11
    scrollRestoration: true,
  },

  // 🔍 Pattern: redirects thường chứa lịch sử URL restructuring
  async redirects() {
    return [
      {
        source: '/san-pham/:id', // URL cũ bằng tiếng Việt
        destination: '/products/:id', // URL mới tiếng Anh
        permanent: true,
        // 🔍 CLUE: Project từng có URL tiếng Việt, sau đổi sang English
      },
    ];
  },

  // 🔍 Headers tiết lộ security posture của team
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          { key: 'X-Frame-Options', value: 'DENY' },
          { key: 'X-Content-Type-Options', value: 'nosniff' },
          // 🔴 Thiếu: Content-Security-Policy → Security gap
        ],
      },
    ];
  },
};

module.exports = nextConfig;
```

### 4.6 Map Data Fetching trong Codebase

Sau khi đọc config, mục tiêu tiếp theo là hiểu mỗi trang dùng strategy gì. Không cần đọc từng file — chỉ cần `grep`:

```bash
# Tìm tất cả pages dùng getServerSideProps
grep -rl "getServerSideProps" pages/ --include="*.js" --include="*.ts"

# Tìm tất cả pages dùng getStaticProps
grep -rl "getStaticProps" pages/ --include="*.js" --include="*.ts"

# Tìm tất cả pages dùng getStaticProps CÓ revalidate (ISR)
grep -rl "revalidate" pages/ --include="*.js" --include="*.ts"

# Tìm getInitialProps còn sót lại
grep -rl "getInitialProps" pages/ --include="*.js" --include="*.ts"

# Tìm pages hoàn toàn client-side (không có data fetching nào)
# → Các pages này là CSR thuần
```

**Vẽ bản đồ data fetching từ kết quả:**


### 4.7 API Routes — Đọc Backend Structure

```javascript
// pages/api/products/index.js
// Đọc file này để hiểu backend pattern của team

export default async function handler(req, res) {

  // 🔍 PATTERN CHECK: HTTP method routing

  // Pattern 1 (cũ): if/else chain
  if (req.method === 'GET') {
    // ...
  } else if (req.method === 'POST') {
    // ...
  } else {
    res.setHeader('Allow', ['GET', 'POST']);
    res.status(405).end(`Method ${req.method} Not Allowed`);
  }

  // Pattern 2 (hiện đại hơn): switch
  // Pattern 3 (best): Route handlers trong App Router
}
```

```javascript
// pages/api/products/index.js — Một ví dụ thực tế hơn

import { getServerSession } from 'next-auth/next';
// 🔍 getServerSession → NextAuth v4 pattern (tốt)

import { authOptions } from '../auth/[...nextauth]';
import { prisma } from '../../../lib/db';

export default async function handler(req, res) {
  const session = await getServerSession(req, res, authOptions);

  // 🔍 Auth check pattern
  if (!session) {
    return res.status(401).json({ error: 'Unauthorized' });
  }

  if (req.method === 'GET') {
    try {
      // 🔍 Prisma → ORM choice
      const products = await prisma.product.findMany({
        where: { shopId: session.user.shopId },
        include: { category: true },
        // 🔴 RED FLAG: Không có pagination → N+1 và memory issues ở scale
        // take: 10, ← Nên có
        // skip: offset, ← Nên có
      });

      return res.status(200).json(products);
    } catch (error) {
      // 🔴 RED FLAG: console.error thay vì structured logging
      console.error('Error fetching products:', error);
      return res.status(500).json({ error: 'Internal server error' });
    }
  }

  // 🔴 RED FLAG: Không handle POST, không return 405
  // Caller sẽ nhận undefined response
}
```

**Taxonomy của API Routes trong legacy project:**

```
/api/auth/[...nextauth].js  → Auth endpoint — KHÔNG touch
/api/webhooks/*             → Webhook handlers — Cẩn thận, có side effects
/api/[resource]/index.js    → CRUD list operations
/api/[resource]/[id].js     → CRUD single resource
```

```bash
# Kết quả: index.js  [id].js
```

**Bước 2 — Đọc data fetching strategy:**

```javascript
// pages/products/index.js — Bạn thấy gì?

import { useState } from 'react';
import ProductGrid from '../../components/features/ProductGrid';
import { fetchProducts } from '../../lib/api';

// 🔍 Không thấy getStaticProps hay getServerSideProps ở bottom file?
// Scroll xuống...

export async function getServerSideProps(context) {
  // 🔍 SSR! Mỗi request đều chạy qua đây

  const { page = 1, category, sort } = context.query;

  // 🔍 Sequential fetches — không phải parallel!
  const categories = await fetchCategories(); // 300ms
  const products = await fetchProducts({    // 500ms
    page,
    category,
    sort,
    limit: 24,
  });
  const featuredBanners = await fetchBanners(); // 200ms

  // Tổng: 1000ms chỉ cho data fetching
  // Chưa tính render time và network round trip

  return {
    props: {
      categories,
      products,
      featuredBanners,
      currentPage: Number(page),
    },
  };
}
```

**Bước 3 — Hiểu vấn đề:**

```
Phát hiện:
1. Trang Products dùng SSR → Mỗi request = 1000ms server time
2. Fetches sequential không parallel → 3 API calls nối tiếp nhau
3. categories và featuredBanners KHÔNG phụ thuộc vào query params
   → Chúng có thể cache được, không cần SSR

Recommendation:
- categories → getStaticProps + revalidate: 3600 (thay đổi ít)
- featuredBanners → getStaticProps + revalidate: 1800
- products → Giữ SSR (phụ thuộc page/category/sort)
- Chuyển categories/banners sang parallel fetch trong SSR (short-term fix)
```

**Bước 4 — Quick fix để report trước khi refactor:**

```javascript
// pages/products/index.js — Quick fix: Parallel fetch
export async function getServerSideProps(context) {
  const { page = 1, category, sort } = context.query;

  // Parallel fetch thay vì sequential → Giảm từ 1000ms xuống ~500ms
  const [categories, products, featuredBanners] = await Promise.all([
    fetchCategories(),
    fetchProducts({ page, category, sort, limit: 24 }),
    fetchBanners(),
  ]);

  return {
    props: {
      categories,
      products,
      featuredBanners,
      currentPage: Number(page),
    },
  };
}
```

**Kết quả có thể report ngay ngày đầu:** Load time giảm từ ~4.5s xuống ~2.5s chỉ bằng cách đổi sequential → parallel fetch, không cần refactor kiến trúc.

## 9. Refactoring Scenario

### Refactor `getInitialProps` sang `getServerSideProps`

Bạn tìm thấy file `pages/legacy-checkout.js` dùng `getInitialProps` — pattern từ Next.js 9:

```javascript
// TRƯỚC — pages/legacy-checkout.js (Next.js 9 pattern)

function CheckoutPage({ cart, user, shippingOptions }) {
  // Component logic
}

// getInitialProps chạy cả ở server lẫn client
// Khi navigate về trang này, sẽ chạy lại ở CLIENT
CheckoutPage.getInitialProps = async (ctx) => {
  // ctx.req chỉ có ở server-side! Ở client-side ctx.req = undefined
  const isServer = !!ctx.req;

  // 🔴 Code phải handle cả hai environments
  const baseUrl = isServer
    ? `http://localhost:${process.env.PORT || 3000}`
    : '';

  const [cart, user, shippingOptions] = await Promise.all([
    fetch(`${baseUrl}/api/cart`).then(r => r.json()),
    fetch(`${baseUrl}/api/user/me`).then(r => r.json()),
    fetch(`${baseUrl}/api/shipping/options`).then(r => r.json()),
  ]);

  return { cart, user, shippingOptions };
};

export default CheckoutPage;
```

```javascript
// SAU — pages/legacy-checkout.js (Modern pattern)

function CheckoutPage({ cart, user, shippingOptions }) {
  // Component logic không đổi
}

// getServerSideProps chỉ chạy ở server — đơn giản hơn nhiều
export async function getServerSideProps(context) {
  // Không cần handle server/client branching
  // Không cần construct base URL

  // Tốt hơn: Import trực tiếp thay vì gọi qua HTTP
  const { getCart, getUserById, getShippingOptions } = await import('../lib/api');
  const session = await getServerSession(context.req, context.res, authOptions);

  if (!session) {
    return {
      redirect: {
        destination: '/login?redirect=/checkout',
        permanent: false,
      },
    };
  }

  const [cart, user, shippingOptions] = await Promise.all([
    getCart(session.user.id),
    getUserById(session.user.id),
    getShippingOptions(),
  ]);

  return {
    props: {
      cart,
      user,
      shippingOptions,
    },
  };
}

export default CheckoutPage;
```

**Lợi ích của migration này:**

```
TRƯỚC (getInitialProps):
→ Runs on server AND client
→ Phải handle isServer branching
→ Không có redirect support native
→ Không có notFound support native
→ API call qua HTTP (unnecessary round trip)

SAU (getServerSideProps):
→ Server only — đơn giản, predictable
→ Direct function import (không qua HTTP)
→ Native redirect support
→ Native notFound support
→ Dễ đọc và maintain hơn
```

### Khi Nào KHÔNG Refactor

```

❌ Nếu sắp có migration lớn (Pages → App Router) — Đừng refactor sang Pages Router mới
❌ Nếu team size < 2 người — Focus vào feature delivery
❌ Nếu deadline < 2 tuần — "If it ain't broke, don't fix it"
```

### Technical Debt Taxonomy

```
Type 1 — Deliberate Debt (có chủ đích):
→ Viết nhanh vì deadline, biết cần sửa sau
→ Nguy hiểm nếu "sau" không bao giờ đến
→ Cần track trong backlog

Type 2 — Accidental Debt (vô tình):
→ Code không biết là anti-pattern
→ getInitialProps trong _app.js là ví dụ
→ Cần education và review process

Type 3 — Bit Rot (mục nát theo thời gian):
→ Code đúng khi viết, sai vì environment thay đổi
→ Ví dụ: target: 'serverless' trong Next.js 12
→ Cần regular dependency upgrade

Type 4 — Environment Debt:
→ Package vulnerabilities, outdated Node.js
→ Dễ fix nhất nhưng thường bị bỏ qua
→ Cần CI/CD check tự động
```

## 11. Interview Questions

**Junior:**

**Q1: `_app.js` và `_document.js` trong Next.js dùng để làm gì? Chúng khác nhau như thế nào?**

<details>
<summary>Đáp án</summary>

`_app.js` là entry point của tất cả trang trong Pages Router. Nó wrap mọi page component và được dùng để thêm global state (Redux Provider, SessionProvider), global CSS, và persistent layout (Header, Footer). Component nhận `{ Component, pageProps }` — `Component` là trang đang render, `pageProps` là data từ `getServerSideProps` hay `getStaticProps` của trang đó.

`_document.js` dùng để tùy chỉnh HTML document structure — tức là thẻ `<html>`, `<head>`, `<body>`. Nó chỉ render ở server, không render ở client. Dùng khi cần thêm custom font trong `<head>`, set `lang` attribute, hoặc collect styles cho CSS-in-JS (styled-components, emotion) để tránh FOUC (Flash of Unstyled Content).

Điểm khác biệt quan trọng: `_app.js` render cả server lẫn client (interactive), còn `_document.js` chỉ render ở server một lần duy nhất.

</details>

**Q2: Nếu bạn thấy `MyApp.getInitialProps` trong `_app.js`, điều đó có nghĩa gì với performance của project?**

<details>
<summary>Đáp án</summary>

Đây là một red flag nghiêm trọng về performance. Khi `getInitialProps` được định nghĩa trong `_app.js`, Next.js tắt Automatic Static Optimization cho toàn bộ tất cả các trang trong project — kể cả những trang không cần server-side data (như trang About, FAQ, Error page).

Điều này có nghĩa là mọi trang, dù đơn giản đến đâu, đều bị force thành Server-Side Rendered, không thể được tối ưu thành trang tĩnh. Kết quả là tăng latency, tăng chi phí server, và giảm performance toàn bộ app.

Giải pháp: Loại bỏ `getInitialProps` khỏi `_app.js`. Chuyển global data sang Context API với lazy loading, hoặc dùng SWR/react-query để fetch global data ở client side. Nếu cần session trên mọi trang, dùng `getServerSideProps` ở từng trang cần hoặc dùng middleware để validate session.

</details>

**Mid:**

**Q3: Khi join một dự án Next.js mới, bạn làm gì trong ngày đầu tiên để nhanh chóng hiểu codebase?**

<details>
<summary>Đáp án</summary>

Tôi đọc codebase theo thứ tự từ ngoài vào trong, không đọc random theo code flow.

Bước một: `package.json` để xác định Next.js version (era), libraries đang dùng (auth, state, styling), và scripts có sẵn. Version Next.js 11 hay 12 cho biết ngay những feature nào có hay không.

Bước hai: `next.config.js` để hiểu kiến trúc — có custom webpack không, có redirects gì, image domains từ đâu, có experimental feature nào đang dùng không. Config này là "bảo tàng" của các quyết định kỹ thuật qua thời gian.

Bước ba: `pages/_app.js` để hiểu global state management (Redux, Zustand), auth provider, CSS solution. Tôi đặc biệt check xem có `MyApp.getInitialProps` không vì nó ảnh hưởng toàn bộ performance.

Bước bốn: Map data fetching strategy bằng `grep`: tìm tất cả files dùng `getServerSideProps`, `getStaticProps`, `revalidate`, `getInitialProps`. Vẽ bản đồ xem mỗi route dùng strategy gì.

Bước năm: Đọc một trang phức tạp nhất (thường là trang product detail hay dashboard) để hiểu coding style và patterns thực tế.

Sau đó tôi có thể bắt đầu làm task, không cần đợi ai explain.

</details>

**Senior:**

**Q4: Bạn được assign vào một Next.js 11 project với 150,000 dòng code. Làm sao bạn đánh giá "technical debt" và đề xuất roadmap cải thiện trong 1 tuần đầu?**

<details>
<summary>Đáp án</summary>

Tôi không đọc hết 150,000 dòng code. Tôi đánh giá theo 7 dimensions, mỗi dimension có thể kiểm tra tự động:

Một: TypeScript coverage — `find . -name "*.ts" -o -name "*.tsx" | wc -l` so với tổng số file. Dưới 30% là red flag.

Hai: Data fetching consistency — `grep -rl "getInitialProps" pages/` và đặc biệt check xem _app.js có getInitialProps không. Đây là impact lớn nhất về performance.

Ba: Bundle size — Chạy `ANALYZE=true npm run build`. Chunk nào trên 500KB là vấn đề.

Bốn: Security — `npm audit` và check xem có API routes không có authentication không.

Năm: Image optimization — `grep -rn "<img " pages/ components/` để đếm raw img tags. Nhiều = LCP score kém.

Sáu: Error handling — Check xem có `_error.js`, `404.js` không. Check xem có monitoring tool (Sentry) không.

Bảy: Dependency health — `npm outdated` để thấy độ lag so với latest versions.

Sau 1 tuần, tôi sẽ có bảng điểm cho từng dimension và đề xuất roadmap theo hai horizon: quick wins trong 2 tuần (parallel fetch, missing error pages, npm audit fix) và structural improvements trong 3-6 tháng (migrate getInitialProps, TypeScript coverage, App Router evaluation).

Quan trọng: Tôi KHÔNG đề xuất refactor toàn bộ sang App Router ngay mà cần evaluation kỹ về business case, team bandwidth, và risk.

</details>

**Q5: Bạn thấy project có cả `react-query` lẫn `swr` trong dependencies. Đây có phải vấn đề không? Bạn xử lý thế nào?**

<details>
<summary>Đáp án</summary>

Đây là technical debt ở dạng "inconsistency" — không phải bug hay security issue, nhưng tốn cost dài hạn: bundle size tăng (2 thư viện làm cùng việc), developer experience kém (team phải học cả hai, không biết nên dùng cái nào), code review khó (mỗi người dùng convention khác nhau).

Trước khi fix, tôi cần hiểu tại sao lại có cả hai. Thường có một trong hai lý do: (1) project bắt đầu với swr, sau có người join dùng react-query và không được review kỹ; (2) có migration đang diễn ra nhưng chưa hoàn thành.

`git log --all --full-history --oneline -- "**/package.json"` và check commit messages sẽ cho biết lý do.

Giải pháp: Chọn một cái dựa trên usage hiện tại. Nếu swr được dùng nhiều hơn và team quen hơn → standardize sang swr và migrate dần các chỗ dùng react-query. Ngược lại thì migrate sang react-query (đặc biệt nếu muốn dùng @tanstack/react-query v5 với features mới hơn).

Migration không cần làm trong một PR. Dùng approach "strangler fig": mỗi khi touch một component, migrate nó sang standard library. Sau 2-3 tháng, project sẽ chỉ còn một library.

</details>

**Bài 1 (Basic):** Đọc thông tin project dưới đây và trả lời các câu hỏi:

```json
{
  "dependencies": {
    "next": "10.2.3",
    "react": "17.0.1",
    "next-auth": "3.29.0",
    "styled-components": "5.2.3",
    "redux": "4.1.0",
    "axios": "0.21.4"
  }
}
```

```javascript
// pages/_app.js
import '../styles/globals.css';
import { Provider } from 'react-redux';
import { store } from '../store';

function MyApp({ Component, pageProps }) {
  return (
    <Provider store={store}>
      <Component {...pageProps} />
    </Provider>
  );
}

MyApp.getInitialProps = async (appContext) => {
  const { data: user } = await axios.get('/api/auth/me');
  return { pageProps: { user } };
};

export default MyApp;
```

Câu hỏi: (a) Project này thuộc era nào? (b) Có bao nhiêu vấn đề trong code trên? Liệt kê và giải thích.

<details>
<summary>Đáp án</summary>

(a) Era: Next.js 10, năm 2020-2021. Dấu hiệu: next@10.2.3, next-auth@3 (v3 là NextAuth pre-v4), React 17 (pre-concurrent mode). Đây là project khá cũ, khoảng 3-4 năm tuổi tính đến 2024.

(b) Các vấn đề:

Vấn đề 1 - Nghiêm trọng nhất: `MyApp.getInitialProps` trong `_app.js` disable Automatic Static Optimization cho TẤT CẢ trang. Kể cả trang About hay FAQ đơn giản cũng bị force sang SSR. Performance impact toàn bộ app.

Vấn đề 2: `MyApp.getInitialProps` dùng `axios.get('/api/auth/me')` với URL tương đối. URL tương đối chỉ hoạt động ở client. Khi chạy ở server (SSR), `/api/auth/me` sẽ fail vì không có hostname. Đây là bug tiềm ẩn nếu code chạy ở server.

Vấn đề 3: Fetch user auth trong `getInitialProps` của _app thay vì dùng NextAuth session. next-auth@3 đã có `getSession()` — không cần custom fetch user mỗi request.

Vấn đề 4: `nextAuth@3` là version rất cũ. v3 dùng `getSession()` và import từ `next-auth/client`, không phải `next-auth/react`. Nếu team muốn upgrade lên v4, sẽ có breaking changes.

Vấn đề 5 - Nhỏ hơn: Không có Layout component bao ngoài. Header/Footer sẽ phải import trong từng trang hoặc sẽ bị tắt đi khi navigate. Thường nên có `<Layout><Component {...pageProps} /></Layout>`.

</details>

Lệnh bash để mapping nhanh:

Bản đồ data fetching:

- ISR: blog/[slug].js (revalidate: 60)
- Legacy (getInitialProps): products/[id].js — cần migrate
- SSR: index.js, about.js, blog/index.js, products/index.js, dashboard/index.js, dashboard/profile.js

Hai improvement có impact cao nhất:

Improvement 1 — Migrate products/[id].js từ getInitialProps sang getStaticProps + ISR. Product page là high-traffic page. Với ISR revalidate: 300, page được serve từ cache (~10ms) thay vì render mỗi request (~300-500ms). Impact lớn nhất về performance.

Improvement 2 — Chuyển about.js từ SSR sang SSG thuần. Company info không thay đổi thường xuyên. `getStaticProps` không có `revalidate` sẽ generate một lần khi build và serve tĩnh mãi mãi. Nếu cần update, có thể trigger rebuild hoặc thêm `revalidate: 86400`. Giải phóng server resource không cần thiết.

Improvement thứ 3 có thể xem xét: blog/index.js và products/index.js. Listing pages có thể chuyển sang ISR với revalidate thấp (60-300 giây) nếu không cần query params cho filtering/pagination. Nếu cần query params, giữ SSR nhưng thêm cache headers phù hợp.

**Bài 3 (Advanced):** Phân tích `next.config.js` dưới đây và viết báo cáo ngắn gọn: (a) project này ở Next.js era nào? (b) config nào đã deprecated hoặc có vấn đề? (c) 3 cải thiện cụ thể bạn đề xuất?

```javascript
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});

module.exports = withBundleAnalyzer({
  target: 'serverless',
  reactStrictMode: false,
  swcMinify: false,

  images: {
    domains: ['cdn.example.com', 'lh3.googleusercontent.com'],
    formats: ['image/webp'],
  },

  env: {
    API_SECRET_KEY: process.env.API_SECRET_KEY,
    STRIPE_PUBLIC_KEY: process.env.STRIPE_PUBLIC_KEY,
  },

  experimental: {
    concurrentFeatures: true,
    serverComponents: false,
  },

  webpack: (config, { buildId, dev, isServer, defaultLoaders, webpack }) => {
    config.module.rules.push(
      { test: /\.svg$/, use: ['@svgr/webpack'] },
      { test: /\.md$/, use: 'raw-loader' },
      { test: /\.pdf$/, use: 'file-loader' }
    );
    config.plugins.push(new webpack.ProvidePlugin({ $: 'jquery' }));
    return config;
  },

  async rewrites() {
    return [
      { source: '/api/v1/:path*', destination: 'https://api.internal.company.com/:path*' },
    ];
  },
});
```

<details>
<summary>Đáp án</summary>

(a) Era phân tích:

`target: 'serverless'` chỉ tồn tại trong Next.js 9, 10, 11 — bị remove hoàn toàn trong Next.js 12. `experimental.concurrentFeatures` là config thử nghiệm từ Next.js 11-12. `swcMinify: false` cho thấy Next.js 12 (swcMinify mới có trong 12, default true từ 13). Kết luận: Project đang chạy Next.js 11 hoặc đầu 12, tức khoảng 2021-2022. Dấu hiệu `bundle-analyzer` setup cho thấy team có ý thức về performance.

(b) Deprecated và vấn đề:

`target: 'serverless'` — Đã bị remove từ Next.js 12. Nếu project đang trên Next.js 12+, config này bị ignore hoặc có thể gây lỗi khi build. Cần xóa ngay.

`images.domains` — Deprecated từ Next.js 12.3, thay bằng `images.remotePatterns` với pattern matching linh hoạt hơn. Nên migrate.

`env.API_SECRET_KEY` — NGHIÊM TRỌNG. `env` trong next.config.js expose biến ra client-side bundle. API_SECRET_KEY là secret — không được expose ra client. Đây là security vulnerability. Biến secret chỉ nên dùng ở server, không cần trong `env` config.

`env.STRIPE_PUBLIC_KEY` — Ít nghiêm trọng hơn nhưng vẫn sai convention. Public key của Stripe nên đặt tên `NEXT_PUBLIC_STRIPE_PUBLIC_KEY` trong .env file để rõ ràng đây là client-safe. Việc đặt trong `env` config là redundant và confusing.

`swcMinify: false` — Tắt SWC minifier, quay về Terser chậm hơn. Không có lý do tốt để tắt trừ khi có bug cụ thể với SWC. Nên bật lại.

`config.plugins.push(new webpack.ProvidePlugin({ $: 'jquery' }))` — jQuery trong một Next.js project hiện đại là anti-pattern nặng. jQuery thêm ~87KB vào bundle, xung đột với React DOM manipulation. Cần tìm tại sao cần jQuery và thay thế.

(c) Ba cải thiện đề xuất theo thứ tự ưu tiên:

Ưu tiên 1 — Fix security: Xóa `API_SECRET_KEY` khỏi `env` config. Biến này chỉ cần ở server-side và có thể access trực tiếp qua `process.env.API_SECRET_KEY` trong API routes và server functions mà không cần config ở next.config.js.

Ưu tiên 2 — Migrate deprecated config: Xóa `target: 'serverless'`, đổi `images.domains` sang `images.remotePatterns`. Đây là prerequisite trước khi upgrade Next.js version.

Ưu tiên 3 — Investigate và loại bỏ jQuery: Tìm tất cả `$()` usage trong codebase và replace bằng vanilla JS hoặc React alternatives. Đây là technical debt lớn cho bundle size và maintainability. Sau khi xóa jQuery, cũng có thể xóa `ProvidePlugin` khỏi webpack config, làm cho config đơn giản hơn.

</details>

## 13. Summary

Legacy Project Anatomy là kỹ năng **đọc**, không phải **viết**. Và đọc code người khác là việc bạn làm 80% thời gian trong một công ty thực tế.

**Framework đọc codebase (theo thứ tự):**

Bắt đầu từ `package.json` để xác định era và tech stack. Sau đó `next.config.js` để hiểu kiến trúc và lịch sử quyết định. Tiếp theo `pages/_app.js` — DNA của project, tiết lộ auth, state management, styling. Rồi `pages/_document.js` nếu project dùng CSS-in-JS. Cuối cùng là map data fetching bằng grep trước khi đọc từng page.

**5 dấu hiệu của legacy project "nguy hiểm":**

Một là `getInitialProps` trong `_app.js` — performance bomb ảnh hưởng toàn bộ app. Hai là `target: 'serverless'` trong `next.config.js` của Next.js 12+ — config zombie. Ba là secrets trong `env` config — security hole. Bốn là custom `server.js` — vô hiệu hóa nhiều Next.js features. Năm là không có error monitoring — blind khi production sự cố.

**Tư duy đúng khi join legacy project:**

Tuần đầu không touch gì, chỉ understand. Không đề xuất "viết lại từ đầu" — đó là thinking của người thiếu kinh nghiệm. Đề xuất roadmap có horizon rõ ràng: quick wins, mid-term improvements, long-term modernization.

**Sang bài tiếp theo (M2.10):** Bạn đã có đủ nền tảng để hiểu Legacy Next.js. M2.10 sẽ đi vào chiến lược Migration cụ thể — làm sao chuyển từ Pages Router sang App Router mà không downtime, không phá codebase hiện tại.
