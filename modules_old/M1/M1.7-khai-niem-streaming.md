# M1.7 — Khái niệm Streaming: Progressive HTML, Hydration có chọn lọc

## 1. Lesson Goal

Bài M1.6 đã trả lời "Hydration làm gì sau khi HTML đã tới browser". Bài này trả lời câu hỏi đứng **trước** nó: "HTML tới browser **như thế nào**, theo từng phần hay tất cả một lúc?"

Sau bài này bạn phải trả lời được:

- Tại sao `renderToString()` kiểu cũ luôn bị chặn bởi **component chậm nhất** trong cây, dù 95% trang đã sẵn sàng?
- "Shell" trong streaming SSR là gì, và tại sao TTFB của trang App Router phụ thuộc vào shell, **không phải** vào toàn bộ trang?
- Cơ chế "out-of-order streaming" — HTML gửi sau lại được chèn đúng vị trí — hoạt động trên nguyên lý gì?
- Tại sao đặt `<Suspense>` sai vị trí (bọc cả trang) khiến bạn **tưởng** đã dùng streaming nhưng thực ra không được lợi gì?

Bài này dùng lại trực tiếp Suspense (M1.4) và Selective Hydration (M1.6) — Streaming và Selective Hydration là **hai cơ chế độc lập nhưng luôn đi cùng nhau**, dùng chung đơn vị: ranh giới `<Suspense>`.

## 2. Evolution Context

**Pain Point gốc:** SSR truyền thống coi việc render là một **giao dịch tất-cả-hoặc-không-gì** (all-or-nothing transaction). Server phải có **toàn bộ** dữ liệu và render xong **toàn bộ** cây thành một HTML string hoàn chỉnh, rồi mới gửi response. Nếu trang cần 5 nguồn dữ liệu và 1 trong số đó (ví dụ gọi API gợi ý sản phẩm của bên thứ ba) mất 2 giây, thì **toàn bộ trang** — kể cả phần header, tên sản phẩm, nút mua hàng đã sẵn sàng từ rất sớm — cũng phải chờ đủ 2 giây mới được gửi đi.

**Cách cũ:** Pages Router (và SSR React nói chung trước React 18) dùng mô hình hai pha cứng:

```
Pha 1: Fetch TẤT CẢ data cần thiết (await Promise.all hoặc tuần tự)
            ↓ (đợi xong toàn bộ)
Pha 2: renderToString(<App/>) → một HTML string hoàn chỉnh
            ↓
        Gửi response (status 200 + toàn bộ HTML trong một lần)
```

```tsx
// pages/product/[id].tsx — Pages Router, mô hình "đợi hết rồi render"
export async function getServerSideProps({ params }) {
  // ❌ Nếu getRecommendations() chậm (API bên thứ 3), TOÀN BỘ request
  // bị giữ lại ở đây — chưa có byte nào được gửi cho browser
  const [product, reviews, recommendations] = await Promise.all([
    getProduct(params.id),
    getReviews(params.id),
    getRecommendations(params.id), // chậm: 2000ms
  ]);
  return { props: { product, reviews, recommendations } };
}
```

**Tại sao thất bại:** TTFB (Time To First Byte) của toàn trang bị kéo dài bằng **đúng** thời gian của tác vụ chậm nhất, bất kể phần còn lại nhanh thế nào. Đây là một dạng "waterfall" ở tầng network response, tương tự waterfall data fetching đã học ở M5.6 — nhưng lần này nó chặn cả người dùng nhìn thấy **bất kỳ nội dung gì**, không chỉ chặn một phần dữ liệu.

**Cách mới — Streaming SSR (React 18):** Server không cần đợi xong toàn bộ cây mới gửi response. Nó gửi response theo **nhiều chunk HTML kế tiếp nhau** (HTTP chunked transfer encoding), bắt đầu bằng phần đã render xong (gọi là **shell**), kèm theo placeholder (fallback) cho phần đang chờ dữ liệu; khi phần chờ đó xong, server gửi tiếp một chunk mới chứa HTML thật, kèm một đoạn script nhỏ để "tráo" placeholder bằng nội dung thật ngay tại đúng vị trí trong DOM — không cần reload, không cần `document.write`.

```
Chunk 1 (gửi ngay): <html><head>...</head><body>
                       <Header/> <ProductInfo/> <BuyButton/>
                       <div id="S:1">⏳ Đang tải đánh giá...</div>
                       <div id="S:2">⏳ Đang tải gợi ý...</div>

Chunk 2 (gửi sau ~300ms, khi Reviews data sẵn sàng):
                     <div hidden id="R:1"><ReviewList .../></div>
                     <script>/* tráo nội dung vào #S:1 */</script>

Chunk 3 (gửi sau ~2000ms, khi Recommendations sẵn sàng):
                     <div hidden id="R:2"><Recommendations .../></div>
                     <script>/* tráo nội dung vào #S:2 */</script>
                     </body></html>
```

## 3. Mental Model

Hãy hình dung server không còn là một "máy in" phải in xong toàn bộ tài liệu rồi mới đưa cho bạn, mà là một **dây chuyền chuyển phát theo lô (batch delivery)**: gói hàng nào đóng xong trước, giao trước; gói nào chưa đóng xong, gửi tạm một tấm bưu thiếp ghi "đang chuẩn bị, sẽ giao sau" — và khi gói thật tới, người giao hàng tự thay tấm bưu thiếp đó bằng gói hàng thật, đặt đúng vào vị trí đã hẹn trước.

Hệ quả tất yếu của mental model này:

- **"Shell"** = phần hàng đã đóng gói xong ngay từ đầu — không phụ thuộc Suspense nào đang treo. Shell quyết định TTFB. Nếu shell nhỏ và nhanh, TTFB nhanh, **bất kể** phần còn lại của trang chậm thế nào.
- **Ranh giới `<Suspense>`** = nơi bạn **chủ động khai báo** "phần này được phép đóng gói trễ, gửi placeholder tạm trước". Không có Suspense = không có điểm chia gói = quay lại mô hình "đợi hết rồi gửi một lần".
- Cơ chế tráo nội dung (out-of-order streaming) chỉ hoạt động vì mỗi placeholder có một **ID cố định trước**, và HTML thật gửi sau **mang đúng ID đó** để biết "tráo vào đâu" — y như tấm bưu thiếp ghi sẵn mã đơn hàng.

## 4. Core Concepts

**4.1. API React 18 đứng sau Streaming**

| API | Runtime | Next.js dùng khi nào |
|---|---|---|
| `renderToString()` | Bất kỳ | Legacy, blocking, không streaming (Pages Router/React ≤17 mindset) |
| `renderToPipeableStream()` | Node.js Streams | Next.js server chạy trên Node.js runtime |
| `renderToReadableStream()` | Web Streams API | Next.js server chạy trên Edge runtime |

Next.js trừu tượng hóa hoàn toàn hai API này — bạn không gọi trực tiếp, nhưng hiểu chúng tồn tại giúp bạn biết **tại sao** một số hành vi (như callback `onShellReady`/`onAllReady`) ảnh hưởng tới cách trang của bạn stream.

**4.2. "Shell" — đơn vị quyết định TTFB**

Shell = toàn bộ phần cây **không** nằm trong một Suspense boundary đang chờ promise resolve. React render shell trước, gọi `onShellReady`, và **bắt đầu gửi response ngay tại đó** — không đợi bất kỳ Suspense con nào.

```tsx
// app/product/[id]/page.tsx
import { Suspense } from 'react';

export default async function ProductPage({ params }) {
  // Đây CHẠY TRONG shell — nếu getProduct() chậm, TTFB vẫn bị chặn,
  // vì nó không nằm trong Suspense nào cả.
  const product = await getProduct(params.id);

  return (
    <>
      <Header />
      <ProductInfo product={product} />
      <BuyButton productId={product.id} />

      {/* Phần NGOÀI shell — được phép trễ */}
      <Suspense fallback={<ReviewsSkeleton />}>
        <Reviews productId={product.id} />
      </Suspense>
      <Suspense fallback={<RecsSkeleton />}>
        <Recommendations productId={product.id} />
      </Suspense>
    </>
  );
}
```

> ⚠️ Điểm hay bị hiểu nhầm nhất: **mọi `await` xảy ra ngoài một Suspense boundary đều nằm trong shell**, kể cả khi nó async. Muốn một phần dữ liệu chậm **không** chặn TTFB, bạn phải tách nó ra một Server Component riêng và bọc bằng `<Suspense>` — không phải chỉ "dùng `await` là tự động streaming".

**4.3. Out-of-order Streaming — cơ chế tráo nội dung**

Khi một Suspense boundary resolve, React không gửi HTML thật **ngay tại vị trí cũ trong document flow** (vì vị trí đó đã được gửi và browser đã parse xong từ lâu). Thay vào đó, React:

1. Gửi một chunk mới chứa HTML thật, đặt trong một `<template>`/`<div hidden>` **ở cuối** dòng stream hiện tại.
2. Kèm một đoạn `<script>` nhỏ, inline, gọi một hàm runtime nội bộ của React để **di chuyển (re-parent)** nội dung đó từ vị trí cuối stream vào đúng vị trí placeholder ban đầu (đã có sẵn một marker ID).
3. Sau khi nội dung được chèn đúng vị trí, **Selective Hydration** (M1.6) tiếp quản: nếu phần đó là Client Component, nó được hydrate độc lập — không phải chờ hydrate lại từ đầu trang.

Đây chính là điểm giao giữa hai khái niệm: **Streaming quyết định HTML tới lúc nào**, **Selective Hydration quyết định JS gắn vào lúc nào** — chúng dùng chung một ranh giới (`<Suspense>`) nhưng giải quyết hai vấn đề khác nhau.

**4.4. `loading.tsx` — Suspense boundary tự động ở route-segment level**

```
app/
  product/
    [id]/
      page.tsx     ← nội dung chính
      loading.tsx  ← Next.js tự bọc page.tsx trong <Suspense fallback={<Loading/>}>
```

```tsx
// app/product/[id]/loading.tsx
export default function Loading() {
  return <ProductSkeleton />;
}
```

Next.js tự sinh ra:

```tsx
<Suspense fallback={<Loading />}>
  <Page />
</Suspense>
```

Đây là streaming **ở cấp route segment** (toàn bộ page coi như một khối). Muốn streaming **chi tiết hơn trong cùng một trang** (ví dụ Reviews và Recommendations stream độc lập với nhau, không chỉ độc lập với Header), bạn phải tự thêm `<Suspense>` thủ công bên trong `page.tsx` như mục 4.2.

## 5. Legacy Recognition

Dấu hiệu nhận diện code Pages Router mang tư duy "đợi hết rồi render" — không có khái niệm streaming ở tầng ứng dụng:

```tsx
// pages/dashboard.tsx — pattern kinh điển, gom hết fetch vào một getServerSideProps
export async function getServerSideProps() {
  // ❌ 4 nguồn dữ liệu, có thể tốc độ rất khác nhau, nhưng response
  // chỉ được gửi sau khi CẢ 4 xong — không có cách "gửi sớm phần nhanh"
  const stats = await fetchStats();
  const notifications = await fetchNotifications();
  const recentOrders = await fetchRecentOrders();
  const slowAnalyticsWidget = await fetchHeavyAnalytics(); // 3s, third-party

  return { props: { stats, notifications, recentOrders, slowAnalyticsWidget } };
}
```

Giải pháp né tạm trong Pages Router (không phải streaming thật, chỉ là che giấu): fetch phần chậm ở client sau khi mount (`useEffect` + `fetch` từ component), khiến phần đó render rỗng/loading **ở client**, không liên quan gì đến cơ chế server streaming — đây là "Client-side waterfall hóa trang", khác hẳn bản chất với Suspense streaming của App Router.

## 6. Modern Approach

**Nguyên tắc 1 — Giữ shell tối thiểu, chỉ chứa phần thực sự cần cho First Paint có ý nghĩa:**

```tsx
// ✅ Header, navigation, layout tĩnh → trong shell, render ngay
// Mọi phần phụ thuộc fetch chậm/không chắc chắn về tốc độ → bọc Suspense
export default async function DashboardPage() {
  return (
    <DashboardLayout>
      <Suspense fallback={<StatsSkeleton />}>
        <StatsWidget />
      </Suspense>
      <Suspense fallback={<NotificationsSkeleton />}>
        <Notifications />
      </Suspense>
      <Suspense fallback={<AnalyticsSkeleton />}>
        <HeavyAnalyticsWidget /> {/* phần chậm nhất, cô lập riêng */}
      </Suspense>
    </DashboardLayout>
  );
}
```

**Nguyên tắc 2 — Mỗi nguồn dữ liệu có tốc độ khác nhau nên có Suspense riêng, không gộp chung:**

```tsx
// ❌ Gộp chung một Suspense → cả Notifications (nhanh) cũng phải chờ
// HeavyAnalyticsWidget (chậm) xong mới hiển thị
<Suspense fallback={<Skeleton />}>
  <Notifications />
  <HeavyAnalyticsWidget />
</Suspense>
```

```tsx
// ✅ Tách riêng → Notifications hiện ngay, không bị kéo theo phần chậm
<Suspense fallback={<NotificationsSkeleton />}>
  <Notifications />
</Suspense>
<Suspense fallback={<AnalyticsSkeleton />}>
  <HeavyAnalyticsWidget />
</Suspense>
```

**Nguyên tắc 3 — Skeleton phải khớp kích thước nội dung thật, tránh Cumulative Layout Shift (CLS):**

```tsx
// ❌ Skeleton cao 40px nhưng nội dung thật cao 320px → khi tráo, layout
// nhảy mạnh, ảnh hưởng Core Web Vitals (sẽ học chi tiết ở M11.6)
function BadSkeleton() {
  return <div style={{ height: 40 }}>Loading...</div>;
}
```

```tsx
// ✅ Skeleton ước lượng đúng kích thước khung chứa nội dung thật
function GoodSkeleton() {
  return <div style={{ minHeight: 320 }} className="animate-pulse">...</div>;
}
```

## 7. Production Scenario

**Bối cảnh:** Trang chi tiết sản phẩm có 3 nguồn dữ liệu: thông tin sản phẩm (DB nội bộ, ~50ms), reviews (DB nội bộ, ~80ms), gợi ý sản phẩm tương tự (gọi một dịch vụ ML bên thứ ba, trung bình 1.5s, đôi khi timeout tới 5s).

**Vấn đề khi chưa tách Suspense:** Toàn bộ trang chờ dịch vụ ML — TTFB trung bình ~1.6s, đôi khi 5s+ khi dịch vụ ML chậm bất thường. Người dùng nhìn màn hình trắng dù 99% nội dung họ cần (tên sản phẩm, giá, nút mua) đã có sẵn từ rất sớm.

**Sau khi tách Suspense đúng:**

- Shell (Header, ProductInfo, BuyButton, Reviews) gửi trong ~80–100ms — đây là phần quyết định TTFB và là phần quan trọng nhất với conversion (người dùng có thể bấm "Mua ngay" trước khi phần gợi ý kịp tải).
- `<Recommendations>` nằm trong `<Suspense>` riêng, stream vào sau, không ảnh hưởng đến khả năng tương tác của phần quan trọng.
- **Quan trọng:** đặt timeout/fallback hợp lý cho service ML (ví dụ Promise race với timeout 3s, trả về danh sách rút gọn nếu quá hạn) — vì dù streaming giúp UX tốt hơn, người dùng vẫn không nên chờ vô hạn cho một phần không thiết yếu.

## 8. Debugging Scenario

**Hiện tượng:** Đã thêm `<Suspense>` và `loading.tsx`, nhưng đo TTFB không cải thiện — toàn trang vẫn xuất hiện cùng lúc, giống như không có streaming.

**Quy trình debug có hệ thống:**

1. **Kiểm tra vị trí await có nằm ngoài Suspense không.** Lỗi phổ biến nhất: `await getRecommendations()` được gọi ngay trong `page.tsx` (component cha, ngoài Suspense), rồi mới truyền data xuống `<Recommendations data={...}>` qua props. Lúc này promise đã được resolve **trong shell**, Suspense con chỉ nhận data tĩnh — không có gì để "treo" cả, không có streaming xảy ra. Phải để component **bên trong** Suspense tự `await` data của riêng nó.

```tsx
// ❌ Sai vị trí await — Recommendations component chỉ nhận props tĩnh,
// không tự fetch, nên Suspense bên ngoài nó vô nghĩa
export default async function Page() {
  const recs = await getRecommendations(); // chặn TTFB ngay tại đây
  return (
    <Suspense fallback={<Skeleton />}>
      <Recommendations data={recs} /> {/* đã resolve từ trước, vô dụng */}
    </Suspense>
  );
}
```

```tsx
// ✅ Đúng — component bên trong Suspense tự await
export default function Page() {
  return (
    <Suspense fallback={<Skeleton />}>
      <Recommendations /> {/* component này tự fetch + await bên trong */}
    </Suspense>
  );
}

async function Recommendations() {
  const recs = await getRecommendations(); // chặn ở ĐÂY, không chặn shell
  return <RecList items={recs} />;
}
```

2. **Kiểm tra buffering ở tầng hạ tầng (infra).** Nhiều reverse proxy (Nginx với `proxy_buffering on`), một số nền tảng serverless, hoặc middleware nén response không hỗ trợ streaming sẽ **đợi toàn bộ response xong rồi mới forward** — về mặt application code mọi thứ đúng, nhưng người dùng cuối vẫn nhận một cục HTML duy nhất. Kiểm tra bằng cách quan sát network tab: nếu thấy response chỉ có **một** "Receive" timestamp thay vì nhiều đợt nhận dữ liệu rải theo thời gian, nghi ngờ buffering ở hạ tầng trước.

3. **Kiểm tra `export const dynamic = 'force-static'` hoặc cấu hình caching toàn trang** — nếu route được render tĩnh hoàn toàn tại build time, không có request-time streaming nào xảy ra (đây không phải bug, chỉ là một chế độ khác — Full Route Cache, sẽ học ở M7.4).

## 9. Refactoring Scenario

**Trước (Pages Router, gom 3 nguồn dữ liệu vào một `getServerSideProps`):**

```tsx
// pages/product/[id].tsx
export async function getServerSideProps({ params }) {
  const [product, reviews, recommendations] = await Promise.all([
    getProduct(params.id),
    getReviews(params.id),
    getRecommendations(params.id), // chậm
  ]);
  return { props: { product, reviews, recommendations } };
}

function ProductPage({ product, reviews, recommendations }) {
  return (
    <div>
      <ProductInfo product={product} />
      <ReviewList reviews={reviews} />
      <RecommendationList items={recommendations} />
    </div>
  );
}
export default ProductPage;
```

**Sau (App Router, mỗi nguồn dữ liệu là một đơn vị stream độc lập):**

```tsx
// app/product/[id]/page.tsx
import { Suspense } from 'react';

export default async function ProductPage({ params }) {
  // Chỉ phần CHẮC CHẮN nhanh và quan trọng nhất nằm trong shell
  const product = await getProduct(params.id);

  return (
    <div>
      <ProductInfo product={product} />

      <Suspense fallback={<ReviewsSkeleton />}>
        <Reviews productId={params.id} />
      </Suspense>

      <Suspense fallback={<RecsSkeleton />}>
        <Recommendations productId={params.id} />
      </Suspense>
    </div>
  );
}

// Mỗi component tự chịu trách nhiệm fetch riêng, độc lập về tốc độ
async function Reviews({ productId }: { productId: string }) {
  const reviews = await getReviews(productId);
  return <ReviewList reviews={reviews} />;
}

async function Recommendations({ productId }: { productId: string }) {
  const recommendations = await getRecommendations(productId);
  return <RecommendationList items={recommendations} />;
}
```

**Điểm khác biệt cốt lõi:** Ở phiên bản cũ, tốc độ trang = tốc độ của nguồn **chậm nhất** trong 3 nguồn (do `Promise.all` rồi mới render). Ở phiên bản mới, tốc độ **hiển thị phần đầu** = tốc độ của `getProduct` duy nhất; Reviews và Recommendations stream vào theo tốc độ riêng của chúng, không cộng dồn, không kéo nhau xuống.

## 10. Architecture Notes

- **Streaming không tự nhiên tương thích với mọi hạ tầng.** Trước khi thiết kế nhiều Suspense boundary kỳ vọng cải thiện TTFB, cần xác minh runtime/hosting thực sự forward response theo dạng chunked (không buffer toàn bộ) — đặc biệt khi tự host (self-hosting) phía sau reverse proxy hoặc một số nền tảng serverless có giới hạn về streaming response.
- **SEO/crawler:** nhiều framework (bao gồm Next.js) có xử lý đặc biệt cho user-agent của các bot tìm kiếm đã biết — chờ toàn bộ nội dung (kể cả phần trong Suspense) hoàn tất trước khi trả response, để đảm bảo bot nhận được HTML đầy đủ thay vì các placeholder rời rạc. Hệ quả: đừng giả định "mọi request đều được stream" khi đo lường hoặc khi thiết kế nội dung quan trọng cho SEO — luôn đặt nội dung quan trọng cho SEO trong shell, không phụ thuộc vào việc bot có đợi đủ lâu hay không.
- **Lỗi sau khi shell đã gửi không thể đổi HTTP status code.** Nếu shell đã được gửi (status 200), một lỗi xảy ra ở Suspense con chỉ có thể hiển thị qua `error.tsx` boundary tại đúng vị trí đó — không thể "rollback" về trang lỗi 500 cho toàn bộ response. Đây là lý do `error.tsx` (M3.6) tồn tại như một cơ chế bắt buộc đi cùng với mô hình streaming, không phải tính năng tùy chọn.
- **Mối liên kết tới M4.6 (Partial Prerendering) và M7 (Caching):** PPR về bản chất là việc đưa khái niệm "shell" này lên một tầm cao hơn — shell được **prerender tại build time** (giống Full Route Cache), còn các Suspense "hole" được stream tại request time giống cơ chế đã học ở bài này. Hiểu rõ Streaming ở cấp request giúp M4.6 trở thành một bước mở rộng tự nhiên, không phải một khái niệm hoàn toàn mới.

## 11. Interview Questions

**[Junior/Mid] Streaming SSR khác `renderToString()` truyền thống ở điểm nào?**

<details>
<summary>Đáp án</summary>

`renderToString()` phải render xong toàn bộ cây component thành một HTML string hoàn chỉnh rồi mới trả về — nếu một phần chậm, toàn bộ response bị chặn theo đúng thời gian đó. Streaming SSR (qua `renderToPipeableStream`/`renderToReadableStream` trong React 18) gửi HTML theo nhiều chunk: phần đã sẵn sàng (shell) được gửi ngay, phần còn phụ thuộc dữ liệu chậm được gửi sau dưới dạng chunk bổ sung, kèm placeholder tạm trong lúc chờ.
</details>

**[Mid/Senior] "Shell" trong streaming SSR là gì, và nó liên hệ thế nào với TTFB?**

<details>
<summary>Đáp án</summary>

Shell là toàn bộ phần cây component nằm ngoài mọi Suspense boundary đang chờ resolve — đây là phần React render và gửi đi đầu tiên (kích hoạt callback `onShellReady`). TTFB của trang bằng thời gian cần để hoàn thành shell, không phải toàn bộ trang. Vì vậy, muốn TTFB nhanh, cần giữ shell nhỏ, chỉ chứa các phần chắc chắn nhanh, và đẩy mọi phần phụ thuộc dữ liệu không chắc chắn về tốc độ vào trong các Suspense boundary riêng.
</details>

**[Senior] Giải thích cơ chế "out-of-order streaming" — làm sao HTML gửi sau lại hiển thị đúng vị trí ban đầu?**

<details>
<summary>Đáp án</summary>

Khi gửi shell, React để lại các placeholder có ID cố định tại vị trí của mỗi Suspense boundary đang chờ. Khi một boundary resolve, React không gửi HTML thật vào đúng vị trí cũ trong luồng HTML (vị trí đó đã được browser parse xong), mà gửi nó trong một chunk mới ở cuối stream hiện tại, kèm một đoạn script nhỏ thực thi ngay khi nhận được — script này di chuyển (re-parent) nội dung thật từ vị trí cuối vào đúng vị trí placeholder dựa trên ID đã khớp trước. Sau khi chèn xong, nếu phần đó là Client Component, Selective Hydration sẽ gắn tương tác cho nó độc lập.
</details>

**[Senior/Staff] Vì sao đặt một `<Suspense>` duy nhất bọc toàn bộ trang có thể khiến bạn lầm tưởng đã tối ưu streaming, nhưng thực ra không được lợi gì?**

<details>
<summary>Đáp án</summary>

Vì một Suspense boundary chỉ release nội dung khi **toàn bộ** promise bên trong nó resolve — nếu tất cả phần của trang nằm trong cùng một boundary, thời điểm hiển thị vẫn bị quyết định bởi phần **chậm nhất**, giống hệt mô hình "đợi hết rồi render" của Pages Router, chỉ khác là có thêm một fallback loading ban đầu. Lợi ích thật của streaming chỉ xuất hiện khi các phần có tốc độ khác nhau được tách vào các Suspense boundary **riêng biệt**, để phần nhanh không bị phần chậm kéo theo.
</details>

**[Staff/Architect] Tại sao hạ tầng (reverse proxy, CDN, serverless platform) có thể "vô hiệu hóa" lợi ích của streaming dù code hoàn toàn đúng?**

<details>
<summary>Đáp án</summary>

Streaming dựa trên việc server gửi response theo nhiều chunk qua thời gian (HTTP chunked transfer encoding) và client nhận từng chunk ngay khi nó tới. Nếu một lớp trung gian giữa server và browser — ví dụ reverse proxy bật buffering, một số gateway/serverless platform có giới hạn về response streaming, hoặc middleware nén toàn bộ response trước khi gửi — chủ động đợi nhận hết toàn bộ response từ server rồi mới forward tiếp cho client, thì về phía browser, mọi chunk sẽ tới cùng một lúc, xóa bỏ hoàn toàn lợi ích TTFB của streaming, dù application code không có lỗi gì.
</details>

## 12. Exercises

**Bài 1 (Mid):** Trang dashboard dưới đây gọi 2 API có tốc độ rất khác nhau (`getQuickStats` ~50ms, `getHeavyReport` ~4000ms) nhưng được viết theo cách khiến cả trang chờ phần chậm. Hãy chỉ ra lỗi và sửa lại để tách streaming đúng.

```tsx
// app/dashboard/page.tsx
export default async function Dashboard() {
  const stats = await getQuickStats();
  const report = await getHeavyReport();
  return (
    <div>
      <StatsPanel stats={stats} />
      <ReportPanel report={report} />
    </div>
  );
}
```

<details>
<summary>Đáp án</summary>

Lỗi: cả hai `await` đều nằm trong shell (component cha `Dashboard`, không có Suspense nào), nên TTFB bị chặn bởi `getHeavyReport` (4000ms), dù `getQuickStats` chỉ cần 50ms.

```tsx
import { Suspense } from 'react';

export default function Dashboard() {
  return (
    <div>
      <Suspense fallback={<StatsSkeleton />}>
        <Stats />
      </Suspense>
      <Suspense fallback={<ReportSkeleton />}>
        <Report />
      </Suspense>
    </div>
  );
}

async function Stats() {
  const stats = await getQuickStats(); // chỉ chặn boundary này, không chặn shell
  return <StatsPanel stats={stats} />;
}

async function Report() {
  const report = await getHeavyReport(); // tự stream vào sau, không kéo theo Stats
  return <ReportPanel report={report} />;
}
```

Sau khi sửa, `Dashboard` không còn `await` nào trong shell, nên shell trống/render gần như ngay lập tức, và TTFB không còn phụ thuộc vào `getHeavyReport`. Stats hiển thị trong khoảng 50ms, Report stream vào sau, độc lập.
</details>

**Bài 2 (Senior):** Bạn nhận thấy dù đã tách Suspense đúng như trên, đo thực tế trên production (qua CDN) vẫn không thấy nhiều đợt nhận dữ liệu trong Network tab — toàn bộ response tới cùng lúc. Liệt kê 2 hướng kiểm tra hợp lý nhất và giải thích ngắn vì sao.

<details>
<summary>Đáp án</summary>

Hướng 1 — Kiểm tra buffering ở tầng hạ tầng: nếu giữa server Next.js và browser có một reverse proxy hoặc CDN bật chế độ buffer toàn bộ response trước khi forward (ví dụ Nginx `proxy_buffering on`, hoặc một số nền tảng serverless tổng hợp response trước khi trả về), thì dù server đã gửi nhiều chunk đúng cách, lớp trung gian này sẽ ghép lại thành một response duy nhất trước khi tới browser — code ứng dụng không có lỗi, vấn đề nằm ở hạ tầng.

Hướng 2 — Kiểm tra lại vị trí `await`: xác nhận rằng các hàm fetch dữ liệu chậm thực sự được gọi **bên trong** component nằm trong Suspense (như `Stats`/`Report` ở bài 1), không bị gọi nhầm ở component cha rồi truyền qua props — nếu promise đã resolve trước khi tới Suspense, boundary đó không có gì để "treo" và sẽ không tạo ra chunk riêng.
</details>

## 13. Summary

- Streaming SSR thay mô hình "render xong toàn bộ rồi gửi một lần" bằng việc gửi HTML theo nhiều chunk: shell gửi ngay, các phần còn lại (mỗi phần nằm trong một `<Suspense>`) gửi tiếp khi sẵn sàng.
- "Shell" = phần cây không nằm trong Suspense đang chờ — quyết định TTFB. Mọi `await` ngoài Suspense đều nằm trong shell và chặn TTFB như cũ.
- Out-of-order streaming dùng placeholder ID + script nhỏ để chèn nội dung thật vào đúng vị trí, dù nội dung đó được gửi ở chunk sau.
- `loading.tsx` tạo Suspense ở cấp route segment; muốn streaming chi tiết hơn trong cùng trang, phải tự thêm `<Suspense>` quanh từng phần có tốc độ khác nhau.
- Streaming và Selective Hydration (M1.6) dùng chung ranh giới `<Suspense>` nhưng giải quyết hai vấn đề khác nhau: streaming quyết định **HTML tới lúc nào**, hydration quyết định **JS gắn vào lúc nào**.
- Lợi ích thực tế phụ thuộc vào hạ tầng (proxy/CDN không buffer response) và vào việc đặt Suspense đúng chỗ — bọc cả trang trong một Suspense duy nhất không mang lại lợi ích gì so với mô hình cũ.
