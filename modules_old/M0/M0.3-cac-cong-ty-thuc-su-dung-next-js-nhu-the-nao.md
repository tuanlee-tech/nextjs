# M0.3 — Các Công Ty Thực Sự Dùng Next.js Như Thế Nào



## 1. Lesson Goal

Sau bài này, bạn có thể:

- **Nhìn vào job description** và biết ngay công ty đó dùng Next.js theo kiến trúc nào
- **Hiểu tại sao** hai công ty cùng dùng Next.js nhưng code lại khác nhau hoàn toàn
- **Không bị shock** khi codebase thực tế khác xa những gì tutorial dạy
- **Đặt câu hỏi đúng** trong technical interview: về infrastructure, team structure, deployment
- **Đưa ra quyết định kiến trúc** khi được hỏi "Chúng ta nên thiết kế hệ thống này như thế nào?"

> **Vấn đề cốt lõi:** Tutorial dạy bạn xây một thứ cụ thể. Production là vô số thứ cụ thể khác nhau. Bài này là bản đồ của khoảng cách đó.



## 2. Evolution Context

### Tại sao Next.js được chọn — và chọn vì lý do gì

Không phải công ty nào chọn Next.js đều vì cùng một lý do. Đây là điều quan trọng nhất để hiểu.

```
Công ty A (E-commerce):
  Vấn đề: Google không index được SPA React
  Giải pháp: Next.js vì SSR/SSG → SEO tốt
  Họ dùng: 80% SSG, 20% SSR, App Router optional

Công ty B (SaaS Dashboard):
  Vấn đề: Cần build admin panel nhanh, không cần SEO
  Giải pháp: Next.js vì routing, API routes, DX tốt
  Họ dùng: 90% CSR (Client Side Rendering), Next.js gần như chỉ là router

Công ty C (Startup):
  Vấn đề: Team 3 người, cần full-stack nhanh
  Giải pháp: Next.js vì không cần backend riêng ban đầu
  Họ dùng: Server Actions, API Routes, Prisma, Vercel

Công ty D (Enterprise):
  Vấn đề: Legacy .NET backend, cần modern frontend
  Giải pháp: Next.js là "frontend layer" trên existing API
  Họ dùng: Next.js chỉ fetch từ .NET API, không có API Routes
```

**Lesson:** Khi thấy Next.js trong job requirement, câu hỏi phải hỏi không phải "Họ dùng Next.js", mà là **"Họ dùng Next.js để làm gì?"**

### Tiến hóa của việc sử dụng Next.js trong industry

```
2019–2020: "SSR cho React" — Companies chọn Next.js vì SSR
  → Hầu hết dùng Pages Router + getServerSideProps
  → App Router chưa tồn tại

2021–2022: "Framework cho mọi thứ" — Companies dùng Next.js rộng hơn
  → API Routes cho simple backend
  → ISR cho content sites
  → Vercel deployment boom

2023–2024: "App Router revolution" — Kỳ vọng cao, adoption chậm
  → Nhiều công ty vẫn ở Pages Router (stable, production-proven)
  → Startup mới → App Router
  → Company cũ → "Sẽ migrate sau"

2025–hiện tại: "Pragmatic adoption" — Reality check
  → App Router phổ biến cho "greenfield" (dự án mới xây dựng từ đầu)
  → Nhiều codebase vẫn Pages Router — không migrate vì cost > benefit
  → Next.js 15/16 features dành cho những ai thực sự cần
```



## 3. Mental Model — Next.js Là Infrastructure (Cơ sở hạ tầng), Không Phải Sản Phẩm

**Sai lầm phổ biến nhất của developer mới:** Nghĩ rằng "dùng Next.js" có nghĩa là xây thứ gì đó trông giống tutorial.

**Thực tế:** Next.js là một **infrastructure layer**. Sản phẩm thực sự là business logic bên trên.

```
TUTORIAL MINDSET:
  Next.js App
  ├── pages (hoặc app)
  ├── components
  ├── styles
  └── public

PRODUCTION MINDSET:
  Business Problem
  └── Giải pháp kỹ thuật
      ├── Next.js (một phần của solution)
      ├── Database (Postgres, MySQL, MongoDB...)
      ├── Auth (NextAuth, Auth0, Clerk, custom...)
      ├── CDN (Cloudflare, CloudFront...)
      ├── Monitoring (Datadog, Sentry...)
      ├── CI/CD (GitHub Actions, Jenkins...)
      └── Infrastructure (Vercel, Docker, K8s...)
```

**Analogy:** Next.js giống như khung nhà. Tutorial dạy bạn dựng khung. Công ty thực tế phải lo: điện, nước, nội thất, hệ thống PCCC, an ninh, bảo trì — và tất cả những thứ đó có ảnh hưởng đến cách dựng khung.



## 4. Core Concepts — 5 Kiến Trúc Thực Tế

### 📐 Pattern A — Full-Stack Monolith (Startup / SMB - Small and Medium-sized Business) : Doanh nghiệp nhỏ và vừa

**Phổ biến ở:** Startup giai đoạn đầu, freelance project, MVP, công ty <20 người

**Đặc điểm:** Next.js làm TẤT CẢ — frontend, backend logic, database access, auth

```
startup-app/
├── app/
│   ├── (auth)/
│   │   ├── login/page.tsx
│   │   └── register/page.tsx
│   ├── (dashboard)/
│   │   ├── layout.tsx
│   │   └── dashboard/page.tsx
│   └── api/           ← Backend sống trong Next.js
│       ├── auth/
│       │   └── [...nextauth]/route.ts
│       └── webhooks/
│           └── stripe/route.ts
├── actions/           ← Server Actions thay thế nhiều API endpoints
│   ├── user.ts
│   ├── product.ts
│   └── order.ts
├── lib/
│   ├── db.ts          ← Prisma client
│   ├── auth.ts        ← Auth config
│   └── stripe.ts      ← Payment
└── prisma/

```

```typescript
// actions/product.ts — Server Action trong Pattern A
// Database access TRỰC TIẾP từ Next.js layer
'use server';

import { db } from '@/lib/db';
import { auth } from '@/lib/auth';
import { revalidatePath } from 'next/cache';
import { z } from 'zod';

const CreateProductSchema = z.object({
  name: z.string().min(1).max(100),
  price: z.number().positive(),
  description: z.string().optional(),
});

export async function createProduct(formData: FormData) {
  // Auth check thẳng trong Server Action
  const session = await auth();
  if (!session?.user) throw new Error('Unauthorized');

  const input = CreateProductSchema.parse({
    name: formData.get('name'),
    price: Number(formData.get('price')),
    description: formData.get('description'),
  });

  // Prisma query thẳng — không qua API layer nào
  const product = await db.product.create({
    data: {
      ...input,
      createdById: session.user.id,
    },
  });

  revalidatePath('/dashboard/products');
  return product;
}
```

**Tại sao Pattern A tồn tại:**

```
PRO:
  + Đơn giản nhất — một repo, một deploy, một team
  + Không có network latency giữa frontend và backend (cùng process)
  + Type safety end-to-end tự nhiên (Prisma → TypeScript → Component)
  + Vercel deployment trivial
  + Iteration speed cực nhanh ở giai đoạn đầu

CON:
  - Không scale tốt khi team lớn (ai cũng có access vào DB logic)
  - Database connection management phức tạp trên serverless
  - Không thể reuse API cho mobile app (nếu cần)
  - Vendor lock-in vào Vercel ecosystem (nếu không cẩn thận)
  - Một Next.js deploy = một monolith to tướng
```

**Signal trong job posting:**
> *"Next.js, Prisma, PostgreSQL, Vercel, Server Actions"*

→ Pattern A. Team nhỏ, move fast, full-stack engineers.

### 📐 Pattern B — Frontend Layer (Decoupled)

**Phổ biến ở:** Công ty đã có backend (Java, .NET, Python, Go), team frontend riêng

**Đặc điểm:** Next.js chỉ là **presentation layer**. Không có API Routes (hoặc rất ít). Không có DB access. Tất cả data từ external API.

```

├── app/
│   ├── layout.tsx
│   ├── page.tsx
│   └── products/
│       └── [id]/page.tsx
├── lib/
│   ├── api/
│   │   ├── client.ts        ← Configured API client (axios/fetch wrapper)
│   │   ├── products.ts      ← Product API calls
│   │   └── users.ts         ← User API calls
│   └── types/
│       ├── product.ts       ← Types mirror backend DTOs
│       └── user.ts
└── next.config.ts
# KHÔNG CÓ:
# - /prisma
# - /actions (hoặc rất ít)
# - /app/api (hoặc chỉ proxy routes)
```

```typescript
// lib/api/client.ts — Configured API client
// Đây là pattern rất phổ biến ở enterprise

const BASE_URL = process.env.API_URL; // Internal backend URL

// Server-side API calls (trong Server Components)
export async function serverFetch<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  const response = await fetch(`${BASE_URL}${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      // Service-to-service auth — không phải user token
      'X-Service-Key': process.env.SERVICE_KEY!,
      ...options?.headers,
    },
  });

  if (!response.ok) {
    throw new Error(`API Error: ${response.status} ${endpoint}`);
  }

  return response.json();
}

// Client-side API calls (trong Client Components)
export async function clientFetch<T>(
  endpoint: string,
  options?: RequestInit
): Promise<T> {
  // Next.js API route đóng vai trò PROXY
  // Không expose backend URL ra client
  const response = await fetch(`/api/proxy${endpoint}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      ...options?.headers,
    },
    credentials: 'include', // Gửi session cookie
  });

  if (!response.ok) throw new Error(`API Error: ${response.status}`);
  return response.json();
}
```

```typescript
// app/api/proxy/[...path]/route.ts
// Proxy pattern — Next.js chỉ forward request
// Pattern B thường có proxy như thế này
import { NextRequest, NextResponse } from 'next/server';
import { auth } from '@/lib/auth';

export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ path: string[] }> }
) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  // Forward to backend với user token
  const { path } = await params;
  const endpoint = path.join('/');
  const url = `${process.env.API_URL}/${endpoint}${request.nextUrl.search}`;

  const response = await fetch(url, {
    headers: {
      Authorization: `Bearer ${session.accessToken}`, // Forward user token
      'X-User-Id': session.user.id,
    },
  });

  const data = await response.json();
  return NextResponse.json(data, { status: response.status });
}

// POST, PUT, DELETE tương tự...
export async function POST(request: NextRequest, { params }: { params: Promise<{ path: string[] }> }) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  const { path } = await params;
  const body = await request.json();

  const response = await fetch(`${process.env.API_URL}/${path.join('/')}`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${session.accessToken}`,
    },
    body: JSON.stringify(body),
  });

  const data = await response.json();
  return NextResponse.json(data, { status: response.status });
}
```

**Tại sao Pattern B tồn tại:**

```
PRO:
  + Backend team có thể làm việc độc lập (Go team, Java team...)
  + Next.js chỉ quan tâm đến UI — không bị lẫn business logic
  + Backend có thể serve nhiều client (web, mobile, partner API)
  + Dễ swap Next.js sang framework khác nếu cần
  + Security tốt hơn: DB credentials không cần ở frontend layer

CON:
  - Thêm một network hop (browser → Next.js → Backend)
  - Type duplication: phải tạo TS types mirror backend DTOs
  - Two team coordination: API contract thay đổi → frontend bị ảnh hưởng
  - Local development phức tạp hơn (cần chạy cả backend)
```

**Signal trong job posting:**
> *"Next.js frontend, REST API consumption, Java/Spring backend (separate team)"*
>
> *"Experience with API integration, OpenAPI/Swagger"*

→ Pattern B. Frontend-only role, backend là black box.

### 📐 Pattern C — BFF (Backend for Frontend)

**Phổ biến ở:** Công ty có microservices, cần aggregation layer, team medium-large

**Đặc điểm:** Next.js đóng vai trò **BFF** — nhận request từ client, gọi nhiều microservices, tổng hợp data, trả về.

```
BFF không phải là "proxy" — BFF có BUSINESS LOGIC:
  - Aggregation: Lấy data từ 3 services, merge thành 1 response
  - Transformation: Convert data format phù hợp với UI cần
  - Orchestration: Gọi service A, dùng kết quả gọi service B
  - Auth gateway: Verify token, enrich request với user context
```

```
bff-app/
├── app/
│   ├── layout.tsx
│   └── [feature]/page.tsx
├── app/api/
│   ├── dashboard/
│   │   └── route.ts      ← Aggregate từ 3 microservices
│   ├── checkout/
│   │   └── route.ts      ← Orchestrate cart + payment + inventory
│   └── search/
│       └── route.ts      ← Query search service + personalization
└── services/              ← Typed clients cho từng microservice
    ├── product-service.ts
    ├── inventory-service.ts
    ├── user-service.ts
    ├── order-service.ts
    └── recommendation-service.ts
```

```typescript
// services/product-service.ts — Typed client cho Product Microservice
const PRODUCT_SERVICE_URL = process.env.PRODUCT_SERVICE_URL;

interface Product {
  id: string;
  name: string;
  price: number;
  categoryId: string;
}

export const productService = {
  async getById(id: string): Promise<Product> {
    const res = await fetch(`${PRODUCT_SERVICE_URL}/products/${id}`, {
      headers: { 'X-Service-Name': 'nextjs-bff' },
      // Service mesh auth (mTLS trong production)
    });
    if (!res.ok) throw new Error(`Product not found: ${id}`);
    return res.json();
  },

  async getMany(ids: string[]): Promise<Product[]> {
    // Batch endpoint — tránh N+1 calls
    const res = await fetch(`${PRODUCT_SERVICE_URL}/products/batch`, {
      method: 'POST',
      body: JSON.stringify({ ids }),
      headers: {
        'Content-Type': 'application/json',
        'X-Service-Name': 'nextjs-bff',
      },
    });
    return res.json();
  },
};
```

```typescript
// app/api/dashboard/route.ts — BFF Aggregation
// Đây là trái tim của Pattern C
import { NextRequest, NextResponse } from 'next/server';
import { auth } from '@/lib/auth';
import { productService } from '@/services/product-service';
import { orderService } from '@/services/order-service';
import { analyticsService } from '@/services/analytics-service';
import { inventoryService } from '@/services/inventory-service';

export async function GET(request: NextRequest) {
  const session = await auth();
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  const userId = session.user.id;

  // BFF PATTERN: Gọi nhiều services PARALLEL
  // Client chỉ cần 1 request thay vì 4 requests
  const [recentOrders, topProducts, analytics, lowStockAlerts] = await Promise.all([
    orderService.getRecent(userId, { limit: 5 }),
    productService.getTopSelling(userId, { limit: 10 }),
    analyticsService.getSummary(userId, { period: '30d' }),
    inventoryService.getLowStockAlerts(userId),
  ]);

  // BFF TRANSFORMATION: Merge và format data phù hợp với UI
  // Client không cần biết data đến từ 4 services khác nhau
  const dashboardData = {
    summary: {
      totalRevenue: analytics.revenue,
      orderCount: analytics.orderCount,
      topProductsCount: topProducts.length,
    },
    recentOrders: recentOrders.map(order => ({
      id: order.id,
      // Flatten nested structure — client nhận format đơn giản hơn
      customerName: `${order.customer.firstName} ${order.customer.lastName}`,
      total: order.lineItems.reduce((sum, item) => sum + item.price * item.quantity, 0),
      status: order.status,
      createdAt: order.createdAt,
    })),
    topProducts,
    alerts: lowStockAlerts.filter(alert => alert.quantity < 10),
    // Thêm computed fields mà không service nào cung cấp
    conversionRate: analytics.orderCount / analytics.visitCount,
  };

  return NextResponse.json(dashboardData);
}
```

**Tại sao Pattern C tồn tại:**

```
PRO:
  + Client chỉ cần 1 request thay vì nhiều requests riêng lẻ
  + Microservices thay đổi nội bộ không ảnh hưởng client
  + Dễ cache ở BFF layer
  + One place để handle cross-cutting concerns: logging, tracing, auth
  + Mobile và web có thể có BFF khác nhau (Mobile BFF vs Web BFF)

CON:
  - Thêm một layer cần maintain
  - BFF có thể trở thành bottleneck
  - Logic aggregation dễ bị trùng lặp nếu nhiều BFF
  - Team cần hiểu cả frontend lẫn service contracts
```

**Signal trong job posting:**
> *"Next.js, microservices architecture, API aggregation, GraphQL (optional)"*
>
> *"Experience with service-to-service communication"*

→ Pattern C. Thường gặp ở tech companies dùng microservices.

### 📐 Pattern D — Content Platform

**Phổ biến ở:** Media, news, blog, documentation site, marketing website

**Đặc điểm:** Tập trung vào **SEO, performance, CDN**. Pages được build trước (SSG/ISR). Content từ CMS.

```
content-platform/
├── app/
│   ├── layout.tsx
│   ├── page.tsx            ← Homepage, ISR 1 hour
│   ├── [category]/
│   │   └── page.tsx        ← Category page, ISR 30 min
│   └── [category]/[slug]/
│       └── page.tsx        ← Article page, ISR 5 min
├── lib/
│   └── cms/
│       ├── contentful.ts   ← Hoặc Sanity, Strapi, Prismic...
│       ├── queries.ts      ← GraphQL/REST queries for CMS
│       └── transforms.ts   ← Transform CMS data → app types
└── next.config.ts
# Đặc trưng: Không có user auth phức tạp
# Đặc trưng: Nhiều generateStaticParams
# Đặc trưng: revalidatePath/revalidateTag từ CMS webhook
```

```typescript
// lib/cms/contentful.ts — Headless CMS integration
import { createClient } from 'contentful';

const client = createClient({
  space: process.env.CONTENTFUL_SPACE_ID!,
  accessToken: process.env.CONTENTFUL_ACCESS_TOKEN!,
  // Preview token khác — dùng cho draft content
  ...(process.env.CONTENTFUL_PREVIEW_MODE && {
    accessToken: process.env.CONTENTFUL_PREVIEW_TOKEN!,
    host: 'preview.contentful.com',
  }),
});

// CMS query được wrap trong typed functions
export async function getArticle(slug: string) {
  const entries = await client.getEntries<ArticleFields>({
    content_type: 'article',
    'fields.slug': slug,
    include: 2, // Depth của linked entries
    limit: 1,
  });

  if (!entries.items.length) return null;
  return transformArticle(entries.items[0]);
}

export async function getAllArticleSlugs(): Promise<string[]> {
  const entries = await client.getEntries<ArticleFields>({
    content_type: 'article',
    select: ['fields.slug'], // Chỉ lấy slug — tối ưu bandwidth
    limit: 1000,
  });

  return entries.items.map(item => item.fields.slug);
}
```

```typescript
// app/[category]/[slug]/page.tsx — Article page, Pattern D
import { getArticle, getAllArticleSlugs } from '@/lib/cms/contentful';
import { notFound } from 'next/navigation';

// ISR: Tái tạo trang sau 5 phút nếu có request
export const revalidate = 300;

// Pre-render top articles tại build time

export async function generateStaticParams() {

  }));

export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const article = await getArticle(slug);
  if (!article) return {};

  return {
    title: `${article.title} | TechNews`,
    description: article.excerpt,
```

```typescript
// middleware.ts — Tenant detection — Trái tim của Pattern E
// Chạy trên EVERY request trước khi đến route handler
import { NextRequest, NextResponse } from 'next/server';

export function middleware(request: NextRequest) {
  const hostname = request.headers.get('host') || '';
  const url = request.nextUrl.clone();

  // Subdomain detection: acme.app.com → tenant: "acme"
  // Hoặc custom domain: acme.com → lookup in DB
  const mainDomain = process.env.NEXT_PUBLIC_ROOT_DOMAIN; // "app.com"

  let tenantSlug: string | null = null;

  if (hostname.endsWith(`.${mainDomain}`)) {
    // Subdomain pattern: acme.app.com
    tenantSlug = hostname.replace(`.${mainDomain}`, '');
  } else if (hostname !== mainDomain && hostname !== `www.${mainDomain}`) {
    // Custom domain: Cần lookup từ DB/cache
    // Thực tế: Cache tenant mapping để tránh DB hit mỗi request
    tenantSlug = getCachedTenantByDomain(hostname);
  }

  if (tenantSlug) {
    // Rewrite URL để route đến tenant-specific handler
    // User thấy URL: acme.app.com/dashboard
    // Next.js nhận: /tenants/acme/dashboard
    url.pathname = `/tenants/${tenantSlug}${url.pathname}`;
    return NextResponse.rewrite(url);
  }

  return NextResponse.next();
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)'],
};
```

**Q1: Tại sao một công ty có thể dùng Next.js nhưng không có file nào trong `/app/api/`?**

Đây là pattern rất phổ biến tại các công ty lớn có team backend riêng, và là lý do tại sao "biết Next.js" không tự động có nghĩa là biết backend development.

**Q2: Developer mô tả project của họ: "Next.js app, PostgreSQL, Prisma, deploy trên Vercel, dùng Server Actions cho form submissions". Đây là pattern gì và trade-offs của nó?**

<details>
<summary>Đáp án</summary>

**Pattern A — Full-Stack Monolith.** Đây là pattern phổ biến nhất ở startup và small team.

**Trade-offs:**

*Ưu điểm:*
- Type safety end-to-end: Schema Prisma → TypeScript → Component
- Không có network hop giữa frontend và database (same process)
- Tốc độ phát triển và thử nghiệm (Iteration speed) rất nhanh
- Deploy đơn giản (trivial) với Vercel

*Nhược điểm:*
- Database connection pooling là vấn đề trên serverless (mỗi Vercel function = new Prisma instance). Cần setup connection pooler (PgBouncer, Supabase pooler, Neon).
- Không thể reuse API cho mobile app nếu dùng Server Actions — phải build REST API riêng sau này
- Scale team khó: Khi team lớn, mọi người có access vào database logic, dễ inconsistent patterns (cách triển khai không nhất quán, làm giảm tính chuẩn hóa của hệ thống)
- Nguy cơ bị phục thuộc (Vendor lock-in) vào Vercel ecosystem nếu không setup để ứng dụng có thể chạy độc lập (standalone output)

</details>

### Mid Level

**Q3: Company có 3 microservices: user-service, product-service, order-service. CTO muốn dùng Next.js. Bạn sẽ suggest kiến trúc gì và tại sao?**

<details>
<summary>Đáp án</summary>

**Pattern C — BFF (Backend for Frontend).**

**Lý do:**

Với 3+ microservices riêng biệt, một client phải gọi nhiều services cho một trang (ví dụ order history cần cả user-service lẫn order-service lẫn product-service). Không có BFF dẫn đến:
- Client phải biết về tất cả service URLs
- Nhiều lượt gọi mạng (network round-trips) từ browser
- Không có nơi tập trung xử lý auth, authorization và user context
- Logic aggregation (gom data) bị đẩy xuống frontend → khó maintain và dễ duplicate

**BFF implementation với Next.js:**

- `/app/api/dashboard/route.ts` → Gọi nhiều services song song bằng `Promise.all()`, sau đó merge và transform response thành format UI cần
- `/services/*.ts` → Typed clients cho từng microservice
- Middleware xử lý auth: verify JWT, extract session, inject user context vào các service calls
- Service-to-service communication dùng **internal URLs (private network/VPC/Kubernetes DNS)** (không expose qua internet)

---

**💡 Senior Insight:** Lưu ý quan trọng khi pitch (thuyết trình / trình bày)

* **BFF cần team riêng để vận hành và maintain**
  BFF không nên được xem là “phần phụ” của frontend hay backend team. Đây là một lớp backend độc lập, có trách nhiệm riêng như một product: quản lý API aggregation, performance, auth flow và integration với nhiều microservices. Nếu không có ownership rõ ràng, BFF rất dễ bị viết chồng chéo, khó scale và khó debug.

* **Cần monitoring và alerting đầy đủ cho BFF (vì đây là single point of failure)**
  BFF là điểm vào duy nhất từ client vào hệ thống backend. Nghĩa là nếu BFF bị lỗi, toàn bộ ứng dụng frontend sẽ không hoạt động dù các microservices phía sau vẫn chạy bình thường. Vì vậy cần theo dõi chặt:

  * latency (p95/p99)
  * error rate
  * traffic / throughput
  * tình trạng các dependency (user-service, order-service, …)
    Đồng thời phải có alert real-time để phát hiện sớm sự cố trước khi người dùng bị ảnh hưởng.

* **Cần Circuit Breaker để tránh một service làm sập toàn bộ page**
  Trong hệ thống microservices, việc một service bị chậm hoặc down là chuyện bình thường. Nếu BFF gọi trực tiếp tất cả services theo kiểu “fail là fail toàn bộ”, chỉ cần 1 service lỗi là cả page bị crash.
  Circuit breaker giúp:

  * phát hiện service đang lỗi hoặc timeout
  * tạm ngưng gọi service đó trong một khoảng thời gian
  * trả fallback data (rỗng, cache, hoặc degraded response)
    → Nhờ đó UI vẫn hiển thị được phần còn lại thay vì trắng trang hoàn toàn.

* **Tránh logic bị “phình to” → BFF không nên biến thành “mini monolith backend”**
  BFF chỉ nên tập trung vào 2 việc chính:

  * API aggregation / composition (gọi nhiều service và merge response)
  * Auth / authorization / user session handling

  **Nó KHÔNG nên chứa:**

  * business logic nghiệp vụ (nếu đã có service riêng)
  * complex data transformation
  * database access trực tiếp
    Khi team lớn lên, nếu BFF quá to, mọi người sẽ “dính” vào nó và khó tách service độc lập sau này.

---

**Q5: "Next.js trên Vercel rất đắt khi scale. Công ty muốn self-host bằng Docker nhưng vẫn muốn giữ ISR và Image Optimization. Điều này khả thi không? Những gì bị mất?"**

<details>
<summary>Đáp án</summary>

**Khả thi — với một số lưu ý quan trọng.**

**Setup cơ bản:**
```typescript
// next.config.ts
const nextConfig: NextConfig = {
  output: 'standalone', // ← BẮT BUỘC cho Docker deployment
};
```

**Những gì hoạt động khi self-host:**
- ISR: Hoạt động, nhưng cache lưu trên local file system của container
- `revalidatePath`/`revalidateTag`: Hoạt động
- Image Optimization (`next/image`): Hoạt động (Next.js có built-in image server)

**Những gì BỊ MẤT hoặc CẦN SETUP THÊM:**
1. **Edge Runtime functions** — Vercel Edge Network không có khi self-host. Middleware chạy trên Node.js runtime, không phải Edge.
2. **ISR với multiple instances** — Nếu deploy 3 Docker containers, mỗi container có cache riêng → inconsistent ISR. Cần shared cache layer (Redis) để sync.
3. **Automatic scaling** — Vercel tự scale. Docker cần Kubernetes hoặc ECS setup.
4. **Analytics và Speed Insights** — Vercel-specific, không khả dụng.
5. **Automatic SSL, CDN, DDoS protection** — Phải tự setup (Cloudflare, Nginx, etc.).

**Về ISR với multiple instances — đây là vấn đề nghiêm trọng nhất:**
```
Container A: Article "ABC" → Cache HIT (fresh)
Container B: Article "ABC" → Cache MISS (chưa có cache)
→ Load balancer gửi user đến B → User thấy stale content

Fix: Dùng Redis làm shared cache, hoặc sticky sessions trên load balancer
```

**Bottom line (Tóm lại):** Self-host được, nhưng phải tự build nhiều infrastructure mà Vercel cung cấp sẵn. Tính toán TCO (Total Cost of Ownership) bao gồm: engineer time setup infrastructure, monitoring, on-call.

Dưới đây là bảng trade-off rõ ràng giữa **Next.js self-host (Docker)** và **Vercel (managed platform)** trong bối cảnh có ISR + Image Optimization:

---

**📊 Trade-off: Vercel vs Self-host (Docker)**

| Tiêu chí                              | Vercel (Managed)                 | Self-host (Docker / Kubernetes)                  |
| ------------------------------------- | -------------------------------- | ------------------------------------------------ |
| **Setup ban đầu**                     | Rất nhanh (zero infra)           | Phức tạp (Docker, Load Balancer, CDN, CI/CD)                |
| **Scale traffic**                     | Auto-scale                       | Tự build (K8s, ECS, autoscaling)                 |
| **ISR (cache)**                       | Shared global cache (consistent) | Local cache per instance → dễ lệch dữ liệu       |
| **ISR consistency**                   | ✔ Consistent across regions      | ❌ Cần Redis / shared storage để sync             |
| **Image Optimization (`next/image`)** | Edge optimized (CDN built-in)    | Node server xử lý → tốn CPU, cần CDN hỗ trợ      |
| **Edge Functions / Middleware**       | Edge network native              | Chạy Node.js server (không phải Edge thật)       |
| **Latency global**                    | Tối ưu sẵn (Edge network)        | Phụ thuộc setup CDN (Cloudflare, AWS CloudFront) |
| **Observability** - Khả năng quan sát                    | Built-in analytics, logs         | Tự setup (Prometheus, Grafana, ELK, etc.)        |
| **Deploy speed**                      | Git push → auto deploy           | CI/CD pipeline tự xây                            |
| **Infrastructure control**            | Hạn chế                          | Toàn quyền control                               |
| **Cost model**                        | Đắt khi scale lớn                | Rẻ hơn ở scale lớn nhưng tốn engineer time       |
| **Vendor lock-in**                    | Cao                              | Thấp                                             |
| **Maintenance burden (gánh nặng bảo trì)**                | Gần như không                    | Cao (on-call, infra, scaling, incidents - sự cố)         |

**⚠️ Điểm khác biệt quan trọng nhất**

**1. ISR consistency problem (self-host)**

**1.1 ISR cache nằm ở đâu?**

Khi self-host: 
- cache nằm trong `.next/cache` hoặc trên filesystem của container instance.
- Mỗi container instance có ISR cache riêng → **không shared** giữa instances

Ví dụ:

```text
Multi-instance deployment
Container A → có ISR cache version 1
Container B → có ISR cache version 2
Container C → cache miss (chưa có cache)
```

👉 Kết quả:

* user mỗi request có thể thấy data khác nhau (inconsistent UI)

✔ Fix:

* Redis shared cache

**1. Pattern cho MVP: Pattern A — Full-Stack Monolith (với điều chỉnh)**

> **"Tôi vẫn dùng Full-Stack Monolith để ship nhanh, nhưng business logic được đặt trong `lib/api/*`, API contract (`/api/v1/*`) tồn tại từ đầu. Khi mobile app xuất hiện, tôi chỉ cần expose API hiện có thay vì rewrite toàn bộ backend."**

Chọn Pattern A vì:
- Team 3 người (2 non-technical + bạn) — không có backend team riêng, không có bandwidth cho multiple services
- Timeline 3 tháng cho MVP — cần iteration speed tối đa, không có thời gian cho API design, contract negotiation
- Budget hạn chế — một repo, một deploy target, một monitoring stack giảm operational overhead
- Restaurant menu/order logic không quá phức tạp ở giai đoạn đầu — CRUD operations, simple auth, Stripe payment

**Điều chỉnh quan trọng để prepare cho mobile app:**
- Không dùng Server Actions cho mutations từ đầu — hoặc dùng nhưng wrap trong abstraction layer
- Tạo `/app/api/internal/*` Route Handlers ngay từ MVP — mobile team sẽ consume API này sau
- Database schema design với REST resource mindset: `menus`, `orders`, `items` tables có clear relationships, không denormalize quá sớm

**2. Chuẩn bị cho mobile app migration:**

- **Tách API layer ra khỏi UI ngay từ đầu:**
  - `lib/api/` chứa business logic callable từ cả Server Components lẫn Route Handlers
  - Route Handlers (`/api/v1/menus`, `/api/v1/orders`) gọi `lib/api/` functions
  - Server Actions (nếu dùng) chỉ là thin wrapper call `lib/api/` — không chứa business logic

- **Auth strategy future-proof:**
  - Dùng JWT access token + refresh token pattern từ đầu
  - Web dùng httpOnly cookie, mobile sẽ dùng Bearer token — cùng auth middleware
  - Không dùng session-based auth (NextAuth mặc định) vì khó integrate với mobile

- **Database:**
  - PostgreSQL trên Supabase hoặc Neon — có connection pooler built-in (PgBouncer/Supabase pooler)
  - Schema thiết kế với API contract mindset: tables có clear IDs, timestamps, foreign keys
  - Dùng Prisma với migration files — schema evolution trackable

- **Code organization prepare cho tách:**
  ```
  lib/
  ├── api/           ← Business logic, callable từ anywhere
  │   ├── menus.ts
  │   ├── orders.ts
  │   └── auth.ts
  ├── db.ts          ← Prisma singleton
  └── types/         ← Shared types
  app/
  ├── api/v1/        ← REST API cho mobile (từ đầu)
  │   ├── menus/
  │   └── orders/
  └── (dashboard)/   ← UI pages
  ```

**3. Vercel vs self-host và budget cuts:**

- **Chọn Vercel cho MVP** — lý do:
  - Zero DevOps overhead: không cần setup CI/CD, SSL, CDN, load balancer
  - Automatic preview deployments cho mỗi PR — founders (non-technical) có thể review features qua preview URL
  - ISR/SSG built-in — restaurant menu pages có thể static render, cải thiện performance không cần effort
  - Scale tự động — nếu có traffic tăng đột biến (spike) (ví dụ: được đăng trên Hacker News), không cần manual intervention

- **Điểm sẽ cắt giảm để tiết kiệm budget:**
  - **Vercel Pro ($20/month) thay vì Enterprise** — Pro plan đủ cho MVP, giới hạn team members và build minutes
  - **Supabase Free tier** cho PostgreSQL — 500MB database, 2GB bandwidth, đủ cho MVP vài trăm restaurants
  - **Không dùng paid monitoring** (Datadog, Sentry paid) — Vercel Analytics free + console logs đủ cho giai đoạn đầu
  - **Stripe test mode** cho payment — không cần production Stripe account cho MVP
  - **Không dùng email service paid** — Resend free tier (100 emails/day) đủ cho auth emails
  - **Self-host khi nào:** Khi monthly Vercel bill vượt quá $100-200 (tương đương 1-2 ngày engineer time setup infrastructure), hoặc khi cần features Vercel không có (custom caching, on-premise compliance)

- **Trade-off của việc cắt giảm:**
  - Không có advanced monitoring → bugs khó detect, nhưng với user base nhỏ, manual testing + user feedback đủ
  - Supabase free tier limits → nếu grow nhanh, cần upgrade hoặc migrate DB — nhưng đó là good problem to have
  - Không có CDN custom → Vercel Edge Network đủ cho global delivery ở scale MVP

<details>
<summary>Đáp án</summary>

**1. Pattern: Pattern C — BFF (Backend for Frontend) với yếu tố Pattern B**

Dấu hiệu xác nhận:

- `"separate backend team running NestJS microservices (8 services)"` → Backend riêng biệt, không phải monolith trong Next.js
- `"consuming gRPC/REST APIs via our API gateway"` → Next.js không gọi trực tiếp microservices mà qua API gateway — đây là aggregation/orchestration layer
- `"API gateway"` → Gateway pattern thường đi kèm BFF; Next.js layer có thể là BFF hoặc consumer của gateway
- `"8 services"` → Đủ nhiều services để cần aggregation; dashboard cần data từ nhiều nguồn
- `"identity service (OAuth2/JWT)"` → Auth externalized, Next.js chỉ handle token management
- `"ECS Fargate + ALB, not Vercel"` → Self-hosted, Docker-based — không dùng Vercel serverless features

Tại sao không phải Pattern B thuần:
- Pattern B thuần: Next.js chỉ là presentation layer, proxy/pass-through requests
- Ở đây có `"performance (users refresh dashboards every 30 seconds)"` + `"real-time data"` → Next.js layer likely cần caching, data transformation, WebSocket management — đây là BFF responsibilities, không phải pure frontend

**2. Day-to-day của role:**

- **60% UI development:** Build dashboard components với real-time data (market charts, portfolio tables, transaction feeds) — dùng Tailwind + shadcn hoặc tương đương
- **20% API integration:** Viết typed clients cho gRPC/REST services, handle data transformation từ microservices format → UI format, error handling cho distributed systems
- **15% Performance optimization:** Implement caching strategies (React Server Components cache, SWR/React Query stale-while-revalidate), optimize re-renders cho 30-second refresh cycles, WebSocket connection management
- **5% Infrastructure/DevOps:** GitHub Actions workflows, ECS deployment troubleshooting, ALB routing config, monitoring dashboards

Ví dụ một task thực tế:
> "Build portfolio dashboard page: gọi `/api/gateway/portfolio` (aggregate từ `user-service`, `holding-service`, `market-data-service`), transform response thành chart data format, implement auto-refresh every 30s với SWR + fallback to cached data, handle WebSocket stream cho real-time price updates."

**3. 3 technical challenges chính:**

- **Challenge 1: Distributed auth + token management**
  - OAuth2 flow với identity service: handle authorization code → access token + refresh token
  - Token refresh không block UI (background refresh trước khi expire)
  - WebSocket connections cần auth — gửi JWT qua connection handshake, handle re-auth khi token refresh
  - CORS và secure cookie handling giữa Next.js (ECS) và identity service

- **Challenge 2: Real-time data + performance ở scale**
  - WebSocket streams cho market data: connection pooling, reconnection logic, message deduplication
  - Dashboard refresh every 30s nhưng users không muốn see loading state → implement optimistic UI + stale-while-revalidate
  - Memory leak risk: WebSocket subscriptions không cleanup khi component unmount → dashboard chạy 8 tiếng liên tục sẽ crash
  - ECS Fargate = container restart → WebSocket state lost → cần reconnect + replay logic

- **Challenge 3: Microservices failure resilience**
  - 8 services = nhiều points of failure. Dashboard cần data từ 3-4 services → một service down không nên crash cả page
  - Implement circuit breaker pattern ở BFF layer: nếu `market-data-service` timeout, show cached data + warning banner thay vì error page
  - Partial data rendering: portfolio value có thể tính được ngay cả khi `news-service` down
  - Graceful degradation: real-time stream fallback sang polling nếu WebSocket server unavailable

**4. Câu hỏi phỏng vấn nên hỏi và đáp án tham khảo:**

- **"API gateway là internal BFF layer hay external product (Kong, AWS API Gateway)? Next.js layer gọi gateway trực tiếp hay qua internal network?"**
  - Đáp án mong đợi: Gateway là NestJS service internal, Next.js gọi qua internal VPC (không qua internet). Quan trọng để biết latency và security model.

* **"Dashboard data được cache ở đâu — React Query/SWR client-side, Next.js server cache, hay Redis ở gateway layer?"**
  - Đáp án mong đợi: **Multi-layer caching** — React Query/SWR dùng *stale-while-revalidate* ở client để tối ưu UX và giảm loading flicker. Ở server layer, Next.js vẫn có thể dùng **`unstable_cache` hoặc fetch cache (revalidate, tags)** để cache dữ liệu theo request/SSR, dù hiện tại **Next.js 16 đang khuyến nghị `use cache` + Cache Components thay thế dần `unstable_cache`** (tuy nhiên `unstable_cache` vẫn dùng được trong nhiều codebase production hiện tại). Ở infrastructure layer, Redis tại gateway (NestJS) dùng để cache cross-container và giảm load microservices (đặc biệt cho market data, portfolio aggregation). Quan trọng nhất là hiểu **cache invalidation cascade**: khi dữ liệu thay đổi (ví dụ sell stock), cần propagate invalidation từ Redis → Next.js cache (`revalidateTag` / invalidate cache boundary) → client cache (React Query invalidate query key), để đảm bảo consistency nhưng vẫn giữ performance cao.


- **"WebSocket streams từ đâu — gateway maintain WebSocket connection hay Next.js proxy đến dedicated WebSocket service?"**
  - Đáp án mong đợi: **WebSocket thường được tách ra thành một dedicated realtime service (ví dụ Socket.io / native WS server) hoặc một module realtime riêng trong backend, thay vì để Next.js xử lý.** Next.js App Router chủ yếu phục vụ request/response (SSR, RSC) nên không phù hợp để giữ long-lived connections. Vì vậy kiến trúc phổ biến là browser (qua Next.js UI) sẽ connect trực tiếp tới WebSocket service (hoặc qua ALB/NLB), còn gateway chỉ xử lý REST/gRPC aggregation và auth. WebSocket service sẽ chịu trách nhiệm maintain connection, fan-out market data stream, reconnect logic và scaling pub-sub. Quan trọng là tách biệt rõ HTTP layer (gateway/Next.js) và streaming layer (WebSocket service) để tránh overload gateway và đảm bảo khả năng scale realtime cho dữ liệu thị trường

- **"Team có 4 frontend engineers — ai làm BFF logic? Frontend team maintain API routes hay có platform/backend team hỗ trợ?"**
  - Đáp án mong đợi: Frontend team own Next.js layer bao gồm API routes (BFF). Backend team own microservices. Clear contract: OpenAPI/Swagger specs. Quan trọng để biết scope và on-call responsibility.

- **"Error handling strategy cho microservices — có standardized error format không? Frontend có retry logic hay gateway handle retry?"**
  - Đáp án mong đợi: Gateway trả về unified error format (RFC 7807 Problem Details hoặc custom). Frontend có exponential backoff retry cho idempotent requests. Non-idempotent (POST orders) không retry tự động.

- **"Monitoring và alerting — có distributed tracing (OpenTelemetry, Jaeger) không? Frontend có visibility vào backend latency không?"**
  - Đáp án mong đợi: Có tracing cross-service (request ID propagate từ Next.js → gateway → microservices). Frontend dùng Datadog/New Relic hoặc CloudWatch. Biết latency bottleneck ở layer nào.

**5. Skills cần mà bài này chưa đề cập:**

- **gRPC client development:** `protobuf` types, gRPC-Web hoặc gRPC over HTTP/2, handling streaming responses. JD mention gRPC nhưng bài chỉ cover REST patterns.
- **WebSocket/Socket.io production:** Connection management, heartbeat, reconnection with exponential backoff, subscription multiplexing, horizontal scaling với Redis adapter.
- **Financial domain knowledge:** Understanding of portfolio valuation, P&L calculation, market data tick formats, regulatory requirements (nếu fintech có compliance constraints).
- **Advanced React performance:** `useDeferredValue`, `useTransition`, concurrent features, virtualized lists cho large datasets (10K+ transactions), canvas/WebGL charts (TradingView, D3).
- **AWS specifics:** ECS service discovery, ALB path-based routing, CloudWatch logs/metrics, IAM roles cho service-to-service auth, Secrets Manager.
- **Security:** XSS prevention với dynamic market data (user-generated content risk), CSRF protection cho state-changing operations, Content Security Policy cho financial apps.
- **Testing distributed systems:** Mocking microservices (MSW, WireMock), contract testing (Pact), E2E với real-time data (Cypress/Playwright với WebSocket intercept).

## 13. Summary

### Bản Đồ 5 Patterns

```
Pattern A — Full-Stack Monolith
  Nhận diện:  prisma/ + actions/ + Vercel deployment
  Tốt cho:    Startup, MVP, small team
  Risk:       DB connections, không reusable API

Pattern B — Decoupled Frontend
  Nhận diện:  lib/api/client.ts, KHÔNG có prisma/
  Tốt cho:    Công ty có existing backend, enterprise
  Risk:       Type duplication, coordination overhead

Pattern C — BFF
  Nhận diện:  /services/*.ts, Promise.all trong API routes
  Tốt cho:    Microservices architecture
  Risk:       BFF là single point of failure

Pattern D — Content Platform
  Nhận diện:  generateStaticParams nhiều, CMS client, revalidate webhook
  Tốt cho:    Media, docs, marketing, blog
  Risk:       ISR invalidation complexity, multi-instance cache

Pattern E — Multi-tenant SaaS
  Nhận diện:  middleware.ts detect subdomain, tenant DB queries
  Tốt cho:    B2B SaaS, platform products
  Risk:       Data isolation, tenant cache pollution
```

### Câu Hỏi Phải Hỏi Khi Join Công Ty Mới

```
1. "Backend là gì? Team backend riêng hay full-stack?"
   → Xác định Pattern B/C (có backend riêng) vs A (monolith)

2. "Deploy như thế nào? Vercel hay Docker?"
   → Quyết định nhiều thứ: Middleware có dùng Edge không? ISR setup?

3. "Có mobile app không hay chỉ web?"
   → Pattern A có thể không đủ nếu mobile app đang planned

4. "Database access như thế nào?"
   → ORM nào? Connection pooling? (Quan trọng cho serverless)

5. "Migration plan từ Pages Router sang App Router đang ở giai đoạn nào?"
   → Biết mình sẽ làm việc với code era nào (bài M0.2)
```

### Điều Quan Trọng Nhất Của Bài Này

Tutorial dạy bạn xây **một** thứ.

Production là bạn phải hiểu **tại sao** thứ đó được xây theo cách đó, và **khi nào** nên xây theo cách khác.

Biết 5 patterns này không phải để bạn "chọn đúng pattern". Đó là để bạn **đọc được context** khi bước vào bất kỳ codebase nào — hiểu những quyết định đã được đưa ra, tại sao, và trade-offs nào đang được chấp nhận.

Đó là tư duy của engineer, không phải coder.
